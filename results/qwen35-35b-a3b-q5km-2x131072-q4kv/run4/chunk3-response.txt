T1 — **`tofu plan`** calculates the execution plan, showing what changes will be made without applying them. **`tofu apply`** executes the changes required to reach the desired state. `plan` is safe for review; `apply` modifies infrastructure.

T2 — **State** is a JSON file mapping real-world resources to your configuration. **Remote state** is crucial for collaboration (preventing race conditions), locking, and security (storing secrets away from local disks).

T3 — **`count`** uses integer indices; **`for_each`** uses maps or sets. `for_each` is preferred when resource names are dynamic or strings, as it avoids index shifting issues when items are added/removed, ensuring stable resource IDs.

T4 — OpenTofu detects this via **state drift**. Running `tofu plan` compares the state file against the live infrastructure. If changes exist, they appear as "drift" in the plan. `tofu apply` can fix it if the config matches the desired state, or fail if the config conflicts.

T5 —
```hcl
resource "aws_instance" "web" {
  for_each = toset(["web1", "web2", "web3"])
  ami      = "ami-12345678"
  instance_type = "t2.micro"
  tags = {
    Name = each.key
  }
}
```

T6 — **`terraform_remote_state`** reads state from a backend directly (bypasses dependency graph). **Module Outputs** pass data through the module hierarchy. **Recommended:** Use Module Outputs. It enforces dependency tracking, improves readability, and avoids direct state file access which is brittle during refactoring.

T7 — **Why:** The `null_resource` provisioner runs asynchronously during apply. The output (reading the file) is computed before the provisioner finishes writing it. **Fix:** Add `depends_on = [null_resource.generate_kubeconfig]` to the resource consuming the kubeconfig output to ensure the file exists first.

T8 — **`tofu import`** CLI command manually adds existing resources to state. **Limitations:** Manual, error-prone, hard to version control. **`import` block (1.5+):** Declarative, version-controlled, allows importing during `tofu apply`, and integrates with the dependency graph.

T9 — A **variable** is an input parameter for a module or root configuration. An **output** is data exposed from a module to the caller. Variables are inputs; outputs are results.

T10 — A **provider** defines the API interaction for a cloud service (e.g., AWS, Azure). Without `tofu init`, providers are not downloaded, and `tofu plan` fails with "Provider not installed" errors.

T11 — The **`.terraform.lock.hcl`** file records exact provider versions and checksums. It should be committed to ensure reproducible builds and prevent unexpected provider upgrades that might break infrastructure.

T12 — **`tofu destroy`** removes all managed resources. To target a single resource: `tofu destroy -target=aws_instance.example`.

T13 — A **data source** reads data from an external system without creating or managing it. **Example:** `data "aws_ami" "ubuntu"` to get the latest AMI ID instead of hardcoding it.

T14 — **Variables** are inputs from outside the module. **Locals** are computed values within the module. Use **Locals** for intermediate calculations or complex expressions that don't need to be exposed as inputs.

T15 — **`lifecycle`** block controls resource behavior.
*   `create_before_destroy`: Creates new resource before deleting old (zero downtime).
*   `prevent_destroy`: Prevents accidental deletion (safety).
*   `ignore_changes`: Ignores drift on specific attributes (e.g., tags managed externally).

T16 — A **`moved`** block tracks resource identity changes during refactoring. It tells OpenTofu that a resource has moved to a new address without destroying and recreating it, preserving state and dependencies.

T17 — The **dependency graph** determines execution order. **Implicit** dependencies come from resource references. **Explicit** dependencies use `depends_on`. `depends_on` reduces parallelism by forcing sequential execution, which should be used sparingly to avoid slowing down plans.

T18 — Strategies:
1.  **`plan -refresh=false`**: Skips state refresh (faster, but risks stale state).
2.  **Split State**: Break large state into multiple files/modules (faster parallelism, more management overhead).
3.  **`plan -target`**: Plan only specific resources (faster, but partial view).
4.  **Optimize Providers**: Use batching or caching (faster, requires provider support).

T19 — **State locking** prevents concurrent writes. If a lock isn't released (e.g., CI crash), subsequent runs hang. **Break lock:** `tofu force-unlock <LOCK_ID>`. **Risk:** If another process is actually running, this can cause state corruption.

T20 — Steps:
1.  Backup local state file.
2.  Configure backend in `terraform.tfvars` or `init`.
3.  Run `tofu init -backend-config="bucket=my-bucket" -backend-config="key=prod.tfstate" -migrate-state`.
4.  Verify: Run `tofu state list` to confirm resources exist in S3, then `tofu plan` to ensure no drift.

A1 — **Playbook**: YAML file defining a set of plays/tasks. **Role**: Reusable structure for organizing tasks/files. **Task**: Single action executed on a host.

A2 — **`ansible_facts`** contains gathered system information. Access OS family via: `{{ ansible_facts['os_family'] }}`.

A3 — **`command`**: Executes without shell (no pipes/globs). **`shell`**: Executes with shell (supports pipes/globs). **`raw`**: Executes without setup (no Python module transfer), useful for early bootstrapping.

A4 — **Idempotency** means running a task multiple times yields the same result. **Idempotent:** `apt: name=nginx` (installs only if missing). **Not Idempotent:** `shell: echo "Hello" > /tmp/test` (overwrites file every time).

A5 —
```yaml
- name: Install htop
  package:
    name: htop
    state: present
  when: ansible_os_family == "Debian"

- name: Install htop and epel
  package:
    name:
      - epel-release
      - htop
    state: present
  when: ansible_os_family == "RedHat"
```

A6 — **Precedence (High to Low):**
1. Command line extra_vars (`-e`)
2. `set_fact` (Runtime)
3. Play vars
4. Host vars
5. Group vars
6. Role defaults (Lowest)

A7 — Structure:
1. Define inventory groups (e.g., `rke2_first`, `rke2_rest`).
2. Use `serial: 3` in the playbook to limit concurrency.
3. Use `wait_for` or `meta: flush_handlers` to ensure first batch completes before next batch starts.

A8 — **`ansible-vault`** encrypts sensitive data. To encrypt a single variable: `ansible-vault encrypt_string 'secret_value' --name 'my_secret'`.

A9 — **Inventory** defines hosts and groups. **Static:** Defined in files (INI/YAML). **Dynamic:** Generated via scripts/API (e.g., AWS EC2), updates automatically.

A10 — `ansible webservers -m shell -a "df -h"`

A11 — A **handler** is a task that runs only when notified by another task (e.g., `notify: restart service`). It differs from regular tasks by running asynchronously at the end of the play or immediately upon notification.

A12 — **`ansible.cfg`** configures Ansible behavior. Common settings: `remote_user`, `inventory`, `forks`, `host_key_checking`.

A13 — **`register`** stores task output in a variable. Example: `register: disk_info`. Use in subsequent task: `debug: msg="{{ disk_info.stdout }}"` or `when: disk_info.stdout == "..."`.

A14 — **`import_tasks`**: Static, evaluated at parse time (before execution). **`include_tasks`**: Dynamic, evaluated at runtime. `when` clauses in `import_tasks` are static; in `include_tasks`, they are dynamic.

A15 — **Ansible Galaxy** is a repository for roles/collections. Use `requirements.yml` to list dependencies, then run `ansible-galaxy install -r requirements.yml`.

A16 — **Connection plugins** manage how Ansible connects. **SSH:** Remote Linux/Unix. **Local:** Control node only. **Network_CLI:** Network devices (Cisco/Juniper).

A17 — **`delegate_to`** runs task on a different host. **`run_once`** runs task once across all hosts. Example: Gather facts from remote hosts, then write a config file to `localhost` once.

A18 — Strategies:
1. **Increase `forks`**: More parallel hosts (Tradeoff: CPU/Memory load).
2. **Enable `pipelining`**: Reduces SSH connections (Tradeoff: SSH config requirements).
3. **Fact caching**: Avoids re-gathering facts (Tradeoff: Stale facts).
4. **Mitogen**: Optimized transport (Tradeoff: Compatibility/Setup complexity).

A19 — **`{{ }}`**: Output variable. **`{% %}`**: Control logic (loops/if). **`{# #}`**: Comment.
Snippet:
```jinja2
upstream backend {
{% for server in backends %}
    server {{ server.ip }} weight={{ server.weight }};
{% endfor %}
}
```

A20 — Implementation:
1. **Serial:** `serial: 5` (batches of 5).
2. **Pre-task:** Deregister from Load Balancer.
3. **Deploy:** Update app.
4. **Health Check:** `uri` module to verify endpoint.
5. **Post-task:** Register with Load Balancer.
6. **Rollback:** Use `block/rescue`. If health check fails in `rescue`, revert to previous version and re-register.
