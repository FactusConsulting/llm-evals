AA1 — **Monolith vs. Microservices:** A monolith is a single, unified codebase deployed as one unit. Microservices are a collection of small, independent services deployed separately. **Problems introduced by microservices:** 1. **Network Latency:** Communication occurs over the network rather than in-process calls. 2. **Distributed Data Consistency:** Managing transactions across multiple databases is complex (no ACID across services). 3. **Operational Complexity:** Requires orchestration (Kubernetes), distributed tracing, and service discovery, unlike a single deployment pipeline.

AA2 — **Sync vs. Async:** Synchronous communication blocks the sender until the receiver responds (request/response). Asynchronous communication allows the sender to continue without waiting (fire-and-forget). **Examples:** Synchronous: HTTP/REST or gRPC. Asynchronous: Apache Kafka or RabbitMQ.

AA3 — **API Gateway vs. Reverse Proxy:** A reverse proxy routes traffic to backend servers and handles SSL termination. An API Gateway adds business logic like authentication, rate limiting, and API versioning. **Replacement:** An API Gateway can replace a reverse proxy for external traffic, but internal service-to-service communication often still requires a lightweight L4 proxy (like Envoy in a service mesh) for low-level routing and health checks that a Gateway might not handle efficiently.

AA4 — **Saga Pattern:** A Saga manages distributed transactions by breaking them into a sequence of local transactions. **Choreography:** Services emit events that trigger other services (loose coupling, harder to debug). **Orchestration:** A central coordinator tells services what to do (tighter coupling, easier flow control). **Appropriateness:** Use Choreography for simple flows; use Orchestration for complex workflows requiring compensation logic and visibility.

AA5 — **CQRS:** Command Query Responsibility Segregation separates read (Query) and write (Command) operations into different models. **Pairing with Event Sourcing:** Event Sourcing stores state changes as an immutable log. **Problem Solved:** It allows scaling reads and writes independently (e.g., denormalized read models for fast queries) and provides a complete audit trail, which a traditional CRUD model (single table per entity) does not.

AA6 — **Circuit Breaker:** A pattern to prevent cascading failures by stopping calls to a failing service. **States:** Closed (normal operation), Open (fail fast), Half-Open (testing recovery). **Difference from Retry:** Retry attempts to recover immediately; Circuit Breaker stops attempts to prevent overload. **Use Together:** Use Retry for transient network glitches, but wrap it in a Circuit Breaker to stop hammering a service that is persistently down.

AA7 — **Tenant Isolation Models:**
1.  **Shared DB/Schema:** Lowest cost, highest security risk (data leakage), lowest complexity, high noisy-neighbor risk.
2.  **Shared DB/Schema per Tenant:** Medium cost, medium security, medium complexity, medium noisy-neighbor risk.
3.  **Separate DB per Tenant:** Highest cost, highest security, highest complexity, lowest noisy-neighbor risk.

AA8 — **Consistency & Inventory:** Inventory requires **Strong Consistency** to prevent overselling. **Architecture:** Use an ACID-compliant Relational Database (e.g., PostgreSQL). **Patterns:** Implement Optimistic Locking (version columns) or Distributed Locking (Redis/Zookeeper) to serialize inventory updates. Eventual consistency is insufficient here.

AA9 — **Sidecar Proxy:** A sidecar is a proxy deployed alongside a service container. **Envoy/Istio:** Envoy acts as the data plane (traffic proxy); Istiod is the control plane (config management). **mTLS:** Application-level TLS secures data in transit, but mTLS between sidecars verifies **service identity** (who is talking to whom) and encrypts traffic without application code changes.

AA10 — **Strangler Fig Pattern:** Incrementally migrate functionality from a legacy system to new services. **Strategy:** Deploy an API Gateway in front of the .NET Framework monolith. Route specific API paths (e.g., `/api/inventory`) to new .NET 8 microservices. Gradually migrate modules until the monolith is "strangled" and removed.

AA11 — **12-Factor App:** Methodology for building cloud-native apps. **Factors:** 1. Codebase (one repo), 2. Dependencies (explicitly declared), 3. Config (env vars), 4. Backing Services (treat as attached resources), 5. Build/Run/Deploy (separate), 6. Processes (stateless). **Importance:** Ensures portability, scalability, and rapid deployment in cloud environments.

AA12 — **Event-Driven vs. Request-Driven:** **Coupling:** Event (Low) vs. Request (High). **Scalability:** Event (High) vs. Request (Moderate). **Debugging:** Event (Hard) vs. Request (Easy). **Ordering:** Event (Possible) vs. Request (Guaranteed). **Failure:** Event (Dead-letter queues) vs. Request (Timeouts). **Choice:** Use Event for decoupling/scaling; use Request for real-time, synchronous interactions.

AA13 — **Load Balancer:** Distributes traffic across servers. **L4 vs. L7:** L4 balances based on IP/Port (TCP/UDP). L7 balances based on HTTP headers/URL. **Use Case:** L7 is required for SSL termination or routing `/api` to one backend and `/static` to another.

AA14 — **Message Queue vs. Log Broker:** **MQ (RabbitMQ):** Tasks, delete after consume, low latency. **Log Broker (Kafka):** Event streaming, retain history, high throughput. **Choice:** Pick MQ for task queues (e.g., email sending); pick Kafka for audit logs or event sourcing.

AA15 — **Container vs. VM:** A container shares the host OS kernel but isolates user space. A VM virtualizes the hardware and runs a full OS. **Popularity:** Containers are lightweight, start instantly, and ensure consistent environments, making them ideal for microservices.

AA16 — **Stateless vs. Stateful:** Stateless services store no session data (e.g., Web Server). Stateful services store data (e.g., Database). **Scaling:** Stateless services are easier to scale horizontally because any instance can handle any request without data synchronization.

AA17 — **ORM:** Object-Relational Mapper maps objects to database tables. **Examples:** Entity Framework (.NET), Hibernate (Java). **Trade-offs:** ORM speeds development and reduces boilerplate but can introduce performance overhead and abstraction leaks compared to raw SQL.

AA18 — **Bulkhead Pattern:** Isolates resources (threads, connections) for specific services. **Implementation:** Use separate thread pools for each downstream dependency. If one service fails, it exhausts only its pool, preventing it from starving the entire application.

AA19 — **OT vs. CRDT:** **OT:** Server resolves conflicts (requires connectivity). **CRDT:** Mathematical guarantees allow convergence without a server. **Offline Support:** CRDT is superior for offline editing as it resolves conflicts locally. **Choice:** Choose CRDT for a system requiring robust offline editing.

AA20 — **Distributed Tracing:** **Propagation:** W3C Trace Context or B3 headers pass context across services. **Sampling:** Head-based (early decision) vs. Tail-based (late decision based on latency). **Storage:** Jaeger/Tempo. **Kafka:** Inject trace context into message headers so downstream consumers can continue the trace.

OP1 — **Hypervisor Types:** **Type 1 (Bare-metal):** Runs directly on hardware (e.g., VMware ESXi, Microsoft Hyper-V). **Type 2 (Hosted):** Runs on top of an OS (e.g., VirtualBox, VMware Workstation). Type 1 is for production; Type 2 is for development.

OP2 — **SAN vs. NAS:** **SAN:** Block storage (e.g., iSCSI, Fibre Channel), used for databases. **NAS:** File storage (e.g., NFS, SMB), used for file shares. **Choice:** Use SAN for high-performance DBs; use NAS for shared user files.

OP3 — **Network Architecture:** **3-Tier:** Core, Distribution, Access. **Leaf-Spine:** Preferred for modern DCs. **Problem Solved:** 3-Tier has oversubscription and latency at the distribution layer. Leaf-Spine provides non-blocking, low-latency connectivity between all servers.

OP4 — **Proxmox HA:** **Architecture:** 3+ nodes for quorum. **Storage:** Ceph or shared iSCSI. **Fencing:** Pacemaker STONITH to isolate failed nodes. **Failure:** If a node fails, VMs restart on remaining nodes automatically.

OP5 — **VLAN Trunking:** Carries multiple VLANs over one link. **802.1Q:** Adds a tag to frames. **Linux Config:** Create sub-interfaces (e.g., `eth0.10`) with `ip link add link eth0 name eth0.10 type vlan id 10`.

OP6 — **RAID Levels:** **RAID 1:** Mirror (50% cap). **RAID 5:** Striping + Parity (1 drive loss). **RAID 6:** Double Parity (2 drive loss). **RAID 10:** Mirror + Strip (50% cap, 2 drive loss). **Choice:** For 6 drives/DB, choose **RAID 10** for best write performance and fault tolerance.

OP7 — **DNS Architecture:** **Internal:** AD DNS for clients. **External:** Public DNS for web. **Split-Horizon:** Internal queries resolve internal IPs. **Laptop:** Connects via DHCP, queries internal DNS for AD, uses forwarders for external. **DNSSEC:** Signs external zone to prevent spoofing.

OP8 — **PKI Chain:** **Root CA:** Offline, trusted anchor. **Subordinate CA:** Issues end-entity certs. **Revocation:** CRL (list) vs. OCSP (real-time). **Compromise:** Subordinate = Revoke chain. Root = Catastrophic (trust lost for all).

OP9 — **WAN Consolidation:** **MPLS:** Reliable, expensive. **IPsec:** Cheap, variable. **SD-WAN:** Smart routing, cost-effective. **WireGuard:** Simple encryption. **Recommendation:** **SD-WAN** for 50 users/site with VoIP (prioritizes traffic, handles internet variability).

OP10 — **iDRAC/iLO/IPMI:** **BMC:** Out-of-band management hardware. **Critical:** Manage server when OS is down. **Exposed:** If exposed to internet, attackers can reset power or access BIOS (hardware compromise).

OP11 — **LVM vs. ZFS:** **LVM:** Logical volumes, snapshots copy-on-write. **LVM-Thin:** Overcommit storage. **ZFS:** Copy-on-write, checksums, snapshots instant/cheap. **Choice:** Use ZFS for data integrity; LVM for simple volume management.

OP12 — **Stretch Cluster:** **Design:** 2 sites + 1 tiebreaker (witness). **Ceph:** CRUSH rules distribute data. **Latency:** <5ms required. **Split-Brain:** If network fails, both sites think they are primary. **Tiebreaker:** Needed to break quorum and prevent data corruption.

OP13 — **DHCP:** **DORA:** Discover, Offer, Request, Acknowledge. **Unavailable:** Client assigns APIPA address (169.254.x.x) and cannot reach network resources.

OP14 — **Managed vs. Unmanaged:** **Managed:** VLANs, ACLs, SNMP, Port Mirroring. **Unmanaged:** Plug-and-play. **Enterprise:** Managed switches are critical for segmentation, security, and monitoring.

OP15 — **NTP:** **Time Sync:** Critical for logs, auth, clustering. **Drift:** Clock skew causes Kerberos auth failures, log confusion, and database replication errors.

OP16 — **Physical vs. VM vs. Container:** **Physical:** Bare metal. **VM:** Virtualized OS. **Container:** Virtualized App. **Bare Metal:** Use for high-performance DBs or low-latency workloads where virtualization overhead is unacceptable.

OP17 — **UPS vs. Generator:** **UPS:** Short-term battery power. **Generator:** Long-term fuel power. **ATS:** Automatic Transfer Switch switches power from Grid to Generator when grid fails. **Need Both:** UPS bridges the gap until generator starts.

OP18 — **Active-Passive vs. Active-Active:** **Active-Passive:** One node handles load. **Active-Active:** Both handle load. **PostgreSQL HA:** Pacemaker/Corosync manages VIP. **Fencing:** STONITH ensures failed node is powered off to prevent split-brain.

OP19 — **Backup Strategy:** **3-2-1:** 3 copies, 2 media, 1 offsite. **Immutable:** Protects against ransomware. **Test:** Regularly restore to verify RPO/RTO. **Backup vs. DR:** Backup is data copy; DR is the process of restoring operations after a disaster.

OP20 — **802.1X:** **Roles:** Supplicant (client), Authenticator (switch), Server (RADIUS). **EAP:** TLS (certs) vs. PEAP (user). **Fail:** Device placed in quarantine VLAN. **MAB:** Switch sends MAC address to RADIUS for devices without 802.1X support.
