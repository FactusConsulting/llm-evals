CL1 — **IaaS** provides virtualized computing resources (e.g., AWS EC2). **PaaS** provides a platform for developing/running apps without managing infrastructure (e.g., AWS Elastic Beanstalk). **SaaS** delivers software over the internet (e.g., Microsoft 365).

CL2 — **Shared Responsibility:** Provider secures the cloud (hardware, hypervisor); Customer secures what's in the cloud (OS, data, config). **IaaS VM:** Provider secures physical host; Customer secures OS/App. **Managed K8s:** Provider secures control plane; Customer secures nodes/workloads. **SaaS (M365):** Provider secures everything; Customer secures identity/access.

CL3 — **VPC:** Isolated cloud network. **Subnet:** IP range within VPC. **Security Group:** Stateful firewall at instance level (allow rules). **NACL:** Stateless firewall at subnet level (allow/deny rules). **Layering:** VPC contains Subnets; Subnets contain Instances protected by SGs; NACLs filter traffic entering/leaving Subnets. **Stateful:** SG remembers return traffic. **Stateless:** NACL requires explicit rules for both directions.

CL4 — **Landing Zone:** Secure baseline environment. **OUs:** Group accounts by function/environment. **Guardrails:** Preventive controls (e.g., deny public S3). **Shared Networking:** Central hub for inter-VPC traffic. **Centralized Logging:** Aggregated audit trails. **Single-Account Risk:** High blast radius, billing opacity, compliance violations, lack of isolation.

CL5 — **IaC:** Managing infra via code. **Terraform/OpenTofu:** Multi-cloud, declarative, stateful. **Pulumi:** Multi-cloud, imperative (general languages). **CloudFormation/Bicep:** Native to AWS/Azure, tightly integrated. **Crossplane:** K8s-native control plane for cloud resources. **Choice:** Terraform for multi-cloud; Bicep/CFN for AWS/Azure specific; Crossplane for K8s-centric teams.

CL6 — **HPA:** Scales pods based on CPU/Memory. **VPA:** Adjusts pod resource requests/limits. **Cluster Autoscaler:** Adds/removes nodes. **KEDA:** Scales based on external events (e.g., queue length). **Use KEDA:** When scaling is driven by event triggers (Kafka, SQS) rather than resource metrics.

CL7 — **DR Strategy:** Define RPO (data loss tolerance) and RTO (downtime tolerance). **Active-Active:** Both regions serve traffic (low RTO, high cost). **Active-Passive:** One serves, one waits (higher RTO, lower cost). **DB:** Async (performance) vs Sync (consistency). **DNS:** Route53 failover routing. **State:** Shared storage or replicated state. **Pilot Light:** Core infra running, data restored on demand. **Warm Standby:** Scaled-down version running. **Active-Active:** Full redundancy.

CL8 — **GitOps:** Git as single source of truth for infra/app state. **ArgoCD:** Push-based (watch Git, sync cluster). **Flux:** Pull-based (cluster pulls Git). **Secrets:** Sealed Secrets (encrypted in Git), External Secrets Operator (fetch from Vault/AWS Secrets Manager). **Drift:** GitOps controller detects manual changes and reverts them to match Git state.

CL9 — **Architecture:** Use **StatefulSet** for stable IDs. **PV/PVC:** Use `local` storage class for low latency. **Headless Service:** For stable DNS names. **Init Containers:** For config setup. **ConfigMaps:** Mount as volumes for config files. **PDB:** Ensure availability during updates. **Sticky Sessions:** Via Service type or annotations.

CL10 — **SA:** K8s identity for pods. **Workload Identity:** Maps K8s SA to Cloud IAM role. **IRSA:** AWS specific implementation of workload identity. **Mounting Creds:** Bad practice (risk of leakage, rotation difficulty). **Federation:** Uses OIDC tokens to exchange K8s identity for temporary cloud credentials without long-lived secrets.

CL11 — **Object (S3):** Unstructured data, high durability (11 9s), low latency. **Block (EBS):** Attached to VM, low latency, persistent. **File (EFS):** Shared file system, network latency. **Use Case:** Object for backups/media, Block for DBs/OS, File for shared app configs. **Durability:** Object (11 9s), Block (99.999%+), File (99.99%).

CL12 — **Zero Trust:** Verify every request. **Network Policies:** Restrict pod-to-pod traffic. **Service Mesh:** mTLS for encryption/auth. **Pod Identity:** Workload identity. **Admission Controllers:** Enforce policies (OPA/Kyverno). **Image Signing:** Verify integrity. **Runtime Security:** Detect anomalies (Falco). **Diff:** Traditional relies on perimeter; Zero Trust assumes breach and verifies identity/context.

CL13 — **CDN:** Distributed network caching content at edge. **Cloud Services:** AWS CloudFront, Azure CDN. **Use CDN:** For static assets, global latency reduction, DDoS mitigation. **Direct Origin:** For dynamic content or when edge caching isn't needed.

CL14 — **Region:** Geographic area with multiple AZs. **AZ:** Isolated data center within a region. **Multi-AZ:** Protects against data center failure. **Zonal:** EBS (tied to AZ). **Regional:** ELB (spanning AZs).

CL15 — **Managed DB:** RDS/Cloud SQL. **Provider Handles:** Patching, backups, HA, failover, monitoring. **Customer Handles:** Schema, queries, user access.

CL16 — **IAM:** Users (people), Groups (collections), Roles (temporary access), Policies (permissions). **Least Privilege:** Grant only necessary permissions. **Apply:** Attach policies to roles/groups, avoid root/admin usage, use MFA.

CL17 — **Serverless:** Event-driven, no server management. **Compare:** Lambda (AWS), Functions (Azure), Cloud Functions (GCP). **Cold Start:** Latency when initializing new instances. **Poor Fit:** Long-running processes, stateful apps, predictable high throughput.

CL18 — **Ingress:** L7 routing rules. **Gateway API:** Extensible, standardized API. **LB:** L4 load balancing. **Mesh:** Sidecar proxy routing. **TLS:** Terminate at LB (Edge), Ingress Controller, or Mesh. **Gateway API:** Use for complex routing, multi-cluster, or when Ingress limitations are hit.

CL19 — **Strategy:** **RI/SP:** Commit to usage for discount. **Spot:** Use for fault-tolerant workloads. **Right-sizing:** Match instance to actual usage. **Storage:** Tier infrequent data. **FinOps:** Cost visibility, accountability. **Showback:** Charge teams for usage. **Tools:** AWS Cost Explorer, CloudHealth, Kubecost.

CL20 — **Strategy:** **Abstraction:** Kubernetes, Terraform. **Networking:** Direct Connect/ExpressRoute. **Identity:** Federation (OIDC). **Trade-offs:** Multi-cloud increases complexity/cost. **Justified:** Regulatory compliance, extreme redundancy. **Unnecessary:** If single-cloud provider meets all needs (simpler, cheaper).

OT1 — **Purdue Model:** ISA-95 standard. **Level 0:** Physical Process (Sensors/Actuators). **Level 1:** Basic Control (PLCs/Relays). **Level 2:** Supervisory Control (SCADA/HMI). **Level 3:** Operations (MES/Plant Ops). **Level 4:** Enterprise (ERP/Business).

OT2 — **IT:** Data/Information systems. **OT:** Physical process control. **Patching:** IT patches frequently; OT requires stability, legacy OS, and downtime is costly/safety-critical.

OT3 — **DMZ:** Buffer zone between IT and OT. **Traffic:** Historian pulls from SCADA (OT to DMZ), DMZ pushes to IT. **Direct L4-L1:** Dangerous because it allows lateral movement from corporate network directly to control devices, risking physical disruption.

OT4 — **Modbus TCP:** L2/L7, Level 1-2. **OPC UA:** L7, Level 2-3. **EtherNet/IP:** L2/L7, Level 1-2. **PROFINET:** L2, Level 1-2. **MQTT:** L7, Level 3-4.

OT5 — **Data Diode:** Hardware enforcing one-way data flow. **Firewall:** Two-way filtering. **Use Diode:** When data must flow out but no control commands can flow in (e.g., export to cloud). **Limitation:** No feedback loop or remote configuration from secure side.

OT6 — **Flow:** PLC -> Historian -> Edge Gateway -> MQTT Broker -> Cloud. **Security:** DMZ between Edge and Cloud. Encrypt traffic (TLS). Isolate Edge Gateway from direct internet access.

OT7 — **Zones:** Logical grouping of assets with similar security requirements. **Conduits:** Controlled paths between zones. **Model:** Corporate (Zone 4) <-> DMZ (Zone 3) <-> SCADA (Zone 2) <-> PLC/SIS (Zone 1). Firewalls between zones. Engineering workstations in Zone 3.

OT8 — **Strategy:** VLANs per production line. Firewalls between lines and DMZ. Switches with port security. **Remote Access:** Jump host in DMZ with MFA. **Transition:** Phased rollout during maintenance windows, parallel testing, ensure no single point of failure.

OT9 — **SIS:** Independent safety system (stops process safely). **BPCS:** Basic process control. **IEC 61511:** Safety lifecycle standard. **Air-gap:** Prevents cyber attacks from disabling safety systems. **Ransomware on SIS:** Safety failure, potential injury, environmental damage, plant shutdown.

OT10 — **PLC:** Logic controller for machines. **DCS:** Distributed control for continuous processes. **SCADA:** Supervisory monitoring. **Coexist:** Yes, PLCs often feed SCADA; DCS may contain PLCs.

OT11 — **Architecture:** Jump host in DMZ. MFA required. Session recording enabled. Time-limited vendor credentials. **Direct RDP:** Unacceptable because it bypasses audit, allows lateral movement, and exposes OT assets to IT threats.

OT12 — **Plan:** Isolate compromised segment. Activate manual ops. Verify SIS integrity. **Monitoring:** IDS for OT protocols. **Operator Role:** Monitor physical safety, execute manual procedures, communicate status. **Coordination:** IT/OT joint response team.

OT13 — **HMI:** Operator interface for monitoring/control. **Role:** Visualize process, allow manual overrides. **Diff:** Engineering workstation is for programming/configuring logic, not day-to-day operations.

OT14 — **Historian:** Time-series database for process data. **Importance:** Trend analysis, compliance, troubleshooting. **Plant:** Real-time ops (Level 3). **Enterprise:** Long-term analytics/reporting (Level 4).

OT15 — **Ladder Logic:** Graphical programming language mimicking relay circuits. **Device:** PLC. **Dominant:** Deterministic execution, safety certification, legacy familiarity, ease of maintenance for electricians.

OT16 — **RTU:** Remote Terminal Unit. **Diff:** RTU optimized for comms and harsh environments; PLC optimized for local control logic. **Deploy RTU:** Remote sites (pipelines, water), limited connectivity.

OT17 — **Analog:** Continuous signal (voltage/current). **Digital:** Discrete (on/off). **4-20mA:** Preferred because 4mA represents "live zero" (detects wire break at 0mA).

OT18 — **Inventory:** Knowing all assets is critical for patching/vulnerability mgmt. **Tools:** Passive (packet sniffing), Active (scanning). **Active Risk:** Can crash legacy devices or trigger alarms in safety systems.

OT19 — **MITRE ICS:** Framework for ICS threats. **Tactics:** Initial Access (Phishing), Execution (Command), Persistence (Backdoor), Impact (Disruption), Defense Evasion (Obfuscation). **Diff:** Focuses on physical impact/process safety. **Use:** Map defenses to tactics to find gaps.

OT20 — **SOC:** Collect logs via Syslog/NetFlow (non-intrusive). **SIEM:** Correlate IT/OT logs. **Rules:** OT-specific (e.g., PLC write attempts). **Staff:** OT engineers for triage. **False Positives:** Tune rules to avoid production stoppages. **Firmware Change:** Alert -> Isolate -> Verify integrity -> Rollback -> Investigate root cause.
