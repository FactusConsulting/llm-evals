## Go (20 questions)

**G1 — Easy:** An array is a fixed-size value type `[N]T`. A slice is a dynamic view descriptor containing a pointer to an underlying array, length, and capacity. When appending to a slice at capacity, the Go runtime allocates a new, larger underlying array (usually doubling capacity), copies elements, and updates the slice header.

**G2:** `var x int` declares a variable with the zero value (`0`) and requires explicit type. `x := 0` is a short variable declaration that infers the type. You cannot use `:=` if the variable already exists in the same scope (redeclaration error) or outside a function body (must use `var`).

**G3:** A goroutine is a lightweight thread managed by the Go runtime, not the OS. The Go scheduler uses the GMP model: **G** (Goroutine), **M** (OS Thread), **P** (Processor, logical CPU context). The scheduler multiplexes many Gs onto fewer Ms, handling blocking syscalls and work stealing to maintain concurrency.

**G4:** Channels are typed conduits for communication. Unbuffered channels block both sender and receiver until both are ready. Buffered channels allow sending without blocking up to capacity. Sending to a full buffered channel blocks until a receiver reads.

**G5:** The `context` package manages deadlines, cancellation, and request-scoped values.
```go
ctx, cancel := context.WithTimeout(context.Background(), 5*time.Second)
defer cancel()
req, _ := http.NewRequestWithContext(ctx, "GET", url, nil)
client.Do(req) // Cancels if timeout exceeded
```

**G6:** Go interfaces are satisfied implicitly (duck typing). Unlike Java/C#, you don't declare implementation. `interface{}` (or `any`) accepts any type. It is powerful for generic-like flexibility but dangerous because type assertions are required to use the value, risking panics if the type is wrong.

**G7:** This is unsafe due to a **data race**. The write to `data` and read of `ready` are unsynchronized. The compiler/CPU may reorder operations. Correct alternatives: use `sync/atomic` for the boolean flag or a `sync.Mutex` protecting both variables.

**G8:** `sync.Mutex` is for exclusive access. `sync.RWMutex` allows multiple readers or one writer. `sync.Map` is optimized for read-heavy workloads with sparse keys. Trade-off: `sync.Map` has higher overhead for writes and lacks iteration safety compared to a regular map protected by a mutex.

**G9:** `go.mod` declares module path and dependencies. `go.sum` stores checksums for integrity. `go mod tidy` removes unused imports and adds missing ones. **MVS** (Minimum Version Selection) selects the lowest compatible version of a dependency that satisfies all requirements in the dependency tree.

**G10:** `defer` schedules a function call to run after the surrounding function returns. Multiple defers execute in **LIFO** (Last In, First Out) order. Gotcha in loops: the deferred function captures the loop variable by reference. Fix by creating a new variable inside the loop (e.g., `i := i`).

**G11:** A pointer holds a memory address. The zero value is `nil`. Create a pointer to a struct literal using `&StructType{Field: value}`.

**G12:** `init()` runs automatically before `main()` during package initialization. A package can have multiple `init()` functions; they execute in the order they appear in the source file.

**G13:** `new(T)` allocates zeroed memory for type `T` and returns a pointer `*T`. `make(T)` initializes slices, maps, or channels and returns the initialized value (not a pointer). `new` works on any type; `make` only works on slices, maps, channels.

**G14:** Errors are values implementing the `error` interface (`Error() string`). Idiomatic pattern: `if err != nil { return err }`. `errors.Is` checks equality against wrapped errors. `errors.As` retrieves the underlying error type. `==` only works for concrete error types, not wrapped ones.

**G15:**
*   **Map:** Zero value is `nil` (uninitialized).
*   **Slice:** Zero value is `nil` (empty).
*   **Channel:** Zero value is `nil` (uninitialized).
*   **Struct:** Zero value has all fields set to their respective zero values.

**G16:** A race condition occurs when two goroutines access shared data concurrently, with at least one write. The `-race` flag instruments code to detect these at runtime.
*Fix:* Use `sync.Mutex` to lock access or channels to pass ownership.
```go
var mu sync.Mutex
mu.Lock()
data = "hello"
mu.Unlock()
```

**G17:** `select` waits on multiple channel operations. Unlike `switch`, it chooses a ready case randomly if multiple are ready. Implement timeout by adding a `case <-time.After(d)`.

**G18:** Escape analysis determines if a variable lives on the stack or heap. If a variable's address escapes the function scope, it moves to the heap. Inspect with `go build -gcflags="-m"`. Heap allocation is slower (GC pressure) than stack allocation.

**G19:** `reflect` allows runtime inspection of types and values. `reflect.Type` is metadata; `reflect.Value` is the data. It is slow and bypasses compile-time checks. Justified for serialization frameworks or generic libraries, but avoided in performance-critical paths.

**G20:** `//go:embed` embeds files/directories at compile time into a `embed.FS`. Works with single files (`//go:embed file.txt`) or directories (`//go:embed images/*`). Limitations: files must exist at compile time. It implements `io/fs.FS`, allowing standard file system operations on embedded content.

## Rust (20 questions)

**R1 — Easy:** Ownership is Rust's core memory safety feature. Rules: 1) Each value has one owner. 2) When owner goes out of scope, value is dropped. 3) You can move ownership, but not copy (unless `Copy` trait).

**R2:** `String` is an owned, heap-allocated, growable buffer. `&str` is a borrowed, immutable slice of UTF-8 bytes. Use `&str` in function signatures for flexibility (accepts both `String` and string literals). Use `String` when you need to own or mutate the data.

**R3:** Borrowing allows temporary access without ownership. Rules: 1) You can have many immutable references (`&T`) OR one mutable reference (`&mut T`). 2) References must always be valid. 3) You cannot mutate through an immutable reference.

**R4:** `Result<T, E>` represents success/failure. `Option<T>` represents `Some(T)` or `None`. The `?` operator propagates errors (returns early if `Err`/`None`).
```rust
fn read_file(path: &str) -> Result<String, std::io::Error> {
    std::fs::read_to_string(path)
}
```

**R5:** `Box<T>` owns data on the heap (single owner). `Rc<T>` is reference-counted for single-threaded shared ownership. `Arc<T>` is atomic reference-counted for thread-safe shared ownership. Use `Arc` when sharing across threads.

**R6:** Traits define shared behavior. Similar to interfaces but support default implementations. `dyn Trait` is a trait object (dynamic dispatch, vtable). `T: Trait` is a generic (static dispatch, monomorphization). Static dispatch is faster; dynamic allows heterogeneous collections.

**R7:** The function fails because the compiler cannot determine which input lifetime (`x` or `y`) the returned reference belongs to. Fix: Add explicit lifetime annotation.
```rust
fn longest<'a>(x: &'a str, y: &'a str) -> &'a str {
    if x.len() > y.len() { x } else { y }
}
```

**R8:** `Send` means ownership can be transferred across threads. `Sync` means `&T` can be shared across threads. `Rc<T>` is not `Send` because its reference count is not atomic. `Arc<T>` uses atomic reference counting, making it `Send` and `Sync`.

**R9:** `async/await` creates a state machine (`Future`). The executor (e.g., Tokio) polls the future until ready. Unlike Go's preemptive green threads, Rust's async is cooperative; the future must yield control explicitly.

**R10:** `unsafe` allows operations the compiler cannot verify. 1) Dereferencing raw pointers. 2) Calling unsafe functions/FFI. 3) Accessing mutable statics. Justified for FFI, low-level optimizations, or implementing safe abstractions over unsafe primitives.

**R11:** Pattern matching (`match`) exhaustively checks all possible values. `if let` matches only one pattern. The compiler enforces exhaustiveness in `match`, preventing logic errors from unhandled enum variants.

**R12:** `[T; N]` is a fixed-size array on the stack. `Vec<T>` is a growable heap-allocated buffer. When `Vec` capacity is exceeded, it allocates a new larger buffer, copies elements, and drops the old one (usually doubling capacity).

**R13:** Enums are algebraic data types holding data variants. Unlike C/Java, they can hold different types per variant. `Option<T>` is `Some(T)` or `None`. Rust has no `null`; `None` is explicit, preventing null pointer exceptions.

**R14:** `impl` defines methods. `&self` borrows immutably. `&mut self` borrows mutably. `self` consumes the value (move). Associated functions (e.g., `String::new()`) do not take `self` and act like static constructors.

**R15:** `derive` macros automatically implement traits. Common: `Debug`, `Clone`, `PartialEq`. Cannot derive if fields contain types that don't support the trait (e.g., `Fn` trait cannot be derived).

**R16:** `Cow<'a, T>` (Clone on Write) holds either a borrowed `&'a T` or an owned `Box<T>`. Useful when you want to avoid allocation if no mutation occurs. Example: Parsing a string; if no modification needed, keep `&str`; if modified, clone to `String`.

**R17:** `mod` defines a module. `use` imports paths. `pub` makes items visible outside the crate. Filesystem maps to modules (`mod.rs` or `mod/` folder). Rust 2018 improved path resolution (no more `extern crate` required for standard library).

**R18:** `Pin<T>` prevents a value from being moved in memory. Needed for self-referential structs and async futures (which may hold references to their own fields). `Unpin` is a marker trait indicating a type *can* be moved safely.

**R19:** `macro_rules!` are declarative macros (pattern matching). Procedural macros (derive, attribute, function-like) are Rust code run at compile time. Hygiene ensures macro variables don't leak into the caller's scope.

**R20:** Zero-cost abstractions mean high-level features compile to efficient machine code. Generics use monomorphization (copy-paste code per type). Iterators are lazy and inlined. No runtime overhead compared to C, enabling safety without performance penalty.
