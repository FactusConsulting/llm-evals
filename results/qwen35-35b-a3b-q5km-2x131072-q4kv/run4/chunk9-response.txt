SC1-A
The pods are failing due to a violation of the **Restricted Pod Security Standard (PSS)**.
1.  **Interaction:** Kubernetes PSS enforces security constraints at the admission level. `securityContext.runAsNonRoot: true` instructs the container runtime to prevent running as UID 0. However, if `runAsUser` is not explicitly set, Kubernetes defaults to the user defined in the container image.
2.  **Image Behavior:** The new base image (`cgr.dev/chainguard/static:latest`) sets `USER root` (UID 0) in its Dockerfile.
3.  **Baseline vs. Restricted:**
    *   **Baseline:** Allows `runAsNonRoot: true` without `runAsUser` set. If the image runs as root, it is allowed (though discouraged).
    *   **Restricted:** Requires `runAsNonRoot: true` **AND** `runAsUser` to be explicitly set to a non-zero value. It explicitly forbids running as root (UID 0).
4.  **Failure:** The admission controller rejects the pod spec because the effective UID is 0 (from the image), violating the Restricted policy's prohibition on root execution, even though `runAsNonRoot` was present.

SC1-C
**Remediation Strategy:**
1.  **Immediate Fix:** Update the container image to a non-root user (e.g., `cgr.dev/chainguard/static:latest-nonroot`) or patch the pod spec to include `securityContext.runAsUser: 1000`.
2.  **Manifest Changes:** Update all Deployment/StatefulSet specs to explicitly define `securityContext.runAsUser` and `runAsGroup` matching the image user.
3.  **CI Pipeline Gate:** Integrate **Kyverno** or **OPA Gatekeeper** in the CI pipeline to validate manifests against the Restricted PSS policy before pushing to the cluster. Add a step to scan images for `USER root` and fail the build if found.
4.  **Rollout:** Use a **Canary Deployment** strategy. Deploy the fixed version to 5% of traffic, monitor CrashLoopBackOff rates and error logs for 15 minutes, then gradually increase to 100%. This ensures the fix works without impacting the entire user base immediately.

SC2-A
**Stale Data Issue:**
1.  **Replication Lag:** Cross-region asynchronous replication introduces latency. When the replica is promoted, it only contains data that has been shipped from the primary. Any data written to the primary just before the outage was not yet replicated.
2.  **WAL Shipping:** PostgreSQL ships Write-Ahead Logs (WAL) asynchronously. The replica applies WAL as it arrives.
3.  **Data Loss:** In async replication, if the primary fails before WAL is shipped, that data is lost.
4.  **Quantification:** Check `pg_stat_replication` on the promoted instance (if it was a replica) or the old primary logs (if accessible) for `write_lag`, `flush_lag`, and `replay_lag`. Calculate the time difference between the last committed transaction on the old primary and the last replayed transaction on the new primary.

SC2-B

SC2-C
**Architecture Design:**
1.  **Connection Management:** Use **RDS Proxy** instead of PgBouncer. RDS Proxy handles connection pooling at the AWS layer, supports IAM auth, and handles failover transparently for the application.
2.  **DNS Failover:** Use **Route53 Health Checks** with Failover Routing Policy. Point the primary endpoint to the US region and the secondary to the EU region. Route53 automatically switches traffic if the US health check fails.
3.  **Application Logic:** Implement exponential backoff with jitter in the connection pooler and application retry logic to handle transient connection errors during failover.
4.  **RPO < 1s:** Standard RDS async replication cannot guarantee <1s RPO across regions reliably due to network jitter. **Recommend Aurora Global Database.** It uses a dedicated replication layer optimized for low latency (typically <1s lag) and allows instant promotion of secondary regions with minimal data loss.

SC3-C
**Hardened Pipeline:**
1.  **Image Signing:** Use **Cosign** (Sigstore) to sign images upon build. `cosign sign --key env://COSIGN_PRIVATE_KEY <image>`.
2.  **Admission Control:** Deploy **Kyverno** or **OPA Gatekeeper** in the cluster to enforce `verify-images` policy. Pods must have a valid signature from a trusted key.

SC4-A
**Top 5 Blockers & Solutions:**
1.  **Local State (NFS):** Kubernetes pods are ephemeral. **Solution:** Migrate to a CSI driver (e.g., `nfs.csi.k8s.io`) or object storage (S3) for uploads.
2.  **Sticky Sessions:** F5 uses IP affinity. **Solution:** Use Kubernetes Ingress with `sessionAffinity: ClientIP` or move session state to Redis.
3.  **Oracle DB:** On-prem Oracle is stateful. **Solution:** Use Oracle Cloud Infrastructure (OCI) Database or containerized Oracle with StatefulSets and persistent storage.
4.  **IBM MQ:** Legacy messaging. **Solution:** Use IBM MQ Kubernetes Operator or migrate to RabbitMQ/Kafka if feasible.
5.  **Monolith Scaling:** Java monolith scales poorly horizontally. **Solution:** Implement horizontal pod autoscaling (HPA) with resource limits, but consider breaking the monolith into smaller services over time (Strangler Fig).

SC6-C
**Connectivity Comparison:**
1.  **AWS Direct Connect:**
    *   *Pros:* Dedicated bandwidth (10Gbps+), lower latency, consistent performance.
    *   *Cons:* High cost, hardware provisioning.
    *   *Zero Trust:* Fits well; traffic is encrypted (MACsec) and isolated.
2.  **Second VPN (ECMP):**
    *   *Pros:* Low cost, quick setup.
    *   *Cons:* Shared bandwidth, higher latency, potential packet reordering.
    *   *Zero Trust:* IPsec encryption is standard, but bandwidth is shared.
3.  **Hybrid Mesh (Tailscale/Netbird):**
    *   *Pros:* Easy identity-based access, no IP management.
    *   *Cons:* Not suitable for high-throughput data replication (200TB MinIO).
    *   *Zero Trust:* Native identity-based routing.
**Recommendation:** Use **Direct Connect** for the primary data path (MinIO/DB replication) to meet bandwidth needs. Use **Tailscale** for administrative access and low-bandwidth service-to-service communication where IP management is difficult. Update routing to use BGP over Direct Connect for dynamic failover.

SC7-A
**Drift Sources & Risks:**
1.  **Manual `tofu apply`:** Bypasses CI/CD. **Risk:** State corruption if two engineers run simultaneously without locking. Race conditions can lead to partial updates or resource deletion.
2.  **Helm Values Divergence:** Untracked changes. **Risk:** Configuration drift leads to inconsistent behavior between environments (e.g., prod has more replicas than staging).
3.  **`kubectl edit`:** Direct API access. **Risk:** Security gaps (RBAC bypass) and inability to audit changes via Git history.
**State Locking:** Without locking, two `apply` commands can read the same state, modify it, and write back, causing the second write to overwrite the first (lost update).

SC10-C
**Automated Failover Orchestration:**
1.  **Detection:** Prometheus monitors site health (ping, DB latency). If Chicago metrics fail for 2 minutes, trigger alert.
2.  **DNS Failover:** GSLB (Route53) switches DNS to Dallas endpoint immediately upon alert.
3.  **Patroni:** Patroni detects primary is down, promotes the Dallas node automatically (via etcd leader election).
4.  **Redis/Rabbit:** Sentinel/Federation detects master loss and promotes replica.
5.  **MinIO:** Site Replication catches up asynchronously.
6.  **State Update:** Ansible playbook updates OpenTofu state variables to reflect Dallas as primary.
7.  **Split-Brain:** Use **etcd quorum** in Patroni. If Chicago etcd is unreachable, Dallas cannot promote unless it has quorum. Use a **witness node** (cloud-based) to break ties.
8.  **Failback:** Once Chicago recovers, demote Dallas, sync data, and switch DNS back. Ensure data consistency before cutover.
