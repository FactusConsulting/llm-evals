SC1-A
The pods are failing because the **Restricted Pod Security Standard (PSS)** forbids running containers as root (UID 0).
1.  **Interaction:** Kubernetes applies PSS policies at the namespace level. Restricted policy requires `securityContext.runAsNonRoot: true` AND explicitly sets `runAsUser` to a non-zero value.
2.  **The Change:** The upgrade enforced Restricted PSS. Previously (Baseline), running as root was allowed.
3.  **The Failure:** The new base image (`chainguard/static:latest`) defaults `USER` to root. The pod spec sets `runAsNonRoot: true` but omits `runAsUser`. Without an explicit `runAsUser`, the container runtime defaults to UID 0 (root). This violates the Restricted policy's requirement that the effective UID must not be 0. The kubelet rejects the pod or the container crashes immediately upon startup due to the policy violation.

```bash
kubectl get pods --all-namespaces -o json | jq -r '.items[] | select(.status.containerStatuses[]?.state.waiting.reason == "CrashLoopBackOff") | "\(.metadata.namespace) \(.metadata.name) \(.spec.securityContext | @json)"'
```
*Note: This assumes `securityContext` is at the pod level. If container-level, adjust selector to `.spec.containers[] | select(.securityContext.runAsNonRoot == true) | select(.securityContext.runAsUser == null)`. The command above captures the pod-level context for the failing pods.*

SC1-C
**Remediation Strategy:**
1.  **Image Fix:** Rebuild the base image with a non-root user (e.g., `USER 1000`) or use a pre-built non-root image.
2.  **Manifest Fix:** Update `securityContext` in the Deployment/StatefulSet to explicitly set `runAsUser: 1000` (matching the image) and keep `runAsNonRoot: true`.
3.  **CI Gate:** Add a step in the CI pipeline using `trivy` or `cosign` to scan the image for `USER` directive. Fail the build if `USER` is root or missing.
4.  **Rollout:** Use a RollingUpdate strategy with `maxSurge: 0` and `maxUnavailable: 10%`. Deploy the fixed manifests to a canary namespace first. Verify health probes pass, then promote to production. This ensures zero downtime as old pods are terminated only after new ones are ready.

SC2-A
**Stale Data Issue:**
1.  **Replication Lag:** Cross-region asynchronous replication introduces latency. Promoting the replica cuts it off from the source's WAL stream. Any WAL generated on the primary but not shipped to the replica before promotion is lost.
2.  **WAL Shipping:** PostgreSQL streams WAL records. In async mode, the replica acknowledges receipt asynchronously. If the primary crashes before the replica receives the WAL, that data is gone.
3.  **Data Loss Window:** The window is the time between the last committed transaction on the primary and the last WAL record received by the replica.
4.  **Quantification:** Check `pg_stat_replication` on the primary (before outage) for `write_lag` and `flush_lag`. After promotion, check `pg_stat_wal_receiver` on the new primary for `received_lsn` vs `replay_lsn`.

SC2-B
1.  **Import:** `terraform import aws_db_instance.promoted_replica <new_replica_arn>` (Ensure the resource block in code matches the new ARN).
2.  **Detach:** `terraform state rm aws_db_instance.old_primary` (Removes old instance from state without destroying it immediately).
3.  **Update:** Update the Terraform config to point to the new ARN. Run `terraform plan` to verify no destroy actions are scheduled for the new instance.
4.  **Cleanup:** Once verified, run `terraform destroy` on the old instance resource block (if it still exists in state) or manually delete it via AWS Console if it was orphaned.
*Crucial:* Ensure `lifecycle { prevent_destroy = false }` is set on the old instance resource before removing it from state to allow manual deletion.

SC2-C
**Architecture Design:**
1.  **Connection Management:** Use **RDS Proxy** instead of PgBouncer. It handles connection pooling at the AWS layer, supports IAM auth, and automatically reconnects on failover without app changes.
2.  **DNS Failover:** Use **Route53 Latency-Based Routing** with health checks. Point the app endpoint to a CNAME that resolves to the primary region. If health checks fail, Route53 routes to the secondary region.
3.  **Retry Logic:** Implement exponential backoff with jitter in the application connection pooler.
4.  **RPO < 1s:** Use **Aurora Global Database**. It uses a shared storage layer with synchronous replication across regions, offering sub-second RPO compared to standard RDS async replication.
5.  **Recommendation:** Yes, recommend Aurora Global DB. Standard RDS async replication cannot guarantee RPO < 1s across regions due to network latency and WAL shipping delays. Aurora Global DB replicates storage blocks, reducing lag significantly.

SC4-A
**Top 5 Blockers:**
1.  **Local Sessions:** Kubernetes pods are ephemeral. **Solution:** Move sessions to Redis (ElastiCache) with sticky sessions disabled.
2.  **Local NFS Mount:** Pods cannot share local filesystems across nodes. **Solution:** Use NFS CSI driver with ReadWriteMany (RWX) PersistentVolumes.
3.  **On-Prem Oracle:** K8s clusters are likely cloud-native. **Solution:** Migrate to RDS/Aurora or use a private link to on-prem DB (if staying hybrid).
4.  **IBM MQ:** On-prem mainframe integration. **Solution:** Use MQ Bridge or Cloud-native MQ (IBM MQ on K8s) with federation.
5.  **Sticky Sessions (F5):** F5 handles affinity. **Solution:** Implement application-level affinity or use Kubernetes Service with `sessionAffinity: ClientIP` (less robust) or move to Redis.

**Connectivity Upgrade:**
1.  **AWS Direct Connect:**
