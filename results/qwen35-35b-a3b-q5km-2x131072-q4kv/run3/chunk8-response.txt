CL1 — **IaaS (Infrastructure as a Service):** Provides virtualized computing resources (VMs, storage, networks). *Example:* AWS EC2. **PaaS (Platform as a Service):** Provides hardware and software tools (runtime, DB, middleware) for development. *Example:* Google App Engine. **SaaS (Software as a Service):** Delivers software applications over the internet. *Example:* Microsoft Office 365.

CL2 — **Shared Responsibility Model:** Provider secures the cloud (physical, network, hypervisor); Customer secures what is in the cloud (OS, data, config). **IaaS VM:** Provider secures hardware/hypervisor; Customer secures OS, patching, app, data. **Managed Kubernetes:** Provider secures control plane/infrastructure; Customer secures node OS, pod config, network policies, app. **SaaS (M365):** Provider secures app/infrastructure; Customer secures identity, data classification, device access.

CL3 — **VPC:** Isolated virtual network. **Subnet:** IP range within VPC (public/private). **Security Group:** Stateful firewall at instance level (allow rules). **NACL:** Stateless firewall at subnet level (allow/deny rules). **Layering:** VPC contains Subnets; Subnets contain Instances; SG protects Instance; NACL protects Subnet. **Stateful:** SG remembers return traffic. **Stateless:** NACL requires explicit rules for both inbound and outbound.

CL4 — **Landing Zone:** Secure baseline environment. **OUs:** Organize accounts by function/environment. **Guardrails:** Preventative controls (e.g., deny public S3). **Shared Networking:** Transit Gateway/Hub-and-Spoke centralizes connectivity. **Centralized Logging:** Aggregates audit trails. **Single-Account Problems:** High blast radius, billing complexity, inability to enforce least privilege across teams, lack of isolation for compliance.

CL5 — **IaC:** Managing infrastructure via code. **Terraform/OpenTofu:** Multi-cloud, declarative, state file management. **Pulumi:** General-purpose languages (TS, Python). **CloudFormation/Bicep:** Native to AWS/Azure, tightly integrated. **Crossplane:** K8s-native IaC. **Choice:** Terraform for multi-cloud; Pulumi for dev-heavy teams; CFN/Bicep for AWS/Azure-only; Crossplane for GitOps/K8s-centric.

CL6 — **HPA:** Scales pods based on CPU/Memory. **VPA:** Scales pod resources (CPU/RAM) vertically. **Cluster Autoscaler:** Adds/removes nodes based on pod scheduling. **KEDA:** Event-driven scaling (e.g., queue length). **Use KEDA over HPA:** When scaling is driven by external events (Kafka, SQS) rather than internal resource metrics.

CL7 — **DR Strategy:** Define RPO (data loss tolerance) and RTO (downtime tolerance). **Active-Active:** Both regions serve traffic (low RTO, high cost). **Active-Passive:** One active, one standby (higher RTO, lower cost). **DB:** Sync (strong consistency, latency) vs Async (replication lag, higher throughput). **DNS:** Route53 Failover routing. **State:** Shared storage or replicated state (Redis/ElastiCache). **Cost:** Active-Active doubles infra cost. **Pilot Light:** Core infra running, app off. **Warm Standby:** Scaled-down app running. **Multi-site:** Full active-active.

CL8 — **GitOps:** Git as single source of truth for infra/app state. **ArgoCD:** Push/Pull model, UI focused. **FluxCD:** Pull model, K8s-native. **Secrets:** Sealed Secrets (encrypted in Git), External Secrets Operator (fetch from Vault/AWS Secrets Manager). **Drift:** GitOps controller detects manual changes and automatically reverts them to match Git state.

CL9 — **Architecture:** Use **StatefulSet** for stable network IDs and ordered deployment. **PV/PVC:** Use `local` storage class for low latency (hostPath). **Headless Service:** For stable DNS names per pod. **Init Containers:** For config file generation before main app starts. **ConfigMaps:** Mount as volumes for specific paths. **PDB:** Ensure minimum available pods during updates to prevent data loss.

CL10 — **SA:** K8s internal identity for pods. **Workload Identity:** Maps K8s SA to Cloud Identity (OIDC). **IRSA:** AWS specific implementation of Workload Identity. **Mounting Creds:** Bad practice (static keys, hard to rotate, risk of leakage). **Federation:** Uses OIDC tokens to request temporary IAM credentials, eliminating static secrets.

CL11 — **Object (S3):** Unstructured data, high durability (11 9s), low latency retrieval. **Block (EBS):** Attached disk, low latency, high IOPS, attached to single instance. **File (EFS):** Shared file system, NFS/SMB, moderate latency. **Use Case:** Object for backups/media, Block for DBs/OS, File for shared app configs. **Durability:** All typically 99.999999999% (11 9s). **Availability:** Varies by service tier (e.g., S3 Standard vs Glacier).

CL12 — **Zero Trust:** Verify every request. **Network Policies:** Restrict pod-to-pod traffic. **Service Mesh:** mTLS for L7 encryption. **Pod Identity:** SPIFFE/SPIRE for workload identity. **Admission Control:** OPA/Kyverno to enforce policies on deploy. **Image Signing:** Cosign to verify integrity. **Runtime:** Falco for anomaly detection. **Difference:** Traditional relies on perimeter trust; Zero Trust assumes breach and verifies every interaction.

CL13 — **CDN:** Distributed network of edge servers caching content. **Function:** Reduces latency, offloads origin. **Services:** AWS CloudFront, Azure CDN. **Use CDN:** For static assets (images, CSS, JS) to improve global performance. **Use Origin:** For dynamic, personalized, or private content not suitable for caching.

CL14 — **Region:** Geographic area with multiple AZs. **AZ:** Isolated data center within a region. **Multi-AZ:** Protects against data center failure. **Zonal:** Resources tied to one AZ (EBS). **Regional:** Resources span AZs (ELB, Route53).

CL15 — **Managed DB:** RDS, Cloud SQL. **Provider Handles:** Patching, backups, failover, hardware provisioning, scaling. **Customer Handles:** Schema, queries, user access, data logic.

CL16 — **IAM:** **Users:** Human identities. **Groups:** Collections of users. **Roles:** Temporary access for services/users. **Policies:** JSON permissions. **Least Privilege:** Grant only necessary permissions. **Application:** Use roles instead of long-term keys, group users by function, audit regularly.

CL17 — **Serverless:** Event-driven execution, no server mgmt. **Compare:** Lambda (AWS), Functions (Azure), Cloud Functions (GCP). **Cold Starts:** Latency when function is invoked after inactivity. **Poor Fit:** Long-running processes, stateful apps, predictable high-throughput workloads.

CL18 — **Ingress:** L7 routing rules (K8s standard). **Gateway API:** Modern, multi-protocol, better extensibility. **LB Annots:** Cloud-specific LB config. **Mesh Gateway:** Service mesh ingress. **TLS:** Ingress (terminates at LB), Mesh (terminates at sidecar). **Use Gateway API:** For complex routing, multi-cluster, or protocol diversity (HTTP/gRPC).

CL19 — **Strategy:** **RI/SP:** Commit to usage for discount. **Spot:** Interruptible for fault-tolerant workloads. **Right-sizing:** Match instance to actual usage. **Tiering:** Move old data to Glacier. **Idle Detection:** Tag and remove unused resources. **FinOps:** Cross-team cost accountability. **Showback:** Report costs to teams. **Tools:** AWS Cost Explorer, CloudHealth, Kubecost.

CL20 — **Strategy:** **Abstraction:** Kubernetes, Terraform. **Networking:** Direct Connect/ExpressRoute. **Identity:** OIDC federation. **Trade-offs:** Increased complexity, cost, skill requirements. **Justified:** Regulatory compliance, risk diversification, legacy constraints. **Unnecessary:** Small teams, single-cloud native tools offer better integration.

OT1 — **Purdue Model:** ISA-95 hierarchy. **Level 0:** Physical Process (sensors). **Level 1:** Basic Control (PLCs, RTUs). **Level 2:** Supervisory Control (SCADA, HMIs). **Level 3:** Operations (MES, Historian). **Level 4:** Enterprise (ERP, Business).

OT2 — **IT:** Information systems, frequent patching, rebootable. **OT:** Operational technology, physical process control, stability critical. **Patching:** IT patches regularly; OT patches rarely due to legacy OS, safety risks, and downtime costs.

OT3 — **DMZ:** Buffer zone between IT and OT. **IDMZ:** Industrial DMZ (secure). **Traffic:** Pull only (OT to IT) or controlled push (IT to OT via DMZ). **Direct L4->L1:** Dangerous because it allows lateral movement from corporate network directly to control devices, risking physical process disruption.

OT4 — **Modbus TCP:** L1/L2, simple polling. **OPC UA:** L2/L3, secure, platform-independent. **EtherNet/IP:** L1/L2, CIP protocol. **PROFINET:** L1/L2, real-time industrial Ethernet. **MQTT:** L3/L4, lightweight pub/sub for IoT.

OT5 — **Data Diode:** Hardware enforcing one-way data flow. **Firewall:** Two-way traffic filtering. **Use Diode:** When data must leave OT but nothing can enter (max security). **Limitation:** No return traffic allowed (e.g., no acknowledgments, no control commands back).

OT6 — **Flow:** PLC -> Historian -> OPC UA Server -> Edge Gateway -> MQTT Broker -> Cloud. **Security:** Edge Gateway acts as DMZ. Encrypt traffic (TLS). Isolate OT network from Internet. Use firewall rules to allow only specific ports/protocols.

OT7 — **Zones:** Logical grouping of assets with similar security requirements. **Conduits:** Controlled paths between zones. **Model:** Corporate (L4) -> DMZ -> SCADA (L3) -> Engineering Workstations -> PLCs (L2) -> SIS (L2). Conduits enforce traffic flow (e.g., Engineering to PLC only).

OT8 — **Segmentation:** VLANs per production line. **Firewalls:** Between VLANs and DMZ. **Switches:** Managed switches with ACLs. **Vendor Access:** Jump host with MFA and session recording. **Transition:** Phased approach, parallel run, maintenance windows, backup before changes.

OT9 — **SIS:** Safety Instrumented System (shutdown on hazard). **BPCS:** Basic Process Control System (normal operation). **IEC 61511:** Safety lifecycle standard. **Isolation:** SIS must be independent to prevent cyber attack disabling safety. **Ransomware on SIS:** Can disable safety shutdowns, leading to physical damage or injury.

OT10 — **PLC:** Programmable Logic Controller (local logic). **DCS:** Distributed Control System (integrated control for large plants). **SCADA:** Supervisory Control and Data Acquisition (wide area monitoring). **Coexist:** Yes, PLCs often feed SCADA/DCS.

OT11 — **Architecture:** Jump host (bastion), MFA, Session Recording, Time-limited access. **No Direct RDP:** Prevents credential theft and lateral movement from corporate network to critical OT assets.

OT12 — **Detection:** NIDS, anomaly detection on OT protocols. **Response:** Isolate compromised segment, notify Control Room. **Ops Role:** Maintain physical safety, manual override if needed. **Coordination:** IT handles network, OT handles process safety.

OT13 — **HMI:** Human-Machine Interface (operator screen). **Role:** Monitor and control process. **Difference:** HMI is for operations; Engineering Workstation is for programming/configuring logic.

OT14 — **Historian:** Time-series database for process data. **Importance:** Trend analysis, compliance, troubleshooting. **Plant (L3):** Local, real-time ops data. **Enterprise (L4):** Aggregated, long-term business analytics.

OT15 — **Ladder Logic:** Graphical programming language for PLCs. **Device:** PLC. **Dominant:** Visual representation mimics electrical circuits, familiar to electricians, deterministic execution.

OT16 — **RTU:** Remote Terminal Unit (long-distance, harsh env). **PLC:** Local control. **RTU Use:** Pipelines, utilities, remote sites with limited connectivity.

OT17 — **Analog:** Continuous signal (4-20mA). **Digital:** Discrete signal (0/1). **4-20mA:** Preferred because 4mA indicates live wire (fail-safe); 0mA indicates break/fault.

OT18 — **Inventory:** Foundation of security (know what to protect). **Tools:** Passive (sniffing), Active (scanning). **Active Risk:** Can crash legacy devices or trigger alarms in OT networks.

OT19 — **MITRE ATT&CK ICS:** Framework for ICS threats. **Tactics:** Initial Access (Spearphishing), Execution (Command Line), Persistence (Scheduled Task), Impact (Data Manipulation), Defense Evasion (Obfuscation). **Difference:** Focuses on physical impact and safety. **Usage:** Map current defenses to tactics to find gaps.

OT20 — **SOC:** Passive log collection (SPAN ports), SIEM integration, OT-specific rules (e.g., Modbus anomalies). **Staffing:** OT engineers for triage. **False Positives:** High due to legacy behavior. **Firmware Alert:** Verify change request, isolate device, check integrity, re-image if unauthorized, investigate source.
