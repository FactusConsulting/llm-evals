AA1 — **Monolith vs. Microservices:** A monolith is a single, unified codebase deployed as one unit. Microservices are independent, loosely coupled services deployed separately.
**3 Problems Microservices Introduce:**
1.  **Network Latency & Reliability:** Communication happens over the network, introducing latency and failure points not present in local method calls.
2.  **Distributed Data Consistency:** Maintaining ACID transactions across multiple services requires complex patterns (e.g., Sagas) rather than simple DB transactions.
3.  **Operational Complexity:** Requires orchestration (Kubernetes), distributed tracing, and service discovery, increasing DevOps overhead compared to a single deployment.

AA2 — **Sync vs. Async Communication:** Synchronous communication blocks the caller until a response is received (request-response). Asynchronous communication allows the caller to send a message and continue without waiting for an immediate response.
**Examples:**
*   **Synchronous:** REST (HTTP) or gRPC.
*   **Asynchronous:** RabbitMQ (Message Queue) or Apache Kafka (Event Streaming).

AA3 — **API Gateway vs. Reverse Proxy:** A reverse proxy routes traffic to backend servers (L4/L7). An API Gateway is a reverse proxy with added API management features (auth, rate limiting, transformation, analytics).
**Can it replace entirely?** No. While an API Gateway handles HTTP/HTTPS traffic, it cannot replace a Layer 4 TCP/UDP load balancer for non-HTTP protocols (e.g., gRPC over TCP, WebSocket passthrough without HTTP framing, or database connections).

AA4 — **Saga Pattern:** A sequence of local transactions where each updates the database and triggers the next step. If a step fails, compensating transactions undo previous changes.
**Choreography vs. Orchestration:**
*   **Choreography:** Services communicate via events; no central coordinator. *Appropriate for:* Simple, low-coupling flows.
*   **Orchestration:** A central coordinator directs the flow. *Appropriate for:* Complex workflows requiring centralized error handling and visibility.

AA5 — **CQRS (Command Query Responsibility Segregation):** Separates read (Query) and write (Command) operations into different models.
**Pairing with Event Sourcing:** Event Sourcing stores state changes as a sequence of events. CQRS uses these events to build optimized read models.
**Problem Solved:** Allows independent scaling of read and write workloads, provides a complete audit trail (history of state changes), and enables temporal queries (viewing state at a past point in time).

AA6 — **Circuit Breaker Pattern:** Prevents an application from repeatedly trying to execute an operation that's likely to fail.
**Three States:** Closed (normal), Open (failures detected, requests blocked), Half-Open (testing if service recovered).
**Difference from Retry:** Retry adds load to a failing service; Circuit Breaker stops load to allow recovery.
**Use Together:** Use Retry for transient network glitches, but wrap the retry logic in a Circuit Breaker to stop hammering a service that is genuinely down.

AA7 — **Multi-Tenant Isolation Models:**
1.  **Shared DB, Shared Schema:** Lowest cost, highest operational complexity (row-level security), highest noisy-neighbor risk.
2.  **Shared DB, Separate Schema:** Medium cost, better isolation (schema-level permissions), moderate noisy-neighbor risk.
3.  **Separate DB per Tenant:** Highest cost (resource overhead), best security/isolation, lowest noisy-neighbor risk.
**Trade-off:** Choose based on tenant size and security requirements. Large enterprise tenants usually get separate DBs; SMB tenants share.

AA8 — **Eventual vs. Strong Consistency:** Eventual consistency guarantees data will match eventually but allows temporary divergence. Strong consistency guarantees immediate visibility of updates across all nodes.
**Inventory Architecture:**
*   **Consistency:** Strong (to prevent overselling).
*   **Data Store:** Relational Database (e.g., PostgreSQL) with row-level locking or a distributed lock manager (e.g., Redis Redlock).
*   **Pattern:** Pessimistic locking (lock row before decrement) or Optimistic locking (version check) within a single transaction.

AA9 — **Sidecar Proxy:** A container deployed alongside the main application container to handle network traffic.
**Envoy in Istio:** Envoy acts as the data plane proxy. Istio Control Plane configures Envoy via xDS API.
**Data Plane vs. Control Plane:** Data Plane handles actual traffic; Control Plane manages configuration and policy.
**mTLS:** Application-level TLS requires code changes and cert management per app. mTLS between sidecars provides transparent encryption and identity verification (service-to-service authentication) without modifying application code.

AA10 — **Strangler Fig Pattern:** Incrementally replacing legacy functionality with new services.
**Migration Strategy:**
1.  Deploy new .NET 8 microservices alongside the legacy .NET Framework app.
2.  Configure the API Gateway to route specific endpoints (e.g., `/api/new-feature`) to the new services.
3.  Route remaining traffic to the legacy monolith.
4.  Gradually migrate more endpoints until the legacy app is empty and can be decommissioned.

AA11 — **12-Factor App:** Methodology for building cloud-native apps.
**6 Factors:**
1.  **Codebase:** One codebase tracked in revision control.
2.  **Dependencies:** Explicitly declare and isolate dependencies.
3.  **Config:** Store config in environment variables.
4.  **Backing Services:** Treat backing services (DBs) as attached resources.
5.  **Build, Release, Run:** Strictly separate build and run stages.
6.  **Processes:** Execute the app as one or more stateless processes.
**Why:** Ensures portability, scalability, and ease of deployment across environments.

AA12 — **Event-Driven (Kafka) vs. Request-Driven (REST):**
*   **Coupling:** Event-driven is loosely coupled; Request-driven is tightly coupled.
*   **Scalability:** Event-driven scales better for high throughput; Request-driven is limited by connection handling.
*   **Debugging:** Event-driven is harder (distributed tracing needed); Request-driven is easier (request/response chain).
*   **Ordering:** Event-driven requires partition keys for ordering; Request-driven is naturally ordered per connection.
*   **Failure:** Event-driven requires idempotency/retry logic; Request-driven uses timeouts.
**Choice:** Use Event-driven for decoupling and high-volume data pipelines. Use Request-driven for real-time user interactions requiring immediate feedback.

AA13 — **Load Balancer:** Distributes incoming network traffic across multiple servers.
**L4 vs. L7:** L4 (Transport) balances based on IP/Port. L7 (Application) balances based on HTTP headers, URL, cookies.
**L7 Use Case:** SSL Termination (decrypting traffic at the LB) or routing `/api` to backend A and `/static` to backend B.

AA14 — **Message Queue vs. Log-based Broker:**
*   **Message Queue (RabbitMQ):** Messages are consumed and deleted. Good for task queues, work distribution.
*   **Log-based (Kafka):** Messages are stored as an immutable log. Good for event replay, high throughput, audit trails.
**Pick One:** Pick RabbitMQ for simple task queues. Pick Kafka for event streaming, data pipelines, or when you need to replay history.

AA15 — **Container vs. VM:** A container virtualizes the OS kernel (shares host kernel), while a VM virtualizes the hardware (runs full guest OS).
**Why Popular:** Containers are lightweight, start in seconds, have lower overhead, and are portable across environments, fitting the ephemeral nature of microservices.

AA16 — **Stateless vs. Stateful:** Stateless services do not store session data locally (e.g., Web Server). Stateful services store data locally (e.g., Database, Cache).
**Scaling:** Stateless services are easier to scale horizontally because any instance can handle any request without data synchronization.
**Examples:** Stateless: Nginx. Stateful: PostgreSQL.

AA17 — **ORM (Object-Relational Mapper):** Maps database tables to programming language objects.
**Examples:** Hibernate (Java), Entity Framework (.NET).
**Trade-offs:** ORMs increase developer productivity and reduce boilerplate but can generate inefficient SQL and hide database-specific optimizations compared to raw SQL.

AA18 — **Bulkhead Pattern:** Isolates resources (threads, connections) for different services to prevent a failure in one from exhausting resources for others.
**Implementation:** Use separate thread pools for each downstream service call. In Kubernetes, use separate namespaces or deployment groups with resource limits (CPU/Memory) to enforce isolation.

AA19 — **OT vs. CRDT:**
*   **OT:** Server authoritative, calculates transformations. High consistency, high server dependency, complex implementation.
*   **CRDT:** Local resolution, mathematically convergent. Lower consistency latency, supports offline editing, simpler server logic.
**Choice:** Choose **CRDTs** for a system requiring offline editing. CRDTs allow clients to edit locally and merge changes automatically without server mediation, whereas OT requires server coordination to resolve conflicts.

AA20 — **Distributed Tracing Challenges:**
*   **Context Propagation:** Must pass TraceID/SpanID across services (W3C Trace Context, B3 headers).
*   **Sampling:** Head-based (decide at start) vs. Tail-based (decide after latency known).
*   **Storage:** High volume requires efficient storage (Jaeger, Tempo).
*   **Kafka Continuity:** Producers must inject trace headers into Kafka messages; Consumers must extract and propagate them to downstream services to maintain the trace chain.

OP1 — **Hypervisor Type 1 vs. Type 2:**
*   **Type 1 (Bare-metal):** Runs directly on hardware. Examples: VMware ESXi, Microsoft Hyper-V Server.
*   **Type 2 (Hosted):** Runs on top of an OS. Examples: Oracle VirtualBox, VMware Workstation.
**Difference:** Type 1 has better performance and security (less attack surface); Type 2 is easier to set up on desktops.

OP2 — **SAN vs. NAS:**
*   **SAN (Storage Area Network):** Block-level storage. Protocols: iSCSI, Fibre Channel. Use for: Databases, high-performance VMs.
*   **NAS (Network Attached Storage):** File-level storage. Protocols: NFS, SMB/CIFS. Use for: File shares, home directories.

OP3 — **3-Tier vs. Leaf-Spine:**
*   **3-Tier:** Core, Distribution, Access layers. Can suffer from oversubscription and high latency in the spine.
*   **Leaf-Spine:** Every leaf connects to every spine.
**Problem Solved:** Leaf-Spine provides non-blocking, predictable latency and higher bandwidth utilization, solving the oversubscription bottleneck of 3-tier architectures in modern data centers.

OP4 — **Proxmox HA Architecture:**
*   **Nodes:** Minimum 3 for quorum.
*   **Quorum:** Majority vote required to make decisions.
*   **Fencing:** STONITH (e.g., iDRAC/ILO) to power off failed nodes to prevent split-brain.
*   **Storage:** Ceph (distributed) or Shared iSCSI/NFS.
*   **Failure:** If a node fails, HA manager restarts VMs on surviving nodes (after fencing the failed node).

OP5 — **VLAN Trunking:** Carries traffic for multiple VLANs over a single link using 802.1Q tags.
**Linux Configuration:** Create sub-interfaces.
`ip link add link eth0 name eth0.10 type vlan id 10`
`ip link add link eth0 name eth0.20 type vlan id 20`
Assign IPs to `eth0.10` and `eth0.20`.

OP6 — **RAID Levels:**
*   **RAID 1:** Mirroring (50% capacity, 1 drive fault tolerance).
*   **RAID 5:** Striping + Parity (N-1 capacity, 1 drive fault tolerance).
*   **RAID 6:** Striping + Double Parity (N-2 capacity, 2 drive fault tolerance).
*   **RAID 10:** Mirroring + Striping (50% capacity, 1 drive per mirror fault tolerance).
*   **RAID Z2:** ZFS double parity (similar to RAID 6).
**Choice:** **RAID 10** for a 6-drive database server. It offers the best performance (striping) and fault tolerance (mirroring) for write-heavy workloads.

OP7 — **DNS Architecture:**
*   **Internal:** AD Integrated DNS (Active Directory).
*   **External:** Public authoritative servers (e.g., Route53).
*   **Split-Horizon:** Internal clients resolve internal IPs; external resolve public IPs.
*   **Forwarders:** Internal DNS forwards unknown queries to ISP/Cloud.
*   **Laptop Move:** DHCP updates DNS A records (DDNS) or client updates via DHCP lease renewal.

OP8 — **PKI Chain of Trust:**
*   **Root CA:** Offline, highly secure.
*   **Subordinate CA:** Issues end-entity certs.
*   **CRL vs. OCSP:** CRL (list of revoked) vs. OCSP (real-time status check).
*   **Auto-enrollment:** Clients request certs from CA automatically.
*   **Consequences:** Compromised Root = All trust revoked (catastrophic). Compromised Subordinate = Revoke sub, issue new sub, reissue certs (contained).

OP9 — **WAN Consolidation:**
*   **MPLS:** High reliability, high cost, hard to manage.
*   **IPsec VPN:** Low cost, variable reliability, complex config.
*   **SD-WAN:** Low cost, dynamic path selection, easy management.
*   **WireGuard:** Simple, encrypted, but lacks enterprise management features.
**Recommendation:** **SD-WAN**. It optimizes VoIP traffic (QoS), reduces MPLS costs, and simplifies management for 50 users/site.

OP10 — **iDRAC/iLO/IPMI:** Out-of-band management interfaces allowing remote power control and console access independent of the OS.
**Critical:** Essential for remote recovery when the OS is unresponsive.
**Exposed to Internet:** High risk. Vulnerabilities in IPMI firmware can lead to Remote Code Execution (RCE) and full server compromise.

OP11 — **LVM vs. LVM-Thin vs. ZFS:**
*   **LVM:** Logical volumes over physical partitions.
*   **LVM-Thin:** Overprovisioning, thin provisioning.
*   **ZFS:** Filesystem + Volume manager, Copy-on-Write.
**Snapshots:** LVM (Copy-on-Write, slower). ZFS (Instant, low overhead, efficient).
**Use:** LVM for simple Linux volumes. ZFS for data integrity and snapshots.

OP12 — **Stretch Cluster / DR:**
*   **Design:** 2 sites + 1 Tiebreaker (disk/VM).
*   **Ceph CRUSH:** Place replicas across sites.
*   **Latency:** < 5ms RTT recommended for Ceph.
*   **Split-Brain:** If network fails, both sites think they are primary.
*   **Tiebreaker:** Breaks the tie to ensure only one site remains writable.
**Why Bad Without Tiebreaker:** Risk of data corruption if both sites write simultaneously during a partition.

OP13 — **DHCP & DORA:**
*   **Process:** **D**iscover, **O**ffer, **R**equest, **A**cknowledge.
*   **Unavailable Server:** Client retries. If lease expires, client must start DORA again to get a new IP.
*   **Windows:** May assign a Link-Local address (169.254.x.x) if DHCP fails.

OP14 — **Managed vs. Unmanaged Switch:**
*   **Unmanaged:** Plug-and-play, no config.
*   **Managed:** Configurable via CLI/GUI.
*   **Features:** VLANs, QoS (prioritize VoIP), ACLs, SNMP monitoring, Port Mirroring.
**Importance:** Critical for segmentation, security, and traffic prioritization in enterprise.

OP15 — **NTP:** Network Time Protocol.
**Critical:** Logs, authentication (Kerberos), clustering (quorum), and certificate validation rely on accurate time.
**Drift Consequences:** Kerberos auth fails (clock skew > 5 mins), log correlation impossible, cluster split-brain.

OP16 — **Physical vs. VM vs. Container:**
*   **Physical:** Bare metal hardware.
*   **VM:** Virtualized hardware.
*   **Container:** Virtualized OS.
**Bare Metal Use:** High-performance computing, low-latency trading, or software licensing that requires physical CPU cores.

OP17 — **UPS vs. Generator:**
*   **UPS:** Battery backup for short-term power (seconds/minutes) to allow graceful shutdown or bridge to generator.
*   **Generator:** Long-term power (hours/days) using fuel.
*   **ATS (Automatic Transfer Switch):** Automatically switches power source from Utility to Generator when utility fails.
**Need Both:** UPS protects against surges and bridges the gap until the generator starts.

OP18 — **Active-Passive vs. Active-Active:**
*   **Active-Passive:** One node handles traffic; standby takes over on failure.
*   **Active-Active:** Both nodes handle traffic.
**PostgreSQL HA (Pacemaker/Corosync):**
*   **Fencing (STONITH):** Power off failed node to prevent data corruption.
*   **VIP:** Virtual IP moves to the active node.
*   **Failover:** Pacemaker detects failure, runs STONITH, promotes standby, moves VIP.

OP19 — **Backup & Recovery Strategy:**
*   **3-2-1 Rule:** 3 copies, 2 media types, 1 offsite.
*   **RPO/RTO:** Recovery Point (data loss tolerance) vs. Recovery Time (downtime tolerance).
*   **Types:** Full, Incremental, Differential.
*   **Immutable Backups:** Prevents ransomware from encrypting backups.
*   **Backup vs. DR:** Backup is the data copy; DR is the plan/process to restore operations.
*   **Testing:** Regularly test restores to verify integrity.

OP20 — **802.1X:**
*   **Roles:** Supplicant (Client), Authenticator (Switch), Authentication Server (RADIUS).
*   **EAP:** EAP-TLS (Certificate-based, secure) vs. PEAP (Username/Password).
*   **Dynamic VLAN:** Switch assigns VLAN based on RADIUS response.
*   **Failure:** Port is blocked or placed in a Guest VLAN.
*   **MAB:** If 802.1X fails, switch sends MAC address to RADIUS as credentials for devices that don't support 802.1X (e.g., printers).
