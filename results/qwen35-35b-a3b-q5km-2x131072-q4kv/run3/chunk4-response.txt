## Go (20 questions)

**G1 — Easy:** An array is a fixed-size value type (`[N]T`), while a slice is a dynamic view over an array (`[]T`) containing a pointer, length, and capacity. When appending to a slice that has reached capacity, the Go runtime allocates a new, larger underlying array (usually doubling capacity), copies the existing elements, and updates the slice header to point to the new array.

**G2:** `var x int` declares a variable with an explicit type and initializes it to the zero value (`0`). `x := 0` is a short variable declaration that infers the type. You cannot use `:=` if the variable is already declared in the current scope, or if you need to declare a variable without initializing it (e.g., `var x int` before assignment).

**G3:** A goroutine is a lightweight thread managed by the Go runtime, not the OS. It differs from an OS thread by having a small initial stack (2KB) and being multiplexed onto fewer OS threads. The Go scheduler (GMP model) maps Goroutines (G) to OS Threads (M) via Processors (P). The scheduler handles preemption and load balancing to maximize CPU utilization.

**G4:** Channels are typed conduits for communication between goroutines. Unbuffered channels block both sender and receiver until the other side is ready. Buffered channels allow sending without blocking up to their capacity. Sending to a full buffered channel blocks until a receiver reads from it.

**G5:** The `context` package manages deadlines, cancellation signals, and request-scoped values across API boundaries.
```go
ctx, cancel := context.WithTimeout(context.Background(), 5*time.Second)
defer cancel()
req, _ := http.NewRequestWithContext(ctx, "GET", url, nil)
resp, err := client.Do(req) // Cancels if timeout occurs
```

**G6:** Go interfaces are satisfied implicitly; a type satisfies an interface if it implements all methods, without explicit declaration. Unlike Java/C#, Go does not require `implements` keywords. The empty interface `interface{}` (or `any`) accepts any type. It is powerful for generic-like behavior but dangerous because it requires type assertions, risking panics if the type is wrong.

**G7:** This code is unsafe due to a data race: one goroutine writes `data` and `ready`, while the main goroutine reads `ready` and `data` without synchronization. The compiler or CPU may reorder operations, or the main goroutine may see `ready=true` before `data` is updated. Correct alternatives include using `sync.Mutex` to protect both variables or `sync/atomic` for the boolean flag.

**G8:** `sync.Mutex` provides exclusive access. `sync.RWMutex` allows multiple concurrent readers or one exclusive writer. `sync.Map` is optimized for read-heavy workloads with dynamic keys (e.g., caches). It trades write performance for read performance and avoids locking for reads, but is not suitable for high-write contention or when key sets are known upfront.

**G9:** Go modules manage dependencies. `go.mod` declares the module path and required dependencies. `go.sum` stores cryptographic checksums of those dependencies. `go mod tidy` removes unused dependencies and adds missing ones. Minimum Version Selection (MVS) ensures the lowest compatible versions of all transitive dependencies are selected to avoid conflicts.

**G10:** `defer` schedules a function call to run immediately before the surrounding function returns. Multiple defers execute in Last-In-First-Out (LIFO) order. A common gotcha in loops is capturing the loop variable in a deferred closure; in Go 1.22+, loop variables are redeclared per iteration, but prior versions captured the same variable, causing all deferred calls to see the final value.

**G11:** A pointer holds the memory address of a value. The zero value of a pointer is `nil`. To create a pointer to a struct literal, use the address-of operator: `&MyStruct{Field: value}`.

**G12:** The `init()` function is called automatically before `main()` runs. It runs once per package during initialization. A package can have multiple `init()` functions; they execute in the order they appear in the source files, and across files, in the order the compiler processes them (typically alphabetical by filename).

**G13:** `new(T)` allocates zeroed memory of type `T` and returns a pointer `*T`. `make(T)` initializes built-in reference types (slice, map, channel) and returns the initialized value (not a pointer). `new` is for any type; `make` is only for slices, maps, and channels.

**G14:** Errors are values implementing the `error` interface (`Error() string`). The idiomatic pattern is `if err != nil { return err }`. `errors.Is` checks if an error matches a target (using wrapping). `errors.As` retrieves the underlying error value. `==` only works for concrete error types, not wrapped errors.

**G15:** Built-in composite types are `map`, `slice`, `channel`, and `struct`. Their zero values are: `map` is `nil`, `slice` is `nil`, `channel` is `nil`, and `struct` has all fields set to their respective zero values.

**G16:** A race condition occurs when two goroutines access shared data concurrently, with at least one write, without synchronization. The `-race` flag instruments code to detect these at runtime.
*Fix:* Use `sync.Mutex` to lock/unlock around the shared access, or use channels to pass ownership of the data.

**G17:** `select` is like a `switch` statement for channels. It waits for one of several channel operations to be ready. If multiple are ready, one is chosen pseudo-randomly. To implement a timeout, use `time.After` to send a signal to a channel case: `case <-time.After(1 * time.Second):`.

**G18:** Escape analysis determines if a variable's lifetime exceeds the function scope. If so, it is allocated on the heap; otherwise, on the stack. Inspect with `go build -gcflags="-m"`. Heap allocation incurs GC overhead; stack allocation is faster. Understanding this helps optimize performance-critical code.

**G19:** `reflect` allows inspecting and modifying values at runtime. `reflect.Type` describes the type; `reflect.Value` holds the value. Reflection is slow due to dynamic dispatch and type checks. It is justified for frameworks (e.g., serialization, ORMs) but avoided in performance-critical paths where code generation (e.g., `go:generate`) is preferred.

**G20:** `//go:embed` embeds static files into the binary at compile time. It supports single files, directories (`//go:embed dir/*`), and file systems (`//go:embed fs`). Limitations: cannot embed symlinks, files must be within the module root. It populates a `fs.FS` interface (e.g., `embed.FS`), allowing standard `io` operations on embedded content.

## Rust (20 questions)

**R1 — Easy:** Ownership is Rust's core memory safety feature. Rules: 1) Each value has a single owner. 2) When the owner goes out of scope, the value is dropped. 3) You can transfer ownership (move), but not copy (unless `Copy` trait).

**R2:** `String` is an owned, heap-allocated, growable string. `&str` is a borrowed slice of UTF-8 bytes (often a string literal). Use `String` for owned data; use `&str` for function arguments to avoid unnecessary allocations and copying.

**R3:** Borrowing allows referencing data without taking ownership. Rules: You can have either one mutable reference (`&mut T`) OR any number of immutable references (`&T`), but not both simultaneously. References must always be valid (no dangling pointers).

**R4:** `Result<T, E>` represents success (`Ok(T)`) or failure (`Err(E)`). `Option<T>` represents a value that may be `Some(T)` or `None`. The `?` operator propagates errors up the call stack.
```rust
fn read_file(path: &str) -> Result<String, std::io::Error> {
    std::fs::read_to_string(path)
}
```

**R5:** `Box<T>` allocates on the heap with a single owner. `Rc<T>` (Reference Counted) allows multiple owners in a single thread. `Arc<T>` (Atomic Reference Counted) allows multiple owners across threads. Use `Box` for heap allocation, `Rc` for shared single-threaded ownership, `Arc` for shared multi-threaded ownership.

**R6:** Traits define shared behavior. Unlike Go interfaces, traits can have default implementations. `dyn Trait` creates a trait object for dynamic dispatch (vtable). `T: Trait` uses static dispatch (monomorphization) for generics, which is generally faster.

**R7:** The function fails because the compiler cannot guarantee the returned reference lives as long as the inputs. Fix: Add a lifetime parameter `'a` to bind the input and output lifetimes.
```rust
fn longest<'a>(x: &'a str, y: &'a str) -> &'a str {
    if x.len() > y.len() { x } else { y }
}
```

**R8:** `Send` means a type can be transferred to another thread. `Sync` means a type can be referenced from another thread (`&T` is `Send` if `T` is `Sync`). `Rc<T>` is not `Send` because its reference count is not atomic. `Arc<T>` uses atomic reference counting, making it `Send` and `Sync`.

**R9:** Rust async/await creates `Future` types (lazy computations). The executor (e.g., Tokio) polls futures until they are ready. Unlike Go's preemptive goroutines, Rust async is cooperative (pauses at `await`). This avoids stack overhead but requires an executor to manage the event loop.

**R10:** `unsafe` allows operations the compiler cannot verify for safety. Examples: dereferencing raw pointers, calling unsafe functions, modifying `static mut`. Justified for FFI, low-level optimizations, or implementing safe abstractions over unsafe primitives.

**R11:** Pattern matching uses `match` to exhaustively handle all possible values of an enum. `if let` matches a single pattern and ignores others. The compiler enforces exhaustiveness in `match`, ensuring no cases are missed, preventing runtime errors.

**R12:** `Vec<T>` is a growable heap-allocated array. `[T; N]` is a fixed-size stack array. When `Vec` capacity is exceeded, it allocates a new, larger buffer (usually doubling), copies elements, and frees the old one.

**R13:** Enums in Rust are tagged unions (can hold data of different types). Unlike C/Java enums, they can store values. `Option<T>` is an enum (`Some(T)` or `None`) that replaces `null`, forcing explicit handling of missing values.

**R14:** `impl` blocks define methods. `&self` borrows the instance; `&mut self` mutably borrows; `self` consumes the instance. Associated functions (e.g., `String::new()`) do not take `self` and act like static methods.

**R15:** `derive` macros automatically implement traits. Common: `Debug`, `Clone`, `PartialEq`. You cannot derive a trait if the type contains fields that don't support it (e.g., `Clone` on a field without `Clone`, or `Debug` on a function pointer).

**R16:** `Cow<'a, T>` (Clone on Write) holds either a borrowed `&'a T` or an owned `Box<T>`. It avoids allocation if the data is only read. If a write is attempted, it clones the data to owned memory first. Useful for functions that accept input but may need to modify it.

**R17:** `mod` declares a module. `use` imports items. `pub` makes items visible outside the module. Modules map to files (`mod.rs` or `module.rs`). Rust 2018 edition improved path resolution, allowing `crate::` and `super::` to work more intuitively across crates.

**R18:** `Pin<T>` prevents a value from being moved in memory. Needed for self-referential structs (e.g., futures holding references to themselves) because moving invalidates pointers. `Unpin` is a marker trait; if a type is `Unpin`, it can be moved freely.

**R19:** `macro_rules!` are declarative macros (pattern matching on tokens). Procedural macros (derive, attribute, function-like) are Rust code that generates code at compile time. Hygiene ensures macro variables don't leak into the caller's scope, preventing naming collisions.

**R20:** Zero-cost abstractions mean high-level features compile to efficient machine code. Generics use monomorphization (code generated per type). Iterators are lazy and inlined. Traits use static dispatch by default. This avoids runtime overhead (like vtables or GC) compared to C, while maintaining safety and ergonomics.
