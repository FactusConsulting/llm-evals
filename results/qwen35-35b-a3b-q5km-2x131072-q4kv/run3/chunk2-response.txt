K1 — **Deployment vs StatefulSet**
A **Deployment** manages stateless applications, providing rolling updates and scaling without guaranteed pod identity. A **StatefulSet** manages stateful applications, providing stable, unique network identities (e.g., `pod-0`, `pod-1`), ordered deployment/termination, and stable persistent storage. Use StatefulSet for databases or apps requiring ordered scaling and persistent identity.

K2 — **ClusterIP Routing**
A **ClusterIP** Service creates a virtual IP (VIP) within the cluster. **kube-proxy** (using iptables or IPVS) manages network rules on each node to intercept traffic sent to the VIP and route it to the healthy backend Pod IPs. It does not route traffic outside the cluster.

K3 — **PV, PVC, StorageClass**
*   **PersistentVolume (PV):** A cluster-level storage resource (e.g., NFS, Cloud Disk).
*   **PersistentVolumeClaim (PVC):** A user request for storage (e.g., "10Gi").
*   **StorageClass:** A template for dynamically provisioning PVs.
*   **Relation:** A PVC binds to a PV. If a StorageClass is specified, the cluster dynamically creates a PV to satisfy the PVC.

K4 — **CrashLoopBackOff Troubleshooting**
1.  **Check Logs:** `kubectl logs <pod>` to see application errors or exceptions.
2.  **Check Events:** `kubectl describe pod <pod>` for OOMKilled, Liveness Probe failures, or ImagePull errors.
3.  **Verify Resources:** Ensure memory/CPU requests/limits are sufficient for the workload.
4.  **Inspect Dependencies:** Verify the app can connect to required services (DB, Redis) and check network policies.

K5 — **CoreDNS Role**
CoreDNS provides DNS resolution for services within the cluster (Service discovery). If CoreDNS pods are down, internal DNS resolution (e.g., `my-service.default.svc`) fails, breaking service-to-service communication. External DNS (e.g., `google.com`) may still work if the pod uses the host network or fallback DNS.

K6 — **Kubernetes Scheduler**
The scheduler places pods on nodes via **Filtering** (scheduling predicates) and **Scoring** (prioritization).
1.  **Filtering:** Removes nodes that don't meet requirements (e.g., nodeSelector, taints, resource capacity).
2.  **Scoring:** Ranks remaining nodes based on fit.
3.  **Factors:** CPU/Memory availability, Node Affinity rules, Pod Topology Spread Constraints.

K7 — **Memory Scheduling & Runtime**
*   **Scheduling:** Node has 4GB total, 3.5GB requested. Available = 500Mi. Pod requests 256Mi. **Yes**, it will be scheduled (256Mi < 500Mi).
*   **Runtime:** Pod uses 600Mi, but limit is 512Mi. The OOM Killer will terminate the container (**OOMKilled**), causing a CrashLoopBackOff.

K8 — **Ingress Traffic Lifecycle**
1.  **External LB:** Routes traffic to the Ingress Controller NodePort.
2.  **Ingress Controller:** (e.g., NGINX Pod) receives traffic, parses Ingress rules, and routes to the correct Service.
3.  **Service:** Routes traffic to the Pod IP via Endpoints.
4.  **kube-proxy:** Manages iptables/IPVS rules on the node to forward traffic to the specific Pod IP.

K9 — **NetworkPolicy**
A **NetworkPolicy** defines how pods are allowed to communicate (firewall rules).
*   **Default:** If no policy exists, all traffic is allowed (unless the cluster has a default deny policy).
*   **Applied:** When a policy selects pods, it enforces the rules. If a policy selects a pod with no ingress rules, traffic to that pod is denied by default.

K10 — **RollingUpdate vs Recreate**
*   **RollingUpdate:** Gradually replaces old pods with new ones (zero downtime).
*   **Recreate:** Terminates all existing pods before creating new ones (downtime).
*   **Use Recreate:** For stateful applications requiring a full stop (e.g., database migrations) or when atomic updates are critical.

K11 — **Namespace & Defaults**
A **Namespace** provides logical isolation for resources within a cluster.
*   **Default Namespaces:** `default`, `kube-system`, `kube-public`, `kube-node-lease`.

K12 — **kubectl describe pod**
Used for detailed debugging. Unlike `get`, it shows:
1.  **Events:** Recent cluster events (e.g., scheduling failures, OOM).
2.  **Conditions:** Pod status conditions (Ready, Initialized).
3.  **Mounts:** Detailed volume mount paths and secrets.

K13 — **ConfigMap**
A ConfigMap stores non-sensitive configuration data.
*   **Methods:**
    1.  **Environment Variables:** Injected into the container environment.
    2.  **Volume Mounts:** Mounted as files inside the container filesystem.

K14 — **Service Types**
*   **ClusterIP:** Internal virtual IP (default). Accessible only within the cluster.
*   **NodePort:** Exposes the service on a static port on each node's IP.
*   **LoadBalancer:** Provisions an external cloud load balancer to expose the service externally.

K15 — **DaemonSet**
A **DaemonSet** ensures a copy of a pod runs on all (or specific) nodes.
*   **Use Case:** Logging agents (Fluentd), monitoring agents (Prometheus Node Exporter), or storage daemons (Ceph) that must run on every node.

K16 — **Taints/Tolerations vs Affinity**
*   **Taints/Tolerations:** Taints repel pods; tolerations allow pods to schedule on tainted nodes. (Push/Pull mechanism).
*   **Node Affinity:** Preferences or requirements for node labels. (Pull mechanism).
*   **Use Case:** Taint a node for GPU workloads (only pods with GPU tolerations go there). Use Affinity to co-locate pods on the same zone.

K17 — **Kubernetes Operator**
An **Operator** is a custom controller that manages complex stateful applications using CRDs (Custom Resource Definitions).
*   **Difference:** Standard controllers manage generic resources; Operators encode operational knowledge (e.g., backup, restore, scaling) for specific apps.
*   **Problem Solved:** Automates lifecycle management of stateful apps (e.g., Kafka, PostgreSQL) that are hard to manage manually.

K18 — **etcd Consistency & Recovery**
*   **Consistency:** Strong consistency (linearizable).
*   **Quorum Loss:** If quorum is lost (e.g., 2 of 3 nodes down), the cluster becomes read-only or unavailable to prevent split-brain.
*   **Recovery:** Restore etcd from a snapshot on new nodes, rejoin them to the cluster, and remove the dead nodes from the member list.

K19 — **RBAC & Manifest**
*   **Role:** Permissions within a namespace.
*   **ClusterRole:** Permissions across the cluster.
*   **RoleBinding:** Links a Role to a user/group in a namespace.
*   **ClusterRoleBinding:** Links a ClusterRole to a user/group cluster-wide.
*   **Manifest:**
    ```yaml
    apiVersion: rbac.authorization.k8s.io/v1
    kind: Role
    metadata: namespace: my-ns
    rules:
    - apiGroups: [""]
      resources: ["pods", "pods/log"]
      verbs: ["get", "list"]
    ---
    apiVersion: rbac.authorization.k8s.io/v1
    kind: RoleBinding
    metadata: namespace: my-ns
    subjects:
    - kind: User
      name: developer
    roleRef:
      kind: Role
      name: read-only-pods
    ```

K20 — **DNS ndots**
`ndots` defines how many dots a domain name must have to be treated as FQDN. Default is 5. If a pod queries `google.com` (0 dots), it tries `google.com.<search-domain>` first, causing retries.
*   **Fix:** Set `dnsConfig` in pod spec: `options: [{name: "ndots", value: "1"}]` to reduce unnecessary search domain queries.

D1 — **Git Rebase vs Merge**
*   **Rebase:** Rewrites history to create a linear commit sequence.
*   **Merge:** Preserves history with a merge commit.
*   **Avoid Rebase:** On shared/public branches, as it rewrites history and disrupts collaborators.

D2 — **TCP vs UDP**
*   **TCP:** Connection-oriented, reliable, ordered. (Protocols: HTTP, SSH).
*   **UDP:** Connectionless, fast, unreliable. (Protocols: DNS, VoIP).

D3 — **Race Condition**
Occurs when multiple threads access shared data concurrently, leading to unpredictable results.
*   **Example:** Two threads incrementing a global counter `x++` without a lock. Final value may be less than expected due to lost updates.

D4 — **Symmetric vs Asymmetric Encryption**
*   **Symmetric:** Single key for encryption/decryption (fast).
*   **Asymmetric:** Public/Private key pair (secure key exchange).
*   **TLS:** Uses Asymmetric for the handshake (key exchange) and Symmetric for the actual data transfer (performance).

D5 — **Container Image Layers**
Each Dockerfile instruction creates a new read-only layer.
*   **Caching:** Layers are cached. If a layer changes, subsequent layers are rebuilt.
*   **Best Practice:** Copy dependency files (e.g., `package.json`) before source code to cache dependencies and speed up rebuilds.

D6 — **HTTP 503 Troubleshooting**
1.  **Upstream Health:** Check if backend services are healthy.
2.  **Database Connectivity:** Verify DB connections are not exhausted.
3.  **Resource Limits:** Check for OOMKilled or CPU throttling.
4.  **Logs/Tracing:** Analyze application logs and distributed traces for specific error codes.

D7 — **CAP Theorem**
A distributed system can only guarantee two of three: Consistency, Availability, Partition Tolerance.
*   **CP (Consistency + Partition Tolerance):** Zookeeper, HBase. (Fails availability during partition).
*   **AP (Availability + Partition Tolerance):** DynamoDB, Cassandra. (Returns stale data during partition).

D8 — **Scaling**
*   **Vertical:** Increase resources (CPU/RAM) of existing instance.
*   **Horizontal:** Add more instances.
*   **Patterns:** Stateless architecture, Sharding, Load Balancing.

D9 — **Environment Variable**
A dynamic value set at runtime.
*   **Persist:** Add `export VAR=value` to `~/.bashrc` or `/etc/environment` on Linux.

D10 — **JSON vs YAML**
*   **JSON:** Strict syntax, no comments. Advantage: Standard for APIs.
*   **YAML:** Human-readable, supports comments. Advantage: Configuration files.

D11 — **REST API**
Architectural style using HTTP methods.
*   **GET:** Retrieve resource.
*   **POST:** Create resource.
*   **PUT:** Update/Replace resource.
*   **DELETE:** Remove resource.

D12 — **Version Pinning**
Specifying exact dependency versions (e.g., `package@1.2.3`).
*   **Importance:** Ensures reproducible builds and prevents breakage from upstream updates.

D13 — **Compiled vs Interpreted**
*   **Compiled:** Source code converted to binary before execution. (Examples: C, Go).
*   **Interpreted:** Source code executed line-by-line at runtime. (Examples: Python, JavaScript).

D14 — **Reverse Proxy**
*   **Reverse Proxy:** Server-side proxy (Client -> Proxy -> Server).
*   **Forward Proxy:** Client-side proxy (Client -> Proxy -> Internet).
*   **Examples:** Nginx (L7 load balancing), HAProxy (L4 load balancing).

D15 — **Deadlock**
A situation where processes wait indefinitely for each other.
*   **Coffman Conditions:** 1. Mutual Exclusion, 2. Hold and Wait, 3. No Preemption, 4. Circular Wait.

D16 — **Eventual Consistency**
Updates propagate asynchronously; systems converge over time.
*   **Cassandra:** Uses Read Repair and Hinted Handoff.
*   **Levels:** `ONE` (fastest, least consistent), `QUORUM` (balanced), `ALL` (strongest).

D17 — **mTLS (Mutual TLS)**
Both client and server authenticate each other via certificates.
*   **Handshake:** Client Hello -> Server Hello -> Server Cert -> Client Cert -> Verify.
*   **Use Case:** Service Mesh (Istio) for secure service-to-service communication.

D18 — **Hash Table**
Maps keys to indices using a hash function.
*   **Collision:** Chaining (linked lists) or Open Addressing (probing).
*   **Complexity:** Average O(1) for insert/lookup/delete. Worst O(n) if all keys collide.

D19 — **Circuit Breaker**
Prevents cascade failures by stopping calls to failing services.
*   **States:** Closed (normal), Open (fail fast), Half-Open (test recovery).
*   **Relation:** Works with retry policies; retrying a failing service without a breaker causes overload.

D20 — **Optimistic vs Pessimistic Concurrency**
*   **Optimistic:** Check version on commit (e.g., `IF version = X`). High concurrency, low conflict.
*   **Pessimistic:** Lock rows on read (`SELECT FOR UPDATE`). Low concurrency, high conflict.
*   **Choice:** Optimistic for read-heavy; Pessimistic for write-heavy/conflict-prone.
