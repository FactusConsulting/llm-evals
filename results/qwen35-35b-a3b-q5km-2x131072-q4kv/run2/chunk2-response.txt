K1 — Easy: A **Deployment** manages stateless applications, creating pods with no guaranteed identity or order. A **StatefulSet** manages stateful applications, providing stable, unique network identities (e.g., `pod-0`, `pod-1`) and ordered, graceful deployment and scaling.

K2 — Easy: A **ClusterIP** Service creates a virtual IP within the cluster. **kube-proxy** (using iptables or IPVS) configures the node's networking rules to route traffic sent to the Service IP to the backend Pod IPs based on the Service's selector.

K3 — Medium: A **PersistentVolume (PV)** is a cluster-level storage resource. A **PersistentVolumeClaim (PVC)** is a user request for storage. A **StorageClass** defines the "class" of storage and how PVs are provisioned dynamically. A PVC binds to a PV; a StorageClass allows a PVC to auto-provision a PV if no static PV exists.

K4 — Medium: 1. Check container logs (`kubectl logs`) for application errors. 2. Check pod events (`kubectl describe`) for OOMKilled or CrashLoopBackOff reasons. 3. Verify resource limits (CPU/Memory) are not being exceeded. 4. Inspect liveness/readiness probe configurations for false positives.

K5 — Medium: **CoreDNS** provides DNS resolution for services within the cluster. If CoreDNS pods are down, internal service discovery (e.g., `my-service.default.svc`) fails, though external DNS resolution may still work if the pod falls back to the node's `/etc/resolv.conf`.

K6 — Hard: The scheduler uses **Filtering** (remove nodes that don't meet requirements) and **Scoring** (rank remaining nodes). Factors considered: 1. **Resource Requests** (CPU/Mem fit), 2. **Node Affinity** (must be on specific node labels), 3. **Pod Affinity** (must be near other pods).

K7 — Hard: **Yes**, it will be scheduled. Node has 4GB total, 3.5GB requested, leaving 0.5GB (512Mi) free. The pod requests 256Mi, which fits. At runtime, if it uses 600Mi (exceeding the 512Mi limit), the container will be **OOMKilled** by the kernel.

K8 — Hard: 1. Request hits **External LB**. 2. LB forwards to **Ingress Controller** (NGINX). 3. Controller parses **Ingress** resource and configures NGINX. 4. NGINX routes to **Service** IP. 5. **kube-proxy** redirects to **Endpoints** (Pod IPs). 6. Traffic reaches the **Application Pod**.

K9 — Medium: A **NetworkPolicy** restricts traffic flow between pods. Default behavior (no policy): All traffic is allowed. When a policy is applied: Traffic to/from pods selected by the policy is restricted to only what the policy explicitly allows; traffic to pods not selected remains unaffected.

K10 — Medium: **RollingUpdate** replaces pods gradually (zero downtime). **Recreate** kills all existing pods before starting new ones. Choose **Recreate** for stateful applications where version mismatches are critical or database migrations require a full stop.

K11 — Easy: A **Namespace** provides logical isolation for resources. Default namespaces: `default`, `kube-system`, `kube-public`, `kube-node-lease`.

K12 — Easy: `kubectl describe` shows detailed status and events. It displays: 1. **Events** (e.g., scheduling failures), 2. **Conditions** (e.g., Ready status details), 3. **Environment Variables** and **Mounts** (vs `get` which only shows status/age).

K13 — Easy: A **ConfigMap** stores non-confidential configuration data. Pods consume it via: 1. **Environment Variables**, 2. **Volume Mounts** (files).

K14 — Easy: **ClusterIP** (internal cluster access), **NodePort** (access via `<NodeIP>:<Port>`), **LoadBalancer** (external cloud provider load balancer with public IP).

K15 — Easy: A **DaemonSet** ensures a copy of a pod runs on all (or specific) nodes. Use it for logging agents, monitoring daemons, or CNI plugins where node-level access is required, unlike Deployments which manage replica counts.

K16 — Medium: **Taints** repel pods (node attribute); **Tolerations** allow pods to schedule on tainted nodes. **Node Affinity** attracts pods to nodes. Use Case: Taint a node for GPU workloads (no other pods allowed); Affinity to co-locate microservices on the same zone.

K17 — Medium: A **Kubernetes Operator** is a custom controller that manages complex stateful applications using CRDs. It differs from a standard controller by encapsulating **domain-specific knowledge** (e.g., backup, scaling, upgrades) to automate lifecycle management.

K18 — Hard: etcd uses **Strong Consistency (Linearizable)**. If quorum is lost (e.g., 2/3 nodes lost in a 3-node cluster), the cluster becomes **read-only/unavailable**. Recovery: Restore the remaining node from a **snapshot**, remove dead nodes, and rejoin new nodes to form a new quorum.

K19 — Hard: **Role** (namespaced permissions), **ClusterRole** (cluster-wide), **RoleBinding** (links Role to Subject in namespace), **ClusterRoleBinding** (links ClusterRole to Subject cluster-wide).
*Manifest:*
```yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata: namespace: dev
rules:
- apiGroups: [""]
  resources: ["pods", "pods/log"]
  verbs: ["get", "list"]
---
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata: namespace: dev
subjects:
- kind: User
  name: developer
roleRef:
  kind: Role
  name: pod-reader
```

K20 — Hard: `ndots` defines how many dots trigger a full FQDN lookup. Default `ndots: 5` causes queries for short names to try all search domains first, causing latency. Fix: Set `dnsConfig: ndots: 1` in the pod spec to query the FQDN immediately.

D1 — Easy: **Rebase** rewrites history to create a linear commit chain. **Merge** preserves history with a merge commit. Avoid rebasing when the branch is **shared/public** to avoid forcing others to rewrite history.

D2 — Medium: **TCP** is connection-oriented and reliable (HTTP, SSH). **UDP** is connectionless and fast (DNS, VoIP).

D3 — Medium: A **race condition** occurs when output depends on the sequence of uncontrollable events. Example: Two threads incrementing a shared counter `count++` without a lock, resulting in lost updates.

D4 — Medium: **Symmetric** uses one key for encryption/decryption. **Asymmetric** uses a public/private key pair. TLS uses Asymmetric for the handshake (key exchange) and Symmetric for the actual data transfer (performance).

D5 — Medium: A **layer** is a read-only filesystem diff in a Docker image. **Layer caching** improves build speed; place frequently changing instructions (e.g., `COPY .`) after stable ones (e.g., `RUN apt-get`) to leverage cache.

D6 — Medium: 1. Check **upstream service health** (is the backend down?). 2. Check **database connectivity**. 3. Check **resource limits** (OOM). 4. Check **rate limiting** or circuit breaker status.

D7 — Hard: **CAP Theorem**: Consistency, Availability, Partition Tolerance. **CP** (Consistency + Partition Tolerance): Zookeeper, HBase. **AP** (Availability + Partition Tolerance): Cassandra, DynamoDB.

D8 — Medium: **Horizontal Scaling** adds more nodes; **Vertical Scaling** adds more resources to a node. Patterns enabling horizontal scaling: **Stateless services**, **Sharding**, **Load Balancing**.

D9 — Easy: An **environment variable** is a dynamic value affecting a process. To persist across sessions on Linux, export it in `~/.bashrc` or `~/.profile`.

D10 — Easy: **JSON** is strict key-value pairs; **YAML** is human-readable with indentation. Advantage JSON: Strict parsing. Advantage YAML: Comments and less verbose.

D11 — Easy: **REST API** is an architectural style using HTTP. **GET** (retrieve), **POST** (create), **PUT** (replace), **DELETE** (remove).

D12 — Easy: **Version pinning** locks dependencies to exact versions (e.g., `==1.0.0`). It ensures **reproducible builds** by preventing unexpected breaking changes from upstream updates.

D13 — Easy: **Compiled** languages convert code to binary before execution (C, Go). **Interpreted** languages execute code line-by-line at runtime (Python, JavaScript).

D14 — Medium: A **Reverse Proxy** sits in front of servers (client -> proxy -> server). A **Forward Proxy** sits in front of clients (client -> proxy -> internet). Examples: **Nginx** (web server), **HAProxy** (load balancer).

D15 — Medium: A **deadlock** is a permanent blocking of processes. **Coffman Conditions**: 1. Mutual Exclusion, 2. Hold and Wait, 3. No Preemption, 4. Circular Wait.

D16 — Medium: **Eventual Consistency** ensures updates propagate over time. Cassandra uses **Read Repair** and **Hinted Handoff**. Tunable levels: `ONE`, `QUORUM`, `ALL`. Read-after-write consistency depends on matching write/read levels (e.g., QUORUM write + QUORUM read).

D17 — Hard: **mTLS** authenticates both client and server via certificates. Handshake: Client sends cert, Server validates; Server sends cert, Client validates. Commonly used in **Service Meshes** (e.g., Istio) for secure service-to-service communication.

D18 — Hard: A **hash table** maps keys to indices. **Collision resolution**: Chaining (linked lists) or Open Addressing (probing). Time Complexity: Average O(1), Worst O(n) (due to collisions).

D19 — Hard: **Circuit Breaker** prevents cascade failures. States: **Closed** (normal), **Open** (fail fast), **Half-open** (test recovery). Relates to retry: Retries are suppressed when the circuit is Open.

D20 — Hard: **Optimistic** (check on commit, e.g., version column/CAS) assumes low conflict. **Pessimistic** (lock on read, e.g., `SELECT FOR UPDATE`) assumes high conflict. Choose Optimistic for high concurrency; Pessimistic for critical data integrity.
