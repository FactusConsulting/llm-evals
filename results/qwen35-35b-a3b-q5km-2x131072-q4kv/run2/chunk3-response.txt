**T1**
`tofu plan` performs a dry run, calculating the execution plan and displaying the differences (add, change, destroy) between the current state and the configuration without making changes. `tofu apply` executes the plan, creating, updating, or destroying resources to match the configuration.

**T2**
State is a JSON file mapping OpenTofu resources to real-world infrastructure. Remote state is important because it enables team collaboration, provides state locking to prevent concurrent writes, and secures sensitive data (like credentials) by storing it off the local machine.

**T3**
`count` creates resources based on an integer index, while `for_each` creates resources based on a map or set of keys. `for_each` is preferred because it is more stable; removing an item from a list changes indices in `count` (causing resource recreation), whereas `for_each` preserves resource identity based on keys.

**T4**
This is called "drift". OpenTofu detects this during `tofu plan` by comparing the real infrastructure against the state file. If drift is found, `tofu plan` shows the changes required to reconcile the state with reality, and `tofu apply` will update the infrastructure to match the configuration.

**T5**
```hcl
resource "aws_instance" "web" {
  for_each = toset(["web1", "web2", "web3"])
  ami      = "ami-12345678"
  instance_type = "t2.micro"
  tags = {
    Name = each.key
  }
}
```

**T6**
`terraform_remote_state` reads the entire state file from another backend to access outputs, creating a dependency on the state file structure. Direct `output` values are only accessible if modules share the same state. The recommended alternative is using a shared state backend with modules (monorepo approach) so outputs are accessed directly without the `terraform_remote_state` data source, reducing coupling and complexity.

**T7**
The output is empty because the `null_resource` provisioner runs asynchronously or the output is evaluated before the file is written to disk. To fix this, use a `local_file` resource to write the kubeconfig. Have the `null_resource` trigger the `local_file` (via `triggers` or `depends_on`), and output the `local_file` content. Ensure dependent resources `depends_on` the `null_resource` or `local_file`.

**T8**
`tofu import` maps existing infrastructure to state manually via CLI. Limitations include no config generation and manual mapping. The `import` block (1.5+) allows importing resources directly within the configuration during `tofu apply`, integrating imports into the plan/apply cycle and improving automation.

**T9**
A `variable` is an input parameter passed into a module or root configuration. An `output` is data exposed from a module or root configuration to be used by other modules or the CLI. Variables are inputs; outputs are data exports.

**T10**
A provider is a plugin that interacts with a specific cloud or service API. If you run `tofu plan` without `tofu init`, the command fails because the provider binaries are not downloaded or initialized, resulting in a "Provider not installed" error.

**T11**
The `.terraform.lock.hcl` file records the exact versions of providers used to ensure reproducible builds. It should be committed to version control to prevent accidental upgrades that could break infrastructure or cause drift.

**T12**
`tofu destroy` removes all resources managed by the state file. To target a single resource, use the `-target` flag: `tofu destroy -target=aws_instance.example`.

**T13**
A data source is a read-only resource used to fetch data from an external system. Use one instead of a managed resource when you need to reference existing infrastructure (e.g., looking up an AMI ID) rather than creating new infrastructure.

**T14**
`variables` are inputs defined for external consumption (CLI, files). `locals` are internal variables for calculations within the configuration. Use `locals` for derived values or complex logic that shouldn't be exposed as inputs.

**T15**
The `lifecycle` block controls resource behavior. `create_before_destroy` creates the new resource before destroying the old one (zero-downtime). `prevent_destroy` prevents accidental deletion (safety). `ignore_changes` ignores specific attribute drift (e.g., tags modified manually).

**T16**
A `moved` block allows you to change the address of a resource in configuration without destroying and recreating it. It updates the state mapping to the new address, preserving the resource's history and data during refactoring.

**T17**
The dependency graph determines execution order. Implicit dependencies are inferred from references (e.g., `aws_instance.id`). Explicit dependencies use `depends_on`. `depends_on` reduces parallelism because it forces sequential execution even if resources are unrelated, slowing down apply times.

**T18**
1. **Disable refresh:** Use `-refresh=false` to skip state checks (faster, less accurate). 2. **Modularize:** Split large states into smaller modules (faster parsing). 3. **Target:** Use `-target` to plan only specific changes (faster, limited scope). 4. **Backend Optimization:** Use S3/DynamoDB instead of local state (faster locking/concurrency).

**T19**
State locking prevents concurrent writes to the state file. If a lock isn't released (e.g., CI crash), subsequent applies fail. Manually break it with `tofu force-unlock <LOCK_ID>`. Risk: Data corruption if another process is actually writing to the state concurrently.

**T20**
1. Backup existing state file. 2. Add `backend` block (S3/DynamoDB) to config. 3. Run `tofu init -migrate-state`. 4. Verify with `tofu state list` to ensure resources are present in the new backend.

**A1**
A **Task** is a single action (e.g., install package). A **Role** is a directory structure containing tasks, handlers, vars, etc., for reusable logic. A **Playbook** is a YAML file defining a list of plays (tasks + hosts) to execute.

**A2**
`ansible_facts` is a dictionary of system information gathered by the `setup` module. Access the OS family via `{{ ansible_facts['os_family'] }}` or `{{ ansible_os_family }}`.

**A3**
`command` runs without a shell (no pipes/globs). `shell` runs via a shell (supports pipes/globs). `raw` runs without installing the Python module (useful for initial bootstrap). Use `command` for safety, `shell` for scripting, `raw` for bootstrapping.

**A4**
Idempotency means running a task multiple times produces the same result. `apt: name=htop` is idempotent (installs only if missing). `shell: rm -rf /tmp/*` is NOT idempotent (deletes every time).

**A5**
```yaml
- name: Install htop
  package:
    name: htop
  become: yes
  when: ansible_os_family == "Debian"

- name: Install htop on RedHat
  package:
    name: htop
  become: yes
  when: ansible_os_family == "RedHat"
```

**A6**
Precedence (High to Low): Extra vars > Command line > Env vars > Play vars > Host vars > Group vars > Role vars > Role defaults. Extra vars override everything.

**A7**
Use `serial: 3` in the play to limit concurrency. Group nodes into `rke2_nodes` and `rke2_workers`. Use `serial` on the group or `meta: wait_for` to ensure the first batch completes before the next batch starts.

**A8**
`ansible-vault` encrypts sensitive data. To encrypt a single variable, use `ansible-vault edit vars.yml` and encrypt specific lines, or use `ansible-vault encrypt_string` to encrypt a value and paste it into the vars file.

**A9**
An inventory file lists hosts and groups. **Static** inventory is a fixed file (INI/YAML). **Dynamic** inventory is generated by a script or API (e.g., AWS EC2) to reflect changing infrastructure.

**A10**
`ansible webservers -m shell -a "df -h"`

**A11**
A **Handler** is a task that runs only when notified by another task. It differs from a regular task by running asynchronously at the end of the play. Trigger it using `notify: handler_name` in a task.

**A12**
`ansible.cfg` configures Ansible behavior. Common settings: `host_key_checking` (disable SSH warnings), `forks` (parallelism), `remote_user` (default SSH user).

**A13**
`register` stores task output in a variable. Example: `result = shell: echo "hi"`. Use it in subsequent tasks: `debug: msg="{{ result.stdout }}"`.

**A14**
`include_tasks` is evaluated at runtime (supports conditionals). `import_tasks` is evaluated at compile time (no conditionals). Use `include_tasks` for dynamic logic, `import_tasks` for static structure.

**A15**
Ansible Galaxy is a repository for roles and collections. Use `requirements.yml` to list dependencies, then run `ansible-galaxy install -r requirements.yml` to install them.

**A16**
Connection plugins manage how Ansible connects to hosts. **SSH** (remote Linux/Unix), **Local** (control node), **Network_CLI** (routers/switches). Use SSH for servers, Local for control tasks, Network_CLI for network devices.

**A17**
`delegate_to` runs a task on a specific host. `run_once` runs a task once per play. Example: Gather app server data, then `delegate_to: localhost` to write a config file on the control node once.

**A18**
1. **Forks:** Increase `forks` in `ansible.cfg` for parallelism. 2. **Pipelining:** Enable `pipelining` to reduce SSH connections. 3. **Fact Caching:** Cache facts to avoid gathering on every run. 4. **Mitogen:** Use Mitogen plugin for faster SSH execution.

**A19**
`{{ }}` evaluates expressions. `{% %}` controls logic (loops/if). `{# #}` comments.
```jinja2
upstream backend {
{% for server in backend_servers %}
    server {{ server }}; # Health check: {{ server_health_check }}
{% endfor %}
}
```

**A20**
1. **Serial:** Set `serial: 5` to deploy in batches. 2. **LB:** Pre-task deregister from LB. 3. **Health:** Use `uri` module to check health after deploy. 4. **Rollback:** Wrap in `block/rescue`; if health check fails, trigger `rescue` to revert changes.
