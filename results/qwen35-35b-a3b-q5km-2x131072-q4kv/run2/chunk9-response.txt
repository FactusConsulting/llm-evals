SC1-A
The pods are failing because the **Restricted** Pod Security Standard (PSS) is stricter than **Baseline**.
1.  **Baseline:** Allows `runAsNonRoot: true` without requiring an explicit `runAsUser`. It only checks that the container does not run as root (UID 0).
2.  **Restricted:** Requires `runAsNonRoot: true` **AND** an explicit `runAsUser` (non-zero UID). It also forbids `privileged: true`, `hostNetwork`, and requires specific capabilities.
3.  **The Failure:** The new base image (`cgr.dev/chainguard/static:latest`) defaults `USER` to root (UID 0). The cluster upgrade enforced Restricted PSS. Kubernetes attempts to start the container as root (per image default), but the Pod Security Admission controller rejects it because `runAsUser` is not explicitly set to a non-zero value in the pod spec, or because the image defaults to root which violates the "non-root" requirement without an override.
4.  **Interaction:** The admission controller checks the Pod spec against the namespace's PSS level. Since `runAsUser` is missing, the pod spec is invalid under Restricted, causing the kubelet to reject the pod or the container to fail immediately upon startup, triggering CrashLoopBackOff.

SC1-C
**Remediation Strategy:**
1.  **Immediate Fix (Image):** Rebuild the Go/Python images with a non-root user defined in the Dockerfile (e.g., `USER 1000`). Push to ECR and update image tags.
2.  **Manifest Fix:** Add `securityContext.runAsUser: 1000` to the Deployment spec to explicitly satisfy Restricted PSS requirements.
3.  **CI Pipeline Gate:** Add a step in CI to run `kubectl apply --dry-run=client` against a staging cluster with Restricted PSS enabled. Fail the build if the dry-run fails.
4.  **Rollout:** Use a rolling update strategy (`strategy: rollingUpdate`). Update the image tag in the Deployment. Monitor pod readiness. Since the fix is image-level, no downtime is required if the new pods become ready before old ones terminate.
5.  **Prevention:** Enforce PSS Restricted at the namespace level via Admission Controller (OPA Gatekeeper or Kyverno) to prevent non-compliant manifests from being applied in the future.

SC2-A
**Stale Data & Replication:**
1.  **Cause:** Cross-region replication is asynchronous. When the EU replica was promoted, it stopped receiving WAL (Write-Ahead Log) from the US primary. Any writes that occurred in US after the last WAL sync were lost or not yet shipped.
2.  **WAL Shipping:** PostgreSQL streams WAL records. Across regions, network latency and bandwidth limit throughput. Asynchronous replication means the primary acknowledges writes before the replica confirms receipt.
3.  **Data Loss:** The "data loss window" is the time between the last successful WAL sync and the failover moment.
4.  **Quantification:** Check `pg_stat_replication` on the old primary (if accessible) for `write_lag` and `flush_lag`. Check the promoted replica's `pg_last_wal_receive_lsn` vs `pg_last_wal_replay_lsn` before promotion to estimate lag.

SC2-B
**Terraform Operations:**
1.  **Import Replica:** Import the promoted EU instance into state under a new resource name.
    `terraform import aws_db_instance.promoted_replica arn:aws:rds:eu-west-1:123456789012:db:promoted-replica`
2.  **Detach Old Instance:** Remove the old US instance from Terraform state to stop managing it (since it's no longer the primary).
    `terraform state rm aws_db_instance.old_primary`
3.  **Update References:** Update the Terraform code to reference `aws_db_instance.promoted_replica` instead of the old ARN.
4.  **Apply:** Run `terraform apply` to ensure state matches code.

SC2-C
**Architecture Design:**
1.  **Aurora Global Database:** Recommend Aurora Global DB over standard RDS. It offers sub-second replication lag (RPO < 1s) across regions using a shared storage layer, whereas standard RDS async replication has higher lag.
2.  **Connection Management:** Use **RDS Proxy** instead of PgBouncer. RDS Proxy handles connection pooling and failover transparently at the AWS layer, reducing connection churn during failover.
3.  **DNS Failover:** Use **Route53 Health Checks** with Failover Routing Policy. Point the primary endpoint to the US RDS/Aurora and the secondary to EU. If US health check fails, Route53 switches traffic to EU.
4.  **Application Retry:** Implement exponential backoff with jitter in the application connection logic to handle transient connection errors during the DNS propagation window.
5.  **RPO < 1s:** Achieved via Aurora Global DB's shared storage architecture which replicates data at the storage layer rather than the WAL layer, minimizing lag.

SC10-C
**Automated Failover Orchestration:**
1.  **Detection:** Prometheus monitors site health (ping, DB latency). If Chicago site is unreachable for > 2 mins, trigger alert.
2.  **DNS Failover:** GSLB (Route53) detects Chicago health check failure. Switches DNS to Dallas.
3.  **Patroni Promotion:** Patroni in Dallas detects Chicago DCS unavailability. Promotes itself to Primary (Synchronous commit ensures no data loss).
4.  **Redis/RabbitMQ:** Sentinel/Federation automatically promotes replicas in Dallas.
5.  **MinIO:** Site Replication catches up.
6.  **Terraform State:** Ansible playbook updates OpenTofu state to reflect Dallas as Primary.
7.  **Split-Brain:** Use **etcd** as DCS. If Chicago is down, etcd in Dallas is the quorum. Chicago cannot promote itself.
8.  **Failback:** Once Chicago recovers, re-establish replication. Demote Dallas to Replica. Update DNS back to Chicago. Verify data consistency before cutover.
