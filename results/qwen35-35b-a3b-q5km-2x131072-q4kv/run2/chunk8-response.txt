CL1 — **IaaS (Infrastructure as a Service):** Provides virtualized computing resources (servers, storage, networking). *Example:* AWS EC2. **PaaS (Platform as a Service):** Provides hardware and software tools (runtime, databases) for application development. *Example:* Google App Engine. **SaaS (Software as a Service):** Delivers software applications over the internet. *Example:* Microsoft Office 365.

CL2 — **Shared Responsibility Model:** The provider secures the cloud (physical, hypervisor, network), the customer secures what is in the cloud. **IaaS VM:** Provider secures hardware/hypervisor; Customer secures OS, patching, data, firewall config. **Managed Kubernetes:** Provider secures control plane/infrastructure; Customer secures node OS, pod security, network policies, data. **SaaS (M365):** Provider secures app/infrastructure; Customer secures user access, data classification, device compliance.

CL3 — **VPC:** Isolated virtual network. **Subnet:** IP range within a VPC (public/private). **Security Group:** Instance-level firewall (stateful, allows traffic). **NACL:** Subnet-level firewall (stateless, allows/denies traffic). **Layering:** VPC contains Subnets; Subnets contain Instances; SG filters Instance traffic; NACL filters Subnet traffic. **Stateful:** SG remembers return traffic. **Stateless:** NACL requires explicit rules for both directions.

CL4 — **Landing Zone:** Secure, compliant baseline environment. **OUs:** Logical grouping of accounts (e.g., Prod, Dev). **Guardrails:** Preventative controls (e.g., "no public S3 buckets"). **Shared Networking:** Central hub (Transit Gateway/Hub) connects spokes (VPCs) for centralized egress. **Centralized Logging:** Aggregates logs to a security account. **Single-Account Risks:** High blast radius (one breach compromises all), billing confusion, permission sprawl, lack of isolation between environments.

CL5 — **IaC:** Managing infrastructure via code. **Terraform/OpenTofu:** Multi-cloud, declarative, state file management. **Pulumi:** General-purpose languages (TS, Python). **CloudFormation/Bicep:** Native to AWS/Azure, tight integration. **Crossplane:** Kubernetes-native IaC. **Choice:** Terraform for multi-cloud portability; Bicep for Azure-native simplicity; Pulumi for developer familiarity; Crossplane for K8s-centric workflows.

CL6 — **HPA:** Scales pods based on CPU/Memory. **VPA:** Adjusts pod resource requests/limits (requires restart). **Cluster Autoscaler:** Adds/removes nodes based on pod scheduling. **KEDA:** Scales based on external events (queues, metrics). **Use KEDA:** When scaling is driven by queue depth (Kafka, SQS) or custom business metrics rather than just CPU/Memory.

CL7 — **DR Strategy:** Define RPO (data loss tolerance) and RTO (downtime tolerance). **Active-Active:** Both regions serve traffic (low RPO/RTO, high cost). **Active-Passive:** One region active, one standby. **Database:** Sync (synchronous replication, low RPO, high latency) vs Async (higher RPO, lower latency). **DNS:** Route53 Health Checks/Traffic Manager for failover. **State:** Shared storage (EFS) vs replicated state. **DR Types:** **Pilot Light:** DB running, app off. **Warm Standby:** Scaled-down app running. **Multi-site Active-Active:** Full capacity in both regions.

CL8 — **GitOps:** Git repository is the single source of truth for infrastructure/app state. **ArgoCD:** Push-based, UI-focused. **FluxCD:** Pull-based, K8s-native. **Secrets:** Sealed Secrets (encrypted in Git), External Secrets Operator (fetches from Vault/AWS Secrets Manager). **Drift:** GitOps controller detects manual changes, automatically reverts them to match Git state.

CL9 — **Architecture:** Use **StatefulSet** for stable network IDs and ordered deployment. **PV/PVC:** Use `local` storage class for low latency (hostPath). **Headless Service:** For stable DNS names for StatefulSet pods. **Init Containers:** For config file generation before main app starts. **ConfigMaps:** Mount as volumes for config files. **PDB:** Ensure minimum available pods during updates to prevent data loss.

CL10 — **SA:** K8s identity for pods. **Workload Identity:** GCP/Azure native identity federation. **IRSA:** AWS IAM role linked to K8s SA via OIDC. **Mounting Creds:** Bad practice (leakage, rotation issues, long-lived keys). **Federation:** Uses OIDC tokens to exchange for short-lived cloud credentials, eliminating static secrets.

CL11 — **Object (S3):** Unstructured data, high durability (11 9s), high availability (99.99%). **Block (EBS):** Attached disk, low latency, attached to single instance. **File (EFS):** Shared file system, NFS protocol. **Use Case:** Object for backups/media, Block for DBs/OS, File for shared app configs.

CL12 — **Zero Trust:** Verify every request, no implicit trust. **Network Policies:** Default deny all traffic. **Service Mesh:** mTLS for service-to-service encryption. **Pod Identity:** SPIFFE/SPIRE for workload identity. **Admission Controllers:** OPA/Kyverno to enforce policies. **Image Signing:** Cosign to verify artifacts. **Runtime:** Falco for anomaly detection. **Difference:** Traditional relies on perimeter firewall; Zero Trust assumes breach and segments internally.

CL13 — **CDN:** Distributed network of edge servers caching content. **Mechanism:** Caches static assets closer to users. **Services:** AWS CloudFront, Azure CDN. **Use CDN:** For static content (images, CSS, JS) to reduce latency and origin load. **Use Origin:** For dynamic, personalized, or private content.

CL14 — **Region:** Geographic area with multiple AZs. **AZ:** Isolated data center within a region. **Deploy across AZs:** To survive data center failure. **Zonal:** Resource exists in one AZ (e.g., EBS). **Regional:** Resource spans AZs (e.g., ELB, RDS Multi-AZ).

CL15 — **Managed DB:** Service like RDS. **Provider Handles:** OS patching, backups, failover, hardware maintenance, scaling. **Customer Handles:** Schema, queries, access control, data.

CL16 — **IAM:** **Users:** Human identity. **Groups:** Collections of users. **Roles:** Temporary access for entities. **Policies:** JSON permissions. **Least Privilege:** Grant only necessary permissions. **Apply:** Assign policies to Groups/Roles, not individual Users; use Role assumption for cross-account access.

CL17 — **Serverless:** Event-driven execution, no server management. **Compare:** Lambda (AWS), Functions (Azure), Cloud Functions (GCP). **Cold Starts:** Latency when function is invoked after inactivity. **Poor Fit:** Long-running processes, stateful apps, high-throughput consistent workloads.

CL18 — **Ingress:** K8s resource for HTTP/HTTPS routing. **Gateway API:** New standard, supports gRPC/TCP, better traffic splitting. **LB Annotations:** Cloud-specific LB config. **Mesh Gateway:** Istio ingress. **TLS:** Ingress terminates at LB/Ingress Controller; Mesh terminates at sidecar. **Use Gateway API:** For complex routing, multi-protocol support, or when moving away from cloud-specific annotations.

CL19 — **Strategy:** **RI/SP:** Commit to usage for discount. **Spot:** Interruptible instances for fault-tolerant workloads. **Right-sizing:** Match instance size to actual usage. **Tiering:** Move old data to Glacier. **Idle Detection:** Tag and terminate unused resources. **FinOps:** Culture of cost accountability. **Tools:** AWS Cost Explorer, CloudHealth. **Showback/Chargeback:** Allocate costs to teams via tags.

CL20 — **Strategy:** **Abstraction:** Kubernetes (K8s) as common runtime. **IaC:** Terraform for portability. **Networking:** Direct Connect/ExpressRoute for low latency. **Identity:** OIDC federation. **Trade-offs:** Increased complexity, latency, cost. **Justified:** Regulatory requirements, redundancy, negotiation leverage. **Unnecessary:** If single cloud meets all needs with lower operational overhead.

OT1 — **Purdue Model:** ISA-95 standard. **Level 0:** Physical Process (Sensors/Actuators). **Level 1:** Basic Control (PLCs/RTUs). **Level 2:** Supervisory Control (SCADA/HMI). **Level 3:** Operations (MES/Production). **Level 4:** Enterprise (ERP/Business).

OT2 — **IT:** Data confidentiality, patching, security focus. **OT:** Safety, uptime, legacy hardware. **Patching:** OT patching risks downtime or safety hazards; legacy systems may not support modern patches; testing cycles are long.

OT3 — **DMZ (IDMZ):** Buffer zone between IT and OT. **Traffic:** Historian to SCADA (down), SCADA to Historian (up). **Direct L4 to L1:** Dangerous because it bypasses supervisory controls, allowing direct manipulation of safety-critical devices from corporate network.

OT4 — **Protocols:** 1. **Modbus TCP:** L1/L2, simple, unencrypted. 2. **OPC UA:** L2/L3, secure, platform-independent. 3. **EtherNet/IP:** L1/L2, Allen-Bradley, CIP. 4. **PROFINET:** L1/L2, Siemens, real-time. 5. **MQTT:** L3/L4, lightweight, IoT.

OT5 — **Data Diode:** Hardware allowing one-way data flow. **Firewall:** Two-way filtering. **Use Diode:** When exporting data from secure OT to IT is required but IT must never send commands to OT. **Limitation:** No feedback loop; cannot receive configuration updates from secure side.

OT6 — **Flow:** PLCs -> Historian -> OPC UA Server -> Edge Gateway (Protocol conversion) -> MQTT Broker -> Cloud Ingestion. **Security:** Place firewall between Edge Gateway and Cloud. Encrypt traffic (TLS). Isolate Edge Gateway in DMZ.

OT7 — **IEC 62443:** **Zone:** Logical grouping of assets with similar security requirements. **Conduit:** Communication path between zones. **Model:** Zone 4 (Corporate) -> Conduit -> Zone 3 (DMZ/SCADA) -> Conduit -> Zone 2 (Engineering) -> Conduit -> Zone 1 (PLCs/SIS). Engineering workstations in Zone 2/3 boundary.

OT8 — **Segmentation:** **VLANs:** One per production line. **Firewalls:** Between VLANs and to DMZ. **Switches:** Layer 3 core with ACLs. **Vendor Access:** Jump host with MFA and session recording. **Transition:** Phased approach, parallel run, backup configurations, maintenance windows.

OT9 — **SIS:** Independent safety system (fails safe). **BPCS:** Basic process control. **IEC 61511:** Safety lifecycle standard. **Air-gap:** Prevents cyber attacks from compromising safety functions. **Ransomware on SIS:** Physical danger, inability to shut down safely, potential for catastrophic failure.

OT10 — **PLC:** Logic controller, L1. **DCS:** Distributed control, L1/L2, process-heavy. **SCADA:** Supervisory, L2/L3, wide area. **Coexist:** Yes, PLCs/DCS control process, SCADA monitors and logs.

OT11 — **Remote Access:** **Jump Host:** Single entry point. **MFA:** Required. **Recording:** Session audit. **Time-limited:** Just-in-time access. **Direct RDP:** Unacceptable because it exposes HMI to corporate malware, lacks granular control, and bypasses OT security controls.

OT12 — **Plan:** **Monitor:** IDS for OT protocols. **Inventory:** Identify compromised assets. **IR Steps:** Isolate segment, preserve evidence, coordinate with ops. **Ops:** Maintain physical safety, manual override if needed. **Operator Role:** Critical for safety decisions, must not be distracted by IT incident response.

OT13 — **HMI:** Operator interface for monitoring/control. **Role:** Visualize process, allow manual intervention. **Diff:** HMI is for running (Level 2); Engineering Workstation is for configuration/programming (Level 2/3).

OT14 — **Historian:** Time-series database for process data. **Importance:** Trend analysis, compliance, troubleshooting. **Plant (L3):** Real-time, high frequency. **Enterprise (L4):** Aggregated, lower frequency, business analytics.

OT15 — **Ladder Logic:** Graphical programming language mimicking relay logic. **Device:** PLC. **Dominant:** Reliable, visual, widely understood by electricians, legacy code base.

OT16 — **RTU:** Remote Terminal Unit. **Diff:** RTU optimized for comms and harsh environments; PLC optimized for local control. **Deploy RTU:** Pipelines, remote substations, long-distance telemetry.

OT17 — **Analog:** Continuous signal (4-20mA). **Digital:** Discrete (On/Off). **4-20mA:** Preferred over 0-20mA because 0mA indicates wire break (fail-safe), whereas 0mA could be a valid reading in 0-20mA.

OT18 — **Asset Inventory:** Foundation of security (cannot protect what you don't know). **Tools:** Passive (packet sniffing), Active (scanning). **Active Scanning:** Dangerous because legacy OT devices may crash or malfunction under scan traffic.

OT19 — **MITRE ATT&CK ICS:** Framework for ICS threats. **Tactics:** 1. Initial Access (Spearphishing). 2. Execution (Command and Scripting Interpreter). 3. Persistence (Scheduled Task). 4. Privilege Escalation (Exploitation). 5. Defense Evasion (Obfuscated Files). **Diff:** Focuses on physical impact and industrial protocols. **Usage:** Map defenses to tactics to find gaps.

OT20 — **SOC:** **Logs:** Passive collection (SPAN ports). **SIEM:** Correlate IT/OT logs. **Rules:** OT-specific (e.g., PLC write attempts). **Triage:** OT engineer involvement required. **False Positives:** High noise due to legacy protocols. **Firmware Alert:** Verify change request, isolate PLC, rollback if unauthorized, investigate source.
