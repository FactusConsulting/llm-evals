AA1 — **Monolith vs. Microservices:** A monolith is a single deployable unit where all components are tightly coupled. Microservices are independently deployable, loosely coupled services. **Problems introduced by microservices:** 1. **Distributed Complexity:** Network latency, partial failures, and distributed tracing are required. 2. **Data Consistency:** ACID transactions across services are hard; eventual consistency is often needed. 3. **Operational Overhead:** Requires CI/CD pipelines, container orchestration, and service discovery infrastructure.

AA2 — **Sync vs. Async:** Synchronous communication blocks the caller until a response is received (e.g., HTTP/gRPC). Asynchronous communication sends a message and continues processing without waiting (e.g., RabbitMQ/Kafka). **Example Tech:** HTTP (Sync) vs. RabbitMQ (Async).

AA3 — **API Gateway vs. Reverse Proxy:** A reverse proxy routes traffic (L4/L7) and handles SSL termination. An API Gateway adds business logic (auth, rate limiting, API versioning). **Can Gateway replace Proxy?** No. Gateways often rely on underlying reverse proxies (like Envoy/Nginx) for low-level traffic management. A Gateway adds logic; a Proxy handles transport.

AA4 — **Saga Pattern:** Manages distributed transactions via a sequence of local transactions. **Choreography:** Services emit events; others react (decentralized). **Orchestration:** A central coordinator tells services what to do (centralized). **Use Case:** Choreography for simple flows; Orchestration for complex workflows requiring compensation logic and visibility.

AA5 — **CQRS:** Separates read (Query) and write (Command) models. **Paired with Event Sourcing:** Event Sourcing stores state changes as events. **Problem Solved:** Allows independent scaling of read/write paths and provides a complete audit trail/history of state changes, which CRUD models lack.

AA6 — **Circuit Breaker:** Prevents repeated calls to a failing service. **States:** Closed (Normal), Open (Fail fast), Half-Open (Test recovery). **Differs from Retry:** Retry attempts to recover; Circuit Breaker stops attempts to prevent cascading failure. **Use Together:** Retry for transient network glitches; Circuit Breaker for persistent service outages.

AA7 — **Multi-Tenant Isolation:**
1. **Shared DB/Schema:** Lowest cost, highest noisy-neighbor risk, complex security logic (row-level).
2. **Shared DB/Schema per Tenant:** Medium cost, better isolation, moderate complexity (schema management).
3. **Separate DB per Tenant:** Highest cost, best isolation/security, highest operational complexity (backup/migration per tenant).

AA8 — **Consistency:** Eventual (data converges over time) vs. Strong (immediate visibility). **Inventory Architecture:** Strong consistency required. **Data Store:** Relational DB (PostgreSQL) with Serializable isolation or distributed locks (Redis). **Pattern:** Optimistic locking (versioning) or Pessimistic locking (SELECT FOR UPDATE) to prevent overselling.

AA9 — **Sidecar Proxy:** A container deployed alongside the main app to handle network traffic. **Envoy/Istio:** Envoy is the data plane (traffic handling); Istiod is the control plane (config distribution). **mTLS:** Solves service identity verification and encryption without modifying application code, unlike app-level TLS which requires code changes.

AA10 — **Strangler Fig:** Incrementally migrate by routing specific traffic to new services. **Strategy:** Deploy .NET 8 services alongside legacy .NET Framework. Configure API Gateway to route `/api/v2/*` to new services and `/api/v1/*` to legacy. Gradually migrate endpoints until legacy is removed.

AA11 — **12-Factor App:** 1. Codebase (one repo), 2. Dependencies (explicit), 3. Config (env vars), 4. Backing Services (treat as attached resources), 5. Build/Release/Run (strict separation), 6. Processes (stateless). **Matter:** Ensures portability, scalability, and maintainability in cloud environments.

AA12 — **Event-Driven (Kafka) vs. Request-Driven (REST):**
*   **Coupling:** Event (Loose) vs. Request (Tight).
*   **Scalability:** Event (High) vs. Request (Moderate).
*   **Debugging:** Event (Hard) vs. Request (Easy).
*   **Ordering:** Event (Per-partition) vs. Request (None).
*   **Failure:** Event (Retry/Dead-letter) vs. Request (Timeout).
*   **Choice:** Event for decoupling/scaling; Request for real-time synchronous workflows.

AA13 — **Load Balancer:** Distributes traffic across servers. **L4 vs. L7:** L4 (IP/Port) vs. L7 (HTTP/URL/Headers). **Use Case:** L7 required for SSL termination or routing `/api` to backend A and `/static` to backend B.

AA14 — **Message Queue vs. Log Broker:** **MQ (RabbitMQ):** Task-oriented, acks, deletes after consumption. **Log Broker (Kafka):** Stream-oriented, retains data, replayable. **Pick MQ:** For task queues (email, processing). **Pick Kafka:** For event streaming, audit logs, real-time analytics.

AA15 — **Container vs. VM:** Container isolates processes (shares OS kernel); VM isolates OS (hypervisor). **Popularity:** Containers are lightweight, start fast, and ensure environment consistency for microservices.

AA16 — **Stateless vs. Stateful:** Stateless (no session data stored) vs. Stateful (session/data stored locally). **Scaling:** Stateless easier to scale horizontally (any instance handles any request). **Ex:** Web Server (Stateless) vs. Redis (Stateful).

AA17 — **ORM:** Maps DB tables to objects. **Examples:** Entity Framework (.NET), Hibernate (Java). **Trade-offs:** ORM speeds development but can generate inefficient SQL and hide performance bottlenecks compared to raw SQL.

AA18 — **Bulkhead Pattern:** Isolates resources (threads, connections) per service to prevent cascade failures. **Implementation:** Use separate thread pools for each downstream service or deploy services to separate Kubernetes nodes/pods to limit resource contention.

AA19 — **OT vs. CRDT:** OT (Server authoritative, complex conflict resolution) vs. CRDT (Client authoritative, mathematically guaranteed convergence). **Choice:** CRDT. **Why:** CRDTs allow offline editing with automatic merge upon reconnection without server intervention, whereas OT requires server coordination.

AA20 — **Distributed Tracing:** **Context Propagation:** Inject Trace ID into headers (W3C/B3). **Sampling:** Head (sample at start) vs. Tail (sample at end based on latency). **Storage:** Jaeger/Tempo. **Kafka:** Trace context must be injected into message headers so downstream consumers can continue the trace.

OP1 — **Hypervisor Type 1 vs. 2:** Type 1 (Bare-metal, e.g., ESXi, Hyper-V) runs directly on hardware. Type 2 (Hosted, e.g., VirtualBox, VMware Workstation) runs on an OS. Type 1 is for production; Type 2 is for development.

OP2 — **SAN vs. NAS:** SAN (Storage Area Network) provides block storage (e.g., iSCSI, Fibre Channel) for DBs. NAS (Network Attached Storage) provides file storage (e.g., NFS, SMB) for file shares. **Use SAN:** For high-performance databases. **Use NAS:** For shared user files.

OP3 — **3-Tier vs. Leaf-Spine:** 3-Tier (Core/Dist/Access) has oversubscription and latency at aggregation. Leaf-Spine (Spine switches connect to all Leaf switches) provides non-blocking, low-latency, scalable fabric ideal for modern data centers.

OP4 — **Proxmox HA:** 3+ nodes for quorum. **Quorum:** Majority vote. **Fencing:** DRAC/iLO to power off failed nodes. **Storage:** Ceph (distributed) or iSCSI/NFS (shared). **Failure:** VMs restart on surviving nodes automatically.

OP5 — **VLAN Trunking:** 802.1Q tags frames with VLAN ID. **Linux Config:** Create sub-interfaces (e.g., `eth0.10`, `eth0.20`) using `ip link` or `vconfig` to tag traffic for 3 VLANs on one NIC.

OP6 — **RAID Levels:** RAID 1 (Mirror), RAID 5 (Striping + Parity), RAID 6 (Double Parity), RAID 10 (Mirror + Stripe), RAID Z2 (ZFS Double Parity). **Choice:** RAID 10. **Why:** Best performance and fault tolerance for DBs with 6 drives (can lose 2 drives if mirrored pairs).

OP7 — **DNS Architecture:** Internal AD DNS (Authoritative for internal). External Public DNS (Authoritative for public). **Split-Horizon:** Internal clients get internal IPs. **Laptop:** DHCP provides DNS server IP. If on LAN, resolves internal AD. If on VPN, forwards to internal DNS. **DNSSEC:** Signs external zone for validation.

OP8 — **PKI Chain:** Root CA (Offline, Trust Anchor) -> Subordinate CA (Online, Issues Certs). **Compromise:** Root compromised = Trust lost globally (must rebuild PKI). Subordinate compromised = Revoke certs issued by it, issue new subordinate. **Auto-enrollment:** Windows AD CS auto-issues certs to domain-joined machines.

OP9 — **WAN Consolidation:** **MPLS:** High cost, reliable. **IPsec:** Low cost, variable latency. **SD-WAN:** Low cost, intelligent routing, QoS. **WireGuard:** Simple mesh. **Recommendation:** SD-WAN. **Why:** Cost-effective, handles VoIP QoS better than IPsec, easier management than MPLS.

OP10 — **iDRAC/iLO/IPMI:** Out-of-band management (power, console, BIOS). **Critical:** Allows recovery if OS is down. **Exposed:** If exposed to internet, attackers can reset passwords, power cycle, or install malware (IPMI is often unauthenticated).

OP11 — **LVM vs. ZFS:** LVM (Volume management) vs. ZFS (Filesystem + Volume). **Snapshots:** LVM (Copy-on-write, slower). ZFS (Instant, cheap, immutable). **Use ZFS:** For data integrity and snapshots. **Use LVM:** For simple volume management on standard filesystems.

OP12 — **Stretch Cluster:** 2 Sites + 1 Witness (Tiebreaker). **Ceph CRUSH:** Distribute replicas across sites. **Latency:** <5ms required for Ceph. **Split-Brain:** Without tiebreaker, both sites think they are primary. **Bad Idea:** 2-site Ceph without witness leads to data corruption on partition.

OP13 — **DHCP:** DORA (Discover, Offer, Request, Ack). **Unavailable:** Client uses APIPA (169.254.x.x) if lease expires and server is down. **Windows:** Tries to renew; if fails, uses APIPA.

OP14 — **Managed vs. Unmanaged:** Unmanaged (Plug-and-play). Managed (VLANs, QoS, ACLs, SNMP, Port Mirroring). **Importance:** Managed switches allow traffic prioritization (VoIP) and security segmentation (VLANs).

OP15 — **NTP:** Network Time Protocol. **Critical:** Cluster consensus, log correlation, certificate validation. **Drift:** Time skew causes Kerberos auth failures, log confusion, and database replication errors.

OP16 — **Physical vs. VM vs. Container:** Physical (Bare metal), VM (Virtual OS), Container (Virtualized Process). **Bare Metal:** Use for high-performance DBs, hardware passthrough needs, or latency-sensitive workloads.

OP17 — **UPS vs. Generator:** UPS (Battery, seconds/minutes) vs. Generator (Fuel, hours). **Need Both:** UPS bridges gap until generator starts. **ATS:** Automatic Transfer Switch switches power source from Utility to UPS to Generator seamlessly.

OP18 — **Active-Passive vs. Active-Active:** Active-Passive (One runs, one waits). Active-Active (Both run). **PostgreSQL HA:** Pacemaker/Corosync cluster. **Fencing:** STONITH (power off failed node). **VIP:** Virtual IP moves to active node on failover.

OP19 — **Backup Strategy:** 3-2-1 Rule (3 copies, 2 media, 1 offsite). **Immutable:** Protects against ransomware. **Test:** Regular restore drills. **Backup vs. DR:** Backup is data copy; DR is the plan/process to recover operations after a disaster.

OP20 — **802.1X:** **Roles:** Supplicant (Client), Authenticator (Switch), Server (RADIUS). **EAP:** EAP-TLS (Cert-based, secure) vs. PEAP (Password-based). **Fail:** Port blocked. **MAB:** Switch sends MAC address to RADIUS for auth if 802.1X fails (for printers/IoT).
