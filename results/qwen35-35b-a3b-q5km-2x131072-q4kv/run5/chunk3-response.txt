T1 — `tofu plan` calculates the execution plan, showing what changes will be made without applying them. `tofu apply` executes the plan, creating, updating, or destroying resources to match the configuration.

T2 — State is a JSON file tracking the mapping between OpenTofu resources and real-world infrastructure. Remote state is important for collaboration (sharing state among team members), locking (preventing concurrent writes), and security (storing sensitive data off local machines).

T3 — `count` uses integer indices; `for_each` uses maps or sets. `for_each` is preferred when resources have unique identifiers (like names or IDs) because adding/removing items doesn't shift indices, preventing unintended resource recreation.

T4 — This is "state drift". OpenTofu detects it during `tofu plan` by comparing the state file to the live infrastructure. If the drift is unmanaged, `tofu apply` will attempt to revert it to match the configuration, or fail if the resource is managed differently.

T5 —
```hcl
resource "aws_instance" "web" {
  for_each = toset(["web1", "web2", "web3"])
  ami      = "ami-12345"
  instance_type = "t2.micro"
  tags = {
    Name = each.key
  }
}
```

T6 — `terraform_remote_state` reads the entire state file of another configuration. Output values are specific attributes. The recommended alternative is using **modules** to encapsulate logic and pass outputs directly, or using a **shared state backend** with proper access control, to avoid tight coupling and state file dependencies between root modules.

T7 — The `null_resource` provisioner runs asynchronously or after the resource creation in the apply cycle, so the output is empty when dependent resources try to read it immediately. Fix: Add `depends_on = [null_resource.kubeconfig]` to the dependent resources to ensure they wait for the provisioner to complete.

T8 — `tofu import` adds existing infrastructure to state without creating config. Limitations: No config generation, manual mapping required. The `import` block (1.5+) allows importing within the configuration file, enabling version control, CI/CD integration, and better state management.

T9 — A variable is an input parameter passed into the configuration (e.g., via CLI or `.tfvars`). An output is a value exposed from the configuration to be used by other modules or the CLI.

T10 — A provider is a plugin that interacts with a specific API (e.g., AWS, Azure). Running `tofu plan` without `tofu init` fails because the required provider plugins are not downloaded or initialized.

T11 — The `.terraform.lock.hcl` file records the exact versions of providers and plugins used. It should be committed to version control to ensure reproducible builds and prevent unexpected provider updates.

T12 — `tofu destroy` removes all resources managed by the configuration. To target a single resource: `tofu destroy -target=aws_instance.example`.

T13 — A data source reads information from an external system without creating or managing it. Example: Using `data "aws_ami"` to find the latest Ubuntu AMI ID instead of hardcoding it.

T14 — `locals` define computed values within the configuration scope. `variables` are inputs from outside. Use `locals` for intermediate calculations or complex logic that shouldn't be exposed as inputs.

T15 — The `lifecycle` block controls resource behavior. `create_before_destroy` creates new resources before destroying old ones (zero downtime). `prevent_destroy` prevents accidental deletion. `ignore_changes` ignores specific attributes to prevent drift detection.

T16 — The `moved` block tracks resource identity changes during refactoring (e.g., moving a resource to a new module). It updates the state file to map the old address to the new address without destroying and recreating the resource.

T17 — The dependency graph is a DAG determining execution order. Implicit dependencies are inferred from references (e.g., `aws_instance.id`). Explicit dependencies use `depends_on`. `depends_on` reduces parallelism because it forces sequential execution, slowing down apply times.

T18 — Strategies: 1. Use `terraform plan -refresh=false` if state is fresh. 2. Modularize to reduce graph complexity. 3. Optimize backend (e.g., S3 vs Local). 4. Increase `parallelism` in `terraform init`. Tradeoffs: `-refresh=false` risks stale state; modularization increases maintenance overhead.

T19 — State locking prevents concurrent modifications. If a lock isn't released (e.g., CI crash), subsequent applies fail. Manually break it with `terraform force-unlock <LOCK_ID>`. Risk: Data corruption if another process is actively writing.

T20 — 1. Update `backend` block in config to S3/DynamoDB. 2. Run `terraform init -migrate-state`. 3. OpenTofu pulls local state, uploads to S3, and deletes local copy. 4. Verify with `terraform state list`. 5. Run `terraform plan` to confirm state matches.

A1 — A **playbook** is a YAML file defining automation steps. A **role** is a reusable unit of logic (tasks, vars, handlers). A **task** is a single action within a playbook or role.

A2 — `ansible_facts` contains gathered system information. Access OS family via `{{ ansible_os_family }}`.

A3 — `command` runs without shell features (no pipes). `shell` runs with shell features (pipes, variables). `raw` runs without setup (no Python module transfer), useful for initial bootstrap.

A4 — Idempotency means running a task multiple times yields the same result. Example (Idempotent): `apt: name=nginx` (installs only if missing). Example (Not Idempotent): `shell: rm -rf /tmp/data` (deletes every time).

A5 —
```yaml
- name: Install htop
  package:
    name: htop
  when: ansible_os_family == "Debian"

- name: Install htop on RedHat
  package:
    name: htop
  when: ansible_os_family == "RedHat"
  # Add epel-release task separately if needed
```

A6 — Priority (High to Low): Extra Vars (`-e`) > Command Line Vars > Play Vars > Host Vars > Group Vars > Role Vars > Default Vars. `set_fact` sets runtime variables that override defaults but are lower than extra vars.

A7 — Use `serial: 3` to limit batch size. Group nodes into `rke2_nodes`. Use `wait_for` or `meta: wait_for` to ensure the first 3 nodes are ready before the next batch starts.

A8 — `ansible-vault` encrypts sensitive data. To encrypt a single variable: `ansible-vault encrypt_string 'secret_value' --name 'my_secret'`.

A9 — An inventory file lists managed hosts. **Static** inventory is a fixed file. **Dynamic** inventory is generated by a script or API (e.g., AWS EC2) to reflect changing infrastructure.

A10 — `ansible webservers -m shell -a "df -h"`.

A11 — A handler is a task that runs only when notified by another task (e.g., `notify: restart service`). It differs from a regular task by its conditional execution based on state changes.

A12 — `ansible.cfg` configures Ansible behavior. Common settings: `inventory`, `remote_user`, `forks`, `host_key_checking`.

A13 — `register` saves a task's output to a variable. Use it in subsequent tasks via `when: my_var.stdout == 'success'` or `debug: var=my_var`.

A14 — `import_tasks` is static (evaluated at compile time). `include_tasks` is dynamic (evaluated at runtime). Use `import` for performance, `include` for conditional loading.

A15 — Ansible Galaxy is a repository for roles and collections. Use `requirements.yml` to list dependencies, then run `ansible-galaxy install -r requirements.yml`.

A16 — Connection plugins define how Ansible talks to hosts. `ssh` (remote Linux), `local` (control node), `network_cli` (network devices like Cisco). Use `ssh` for servers, `network_cli` for routers/switches.

A17 — `delegate_to: localhost` runs a task on the control node. `run_once` ensures it runs once even in a loop. Example: Gather facts from remote hosts, then process them on localhost to generate a config file.

A18 — Strategies: 1. Increase `forks` (parallel connections). 2. Enable `pipelining` (reduce SSH overhead). 3. Use `fact caching` (avoid re-gathering facts). 4. Use `Mitogen` (optimized transport). Tradeoffs: Higher forks increase network load; caching risks stale data.

A19 — `{{ }}` outputs variables. `{% %}` controls logic (loops, ifs). `{# #}` comments.
Snippet:
```jinja2
upstream backend_servers {
{% for server in servers %}
    server {{ server.ip }} weight={{ server.weight }}; # Health check: {{ server.health }}
{% endfor %}
}
```

A20 — 1. Set `serial: 1` for one-by-one deployment. 2. `pre_tasks`: Deregister from load balancer. 3. Deploy app. 4. `post_tasks`: Health check via `uri` module. 5. If check fails, `rescue` block reverts changes and re-registers previous version. 6. Register new version on success.
