G1 — **Slice vs Array:** An array is a fixed-size value type `[T]N`. A slice is a dynamic view over an array, defined as a struct containing a pointer to data, length, and capacity (`[T]`). When appending to a slice with no remaining capacity, the Go runtime allocates a new, larger underlying array (usually doubling capacity), copies elements, and updates the slice header.

G2 — **Variable Declaration:** `var x int` declares a variable with explicit type and zero value. `x := 0` is a short declaration that infers the type. You cannot use `:=` if the variable is already declared in the same scope, if you are at package level (outside functions), or if the type cannot be inferred.

G3 — **Goroutines vs Threads:** A goroutine is a lightweight thread managed by the Go runtime, not the OS. It uses less memory and context-switching overhead. The GMP model: **G** (goroutine), **M** (OS thread), **P** (logical processor). The scheduler maps Gs to Ps, which run on Ms, enabling efficient multiplexing of goroutines onto OS threads.

G4 — **Channels:** Channels are typed conduits for communication. **Unbuffered** channels block both sender and receiver until the other side is ready. **Buffered** channels allow sending without blocking up to capacity. Sending to a full buffered channel blocks until a receiver reads from it.

G5 — **Context:** The `context` package manages deadlines, cancellation, and request-scoped values across API boundaries.
```go
ctx, cancel := context.WithTimeout(context.Background(), 5*time.Second)
defer cancel()
req, _ := http.NewRequestWithContext(ctx, "GET", url, nil)
client.Do(req) // Cancels if timeout exceeded
```

G6 — **Interfaces:** Go interfaces are satisfied implicitly (duck typing). Unlike Java/C#, you do not declare implementation. The empty interface `interface{}` (or `any`) accepts any type. It is powerful for generic storage but dangerous because type assertions can panic if the type is wrong, requiring careful handling.

G7 — **Memory Model:** This code has a data race. The writer goroutine writes `data` and `ready` without synchronization. The reader loop spins on `ready` without a memory barrier, potentially seeing `true` while `data` is still zero. Fix using `sync/atomic` or `sync.Mutex` to ensure visibility.

G8 — **Mutex vs RWMutex:** `sync.Mutex` allows exclusive access. `sync.RWMutex` allows multiple readers or one writer. `sync.Map` is optimized for sparse keys or read-heavy workloads where keys are added rarely. Regular maps with mutexes are better for dense keys or frequent writes due to lower overhead.

G9 — **Modules:** `go.mod` declares dependencies and the module path. `go.sum` stores checksums for reproducibility. `go mod tidy` removes unused imports and adds missing ones. **MVS** (Minimum Version Selection) selects the lowest compatible version of a dependency that satisfies all requirements.

G10 — **Defer:** `defer` schedules a function call to run after the surrounding function returns. Multiple defers execute in **LIFO** (Last In, First Out) order. A common gotcha in loops: `defer` captures the loop variable by reference, so all deferred calls often see the final loop value unless captured in a closure.

G11 — **Pointers:** A pointer holds a memory address. The zero value is `nil`. To create a pointer to a struct literal: `p := &MyStruct{Field: "value"}`. Dereference with `*p`.

G12 — **Init:** `init()` runs automatically before `main()` when a package is loaded. A package can have multiple `init()` functions; they execute in the order they appear in the source files, then in the order of file names.

G13 — **Make vs New:** `new(T)` allocates zeroed memory for type `T` and returns a pointer `*T`. `make(T)` initializes slices, maps, and channels and returns the initialized type (not a pointer). `new` is for any type; `make` is only for built-in reference types.

G14 — **Error Handling:** Errors are values implementing the `error` interface (`Error() string`). Idiom: `if err != nil { return err }`. `errors.Is` checks semantic equality (wrapping). `errors.As` performs type assertion on wrapped errors. `==` only checks exact pointer/value equality, not wrapped chains.

G15 — **Composite Types:**
*   **Map:** Zero value is `nil` (uninitialized).
*   **Slice:** Zero value is `nil`.
*   **Channel:** Zero value is `nil`.
*   **Struct:** Zero value has all fields set to their respective zero values.

G16 — **Race Condition:** Occurs when two goroutines access shared data concurrently, and at least one writes. The `-race` flag instruments code to detect these at runtime. Fix with `sync.Mutex` (lock before access) or channels (communicate, don't share memory).

G17 — **Select:** `select` waits on multiple channel operations. Unlike `switch`, it chooses a ready case randomly if multiple are ready. Implement timeout by adding a case with `time.After(duration)`. If timeout fires first, the default case or timeout case executes.

G18 — **Escape Analysis:** The compiler determines if a variable lives on the stack or heap. If a variable's address escapes the function scope, it moves to the heap. Inspect with `go build -gcflags="-m"`. Heap allocation is slower (GC pressure); stack allocation is faster.

G19 — **Reflection:** `reflect.Type` describes the type metadata. `reflect.Value` holds the dynamic value. Reflection is slow due to runtime type inspection and dynamic dispatch. Justified for serialization frameworks or generic libraries where compile-time types are unknown.

G20 — **Embed:** `//go:embed` embeds files/directories at compile time into a `fs.FS`. Works with single files (`//go:embed file.txt`) or directories (`//go:embed images/*`). Limitations: files must exist at compile time. Interacts with `io/fs` to read embedded content as standard file system interfaces.

R1 — **Ownership:** Rust ensures memory safety without GC. Rules: 1) Each value has one owner. 2) When owner goes out of scope, value is dropped. 3) You can transfer ownership (move) or borrow.

R2 — **String vs &str:** `String` is an owned, growable heap buffer. `&str` is a borrowed slice of bytes (string slice). Use `String` when you need to own/modify data. Use `&str` for function arguments to avoid allocation and copying.

R3 — **Borrowing:** You can have many immutable references (`&T`) OR one mutable reference (`&mut T`), but not both simultaneously. This prevents data races at compile time. References must be valid (no dangling pointers).

R4 — **Result/Option:** `Result<T, E>` represents success or error. `Option<T>` represents value or nothing (`None`). The `?` operator propagates errors up the call stack.
```rust
fn read_file() -> Result<String, io::Error> {
    std::fs::read_to_string("file.txt")
}
```

R5 — **Box/Rc/Arc:** `Box<T>` owns data on heap (single owner). `Rc<T>` (Reference Counted) allows multiple owners in single-threaded contexts. `Arc<T>` (Atomic Reference Counted) allows multiple owners across threads. Use `Arc` when sharing across threads.

R6 — **Traits:** Traits define shared behavior. Unlike Java interfaces, they can have default implementations. `dyn Trait` enables dynamic dispatch (vtable). `T: Trait` enables static dispatch (monomorphization), which is generally faster.

R7 — **Lifetimes:** The compiler cannot infer that the returned reference is valid for the lifetime of `x` or `y`. Fix by annotating lifetimes:
```rust
fn longest<'a>(x: &'a str, y: &'a str) -> &'a str {
    if x.len() > y.len() { x } else { y }
}
```

R8 — **Send/Sync:** `Send` means a type can be transferred to another thread. `Sync` means a type can be referenced from another thread. `Rc<T>` is not `Send` because its ref count is not atomic. `Arc<T>` uses atomic ref counts, making it `Send` and `Sync`.

R9 — **Async/Await:** Rust `async` creates a `Future` (state machine). The executor (e.g., Tokio) polls the future until completion. Unlike Go's stack-switching goroutines, Rust futures are polled cooperatively on a single thread (usually), avoiding stack overflow but requiring explicit executor management.

R10 — **Unsafe:** `unsafe` blocks allow raw pointer dereferencing, union access, and calling unsafe functions. Justified for FFI, implementing unsafe abstractions, or performance-critical low-level ops where safety checks are unnecessary overhead.

R11 — **Pattern Matching:** `match` is exhaustive; the compiler forces handling of all possible enum variants. `if let` is a shorthand for matching a single pattern. `match` is preferred for complex logic; `if let` for simple cases.

R12 — **Vec vs Array:** `Vec<T>` is a growable heap-allocated array. `[T; N]` is a fixed-size stack-allocated array. When `Vec` capacity is exceeded, it allocates a new larger buffer and moves elements (reallocation).

R13 — **Enums:** Rust enums are algebraic data types (can hold data). Unlike C/Java, they can store variants with different types. `Option<T>` is `enum { Some(T), None }`. Rust has no `null`; `None` replaces it, forcing explicit handling.

R14 — **Impl Block:** `impl` defines methods. `&self` borrows the instance. `&mut self` mutably borrows. `self` consumes the instance (move). Associated functions (e.g., `String::new`) do not take `self` and act like constructors.

R15 — **Derive Macro:** `#[derive(Debug)]` generates boilerplate code. Common traits: `Debug`, `Clone`, `PartialEq`. Cannot derive if the type contains fields that don't support the trait (e.g., `Clone` on `&T` or `Fn` traits).

R16 — **Cow:** `Cow<'a, T>` (Clone on Write) borrows data if possible, clones only if mutation is needed. Useful for functions returning data that might be borrowed or owned, avoiding unnecessary heap allocations when no mutation occurs.

R17 — **Modules:** `mod` declares a module. `use` imports paths. `pub` makes items visible outside the crate. Rust 2018 allows `mod.rs` or `mod.rs` in subdirectories to map to file structure more intuitively than 2015's strict `mod.rs` requirement.

R18 — **Pin:** `Pin<T>` prevents a value from being moved in memory. Needed for self-referential structs and async futures (which may hold references to themselves). `Unpin` is a marker trait indicating a type *can* be moved safely.

R19 — **Macros:** `macro_rules!` are declarative (pattern matching on tokens). Procedural macros (derive, attribute, function-like) are Rust code that runs at compile time to generate code. Hygiene ensures macro variables don't clash with local scope variables.

R20 — **Zero-Cost Abstractions:** Rust abstractions compile down to efficient machine code. Generics use monomorphization (code duplication per type). Iterators are inlined loops. Traits use static dispatch by default. No runtime overhead compared to C.
