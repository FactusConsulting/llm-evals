AA1 — Monoliths are single deployable units where all components run in one process. Microservices are independent, loosely coupled services deployed separately. Microservices introduce: 1) Network latency and failure points between services, 2) Distributed data consistency challenges (no ACID across services), 3) Operational complexity (deployment orchestration, monitoring, tracing across many services).

AA2 — Synchronous communication blocks the sender until a response is received (e.g., HTTP/REST, gRPC). Asynchronous communication sends a message without waiting for an immediate response (e.g., RabbitMQ, Kafka). Sync is for real-time requests; Async is for decoupling and background processing.

AA3 — A reverse proxy routes traffic to backend servers (L4/L7). An API Gateway adds business logic (auth, rate limiting, transformation). An API Gateway cannot always replace a reverse proxy entirely; a dedicated L4 proxy (e.g., HAProxy) is often kept in front for DDoS protection, SSL offloading, or high-performance routing before traffic hits the Gateway.

AA4 — The Saga pattern manages distributed transactions via a sequence of local transactions. Choreography uses events to trigger the next step (decentralized); Orchestration uses a central coordinator to manage the flow. Choreography is appropriate for simple, decoupled flows. Orchestration is better for complex workflows requiring compensation logic and visibility.

AA5 — CQRS separates read (Query) and write (Command) models. It is paired with Event Sourcing because events provide the immutable log needed to rebuild read models. This solves the problem of scaling reads and writes independently and provides a complete audit trail, which a traditional CRUD model (single table per entity) does not.

AA6 — The Circuit Breaker pattern stops calling a failing service to prevent cascading failures. States: Closed (normal), Open (fail fast), Half-Open (testing recovery). It differs from retry by stopping traffic rather than persisting attempts. Use both together: Retry for transient network glitches, Circuit Breaker for persistent service outages.

AA7 — 1. Shared DB/Schema: Lowest cost, highest noisy-neighbor risk, security risk (tenant ID leakage). 2. Shared DB/Schema per tenant: Better isolation, moderate cost, complex schema migrations. 3. Separate DB per tenant: Best isolation/security, highest cost/ops complexity, no noisy-neighbor risk.

AA8 — Strong consistency ensures immediate visibility of updates; eventual consistency allows temporary divergence. For inventory (no overselling), use Strong Consistency. Architecture: SQL database (e.g., PostgreSQL) with distributed locking (Redis) or optimistic locking (version columns). At scale, use a single-writer queue or sharding with strict transactional boundaries.

AA9 — A sidecar proxy runs alongside a service to handle network traffic. In Istio, Envoy is the data plane (handles traffic), while Istiod is the control plane (configures Envoy). mTLS between sidecars solves identity verification and encryption at the network layer, removing the burden of certificate management from the application code.

AA10 — The Strangler Fig pattern incrementally migrates functionality from a legacy system to new services. Strategy: Place an API Gateway in front of the monolith. Route specific endpoints (e.g., `/api/inventory`) to new .NET 8 microservices while routing others to the legacy .NET Framework app. Gradually move all endpoints until the monolith is decommissioned.

AA11 — 12-Factor App: 1. Codebase (one repo), 2. Dependencies (explicit), 3. Config (env vars), 4. Backing services (treat as attached resources), 5. Build/Release/Run (separate), 6. Processes (stateless). Matters for cloud-native because it ensures portability, scalability, and separation of concerns.

AA12 — Event-driven (Kafka): Low coupling, high scalability, hard debugging, ordering per-partition. Request-driven (REST): High coupling, moderate scalability, easy debugging, global ordering. Choose Event-driven for decoupling and high throughput. Choose Request-driven for real-time, synchronous interactions.

AA13 — A load balancer distributes traffic across servers. Layer 4 (TCP/UDP) routes based on IP/Port. Layer 7 (HTTP) routes based on headers/URL. Layer 7 is required for SSL termination or routing `/api` to one service and `/web` to another on the same IP.

AA14 — A message queue stores messages for processing. RabbitMQ (Queue) is for task distribution and acks. Kafka (Log) is for event streaming and replay. Pick RabbitMQ for task queues (fire-and-forget). Pick Kafka for event sourcing or high-volume streaming data.

AA15 — A container packages code and dependencies into a lightweight image. Unlike a VM, it shares the host OS kernel. Containers are popular for microservices because they start faster, use less memory, and scale easily compared to VMs.

AA16 — Stateless services store no session data (any instance can handle any request). Stateful services store session/data locally. Stateless is easier to scale horizontally because you can add/remove instances without data migration. Example: Stateless (Web Server), Stateful (Database).

AA17 — An ORM maps database tables to objects. Examples: Hibernate (Java), Entity Framework (.NET). Trade-offs: ORMs increase development speed but can generate inefficient SQL and hide performance bottlenecks compared to raw SQL.

AA18 — The Bulkhead pattern isolates resources (threads, connections) for different services. If one service fails, it doesn't exhaust shared resources. Implementation: Use separate thread pools for each downstream service call or deploy critical services on separate node groups.

AA19 — OT requires server authority to resolve conflicts (complex, high latency). CRDTs allow clients to merge changes autonomously (eventual consistency). Choose CRDTs for offline editing because they allow local changes to merge automatically when connectivity returns without server intervention.

AA20 — Challenges include context propagation across languages. W3C Trace Context/B3 headers carry TraceID/SpanID. Sampling: Head-based (decide early), Tail-based (decide after). Storage: Jaeger/Tempo. Kafka continuity: Inject trace headers into message metadata so downstream consumers can resume the trace.

OP1 — Type 1 (Bare-metal): Runs directly on hardware (e.g., VMware ESXi, Microsoft Hyper-V). Type 2 (Hosted): Runs on an OS (e.g., VirtualBox, VMware Workstation). Type 1 is for production; Type 2 is for development.

OP2 — SAN (Storage Area Network): Block storage, uses iSCSI or Fibre Channel. Used for databases. NAS (Network Attached Storage): File storage, uses NFS or SMB. Used for file shares. SAN offers lower latency for DBs; NAS is easier to manage for files.

OP3 — 3-tier (Core/Distribution/Access) is hierarchical. Leaf-Spine is preferred in modern DCs because it provides non-blocking, low-latency connectivity between all servers, solving the oversubscription and latency bottlenecks of the 3-tier model.

OP4 — Proxmox HA: 3+ nodes for quorum. Use DRAC/iLO for fencing. Shared storage: Ceph (distributed) or iSCSI/NFS. During node failure: Quorum votes, fencing isolates failed node, VMs restart on surviving nodes.

OP5 — VLAN trunking carries multiple VLANs over one link using 802.1Q tags. On Linux: Create sub-interfaces (e.g., `eth0.10`) with `ip link add link eth0 name eth0.10 type vlan id 10`. Assign IPs to sub-interfaces.

OP6 — RAID 1 (Mirror), 5 (Striped+Parity), 6 (Double Parity), 10 (Striped+Mirror), ZFS (RAID-Z). For 6 drives/DB: Choose RAID 10. It offers the best performance and fault tolerance (can lose 2 drives if mirrored pairs) compared to RAID 6's slower write performance.

OP7 — Internal: AD-integrated DNS (Active Directory). External: Public DNS with DNSSEC. Split-horizon: Internal clients get internal IPs. Forwarders: Internal DNS forwards external queries. Laptop: DHCP updates DNS records; internal resolver resolves internal names, external resolves public.

OP8 — PKI Chain: Root CA (offline) -> Subordinate CA (online) -> End-entity. CRL/OCSP check revocation. Compromised Subordinate: Revoke certs, issue new. Compromised Root: Trust chain broken, must rebuild entire PKI. Root must be kept offline.

OP9 — SD-WAN is recommended. MPLS is expensive. IPsec is complex to manage. WireGuard is good but lacks enterprise management. SD-WAN offers cost savings, reliability (bonding), low latency (QoS for VoIP), and centralized management.

OP10 — iDRAC/iLO/IPMI are out-of-band management interfaces. Critical for remote server recovery when OS is down. Exposed IPMI to the internet allows attackers to gain root access to the hardware (power cycle, console access).

OP11 — LVM: Volume management. LVM-Thin: Overprovisioning. ZFS: Filesystem + Volume + Integrity. Snapshots: LVM (copy-on-write, slower), ZFS (instant, cheap, immutable). Use ZFS for data integrity/snapshots; LVM for simple volume management.

OP12 — Stretch Cluster: 2 sites + 1 witness (tiebreaker). Ceph CRUSH rules must distribute replicas across sites. Network latency < 5ms required. 2-site without tiebreaker risks split-brain (both sites think they are primary). Witness ensures quorum.

OP13 — DHCP DORA: Discover, Offer, Request, Acknowledge. If server unavailable and lease expires: Client assigns itself an APIPA address (169.254.x.x) and cannot communicate on the network until a server is found.

OP14 — Managed switch: Configurable (VLANs, QoS, ACLs, SNMP). Unmanaged: Plug-and-play. Managed is critical for enterprise to segment traffic (VLANs), prioritize VoIP (QoS), and monitor health (SNMP).

OP15 — NTP synchronizes time. Critical for logs, certificate validation, and cluster consensus. Clock drift causes authentication failures (Kerberos), split-brain in clusters, and log correlation issues.

OP16 — Physical: Hardware only. VM: OS virtualized on hardware. Container: Process virtualized on OS. Run on bare metal for high I/O, low latency, or hardware passthrough (e.g., GPU, high-performance DB).

OP17 — UPS: Short-term battery power (minutes). Generator: Long-term power (hours/days). ATS: Automatically switches power source from grid to generator when grid fails. Both needed to ensure uptime during outages.

OP18 — Active-Passive: One node runs, one waits. Active-Active: Both run. PostgreSQL HA: Pacemaker/Corosync manages virtual IP. Fencing (STONITH) ensures failed node is powered off to prevent split-brain. Failover: VIP moves to survivor.

OP19 — 3-2-1 Rule: 3 copies, 2 media, 1 offsite. RPO/RTO define recovery goals. Immutable backups prevent ransomware encryption. Test restores regularly. Backup = Data copy; Disaster Recovery = Process to restore operations.

OP20 — 802.1X: Supplicant (Client), Authenticator (Switch), Server (RADIUS). EAP-TLS (Cert) is secure; PEAP (User/Pass) is easier. Dynamic VLAN assigns VLAN based on identity. Failed auth: Port blocked. MAB: Switch sends MAC address to RADIUS for auth if 802.1X fails.
