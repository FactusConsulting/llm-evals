CL1 — **IaaS, PaaS, SaaS**
*   **IaaS (Infrastructure as a Service):** Provides virtualized computing resources over the internet. You manage OS, middleware, runtime, data, and apps. *Example: AWS EC2, Azure VMs.*
*   **PaaS (Platform as a Service):** Provides hardware and software tools (usually for app development) over the internet. You manage data and apps; provider manages OS and runtime. *Example: AWS Elastic Beanstalk, Azure App Service.*
*   **SaaS (Software as a Service):** Delivers software applications over the internet. Provider manages everything; user manages only data and access. *Example: Microsoft 365, Salesforce.*

CL2 — **Shared Responsibility Model**
*   **Concept:** Security is a joint effort. Provider secures the cloud (infrastructure); Customer secures what is in the cloud (data, config).
*   **IaaS VM:** Provider secures physical host, hypervisor, network. Customer secures OS, patching, firewall config, data.
*   **Managed Kubernetes:** Provider secures control plane, nodes, etcd. Customer secures workloads, network policies, RBAC, secrets.
*   **SaaS (M365):** Provider secures app, infrastructure, data at rest/transit. Customer secures identity (MFA), access permissions, data classification.

CL3 — **Networking Components (VPC, Subnet, SG, NACL)**
*   **VPC:** Isolated virtual network in the cloud.
*   **Subnet:** Range of IP addresses within a VPC (public/private).
*   **Security Group (SG):** Stateful firewall at the instance level (allow/deny traffic).
*   **NACL:** Stateless firewall at the subnet level (allow/deny traffic).
*   **Layering:** VPC contains Subnets; Subnets contain Instances; SG protects Instances; NACL protects Subnets.
*   **Stateful vs Stateless:** SG is stateful (return traffic allowed automatically). NACL is stateless (must explicitly allow return traffic).

CL4 — **Multi-Account Strategy**
*   **Landing Zone:** Secure, compliant baseline environment (networking, identity, logging) for new accounts.
*   **Organizational Units (OUs):** Logical grouping of accounts (e.g., Prod, Dev, Security) for policy application.
*   **Guardrails:** SCPs (Service Control Policies) to prevent risky actions (e.g., disabling CloudTrail).
*   **Shared Networking:** Transit Gateway (AWS) or Hub-and-Spoke (Azure) connects VPCs/VNets centrally for traffic inspection.
*   **Centralized Logging:** Aggregates logs to a dedicated Security/Log account for audit and monitoring.
*   **Single-Account Problems:** High blast radius (compromise affects all), billing confusion, compliance violations (separation of duties), resource contention.

CL5 — **Infrastructure as Code (IaC)**
*   **Definition:** Managing infrastructure via machine-readable definition files rather than physical hardware configuration.
*   **Terraform/OpenTofu:** Declarative, multi-cloud, state-based. *Choose:* Multi-cloud, complex dependencies, community modules.
*   **Pulumi:** Imperative/Declarative using general languages (TS, Python). *Choose:* Complex logic, teams comfortable with coding.
*   **CloudFormation/Bicep:** Native to AWS/Azure. *Choose:* Single-cloud, tight integration with native features, CI/CD pipelines.
*   **Crossplane:** Kubernetes-native IaC. *Choose:* Teams already using K8s, managing cloud resources via K8s API.

CL6 — **Autoscaling Patterns**
*   **HPA:** Scales pods based on CPU/Memory metrics.
*   **VPA:** Adjusts resource requests/limits per pod (requires restart).
*   **Cluster Autoscaler:** Adds/removes nodes based on pending pods.
*   **KEDA:** Event-driven scaling (e.g., queue length, Kafka lag).
*   **KEDA vs HPA:** Use KEDA when scaling needs to be based on business events (e.g., message queue depth) rather than resource utilization.

CL7 — **Disaster Recovery (DR) Strategy**
*   **RPO/RTO:** RPO (data loss tolerance) dictates replication sync (sync for low RPO). RTO (downtime tolerance) dictates failover speed.
*   **Active-Active:** Traffic split across regions. High cost, low RTO/RPO.
*   **Active-Passive:** One region active, one standby. Lower cost, higher RTO.
*   **Database:** Sync (synchronous replication) for zero data loss (Active-Active); Async for Active-Passive.
*   **DNS Failover:** Route53 Health Checks trigger failover to secondary region.
*   **State:** Externalize state (S3, DynamoDB) to avoid local state loss.
*   **DR Models:** Pilot Light (min infra running), Warm Standby (scaled down prod), Multi-site (full active-active).

CL8 — **GitOps Model**
*   **Concept:** Git repository is the single source of truth for infrastructure and app state. Controllers sync cluster state to Git state.
*   **ArgoCD vs Flux:** ArgoCD (Push/Pull, UI focused), Flux (Pull only, K8s native).
*   **Secrets:** Sealed Secrets (encrypted in Git), External Secrets Operator (fetch from Vault/AWS Secrets Manager).
*   **Drift:** If manual change occurs, GitOps controller detects drift and automatically reverts changes to match Git.

CL9 — **Stateful Legacy App to Kubernetes**
*   **StatefulSet:** Ensures stable network IDs and ordered deployment/termination.
*   **PV/PVC:** Use `local` storage class for low latency (node-local disk).
*   **Headless Service:** Provides stable DNS names for StatefulSet pods.
*   **Init Containers:** Handle config file generation or permission setup before main app starts.
*   **ConfigMaps:** Mount as volumes for specific file paths.
*   **PDB:** Ensure minimum available pods during disruptions to maintain service.

CL10 — **Identity (SA, Workload Identity, IRSA)**
*   **Service Account:** K8s identity for pods.
*   **Workload Identity:** GCP/Azure mechanism to bind K8s SA to cloud IAM roles.
*   **IRSA:** AWS mechanism using OIDC to map K8s SA to IAM Role.
*   **Mounting Creds:** Bad practice (static keys, risk of leakage, hard to rotate).
*   **Federation:** Uses short-lived tokens via OIDC, no long-term secrets stored in pods.

CL11 — **Storage Types**
*   **Object (S3/Blob):** Unstructured data, HTTP API. High durability (11 9s), high availability. *Use:* Backups, media, logs.
*   **Block (EBS/Managed Disk):** Raw storage, attached to VM. High IOPS, low latency. *Use:* OS, DBs.
*   **File (EFS/Azure Files):** Shared file system (NFS/SMB). *Use:* Shared configs, legacy app data.
*   **Durability:** All offer high durability (11 9s+), but availability differs (Object is higher than Block).

CL12 — **Zero-Trust Network Architecture (K8s)**
*   **Network Policies:** Restrict pod-to-pod traffic (default deny).
*   **Service Mesh (mTLS):** Encrypts service-to-service traffic, authenticates identities.
*   **Pod Identity:** Workload identity federation (no static keys).
*   **Admission Controllers (OPA/Kyverno):** Enforce policies on deployment (e.g., no root user).
*   **Image Signing:** Verify image integrity before deployment.
*   **Runtime Security (Falco):** Detects anomalous behavior (e.g., shell spawn).
*   **Diff:** Traditional = Perimeter defense. Zero Trust = Assume breach, verify every request.

CL13 — **CDN (Content Delivery Network)**
*   **Function:** Distributed edge servers cache content closer to users.
*   **Services:** AWS CloudFront, Azure CDN.
*   **Use CDN:** Static assets (images, JS), reduce origin load, improve latency, DDoS protection.
*   **Use Origin:** Dynamic content, real-time data, sensitive data not meant for caching.

CL14 — **Regions and Availability Zones**
*   **Region:** Geographic area (e.g., us-east-1).
*   **AZ:** Isolated data centers within a region (power, network independent).
*   **Multi-AZ:** Deploy across AZs for High Availability (HA) against data center failure.
*   **Zonal:** Resource lives in one AZ (e.g., EBS). **Regional:** Resource spans AZs (e.g., ELB, RDS Multi-AZ).

CL15 — **Managed Database Service**
*   **Definition:** DBaaS (e.g., RDS, Cloud SQL).
*   **Provider Tasks:** Patching, backups, failover, hardware provisioning, scaling.
*   **User Tasks:** Schema design, queries, data security, access control.
*   **Benefit:** Reduces operational overhead, improves availability.

CL16 — **Cloud Identity (IAM)**
*   **Users:** Human/Service identity. **Groups:** Collection of users. **Roles:** Temporary credentials for entities. **Policies:** JSON permissions.
*   **Least Privilege:** Grant only necessary permissions.
*   **Application:** Use Groups for permissions, Roles for cross-account/EC2 access, avoid long-term user keys.

CL17 — **Serverless Computing**
*   **Definition:** Event-driven code execution without managing servers.
*   **Comparison:** Lambda (AWS), Functions (Azure), Cloud Functions (GCP).
*   **Cold Starts:** Delay when initializing a new instance. Matters for latency-sensitive apps.
*   **Poor Fit:** Long-running processes, stateful apps, predictable high-throughput (reserved instances cheaper).

CL18 — **Kubernetes Ingress Options**
*   **Ingress Resource:** L7 routing rules (standard).
*   **Gateway API:** Enhanced, standardized, supports multiple gateways.
*   **Cloud LB:** L4 load balancing (Service type LoadBalancer).
*   **Mesh Ingress:** L7 termination at service mesh edge.
*   **TLS:** LB (offload), Ingress (terminate at controller), Mesh (mTLS between services).
*   **Gateway API:** Use for complex routing, multi-cluster, or when Ingress limitations are hit.

CL19 — **Cost Optimization Strategy ($80k/mo)**
*   **Commitments:** Reserved Instances/Savings Plans (up to 72% off) for steady state.
*   **Spot/Preemptible:** For fault-tolerant, interruptible workloads (up to 90% off).
*   **Right-sizing:** Match instance size to actual usage (monitor metrics).
*   **Storage Tiering:** Move old data to Glacier/Archive.
*   **Idle Detection:** Find unattached volumes, unused IPs.
*   **FinOps:** Culture of cost accountability. Showback/Chargeback to teams.
*   **Tools:** AWS Cost Explorer, CloudHealth, Kubecost.

CL20 — **Multi-Cloud Strategy**
*   **Abstraction:** Kubernetes (runtime), Terraform (IaC).
*   **Networking:** Direct Connect/ExpressRoute or Interconnect for low latency.
*   **Identity:** Federation (SAML/OIDC) across clouds.
*   **Trade-offs:** Increased complexity, higher networking costs, skill fragmentation.
*   **Justified:** Regulatory requirements, avoiding single-vendor outage, best-of-breed services.
*   **Unnecessary:** If single cloud meets all needs (multi-cloud often adds cost without value).

OT1 — **Purdue Model (ISA-95)**
*   **Level 0:** Physical Process (Sensors, Actuators).
*   **Level 1:** Basic Control (PLCs, Relays).
*   **Level 2:** Control Systems (SCADA, HMIs).
*   **Level 3:** Operations (MES, Historians).
*   **Level 4:** Enterprise (ERP, Business Planning).

OT2 — **IT vs OT Networks**
*   **IT:** Confidentiality/Integrity focus, frequent patching, standard protocols.
*   **OT:** Safety/Availability focus, legacy systems, real-time requirements.
*   **Patching:** OT patching risks downtime or safety hazards; often requires vendor approval and maintenance windows.

OT3 — **DMZ (IDMZ)**
*   **Role:** Buffer zone between IT (L4) and OT (L0-L3).
*   **Traffic:** One-way data flow (OT to IT) preferred. If bidirectional, strictly controlled (e.g., OPC UA).
*   **Danger:** Direct L4 to L1 allows attackers to bypass security controls and directly manipulate physical processes.

OT4 — **OT Protocols**
*   **Modbus TCP:** L2/L3, Master/Slave, simple.
*   **OPC UA:** L7, secure, platform-independent, modern.
*   **EtherNet/IP:** L2, CIP protocol, Allen-Bradley dominant.
*   **PROFINET:** L2, real-time Ethernet, Siemens dominant.
*   **MQTT:** L7, lightweight, publish/subscribe, cloud integration.

OT5 — **Data Diode vs Firewall**
*   **Data Diode:** Hardware enforcing one-way data flow.
*   **Firewall:** Two-way traffic filtering.
*   **Use Diode:** When data must leave OT but nothing can enter (e.g., exporting telemetry).
*   **Limitation:** Cannot send commands back to OT; no feedback loop.

OT6 — **Data Flow to Cloud**
*   **Flow:** PLC -> Historian (L3) -> Edge Gateway (Protocol conversion) -> Cloud Ingestion (MQTT/HTTPS).
*   **Security:** Edge Gateway acts as DMZ. Encrypt traffic (TLS). Authenticate gateway to cloud. Isolate OT network from public internet.

OT7 — **IEC 62443 Zones and Conduits**
*   **Zone:** Logical grouping of assets with similar security requirements.
*   **Conduit:** Controlled path for traffic between zones.
*   **Model:**
    *   Zone 4: Corporate Network.
    *   Zone 3: Engineering/SCADA Servers (DMZ).
    *   Zone 2: SCADA/Control Network.
    *   Zone 1: PLCs/Field Devices.
    *   Zone 0: Physical Process.
    *   Firewalls between zones.

OT8 — **Network Segmentation Strategy**
*   **VLANs:** Separate production lines, engineering, and management.
*   **Firewalls:** Place between VLANs (e.g., Line 1 to Line 2).
*   **Switches:** Managed industrial switches with ACLs.
*   **Remote Access:** Jump host in DMZ, MFA, session recording.
*   **Transition:** Phased rollout during maintenance windows. Parallel run old/new. Test thoroughly before cutover.

OT9 — **SIS vs BPCS**
*   **BPCS:** Basic Process Control System (PLC/DCS) for normal operation.
*   **SIS:** Safety Instrumented System (independent) for emergency shutdown.
*   **IEC 61511:** Standard for safety instrumented systems.
*   **Isolation:** SIS must be air-gapped/isolated to prevent cyber attacks from disabling safety functions.
*   **Ransomware on SIS:** Could prevent safe shutdown, leading to physical damage or injury.

OT10 — **PLC, DCS, SCADA**
*   **PLC:** Logic controller for specific machines.
*   **DCS:** Distributed Control System for continuous process plants.
*   **SCADA:** Supervisory Control and Data Acquisition (monitoring/visualization).
*   **Coexist:** Yes. PLCs control machines, DCS controls process, SCADA monitors both.

OT11 — **Remote Access Architecture**
*   **Architecture:** Jump Host (bastion) in DMZ, MFA required, Session Recording enabled.
*   **Vendor Access:** Time-limited, audited, no direct network access.
*   **Direct RDP:** Unacceptable because it exposes HMI to corporate network threats, bypassing OT security controls.

OT12 — **Nation-State Incident Response**
*   **Detection:** Network IDS (Zeek/Suricata), Anomaly detection on traffic patterns.
*   **Inventory:** Identify compromised assets immediately.
*   **Response:** Isolate network segments (VLAN quarantine). Do not shut down production if safety risk.
*   **Coordination:** Control room must know status to maintain physical safety.
*   **Operator Role:** Maintain safe plant state, follow emergency procedures, do not operate compromised systems.

OT13 — **HMI (Human-Machine Interface)**
*   **Role:** Operator interface for monitoring and controlling processes.
*   **Vs Engineering Workstation:** HMI is for operations (run mode). Engineering Workstation is for configuration/programming (engineering mode).

OT14 — **Historian**
*   **Role:** Time-series database for process data (temperature, pressure).
*   **Importance:** Trend analysis, compliance, troubleshooting.
*   **Plant (L3):** Real-time ops data. **Enterprise (L4):** Aggregated data for business analytics.

OT15 — **Ladder Logic**
*   **Definition:** Graphical programming language mimicking relay logic.
*   **Device:** PLC.
*   **Dominance:** Safety certified, deterministic, widely understood by electricians/engineers, legacy stability.

OT16 — **RTU vs PLC**
*   **RTU:** Remote Terminal Unit. Designed for harsh environments, long-distance comms, SCADA focus.
*   **PLC:** Local control, high-speed logic.
*   **RTU Use:** Pipelines, water treatment plants, remote sites.

OT17 — **Analog vs Digital Signals**
*   **Analog:** Continuous voltage/current (4-20mA). **Digital:** Discrete on/off.
*   **4-20mA:** Preferred over 0-20mA because 4mA indicates "live" wire; 0mA indicates break/fault.

OT18 — **OT Asset Inventory**
*   **Foundation:** You cannot secure what you don't know exists.
*   **Tools:** Passive monitoring (packet capture), Active scanning (ping/scan).
*   **Active Risk:** Scanning can crash legacy devices or disrupt control loops. Use passive first.

OT19 — **MITRE ATT&CK for ICS**
*   **Tactics:** Initial Access, Execution, Persistence, Impact, Defense Evasion.
*   **Techniques:** Spearphishing (Initial Access), Malware (Execution), Scheduled Task (Persistence), Data Destruction (Impact), Obfuscation (Defense Evasion).
*   **Difference:** Focuses on physical impact (safety, environment) vs enterprise data theft.
*   **Usage:** Map defenses to tactics to find gaps (e.g., no detection for lateral movement).

OT20 — **Converged IT/OT SOC**
*   **Log Collection:** Passive taps (SPAN ports) to avoid impacting OT.
*   **SIEM:** Centralized correlation.
*   **Rules:** OT-specific (e.g., PLC write attempts).
*   **Staffing:** OT engineers needed to validate alerts (false positives common).
*   **Firmware Change Alert:** Isolate PLC, verify change request, rollback if unauthorized, investigate source.
