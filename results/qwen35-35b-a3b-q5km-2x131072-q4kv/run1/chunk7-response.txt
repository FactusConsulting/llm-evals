AA1 — **Monolith vs. Microservices:** A monolith is a single unified codebase deployed as one unit. Microservices decompose this into independent, loosely coupled services deployed separately.
**Microservices Problems:**
1.  **Distributed Complexity:** Debugging requires tracing across network calls rather than a single stack trace.
2.  **Data Consistency:** Maintaining ACID transactions across services requires complex patterns (e.g., Sagas) rather than local DB transactions.
3.  **Operational Overhead:** Requires orchestration (Kubernetes), service discovery, and monitoring for each service instance.

AA2 — **Sync vs. Async:** Synchronous communication waits for an immediate response (request-response). Asynchronous communication sends a message and continues processing without waiting.
**Examples:** Synchronous: REST/HTTP, gRPC. Asynchronous: RabbitMQ, Apache Kafka.

AA3 — **API Gateway vs. Reverse Proxy:** A reverse proxy routes traffic to backend servers (L4/L7). An API Gateway adds business logic (auth, rate limiting, transformation) on top of routing.
**Replacement:** An API Gateway cannot *entirely* replace a reverse proxy in all cases. You often need a Layer 4 reverse proxy (e.g., HAProxy) in front of the Gateway for SSL termination or DDoS protection before traffic hits the Gateway's application logic.

AA4 — **Saga Pattern:** A sequence of local transactions where each updates data and triggers the next. If one fails, compensating transactions undo previous changes.
**Choreography:** Services listen for events and react (decentralized). Good for simple flows.
**Orchestration:** A central coordinator tells services what to do (centralized). Good for complex flows requiring visibility and control.

AA5 — **CQRS:** Separates read (Query) and write (Command) models. Often paired with Event Sourcing because the write model stores events, which are used to project the read model.
**Problem Solved:** Allows independent scaling of read and write workloads and provides a complete audit trail of state changes, which traditional CRUD (overwriting state) does not.

AA6 — **Circuit Breaker:** Prevents repeated calls to a failing service.
**States:** Closed (normal), Open (failures detected, requests fail fast), Half-Open (testing if service recovered).
**Retry vs. Breaker:** Retry adds load to a failing service. Breaker stops load. Use together: Retry for transient network glitches, Breaker for persistent service outages.

AA7 — **Tenant Isolation:**
1.  **Shared DB/Schema:** Lowest cost, highest noisy-neighbor risk, complex security (row-level).
2.  **Shared DB/Schema:** Medium cost, better isolation, easier backups per tenant, schema migration complexity.
3.  **Separate DB:** Highest cost, best security/isolation, easiest backups, complex provisioning.
**Trade-off:** Choose based on compliance needs and budget. High-security tenants need DB 3; cost-sensitive SaaS often uses DB 1.

AA8 — **Consistency:** Strong consistency ensures all nodes see the same data immediately. Eventual consistency allows temporary divergence.
**Inventory Architecture:** Requires **Strong Consistency**.
**Solution:** Use a relational database with row-level locking (e.g., PostgreSQL `SELECT ... FOR UPDATE`). Implement optimistic concurrency control (versioning) to prevent race conditions. At scale, use a distributed lock manager (e.g., Redis Redlock) or a dedicated inventory service with strict transactional boundaries.

AA9 — **Sidecar Proxy:** A container deployed alongside the main app to handle network traffic.
**Envoy/Istio:** Envoy handles the data plane (traffic routing, mTLS). Istio Control Plane configures Envoy.
**mTLS:** Application TLS secures data in transit. mTLS between sidecars authenticates the *service identity* (not just the host) and encrypts traffic automatically without app code changes.

AA10 — **Strangler Fig:** Incrementally replace legacy functionality with new services.
**Strategy:** Deploy an API Gateway in front of the legacy .NET monolith. Route specific endpoints (e.g., `/api/orders`) to new .NET 8 microservices. Gradually move more routes to the new services until the monolith is empty and can be decommissioned.

AA11 — **12-Factor App:**
1.  **Codebase:** One repo, many deploys.
2.  **Dependencies:** Explicitly declared.
3.  **Config:** Stored in environment.
4.  **Backing Services:** Treated as attached resources.
5.  **Build/Run/Execute:** Strict separation.
6.  **Processes:** Stateless, state stored in backing services.
7.  **Port Binding:** Expose via port.
8.  **Concurrency:** Scale via process model.
9.  **Disposability:** Fast startup/shutdown.
10. **Dev/Prod Parity:** Minimize differences.
11. **Logs:** Streamed as event streams.
12. **Admin Processes:** Run as one-off commands.
**Why:** Ensures portability, scalability, and maintainability in cloud environments.

AA12 — **Event-Driven (Kafka) vs. Request-Driven (REST):**
**Coupling:** Event-driven is loosely coupled (producer doesn't know consumer). REST is tightly coupled.
**Scalability:** Kafka handles high throughput better. REST is easier to scale horizontally with stateless services.
**Debugging:** REST is easier (request/response). Kafka requires replaying logs.
**Ordering:** Kafka guarantees ordering per partition. REST has no inherent ordering.
**Failure:** Kafka allows retry/replay. REST requires immediate handling.
**Choice:** Use REST for real-time user interaction. Use Kafka for high-volume data processing, decoupling, and audit trails.

AA13 — **Load Balancer:** Distributes traffic across servers.
**L4 vs. L7:** L4 (Transport) routes based on IP/Port. L7 (Application) routes based on HTTP headers/URL.
**Use Case:** L7 is required for SSL termination or routing `/api` to Service A and `/static` to Service B on the same IP.

AA14 — **Message Queue:** Stores messages for delivery.
**MQ (RabbitMQ):** Task-oriented, acknowledges delivery, deletes after consumption. Good for work queues.
**Log Broker (Kafka):** Retains messages, allows replay, high throughput. Good for event streaming.
**Pick:** RabbitMQ for task execution. Kafka for event sourcing or analytics.

AA15 — **Container vs. VM:** Containers share the host OS kernel; VMs virtualize hardware.
**Popularity:** Containers are lightweight, start instantly, and have lower overhead, making them ideal for microservices scaling.

AA16 — **Stateless vs. Stateful:** Stateless services store no session data (e.g., API server). Stateful services store data locally (e.g., Redis cache).
**Scaling:** Stateless is easier to scale horizontally because any instance can handle any request. Stateful requires data replication or sharding.

AA17 — **ORM:** Maps DB tables to objects.
**Examples:** Hibernate (Java), Entity Framework (.NET).
**Trade-offs:** ORM speeds up development and reduces boilerplate but can generate inefficient SQL and hide performance issues compared to raw SQL.

AA18 — **Bulkhead:** Isolates resources (threads, connections) for specific services.
**Implementation:** Use separate thread pools for each downstream dependency. If one service hangs, it only exhausts its pool, not the whole application.

AA19 — **OT vs. CRDT:** OT requires a central server to resolve conflicts sequentially. CRDTs allow local edits that mathematically converge without coordination.
**Choice:** **CRDTs.** For offline editing, OT fails because it needs server coordination. CRDTs allow conflict-free merging when connectivity returns.

AA20 — **Distributed Tracing:**
**Context Propagation:** Inject TraceID/SpanID into headers (W3C/B3).
**Sampling:** Head-based (decide at start) vs. Tail-based (decide after latency).
**Storage:** Jaeger/Tempo store spans for querying.
**Kafka:** Trace context must be injected into Kafka message headers. Consumers must extract and continue the trace chain to ensure continuity.

OP1 — **Hypervisor Type 1:** Bare-metal (e.g., VMware ESXi, Microsoft Hyper-V Server). Direct hardware access.
**Hypervisor Type 2:** Hosted (e.g., VirtualBox, VMware Workstation). Runs on top of an OS.

OP2 — **SAN vs. NAS:** SAN provides block storage (raw disks) via iSCSI/Fibre Channel. NAS provides file storage via NFS/SMB.
**Use:** SAN for databases (low latency, block access). NAS for file shares (easy sharing).

OP3 — **3-Tier:** Core (backbone), Distribution (policy), Access (edge).
**Leaf-Spine:** Preferred for modern DCs. Solves oversubscription and latency issues of 3-tier by providing non-blocking, equal-cost paths between any two nodes.

OP4 — **Proxmox HA:**
**Architecture:** 3+ nodes for quorum.
**Quorum:** Majority vote required to avoid split-brain.
**Fencing:** STONITH (e.g., iDRAC) to power off failed nodes.
**Storage:** Ceph (distributed) or iSCSI/NFS (shared).
**Failure:** VMs on failed node are restarted on surviving nodes.

OP5 — **VLAN Trunking:** Carries multiple VLANs over one link using 802.1Q tags.
**Linux Config:** Create sub-interfaces (e.g., `eth0.10`, `eth0.20`) with `ip link add link eth0 name eth0.10 type vlan id 10`. Assign IPs to sub-interfaces.

OP6 — **RAID:**
**RAID 1:** Mirror (50% capacity).
**RAID 5:** Striping + Parity (N-1 capacity).
**RAID 6:** Double Parity (N-2 capacity).
**RAID 10:** Mirror + Stripe (50% capacity, high perf).
**RAID Z2:** ZFS double parity.
**Choice:** **RAID 10** (or Z2) for 6 drives. Best balance of read/write performance and fault tolerance for a database.

OP7 — **DNS Arch:**
**Internal:** AD DNS servers (Active Directory integrated).
**External:** Public authoritative servers (e.g., Route53).
**Split-Horizon:** Internal clients get internal IPs; external get public IPs.
**Forwarders:** Internal DNS forwards unknown queries to public resolvers.
**VPN:** Laptop uses split-tunneling or DNS forwarding to resolve internal names while on VPN.

OP8 — **PKI Chain:** Root CA (offline) -> Subordinate CA (online) -> End Entity.
**CRL/OCSP:** Check revocation status.
**Compromise:** Root CA compromised = revoke all certs, rebuild PKI. Subordinate compromised = revoke that CA, issue new certs.

OP9 — **WAN Consolidation:**
**MPLS:** High cost, high reliability.
**IPsec:** Low cost, variable reliability.
**SD-WAN:** Low cost, intelligent routing, VoIP optimization.
**WireGuard:** Simple, encrypted mesh.
**Recommendation:** **SD-WAN.** Best for cost, management, and VoIP QoS across 5 sites.

OP10 — **iDRAC/iLO/IPMI:** Out-of-band management interfaces.
**Critical:** Allows server control (power, console) even if OS is down.
**Exposed:** If exposed to internet, attackers can gain root access to the hardware (power cycle, boot from ISO).

OP11 — **LVM vs. ZFS:**
**LVM:** Logical volume manager. Thin provisioning possible. Snapshots are copy-on-write.
**ZFS:** Filesystem + Volume manager. Copy-on-write, checksums, snapshots (instant).
**Use:** LVM for simple Linux volumes. ZFS for data integrity and snapshots.

OP12 — **Stretch Cluster:**
**Ceph CRUSH:** Distributes data across sites.
**Latency:** Must be low (<5ms) for monitors.
**Split-Brain:** If network fails, both sites think they are primary.
**Tiebreaker:** A 3rd site or cloud witness is needed to break quorum. 2-site Ceph is bad without a tiebreaker because both sites may write data, causing corruption.

OP13 — **DHCP DORA:** Discover, Offer, Request, Acknowledge.
**Unavailable:** Client uses APIPA (169.254.x.x) or retries lease. If lease expires, client broadcasts Discover again.

OP14 — **Managed vs. Unmanaged:**
**Managed:** Configurable (VLANs, QoS, ACLs, Port Mirroring).
**Unmanaged:** Plug-and-play.
**Importance:** Managed switches allow segmentation and traffic prioritization (QoS) for VoIP/Video.

OP15 — **NTP:** Network Time Protocol.
**Critical:** Certificates, logs, and distributed transactions rely on synchronized time.
**Drift:** Cluster failover may fail; logs become unreadable; security tokens may validate incorrectly.

OP16 — **Physical vs. VM vs. Container:**
**Physical:** Hardware only.
**VM:** OS virtualization.
**Container:** Process virtualization.
**Bare Metal:** Use for high-performance databases or hardware passthrough needs where virtualization overhead is unacceptable.

OP17 — **UPS vs. Generator:**
**UPS:** Short-term battery power (minutes).
**Generator:** Long-term power (hours/days).
**ATS:** Automatically switches load from grid to generator when power fails.
**Need Both:** UPS bridges the gap until generator starts.

OP18 — **Active-Passive vs. Active-Active:**
**Active-Passive:** One node handles traffic; standby takes over on failure.
**Active-Active:** Both handle traffic.
**PostgreSQL HA:** Pacemaker/Corosync cluster. Use STONITH (fencing) to kill failed node. Use Virtual IP (VIP) for failover.

OP19 — **Backup Strategy:**
**3-2-1:** 3 copies, 2 media types, 1 offsite.
**RPO/RTO:** Define acceptable data loss and downtime.
**Immutable:** Prevent ransomware deletion.
**Test:** Regularly restore random files and full systems.
**Backup vs. DR:** Backup is data copy; DR is the plan/process to restore operations after a disaster.

OP20 — **802.1X:**
**Roles:** Supplicant (client), Authenticator (switch), Server (RADIUS).
**EAP:** EAP-TLS (certs, secure) vs PEAP (username/password).
**Dynamic VLAN:** Assign VLAN based on user role.
**Failure:** Port goes to "Guest" VLAN or blocks access.
**MAB:** Switch sends MAC address to RADIUS as credential for devices without 802.1X supplicant.
