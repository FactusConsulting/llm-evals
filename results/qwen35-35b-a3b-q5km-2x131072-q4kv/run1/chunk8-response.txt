CL1 — **IaaS (Infrastructure as a Service):** Provides virtualized computing resources (servers, storage, networking). *Example:* AWS EC2. **PaaS (Platform as a Service):** Provides hardware and software tools (runtime, databases) for developers. *Example:* Google App Engine. **SaaS (Software as a Service):** Delivers software applications over the internet. *Example:* Microsoft 365.

CL2 — **Shared Responsibility Model:** Provider secures the cloud (physical, hypervisor); Customer secures what is in the cloud. **IaaS VM:** Provider secures physical host/hypervisor; Customer secures OS, patching, app, data. **Managed Kubernetes:** Provider secures control plane/infrastructure; Customer secures node OS, app, network policies, secrets. **SaaS (M365):** Provider secures app/infra; Customer secures identity, data classification, device access.

CL3 — **VPC:** Isolated virtual network. **Subnet:** IP range within VPC (public/private). **Security Group:** Stateful firewall at instance level (allow rules). **NACL:** Stateless firewall at subnet level (allow/deny rules). **Layering:** VPC contains Subnets; Subnets contain Instances protected by SGs; NACLs filter traffic entering/leaving Subnets. **Stateful:** SG remembers return traffic. **Stateless:** NACL requires explicit rules for both directions.

CL4 — **Landing Zone:** Secure baseline environment. **OUs:** Logical grouping for billing/policy. **Guardrails:** Preventive controls (e.g., no public S3 buckets). **Shared Networking:** Central hub for connectivity (Transit Gateway). **Centralized Logging:** Aggregated audit trails. **Single-Account Risk:** High blast radius, billing complexity, compliance violations, lack of isolation between environments (prod/dev).

CL5 — **IaC:** Managing infrastructure via code files. **Terraform/OpenTofu:** Multi-cloud, declarative, state-managed. **Pulumi:** Multi-cloud, imperative (general purpose languages). **CloudFormation/Bicep:** Cloud-native, tightly coupled, AWS/Azure specific. **Crossplane:** K8s API for cloud resources. **Choice:** Terraform for multi-cloud portability; CloudFormation for AWS-native simplicity; Pulumi for dev-friendly languages; Crossplane for K8s-centric workflows.

CL6 — **HPA:** Scales pods based on CPU/Memory metrics. **VPA:** Scales pod resource requests/limits. **Cluster Autoscaler:** Adds/removes nodes based on pod scheduling. **KEDA:** Scales based on external events (Kafka, SQS). **Use KEDA:** When scaling needs to be driven by event queues or custom metrics (e.g., message backlog) rather than just CPU utilization.

CL7 — **Strategy:** Define RPO (data loss tolerance) and RTO (downtime tolerance). **Active-Active:** Both regions serve traffic (low RTO, high cost). **Active-Passive:** One region active, one standby (higher RTO, lower cost). **DB:** Async replication (performance, eventual consistency) vs Sync (consistency, latency). **DNS:** Route53 Health Checks trigger failover. **State:** Shared storage (S3) vs replicated state (Redis). **Patterns:** Pilot Light (DB running, app off), Warm Standby (scaled down app), Multi-site (full active-active).

CL8 — **GitOps:** Git repository is the single source of truth for desired state. **ArgoCD:** Push-based (server watches Git, syncs cluster). **Flux:** Pull-based (cluster pulls Git, syncs). **Secrets:** Sealed Secrets (encrypted in Git), External Secrets Operator (fetch from Vault/AWS Secrets Manager). **Drift:** GitOps controller detects manual changes and automatically reverts them to match Git state.

CL9 — **Architecture:** Use **StatefulSet** for stable network IDs and ordered deployment. **PV/PVC:** Use `local` storage class with `hostPath` for low latency. **Headless Service:** For stable DNS names. **Init Containers:** For config file generation before main app starts. **ConfigMaps:** Mount as volumes for specific paths. **PDB:** Ensure minimum available pods during updates to prevent data loss.

CL10 — **SA:** K8s internal identity for pods. **Workload Identity:** GCP/AWS specific identity federation. **IRSA:** AWS OIDC for K8s SA to assume IAM roles. **Mounting Creds:** Bad practice (leak risk, rotation difficulty). **Federation:** Uses OIDC tokens to assume temporary IAM roles without static keys, reducing credential exposure.

CL11 — **Object (S3):** Unstructured data, high durability (11 9s), low latency. **Block (EBS):** Attached to VM, low latency, high IOPS. **File (EFS):** Shared file system, network latency. **Use Object:** Backups, media, logs. **Use Block:** OS disks, DBs. **Use File:** Shared configs, home dirs. **Durability:** All high (11 9s), **Availability:** Object (99.99%), Block (99.99%), File (99.99%).

CL12 — **Zero Trust:** Assume breach. **Network Policies:** Default deny all traffic. **Service Mesh:** mTLS for service-to-service auth. **Pod Identity:** SPIFFE/SPIRE for workload identity. **Admission:** OPA/Kyverno to enforce policies. **Image Signing:** Cosign to verify artifacts. **Runtime:** Falco for anomaly detection. **Diff:** Traditional relies on perimeter firewall; Zero Trust verifies every request regardless of location.

CL13 — **CDN:** Distributed network of edge servers caching content. **Cloud Services:** AWS CloudFront, Azure CDN. **Use CDN:** Static assets (images, JS), global latency reduction, DDoS protection. **Direct Origin:** Dynamic content, real-time data, or when caching is not beneficial.

CL14 — **Region:** Geographic area with multiple AZs. **AZ:** Isolated data center with independent power/cooling. **Deploy across AZs:** To survive data center failure. **Zonal:** Resource tied to one AZ (e.g., EBS). **Regional:** Resource spans AZs (e.g., ELB, RDS Multi-AZ).

CL15 — **Managed DB:** Service where provider manages hardware, OS, DB software. **Provider Tasks:** Patching, backups, failover, scaling, monitoring. **Your Tasks:** Schema design, queries, access control. **Benefit:** Reduced operational overhead, higher availability.

CL16 — **IAM Users:** Individual identities. **Groups:** Collections of users. **Roles:** Temporary credentials for entities. **Policies:** JSON permissions. **Least Privilege:** Grant only necessary permissions. **Apply:** Use roles for access, avoid root keys, audit permissions regularly.

CL17 — **Serverless:** Event-driven execution without server management. **Functions:** Lambda, Azure Functions, GCP Functions. **Cold Start:** Delay when initializing a new instance. **Poor Fit:** Long-running processes, stateful apps, predictable high-throughput workloads where reserved capacity is cheaper.

CL18 — **Ingress:** K8s resource for HTTP/HTTPS routing. **Gateway API:** Standardized, multi-protocol, better separation of concerns. **LB Annotations:** Cloud-specific load balancer config. **Mesh Gateway:** Istio/Linkerd ingress. **TLS:** Terminates at LB (cloud), Ingress Controller, or Mesh. **Use Gateway API:** For complex routing, multi-cluster, or when standard Ingress is insufficient.

CL19 — **Strategy:** **RI/SP:** Commit to usage for discount. **Spot:** Interruptible for fault-tolerant workloads. **Right-sizing:** Match instance to actual usage. **Storage:** Move old data to Glacier. **FinOps:** Cost ownership culture. **Tools:** AWS Cost Explorer, Kubecost. **Showback:** Tag resources by team/project to allocate costs.

CL20 — **Strategy:** **Abstraction:** Kubernetes, Terraform. **Networking:** Direct Connect/ExpressRoute. **Identity:** OIDC federation. **Trade-offs:** Increased complexity, latency, cost vs. resilience, compliance. **Justified:** Regulatory requirements, avoiding single-vendor risk. **Unnecessary:** If one cloud meets all needs, multi-cloud adds cost without value.

OT1 — **Purdue Model:** ISA-95 standard for IT/OT integration. **Level 0:** Physical Process (sensors). **Level 1:** Basic Control (PLCs, RTUs). **Level 2:** Supervisory Control (SCADA, HMIs). **Level 3:** Operations (MES, Historian). **Level 4:** Enterprise (ERP, Business).

OT2 — **IT:** Data confidentiality, patching, security focus. **OT:** Safety, uptime, availability focus. **Patching:** OT systems often run legacy OS (XP/7) that cannot be patched without breaking functionality or safety certification. Patching risks downtime or physical harm.

OT3 — **DMZ (IDMZ):** Buffer zone between IT and OT. **Traffic:** Historian pulls from SCADA (push), not direct control. **Direction:** IT to OT is restricted; OT to IT is monitored. **Danger:** Direct L4 to L1 bypasses safety controls, allowing IT malware to directly manipulate physical processes.

OT4 — **Modbus TCP:** L1/L2, simple, unencrypted. **OPC UA:** L2/L3, secure, platform independent. **EtherNet/IP:** L1/L2, industrial Ethernet. **PROFINET:** L1/L2, real-time Ethernet. **MQTT:** L3/L4, lightweight pub/sub for cloud.

OT5 — **Data Diode:** Hardware allowing one-way data flow. **Firewall:** Two-way traffic control. **Use Diode:** When data must leave OT but nothing can enter (e.g., telemetry to cloud). **Limitation:** Cannot send commands back to OT, no feedback loop.

OT6 — **Flow:** PLC -> Historian -> OPC UA Server -> Edge Gateway (MQTT) -> Cloud Ingestion. **Security:** Edge Gateway in DMZ. TLS encryption. Firewall rules allow only specific ports (e.g., 8883 for MQTT). No direct internet access from PLCs.

OT7 — **Zones:** Logical grouping of assets with similar security requirements. **Conduits:** Controlled paths for traffic between zones. **Model:** Zone 1: Corporate (L4). Zone 2: Engineering (L3). Zone 3: SCADA (L3). Zone 4: PLC/SIS (L1/L2). **Conduits:** Firewalls between zones, allowing only necessary traffic (e.g., Engineering to SCADA).

OT8 — **Segmentation:** VLANs per production line. **Firewalls:** Between VLANs and DMZ. **Switches:** Managed industrial switches with ACLs. **Remote Access:** Jump host in DMZ, MFA, session recording. **Transition:** Phased rollout, parallel run, backup network, vendor coordination during maintenance windows.

OT9 — **SIS:** Independent safety system (stops process safely). **BPCS:** Basic control (maintains process). **IEC 61511:** Safety lifecycle standard. **Air-gap:** Prevents cyber attacks from disabling safety systems. **Ransomware on SIS:** Safety system fails to stop hazardous process, leading to physical damage or injury.

OT10 — **PLC:** Logic controller for specific machines. **DCS:** Distributed control for continuous processes. **SCADA:** Monitoring and data collection. **Coexist:** Yes, PLCs often feed data to SCADA, DCS manages large scale processes.

OT11 — **Architecture:** Jump host in DMZ. **MFA:** Required for access. **Recording:** Session logging. **Vendor:** Time-limited accounts. **Direct RDP:** Unacceptable because it exposes OT assets to IT network threats, lacks granular control, and bypasses audit trails.

OT12 — **Plan:** Isolate compromised segment immediately. **Monitoring:** IDS for anomalous traffic. **Inventory:** Verify asset status. **IR:** Contain, Eradicate, Recover. **Ops:** Maintain manual control, prioritize safety over data. **Operator:** Monitor physical safety, execute manual overrides if needed, do not trust compromised systems.

OT13 — **HMI:** Interface for operators to view and control processes. **Role:** Visualization and input. **Difference:** HMI is for operation (real-time); Engineering Workstation is for configuration and programming (offline/development).

OT14 — **Historian:** Time-series database for process data. **Importance:** Trend analysis, compliance, troubleshooting. **Plant:** Real-time operational data (Level 3). **Enterprise:** Aggregated data for business analytics (Level 4).

OT15 — **Ladder Logic:** Graphical programming language resembling relay circuits. **Device:** PLC. **Dominant:** Familiarity, deterministic execution, safety certification, ease of maintenance for electricians.

OT16 — **RTU:** Remote Terminal Unit. **Diff:** RTU focuses on comms and harsh environments; PLC focuses on local control logic. **Deploy RTU:** Remote sites (pipelines, water), long-distance comms, low power.

OT17 — **Analog:** Continuous signal (voltage/current). **Digital:** Discrete (on/off). **4-20mA:** Preferred because 4mA represents "live zero" (detects wire break at 0mA vs 4mA).

OT18 — **Inventory:** Knowing all assets is critical for vulnerability management. **Tools:** Passive (packet sniffing), Active (scanning). **Active Risk:** Can crash legacy devices or trigger alarms in safety systems.

OT19 — **MITRE ICS:** Framework for ICS threats. **Tactics:** Initial Access (Phishing), Execution (Command Line), Persistence (Scheduled Task), Impact (Disruption), Defense Evasion (Obfuscation). **Diff:** Focuses on physical impact/safety. **Use:** Map defenses to tactics to find gaps in detection/prevention.

OT20 — **SOC:** Passive log collection (SPAN ports). **SIEM:** Correlate IT/OT logs. **Rules:** OT-specific (e.g., PLC write attempts). **Triage:** OT engineer validates before IT action. **Staffing:** OT engineers for physical safety context. **Firmware Alert:** Isolate PLC, verify safety status, do not reboot without approval, investigate source, restore from known good image.
