SC1-A
The failure stems from the Kubernetes Pod Security Standards (PSS) enforcement change from Baseline to Restricted.
1.  **Restricted Policy:** Prohibits running containers as root (UID 0). It requires `runAsNonRoot: true` AND an explicit `runAsUser` > 0 in the Pod spec, OR a container image that defaults to a non-root user.
2.  **Image Behavior:** The new base image (`cgr.dev/chainguard/static:latest`) defaults `USER` to root.
3.  **Manifest Behavior:** The pods have `securityContext.runAsNonRoot: true` but lack `runAsUser`.
4.  **Conflict:** When the Pod Security Admission controller enforces Restricted, it sees the container image defaults to root. Since `runAsUser` isn't explicitly set to override the image default, the effective UID is 0. This violates the Restricted policy, causing the kubelet to reject the pod or the container runtime to fail immediately (often resulting in CrashLoopBackOff if the container starts and exits due to permission errors, or admission rejection).
5.  **Baseline vs. Restricted:** Baseline allows privileged containers and root users (mostly). Restricted enforces non-root and drops capabilities. The upgrade triggered the stricter policy, exposing the misconfiguration.

SC1-C
**Immediate Remediation:**
1.  **Image Fix:** Rebuild the image with a non-root user (e.g., `USER 1000` in Dockerfile).
2.  **Manifest Fix:** Add `securityContext.runAsUser: 1000` to the Pod/Deployment spec.
3.  **Rollout:** Use a rolling update strategy (`kubectl rollout restart`) to replace pods without downtime.

**Prevention:**
1.  **CI Gate:** Integrate `kube-score` or `conftest` in CI to validate PSS compliance before push.
2.  **Admission Controller:** Enforce PSS Restricted at the namespace level via OPA Gatekeeper or Kyverno policies that require `runAsUser` if `runAsNonRoot` is true.
3.  **Image Scanning:** Use Trivy/Grype to scan for root users in the final image layer.

SC2-A
**Stale Data:** Caused by asynchronous replication lag. When the EU replica is promoted, it becomes the primary. Any WAL (Write-Ahead Log) entries generated on the US primary *after* the last shipped WAL segment but *before* the failover decision are lost.
**Replication:** PostgreSQL streaming replication ships WAL files asynchronously across regions. If the network blips or lag exceeds the timeout, the replica falls behind.
**Data Loss:** The "RPO window" is the time between the last committed transaction on the primary and the last shipped WAL to the replica.
**Quantification:** Check `pg_stat_replication` on the primary (before failover) for `replay_lag`. After failover, check the new primary's `pg_stat_wal_receiver` for `received_lsn` vs `replay_lsn`.

SC2-B
1.  **Import Replica:** `terraform import aws_db_instance.promoted_replica <new_instance_id>`
2.  **Update State:** `terraform state mv aws_db_instance.old_primary aws_db_instance.promoted_replica` (if using same resource name).
3.  **Update Config:** Modify Terraform code to point to the new instance ID/ARN.
4.  **Apply:** `terraform apply` to update dependent resources (Security Groups, Route53) to reference the new ARN.
5.  **Cleanup:** `terraform state rm aws_db_instance.old_primary` (if it still exists in state) and destroy the old instance manually or via `terraform destroy` after confirming state is clean.

SC2-C
**Architecture:**
1.  **Database:** Use **Aurora Global Database**. It offers sub-second replication lag (RPO < 1s) and faster failover (minutes) compared to standard RDS.
2.  **Connection:** Use **RDS Proxy** instead of PgBouncer. It handles connection pooling centrally and supports failover transparently.
3.  **DNS:** Use **Route53 Latency-Based Routing** with health checks. If US endpoint fails, traffic routes to EU.
4.  **App Logic:** Implement exponential backoff retry logic with jitter to handle transient connection failures during failover.
5.  **Recommendation:** Aurora Global DB is recommended over RDS because standard RDS cross-region replication is asynchronous and slower to promote, increasing RPO/RTO risk.

SC3-C
**Hardened Pipeline:**
1.  **Signing:** Use **Cosign** to sign images upon build. `cosign sign --key env://COSIGN_PRIVATE_KEY <image>`.
2.  **Admission:** Deploy **Kyverno** or **OPA Gatekeeper** to verify signatures at deployment time.
3.  **Pinning:** Pin base images by **digest** (e.g., `node@sha256:...`) in Dockerfiles, not tags.
4.  **SBOM:** Generate Software Bill of Materials using **Syft** and upload to a registry (e.g., S3).
5.  **Notary v2:** Integrates with Sigstore. It provides a trust root for images. In this pipeline, Cosign acts as the signer, and Notary v2 (via Sigstore) verifies the trust chain.

SC4-A
1.  **Stateful Storage (NFS):** Kubernetes pods are ephemeral. Local NFS mounts break on pod restart. **Solution:** Use PersistentVolumes (PV) with NFS CSI driver or migrate to object storage (S3).
2.  **Session Affinity:** Sticky sessions rely on F5. Kubernetes uses Service discovery. **Solution:** Move sessions to Redis or use Kubernetes Session Affinity (ClientIP) with caution.
3.  **Database (Oracle):** Oracle licensing and deployment on K8s is complex. **Solution:** Use Oracle Cloud Infrastructure (OCI) RDS or containerized Oracle with StatefulSets.
4.  **IBM MQ:** Legacy integration. **Solution:** Use MQ Bridge or migrate to RabbitMQ/Kafka if possible.
5.  **Bare Metal OS:** Monolith likely relies on host OS features. **Solution:** Containerize the app fully, ensuring no host-specific dependencies.

SC5-C
**Architectural Solution:**
1.  **CQRS:** Separate read/write models. Reconciliation writes to a write model; checkout reads from a read replica.
2.  **Read Replica:** Run the reconciliation job against a read-only replica to avoid locking production tables.
3.  **Bulkheads:** Configure separate connection pools for the CronJob vs. the API service.
4.  **Scheduling:** Move CronJob to off-peak hours (e.g., 03:00 UTC).
5.  **PDBs:** Set `minAvailable` for the inventory-service pods to ensure the CronJob (running as a Job) doesn't evict serving pods during disruptions.

SC6-A
**Identity Layer:**
1.  **Workload Identity:** Deploy **SPIRE** (SPIFFE Runtime Environment) on all clusters (K8s, VMs). SPIRE issues short-lived X.509 certificates.
2.  **Cloud Integration:**
    *   **AWS:** Map SPIFFE IDs to IAM Roles via OIDC (IRSA).
    *   **Azure:** Map SPIFFE IDs to Managed Identities.

3.  **Federation:** Use OIDC federation. A workload in AWS presents its SPIFFE ID to the on-prem service. The on-prem service validates the certificate chain against the shared SPIRE trust root. No shared secrets.

SC6-C
**Connectivity Options:**
1.  **AWS Direct Connect:**
    *   *Pros:* Dedicated bandwidth, lower latency, consistent throughput.
    *   *Cons:* High cost, hardware provisioning.
    *   *Zero-Trust:* Fits well; traffic is encrypted (MACsec) or tunneled.
2.  **Dual VPN with ECMP:**
    *   *Pros:* Lower cost, faster setup.
    *   *Cons:* Shared bandwidth, higher latency variance.
    *   *Zero-Trust:* IPsec encrypts traffic.
3.  **Hybrid Mesh (Tailscale/Netbird):**
    *   *Pros:* Zero-config, identity-based access, no IP management.
    *   *Cons:* Performance overhead, dependency on third-party control plane.
    *   *Zero-Trust:* Native (identity-based).
**Routing:** DX requires BGP. VPN requires static routes. Mesh requires agent routing.
**Recommendation:** Use **Direct Connect** for core data traffic (DB, Storage) to meet bandwidth needs, and **Tailscale** for administrative access and micro-segmentation between workloads to enforce zero-trust at the identity layer.

SC10-A
**Architecture:**
1.  **Active-Passive (Velero):**
    *   *RPO:* Depends on backup frequency (15 mins achievable).
    *   *RTO:* Restore time for 200TB MinIO is high (>1 hour).
    *   *Verdict:* Fails RTO for MinIO.
2.  **Active-Active (Istio/Cilium):**
    *   *RPO:* Near zero with synchronous replication.
    *   *RTO:* Instant failover via DNS.
    *   *Verdict:* Meets RPO/RTO.
**Replication:**
*   **PostgreSQL:** Patroni with synchronous commit (Chicago -> Dallas).
*   **Redis:** Sentinel with cross-region replication (or Redis Enterprise).
*   **RabbitMQ:** Federation or Sharding across clusters.
*   **MinIO:** Site Replication (S3 compatible) configured for async sync.

SC10-C
**Failover Orchestration:**
1.  **Detection:** Prometheus monitors site health (ping, DB latency). If Chicago is down > 30s, trigger alert.
2.  **DNS:** GSLB (Route53) detects health check failure and updates A records to Dallas.
3.  **Database:** Patroni detects primary down, promotes Dallas node (synchronous commit ensures no data loss).
4.  **Cache/Queue:** Redis Sentinel promotes new master. RabbitMQ federation syncs queues.
5.  **Storage:** MinIO site replication catches up (async).
6.  **State:** Ansible/Terraform updates state to mark Dallas as Primary.
**Split-Brain:** Use **etcd** quorum in Patroni. Only one node can win the lock.
**Failback:** Once Chicago is up, demote Dallas (manual or automated via script), sync data, switch DNS back.
