K1 — Easy: A **Deployment** manages stateless applications, allowing for rolling updates, rollbacks, and scaling of identical pods without stable identities. A **StatefulSet** manages stateful applications, providing stable, unique network identities (e.g., `pod-0`, `pod-1`), ordered deployment/termination, and stable persistent storage.

K2 — Easy: A **ClusterIP** Service exposes a stable internal IP and port within the cluster. **kube-proxy** (using iptables or IPVS) maintains network rules on each node to route traffic from the Service IP to the backend Pod IPs based on the Service's selector.

K3 — Medium: A **PersistentVolume (PV)** is a cluster-level storage resource. A **PersistentVolumeClaim (PVC)** is a user request for storage. A **StorageClass** defines the provisioning mechanism (dynamic provisioning). A PVC binds to a PV; if no PV exists, the StorageClass triggers dynamic creation of a PV to satisfy the PVC.

K4 — Medium: 1. Check container logs (`kubectl logs <pod>`) for application errors. 2. Check events (`kubectl describe pod`) for OOMKilled or CrashLoopBackOff reasons. 3. Verify liveness/readiness probe configurations. 4. Check resource limits (CPU/Memory) to ensure the container isn't being throttled or killed.

K5 — Medium: CoreDNS provides DNS resolution for services within the cluster. If CoreDNS pods are down, internal service discovery (e.g., `my-service.default.svc.cluster.local`) fails. External DNS resolution may still work if the pod uses the host network or fallback DNS, but internal Kubernetes networking breaks.

K6 — Hard: The scheduler uses **Filtering** (remove nodes that don't meet requirements like taints, resources) and **Scoring** (rank remaining nodes). Factors considered: 1. Resource requests (CPU/Memory) vs. capacity. 2. Node Affinity/Anti-Affinity rules. 3. Pod Affinity/Anti-Affinity rules.

K7 — Hard: **Yes**, the pod will be scheduled. Node has 4GB total, 3.5GB requested, leaving 0.5GB (512Mi). The pod requests 256Mi, which fits. At runtime, if it uses 600Mi (exceeding the 512Mi limit), the kernel will **OOMKill** the container, causing a CrashLoopBackOff.

K8 — Hard: 1. Request hits External Load Balancer. 2. LB forwards to Ingress Controller (NGINX) Pod. 3. NGINX matches Ingress rules to a Service. 4. Service resolves to **Endpoints** (Pod IPs). 5. **kube-proxy** (iptables/IPVS) on the node routes traffic to the specific Pod IP.

K9 — Medium: A **NetworkPolicy** restricts traffic flow between pods. Default behavior (no policy): All traffic is allowed. When a policy is applied: Traffic is denied by default unless explicitly allowed by the policy's `ingress` or `egress` rules (whitelist model).

K10 — Medium: **RollingUpdate** replaces pods gradually (zero downtime). **Recreate** kills all existing pods before creating new ones. Choose **Recreate** for stateful applications requiring a full shutdown before upgrade (e.g., database migrations) or when zero downtime is not required.

K11 — Easy: A **Namespace** provides logical isolation for resources within a cluster. Default namespaces: `default`, `kube-system`, `kube-public`, and `kube-node-lease`.

K12 — Easy: `kubectl describe` shows detailed status and events. 1. **Events** (e.g., scheduling failures, OOMKilled). 2. **Conditions** (e.g., Ready status details). 3. **Mounts/Environment** (exact config values). `kubectl get` only shows summary status.

K13 — Easy: A **ConfigMap** stores non-confidential configuration data. Pods consume it via: 1. **Environment Variables** (injected into container). 2. **Volume Mounts** (mounted as files in the filesystem).

K14 — Easy: **ClusterIP** (internal cluster access), **NodePort** (exposes port on every node's IP), **LoadBalancer** (provisions external cloud load balancer with public IP).

K15 — Easy: A **DaemonSet** ensures a copy of a pod runs on all (or specific) nodes. Use instead of Deployment for cluster-level agents like logging (Fluentd), monitoring (Prometheus Node Exporter), or CNI plugins.

K16 — Medium: **Taints** repel pods (node property); **Tolerations** allow pods to land on tainted nodes. **Node Affinity** pulls pods to nodes. Use Case: Taint/Toleration for dedicated hardware (e.g., GPU nodes); Affinity for grouping related services.

K17 — Medium: An **Operator** is a custom controller that manages complex stateful applications using CRDs. It differs from a standard controller by encoding human operational knowledge (e.g., backup, restore, scaling) into software. It solves the complexity of managing stateful apps in Kubernetes.

K18 — Hard: etcd uses **Raft consensus** (strong consistency). If quorum is lost (e.g., 2 of 3 nodes lost), the cluster becomes read-only/unavailable. Recovery: Restore etcd from a snapshot on new nodes, rejoin them to the cluster, and ensure quorum is restored.

K19 — Hard: **RBAC** controls access. **Role** (namespace-scoped permissions), **ClusterRole** (cluster-wide), **RoleBinding** (links Role to Subject in namespace), **ClusterRoleBinding** (links ClusterRole to Subject cluster-wide).
*Manifest:*
```yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata: namespace: my-ns
rules:
- apiGroups: [""]
  resources: ["pods", "pods/log"]
  verbs: ["get", "list"]
---
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata: namespace: my-ns
subjects:
- kind: User
  name: developer
roleRef:
  kind: Role
  name: read-pods
  apiGroup: rbac.authorization.k8s.io
```

K20 — Hard: `ndots` defines how many dots trigger an FQDN search. Default `ndots: 5` causes queries for `google.com` to try `google.com`, `google.com.svc`, `google.com.svc.cluster`, etc. Fix: Set `dnsConfig` in pod spec with `ndots: 1` to reduce unnecessary suffix queries.

D1 — Easy: **Rebase** rewrites history to create a linear commit chain. **Merge** creates a new commit preserving history. Avoid rebasing on public/shared branches to avoid disrupting collaborators.

D2 — Easy: **TCP** is connection-oriented and reliable (HTTP, SSH). **UDP** is connectionless and faster (DNS, VoIP).

D3 — Medium: A **race condition** occurs when output depends on the sequence of uncontrollable events. Example: Two threads incrementing a shared counter `x++` without a lock, resulting in lost updates.

D4 — Medium: **Symmetric** uses one key for encryption/decryption. **Asymmetric** uses a public/private key pair. TLS uses Asymmetric for the handshake (key exchange) and Symmetric for the actual data transfer (performance).

D5 — Medium: A **layer** is a read-only filesystem diff in a Docker image. Caching: Place frequently changing instructions (like copying source code) *after* static dependencies (like `npm install`) to maximize cache reuse during builds.

D6 — Medium: 1. Check application logs for stack traces. 2. Verify upstream service health. 3. Check database connectivity. 4. Monitor resource usage (CPU/Memory) for throttling.

D7 — Hard: **CAP** states a distributed system can only guarantee 2 of 3: Consistency, Availability, Partition Tolerance. **CP** (Zookeeper, HBase): Prioritizes data consistency. **AP** (Cassandra, DynamoDB): Prioritizes availability during network partitions.

D8 — Medium: **Horizontal** adds more nodes; **Vertical** adds resources to existing nodes. Patterns enabling Horizontal: Stateless services, Sharding, Load Balancing.

D9 — Easy: An **environment variable** is a dynamic value affecting a process. To persist across sessions on Linux, export it in `~/.bashrc` or `/etc/environment`.

D10 — Easy: **JSON** is strict and machine-readable. **YAML** is human-readable and supports comments. Advantage JSON: Standard for APIs. Advantage YAML: Concise for config files.

D11 — Easy: **REST** is an architectural style for APIs. **GET** (retrieve), **POST** (create), **PUT** (replace/update), **DELETE** (remove).

D12 — Easy: **Version pinning** locks dependencies to specific versions (e.g., `package@1.0.0`). Important for reproducible builds to prevent unexpected breakage from upstream updates.

D13 — Easy: **Compiled** languages translate code to binary before execution (C, Go). **Interpreted** languages execute code line-by-line at runtime (Python, JavaScript).

D14 — Medium: A **reverse proxy** receives client requests and forwards them to backend servers (NGINX, HAProxy). A **forward proxy** acts on behalf of clients to access the internet. Use Case: NGINX for SSL termination; HAProxy for load balancing.

D15 — Medium: A **deadlock** is a situation where processes wait indefinitely for each other. Coffman Conditions: 1. Mutual Exclusion, 2. Hold and Wait, 3. No Preemption, 4. Circular Wait.

D16 — Medium: **Eventual consistency** ensures updates propagate over time. Cassandra uses **tunable consistency levels** (e.g., ONE, QUORUM, ALL). For read-after-write, `LOCAL_QUORUM` ensures the latest write is visible locally.

D17 — Medium: **mTLS** authenticates both client and server via certificates. Handshake: Client sends cert, Server validates; Server sends cert, Client validates. Commonly used in **Service Meshes** (e.g., Istio) for secure service-to-service communication.

D18 — Hard: A **hash table** maps keys to array indices. **Collision resolution**: Chaining (linked lists) or Open Addressing (probing). Time Complexity: Average O(1) for insert/lookup/delete; Worst O(n) if collisions are high.

D19 — Medium: **Circuit Breaker** prevents cascade failures. **Closed** (normal), **Open** (fail fast), **Half-open** (test recovery). Relates to retries: If circuit is Open, retries are skipped to prevent overload.

D20 — Hard: **Optimistic** (check version on commit, e.g., `SELECT ... FOR UPDATE` is not used) assumes low conflict. **Pessimistic** (lock rows immediately) assumes high conflict. Choose Optimistic for high concurrency/low conflict; Pessimistic for critical financial data.
