T1 — `tofu plan` calculates the execution plan and displays the changes that would be made without applying them. `tofu apply` executes the plan, creating, updating, or destroying resources to match the configuration.

T2 — State is a JSON file mapping real-world resources to configuration objects. Remote state is important for collaboration (multiple users), locking (preventing concurrent writes), and storing sensitive data securely off the local machine.

T3 — `count` uses integer indices; `for_each` uses map/set keys. `for_each` is preferred when resources have unique identifiers (like names) because adding/removing items doesn't shift indices, preventing unintended resource recreation.

T4 — OpenTofu detects this via **state drift**. Running `tofu plan` compares the state file to the actual infrastructure. If the config matches the manual change, `apply` will revert it. If the config differs, `apply` will update the resource to match the config.

T5 —
```hcl
resource "aws_instance" "web" {
  for_each = toset(["web1", "web2", "web3"])
  ami      = "ami-12345678"
  instance_type = "t2.micro"
  tags = {
    Name = each.key
  }
}
```

T6 — `terraform_remote_state` reads outputs from another state file (decoupled). `output` values are used within the same state. The recommended alternative is using a **shared backend (single state)** so outputs are accessible directly via module calls, reducing coupling and latency.

T7 — The `null_resource` provisioner runs during apply, but the output is computed before the provisioner finishes if not explicitly waited on. **Fix:** Add `depends_on = [null_resource.kubeconfig]` to the dependent resource, or ensure the output is generated by a resource that waits for cluster readiness (e.g., a `null_resource` with a `local-exec` that waits for the API).

T8 — `terraform import` is a CLI command to import existing resources into state. Limitations: manual, error-prone, no declarative tracking. The `import` block (1.5+) allows importing resources declaratively in code, enabling version control and automated CI/CD imports.

T9 — A `variable` is an input parameter passed into the module (e.g., `tfvars`). An `output` is a value exposed from the module to be used by other modules or the CLI.

T10 — A provider is a plugin that interacts with a specific API (e.g., AWS, Azure). Without `tofu init`, `tofu plan` fails because the provider binaries are not downloaded and initialized.

T11 — The `.terraform.lock.hcl` file records the exact versions of providers used. It should be committed to ensure reproducible builds and prevent breaking changes from provider updates.

T12 — `tofu destroy` removes all managed resources. To target a single resource: `tofu destroy -target=aws_instance.example`.

T13 — A data source reads data from an external system without creating or managing it. Example: `data "aws_ami" "ubuntu"` to get the latest Ubuntu AMI ID instead of hardcoding it.

T14 — `locals` define computed values within the configuration scope. `variable` blocks define inputs from outside. Use `locals` for intermediate calculations or constants that don't need external configuration.

T15 — `lifecycle` block controls resource behavior. `create_before_destroy`: creates new resource before destroying old (zero downtime). `prevent_destroy`: prevents accidental deletion. `ignore_changes`: ignores specific attribute changes in state (e.g., tags managed externally).

T16 — A `moved` block tracks resource identity changes during refactoring (e.g., moving a resource to a new module). It updates the state file to map the old address to the new address without destroying/recreating the resource.

T17 — The dependency graph determines execution order. **Implicit:** inferred from references (e.g., `aws_instance.id`). **Explicit:** defined via `depends_on`. `depends_on` reduces parallelism by forcing sequential execution, which should be used sparingly to avoid bottlenecks.

T18 — 1. **Reduce State Size:** Remove unused resources (tradeoff: manual cleanup). 2. **Plan Caching:** Use Terraform Cloud/Enterprise (tradeoff: cost/dependency). 3. **Optimize Data Sources:** Cache results or reduce calls (tradeoff: complexity). 4. **Modularize:** Reduce graph complexity (tradeoff: refactoring effort).

T19 — State locking prevents concurrent writes. If a lock isn't released (e.g., crash), subsequent runs hang. **Manual Break:** `tofu force-unlock <LOCK_ID>`. **Risk:** State corruption if another process is actively writing.

T20 — 1. Configure new backend in `backend` block. 2. Run `tofu init -migrate-state`. 3. Confirm migration. 4. Verify state file location in S3. 5. Run `tofu plan` to ensure state is accessible. Existing state file is uploaded to S3.

A1 — **Playbook:** YAML file defining automation steps. **Role:** Reusable structure for organizing tasks/files. **Task:** Single action executed on a host.

A2 — `ansible_facts` is a dictionary of system information. Access OS family via `{{ ansible_os_family }}`.

A3 — `command`: executes without shell (no pipes). `shell`: executes with shell (supports pipes). `raw`: executes without setup (no Python module transfer), useful for initial bootstrap.

A4 — Idempotency means running a task multiple times yields the same result. **Idempotent:** `apt: state=present` (installs only if missing). **Not Idempotent:** `shell: rm -rf /tmp/data` (deletes every time).

A5 —
```yaml
- name: Install htop
  package:
    name: htop
    state: present
  when: ansible_os_family == "Debian"

- name: Install htop and epel-release
  package:
    name:
      - epel-release
      - htop
    state: present
  when: ansible_os_family == "RedHat"
```

A6 — Precedence (Low to High): `role defaults` < `group_vars` < `host_vars` < `playbook vars` < `extra_vars`. `set_fact` overrides all at runtime.

A7 — Use `serial: 3` to limit concurrency. Group nodes into `agents` and `agents_batch`. Use `wait_for` or `async`/`poll` to ensure the first 3 complete before the next batch starts.

A8 — `ansible-vault encrypt_string`. Encrypts a specific string value. To encrypt a variable in a file, you must encrypt the whole file or use `vaulted_string` in a vars file.

A9 — Inventory lists managed hosts. **Static:** Defined in a file (INI/YAML). **Dynamic:** Generated by a script/API (e.g., AWS EC2) to reflect dynamic infrastructure.

A10 — `ansible webservers -m shell -a "df -h"`.

A11 — A handler is a task triggered by `notify`. It runs only if notified and executes after all other tasks in the play complete.

A12 — `ansible.cfg` configures Ansible behavior. Common settings: `inventory`, `remote_user`, `forks`, `host_key_checking`.

A13 — `register` saves a task's output to a variable. Use it in subsequent tasks via `when: my_var.stdout == "success"` or `debug: var=my_var`.

A14 — `import_tasks`: static, evaluated at parse time (faster, no conditionals). `include_tasks`: dynamic, evaluated at runtime (allows conditionals). Use `import` for performance, `include` for logic.

A15 — Ansible Galaxy is a repository for roles/collections. `requirements.yml` lists them. Run `ansible-galaxy install -r requirements.yml` to install.

A16 — Connection plugins manage communication. `ssh`: standard remote Linux. `local`: runs on control node. `network_cli`: for network devices (Cisco, Juniper).

A17 — `delegate_to`: runs task on a specific host. `run_once`: runs task once regardless of host count. Example: Gather facts on one node (`delegate_to: localhost`), then use data to configure others.

A18 — 1. **Increase `forks`:** More parallel hosts (tradeoff: memory). 2. **Enable `pipelining`:** Reduces SSH overhead (tradeoff: SSH config). 3. **Fact Caching:** Reduces gathering time (tradeoff: storage). 4. **Mitogen:** Faster transport (tradeoff: installation).

A19 — `{{ }}` outputs variables. `{% %}` controls logic (loops/if). `{# #}` comments.
```jinja
upstream backend_servers {
{% for server in servers %}
    server {{ server.ip }} weight={{ server.weight }}; # Health check: {{ server.health }}
{% endfor %}
}
```

A20 — 1. **Serial:** `serial: 5`. 2. **Pre-task:** Deregister from LB. 3. **Deploy:** Update app. 4. **Post-task:** Register to LB, `uri` health check. 5. **Block/Rescue:** If health check fails, `rescue` block re-registers previous version or rolls back.
