G1 — Easy: An array is a fixed-size value type `[T; N]` stored on the stack (usually). A slice is a dynamic view `[T]` containing a pointer to data, length, and capacity. When appending to a slice at capacity, the Go runtime allocates a new, larger backing array (typically doubling capacity), copies existing elements, and returns a new slice pointing to it.

G2 — `var x int` declares a variable with an explicit type and zero value (`0`). `x := 0` is a short variable declaration that infers the type. You cannot use `:=` to redeclare a variable in the same scope (e.g., `x := 0` followed by `x := 1` in the same block).

G3 — A goroutine is a lightweight thread managed by the Go runtime, not the OS. It differs from an OS thread by having a smaller stack (starts at 2KB) and being multiplexed onto fewer OS threads. The Go scheduler (GMP model) maps Goroutines (G) to OS Threads (M) via Processors (P), which hold execution contexts to manage concurrency efficiently without OS kernel intervention.

G4 — Channels are typed conduits for communication between goroutines. Unbuffered channels block both sender and receiver until the other side is ready. Buffered channels have an internal queue; sending blocks only if the buffer is full. Receiving blocks only if the buffer is empty.

G5 — The `context` package manages deadlines, cancellation, and request-scoped values across API boundaries.
```go
ctx, cancel := context.WithTimeout(context.Background(), 1*time.Second)
defer cancel()
resp, err := http.GetContext(ctx, "http://example.com")
```

G6 — Go interfaces are satisfied implicitly (duck typing); you don't declare implementation. Java/C# require explicit `implements`/`implements` keywords. `interface{}` (or `any`) accepts any type. It is powerful for generic code but dangerous because it bypasses type safety, requiring type assertions that can panic if incorrect.

G7 — This is unsafe due to a data race: one goroutine writes `data` and `ready`, while the main goroutine reads `ready` and `data` without synchronization. The compiler/runtime cannot guarantee visibility. Fix using `sync.Once`, `atomic.Bool`, or a channel to signal readiness.

G8 — `sync.Mutex` provides exclusive write access. `sync.RWMutex` allows multiple readers or one writer. `sync.Map` is optimized for read-heavy workloads with disjoint keys (no locking on reads). Trade-off: `sync.Map` has higher overhead for writes and contention compared to a standard `map` protected by a `Mutex`.

G9 — Go modules manage dependencies. `go.mod` declares the module path and direct dependencies. `go.sum` stores cryptographic checksums for reproducibility. `go mod tidy` removes unused dependencies and adds missing ones. MVS (Minimum Version Selection) selects the lowest version of a dependency that satisfies all constraints in the dependency graph.

G10 — `defer` schedules a function call to run after the surrounding function returns. Multiple defers execute in Last-In-First-Out (LIFO) order. Gotcha in loops: `defer` captures the loop variable by reference, not value. Fix by passing the variable as an argument to the deferred function.

G11 — A pointer holds a memory address. Its zero value is `nil`. Create a pointer to a struct literal using the address-of operator: `&MyStruct{Field: value}`.

G12 — `init()` functions run automatically before `main()` executes. They run during package initialization. A package can have multiple `init()` functions; they execute in the order they appear in the source files.

G13 — `new(T)` allocates zeroed memory for type `T` and returns a pointer `*T`. `make(T)` allocates and initializes internal data structures for slices, maps, and channels, returning the initialized value (not a pointer).

G14 — Errors are values implementing the `error` interface (`Error() string`). Idiomatic pattern: `if err != nil { return err }`. `errors.Is(err, target)` checks if `err` wraps `target`. `errors.As(err, &target)` checks if `err` wraps a specific type. `==` only works for concrete error types, not wrapped errors.

G15 — Map: nil (no backing array). Slice: nil (no backing array). Channel: nil (no buffer). Struct: zero values for all fields.

G16 — A race condition occurs when concurrent goroutines access shared data without synchronization, leading to unpredictable results. The `-race` flag adds instrumentation to detect data races at runtime. Fix by using `sync.Mutex` to protect access or channels to transfer ownership.

G17 — `select` waits on multiple channel operations, similar to `switch` but for channels. If multiple cases are ready, one is chosen randomly. Implement timeout by adding a `case <-time.After(duration)` branch.

G18 — Escape analysis determines if a variable lives on the stack or heap. If a variable's lifetime exceeds the function scope (e.g., returned pointer), it escapes to the heap. Inspect with `go build -gcflags="-m"`. It matters because heap allocations trigger GC, impacting latency and throughput.

G19 — Reflection allows runtime inspection of types and values. `reflect.Type` describes metadata; `reflect.Value` holds the data. Costs include performance overhead and loss of compile-time type safety. Justified for frameworks (e.g., serialization, ORM) where code generation is too complex.

G20 — `//go:embed` embeds files/directories into the binary at compile time. Works with `embed.FS`. Limitations: files must exist at compile time, no symlinks, and content is static. Interacts with `io/fs.FS` by providing a virtual filesystem interface to the embedded content.

R1 — Ownership manages memory without a garbage collector. Rules: 1. Each value has exactly one owner. 2. When the owner goes out of scope, the value is dropped. 3. You can transfer ownership (move) or borrow (reference), but not have multiple owners simultaneously.

R2 — `String` is an owned, mutable heap-allocated string. `&str` is an immutable borrowed slice of bytes. Use `String` when you need ownership or mutation. Use `&str` in function signatures for efficiency (no allocation) when the function only reads the data.

R3 — Borrowing allows temporary access without ownership. Rules: You can have either one mutable reference (`&mut T`) OR any number of immutable references (`&T`), but not both simultaneously. References must always be valid.

R4 — `Result<T, E>` represents success (`Ok(T)`) or failure (`Err(E)`). `Option<T>` represents a value (`Some(T)`) or absence (`None`). The `?` operator propagates errors.
```rust
fn read_file() -> Result<String, std::io::Error> {
    std::fs::read_to_string("file.txt")
}
```

R5 — `Box<T>` allocates on the heap with a single owner. `Rc<T>` (Reference Counted) allows multiple owners in a single thread. `Arc<T>` (Atomic Reference Counted) allows multiple owners across threads. Use `Arc` when sharing data across threads.

R6 — Traits define shared behavior. Unlike Go interfaces, traits can have default implementations. `dyn Trait` enables dynamic dispatch (vtable, runtime cost). `T: Trait` enables static dispatch (monomorphization, zero cost).

R7 — This fails because the compiler cannot determine which input's lifetime the returned reference belongs to. Fix by explicitly annotating lifetimes:
```rust
fn longest<'a>(x: &'a str, y: &'a str) -> &'a str {
    if x.len() > y.len() { x } else { y }
}
```

R8 — `Send` means ownership can be transferred across threads. `Sync` means shared references can be sent across threads. `Rc<T>` is not `Send` because its ref count is not atomic. `Arc<T>` uses atomic ref counts, making it `Sync` and `Send`.

R9 — `async/await` creates a state machine (Future) that can be polled. The executor (e.g., Tokio) polls the Future until completion. Unlike Go's stackful goroutines, Rust's are stackless (allocated on heap), reducing overhead but requiring manual pinning for self-referential data.

R10 — `unsafe` bypasses compiler safety checks. You can: 1. Dereference raw pointers. 2. Call unsafe functions/FFI. 3. Access/modify union fields. Justified for performance-critical code or interfacing with C libraries.

R11 — Pattern matching uses `match` to exhaustively check values. `if let` matches a single pattern. `match` enforces exhaustiveness (compiler error if cases are missing), ensuring all possibilities are handled.

R12 — `Vec<T>` is a growable heap array. `[T; N]` is a fixed-size stack array. When `Vec` capacity is exceeded, it allocates a new larger buffer, copies elements, and drops the old one (geometric growth).

R13 — Enums are Algebraic Data Types (ADTs) holding data variants. Unlike C/Java, they can hold different types per variant. `Option<T>` is `Some(T)` or `None`. Rust has no `null`; `None` forces explicit error handling.

R14 — `impl` defines methods. `&self` borrows the instance. `&mut self` mutates it. `self` consumes it. Associated functions (e.g., `String::new()`) do not take `self` and act like static methods.

R15 — `derive` macros generate boilerplate code. Common traits: `Debug`, `Clone`, `PartialEq`. Cannot derive if fields contain types that don't implement the trait (e.g., `Fn` closures).

R16 — `Cow<'a, T>` (Clone on Write) holds either a borrowed reference or an owned value. Useful when you want to avoid cloning unless modification is needed. Example: Parsing a config string; if you modify it, it clones; if not, it borrows.

R17 — Modules organize code. `mod` declares a module. `use` imports paths. `pub` makes items public. File system maps to modules (`mod.rs` or `mod.rs` in folder). Rust 2018 improved path resolution (`crate::` vs `extern crate`).

R18 — `Pin<T>` prevents a value from being moved in memory. Needed for self-referential structs and async `Future`s (which may hold self-references). `Unpin` is a marker trait; most types are `Unpin` by default.

R19 — Macros generate code. `macro_rules!` (declarative) matches patterns. Procedural macros (derive, attribute) operate on tokens via Rust code. Hygiene ensures macro variables don't collide with caller scope variables.

R20 — Zero-cost abstractions mean high-level features compile to efficient machine code. Iterators compile to loops. Generics use monomorphization (copying code per type). Traits use static dispatch. This avoids runtime overhead compared to dynamic languages.
