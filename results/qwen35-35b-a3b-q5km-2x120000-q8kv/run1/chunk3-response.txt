T1 — Easy: `tofu plan` calculates the execution plan and displays the changes that would be made without applying them. `tofu apply` executes the plan, creating, updating, or destroying resources to match the configuration.

T2 — Easy: State is a JSON file mapping real-world resources to your configuration. Remote state is important for collaboration (sharing state among team members) and enabling state locking to prevent concurrent modifications.

T3 — Medium: `count` uses integer indices, making resource ordering dependent on index changes. `for_each` uses map/set keys, ensuring stable resource addresses even if items are added/removed. `for_each` is preferred for stability and readability when managing collections of resources.

T4 — Medium: This is called "drift." OpenTofu detects it during `plan` by comparing the state file against the live infrastructure. If drift is found, `plan` shows the difference. `apply` can reconcile it if the configuration matches the desired state, or fail if the drift is unexpected.

T5 — Medium:
```hcl
resource "aws_instance" "web" {
  for_each = toset(["web1", "web2", "web3"])
  ami      = "ami-12345678"
  instance_type = "t2.micro"
  tags = {
    Name = each.key
  }
}
```

T6 — Hard: `terraform_remote_state` reads the entire state file of another stack (heavy coupling). `output` values are accessed via `module` blocks (lighter coupling). The recommended alternative is using `data` sources that query the cloud directly (e.g., `aws_ami`) or passing outputs via module blocks to decouple state management.

T7 — Hard: **Why:** `null_resource` provisioners run *after* resource creation. If the output depends on the provisioner's result (e.g., a file written by `remote-exec`), it may be empty when dependent resources try to read it during the same apply. **Fix:** Add `depends_on = [null_resource.kubeconfig]` to dependent resources, or better, use a `local_file` resource to store the kubeconfig and have the output depend on that file.

T8 — Medium: `tofu import` maps existing infrastructure to state without creating config. Limitations: No config generation, manual mapping required. The `import` block (1.5+) allows importing within the configuration file, enabling CI/CD automation and version control of import logic.

T9 — Easy: A `variable` is an input parameter passed into the module (e.g., from CLI or `.tfvars`). An `output` is a value exposed from the module to the caller. Variables are inputs; outputs are outputs.

T10 — Easy: A provider is a plugin that interacts with a specific API (e.g., AWS, Azure). Without `tofu init`, providers are not downloaded, and `tofu plan` will fail with errors about missing providers.

T11 — Easy: The `.terraform.lock.hcl` file records the exact versions of providers used. It should be committed to version control to ensure reproducible builds and prevent unexpected provider updates.

T12 — Easy: `tofu destroy` removes all resources managed by the configuration. To target a single resource: `tofu destroy -target=aws_instance.example`.

T13 — Easy: A data source reads data from an external system without creating or managing it. Example: Using `data "aws_ami"` to find the latest Ubuntu AMI ID instead of hardcoding it.

T14 — Medium: `locals` define computed values within the configuration scope. `variables` are inputs from outside. Use `locals` for intermediate calculations, string manipulation, or grouping values that don't need external input.

T15 — Medium: `lifecycle` controls resource behavior. `create_before_destroy`: Creates new resource before destroying old (zero-downtime). `prevent_destroy`: Prevents accidental deletion (e.g., production DB). `ignore_changes`: Ignores drift on specific attributes (e.g., tags managed externally).

T16 — Hard: A `moved` block allows renaming resources or moving them between modules without destroying and recreating them. It updates the state file to reflect the new address, preserving the resource's identity.

T17 — Hard: The dependency graph is a DAG determining execution order. Implicit dependencies come from resource references in config. Explicit dependencies use `depends_on`. `depends_on` forces serialization, reducing parallelism and slowing down apply. Use sparingly to avoid bottlenecks.

T18 — Hard: 1. `refresh=false`: Skips state refresh (faster, risks drift). 2. State splitting: Use modules to isolate state (faster parallelism, higher complexity). 3. Provider caching: Use local provider cache (faster downloads, disk usage). 4. Reduce `parallelism`: Wait, increase `parallelism` setting (faster apply, higher resource usage).

T19 — Hard: Locking prevents concurrent writes to the state file. If a lock isn't released (crash), subsequent commands fail. Break it with `tofu force-unlock <LOCK_ID>`. Risk: Data corruption if another process is actually running.

T20 — Hard: 1. Update config with `backend "s3" { ... }`. 2. Run `tofu init -migrate-state`. 3. OpenTofu moves the local state file to S3. 4. Verify with `tofu state list`. 5. Run `tofu apply` to ensure remote state is active.

A1 — Easy: A **Playbook** is a YAML file defining the execution steps. A **Role** is a reusable directory structure for organizing tasks, vars, and templates. A **Task** is a single action (e.g., install a package) within a playbook or role.

A2 — Easy: `ansible_facts` contains gathered system information. Access OS family via `{{ ansible_os_family }}`.

A3 — Medium: `command`: Runs without shell (no pipes). `shell`: Runs with shell (supports pipes). `raw`: Runs without setup (no Python module transfer), useful for initial bootstrap.

A4 — Medium: Idempotency means running a task multiple times yields the same result. **Idempotent:** `apt: state=present` (installs only if missing). **Not Idempotent:** `shell: echo "hello" > /tmp/test` (overwrites file every time).

A5 — Medium:
```yaml
- name: Install htop
  package:
    name: htop
    state: present
  when: ansible_os_family == "Debian"

- name: Install htop and epel-release
  package:
    name:
      - htop
      - epel-release
    state: present
  when: ansible_os_family == "RedHat"
```

A6 — Hard: Priority (High to Low): Command line (`-e`) > Playbook vars > Host vars > Group vars > Role defaults. `set_fact` overrides most, but `extra_vars` usually wins.

A7 — Hard: Use `serial: 3` in the play to limit concurrency. Group nodes into `agents` and `batch`. Use `wait_for` or `meta: flush_handlers` to ensure the first 3 nodes are ready before the next batch starts.

A8 — Medium: `ansible-vault` encrypts sensitive data. To encrypt a single variable: `ansible-vault encrypt_string 'secret_value' --name 'my_secret'`.

A9 — Easy: Inventory is a list of hosts. **Static:** Defined in a file (INI/YAML). **Dynamic:** Generated by a script or API (e.g., AWS EC2), useful for cloud environments.

A10 — Easy: `ansible webservers -m shell -a "df -h"`.

A11 — Easy: A **handler** is a task triggered by `notify` when a task reports a change. It runs at the end of the play. Regular tasks run immediately.

A12 — Easy: `ansible.cfg` configures Ansible behavior. Common settings: `inventory`, `remote_user`, `forks`, `host_key_checking`.

A13 — Easy: `register` saves a task's output to a variable. Use it in subsequent tasks via `when: my_var.stdout == "success"` or `debug: var=my_var`.

A14 — Medium: `import_tasks` is static (evaluated at parse time). `include_tasks` is dynamic (evaluated at runtime). `include_tasks` supports `when` on the include line; `import_tasks` does not.

A15 — Medium: Ansible Galaxy is a repository for roles/collections. `requirements.yml` lists them. Install with `ansible-galaxy install -r requirements.yml`.

A16 — Medium: Connection plugins define how Ansible connects. `ssh`: Remote Linux/Unix. `local`: Control node only. `network_cli`: Network devices (Cisco/Juniper).

A17 — Hard: `delegate_to: localhost` runs the task on the control node. `run_once` ensures it runs once even for multiple hosts. Example: Gather facts from remote hosts, then generate a config file on the control node.

A18 — Hard: 1. `forks`: Increase parallelism (Tradeoff: RAM usage). 2. `pipelining`: Reduces SSH connections (Tradeoff: SSH config). 3. `fact_caching`: Caches gathered facts (Tradeoff: Cache backend setup). 4. `mitogen`: Optimized transport (Tradeoff: Installation complexity).

A19 — Hard: Jinja2 is the templating engine. `{{ }}` (expression), `{% %}` (control flow), `{# #}` (comment).
```jinja2
upstream backend_servers {
{% for server in backend_servers %}
    server {{ server.ip }} weight={{ server.weight }}; # Health check: {{ server.health_check }}
{% endfor %}
}
```

A20 — Hard: 1. Set `serial: 1` (or batch size). 2. `pre_tasks`: Deregister from LB. 3. `tasks`: Deploy app. 4. `post_tasks`: Register to LB + `uri` health check. 5. `block/rescue`: If health check fails, rollback (revert deployment, re-register).
