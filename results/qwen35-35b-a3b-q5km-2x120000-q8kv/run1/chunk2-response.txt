K1 — Easy: A **Deployment** manages stateless applications, ensuring a desired number of identical pod replicas are running. A **StatefulSet** manages stateful applications, providing stable network identities (pod names), ordered deployment/termination, and persistent storage guarantees.

K2 — Easy: A **ClusterIP** Service assigns an internal virtual IP. **kube-proxy** (using iptables or IPVS) updates node-level rules to route traffic from the Service IP to the backend Pod IPs listed in the Service's **Endpoints** object.

K3 — Medium: A **PersistentVolume (PV)** is a cluster-level storage resource. A **PersistentVolumeClaim (PVC)** is a user request for storage. A **StorageClass** defines the provisioning class for dynamic PV creation. A PVC binds to an existing PV (static) or triggers the StorageClass to provision a new PV (dynamic).

K4 — Medium: 1. Check application logs (`kubectl logs`) for errors. 2. Check events (`kubectl describe`) for OOMKilled or probe failures. 3. Verify resource limits vs. node capacity. 4. Inspect liveness/readiness probe configuration for incorrect paths or timeouts.

K5 — Medium: **CoreDNS** provides DNS resolution for Kubernetes services (e.g., `svc.namespace`). If CoreDNS pods are down, internal service discovery fails, though external DNS resolution may still work depending on the cluster's `resolv.conf` configuration.

K6 — Hard: The scheduler uses **Filtering** (remove nodes that don't meet requirements like taints/resources) and **Scoring** (rank remaining nodes). Factors considered: 1. CPU/Memory requests vs. capacity. 2. Node Affinity rules. 3. Pod Affinity/Anti-Affinity rules.

K7 — Hard: Yes, it will be scheduled (256Mi request < 500Mi free). At runtime, if it uses 600Mi (exceeding 512Mi limit), the container will be **OOMKilled** and restarted.

K8 — Hard: 1. Request hits External Load Balancer. 2. LB forwards to Ingress Controller (NGINX). 3. NGINX matches Ingress rules to Service. 4. Service routes to **Endpoints** (Pod IPs). 5. **kube-proxy** (iptables/IPVS) on the node directs traffic to the specific Pod.

K9 — Medium: A **NetworkPolicy** restricts traffic flow between pods. Default behavior (no policy): Allow all traffic. When applied: Traffic is denied by default unless explicitly allowed by the policy rules.

K10 — Medium: **RollingUpdate** replaces pods gradually (zero downtime). **Recreate** kills all existing pods before starting new ones. Choose **Recreate** for stateful applications requiring exclusive access or atomic configuration changes.

K11 — Easy: A **Namespace** provides logical isolation for resources. Default namespaces: `default`, `kube-system`, `kube-public`, `kube-node-lease`.

K12 — Easy: `kubectl describe` shows detailed status. Information shown: 1. **Events** (warnings/errors). 2. **Conditions** (Ready status). 3. **Mounts** (volumes/configmaps). `kubectl get` only shows summary status/age.

K13 — Easy: A **ConfigMap** stores non-sensitive configuration data. Pods consume it via: 1. **Environment Variables**. 2. **Volume Mounts** (files).

K14 — Easy: **ClusterIP** (internal cluster access), **NodePort** (access via `<NodeIP>:<Port>`), **LoadBalancer** (external cloud load balancer provisioned).

K15 — Easy: A **DaemonSet** ensures a copy of a pod runs on all (or specific) nodes. Use for logging agents, monitoring daemons, or storage daemons.

K16 — Medium: **Taints** repel pods; **Tolerations** allow pods to schedule on tainted nodes. **Node Affinity** attracts pods to nodes. Use case: Taint for dedicated GPU nodes; Affinity to co-locate app and cache pods.

K17 — Medium: An **Operator** is a custom controller managing a complex application via Custom Resource Definitions (CRDs). It differs from standard controllers by automating operational tasks (backup, scaling, upgrades) specific to that application.

K18 — Hard: etcd uses **strong consistency** (linearizable). If quorum is lost, the cluster becomes read-only/unavailable. Recovery: Restore from a snapshot/backup onto new etcd nodes, remove dead members, and rejoin the cluster.

K19 — Hard: **Role/ClusterRole** define permissions. **RoleBinding/ClusterRoleBinding** grant them to users/groups.
```yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata: namespace: my-ns
rules:
- apiGroups: [""]
  resources: ["pods", "pods/log"]
  verbs: ["get", "list", "watch"]
---
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata: namespace: my-ns
subjects:
- kind: User
  name: developer
roleRef:
  kind: Role
  name: pod-reader
```

K20 — Hard: `ndots` (default 5) causes the resolver to try appending up to 5 suffixes to the domain before querying upstream. This causes excessive queries for short names. Fix: Set `dnsConfig: ndots: 1` in the pod spec to reduce retries.

D1 — Easy: **Rebase** rewrites history to create a linear commit chain. **Merge** preserves history with a merge commit. Avoid rebase on shared branches to prevent history rewriting conflicts.

D2 — Easy: **TCP** is connection-oriented and reliable (HTTP, SSH). **UDP** is connectionless and fast (DNS, VoIP).

D3 — Medium: A **race condition** occurs when output depends on the sequence of uncontrollable events. Example: Two threads incrementing a shared counter without a lock, resulting in lost updates.

D4 — Medium: **Symmetric** uses one key for encryption/decryption. **Asymmetric** uses a public/private key pair. TLS uses Asymmetric for the handshake (key exchange) and Symmetric for data transfer (performance).

D5 — Medium: A **layer** is a diff of the filesystem in a Docker image. Layer caching improves build speed. Best practice: Copy dependency files (e.g., `package.json`) before source code to cache dependency installation layers.

D6 — Medium: 1. Check upstream service health. 2. Verify database connectivity. 3. Check resource limits (CPU/Mem). 4. Review application logs for exceptions.

D7 — Medium: **CAP Theorem**: Consistency, Availability, Partition Tolerance. **CP** (Zookeeper, HBase) prioritizes consistency. **AP** (Cassandra, DynamoDB) prioritizes availability during partitions.

D8 — Medium: **Horizontal** adds nodes; **Vertical** adds resources to a node. Patterns enabling horizontal scaling: Stateless services, sharding, load balancing.

D9 — Easy: An **environment variable** is a dynamic value set at runtime. To persist across sessions on Linux, export it in `~/.bashrc` or `~/.profile`.

D10 — Easy: **JSON** is strict key-value (machine-friendly). **YAML** is human-readable (indentation-based). Advantage JSON: Strict parsing. Advantage YAML: Readable config files.

D11 — Easy: **REST** is an architectural style. **GET** (retrieve), **POST** (create), **PUT** (replace), **DELETE** (remove).

D12 — Easy: **Version pinning** specifies exact dependency versions. Important for reproducible builds to prevent breakage from upstream updates.

D13 — Easy: **Compiled** (source to binary) vs **Interpreted** (source to execution). Compiled: C, Go. Interpreted: Python, JavaScript.

D14 — Medium: A **reverse proxy** receives client requests and forwards them to servers (e.g., Nginx for web, HAProxy for load balancing). A **forward proxy** acts on behalf of clients to access external resources.

D15 — Medium: A **deadlock** is a circular wait where processes cannot proceed. Coffman conditions: 1. Mutual exclusion. 2. Hold and wait. 3. No preemption. 4. Circular wait.

D16 — Hard: **Eventual consistency** ensures updates propagate over time. Cassandra uses **tunable consistency levels** (ONE, QUORUM, ALL). For read-after-write, use `LOCAL_QUORUM` or `SERIAL`.

D17 — Hard: **mTLS** authenticates both client and server via certificates. Handshake: Client sends cert, server verifies it; server sends cert, client verifies it. Commonly used in **Service Meshes** (e.g., Istio).

D18 — Hard: A **hash table** maps keys to buckets via a hash function. Collision resolution: **Chaining** (linked lists) or **Open Addressing** (probing). Avg complexity: O(1). Worst case: O(n).

D19 — Hard: **Circuit breaker** prevents cascade failures. States: **Closed** (normal), **Open** (fail fast), **Half-open** (test recovery). Relates to retries: Do not retry if circuit is open.

D20 — Hard: **Optimistic** (check version on commit) vs **Pessimistic** (lock on read). Choose Optimistic for high concurrency/low conflict. Choose Pessimistic for high conflict/critical data (e.g., `SELECT FOR UPDATE`).
