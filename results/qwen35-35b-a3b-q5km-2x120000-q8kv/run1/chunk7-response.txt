AA1 — **Monolith vs. Microservices:** A monolith is a single, unified codebase deployed as one unit. Microservices are independent, loosely coupled services deployed separately. **Problems introduced by microservices:** 1. **Network Latency:** Communication is over the network, not in-process. 2. **Distributed Tracing:** Debugging requires correlating logs across multiple services. 3. **Data Consistency:** Maintaining ACID transactions across services requires complex patterns (e.g., Sagas) rather than local DB transactions.

AA2 — **Sync vs. Async:** Synchronous communication blocks the caller until a response is received (e.g., HTTP/REST). Asynchronous communication sends a message and continues processing without waiting (e.g., Message Queues). **Examples:** HTTP (Sync) vs. RabbitMQ (Async).

AA3 — **API Gateway vs. Reverse Proxy:** A reverse proxy forwards traffic to backend servers (L4/L7 routing). An API Gateway adds business logic like authentication, rate limiting, and API versioning. **Replacement:** An API Gateway cannot fully replace a reverse proxy in all cases; a Layer 4 load balancer is often still needed upstream for TCP health checks and SSL termination before the Gateway handles HTTP logic.

AA4 — **Saga Pattern:** A sequence of local transactions where each updates data and triggers the next. If a step fails, compensating transactions undo previous steps. **Choreography:** Services communicate via events (loose coupling, hard to debug). **Orchestration:** A central coordinator directs steps (tighter coupling, easier to manage flow). **Use:** Choreography for simple flows; Orchestration for complex workflows requiring visibility and control.

AA5 — **CQRS:** Command Query Responsibility Segregation separates read (Query) and write (Command) models. **Paired with Event Sourcing:** Writes are stored as immutable events; reads are projections of those events. **Problem Solved:** Allows independent scaling of read/write paths and provides a complete audit trail/history of state changes, which traditional CRUD (current state only) does not.

AA6 — **Circuit Breaker:** Prevents repeated calls to a failing service. **States:** Closed (normal), Open (failures detected, requests blocked), Half-Open (testing if service recovered). **Difference from Retry:** Retry attempts to fix transient failures; Circuit Breaker stops hammering persistent failures. **Use Together:** Use retry for transient network glitches, but wrap the retry logic in a Circuit Breaker to prevent cascading failures if the service is down.

AA7 — **Tenant Isolation Models:**
1. **Shared DB/Schema:** Lowest cost, highest noisy-neighbor risk, complex security (row-level).
2. **Shared DB/Schema:** Medium cost, better isolation (schema level), harder migrations.
3. **Separate DB:** Highest cost, best isolation/security, highest operational complexity.
**Trade-off:** Choose based on compliance needs and budget. High-security tenants need separate DBs; low-risk SaaS can use shared schema.

AA8 — **Consistency:** Strong consistency ensures immediate visibility of updates; eventual consistency allows temporary divergence. **Inventory Architecture:** Use **Strong Consistency** to prevent overselling. **Data Store:** Relational DB (PostgreSQL) with ACID transactions. **Patterns:** Pessimistic locking (SELECT FOR UPDATE) or a queue-based serial processing model to ensure only one transaction modifies stock at a time.

AA9 — **Sidecar Proxy:** A container deployed alongside the main app to handle network traffic. **Envoy/Istio:** Envoy is the data plane proxy; Pilot is the control plane managing config. **mTLS:** Encrypts traffic between services without app code changes. **Benefit:** Solves service-to-service authentication and encryption centrally, whereas app-level TLS requires managing certs per application.

AA10 — **Strangler Fig:** Incrementally migrate functionality from a legacy system to new services. **Strategy:** Deploy a Gateway in front of the .NET Framework monolith. Route specific endpoints (e.g., `/api/orders`) to new .NET 8 microservices. Gradually move more endpoints until the monolith is "strangled" and can be decommissioned.

AA11 — **12-Factor App:** Methodology for building cloud-native apps. **Factors:** 1. Codebase (one repo), 2. Dependencies (explicitly declared), 3. Config (env vars), 4. Backing services (treated as attached resources), 5. Build/Release/Run (strict separation), 6. Processes (stateless). **Importance:** Ensures portability, scalability, and maintainability in dynamic cloud environments.

AA12 — **Event-Driven (Kafka) vs. Request-Driven (REST):**
*   **Coupling:** Event-driven is loosely coupled; Request-driven is tightly coupled.
*   **Scalability:** Event-driven scales better for high throughput; Request-driven is simpler for low latency.
*   **Debugging:** Event-driven is harder (distributed state); Request-driven is easier (stack traces).
*   **Ordering:** Kafka guarantees ordering; REST does not.
*   **Failure:** Kafka uses dead-letter queues; REST uses timeouts/retries.
**Choice:** Use Event-driven for decoupled, high-volume processing; Request-driven for real-time user interactions.

AA13 — **Load Balancer:** Distributes traffic across servers. **L4:** Routes based on IP/Port (TCP/UDP). **L7:** Routes based on HTTP headers/URL. **Use Case:** L7 is required for SSL termination or routing `/api` to one service and `/web` to another on the same IP.

AA14 — **Message Queue vs. Log Broker:** **Queue (RabbitMQ):** Tasks are consumed once and deleted. **Log Broker (Kafka):** Messages are retained and replayed. **Pick Queue:** For task processing (email, jobs). **Pick Log:** For analytics, audit trails, or event streaming where history matters.

AA15 — **Container vs. VM:** **Container:** OS-level virtualization (shares kernel). **VM:** Hardware-level virtualization (full OS). **Popularity:** Containers are lighter, start faster, and have higher density, making them ideal for microservices scaling.

AA16 — **Stateless vs. Stateful:** **Stateless:** No session data stored on the server (e.g., Web Server). **Stateful:** Maintains state (e.g., Database). **Scaling:** Stateless is easier to scale horizontally because any instance can handle any request without data migration.

AA17 — **ORM:** Maps database tables to programming language objects. **Examples:** Entity Framework (.NET), Hibernate (Java). **Trade-offs:** ORMs increase developer productivity but can generate inefficient SQL and hide performance bottlenecks compared to raw SQL.

AA18 — **Bulkhead Pattern:** Isolates resources (threads, connections) for different services. **Implementation:** Use separate thread pools for each downstream dependency. If one service hangs, it exhausts only its pool, not the entire application's resources.

AA19 — **OT vs. CRDT:** **OT:** Server-authoritative, complex conflict resolution, high latency. **CRDT:** Client-authoritative, mathematically convergent, low latency. **Offline Support:** CRDTs handle offline editing better as conflicts resolve automatically on merge. **Choice:** CRDT for offline-first collaborative editing.

AA20 — **Distributed Tracing:** **Context Propagation:** TraceID/SpanID passed via headers (W3C/B3). **Sampling:** Head-based (early decision) vs. Tail-based (late decision based on latency). **Storage:** Jaeger/Tempo. **Kafka Continuity:** Inject trace context into message headers so the consumer can resume the trace. **Challenge:** Ensuring context survives serialization and async hops.

OP1 — **Hypervisor Type 1 vs. 2:** **Type 1 (Bare-metal):** Runs directly on hardware (e.g., VMware ESXi, Microsoft Hyper-V). **Type 2 (Hosted):** Runs on an OS (e.g., VirtualBox, VMware Workstation). Type 1 is for production; Type 2 is for development.

OP2 — **SAN vs. NAS:** **SAN:** Block storage (FC/iSCSI), high performance, used for databases. **NAS:** File storage (NFS/SMB), easier sharing, used for file servers. **Use:** SAN for VM/DB performance; NAS for shared user files.

OP3 — **3-Tier vs. Leaf-Spine:** **3-Tier:** Core/Distribution/Access. **Leaf-Spine:** All-to-all connectivity. **Problem Solved:** Leaf-Spine eliminates oversubscription and reduces latency compared to 3-tier, which has bottlenecks at the distribution layer.

OP4 — **Proxmox HA:** **Architecture:** 3+ nodes for quorum. **Storage:** Ceph or shared iSCSI. **Fencing:** STONITH (DRAC/ILO) to power off failed nodes. **Failure:** VMs restart on surviving nodes automatically. **Quorum:** Prevents split-brain.

OP5 — **VLAN Trunking:** **802.1Q:** Tags frames with VLAN ID. **Linux Config:** Create sub-interfaces (e.g., `eth0.10`) with `ip link add link eth0 name eth0.10 type vlan id 10`. Allows one NIC to carry traffic for multiple VLANs.

OP6 — **RAID Levels:** **RAID 1:** Mirror (50% cap). **RAID 5:** Striping + Parity (1 drive loss). **RAID 6:** Double Parity (2 drive loss). **RAID 10:** Mirror + Stripe (High perf, 50% cap). **RAID Z2:** ZFS equivalent to RAID 6. **Choice:** RAID 10 for DB (6 drives) due to write performance and fault tolerance.

OP7 — **DNS Architecture:** **Internal:** AD DNS servers (Active Directory integrated). **External:** Public authoritative servers. **Split-Horizon:** Internal queries resolve private IPs; external resolve public. **Laptop VPN:** Tunnel DNS traffic to internal resolver to access internal resources. **DNSSEC:** Signs external zone for validation.

OP8 — **PKI Chain:** **Root CA:** Offline, trusted anchor. **Subordinate CA:** Issues end-entity certs. **Revocation:** CRL (list) vs. OCSP (real-time). **Compromise:** Subordinate = Revoke Issuing CA. Root = Trust chain broken, requires new Root CA deployment.

OP9 — **WAN Consolidation:** **MPLS:** Reliable, expensive. **IPsec:** Cheap, variable latency. **SD-WAN:** Intelligent routing, cost-effective. **WireGuard:** Simple, secure. **Recommendation:** SD-WAN for 50 users/site with VoIP (prioritizes traffic, handles internet variability better than MPLS).

OP10 — **iDRAC/iLO/IPMI:** **Out-of-Band:** Management interface independent of OS. **Critical:** Allows reboot/install if OS crashes. **Exposed Risk:** If exposed to internet, attackers can gain root access to hardware (power cycle, BIOS flash).

OP11 — **LVM vs. ZFS:** **LVM:** Volume management only. **ZFS:** Filesystem + Volume + RAID. **Snapshots:** LVM (copy-on-write, slower). ZFS (instant, cheap, immutable). **Use:** ZFS for data integrity/RAID; LVM for simple volume management.

OP12 — **Stretch Cluster:** **Design:** 2 sites + 1 Tiebreaker (witness). **Ceph:** CRUSH rules distribute data across sites. **Latency:** <5ms between sites. **Split-Brain:** Without tiebreaker, network partition causes both sites to write, corrupting data. Tiebreaker breaks the tie.

OP13 — **DHCP:** **DORA:** Discover, Offer, Request, Acknowledge. **Unavailable:** Client uses APIPA (169.254.x.x) if lease expires and server is down. **Windows:** Tries to renew; if fails, reboots or uses static fallback.

OP14 — **Managed vs. Unmanaged:** **Managed:** Supports VLANs, QoS, ACLs, SNMP. **Unmanaged:** Plug-and-play. **Importance:** Managed switches allow network segmentation (security) and traffic prioritization (VoIP).

OP15 — **NTP:** **Time Sync:** Critical for Kerberos auth, log correlation, and clustering. **Drift:** Causes auth failures (tickets expire), log confusion, and cluster split-brain.

OP16 — **Physical vs. VM vs. Container:** **Physical:** Bare metal. **VM:** Virtualized OS. **Container:** Virtualized App. **Bare Metal Use:** High-performance computing, low-latency trading, or when hardware passthrough is required.

OP17 — **UPS vs. Generator:** **UPS:** Immediate battery power (seconds). **Generator:** Long-term power (minutes/hours). **ATS:** Switches load from Grid to UPS to Generator. **Need Both:** UPS bridges the gap until generator starts.

OP18 — **Active-Passive vs. Active-Active:** **Active-Passive:** One node handles load; failover to standby. **Active-Active:** Both handle load. **PostgreSQL HA:** Pacemaker/Corosync manages VIP. **STONITH:** Fences failed node. **Failover:** VIP moves, DB starts on new node.

OP19 — **Backup/Recovery:** **3-2-1:** 3 copies, 2 media, 1 offsite. **Immutable:** Protects against ransomware. **Test:** Regular restore drills. **Difference:** Backup is data copy; DR is the plan/process to restore operations after disaster.

OP20 — **802.1X:** **Roles:** Supplicant (Client), Authenticator (Switch), Server (RADIUS). **EAP:** EAP-TLS (Cert) vs. PEAP (User). **Dynamic VLAN:** Assigns VLAN based on identity. **Fail:** Port blocked or placed in Guest VLAN. **MAB:** Authenticates device by MAC address if supplicant is missing (IoT).
