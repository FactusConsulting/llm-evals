G1 — Easy: An array is a fixed-size value type `[T]N`. A slice is a dynamic view descriptor containing a pointer, length, and capacity `[]T`. When appending to a slice that exceeds its capacity, Go allocates a new underlying array (usually doubling capacity), copies data, and returns a new slice header.

G2 — Easy: `var x int` declares a variable with the zero value. `x := 0` declares and assigns in one statement (short declaration). You cannot use `:=` if the variable already exists in the current scope (e.g., redeclaring `x`), or outside a function (e.g., package level).

G3 — Medium: A goroutine is a lightweight thread managed by the Go runtime, not the OS. The Go scheduler (GMP model) maps Goroutines (G) to OS Threads (M) via Processor contexts (P), allowing efficient multiplexing of thousands of goroutines on fewer OS threads.

G4 — Medium: Channels are typed pipes for communication. Unbuffered channels block both sender and receiver until handshake. Buffered channels have a queue; sending blocks only if full. Receiving blocks only if empty.

G5 — Medium: The `context` package manages deadlines, cancellation, and request-scoped values.
```go
ctx, cancel := context.WithTimeout(context.Background(), 5*time.Second)
defer cancel()
resp, err := http.GetContext(ctx, url)
```

G6 — Medium: Go interfaces are satisfied implicitly (duck typing). Java/C# require explicit declaration. `interface{}` (or `any`) accepts any type. It is powerful for generic containers but dangerous because it requires type assertions, losing compile-time safety.

G7 — Hard: This is unsafe due to a data race. The main goroutine reads `data` without synchronization after `ready` is set. The Go memory model requires a "happens-before" relationship. Fix: Use `sync.Once`, `atomic.Bool`, or a channel to signal readiness.

G8 — Medium: `sync.Mutex` locks for exclusive access. `sync.RWMutex` allows multiple readers or one writer. `sync.Map` is optimized for read-heavy, sparse key scenarios. Regular map + mutex is better for write-heavy or dense maps due to `sync.Map` overhead.

G9 — Medium: Go modules manage dependencies. `go.mod` lists direct dependencies; `go.sum` stores checksums for integrity. `go mod tidy` removes unused imports and adds missing ones. MVS (Minimum Version Selection) selects the lowest version satisfying all constraints.

G10 — Medium: `defer` postpones function execution until the surrounding function returns. Multiple defers execute in LIFO (Last In, First Out) order. A common gotcha is capturing loop variables in closures within loops; defer captures the variable by reference, not value.

G11 — Easy: A pointer holds a memory address. Its zero value is `nil`. Create a pointer to a struct literal using `&Struct{Field: value}`.

G12 — Easy: `init()` runs automatically before `main()` during package initialization. A package can have multiple `init()` functions; they execute in the order they appear in the source file.

G13 — Easy: `new(T)` allocates zeroed memory for type `T` and returns a pointer `*T`. `make(T)` initializes built-in types (map, slice, channel) and returns the type itself (not a pointer).

G14 — Easy: Errors are values implementing the `error` interface (`Error() string`). Idiom: `if err != nil { return err }`. `errors.Is` checks wrapped errors against a sentinel. `errors.As` performs type assertion on wrapped errors. `==` only works for exact matches.

G15 — Easy: Built-in composites: `map`, `slice`, `channel`, `struct`. Zero values: `map`/`slice`/`channel` are `nil`; `struct` fields are their respective zero values (e.g., `0`, `""`, `false`).

G16 — Medium: A race condition occurs when concurrent accesses to shared data lack synchronization. The `-race` flag instruments code to detect races at runtime. Fix: Use `sync.Mutex` to protect shared state or channels to transfer ownership.

G17 — Medium: `select` waits on multiple channel operations. Unlike `switch`, it chooses a ready case randomly if multiple are ready. Implement timeout by adding a case with `time.After(duration)`.

G18 — Hard: Escape analysis determines if a variable lives on the stack or heap. If a variable's address escapes the function scope, it moves to the heap. Inspect with `go build -gcflags="-m"`. Heap allocation increases GC pressure and latency.

G19 — Hard: `reflect` allows runtime introspection. `reflect.Type` is metadata; `reflect.Value` is the data. Reflection is slow due to dynamic dispatch and type checks. Justified for frameworks (serialization, DI) but avoid in hot paths.

G20 — Hard: `//go:embed` embeds files/directories at compile time into a `embed.FS`. It works with single files, directories, or `fs.FS`. Limitations: static content only, no runtime modification. Interacts with `io/fs` for traversal.

R1 — Easy: Ownership ensures memory safety without GC. Rules: 1) Each value has one owner. 2) When owner goes out of scope, value is dropped. 3) Values can be moved, not copied.

R2 — Easy: `String` is owned, heap-allocated, mutable. `&str` is a borrowed, immutable slice of bytes. Use `&str` for function arguments (zero-copy), `String` for owned/modifiable data.

R3 — Medium: Borrowing allows access without ownership. Rules: 1) Immutable references (`&T`) can be shared. 2) Mutable reference (`&mut T`) is exclusive. 3) Cannot have mutable and immutable references simultaneously.

R4 — Medium: `Result<T, E>` represents success or error. `Option<T>` represents `Some(T)` or `None`. The `?` operator propagates errors.
```rust
fn read_file() -> Result<String, io::Error> {
    fs::read_to_string("file.txt")
}
```

R5 — Medium: `Box<T>` owns data on heap. `Rc<T>` is reference-counted, single-threaded. `Arc<T>` is atomic reference-counted, thread-safe. Use `Arc` when sharing data across threads.

R6 — Medium: Traits define shared behavior. `dyn Trait` uses dynamic dispatch (vtable, runtime cost). `T: Trait` uses static dispatch (monomorphization, zero cost). Rust traits are more powerful than Java interfaces (can define default impls).

R7 — Hard: The compiler cannot infer which input lifetime the return value borrows. Fix: Explicitly bind lifetimes.
```rust
fn longest<'a>(x: &'a str, y: &'a str) -> &'a str {
    if x.len() > y.len() { x } else { y }
}
```

R8 — Medium: `Send` means ownership can be transferred across threads. `Sync` means `&T` can be shared. `Rc<T>` is not `Send` because ref counting is not atomic. `Arc<T>` uses atomic ref counting, making it `Send` and `Sync`.

R9 — Medium: `async/await` creates a state machine (`Future`). The executor (e.g., Tokio) polls the future. Unlike Go's preemptive goroutines, Rust's async is cooperative, requiring explicit yielding.

R10 — Easy: `unsafe` allows raw pointer manipulation, FFI, and union access. Justified for low-level systems programming, performance-critical code, or interfacing with C libraries where safety cannot be guaranteed by the compiler.

R11 — Easy: `match` exhaustively checks all patterns. `if let` checks a single pattern. The compiler enforces exhaustiveness in `match` to prevent logic errors from unhandled cases.

R12 — Easy: `Vec<T>` is a growable heap array. `[T; N]` is a fixed-size stack array. `Vec` reallocates memory (doubling capacity) when pushing beyond capacity, copying elements.

R13 — Easy: Enums (ADTs) can hold data in variants. `Option<T>` is `Some(T)` or `None`. Rust has no `null`; `Option` forces handling of absence at compile time, preventing null pointer exceptions.

R14 — Easy: `impl` defines methods. `&self` borrows, `&mut self` mutates, `self` consumes. Associated functions (e.g., `String::new()`) do not take `self` and act like static methods.

R15 — Easy: `derive` auto-implements traits. Common: `Debug`, `Clone`, `PartialEq`. Cannot derive if fields contain types that don't support the trait (e.g., `Fn` closures in `Clone`).

R16 — Medium: `Cow<'a, T>` (Clone on Write) holds either a borrowed `&'a T` or an owned `Box<T>`. Useful when data is often read but sometimes modified, avoiding allocation until mutation occurs.

R17 — Medium: `mod` defines a module, `use` imports paths, `pub` exposes items. Filesystem maps to modules (`mod.rs` or `mod/`). Rust 2018 improved path resolution (`crate::` vs `self::`).

R18 — Hard: `Pin<T>` prevents a value from being moved in memory. Needed for self-referential structs and async (where `Future` must not move while awaiting). `Unpin` indicates a type can be safely moved.

R19 — Medium: `macro_rules!` are declarative (token matching). Procedural macros (derive, attribute) are Rust code generating code. Hygiene ensures macro variables don't conflict with local scope.

R20 — Hard: Zero-cost abstractions mean high-level code compiles to efficient machine code. Iterators compile to loops. Generics use monomorphization (copying code per type). Traits use static dispatch, avoiding vtable overhead.
