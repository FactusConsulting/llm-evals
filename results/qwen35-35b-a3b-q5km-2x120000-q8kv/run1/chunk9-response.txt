SC1-A
The pods are failing due to a conflict between the **Restricted Pod Security Standard (PSS)** and the container image configuration.
1.  **PSS Restricted:** Requires `runAsNonRoot: true` and forbids running as root (UID 0). It also requires explicit `runAsUser` if `runAsNonRoot` is true.
2.  **Image Behavior:** The new base image (`cgr.dev/chainguard/static:latest`) defaults `USER` to root (UID 0).
3.  **Manifest Behavior:** The pods have `securityContext.runAsNonRoot: true` but **no** `runAsUser` set.
4.  **The Failure:** When the kubelet attempts to start the container, it sees `runAsNonRoot: true`. It checks the image's default user. Since the image defaults to root (UID 0), the kubelet attempts to run as root. This violates the Restricted PSS policy (which forbids UID 0). Kubernetes rejects the pod start, causing a CrashLoopBackOff.
5.  **Baseline vs. Restricted:** Baseline allows root users and privileged containers. Restricted enforces non-root users and drops capabilities. The upgrade changed the namespace policy from Baseline to Restricted, exposing the implicit root user in the new images.

SC1-C
**Remediation Strategy:**
1.  **Immediate Fix (Image):** Rebuild the Go/Python images with a non-root user. Add `USER 1000` (or similar) to the Dockerfile.
2.  **Manifest Fix:** Update the Deployment manifests to explicitly set `securityContext.runAsUser: 1000` to ensure determinism.
3.  **CI Gate:** Add an OPA Gatekeeper or Kyverno policy in CI that scans Helm charts/Deployments. Policy: "If `runAsNonRoot` is true, `runAsUser` must be explicitly set and non-zero."
4.  **Rollout:** Use a **Canary Deployment**. Deploy the new images to 5% of pods first. Monitor CrashLoopBackOff and error rates. If stable, ramp to 100%. This prevents a full outage if the new image has other issues.

SC2-A
**Stale Data & Replication:**
1.  **Cause:** Cross-region replication is typically **asynchronous**. The primary writes WAL, ships it to the replica, and the replica applies it. If the primary fails before WAL ships, that data is lost.
2.  **Lag:** Promoting a replica makes it the new primary. Any data written to the old primary after the last WAL shipped to the replica is lost.
3.  **Data Loss Window:** Check `pg_stat_replication` on the old primary (if accessible) or `pg_last_wal_receive_lsn` vs `pg_last_wal_replay_lsn` on the replica. The difference indicates the lag.
4.  **Quantification:** Query `SELECT pg_wal_lsn_diff(pg_current_wal_lsn(), pg_last_wal_replay_lsn()) AS lag_bytes FROM pg_stat_replication;` (on primary) or check replica lag metrics.

SC2-B
**Terraform Operations:**
1.  **Import Replica:** `terraform import aws_db_instance.replica <replica_instance_arn>`
2.  **Update State:** The old instance ARN is hardcoded. You must update the Terraform configuration to reference the new instance ARN.
3.  **Detach Old:** `terraform state rm aws_db_instance.old_primary` (after ensuring it's decommissioned).
4.  **Update Dependencies:** Run `terraform plan` to ensure dependent resources (SGs, Route53) update to the new ARN. If they reference the old ARN directly in code, update the code to use the new ARN variable.
5.  **Apply:** `terraform apply` to reconcile state.

SC2-C
**Architecture Design:**
1.  **RDS Proxy:** Use RDS Proxy instead of PgBouncer. It handles connection pooling at the AWS layer, reducing connection churn during failover.
2.  **Route53:** Use **Route53 Failover Routing** with health checks. The primary endpoint is active; the replica is secondary. If health check fails, DNS TTL propagates the new IP.
3.  **Retry Logic:** Implement exponential backoff with jitter in the application.
4.  **RPO < 1s:** Standard RDS async replication cannot guarantee <1s RPO. **Recommend Aurora Global Database.** It uses a dedicated replication layer with synchronous commit options and lower latency replication (typically <1s RPO).
5.  **Why Aurora:** Native cross-region replication, faster failover (minutes vs hours), and better consistency guarantees than standard RDS async.

SC4-A
**Technical Blockers:**
1.  **Local Sessions:** Kubernetes pods are ephemeral. **Solution:** Move sessions to Redis (ElastiCache).
2.  **Local NFS Mount:** Pods cannot share local filesystems easily. **Solution:** Use EFS or NFS CSI driver with RWX access.
3.  **On-Prem Oracle:** Network connectivity and licensing. **Solution:** Oracle Cloud or RDS for Oracle, or keep on-prem but use secure tunneling (Direct Connect).
4.  **IBM MQ:** Legacy protocol. **Solution:** Use MQ Bridge or migrate to RabbitMQ/Kafka if possible.
5.  **Sticky Sessions:** F5 relies on IP affinity. **Solution:** Use Kubernetes Service (ClusterIP) with Session Affinity or move logic to Redis.

SC6-C
**Comparison:**
1.  **Direct Connect:** High bandwidth, low latency, dedicated. Costly. Fits Zero Trust (private link).
2.  **ECMP VPN:** Cheaper, shared bandwidth. Encryption overhead. Failover is automatic but slower.
3.  **Mesh (Tailscale/Netbird):** Software-defined, identity-based. Easier for Zero Trust (no IP trust). Lower bandwidth than DX.
**Recommendation:** **Direct Connect** for core data center traffic (bandwidth). **Mesh** for developer access and micro-segmentation.
**Routing:** DX requires BGP. Mesh requires agent routing.

SC8-C

2.  **Logs:** Loki in **Microservices Mode**. Separate `ingester`, `querier`, `index`. Scale read/write independently.

SC10-A
**Architecture:**
1.  **Active-Passive (Velero):** RPO = Backup frequency (15m). RTO = Restore time. 200TB restore is slow. **Fails RTO.**
2.  **Active-Active (Istio):** RPO = Replication lag. RTO = DNS failover.
3.  **Recommendation:** **Active-Passive with Async Replication.**
4.  **Replication:**
    *   **Postgres:** Patroni Async (RPO < 15m).
    *   **Redis:** Sentinel (RPO ~0).
    *   **RabbitMQ:** Federation (RPO ~0).
    *   **MinIO:** Site Replication (Async).
5.  **Latency:** 12ms is acceptable for async replication. Sync replication would degrade performance.

SC10-C
**Failover Orchestration:**
1.  **Detection:** Prometheus monitors site health. If 2 consecutive checks fail, trigger alert.
2.  **DNS:** GSLB (Route53) detects health check failure. Switches DNS to Dallas.
3.  **DB:** Patroni detects primary down. Promotes standby (Dallas).
4.  **Cache/Queue:** Redis Sentinel promotes master. RabbitMQ federation syncs.
5.  **Storage:** MinIO Site Replication catches up.
6.  **State:** Ansible/Terraform updates state to mark Dallas as Primary.
7.  **Split-Brain:** Use **Etcd Quorum**. Only one Patroni leader can be elected.
8.  **Failback:** Once Chicago is up, re-sync data. Promote Chicago. Update DNS.
**RPO/RTO:** 12ms latency allows async replication to stay within 15m RPO. DNS TTL + Failover time must be < 1 hour RTO.
