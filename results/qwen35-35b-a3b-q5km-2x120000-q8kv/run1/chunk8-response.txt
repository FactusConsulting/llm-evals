CL1 — **IaaS (Infrastructure as a Service):** Provides virtualized computing resources (VMs, storage, networks). *Example: AWS EC2.* **PaaS (Platform as a Service):** Provides hardware and software tools (runtime, DB, middleware) for developers. *Example: Google App Engine.* **SaaS (Software as a Service):** Delivers software applications over the internet. *Example: Microsoft 365.*

CL2 — **Shared Responsibility Model:** Provider secures the cloud (physical, network, hypervisor); Customer secures what is *in* the cloud. **IaaS VM:** Provider = Physical/Hypervisor; Customer = OS, Patching, Data, Firewall. **Managed K8s:** Provider = Control Plane, Nodes; Customer = Workloads, Network Policies, Secrets. **SaaS (M365):** Provider = App/Infra; Customer = Data, User Access, Device Security.

CL3 — **VPC:** Isolated virtual network. **Subnet:** IP range within VPC (public/private). **Security Group:** Instance-level firewall (stateful). **NACL:** Subnet-level firewall (stateless). **Layering:** VPC contains Subnets; Subnets contain Instances protected by SGs; NACLs filter traffic entering/leaving Subnets. **Stateful:** Allows return traffic automatically (SG). **Stateless:** Requires explicit rules for return traffic (NACL).

CL4 — **Landing Zone:** Secure, compliant baseline environment. **OUs:** Logical grouping for billing/policy. **Guardrails:** Preventative controls (e.g., no public S3 buckets). **Shared Networking:** Central hub (Transit Gateway) connects VPCs for traffic flow. **Centralized Logging:** Aggregates logs for security monitoring. **Single-Account Problems:** High blast radius, billing confusion, permission sprawl, lack of isolation between teams/environments.

CL5 — **IaC:** Managing infrastructure via code. **Terraform/OpenTofu:** Declarative, multi-cloud, state file management. **Pulumi:** Imperative/Declarative using general languages (TS/Python). **CloudFormation/Bicep:** Native AWS/Azure, tight integration, less portable. **Crossplane:** K8s API for managing cloud resources. **Choice:** Terraform for multi-cloud portability; Native for cost/depth; Pulumi for dev familiarity; Crossplane for K8s-centric ops.

CL6 — **HPA:** Scales pods based on CPU/Memory. **VPA:** Adjusts pod resource requests/limits. **Cluster Autoscaler:** Adds/removes nodes based on pending pods. **KEDA:** Event-driven scaling (queues, Kafka). **Use KEDA over HPA:** When scaling based on custom metrics (e.g., queue depth) or scaling to zero (HPA usually requires min replicas).

CL7 — **DR Strategy:** Define RPO (data loss tolerance) and RTO (downtime tolerance). **Active-Active:** Both regions serve traffic (low RTO, high cost). **Active-Passive:** One region active (higher RTO, lower cost). **DB:** Async replication (lower latency, potential data loss) vs Sync (strong consistency, latency). **DNS:** Route53 Health Checks trigger failover. **Pilot Light:** DB running, app off. **Warm Standby:** App running scaled down. **Active-Active:** Full redundancy.

CL8 — **GitOps:** Git repository is source of truth; controllers sync cluster state to Git state. **ArgoCD:** Push/Pull model, UI focused. **Flux:** Pull model, K8s CRD focused. **Secrets:** Sealed Secrets (encrypted in Git), External Secrets Operator (fetch from Vault/AWS Secrets Manager). **Drift:** Reconciliation loop detects manual changes and reverts them to match Git.

CL9 — **Architecture:** Use **StatefulSet** for stable network IDs and ordered deployment. **PV/PVC:** Use `local` storage class for low latency (binds to specific node). **Headless Service:** Provides DNS for StatefulSet pods. **Init Containers:** Handle config file generation before main app starts. **ConfigMaps:** Mount as volumes for config files. **PDB:** Ensure minimum availability during updates.

CL10 — **SA:** K8s identity for pods. **Workload Identity:** Maps SA to Cloud IAM identity. **IRSA:** AWS specific implementation of Workload Identity. **Mounting Creds:** Bad practice (risk of leakage, rotation difficulty). **Federation:** Uses OIDC tokens to exchange SA identity for temporary cloud credentials without long-lived keys.

CL11 — **Object Storage (S3):** Unstructured data, high durability (11 9s), low latency. **Block Storage (EBS):** Attached to VM, low latency, high IOPS. **File Storage (EFS):** Shared file system, NFS/SMB. **Use Case:** Object for backups/media, Block for DB/OS, File for shared app configs. **Durability:** All high (11 9s+), **Availability:** Varies by tier (e.g., S3 Standard vs Glacier).

CL12 — **Zero Trust:** Verify every request, never trust by default. **Components:** Network Policies (K8s), Service Mesh (mTLS), Pod Identity (RBAC), Admission Controllers (OPA/Kyverno), Image Signing (Cosign), Runtime Security (Falco). **Difference:** Traditional relies on perimeter (firewall); Zero Trust assumes breach and verifies identity/context for every interaction.

CL13 — **CDN:** Distributed network of servers caching content at edge. **Mechanism:** Caches static assets closer to users. **Services:** AWS CloudFront, Cloudflare. **Use CDN:** For static content (images, JS) to reduce latency and origin load. **Direct Origin:** Only for dynamic, personalized, or sensitive content not suitable for caching.

CL14 — **Region:** Geographic area (e.g., us-east-1). **AZ:** Isolated data center within a Region. **Multi-AZ:** Protects against data center failure. **Zonal Resources:** Bound to one AZ (e.g., EBS). **Regional Resources:** Span multiple AZs (e.g., ELB, RDS Multi-AZ).

CL15 — **Managed DB:** Service where provider manages hardware, OS, DB software. **Provider Tasks:** Patching, backups, failover, scaling, monitoring. **Customer Tasks:** Schema design, queries, access control. **Benefit:** Reduced operational overhead, higher availability SLAs.

CL16 — **IAM:** **Users:** Human identities. **Groups:** Collections of users. **Roles:** Temporary credentials for entities. **Policies:** JSON permissions. **Least Privilege:** Grant only necessary permissions. **Application:** Use roles for services, groups for users, deny-by-default policies, and regular access reviews.

CL17 — **Serverless:** Event-driven execution without server management. **Comparison:** Lambda (AWS), Functions (Azure), Cloud Functions (GCP). **Cold Starts:** Delay when function initializes (memory allocation). **Poor Fit:** Long-running processes, stateful apps, predictable high-throughput workloads where provisioned capacity is cheaper.

CL18 — **Ingress:** K8s resource for HTTP/HTTPS routing. **Gateway API:** Standardized, multi-protocol, better traffic splitting. **LB Annotations:** Cloud-specific load balancer config. **Mesh Gateway:** Sidecar-based ingress. **TLS:** Terminates at LB (Cloud), Ingress Controller, or Mesh. **Use Gateway API:** For complex routing, multi-cluster, or protocol diversity (gRPC, TCP).

CL19 — **Strategy:** **RI/SP:** Commit for 1-3 years for predictable workloads. **Spot:** Use for fault-tolerant/batch workloads. **Right-sizing:** Match instance to actual usage. **Tiering:** Move old data to Glacier. **FinOps:** Culture of cost accountability. **Tools:** AWS Cost Explorer, CloudHealth. **Showback:** Report costs to teams. **Chargeback:** Bill teams directly.

CL20 — **Strategy:** **Abstraction:** Kubernetes, Terraform. **Networking:** Direct Connect/ExpressRoute. **Identity:** OIDC Federation. **Trade-offs:** Multi-cloud increases complexity, networking costs, and skill requirements. **Justified:** Regulatory compliance, catastrophic redundancy, vendor negotiation leverage. **Unnecessary:** If single cloud meets all needs, multi-cloud adds risk without benefit.

OT1 — **Purdue Model:** ISA-95 standard for IT/OT integration. **L0:** Physical Process (Sensors/Actuators). **L1:** Basic Control (PLCs/Relays). **L2:** Area Supervisory (SCADA/HMI). **L3:** Site Operations (MES/Historian). **L4:** Enterprise (ERP/Business).

OT2 — **IT:** Data, Information, Security focused. **OT:** Control, Physical Process, Safety focused. **Patching:** IT patches frequently; OT patches rarely due to stability requirements, vendor support cycles, and risk of downtime affecting physical processes.

OT3 — **DMZ (IDMZ):** Buffer zone between IT and OT. **Traffic:** Historian pulls data from SCADA (L3 to L2). Engineering pushes configs to DMZ. **Direct L4 to L1:** Dangerous because it bypasses segmentation, allowing malware to move from corporate network directly to control logic.

OT4 — **Modbus TCP:** L1/L2, Master/Slave, simple. **OPC UA:** L2/L3, Secure, platform-independent. **EtherNet/IP:** L1/L2, CIP protocol, Allen-Bradley dominant. **PROFINET:** L1/L2, Real-time Ethernet, Siemens dominant. **MQTT:** L2/L3/Cloud, Lightweight, Pub/Sub.

OT5 — **Data Diode:** Hardware allowing one-way data flow. **Firewall:** Two-way filtering. **Use Diode:** When exporting data from secure OT to IT is required but IT must never initiate connection to OT. **Limitation:** Cannot send commands back to OT; no bidirectional control.

OT6 — **Flow:** PLC -> Historian -> OPC UA Server -> Edge Gateway -> MQTT Broker -> Cloud. **Security Boundaries:** Firewall between Edge and Cloud (DMZ). Encrypted MQTT (TLS). OT network segmented from IT. Edge device acts as security gateway.

OT7 — **IEC 62443:** Security standard for ICS. **Zone:** Logical grouping of assets with similar security requirements. **Conduit:** Communication path between zones. **Model:** Corporate (L4) -> DMZ -> SCADA (L3) -> Engineering (L3) -> PLC (L1). Conduits enforce traffic rules between zones.

OT8 — **Segmentation:** VLANs per production line. **Firewall:** Between VLANs and DMZ. **Switch:** Managed Layer 2/3 switches with ACLs. **Remote Access:** Jump host with MFA and session recording. **Transition:** Parallel run (new network alongside old), phased migration during maintenance windows, rollback plan.

OT9 — **SIS:** Independent safety system (stops process safely). **BPCS:** Basic Process Control System (maintains process). **IEC 61511:** Safety lifecycle standard. **Air-gap:** Prevents cyber attacks from disabling safety functions. **Ransomware on SIS:** Physical danger, potential injury, environmental release, inability to shut down safely.

OT10 — **PLC:** Localized logic controller. **DCS:** Distributed control for continuous processes. **SCADA:** Supervisory monitoring and control. **Coexist:** Yes, PLCs often feed into SCADA/DCS. **Use:** PLC for discrete manufacturing, DCS for continuous (chemical/pipeline), SCADA for wide-area monitoring.

OT11 — **Architecture:** Jump host (bastion) in DMZ. **MFA:** Required for all access. **Recording:** Session recording for audit. **Vendor:** Time-limited, scoped access. **No Direct RDP:** Prevents lateral movement from corporate network to critical OT assets.

OT12 — **Plan:** **Monitor:** IDS for OT protocols (e.g., Nozomi, Claroty). **Inventory:** Identify compromised assets. **IR:** Isolate network segments, preserve logs. **Ops:** Coordinate with control room for manual override. **Operator Role:** Prioritize physical safety, maintain process stability, follow emergency procedures, do not reset systems without authorization.

OT13 — **HMI:** Interface for operators to view and control process. **Role:** Visualization and manual intervention. **Engineering Workstation:** Used by engineers to configure logic, download code, and troubleshoot. **Difference:** HMI is for operation; EWS is for maintenance/configuration.

OT14 — **Historian:** Time-series database for process data. **Importance:** Trend analysis, compliance, troubleshooting. **Plant Historian (L3):** Real-time, high frequency, local. **Enterprise Historian (L4):** Aggregated, lower frequency, for business analytics.

OT15 — **Ladder Logic:** Graphical programming language resembling relay logic. **Device:** PLC. **Dominance:** Deterministic execution, safety certification, legacy familiarity, ease of troubleshooting for electricians.

OT16 — **RTU:** Remote Terminal Unit. **Difference:** RTU optimized for comms and harsh environments (long distance); PLC optimized for local control speed. **Deploy RTU:** Pipelines, water distribution, remote sites with limited connectivity.

OT17 — **Analog:** Continuous signal (voltage/current). **Digital:** Discrete (On/Off). **4-20mA:** Preferred over 0-20mA because 4mA indicates a live wire (fail-safe); 0mA indicates a broken wire or fault.

OT18 — **Inventory:** Knowing all assets is critical for vulnerability management. **Tools:** Passive (Network TAPs, Sniffers), Active (Scanners). **Active Scanning:** Dangerous because legacy OT devices may crash or lock up when probed with unexpected traffic.

OT19 — **MITRE ICS:** Framework for ICS threats. **Tactics:** Reconnaissance (Scanning), Initial Access (Phishing), Execution (Malware), Persistence (Backdoor), Impact (Disruption). **Difference:** Focuses on physical impact and safety, not just data theft. **Usage:** Map defenses to tactics to find gaps (e.g., no network segmentation for Execution).

OT20 — **SOC:** **Logs:** Syslog agents (non-intrusive), SIEM integration. **Rules:** OT-specific (e.g., PLC write attempts). **Staffing:** IT analysts + OT Engineers. **False Positives:** Tuned to OT baselines. **Firmware Change:** Alert triggers immediate investigation, isolate PLC, verify with engineering, rollback if unauthorized, patch if needed.
