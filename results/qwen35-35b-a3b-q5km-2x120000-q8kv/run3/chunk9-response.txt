SC1-A
The pods are failing due to a conflict between the **Restricted Pod Security Standard (PSS)** and the container image configuration.
1.  **Restricted PSS:** Requires `runAsNonRoot: true` and forbids `runAsUser: 0` (root). It also restricts capabilities and volumes.
2.  **Image Configuration:** The new base image (`cgr.dev/chainguard/static:latest`) defaults `USER` to `root` (UID 0).
3.  **Manifest Configuration:** The pods have `securityContext.runAsNonRoot: true` but lack an explicit `securityContext.runAsUser`.
4.  **The Failure:** Kubernetes attempts to run the container as the image's default user (root). The Restricted PSS admission controller rejects this because `runAsUser` would be 0, violating the "non-root" requirement. The pod enters `CrashLoopBackOff` because the kubelet cannot start the container due to the security policy violation.
5.  **Baseline vs. Restricted:** Baseline allows root users and most capabilities. Restricted enforces non-root, drops capabilities, and forbids privileged containers. The upgrade enforced the stricter policy, exposing the misconfiguration.

SC1-C
**Remediation Strategy:**
1.  **Image Fix:** Rebuild the base image or override the `USER` directive in the Dockerfile to a non-root UID (e.g., `USER 1000`).
2.  **Manifest Fix:** Explicitly set `securityContext.runAsUser: 1000` in the Deployment spec to ensure consistency regardless of image defaults.
3.  **CI Gate:** Integrate a PSS scanner (e.g., `kube-score` or `checkov`) into the CI pipeline. Fail the build if the manifest violates the target PSS level (Restricted) or if the image is not scanned for root users.
4.  **Rollout:** Use a **Canary Deployment**. Deploy the fixed version to 5% of traffic first. Monitor CrashLoopBackOff rates and error logs. If stable, gradually increase to 100%. This ensures immediate rollback capability if the fix introduces new issues.

SC2-A
**Stale Data & Replication:**
1.  **Cause:** Cross-region read replicas use **asynchronous streaming replication**. WAL (Write-Ahead Log) shipping is not guaranteed to be instant.
2.  **Lag:** During the failover, the promoted replica may have missed recent WAL segments generated in us-east-1 before the network cut.
3.  **Data Loss:** Asynchronous replication allows data loss if the primary fails before WAL is shipped. The "data loss window" is the time between the last committed transaction on the primary and the last shipped WAL on the replica.
4.  **Quantification:** Check `pg_stat_replication` on the new primary (formerly replica) for `pg_wal_lsn_diff` (lag in bytes) and `pg_last_xact_replay_timestamp`. Compare the `pg_current_wal_lsn()` on the old primary (if accessible) vs the new primary.

SC2-B
1.  **Import Replica:** `terraform import aws_db_instance.promoted_replica <new_instance_arn>`
2.  **Update State:** Use `terraform state mv` to move the old instance resource to the new ARN if the resource type matches, or create a new resource and import it.
3.  **Detach Old:** `terraform taint aws_db_instance.old_primary` (to force destroy) or manually delete the old instance in AWS console after confirming data is safe.
4.  **Update Deps:** Run `terraform plan` to verify dependent resources (SGs, Route53) reference the new ARN. If ARNs are hardcoded in code, update the Terraform variables and run `terraform apply`.
5.  **Command:** `terraform state mv 'aws_db_instance.old_primary' 'aws_db_instance.promoted_replica'` (if renaming resource).

SC2-C
**Architecture Design:**
1.  **RDS Proxy:** Use RDS Proxy instead of PgBouncer. It handles connection pooling at the AWS layer, supports IAM auth, and retries transient failures automatically.
2.  **Route53:** Configure **Health Checks** on the primary endpoint. Use **Failover Routing Policy** to switch DNS to the secondary region upon failure detection.
3.  **Retry Logic:** Implement exponential backoff with jitter in the application connection pool.
4.  **RPO < 1s:** Standard RDS async replication cannot guarantee <1s RPO. **Recommend Aurora Global Database.** It uses a single storage volume replicated synchronously across regions (125ms latency) with a dedicated replication layer, ensuring near-zero data loss and sub-second failover.
5.  **Failover:** Aurora Global Database handles promotion automatically via the control plane, reducing manual intervention.

SC3-C

1.  **Image Signing:** Use **Cosign** to sign images upon build. `cosign sign --key cosign.key <image>`.
2.  **Admission Control:** Deploy **Kyverno** or **OPA Gatekeeper** to enforce signature verification on pod creation. Reject unsigned images.

2.  **Local NFS Mount:** `/data/uploads` on bare metal. **Solution:** Use Kubernetes PersistentVolumes (PV) with NFS CSI or S3.

5.  **Stateful Monolith:** Single binary scaling. **Solution:** Horizontal Pod Autoscaler (HPA) requires stateless app. Refactor to stateless or use StatefulSets if state is required.

SC5-C
**Architectural Solution:**
1.  **CQRS:** Separate Read (Checkout) and Write (Reconciliation) models. Reconciliation writes to a separate table or replica.
2.  **Read Replica:** Run the reconciliation CronJob against a **PostgreSQL Read Replica**, not the primary.
3.  **Bulkhead:** Configure separate connection pools for the CronJob and the API service in the application config.
4.  **Scheduling:** Stagger CronJob execution (e.g., 14:05) to avoid peak load.
5.  **PDBs:** Define `PodDisruptionBudget` for the Inventory Service to ensure at least 1 pod remains available during voluntary disruptions (like CronJob scaling).
6.  **Resource Quotas:** Limit CPU/Memory for the CronJob namespace to prevent starvation of serving pods.

SC6-C
**Upgrade Comparison:**
1.  **AWS Direct Connect:**
    *   **Pros:** Dedicated bandwidth (10Gbps+), lower latency, consistent performance.
    *   **Cons:** High cost, hardware provisioning.
    *   **Zero-Trust:** Fits well (private link).
2.  **Second VPN (ECMP):**
    *   **Pros:** Low cost, quick setup.
    *   **Cons:** Shared bandwidth, higher latency, encryption overhead.
    *   **Zero-Trust:** Fits (IPsec).
3.  **Mesh (Tailscale/Netbird):**
    *   **Pros:** Zero-config, identity-based, encrypted.
    *   **Cons:** Not suitable for high-throughput data center traffic (10Gbps).
    *   **Zero-Trust:** Best fit for identity, worst for bandwidth.
**Recommendation:** Upgrade to **Direct Connect** for the core data plane to meet bandwidth needs. Use **SPIRE/Zero-Trust** for the control plane.

**Failover Orchestration:**

2.  **DNS:** GSLB (Route53) switches traffic to Dallas.
3.  **Patroni:** Patroni detects primary down, promotes Dallas node (Synchronous commit ensures no data loss).
4.  **Redis:** Sentinel promotes replica.
