CL1 — **IaaS, PaaS, SaaS:**
*   **IaaS (Infrastructure as a Service):** Provides virtualized computing resources over the internet. You manage OS, middleware, runtime, data, and apps. *Example:* AWS EC2.
*   **PaaS (Platform as a Service):** Provides hardware and software tools (usually for app development) hosted on the provider's infrastructure. You manage data and apps. *Example:* Google App Engine.
*   **SaaS (Software as a Service):** Fully managed software applications delivered over the internet. You manage nothing but user access. *Example:* Salesforce.

CL2 — **Shared Responsibility Model:**
*   **Concept:** Security is shared between provider (cloud) and customer (user).
*   **IaaS VM:** Provider secures physical hardware, hypervisor, and network infrastructure. Customer secures OS, patching, firewall config, data, and IAM.
*   **Managed Kubernetes:** Provider secures control plane, nodes, and underlying infrastructure. Customer secures node OS (if self-managed), pod security, network policies, and application code.
*   **SaaS (M365):** Provider secures infrastructure, app, and data center. Customer secures identity, access management, data classification, and device compliance.

CL3 — **Networking Components:**
*   **VPC:** Isolated virtual network in the cloud.
*   **Subnet:** Segment of a VPC IP range (public/private).
*   **Security Group:** Stateful virtual firewall attached to instances (allow rules only).
*   **NACL:** Stateless virtual firewall at subnet level (allow/deny rules).
*   **Layering:** VPC contains Subnets; Instances in Subnets have Security Groups; Traffic entering/leaving Subnets hits NACLs.
*   **Stateful vs. Stateless:** Security Groups are stateful (return traffic allowed automatically). NACLs are stateless (must explicitly allow return traffic).

CL4 — **Multi-Account Strategy:**
*   **Landing Zone:** Pre-configured environment establishing baseline security, networking, and identity.
*   **Organizational Units (OUs):** Logical grouping of accounts (e.g., Prod, Dev, Security).
*   **Guardrails:** SCPs (Service Control Policies) to prevent risky actions (e.g., disabling CloudTrail).
*   **Shared Networking:** Transit Gateway (AWS) or Hub-and-Spoke (Azure) connects accounts centrally for traffic flow.
*   **Centralized Logging:** Aggregates logs from all accounts to a dedicated Security/Log account.
*   **Single-Account Problem:** High blast radius if compromised, no isolation between teams, difficult compliance auditing, resource contention.

CL5 — **Infrastructure as Code (IaC):**
*   **Terraform/OpenTofu:** Declarative, multi-cloud, state-file based. Best for general-purpose, multi-cloud, or complex provisioning.
*   **Pulumi:** Imperative/Declarative, uses general languages (TS, Python). Best for teams wanting code flexibility.
*   **CloudFormation/Bicep:** Native to AWS/Azure. Best for deep integration with specific provider features, CI/CD tight coupling.
*   **Crossplane:** Kubernetes-native IaC. Best for managing cloud resources *from* within a K8s cluster.

CL6 — **Autoscaling Patterns:**
*   **HPA:** Scales pods based on CPU/Memory metrics.
*   **VPA:** Adjusts resource requests/limits for pods (requires restart).
*   **Cluster Autoscaler:** Adds/removes nodes based on pending pods.
*   **KEDA:** Scales based on external events (e.g., Kafka queue depth, AWS SQS).
*   **KEDA vs. HPA:** Use KEDA when scaling needs to be driven by non-standard metrics (e.g., message queue length) or event-driven workloads, not just CPU.

CL7 — **Disaster Recovery Strategy:**
*   **RPO/RTO:** RPO (data loss tolerance) dictates replication sync (sync for low RPO). RTO (downtime tolerance) dictates failover speed.
*   **Active-Active:** Both regions serve traffic. High cost, low RTO/RPO.
*   **Active-Passive:** One region active, one standby. Lower cost, higher RTO.
*   **Database:** Sync replication (synchronous) for zero data loss (Active-Active); Async for Active-Passive.
*   **DNS:** Route53 Health Checks + Failover Routing Policy.
*   **State:** Externalize state (S3, DynamoDB) rather than local disk.
*   **DR Types:**
    *   *Pilot Light:* Core DB running, app code ready to deploy.
    *   *Warm Standby:* Scaled-down version of prod running.
    *   *Multi-Site Active-Active:* Full production in both regions.

CL8 — **GitOps Model:**
*   **Concept:** Git repository is the single source of truth. Controllers sync cluster state to Git state.
*   **ArgoCD vs. Flux:** ArgoCD has a UI, Flux is CLI-first. Both reconcile state.
*   **Secrets:** Sealed Secrets (encrypted in Git), External Secrets Operator (fetches from Vault/AWS Secrets Manager), Vault Agent.
*   **Drift:** If manual change occurs, GitOps controller detects drift and reverts the cluster to match Git state automatically.

CL9 — **Stateful Legacy App to K8s:**
*   **StatefulSet:** Ensures stable pod identity and ordered deployment/termination.
*   **PV/PVC:** Use `local` storage class for low latency (node-local disk).
*   **Headless Service:** Provides stable DNS names for StatefulSet pods (no load balancing).
*   **Init Containers:** Handle config file generation or DB initialization before main app starts.
*   **ConfigMaps:** Mount as volumes at specific paths required by legacy app.
*   **PDB:** Ensure minimum available pods during updates to prevent service outage.

CL10 — **Identity (SA vs. Workload vs. IRSA):**
*   **Service Account:** K8s native identity for pods.
*   **Workload Identity:** GCP/Azure native mapping of K8s SA to cloud IAM.
*   **IRSA:** AWS mapping of K8s SA to IAM Role via OIDC.
*   **Mounting Credentials:** Bad practice (leaks secrets, hard to rotate, violates least privilege).
*   **Federation:** Uses OIDC tokens to exchange K8s identity for temporary cloud credentials without long-lived secrets.

CL11 — **Storage Types:**
*   **Object (S3/Blob):** Unstructured data, HTTP access. High durability (11 9s), high availability. Use for backups, media, logs.
*   **Block (EBS/Disk):** Attached to VM, file system level. High IOPS, low latency. Use for OS, DBs.
*   **File (EFS/Azure Files):** Network file system (NFS/SMB). Shared access. Use for shared configs, legacy apps.
*   **Durability:** Object is highest (replicated across zones). Block is zone-reliant (unless multi-attach). File is zone-reliant.

CL12 — **Zero-Trust K8s:**
*   **Network Policies:** Deny-all default, allow specific pod-to-pod traffic.
*   **Service Mesh (mTLS):** Encrypts service-to-service traffic (Istio/Linkerd).
*   **Pod Identity:** Workload identity (IRSA/Workload Identity) for cloud access.
*   **Admission Controllers:** OPA/Kyverno to enforce policies (e.g., no root users).
*   **Image Signing:** Cosign/Notary to verify image integrity.
*   **Runtime Security:** Falco to detect anomalies (e.g., shell in container).
*   **Secrets:** External Secrets Operator (Vault).
*   **Difference:** Traditional uses perimeter firewall; Zero Trust assumes breach, verifies every request internally.

CL13 — **CDN:**
*   **Definition:** Distributed network of servers caching content closer to users.
*   **How:** Caches static assets at edge locations; reduces origin load and latency.
*   **Services:** AWS CloudFront, Azure CDN.
*   **Use CDN:** For static content (images, JS, CSS), high traffic, global users.
*   **Use Origin:** For dynamic, personalized, or sensitive data requiring real-time processing.

CL14 — **AZ vs. Region:**
*   **Region:** Geographic area (e.g., us-east-1).
*   **Availability Zone (AZ):** Isolated data centers within a Region (power, network, cooling).
*   **Multi-AZ:** Deploy across AZs to survive a data center failure.
*   **Zonal:** Resources tied to one AZ (e.g., EBS).
*   **Regional:** Resources replicated across AZs (e.g., RDS Multi-AZ, S3).

CL15 — **Managed Database:**
*   **Definition:** DB service where provider manages hardware, OS, and DB software.
*   **Provider Tasks:** Patching, backups, failover, scaling, monitoring, maintenance windows.
*   **Customer Tasks:** Schema, queries, data, access control, parameter tuning.

CL16 — **IAM & Least Privilege:**
*   **Users:** Human identities.
*   **Groups:** Collections of users for permission assignment.
*   **Roles:** Temporary credentials for services/users to assume.
*   **Policies:** JSON permissions defining actions/resources.
*   **Least Privilege:** Grant only necessary permissions. Apply by using specific resource ARNs and denying by default, avoiding `*` actions.

CL17 — **Serverless:**
*   **Definition:** Event-driven compute without managing servers (AWS Lambda, Azure Functions, GCP Functions).
*   **Cold Starts:** Latency when a function is invoked after inactivity (initialization overhead).
*   **Poor Fit:** Long-running processes, high stateful requirements, strict latency needs (cold start), heavy CPU usage.

CL18 — **Ingress Options:**
*   **Ingress Resource:** Standard K8s resource, requires controller (Nginx, ALB). TLS terminates at controller.
*   **Gateway API:** Newer standard, more flexible routing, supports multiple gateways.
*   **Cloud LB Annotations:** Direct cloud LB (ALB/NLB) integration. TLS terminates at LB.
*   **Service Mesh:** Istio Ingress Gateway. TLS terminates at mesh sidecar.
*   **Gateway API:** Use when needing advanced routing (header-based, multiple gateways) or standardizing across providers.

CL19 — **Cost Optimization ($80k/mo):**
*   **RI/SP:** Commit to 1-3 years for baseline steady-state workloads (up to 72% savings).
*   **Spot/Preemptible:** For fault-tolerant, stateless workloads (up to 90% savings).
*   **Right-sizing:** Use tools (AWS Compute Optimizer) to match instance size to actual usage.
*   **Storage Tiering:** Move old data to Glacier/Archive.
*   **Idle Detection:** Tag and alert on unused EBS volumes, idle LBs.
*   **FinOps:** Cross-functional team, showback/chargeback to teams, budget alerts.
*   **Tools:** AWS Cost Explorer, CloudHealth, Kubecost.

CL20 — **Multi-Cloud Strategy:**
*   **Abstraction:** Kubernetes (K8s) as runtime, Terraform for IaC.
*   **Networking:** Direct Connect/ExpressRoute or Cloud Interconnect for low latency.
*   **Identity:** Federation (OIDC) to central IdP.
*   **Trade-offs:** Increased complexity, higher networking costs, harder compliance, vendor lock-in risk reduced but operational lock-in increased.
*   **Justified:** Regulatory requirements, avoiding single-vendor outage risk, best-of-breed services.
*   **Unnecessary:** If one provider meets all needs, multi-cloud adds cost without value.

OT1 — **Purdue Model:**
*   **Level 0:** Physical Process (Sensors, Actuators).
*   **Level 1:** Basic Control (PLCs, RTUs).
*   **Level 2:** Area Supervisory (SCADA, HMI).
*   **Level 3:** Site Operations (MES, Historian).
*   **Level 4:** Enterprise (ERP, Business).
*   *(Note: Level 5 is Enterprise, Level 0-1 is Field, Level 2-3 is Control).*

OT2 — **IT vs. OT:**
*   **IT:** Confidentiality/Integrity focus, frequent patching, standard protocols (TCP/IP).
*   **OT:** Availability/Safety focus, stability over security, proprietary protocols, long lifecycles.
*   **Patching:** OT patching requires downtime validation; IT patching is routine. OT systems often run unsupported OS (XP/7).

OT3 — **DMZ (IDMZ):**
*   **Role:** Buffer zone between IT (Level 4) and OT (Level 0-3).
*   **Traffic:** Unidirectional or strictly controlled bidirectional (e.g., DB sync from OT to IT).
*   **Danger:** Direct L4 to L1 allows attackers to bypass controls and directly manipulate physical processes.

OT4 — **OT Protocols:**
*   **Modbus TCP:** Level 1-2, Master/Slave, unencrypted.
*   **OPC UA:** Level 1-3, Secure, platform-independent, modern standard.
*   **EtherNet/IP (CIP):** Level 1-2, Allen-Bradley dominant.
*   **PROFINET:** Level 1-2, Siemens dominant, real-time.
*   **MQTT:** Level 2-3, Lightweight, publish/subscribe for cloud connectivity.

OT5 — **Data Diode vs. Firewall:**
*   **Data Diode:** Hardware allowing data flow in *one direction only* (physically).
*   **Firewall:** Software/Hardware filtering bidirectional traffic.
*   **Use Diode:** When moving data from OT to IT (e.g., Historian to Cloud) where OT must never receive commands from IT.
*   **Limitation:** Cannot allow bidirectional communication (e.g., remote configuration).

OT6 — **Data Flow Design:**
*   **Flow:** PLC -> OPC UA Server -> Edge Gateway -> MQTT Broker -> Cloud Ingestion.
*   **Security:** Edge Gateway acts as DMZ. Firewall between SCADA and Edge. TLS for MQTT.
*   **Boundary:** Place security boundary at the Edge Gateway (Level 2/3 boundary).

OT7 — **IEC 62443 Zones & Conduits:**
*   **Zone:** Group of assets with similar security requirements.
*   **Conduit:** Logical path for traffic between zones.
*   **Design:**
    *   *Zone 1:* Corporate Network (Level 4).
    *   *Zone 2:* Engineering Workstations (Level 3).
    *   *Zone 3:* SCADA Servers (Level 3).
    *   *Zone 4:* PLCs/SIS (Level 1-2).
    *   *Conduits:* Firewall between Z1-Z2, Z2-Z3, Z3-Z4.

OT8 — **Segmentation Strategy:**
*   **VLANs:** Segment by production line (VLAN 10, 20, 30, 40).
*   **Firewall:** Layer 3 firewall between VLANs and DMZ.
*   **Switch:** Managed switches with port security, disable unused ports.
*   **Remote Access:** Jump host in DMZ, MFA, session recording, time-bound tickets.
*   **Transition:** Phase migration during maintenance windows; parallel run old/new network; validate PLC comms before cutting over.

OT9 — **SIS vs. BPCS:**
*   **BPCS:** Basic Process Control (PLC/DCS) for normal operation.
*   **SIS:** Safety Instrumented System (Emergency Shutdown) for safety.
*   **IEC 61511:** Standard for Safety Instrumented Systems.
*   **Isolation:** SIS must be independent to ensure safety function works even if BPCS/IT is compromised.
*   **Ransomware:** If SIS is compromised, safety shutdowns may fail, leading to physical damage or injury.

OT10 — **PLC vs. DCS vs. SCADA:**
*   **PLC:** Discrete control, fast logic, Level 1.
*   **DCS:** Distributed control, continuous process, integrated I/O, Level 1-2.
*   **SCADA:** Supervisory, data collection, HMI, Level 2-3.
*   **Coexist:** Yes, PLCs often feed into DCS/SCADA.

OT11 — **Remote Access:**
*   **Architecture:** Jump Host in DMZ, MFA required, Session Recording (video/log).
*   **Vendor Access:** Time-limited credentials, specific IP whitelisting.
*   **Unacceptable:** Direct RDP to HMI exposes HMI to corporate network threats; HMI should not be directly internet-facing.

OT12 — **Incident Response (Nation-State):**
*   **Monitoring:** IDS on OT network (e.g., Nozomi, Claroty), anomaly detection on traffic.
*   **Inventory:** Know all assets to identify compromised ones.
*   **Steps:** Isolate network segment, preserve logs, notify control room.
*   **Coordination:** Operators prioritize safety; IT handles containment.
*   **Operator Role:** Monitor physical process, initiate manual shutdown if automated systems are untrusted.

OT13 — **HMI:**
*   **Definition:** Interface for operators to view/process data (Level 2).
*   **Role:** Visualization, manual control.
*   **Vs. Engineering Workstation:** HMI is for operations; Engineering Workstation is for programming/configuring PLCs (Level 2/3).

OT14 — **Historian:**
*   **Definition:** Time-series database for process data.
*   **Importance:** Trend analysis, compliance, troubleshooting.
*   **Plant (L3):** Real-time process data, high frequency.
*   **Enterprise (L4):** Aggregated data for business reporting, lower frequency.

OT15 — **Ladder Logic:**
*   **Definition:** Graphical programming language mimicking relay logic.
*   **Device:** PLC.
*   **Dominance:** Familiar to electricians, deterministic, robust, industry standard for decades.

OT16 — **RTU vs. PLC:**
*   **RTU:** Remote Terminal Unit, rugged, long-distance comms, low power.
*   **PLC:** Local control, high I/O density.
*   **RTU Use:** Pipelines, SCADA remote sites, harsh environments.

OT17 — **Analog vs. Digital:**
*   **Analog:** Continuous signal (4-20mA, 0-10V).
*   **Digital:** Discrete signal (0/1).
*   **4-20mA:** 4mA represents zero (live zero) allows detection of wire break (0mA = fault).

OT18 — **Asset Inventory:**
*   **Foundation:** You can't secure what you don't know.
*   **Tools:** Passive monitoring (packet capture), Active scanning (ping/scan).
*   **Risk:** Active scanning can crash legacy OT devices (unpatched OS). Use passive monitoring primarily.

OT19 — **MITRE ATT&CK for ICS:**
*   **Tactics:**
    1.  *Reconnaissance:* Scanning network.
    2.  *Initial Access:* Phishing vendor.
    3.  *Execution:* Running malicious script on HMI.
    4.  *Impact:* Stopping pumps (Denial of Service).
    5.  *Defense Evasion:* Disabling logging.
*   **Difference:** Focuses on physical impact and industrial protocols vs. enterprise data theft.
*   **Usage:** Map current defenses to tactics to find gaps (e.g., no network segmentation).

OT20 — **Converged SOC:**
*   **Log Collection:** Agents on OT assets (low impact), network taps.
*   **SIEM:** Centralized correlation (IT + OT).
*   **Rules:** OT-specific (e.g., unexpected PLC write).
*   **Staffing:** IT analysts for triage, OT engineers for validation.
*   **False Positives:** High in OT due to normal maintenance; tune rules to reduce noise.
*   **Firmware Alert:** Verify change request, isolate PLC, check backup, restore if unauthorized.
