AA1 — **Monolith vs. Microservices:** A monolith is a single, unified codebase deployed as one unit. Microservices are a collection of small, independent services communicating over a network.
**3 Problems Microservices Introduce:**
1.  **Distributed Complexity:** Debugging requires tracing across multiple services; network latency and partial failures are common.
2.  **Data Consistency:** Maintaining ACID transactions across services is difficult (requires patterns like Sagas).
3.  **Operational Overhead:** Requires robust CI/CD, service discovery, and monitoring infrastructure compared to a single deployable artifact.

AA2 — **Sync vs. Async:** Synchronous communication blocks the sender until the receiver responds (request/response). Asynchronous communication allows the sender to continue without waiting for an immediate response (fire-and-forget).
**Examples:** Synchronous: HTTP/REST or gRPC. Asynchronous: RabbitMQ (Message Queue) or Kafka (Event Log).

AA3 — **API Gateway vs. Reverse Proxy:** A reverse proxy routes traffic to backend servers (Layer 4/7 routing, SSL termination). An API Gateway adds business logic capabilities like authentication, rate limiting, API versioning, and analytics.
**Replacement:** No. An API Gateway is a specialized reverse proxy. A standard reverse proxy lacks the specific API management features (e.g., OAuth validation, quota enforcement) required for public-facing APIs.

AA4 — **Saga Pattern:** A sequence of local transactions where each updates data within a single service. If a step fails, compensating transactions undo previous changes.
**Choreography vs. Orchestration:** Choreography uses events to trigger the next step (decentralized). Orchestration uses a central coordinator service to tell services what to do (centralized).
**Appropriateness:** Choreography for simple, loosely coupled flows. Orchestration for complex workflows requiring visibility, error handling, and state management.

AA5 — **CQRS:** Command Query Responsibility Segregation separates read (Query) and write (Command) operations into different models.
**Pairing with Event Sourcing:** Event Sourcing stores state changes as a sequence of events. CQRS uses these events to build optimized read models.
**Problem Solved:** It solves the scaling bottleneck of traditional CRUD where read and write models compete for the same database resources. It also provides a complete audit trail of state changes.

AA6 — **Circuit Breaker:** A pattern that stops calling a failing service to prevent cascading failures.
**Three States:** Closed (normal operation), Open (failures detected, requests blocked), Half-Open (testing if service recovered).
**Difference from Retry:** Retry attempts to fix the failure immediately; Circuit Breaker stops traffic to allow the service to recover.
**Use Together:** Use Circuit Breaker to stop the cascade, and Retry with backoff within the Half-Open state to verify recovery.

AA7 — **Tenant Isolation Models:**
1.  **Shared DB, Shared Schema:** Lowest cost, highest operational complexity (schema changes affect all), high noisy-neighbor risk, lower security.
2.  **Shared DB, Separate Schema:** Balanced cost and security. Schema changes require migration per tenant. Moderate noisy-neighbor risk.
3.  **Separate DB per Tenant:** Highest cost, lowest operational complexity (easy backups), lowest noisy-neighbor risk, highest security (data isolation).

AA8 — **Consistency:** Strong consistency ensures all nodes see the same data immediately. Eventual consistency allows temporary divergence.
**Architecture for Inventory:** Use **Strong Consistency**.
**Data Store:** Relational Database (SQL) with row-level locking or optimistic concurrency control.
**Patterns:** Use a distributed lock (e.g., Redis Redlock) or database transactions to reserve stock before purchase. At scale, use a dedicated inventory service with a high-performance cache (Redis) for availability, but enforce strong consistency on the write path.

AA9 — **Sidecar Proxy:** A container deployed alongside the application container to handle network traffic.
**Envoy in Istio:** Envoy handles the data plane (traffic routing, mTLS). The Istio Control Plane (Pilot) configures Envoy.
**Data vs. Control Plane:** Data plane processes traffic; Control plane manages configuration and policy.
**mTLS:** Application-level TLS secures data in transit. mTLS between sidecars authenticates the *identity* of the service (not just the host) and encrypts traffic automatically without code changes.

AA10 — **Strangler Fig Pattern:** Incrementally replacing a legacy system by routing specific functionality to new services.
**Migration Strategy:** Deploy an API Gateway in front of the .NET Framework monolith. Create new .NET 8 microservices for specific features. Configure the Gateway to route requests for those features to the new services while routing the rest to the monolith. Gradually "strangle" the monolith until it is removed.

AA11 — **12-Factor App:** Methodology for building cloud-native apps.
**6 Factors:**
1.  **Codebase:** One codebase tracked in version control.
2.  **Dependencies:** Explicitly declared and isolated.
3.  **Config:** Stored in environment variables.
4.  **Backing Services:** Treated as attached resources.
5.  **Build, Release, Run:** Strictly separated stages.
6.  **Processes:** Stateless, short-lived processes.
**Why:** Ensures portability, scalability, and ease of deployment across environments.

AA12 — **Event-Driven (Kafka) vs. Request-Driven (REST/gRPC):**
*   **Coupling:** Event-driven is loosely coupled; Request-driven is tightly coupled.
*   **Scalability:** Event-driven scales better for high throughput; Request-driven is simpler for low latency.
*   **Debugging:** Event-driven is harder (distributed tracing needed); Request-driven is easier (stack traces).
*   **Ordering:** Kafka guarantees ordering per partition; REST does not.
*   **Failure:** Event-driven requires retry/dead-letter queues; Request-driven returns errors immediately.
**Choice:** Use Event-driven for decoupled, high-volume data pipelines. Use Request-driven for real-time user interactions.

AA13 — **Load Balancer:** Distributes traffic across multiple servers.
**L4 vs. L7:** L4 (Transport) routes based on IP/Port. L7 (Application) routes based on HTTP headers, URL, cookies.
**L7 Use Case:** SSL Termination or routing `/api` traffic to backend A and `/static` to backend B.

AA14 — **Message Queue:** Stores messages for delivery.
**MQ (RabbitMQ) vs. Log (Kafka):** MQ is for task distribution (consume once). Kafka is for event streaming (retain logs, replay).
**Pick One:** Use RabbitMQ for job queues (email, processing). Use Kafka for audit logs, analytics, or event sourcing.

AA15 — **Container vs. VM:** A container virtualizes the OS kernel (shares host kernel), while a VM virtualizes hardware (includes guest OS).
**Popularity:** Containers are lightweight, start faster, and have higher density, making them ideal for microservices.

AA16 — **Stateless vs. Stateful:** Stateless services store no session data (state is in DB/Cache). Stateful services store session data locally.
**Scaling:** Stateless services scale horizontally easily (any instance can handle any request). Stateful services require sticky sessions or data replication.
**Example:** Stateless: Web server. Stateful: Database.

AA17 — **ORM:** Maps database tables to programming language objects.
**Examples:** Hibernate (Java), Entity Framework (.NET).
**Trade-offs:** ORMs increase developer productivity but can generate inefficient SQL and hide database complexity. Raw SQL offers performance control but increases maintenance.

AA18 — **Bulkhead Pattern:** Isolates resources (threads, connections) for different services to prevent one failure from exhausting shared resources.
**Implementation:** Use separate thread pools for each downstream service call. Alternatively, deploy critical services on separate physical nodes or Kubernetes namespaces with resource quotas.

AA19 — **OT vs. CRDT:** OT requires a central server to resolve conflicts sequentially. CRDTs allow local edits that mathematically converge without a central server.
**Comparison:** OT has lower latency for server-synced but complex logic. CRDTs have higher latency for convergence but better offline support.
**Choice:** **CRDTs**. They allow offline editing and automatic conflict resolution without server coordination, essential for a Google Docs-like experience.

AA20 — **Distributed Tracing Challenges:**
*   **Context Propagation:** Must pass trace IDs via headers (W3C Trace Context) or message headers.
*   **Sampling:** Head-based samples at start (low overhead); Tail-based samples after latency analysis (high accuracy).
*   **Storage:** Jaeger/Tempo store spans; querying requires indexing.
*   **Async Messaging:** Inject trace context into Kafka message headers.
*   **Continuity:** Ensure the producer adds trace context and the consumer extracts it to link the span.

OP1 — **Hypervisor Type 1 vs. Type 2:** Type 1 runs directly on hardware (Bare-metal). Type 2 runs on top of an OS.
**Examples:** Type 1: VMware ESXi, Microsoft Hyper-V Server. Type 2: VirtualBox, VMware Workstation.

OP2 — **SAN vs. NAS:** SAN provides block-level storage (appears as a disk). NAS provides file-level storage (appears as a network share).
**Usage:** SAN for databases (high I/O). NAS for file sharing.
**Protocols:** SAN: iSCSI, Fibre Channel. NAS: NFS, SMB/CIFS.

OP3 — **3-Tier Network:** Core (backbone), Distribution (policy), Access (endpoints).
**Leaf-Spine:** Preferred for modern data centers.
**Problem Solved:** 3-tier has oversubscription and high latency between tiers. Leaf-Spine provides non-blocking, low-latency, predictable bandwidth between any two nodes.

OP4 — **Proxmox HA Architecture:**
*   **Nodes:** Minimum 3 for quorum.
*   **Quorum:** Majority vote required to make decisions.
*   **Fencing:** STONITH (e.g., iDRAC) to power off failed nodes.
*   **Storage:** Ceph (distributed) or iSCSI/NFS (shared).
*   **Failure:** If a node fails, fencing triggers, and VMs restart on surviving nodes.

OP5 — **VLAN Trunking:** Carries multiple VLAN traffic over a single link using 802.1Q tags.
**Linux Config:** Create sub-interfaces. Example: `ip link add link eth0 name eth0.10 type vlan id 10`. Assign IP to `eth0.10`.

OP6 — **RAID Levels:**
*   **RAID 1:** Mirroring (50% capacity).
*   **RAID 5:** Striping + Parity (N-1 capacity, 1 drive fault).
*   **RAID 6:** Striping + Double Parity (N-2 capacity, 2 drive fault).
*   **RAID 10:** Mirroring + Striping (50% capacity, fast).
*   **RAID Z2:** ZFS double parity (similar to RAID 6).
**Choice:** **RAID 10** (or ZFS RAID-Z2). For a DB server, performance is critical. RAID 10 offers the best read/write performance with redundancy for 6 drives.

OP7 — **DNS Architecture:**
*   **Internal:** AD DNS servers (Active Directory integrated).
*   **External:** Public authoritative servers (e.g., Route53).
*   **Split-Horizon:** Internal clients resolve internal IPs; external resolve public IPs.
*   **Forwarders:** Internal DNS forwards unknown queries to external resolvers.
*   **Laptop:** Connects via DHCP, receives internal DNS server IP. Resolves internal names via AD, external via forwarders.

OP8 — **PKI Chain:** Root CA (offline, trust anchor) -> Subordinate CA (issues certs) -> End-entity.
**Internal PKI:** Root offline. Subordinate issues certs via templates. Auto-enrollment via Group Policy.
**Revocation:** CRL (list) vs OCSP (real-time check).
**Consequences:** Compromised Root = Revoke all trust (catastrophic). Compromised Subordinate = Revoke that CA and reissue certs (contained).

OP9 — **WAN Consolidation:**
*   **MPLS:** High cost, high reliability, low latency.
*   **IPsec VPN:** Low cost, variable reliability, higher latency.
*   **SD-WAN:** Low cost, intelligent routing, high reliability.
*   **WireGuard:** Lightweight, secure, requires mesh management.
**Recommendation:** **SD-WAN**. It offers MPLS-like reliability with internet cost savings and handles VoIP latency via QoS policies better than standard IPsec.

OP10 — **iDRAC/iLO/IPMI:** Out-of-band management interfaces allowing remote power control and console access.
**Critical:** Allows recovery if the OS is down.
**Exposure:** If exposed to the internet, attackers can gain root access to the physical server (power cycle, BIOS access).

OP11 — **LVM vs. ZFS:**
*   **LVM:** Volume management on top of block devices. Thin provisioning available.
*   **ZFS:** Filesystem + Volume Manager. Copy-on-write, checksums.
*   **Snapshots:** LVM (fast, copy-on-write). ZFS (instant, space-efficient, immutable).
*   **Use:** LVM for simple VM storage. ZFS for data integrity and snapshots.

OP12 — **Stretch Cluster:**
*   **Design:** 2 sites + 1 Witness node (tiebreaker).
*   **Ceph:** CRUSH rules distribute replicas across sites.
*   **Latency:** <10ms RTT required for synchronous replication.
*   **Split-Brain:** If network fails, both sites think they are primary.
*   **Tiebreaker:** Without a witness, Ceph cannot decide which site to keep, risking data corruption.

OP13 — **DHCP:** Dynamic Host Configuration Protocol.
**DORA:** Discover, Offer, Request, Acknowledge.
**Unavailable:** Client retries. If lease expires, client enters 0.0.0.0 state and broadcasts Discover again.

OP14 — **Managed vs. Unmanaged Switch:**
*   **Unmanaged:** Plug-and-play.
*   **Managed:** Configurable (VLANs, ACLs, QoS, SNMP).
*   **Importance:** VLANs for segmentation, QoS for VoIP, SNMP for monitoring.

OP15 — **NTP:** Network Time Protocol.
**Critical:** Cluster consensus (Kafka, DB) and logging depend on accurate timestamps.
**Drift:** Clock skew causes replication errors, authentication failures (Kerberos), and log analysis issues.

OP16 — **Physical vs. VM vs. Container:**
*   **Physical:** Hardware only.
*   **VM:** Hardware virtualized.
*   **Container:** OS virtualized.
*   **Bare Metal:** Use for high I/O (DB), low latency (HPC), or licensing restrictions.

OP17 — **UPS vs. Generator:**
*   **UPS:** Short-term battery power (seconds/minutes).
*   **Generator:** Long-term power (hours/days).
*   **ATS:** Automatic Transfer Switch switches power from Grid to UPS to Generator seamlessly.

OP18 — **Active-Passive vs. Active-Active:**
*   **Active-Passive:** One node handles traffic; standby takes over on failure.
*   **Active-Active:** All nodes handle traffic.
*   **PostgreSQL HA:** Pacemaker/Corosync manages the cluster. Virtual IP floats. STONITH (fencing) ensures failed node is killed. Failover triggers `pg_ctl promote` on standby.

OP19 — **Backup Strategy:**
*   **3-2-1:** 3 copies, 2 media types, 1 offsite.
*   **RPO/RTO:** Recovery Point/Time Objective.
*   **Types:** Full (base), Incremental (changes since last), Differential (changes since full).
*   **Protection:** Immutable backups prevent ransomware deletion.
*   **Test:** Regular restore drills.
*   **Backup vs. DR:** Backup is data copy; DR is the plan to restore operations at a new site.

OP20 — **802.1X:** Port-based Network Access Control.
*   **Roles:** Supplicant (Client), Authenticator (Switch), Server (RADIUS).
*   **EAP:** EAP-TLS (Certificate-based, secure). PEAP (Password-based, tunneled).
*   **Dynamic VLAN:** Server assigns VLAN based on user identity.
*   **Failure:** Port blocked or placed in Guest VLAN.
*   **MAB:** Fallback for devices without 802.1X; authenticates based on MAC address.
