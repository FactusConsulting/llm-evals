K1 — Easy: A **Deployment** manages stateless applications, creating pods with random names and allowing arbitrary scaling or rolling updates. A **StatefulSet** manages stateful applications, providing stable, unique network identities (pod-0, pod-1), ordered deployment/termination, and stable persistent storage.

K2 — Easy: A **ClusterIP** Service creates a virtual IP (VIP) within the cluster. **kube-proxy** (using iptables or IPVS) configures the node's networking rules to route traffic sent to the VIP to the backend Pod IPs based on the Service's selector.

K3 — Medium: A **PersistentVolume (PV)** is a cluster-wide storage resource. A **PersistentVolumeClaim (PVC)** is a user request for storage. A **StorageClass** defines the provisioning mechanism (dynamic). A PVC binds to a PV (static) or triggers a PV creation via StorageClass (dynamic).

K4 — Medium: 1. Check `kubectl logs` for application errors. 2. Check `kubectl describe pod` for OOMKilled or CrashLoopBackOff events. 3. Verify liveness/readiness probe configurations. 4. Check resource limits (CPU/Memory) against node capacity.

K5 — Medium: CoreDNS provides DNS resolution for services within the cluster. If CoreDNS pods are down, internal service discovery (e.g., `my-service.default.svc`) fails, though external DNS resolution may still work if the pod's `/etc/resolv.conf` points to external nameservers.

K6 — Hard: The scheduler uses **Filtering** (remove nodes that don't meet requirements like taints, resources) and **Scoring** (rank remaining nodes). Factors: 1. Resource requests (CPU/Mem). 2. Node Affinity/Anti-affinity rules. 3. Topology Spread Constraints (pod distribution).

K7 — Hard: Yes, it will be scheduled (256Mi request < 0.5GB free). At runtime, if it uses 600Mi (exceeding 512Mi limit), the kernel will send SIGKILL, and the pod will restart in an OOMKilled state.

K8 — Hard: 1. Client hits External LB. 2. LB forwards to Ingress Controller (NGINX). 3. NGINX matches Ingress rules to Service. 4. Service routes to Endpoints (Pod IPs). 5. kube-proxy (iptables) redirects traffic to Pod. 6. Pod receives request.

K9 — Medium: A **NetworkPolicy** restricts traffic flow between pods. Default behavior: all traffic is allowed. When a policy is applied, it acts as a deny-all baseline for the selected pods unless explicitly allowed by the policy rules.

K10 — Medium: **RollingUpdate** replaces pods gradually (zero downtime). **Recreate** kills all existing pods before starting new ones. Choose **Recreate** for stateful applications that cannot handle concurrent versions or require exclusive access to resources during update.

K11 — Easy: A **Namespace** is a virtual cluster within a physical cluster for resource isolation. Default namespaces: `default`, `kube-system`, `kube-public`, `kube-node-lease`.

K12 — Easy: `describe` shows detailed status and events. 1. **Events** (warnings/errors). 2. **Conditions** (Ready status). 3. **Mounts/Env Vars** (configuration details).

K13 — Easy: A **ConfigMap** stores non-sensitive configuration data. Pods consume it via: 1. **Environment Variables**. 2. **Volume Mounts** (files).

K14 — Easy: **ClusterIP** (internal only), **NodePort** (exposes port on every node), **LoadBalancer** (provisions external cloud load balancer).

K15 — Easy: A **DaemonSet** ensures a copy of a pod runs on all (or specific) nodes. Use for logging agents, monitoring daemons, or storage daemons that need node-level access.

K16 — Medium: **Taints** repel pods from nodes; **Tolerations** allow pods to schedule on tainted nodes. **Node Affinity** attracts pods to nodes. Use case: Taint for dedicated GPU nodes; Affinity for co-locating related services.

K17 — Medium: An **Operator** is a custom controller that manages complex stateful applications using CRDs. It differs from a standard controller by automating the lifecycle (backup, restore, scaling) of the custom resource, solving operational complexity.

K18 — Hard: etcd uses **linearizability** (strong consistency). If quorum is lost, the cluster becomes read-only/unavailable. Recovery: Restore from a snapshot on a new etcd cluster, then rejoin remaining nodes to the new cluster.

K19 — Hard: **Role** (namespaced permissions), **ClusterRole** (cluster-wide), **RoleBinding** (links Role to Subject in NS), **ClusterRoleBinding** (links ClusterRole to Subject cluster-wide).
```yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata: namespace: default
name: pod-reader
rules:
- apiGroups: [""]
  resources: ["pods", "pods/log"]
  verbs: ["get", "list", "watch"]
---
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata: namespace: default
name: read-pods
subjects:
- kind: User
  name: developer
  apiGroup: rbac.authorization.k8s.io
roleRef:
  kind: Role
  name: pod-reader
  apiGroup: rbac.authorization.k8s.io
```

K20 — Hard: `ndots` (default 5) causes the resolver to try appending search domains before querying the FQDN. If `ndots` is high, a query for `google.com` triggers 5+ queries. Fix: Set `dnsConfig: search: [""]` and `ndots: 1` in the pod spec to query FQDN immediately.

D1 — Easy: **Rebase** rewrites history to create a linear commit chain. **Merge** creates a new commit preserving branch history. Avoid rebasing on shared/public branches to prevent history rewriting for collaborators.

D2 — Easy: **TCP** is connection-oriented and reliable (HTTP, SSH). **UDP** is connectionless and faster (DNS, VoIP).

D3 — Medium: A **race condition** occurs when output depends on the sequence/timing of uncontrollable events. Example: Two threads incrementing a shared counter without a lock results in lost updates.

D4 — Medium: **Symmetric** uses one key for encryption/decryption. **Asymmetric** uses a public/private key pair. TLS uses Asymmetric for the handshake (key exchange) and Symmetric for the data transfer (performance).

D5 — Medium: A **layer** is a read-only filesystem diff in a container image. Caching: Place frequently changing commands (like copying source code) after stable commands (like installing dependencies) to maximize build cache reuse.

D6 — Medium: 1. Check upstream service health. 2. Verify database connectivity. 3. Check resource limits (OOM). 4. Review rate limiting or circuit breaker logs.

D7 — Medium: **CAP** states a distributed system can only guarantee 2 of 3: Consistency, Availability, Partition Tolerance. **CP**: Zookeeper (consistency over availability during partition). **AP**: Cassandra (availability over consistency during partition).

D8 — Medium: **Horizontal** adds more nodes; **Vertical** adds more resources to a node. Patterns enabling horizontal scaling: Stateless services, sharding, load balancing.

D9 — Easy: An **environment variable** is a dynamic value affecting process behavior. Persist across sessions on Linux by adding `export VAR=value` to `~/.bashrc` or `~/.profile`.

D10 — Easy: **JSON** is strict key-value (machine-friendly). **YAML** is human-readable (indentation-based). Advantage JSON: Strict parsing. Advantage YAML: Readability for config.

D11 — Easy: **REST** is an architectural style using HTTP. **GET** (retrieve), **POST** (create), **PUT** (replace), **DELETE** (remove).

D12 — Easy: **Version pinning** specifies exact dependency versions (e.g., `1.0.1` vs `^1.0.0`). Important for reproducible builds to prevent unexpected breakage from upstream updates.

D13 — Easy: **Compiled** (source to binary before run: C, Go). **Interpreted** (source executed line-by-line: Python, JavaScript).

D14 — Medium: A **reverse proxy** sits in front of servers (client -> proxy -> server). A **forward proxy** sits in front of clients (client -> proxy -> internet). Servers: Nginx (web), HAProxy (TCP). Use: SSL termination, caching.

D15 — Medium: A **deadlock** is a permanent blocking of processes. Coffman conditions: 1. Mutual Exclusion. 2. Hold and Wait. 3. No Preemption. 4. Circular Wait.

D16 — Medium: **Eventual consistency** ensures updates propagate over time. Cassandra uses **Read Repair** and **Hinted Handoff**. Tunable levels: ONE, QUORUM, ALL.

D17 — Medium: **mTLS** authenticates both client and server via certificates. Handshake: Client sends cert, Server validates; Server sends cert, Client validates. Common in **Service Meshes** (Istio) for zero-trust networking.

D18 — Medium: A **hash table** maps keys to array indices. Collision resolution: **Chaining** (linked lists) or **Open Addressing** (probing). Complexity: Avg O(1), Worst O(n).

D19 — Medium: A **circuit breaker** prevents cascade failures. States: **Closed** (normal), **Open** (fail fast), **Half-open** (test recovery). Relates to retry: Do not retry if circuit is open; use exponential backoff when transitioning to half-open.

D20 — Medium: **Optimistic** (check version on commit, e.g., `IF version = X`). **Pessimistic** (lock row on read, e.g., `SELECT FOR UPDATE`). Choose Optimistic for low contention; Pessimistic for high contention/conflict.
