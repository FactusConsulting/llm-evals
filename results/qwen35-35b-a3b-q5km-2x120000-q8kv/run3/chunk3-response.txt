T1 — `tofu plan` generates an execution plan showing what changes will be made without applying them. `tofu apply` executes the plan, creating, updating, or destroying resources to match the configuration.

T2 — State is a JSON file mapping real-world resources to your configuration. Remote state is important for collaboration (team access), locking (preventing concurrent writes), and security (storing sensitive data off local machines).

T3 — `count` uses integer indices; `for_each` uses map/set keys. `for_each` is preferred when resources have unique identifiers (like names) because adding/removing items doesn't shift indices, preventing unintended resource recreation.

T4 — This is "state drift." `tofu plan` detects the difference between state and reality. `tofu apply` will attempt to reconcile the drift by updating the resource to match the configuration, or fail if the resource is immutable.

T5:
```hcl
resource "aws_instance" "web" {
  for_each = toset(["web1", "web2", "web3"])
  ami      = "ami-12345678"
  instance_type = "t2.micro"
  tags = {
    Name = each.key
  }
}
```

T6 — `terraform_remote_state` reads the raw state file of another module. `output` values are the intended interface. Recommended alternative: Use `output` values if modules are in the same state file (direct reference). If separate, `terraform_remote_state` is standard, but consolidating state is preferred to reduce coupling and complexity.

T7 — The output is computed before the `null_resource` provisioner finishes executing. Fix: Add `depends_on = [null_resource.kubeconfig]` to any resource consuming the kubeconfig output, or use `local-exec` to write the config to a file and read it back.

T8 — `tofu import` maps existing infrastructure to state without creating config. Limitations: No config generation, manual mapping. `import` block (1.5+) allows importing within the configuration file, enabling CI/CD pipelines to manage imports declaratively.

T9 — A `variable` is an input parameter passed into the module. An `output` is a value exposed by the module to be used by the caller.

T10 — A provider is a plugin that interacts with a specific API (e.g., AWS, Azure). Without `tofu init`, providers are not downloaded, and `tofu plan` fails with "Provider not installed" errors.

T11 — The `.terraform.lock.hcl` file records exact provider versions and checksums. It should be committed to ensure reproducible builds and prevent breaking changes from provider updates.

T12 — `tofu destroy` removes all managed resources. To target a single resource: `tofu destroy -target=aws_instance.example`.

T13 — A data source reads information from an external system without creating or managing it. Example: `data "aws_ami" "ubuntu"` to get the latest Ubuntu AMI ID instead of managing an AMI resource.

T14 — `locals` define computed values within the configuration scope. `variables` are inputs from outside. Use `locals` for internal logic, transformations, or constants that don't need to be configurable by the user.

T15 — `lifecycle` manages resource behavior. `create_before_destroy` (zero-downtime updates), `prevent_destroy` (protects resources from accidental deletion), `ignore_changes` (tolerates manual drift on specific attributes).

T16 — A `moved` block tracks resource identity changes during refactoring (e.g., moving a resource to a new module). It updates the state file to point to the new address without destroying and recreating the resource.

T17 — The dependency graph is a DAG determining execution order. Implicit dependencies come from resource references. Explicit `depends_on` forces order. `depends_on` should be used sparingly because it prevents parallel execution of independent resources, slowing down apply.

T18:
1. **Increase `parallelism`**: Faster apply, but higher API rate limit risk.
2. **Optimize State Backend**: Use S3 + DynamoDB for locking, reduces network latency.
3. **Module Isolation**: Split large configs into smaller modules to reduce graph size.
4. **State Caching**: Use `tofu state` commands to cache remote state locally during development.

T19 — State locking prevents concurrent modifications. If a lock isn't released (crash), subsequent runs hang. Manually break with `tofu force-unlock <LOCK_ID>`. Risk: Data corruption if another process is actually writing.

T20:
1. Backup local state file.
2. Configure `backend` in `tofu.tf` (S3 + DynamoDB).
3. Run `tofu init -migrate-state`.
4. Verify state is in S3 (`aws s3 ls`).
5. Run `tofu plan` to ensure no drift.

A1 — A **playbook** is a YAML file defining automation steps. A **role** is a reusable package of tasks, handlers, and templates. A **task** is a single action executed on a host.

A2 — `ansible_facts` is a dictionary of gathered system information. Access OS family via `{{ ansible_os_family }}`.

A3 — `command` runs without shell features (no pipes). `shell` runs with shell features (pipes, variables). `raw` runs without setup (no Python interpreter), useful for initial bootstrap.

A4 — Idempotency means running a task multiple times yields the same result. `apt: state=present` is idempotent. `shell: rm -rf /tmp` is NOT idempotent (deletes every time).

A5:
```yaml
- name: Install htop
  package:
    name: htop
    state: present
  when: ansible_os_family == "Debian"

- name: Install htop and epel-release
  package:
    name:
      - htop
      - epel-release
    state: present
  when: ansible_os_family == "RedHat"
```

A6 — Precedence (High to Low): Extra vars (`-e`) > Play vars > Host vars > Group vars > Role vars > Role defaults > Inventory vars. `set_fact` overrides most at runtime.

A7 — Use `serial: 3` in the play to process 3 nodes at a time. Group nodes in inventory. Use `wait_for` or `wait_for_connection` to ensure the first 3 are ready before the next batch starts.

A8 — `ansible-vault` encrypts sensitive data. To encrypt a single variable: `ansible-vault encrypt_string 'secret_value' --name 'my_secret'`.

A9 — Inventory lists hosts and groups. **Static** is a file (INI/YAML). **Dynamic** is a script/API that generates inventory dynamically (e.g., from cloud provider).

A10 — `ansible webservers -m shell -a "df -h"`.

A11 — A **handler** is a task triggered by `notify`. It runs only if notified and executes after all tasks in the play complete.

A12 — `ansible.cfg` configures Ansible behavior. Common settings: `inventory`, `remote_user`, `forks`, `timeout`.

A13 — `register` saves a task's output to a variable. Use it in subsequent tasks via `when: my_var.stdout == "success"`.

A14 — `include_tasks` is dynamic (evaluated at runtime, supports `when`). `import_tasks` is static (evaluated at compile time, faster, no `when`).

A15 — Ansible Galaxy is a repository for roles/collections. `requirements.yml` lists them. Install via `ansible-galaxy install -r requirements.yml`.

A16 — Connection plugins define how Ansible talks to hosts. **ssh** (remote Linux), **local** (control node), **network_cli** (network devices).

A17 — `delegate_to: localhost` runs a task on the control node. `run_once` ensures it runs once even in a loop. Example: Generate a config file on control node using facts gathered from remote hosts.

A18:
1. **Increase `forks`**: More parallelism, higher control node CPU/RAM usage.
2. **Enable `pipelining`**: Reduces SSH connections, requires SSH config changes.
3. **Fact Caching**: Reduces gathering time, risks stale facts.
4. **Mitogen**: Optimizes SSH, adds complexity/dependencies.

A19 — Jinja2 is the templating engine. `{{ }}` (expression), `{% %}` (control flow), `{# #}` (comment).
Snippet:
```jinja2
upstream backend {
{% for server in backend_servers %}
    server {{ server }}; # Health check: {{ server }}
{% endfor %}
}
```

A20 — Use `serial: 5` for batches. Pre-task: Deregister from LB. Deploy. Post-task: Health check (`uri`). If fail, `rescue` block: Re-register old version, rollback deployment. Use `block/rescue` for atomicity.
