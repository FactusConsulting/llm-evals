## Go

**G1 — Easy:** An array is a fixed-size value type `[N]T`. A slice is a dynamic view over an array, defined as a struct containing a pointer to the array, length, and capacity (`[len]T`). When appending to a slice that has reached its capacity, the Go runtime allocates a new, larger underlying array (typically double the size), copies the existing elements, and returns a new slice header pointing to it.

**G2:** `var x int` declares a variable with the zero value (`0`) and requires an explicit type. `x := 0` is a short variable declaration that infers the type. You cannot use `:=` to redeclare a variable in the same scope (e.g., `x := 1` after `x := 0` in the same block) or to declare a variable outside a function (e.g., at package level).

**G3:** A goroutine is a lightweight thread managed by the Go runtime, not the OS. It uses less memory and context-switching overhead than OS threads. The Go scheduler uses the GMP model: **G** (Goroutine), **M** (OS Thread), and **P** (Logical Processor). The scheduler maps Gs to Ps, which run on Ms, allowing efficient multiplexing of many goroutines on fewer OS threads.

**G4:** Channels are typed conduits for communication between goroutines. An **unbuffered channel** blocks the sender until a receiver is ready and vice versa. A **buffered channel** allows sending without blocking up to its capacity. Sending to a full buffered channel blocks until a receiver reads from it.

**G5:** The `context` package manages deadlines, cancellation signals, and request-scoped values across API boundaries.
```go
ctx, cancel := context.WithTimeout(context.Background(), 5*time.Second)
defer cancel()
req, _ := http.NewRequestWithContext(ctx, "GET", url, nil)
resp, err := client.Do(req) // Cancels if timeout exceeded
```

**G6:** Go interfaces are satisfied implicitly (duck typing); a type implements an interface if it implements all methods. Unlike Java/C#, you don't declare `implements`. The empty interface `interface{}` (or `any`) accepts any type. It is powerful for generic-like flexibility but dangerous because it requires type assertions to use values, losing compile-time type safety.

**G7:** This is unsafe due to a **data race**. The main goroutine reads `data` without synchronization while the worker writes it. The Go memory model guarantees no visibility of writes across goroutines without synchronization.
**Fix:** Use `sync/atomic` for the boolean flag or a `sync.Mutex` to protect both variables.
```go
var ready atomic.Bool
var data string
go func() {
    data = "hello"
    ready.Store(true)
}()
for !ready.Load() {}
fmt.Println(data)
```

**G8:** `sync.Mutex` provides exclusive access. `sync.RWMutex` allows multiple readers or one writer. `sync.Map` is optimized for cases where keys are rarely deleted and reads vastly outnumber writes. It avoids locking for reads but has higher overhead for writes compared to a `map` with a `Mutex` due to internal sharding and atomic operations.

**G9:** Go modules manage dependencies. `go.mod` declares the module path and required dependencies. `go.sum` stores checksums for integrity. `go mod tidy` removes unused dependencies and adds missing ones. **MVS (Minimum Version Selection)** ensures the lowest compatible version of a dependency is selected to minimize breaking changes.

**G10:** `defer` schedules a function call to run after the surrounding function returns. Multiple defers execute in **LIFO** (Last In, First Out) order. A common gotcha in loops is capturing the loop variable by reference in the deferred closure, causing all deferred calls to see the final loop value.

**G11:** A pointer holds a memory address. Its zero value is `nil`. To create a pointer to a struct literal: `p := &MyStruct{Field: "value"}`.

**G12:** `init()` is a function executed automatically before `main()` runs. It runs once per package. A package can have multiple `init()` functions; they execute in the order they appear in the source files, and across files, in the order the compiler loads them (typically alphabetical).

**G13:** `new(T)` allocates zeroed memory of type `T` and returns a pointer `*T`. `make(T)` initializes built-in types (`slice`, `map`, `chan`) and returns the initialized value (not a pointer). `new` is for pointers to any type; `make` is for built-in reference types.

**G14:** Errors are values implementing the `error` interface (`Error() string`). The idiomatic pattern is `if err != nil { return err }`. `errors.Is(err, target)` checks if the error chain contains `target`. `errors.As(err, &target)` checks if the error chain contains a specific type. `==` compares interface values (pointers), which fails for wrapped errors.

**G15:**
*   **Map:** Zero value is `nil`.
*   **Slice:** Zero value is `nil` (empty slice `[]T` is distinct).
*   **Channel:** Zero value is `nil`.
*   **Struct:** Zero value has all fields set to their respective zero values.

**G16:** A race condition occurs when two goroutines access shared data concurrently, and at least one writes. The `-race` flag instruments code to detect these at runtime.
**Fix:** Use `sync.Mutex` to lock access.
```go
var mu sync.Mutex
mu.Lock()
// access shared data
mu.Unlock()
```

**G17:** `select` waits on multiple channel operations. Unlike `switch`, it chooses a case **randomly** if multiple are ready. It can implement timeouts by adding a `case <-time.After(d)`.
```go
select {
case data := <-ch:
    // handle
case <-time.After(1 * time.Second):
    // timeout
}
```

**G18:** Escape analysis determines if a variable lives on the stack or heap. If a variable's address escapes the function scope (e.g., returned), it is allocated on the heap. Inspect with `go build -gcflags="-m"`. Heap allocations increase GC pressure; stack allocations are faster.

**G19:** `reflect` allows inspecting values at runtime. `reflect.Type` is the type metadata; `reflect.Value` is the value itself. Reflection is slow due to type introspection and dynamic dispatch. It is justified for frameworks (e.g., serialization, ORMs) but avoided in performance-critical paths.

**G20:** `//go:embed` embeds files/directories at compile time into a `embed.FS`. It works with single files (`//go:embed file.txt`) or directories (`//go:embed images/*`). Limitations: files must exist at compile time. It implements `io/fs.FS`, allowing standard library functions like `os.ReadFile` to work on embedded content.

## Rust

**R1:** Ownership is Rust's core memory safety feature. Rules: 1. Each value has exactly one owner. 2. When the owner goes out of scope, the value is dropped. 3. You can have one mutable reference OR any number of immutable references, but not both simultaneously.

**R2:** `String` is an owned, heap-allocated, growable string. `&str` is a borrowed, immutable string slice (usually pointing to static memory or a `String`). Use `&str` in function signatures for flexibility (accepts both `String` and string literals) unless ownership transfer is required.

**R3:** Borrowing allows referencing data without taking ownership. **Immutable (`&T`)**: Multiple allowed, read-only. **Mutable (`&mut T`)**: Only one allowed, read-write. The borrow checker enforces these rules at compile time to prevent data races and dangling pointers.

**R4:** `Result<T, E>` represents success (`Ok(T)`) or failure (`Err(E)`). `Option<T>` represents `Some(T)` or `None`. The `?` operator propagates errors.
```rust
fn read_file() -> Result<String, std::io::Error> {
    std::fs::read_to_string("file.txt")
}
```

**R5:** `Box<T>` owns data on the heap (single owner). `Rc<T>` (Reference Counted) allows multiple owners in a single thread. `Arc<T>` (Atomic Reference Counted) allows multiple owners across threads. Use `Box` for heap allocation, `Rc` for shared ownership (single-thread), `Arc` for shared ownership (multi-thread).

**R6:** Traits define shared behavior. Unlike Java interfaces, they can have default implementations. **Trait Object (`dyn Trait`)**: Dynamic dispatch (vtable), runtime cost. **Generic (`T: Trait`)**: Static dispatch (monomorphization), zero-cost. Use generics for performance, trait objects for heterogeneous collections.

**R7:** This code technically compiles in modern Rust due to lifetime elision, but it highlights the need for explicit lifetimes to ensure safety. The compiler must know which input's lifetime the return value is tied to.
**Fix:** Explicitly annotate lifetimes.
```rust
fn longest<'a>(x: &'a str, y: &'a str) -> &'a str {
    if x.len() > y.len() { x } else { y }
}
```

**R8:** `Send` means a type can be transferred across thread boundaries. `Sync` means `&T` can be shared across threads. `Rc<T>` is not `Send` because its reference count is not atomic. `Arc<T>` uses atomic reference counting, making it `Send` and `Sync`.

**R9:** `async/await` creates a `Future` (a state machine). The executor (e.g., Tokio) polls the future until it completes. Unlike Go's preemptive goroutines, Rust's async is cooperative; the task yields control explicitly. This avoids stack overhead but requires an executor.

**R10:** `unsafe` allows bypassing safety checks. 1. Dereference raw pointers. 2. Call unsafe functions/FFI. 3. Modify `static mut`. Justified when interfacing with C, implementing low-level primitives, or optimizing performance where safety is manually verified.

**R11:** Pattern matching (`match`) exhaustively checks all possible values. `if let` matches a single pattern. The compiler enforces exhaustiveness in `match` to prevent logic errors (e.g., forgetting an enum variant).

**R12:** `Vec<T>` is a growable heap-allocated array. `[T; N]` is a fixed-size stack-allocated array. When `Vec` capacity is exceeded, it allocates a new, larger buffer and copies elements (usually doubling capacity).

**R13:** Enums are Algebraic Data Types (ADTs) that can hold data. Unlike C (integers) or Java (classes), Rust enums are memory-efficient and can hold variants with different types. `Option<T>` replaces `null` with `Some(T)` or `None`, forcing explicit handling of absence.

**R14:** `impl` defines methods. `&self` borrows, `&mut self` mutably borrows, `self` takes ownership. Associated functions (e.g., `String::new()`) do not take `self` and act like constructors.

**R15:** `derive` macros automatically implement traits. Common: `Debug`, `Clone`, `PartialEq`. Cannot derive if fields contain types that don't support the trait (e.g., `Fn` closures, raw pointers).

**R16:** `Cow<'a, T>` (Clone on Write) holds either a borrowed `&'a T` or an owned `Box<T>`. It avoids allocation if no mutation occurs. If mutation is needed, it clones the data to owned memory. Useful for functions that return strings but might accept borrowed input.

**R17:** `mod` defines a module. `use` imports paths. `pub` makes items visible outside the crate. File system maps to modules (`mod.rs` or `mod/` directory). Rust 2018 edition improved path resolution (`crate::` vs `super::`) and macro hygiene.

**R18:** `Pin<T>` prevents a value from being moved in memory. Needed for self-referential structs (e.g., async futures holding references to themselves) because moving invalidates pointers. `Unpin` is a marker trait indicating a type *can* be moved safely.

**R19:** `macro_rules!` (declarative) matches patterns and expands text. Procedural macros (derive, attribute) manipulate the AST directly. Hygiene ensures macro variables don't leak into the caller's scope, preventing naming collisions.

**R20:** Zero-cost abstractions mean high-level code compiles to efficient machine code. **Iterators** avoid intermediate allocations. **Generics** use monomorphization (code duplication per type) for static dispatch, matching hand-written C performance while providing type safety.
