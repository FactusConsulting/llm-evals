AA1 — **Monolith vs. Microservices:** A monolith is a single, unified codebase deployed as one unit. Microservices are independent, loosely coupled services deployed separately.
**Problems introduced by Microservices:**
1.  **Distributed Complexity:** Debugging requires tracing across network calls; no single log file contains the full story.
2.  **Data Consistency:** ACID transactions across services are hard; requires eventual consistency or sagas.
3.  **Operational Overhead:** Requires orchestration (Kubernetes), service discovery, and complex CI/CD pipelines compared to a single deployment.

AA2 — **Sync vs. Async:** Synchronous communication blocks the caller until a response is received (request/response). Asynchronous communication allows the caller to continue processing without waiting for an immediate response (fire-and-forget).
**Examples:** Synchronous: HTTP/REST or gRPC. Asynchronous: Message Queue (RabbitMQ) or Event Broker (Kafka).

AA3 — **API Gateway vs. Reverse Proxy:** A reverse proxy forwards traffic to backend servers (Layer 4/7). An API Gateway adds business logic like authentication, rate limiting, API versioning, and analytics.
**Replacement:** An API Gateway cannot *entirely* replace a reverse proxy in all cases. While it acts as an edge proxy, internal service-to-service communication often bypasses the gateway for lower latency, requiring a lightweight internal reverse proxy or service mesh sidecar.

AA4 — **Saga Pattern:** A sequence of local transactions where each updates the database and triggers the next. If one fails, compensating transactions undo previous steps.
**Choreography vs. Orchestration:** Choreography uses events to trigger steps (loose coupling, hard to trace). Orchestration uses a central coordinator (tighter control, single point of failure risk).
**Appropriateness:** Use Choreography for simple, decoupled flows. Use Orchestration for complex workflows requiring visibility and control.

AA5 — **CQRS:** Command Query Responsibility Segregation separates read (Query) and write (Command) data models.
**Pairing with Event Sourcing:** Event Sourcing stores state changes as immutable events. CQRS uses these events to build optimized read projections.
**Problem Solved:** Allows independent scaling of read and write workloads and provides a complete audit trail of state changes, which traditional CRUD (single table model) obscures.

AA6 — **Circuit Breaker:** Prevents repeated calls to a failing service.
**States:** Closed (normal), Open (failures detected, requests blocked), Half-Open (testing if service recovered).
**Difference from Retry:** Retry attempts to fix transient failures; Circuit Breaker stops traffic to prevent cascading failures during persistent outages.
**Usage:** Use Retry for transient network blips; use Circuit Breaker when a service is consistently down to save resources.

AA7 — **Multi-tenant Isolation:**
1.  **Shared DB/Schema:** Lowest cost, highest noisy-neighbor risk, complex security (row-level).
2.  **Shared DB/Schema per Tenant:** Moderate cost, better isolation, easier backup/restore per tenant.
3.  **Separate DB per Tenant:** Highest cost/complexity, best security/isolation, easiest tenant-specific maintenance.
**Trade-off:** Choose based on compliance needs and budget. High-security tenants need separate DBs; low-risk SaaS can use shared schema.

AA8 — **Consistency:** Strong consistency ensures all nodes see the same data immediately. Eventual consistency allows temporary divergence.
**Architecture:** For inventory (no oversell), use **Strong Consistency**.
**Data Store:** Relational DB (e.g., PostgreSQL) with row-level locking or a distributed lock manager (e.g., Redis/Zookeeper).
**Pattern:** Implement a reservation system or pessimistic locking on stock counts during checkout to prevent race conditions.

AA9 — **Sidecar Proxy:** A container deployed alongside the main application container to handle network traffic.
**Envoy/Istio:** Envoy handles the data plane (traffic routing, mTLS). Pilot (Control Plane) configures Envoy.
**mTLS:** Application-level TLS secures data in transit but requires code changes. mTLS between sidecars provides automatic encryption and identity verification without modifying application code.

AA10 — **Strangler Fig:** Incrementally migrate functionality from a legacy system to a new one.
**Strategy:** Deploy .NET 8 microservices alongside the .NET Framework monolith. Configure an API Gateway to route specific endpoints (e.g., `/api/new`) to the new services while routing others (`/api/legacy`) to the monolith. Gradually migrate endpoints until the monolith is retired.

AA11 — **12-Factor App:** Methodology for building cloud-native apps.
**6 Factors:**
1.  **Codebase:** One codebase tracked in version control.
2.  **Dependencies:** Explicitly declared and isolated.
3.  **Config:** Stored in environment variables.
4.  **Backing Services:** Treated as attached resources.
5.  **Build/Release/Run:** Strict separation of stages.
6.  **Port Binding:** Services export ports, not bind to them.
**Why:** Ensures portability, scalability, and consistent environments across dev/prod.

AA12 — **Event-Driven (Kafka) vs. Request-Driven (REST):**
*   **Coupling:** Event-driven is loosely coupled; Request-driven is tightly coupled.
*   **Scalability:** Event-driven scales better for high throughput; Request-driven scales linearly with load.
*   **Debugging:** Event-driven is harder (distributed state); Request-driven is easier (traceable flow).
*   **Ordering:** Kafka guarantees ordering; REST does not.
*   **Failure:** Event-driven uses Dead Letter Queues; Request-driven uses timeouts/retries.
**Choice:** Use Event-driven for decoupling and high-volume data processing. Use Request-driven for real-time user interactions requiring immediate feedback.

AA13 — **Load Balancer:** Distributes traffic across multiple servers.
**L4 vs. L7:** L4 (Transport) routes based on IP/Port. L7 (Application) routes based on HTTP headers/URL.
**Use Case:** L7 is required for SSL termination or routing `/api` to backend A and `/static` to backend B.

AA14 — **Message Queue vs. Log Broker:**
*   **MQ (RabbitMQ):** Tasks are consumed once and deleted. Good for job processing.
*   **Log Broker (Kafka):** Messages persist and can be replayed. Good for event streaming and audit logs.
**Choice:** Pick MQ for task queues (email sending). Pick Kafka for data pipelines (analytics, logging).

AA15 — **Container vs. VM:** A container shares the host OS kernel, while a VM virtualizes the hardware and runs a full guest OS.
**Popularity:** Containers are faster to start, more resource-efficient, and portable, making them ideal for microservices scaling.

AA16 — **Stateless vs. Stateful:** Stateless services store no session data on the server (e.g., Web Server). Stateful services store data locally (e.g., Database).
**Scaling:** Stateless services scale horizontally easily because any node can handle any request. Stateful services require complex data replication to scale.

AA17 — **ORM:** Maps database tables to programming language objects.
**Examples:** Entity Framework (.NET), Hibernate (Java).
**Trade-offs:** ORMs increase development speed but can generate inefficient SQL and hide database-specific optimizations compared to raw SQL.

AA18 — **Bulkhead Pattern:** Isolates resources (threads, connections) for specific services to prevent one failure from exhausting all resources.
**Implementation:** Use separate thread pools for different microservices or deploy services in separate Kubernetes namespaces with resource quotas (CPU/Memory limits).

AA19 — **OT vs. CRDT:**
*   **OT:** Server authoritative, requires coordination, complex conflict resolution.
*   **CRDT:** Client authoritative, mathematically guaranteed convergence, no coordination needed.
**Choice:** **CRDT**. It supports offline editing better because clients can merge changes locally without server intervention, ensuring eventual consistency when reconnected.

AA20 — **Distributed Tracing Challenges:**
*   **Context Propagation:** Must pass Trace ID/Span ID across services (W3C Trace Context/B3 headers).
*   **Sampling:** Head-based (random at start) vs. Tail-based (sample based on latency/errors).
*   **Storage:** High volume requires efficient storage (Jaeger/Tempo).
*   **Kafka:** Trace context must be injected into message headers so downstream consumers can continue the trace.

OP1 — **Hypervisor Type 1 vs. Type 2:**
*   **Type 1 (Bare-metal):** Runs directly on hardware (e.g., VMware ESXi, Proxmox VE). Better performance.
*   **Type 2 (Hosted):** Runs on top of an OS (e.g., VirtualBox, VMware Workstation). Easier setup for desktops.

OP2 — **SAN vs. NAS:**
*   **SAN (Storage Area Network):** Block-level storage. Protocols: iSCSI, Fibre Channel. Use for databases requiring low latency.
*   **NAS (Network Attached Storage):** File-level storage. Protocols: NFS, SMB/CIFS. Use for file shares and backups.

OP3 — **3-Tier vs. Leaf-Spine:**
*   **3-Tier:** Core, Distribution, Access layers. Can suffer from oversubscription and latency at the core.
*   **Leaf-Spine:** Every leaf connects to every spine. Solves oversubscription and provides consistent low latency, preferred for modern data centers.

OP4 — **Proxmox HA Architecture:**
*   **Nodes:** Minimum 3 for quorum.
*   **Quorum:** Majority vote required to make decisions.
*   **Fencing:** STONITH (e.g., iDRAC) to power off failed nodes to prevent split-brain.
*   **Storage:** Ceph or shared iSCSI/NFS.
*   **Failure:** If a node dies, VMs restart on remaining nodes automatically.

OP5 — **VLAN Trunking:** Carries multiple VLAN traffic over a single link using 802.1Q tags.
**Linux Config:** Create sub-interfaces. Example: `ip link add link eth0 name eth0.10 type vlan id 10`. Assign IP to `eth0.10`.

OP6 — **RAID Levels:**
*   **RAID 1:** Mirroring (50% capacity).
*   **RAID 5:** Striping + Parity (N-1 capacity).
*   **RAID 6:** Striping + Double Parity (N-2 capacity).
*   **RAID 10:** Mirroring + Striping (50% capacity, high speed).
*   **RAID Z2:** ZFS equivalent of RAID 6.
**Choice:** **RAID 10** (or Z2) for a DB server with 6 drives. It offers the best balance of read/write performance and fault tolerance (can survive 2 drive failures in Z2, 1 in mirrored pairs).

OP7 — **DNS Architecture:**
*   **Internal:** AD DNS servers handle internal names.
*   **External:** Public DNS servers handle public names.
*   **Split-Horizon:** Internal clients resolve internal IPs; external resolve public IPs.
*   **Forwarders:** Internal DNS forwards unknown queries to external resolvers.
*   **Laptop:** Connects via DHCP, gets DNS server IP. Resolves internal names via AD, external via forwarders.

OP8 — **PKI Chain:** Root CA (Offline) -> Subordinate CA (Online) -> End-Entity Certs.
*   **CRL/OCSP:** Check certificate revocation status.
*   **Compromise:** Subordinate CA compromised -> Revoke subordinate, issue new. Root CA compromised -> Trust chain broken, must rebuild entire PKI.

OP9 — **WAN Consolidation:**
*   **MPLS:** High reliability, high cost.
*   **IPsec:** Low cost, lower reliability/management.
*   **SD-WAN:** Dynamic path selection, QoS for VoIP, lower cost than MPLS.
*   **WireGuard:** Simple, secure, but less enterprise management.
**Recommendation:** **SD-WAN**. Best for VoIP QoS, cost efficiency, and centralized management for 50 users/site.

OP10 — **iDRAC/iLO/IPMI:** Out-of-band management interfaces allowing remote hardware control (power, console) independent of OS.
**Critical:** Essential for remote recovery if OS crashes.
**Risk:** If exposed to the internet, attackers can gain full hardware control (power cycle, BIOS access) even if the OS is secure.

OP11 — **LVM vs. LVM-Thin vs. ZFS:**
*   **LVM:** Block management, snapshots are copy-on-write.
*   **LVM-Thin:** Overprovisioning, faster snapshots.
*   **ZFS:** Filesystem + Volume manager, checksums, copy-on-write.
**Use:** ZFS for data integrity (checksums). LVM for flexible block management on standard filesystems.

OP12 — **Stretch Cluster:**
*   **Design:** 2 sites (2 nodes each) + 1 Witness node (tiebreaker).
*   **Ceph:** CRUSH rules distribute OSDs across sites.
*   **Latency:** <5ms recommended for monitors.
*   **Split-Brain:** If network fails, both sites think they are primary. Tiebreaker prevents this.
*   **Bad Idea:** 2-site Ceph without witness risks data corruption during network partition.

OP13 — **DHCP DORA:** Discover, Offer, Request, Acknowledge.
**Unavailable + Expired:** Client assigns itself an APIPA address (169.254.x.x) and cannot reach network resources.

OP14 — **Managed vs. Unmanaged Switch:**
*   **Unmanaged:** Plug-and-play.
*   **Managed:** Configurable VLANs, QoS, ACLs, SNMP monitoring.
*   **Importance:** Critical for segmentation (VLANs) and traffic prioritization (QoS) in enterprise.

OP15 — **NTP:** Network Time Protocol.
**Critical:** Required for Kerberos auth, log correlation, and cluster consensus.
**Drift:** Causes authentication failures, log confusion, and potential split-brain in clustered systems.

OP16 — **Physical vs. VM vs. Container:**
*   **Physical:** Direct hardware access.
*   **VM:** Virtualized hardware.
*   **Container:** Virtualized OS.
**Bare Metal:** Use for high I/O, low latency, or hardware passthrough (GPUs) where virtualization overhead is unacceptable.

OP17 — **UPS vs. Generator:**
*   **UPS:** Provides immediate short-term power (minutes) to allow graceful shutdown or bridge to generator.
*   **Generator:** Provides long-term power (hours/days).
*   **ATS:** Automatically switches load from utility to generator when power fails.

OP18 — **Active-Passive vs. Active-Active:**
*   **Active-Passive:** One node handles traffic; standby takes over on failure.
*   **Active-Active:** Both handle traffic.
*   **PostgreSQL HA:** Use Pacemaker/Corosync. Configure Fencing (STONITH) to kill failed node. Use Virtual IP (VIP) that floats to the active node.

OP19 — **Backup Strategy:**
*   **3-2-1:** 3 copies, 2 media types, 1 offsite.
*   **RPO/RTO:** Define acceptable data loss and downtime.
*   **Immutable:** Prevents ransomware from encrypting backups.
*   **Test:** Regularly restore to verify integrity.
*   **Backup vs. DR:** Backup is data copy; DR is the plan/process to restore operations.

OP20 — **802.1X:**
*   **Roles:** Supplicant (Client), Authenticator (Switch), Server (RADIUS).
*   **EAP:** EAP-TLS (Certificate) is more secure than PEAP (Password).
*   **Dynamic VLAN:** Assigns VLAN based on user identity.
*   **Failure:** Port is blocked or placed in a guest VLAN.
*   **MAB:** Fallback for printers/phones; authenticates using MAC address as credentials.
