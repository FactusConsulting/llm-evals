K1 — Easy: What is the difference between a Deployment and a StatefulSet?
**Deployment** manages stateless applications, providing rolling updates, scaling, and self-healing without guaranteeing pod identity or order. **StatefulSet** manages stateful applications, guaranteeing stable, unique network identities (e.g., `pod-0`, `pod-1`), ordered deployment/termination, and persistent storage retention.

K2 — Easy: How does a Kubernetes Service of type ClusterIP route traffic to pods?
A **ClusterIP** creates a virtual IP within the cluster. **kube-proxy** (using iptables or IPVS) on each node maintains network rules that map the Service IP to the current set of Pod IPs (Endpoints). Traffic to the Service IP is load-balanced across the backend Pods.

K3 — Medium: Explain the difference between a PersistentVolume, a PersistentVolumeClaim, and a StorageClass. How do they relate?
*   **PersistentVolume (PV):** A cluster-level storage resource (e.g., NFS, Cloud Disk).
*   **PersistentVolumeClaim (PVC):** A user request for storage (size/access mode).
*   **StorageClass:** A provisioner class defining how PVs are dynamically created.
*   **Relation:** A PVC binds to a PV. If no PV exists, the PVC triggers dynamic provisioning via a StorageClass.

K4 — Medium: A pod is stuck in CrashLoopBackOff. Describe your troubleshooting steps (at least 4 distinct checks).
1.  **Check Logs:** `kubectl logs <pod>` to see application errors.
2.  **Check Events:** `kubectl describe pod <pod>` for OOMKilled, probe failures, or scheduling issues.
3.  **Check Resources:** Verify if memory/CPU limits are too low causing OOM kills.
4.  **Check Probes:** Inspect liveness/readiness probe configurations (e.g., wrong port, timeout too short).

K5 — Medium: What is the role of CoreDNS in a Kubernetes cluster? What happens to DNS resolution if the CoreDNS pods are down?
CoreDNS provides DNS service discovery for Services and Pods within the cluster. If CoreDNS pods are down, internal service discovery (e.g., `my-service.default.svc`) fails. External DNS resolution may still work if the cluster uses external DNS servers, but internal communication breaks.

K6 — Hard: Explain how the Kubernetes scheduler decides which node to place a pod on. Cover: filtering, scoring, and at least 3 specific factors it considers.
The scheduler uses **Filtering** (predicates) to eliminate nodes that don't meet requirements (e.g., taints, resource capacity), then **Scoring** (priorities) to rank remaining nodes. Factors include: 1. **Resource Requests** (CPU/Mem balance), 2. **Node Affinity** (preferred/required), 3. **Pod Affinity/Anti-Affinity** (co-location or separation).

K7 — Hard: You have a pod with requests.memory: 256Mi and limits.memory: 512Mi. The node has 4GB RAM with 3.5GB already requested. Will the pod be scheduled? What happens if it actually uses 600Mi at runtime?
**Yes**, it will be scheduled. 4GB - 3.5GB = 500MB free. 256Mi (~262MB) < 500MB. At runtime, if it uses 600Mi, it exceeds the **limit** (512Mi). The kernel will trigger an **OOMKilled** event, and the pod will restart.

K8 — Hard: Explain the full lifecycle of a request hitting a Kubernetes Ingress backed by an NGINX ingress controller, all the way to the application pod. Include at least: external LB, ingress controller, Service, Endpoints, kube-proxy/iptables.
1.  **External LB:** Routes traffic to the Ingress Controller NodePort.
2.  **Ingress Controller:** NGINX reads Ingress rules, matches host/path, and forwards to the Service ClusterIP.
3.  **Service:** Routes to the **Endpoints** (Pod IPs).
4.  **kube-proxy:** Uses **iptables/IPVS** rules on the node to forward traffic to the specific Pod IP.
5.  **Pod:** Application receives the request.

K9 — Medium: What is a NetworkPolicy? If no NetworkPolicy exists in a namespace, what is the default behavior? What happens when you apply one that selects certain pods?
A **NetworkPolicy** restricts traffic flow between pods. Default behavior (no policy): **Allow all** traffic. When a policy is applied to specific pods, it acts as a **default deny** for those pods regarding traffic not explicitly allowed by the policy rules.

K10 — Medium: Explain the difference between RollingUpdate and Recreate deployment strategies. When would you choose Recreate?
**RollingUpdate** replaces pods gradually to ensure zero downtime. **Recreate** kills all existing pods before starting new ones. Choose **Recreate** for stateful applications where simultaneous instances cause conflicts (e.g., database locks) or when the application cannot handle multiple versions running at once.

K11 — Easy: What is a Kubernetes namespace, and what are the default namespaces created in a new cluster?
A **Namespace** provides logical isolation within a cluster (resource grouping, quotas, RBAC). Default namespaces: `default`, `kube-system`, `kube-public`, `kube-node-lease`.

K12 — Easy: What is `kubectl describe pod <name>` used for? Name at least 3 pieces of information it shows that `kubectl get pod` does not.
It provides detailed diagnostic information. `kubectl get pod` only shows status/age. `describe` shows: 1. **Events** (scheduling, pull errors), 2. **Conditions** (Ready, Initialized), 3. **Mounts/Env Vars** (detailed configuration).

K13 — Easy: What is a ConfigMap, and how can a pod consume it? Name at least two different methods.
A **ConfigMap** stores non-sensitive configuration data as key-value pairs. Pods consume it via: 1. **Environment Variables**, 2. **Volume Mounts** (files).

K14 — Easy: What is the difference between a Service of type ClusterIP, NodePort, and LoadBalancer?
*   **ClusterIP:** Internal virtual IP (default).
*   **NodePort:** Exposes service on a static port on every node's IP.
*   **LoadBalancer:** Provisions an external cloud load balancer to expose the service externally.

K15 — Easy: What is a DaemonSet, and when would you use one instead of a Deployment?
A **DaemonSet** ensures a copy of a pod runs on all (or selected) nodes. Use it for **node-level agents** like logging collectors (Fluentd), monitoring agents (Prometheus Node Exporter), or network plugins.

K16 — Medium: What are taints and tolerations? How do they differ from node affinity, and give a practical use case for each?
**Taints** repel pods from a node; **Tolerations** allow pods to schedule on tainted nodes. **Node Affinity** attracts pods to nodes. Use Case: **Taint** a node for GPU workloads (only pods with GPU tolerations go there). **Affinity** to co-locate web and cache pods on the same zone.

K17 — Medium: Explain what a Kubernetes Operator is. How does it differ from a standard controller, and what problems does the Operator pattern solve?
An **Operator** is a custom controller that manages complex stateful applications using Custom Resource Definitions (CRDs). Unlike standard controllers (which manage generic resources), Operators encode **human operational knowledge** (e.g., backup, scaling, upgrades) to automate lifecycle management of specific apps (e.g., Kafka, PostgreSQL).

K18 — Hard: Explain the etcd consistency model used by Kubernetes. What happens to the cluster if the etcd quorum is lost? How do you recover from a situation where 2 out of 3 etcd nodes are permanently lost?
etcd uses **strong consistency** (linearizable). If quorum is lost (e.g., 2/3 nodes down), the cluster becomes **read-only/unavailable** (cannot write). Recovery: Restore from a **snapshot backup** on a new node, then join it to the cluster using `etcdctl member add` and restart the cluster.

K19 — Hard: Describe how Kubernetes RBAC works. Explain the relationship between Role, ClusterRole, RoleBinding, and ClusterRoleBinding. A developer needs read-only access to pods and logs in a single namespace — write the RBAC manifests.
**RBAC** uses Roles (permissions) and Bindings (subjects). **Role** is namespace-scoped; **ClusterRole** is cluster-wide. **RoleBinding** links a Role to a subject in a namespace.
```yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  namespace: default
  name: pod-reader
rules:
- apiGroups: [""]
  resources: ["pods", "pods/log"]
  verbs: ["get", "list", "watch"]
---
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  name: read-pods-binding
  namespace: default
subjects:
- kind: User
  name: developer
  apiGroup: rbac.authorization.k8s.io
roleRef:
  kind: Role
  name: pod-reader
  apiGroup: rbac.authorization.k8s.io
```

K20 — Hard: You notice pod DNS resolution is slow (>5s) for external domains. Explain how ndots in /etc/resolv.conf inside a pod causes excessive DNS queries, and how you would fix it using dnsConfig in the pod spec.
**ndots** (default 5) dictates how many dots must appear in a name before it is tried as a fully qualified domain name (FQDN). If `ndots=5`, a query for `google.com` triggers 5 queries (adding `.default.svc.cluster.local`, etc.). Fix: Set `dnsConfig` in pod spec with `ndots: 1` to reduce suffix queries.

D1 — Easy: What is the difference between git rebase and git merge? When should you avoid rebasing?
**Rebase** rewrites history to create a linear commit sequence. **Merge** preserves history with a merge commit. Avoid rebasing **public/shared branches** as it rewrites history and disrupts collaborators.

D2 — Easy: Explain the difference between TCP and UDP. Name 2 protocols that use each.
**TCP** is connection-oriented, reliable, ordered. **UDP** is connectionless, fast, unreliable. TCP: HTTP, SSH. UDP: DNS, VoIP.

D3 — Medium: What is a race condition? Give a concrete example in any language.
A **race condition** occurs when output depends on the sequence/timing of uncontrollable events. Example: Two threads increment a shared counter `x++` without a lock. Final value may be less than expected due to lost updates.

D4 — Medium: Explain the difference between symmetric and asymmetric encryption. How are both used together in TLS?
**Symmetric** uses one key for encryption/decryption (fast). **Asymmetric** uses public/private key pairs (secure key exchange). **TLS:** Asymmetric keys exchange a session key; Symmetric key encrypts the actual data transfer.

D5 — Medium: What is a container image layer? How does layer caching affect Dockerfile best practices?
A **layer** is a read-only filesystem diff. **Caching:** Place stable instructions (e.g., `COPY dependencies`) before volatile ones (e.g., `COPY source code`) to leverage cache and speed up builds.

D6 — Medium: You have a REST API returning HTTP 503 intermittently. Describe at least 4 things you would check, and in what order.
1.  **Upstream Health:** Are backend services healthy?
2.  **Database Connectivity:** Is the DB connection pool exhausted?
3.  **Resource Limits:** Are pods hitting CPU/Memory limits (OOM)?
4.  **Rate Limiting:** Is the API hitting rate limits or queue depth limits?

D7 — Hard: Explain the CAP theorem. Give a real-world example of a system that chooses CP over AP, and one that chooses AP over CP.
**CAP:** Consistency, Availability, Partition Tolerance. You can only guarantee 2 of 3. **CP:** Zookeeper, HBase (prioritize consistency during partition). **AP:** Cassandra, DynamoDB (prioritize availability during partition).

D8 — Medium: What is the difference between horizontal and vertical scaling? What architectural patterns enable horizontal scaling?
**Horizontal:** Add more nodes/instances. **Vertical:** Increase resources (CPU/RAM) of existing node. Patterns enabling Horizontal: **Stateless services**, **Sharding**, **Load Balancing**.

D9 — Easy: What is an environment variable? How do you set one that persists across shell sessions on Linux?
An **environment variable** is a dynamic value affecting process behavior. To persist: Add `export VAR=value` to `~/.bashrc` or `~/.profile`.

D10 — Easy: What is JSON, and how does it differ from YAML? Name one advantage of each format.
**JSON** is strict key-value pairs. **YAML** is human-readable with indentation. **JSON:** Machine parsing standard. **YAML:** Easier for humans to write config files.

D11 — Easy: What is a REST API? Explain the difference between GET, POST, PUT, and DELETE HTTP methods.
**REST** is an architectural style using HTTP methods. **GET:** Retrieve data. **POST:** Create resource. **PUT:** Update/Replace resource. **DELETE:** Remove resource.

D12 — Easy: What is version pinning in dependency management, and why is it important for reproducible builds?
**Version pinning** specifies exact dependency versions (e.g., `package==1.0.0`). Important for **reproducibility** to ensure builds don't break due to unexpected upstream updates.

D13 — Easy: What is the difference between a compiled language and an interpreted language? Give two examples of each.
**Compiled:** Source code converts to binary before execution. **Interpreted:** Code executes line-by-line at runtime. Compiled: C, Go. Interpreted: Python, JavaScript.

D14 — Medium: Explain what a reverse proxy is and how it differs from a forward proxy. Name two common reverse proxy servers and a use case for each.
**Reverse Proxy:** Client -> Proxy -> Server (hides backend). **Forward Proxy:** Client -> Proxy -> Internet (hides client). Servers: **Nginx** (Web serving/SSL), **HAProxy** (Load balancing).

D15 — Medium: What is a deadlock? What four conditions (Coffman conditions) must hold simultaneously for a deadlock to occur?
**Deadlock:** Processes blocked waiting for each other. Conditions: 1. **Mutual Exclusion**, 2. **Hold and Wait**, 3. **No Preemption**, 4. **Circular Wait**.

D16 — Hard: Explain the concept of eventual consistency. How do distributed databases like Cassandra handle read-after-write consistency, and what tunable consistency levels exist?
**Eventual Consistency:** Updates propagate asynchronously; reads may return stale data temporarily. **Cassandra:** Uses **Tunable Consistency** (ONE, QUORUM, LOCAL_QUORUM, ALL). For read-after-write, use **LOCAL_QUORUM** to ensure local data is read.

D17 — Hard: What is mTLS (mutual TLS)? Explain the full handshake process, including how both client and server authenticate each other. Where is mTLS commonly used in infrastructure?
**mTLS:** Both client and server present certificates. **Handshake:** Server sends cert -> Client verifies -> Client sends cert -> Server verifies. Used in **Service Meshes** (Istio) and internal API security.

D18 — Hard: Explain how a hash table works internally, including collision resolution strategies (chaining vs. open addressing). What is the time complexity for insertion, lookup, and deletion in average and worst cases?
**Hash Table:** Array indexed by hash of key. **Collision:** **Chaining** (linked list at index) or **Open Addressing** (probe next slot). **Complexity:** Average O(1). Worst O(n) (all keys collide).

D19 — Hard: What is a circuit breaker pattern in microservices? Explain its three states (closed, open, half-open) and how it prevents cascade failures. How does it relate to retry policies and exponential backoff?
**Circuit Breaker:** Prevents repeated calls to failing services. **States:** Closed (normal), Open (fail fast), Half-open (test). Prevents cascade by stopping traffic to failing deps. **Retry:** Stop retrying when Open. **Backoff:** Used during Half-open to space out test requests.

D20 — Hard: Explain the difference between optimistic and pessimistic concurrency control in databases. When would you choose each? Give a concrete example using database features (e.g., row versioning, SELECT FOR UPDATE).
**Optimistic:** Check conflicts at commit time (row versioning). **Pessimistic:** Lock rows during read (SELECT FOR UPDATE). Choose **Optimistic** for high concurrency/low conflict. Choose **Pessimistic** for high conflict/financial transactions.
