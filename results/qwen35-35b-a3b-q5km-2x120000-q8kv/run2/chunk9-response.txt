SC1-A
The pods are failing because the **Restricted** Pod Security Standard (PSS) forbids running containers as root.
1.  **PSS Interaction:** Restricted PSS requires `securityContext.runAsNonRoot: true` AND `securityContext.runAsUser` to be set to a non-zero UID.
2.  **The Failure:** The new base image (`cgr.dev/chainguard/static:latest`) defaults `USER` to root (UID 0). The pod spec sets `runAsNonRoot: true` but omits `runAsUser`.
3.  **Baseline vs. Restricted:** Baseline allows `runAsNonRoot: true` without `runAsUser` (defaults to image USER). Restricted enforces that the effective UID must be non-zero. Since the image defaults to root and no override is provided, the container attempts to run as root, violating the Restricted policy, causing the kubelet to reject the pod or the container to crash immediately upon startup depending on the specific enforcement mode (RuntimeClass vs Admission). In this case, the crash loop is likely due to the container runtime enforcing the policy or the application failing permission checks immediately.

SC1-C
**Remediation Strategy:**
1.  **Image Fix:** Rebuild the base image or override the `USER` in the Dockerfile to a non-root UID (e.g., `USER 1000`).
2.  **Manifest Fix:** Add `securityContext.runAsUser: 1000` to the pod spec to explicitly satisfy Restricted PSS.
3.  **CI Gate:** Integrate `kube-score` or `conftest` in the CI pipeline to validate manifests against the Restricted PSS policy before merging.
4.  **Rollout:** Use a **Canary Deployment**. Deploy the fixed version to 5% of pods first. Monitor CrashLoopBackOff rates and error logs. If stable, gradually increase to 100%. This ensures no downtime as healthy pods continue serving traffic while the new version is validated.

SC2-A
**Stale Data & Replication:**
1.  **Cause:** Cross-region replication is asynchronous. When the primary (us-east-1) fails, the replica (eu-west-1) is promoted. Any WAL (Write-Ahead Log) entries generated on the primary but not shipped to the replica before the outage are lost.
2.  **WAL Shipping:** PostgreSQL streams WAL records. Across regions, network latency and bandwidth limit throughput.
3.  **Data Loss:** Uncommitted transactions or committed transactions not yet replicated are lost.
4.  **Quantification:** Check `pg_stat_replication` on the old primary (if accessible) or the replica's `pg_last_wal_receive_lsn` vs `pg_last_wal_replay_lsn` before promotion. The difference indicates the lag window.

SC2-B
1.  **Import Replica:** `terraform import aws_db_instance.promoted_replica <new_replica_arn>`
2.  **Remove Old:** `terraform state rm aws_db_instance.old_primary` (This removes the old resource from state without destroying it, allowing you to manage the new one).
3.  **Update State:** Run `terraform plan` to ensure the state matches the promoted replica.
4.  **Dependencies:** Update Terraform variables referencing the old ARN to the new ARN. Run `terraform apply` to update dependent resources (Security Groups, Route53) to point to the new instance.

SC2-C
**Architecture Design:**
1.  **Connection Management:** Use **RDS Proxy** instead of PgBouncer. It handles connection pooling at the AWS layer, supports IAM auth, and retries automatically on failover without app changes.
2.  **DNS Failover:** Use **Route53 Health Checks** with Failover Routing Policy. Point the primary endpoint to us-east-1 and secondary to eu-west-1.
3.  **Retry Logic:** Implement exponential backoff with jitter in the application connection pool.
4.  **RPO < 1s:** Standard RDS async replication cannot guarantee <1s RPO across regions reliably. **Recommend Aurora Global Database.** It uses a shared storage layer with synchronous replication to the primary region and async to secondary, offering RPO < 1 second and faster failover (typically < 60s).

SC3-A

SC4-A
**Top 5 Blockers:**
1.  **Local Sessions:** Sticky sessions via F5 break stateless K8s scaling. **Fix:** Move sessions to Redis (ElastiCache).
2.  **NFS Mounts:** Local NFS `/data/uploads` breaks pod mobility. **Fix:** Use EFS or S3 with S3FS/MinIO CSI.
3.  **Oracle DB:** On-prem Oracle is stateful. **Fix:** Use Oracle Cloud or migrate to PostgreSQL/Aurora.
4.  **IBM MQ:** Mainframe integration is stateful. **Fix:** Use MQ Kubernetes Operator or bridge via API.

SC5-C
**Architectural Solution:**
1.  **CQRS:** Separate read/write models. Reconciliation writes to a write model; checkout reads from a read replica.
2.  **Read Replica:** Run reconciliation job against a read replica to avoid locking production tables.
3.  **Bulkhead:** Configure separate connection pools for reconciliation jobs vs. serving traffic.
4.  **Scheduling:** Move CronJob to off-peak hours (e.g., 03:00 UTC).
5.  **PDBs:** Set `PodDisruptionBudget` for inventory-service to ensure at least 1 pod remains available during the CronJob's resource spike. Use `resourceQuotas` to limit the CronJob's CPU/Memory.

SC6-C
**Connectivity Upgrade:**
1.  **AWS Direct Connect:**
    *   **Pros:** Dedicated bandwidth, lower latency, consistent throughput.
    *   **Cons:** High cost, physical provisioning.
    *   **Zero Trust:** Fits well; traffic is encrypted (MACsec) and isolated.
2.  **VPN ECMP:**
    *   **Pros:** Faster deployment, lower cost.
    *   **Cons:** Shared bandwidth, higher latency variance.
    *   **Zero Trust:** IPsec encryption maintained.
3.  **Mesh (Tailscale/Netbird):**
    *   **Pros:** Zero config, identity-based access.
    *   **Cons:** Performance overhead, dependency on third-party control plane.
    *   **Zero Trust:** Native identity-based routing.
**Routing:** DX requires BGP peering. VPN requires static routes. Mesh requires agent routing tables. **Recommendation:** DX for production core, Mesh for developer access.

SC10-A
**Multi-Cluster Architecture:**
1.  **Active-Passive (Velero):**
    *   **RPO/RTO:** RPO depends on backup frequency (15 mins). RTO depends on restore time.
    *   **Feasibility:** 200TB restore is slow. **Not feasible** for 1-hour RTO without pre-warmed state.
2.  **Active-Active (Istio/Cilium):**
    *   **RPO/RTO:** Requires synchronous replication. 12ms latency allows sync.
    *   **Feasibility:** Better for RTO.
3.  **Replication:**
    *   **PostgreSQL:** Patroni with async replication (RPO < 15m).
    *   **Redis:** Sentinel with cross-region replication.
    *   **RabbitMQ:** Federation or Quorum Queues.
    *   **MinIO:** Site Replication (S3 compatible).
**Recommendation:** Active-Active with Async Replication for DBs to meet RPO 15m.

SC10-C
**Failover Orchestration:**
1.  **Detection:** Prometheus monitors site health. If Chicago nodes unreachable for 2 mins -> Site Down.
2.  **DNS:** GSLB (Route53) switches traffic to Dallas.
3.  **DB:** Patroni promotes Dallas replica (Synchronous commit ensures no data loss).
4.  **Cache/Queue:** Redis Sentinel promotes master. RabbitMQ mirrors queues.
5.  **Storage:** MinIO Site Replication catches up.
6.  **State:** Ansible/Tofu updates state to mark Dallas as Primary.
**Split-Brain:** Use **Etcd** quorum. If Chicago is down, Etcd in Dallas cannot form quorum with Chicago, preventing double promotion.
**Failback:** Once Chicago recovers, re-establish replication. Promote Chicago back to Primary manually or via automation after data sync.
