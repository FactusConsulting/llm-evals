G1 — Easy: An array is a fixed-size value type `[T]N`, while a slice is a dynamic view over an array with metadata (pointer, length, capacity). When appending to a slice that exceeds its capacity, Go allocates a new, larger underlying array, copies the existing elements, and updates the slice header to point to the new array.

G2 — `var x int` declares a variable with a zero value; `x := 0` declares and assigns. You cannot use `:=` outside a function body (e.g., at package level) or if you are redeclaring a variable with the same name in the same scope.

G3 — A goroutine is a lightweight thread managed by the Go runtime, not the OS. The Go scheduler uses the GMP model: **G** (Goroutine), **M** (OS Thread), and **P** (Processor, logical CPU context). The scheduler multiplexes Gs onto Ms using Ps to maximize CPU utilization.

G4 — Channels are typed conduits for communication between goroutines. Unbuffered channels block both sender and receiver until the other side is ready. Buffered channels have a queue; sending blocks only if the buffer is full. Receiving blocks if the buffer is empty.

G5 — The `context` package manages deadlines, cancellation, and request-scoped values.
```go
ctx, cancel := context.WithTimeout(context.Background(), 5*time.Second)
defer cancel()
resp, err := http.GetContext(ctx, "http://example.com")
```

G6 — Go interfaces are satisfied implicitly (duck typing), unlike Java/C# where explicit declaration is required. The empty interface `interface{}` (or `any`) accepts any type. It is powerful for generic storage but dangerous because it requires type assertions, risking panics if the type is incorrect.

G7 — This code has a data race: one goroutine writes `data` and `ready`, another reads `ready` and `data` without synchronization. The compiler cannot guarantee visibility. Fix using `sync/atomic` for `ready` and `data`, or a `sync.Mutex`/`sync.WaitGroup` to ensure happens-before ordering.

G8 — `sync.Mutex` provides exclusive access; `sync.RWMutex` allows multiple readers or one writer. Use `sync.Map` when reads vastly outnumber writes and keys are static, as it avoids mutex contention. However, `sync.Map` is slower for writes and iteration compared to a regular `map` with a mutex.

G9 — Go modules manage dependencies. `go.mod` declares the module path and dependencies; `go.sum` stores cryptographic checksums. `go mod tidy` removes unused dependencies and adds missing ones. MVS (Minimum Version Selection) ensures the lowest compatible versions of dependencies are selected to avoid conflicts.

G10 — `defer` schedules a function call to run after the surrounding function returns. Multiple defers execute in LIFO (Last In, First Out) order. A common gotcha in loops is capturing the loop variable by reference; use a local copy (e.g., `i := i`) inside the loop to capture the current value.

G11 — A pointer holds a memory address. Its zero value is `nil`. Create a pointer to a struct literal using `&MyStruct{Field: value}`.

G12 — `init()` runs automatically before `main()` executes. A package can have multiple `init()` functions; they execute in the order they appear in the source file.

G13 — `new(T)` allocates zeroed memory for type `T` and returns a pointer `*T`. `make(T)` initializes built-in types (slice, map, channel) and returns the initialized value (not a pointer).

G14 — Errors are returned as values. The `error` interface has a single `Error() string` method. Idiomatic pattern: `if err != nil { return err }`. `errors.Is` checks if an error matches a target (wrapping), while `errors.As` checks if an error can be cast to a specific type.

G15 — Built-in composite types: `map` (nil), `slice` (nil), `channel` (nil), `struct` (zero values for fields).

G16 — A race condition occurs when concurrent goroutines access shared data without synchronization. The `-race` flag instruments code to detect data races at runtime. Fix by using `sync.Mutex` to protect the shared variable or channels for communication.

G17 — `select` waits on multiple channel operations. Unlike `switch`, it evaluates channel readiness dynamically. If multiple cases are ready, one is chosen pseudo-randomly. Implement timeout using `case <-time.After(duration)`.

G18 — Escape analysis determines if a variable lives on the stack or heap. If a variable's address escapes the function scope, it is allocated on the heap. Inspect with `go build -gcflags=-m`. Heap allocation increases GC pressure; stack allocation is faster.

G19 — `reflect.Type` describes the type; `reflect.Value` holds the value. Reflection is slow due to runtime type inspection and dynamic dispatch. It is justified for frameworks (e.g., serialization, dependency injection) but avoided in performance-critical paths.

G20 — `go:embed` embeds files/directories at compile time into a `fs.FS`. It works with single files, directories, and glob patterns. Limitations: files must exist at compile time. It integrates with `io/fs` to provide a virtual file system interface.

R1 — Ownership ensures memory safety without garbage collection. Rules: 1) Each value has exactly one owner. 2) When the owner goes out of scope, the value is dropped. 3) Values can be moved, transferring ownership.

R2 — `String` is an owned, heap-allocated, mutable string. `&str` is a borrowed, immutable string slice. Use `&str` for function arguments to avoid copying; use `String` when ownership or mutation is required.

R3 — Borrowing allows temporary access without ownership. Rules: 1) You can have any number of immutable references (`&T`) OR exactly one mutable reference (`&mut T`). 2) References must always be valid. 3) You cannot mutate through an immutable reference.

R4 — `Result<T, E>` represents success or error; `Option<T>` represents a value or `None`. The `?` operator propagates errors.
```rust
fn read_file() -> Result<String, std::io::Error> {
    std::fs::read_to_string("file.txt")
}
```

R5 — `Box<T>` owns data on the heap. `Rc<T>` (Reference Counted) allows multiple owners in a single thread. `Arc<T>` (Atomic Reference Counted) allows multiple owners across threads. Use `Arc` when sharing data between threads.

R6 — Traits define shared behavior. `dyn Trait` enables dynamic dispatch (vtable) at runtime; `T: Trait` enables static dispatch (monomorphization) at compile time. Static dispatch is generally faster.

R7 — The function fails because the compiler cannot determine which input lifetime the return value borrows. Fix by adding a lifetime parameter: `fn longest<'a>(x: &'a str, y: &'a str) -> &'a str`.

R8 — `Send` means a type can be transferred between threads; `Sync` means references can be shared. `Rc<T>` is not `Send` because its reference count is not atomic. `Arc<T>` uses atomic reference counting, making it `Send` and `Sync`.

R9 — `async/await` creates a `Future` (state machine). The executor (e.g., Tokio) polls the future until completion. Unlike Go's preemptive goroutines, Rust's async is cooperative; the executor controls scheduling.

R10 — `unsafe` allows operations the compiler cannot verify. Examples: dereferencing raw pointers, calling unsafe functions, modifying `static mut`. Justified for FFI, low-level optimizations, or implementing safe abstractions.

R11 — `match` exhaustively checks all patterns; `if let` checks a single pattern. The compiler enforces exhaustiveness in `match`, ensuring no logic is missed for enums.

R12 — `Vec<T>` is a growable heap array; `[T; N]` is a fixed-size stack array. When `Vec` capacity is exceeded, it allocates a new, larger buffer and moves elements.

R13 — Enums in Rust are Algebraic Data Types (ADTs) that can hold data. `Option<T>` is an enum with `Some(T)` or `None`. Rust avoids `null` to prevent null pointer exceptions at compile time.

R14 — `impl` defines methods. `&self` borrows, `&mut self` mutates, `self` consumes the value. Associated functions (e.g., `String::new()`) do not take `self` and act as constructors.

R15 — `derive` macros generate boilerplate trait implementations. Common: `Debug`, `Clone`, `PartialEq`. Cannot derive if fields contain types that don't support the trait (e.g., lifetimes in `Clone`).

R16 — `Cow<'a, T>` (Clone on Write) borrows data if possible, cloning only when mutation occurs. Useful for functions that accept borrowed data but might need to return owned data, avoiding unnecessary allocations.

R17 — `mod` declares a module, `use` imports paths, `pub` exports visibility. File system maps to modules (e.g., `mod.rs` or `mod/`). Rust 2018 improved path resolution and module visibility rules compared to 2015.

R18 — `Pin<T>` prevents a value from being moved in memory. Needed for self-referential structs and `Future` implementations where the future borrows its own fields. `Unpin` indicates a type can be safely moved.

R19 — Declarative macros (`macro_rules!`) transform syntax. Procedural macros (derive, attribute) transform AST. Hygiene ensures macro variables don't conflict with local variables.

R20 — Zero-cost abstractions mean high-level code compiles to efficient machine code. Generics use monomorphization (copying code per type). Iterators are lazy and inlined. No runtime overhead compared to C.
