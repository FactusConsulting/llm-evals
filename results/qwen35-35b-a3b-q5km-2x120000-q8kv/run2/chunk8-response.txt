CL1 — **IaaS, PaaS, SaaS:**
*   **IaaS (Infrastructure as a Service):** Provides virtualized computing resources (VMs, storage, networks). You manage OS, runtime, data, and apps. *Example:* AWS EC2.
*   **PaaS (Platform as a Service):** Provides hardware and software tools (runtime, middleware) for developers. You manage data and apps. *Example:* Google App Engine.
*   **SaaS (Software as a Service):** Provides fully managed software applications. You manage nothing but user access. *Example:* Salesforce.

CL2 — **Shared Responsibility Model:**
*   **Provider:** Manages physical infrastructure, hypervisor, and host OS.
*   **IaaS VM:** Provider secures hardware/hypervisor. You secure Guest OS, network config, firewall, and data.
*   **Managed Kubernetes:** Provider secures control plane and nodes. You secure worker node OS, pod security, network policies, and app code.
*   **SaaS (M365):** Provider secures everything. You manage user identity, data classification, and device compliance.

CL3 — **Networking Components:**
*   **VPC:** Isolated virtual network in the cloud.
*   **Subnet:** Segment of a VPC (public/private) mapped to an Availability Zone.
*   **Security Group (SG):** Stateful virtual firewall at the instance level (allow rules only).
*   **NACL:** Stateless virtual firewall at the subnet level (allow/deny rules).
*   **Layering:** VPC contains Subnets; Subnets contain Instances; SG protects Instances; NACL protects Subnets.
*   **Stateful vs Stateless:** SG tracks connection state (return traffic allowed automatically). NACL does not track state (must explicitly allow return traffic).

CL4 — **Multi-Account Strategy:**
*   **Landing Zone:** Pre-configured environment with baseline security, networking, and identity.
*   **Organizational Units (OUs):** Logical grouping (e.g., Prod, Dev, Security) for policy application.
*   **Guardrails:** SCPs (Service Control Policies) to prevent risky actions (e.g., disabling CloudTrail).
*   **Shared Networking:** Transit Gateway (AWS) or Hub-and-Spoke (Azure) connects accounts centrally.
*   **Centralized Logging:** All logs sent to a dedicated Security/Logging account.
*   **Single-Account Risks:** Blast radius of compromise, permission sprawl, noisy neighbors affecting performance, lack of isolation for compliance.

CL5 — **Infrastructure as Code (IaC):**
*   **Terraform/OpenTofu:** Declarative, multi-cloud, state-file based. *Choose:* Multi-cloud, complex state dependencies.
*   **Pulumi:** Imperative/Declarative, uses general languages (TS, Python). *Choose:* Teams with strong software engineering skills.
*   **CloudFormation/Bicep:** Native to AWS/Azure. *Choose:* Single-cloud, tight integration with native services.
*   **Crossplane:** Kubernetes-native IaC. *Choose:* Managing cloud resources from within a K8s cluster.

CL6 — **Autoscaling Patterns:**
*   **HPA:** Scales pods based on CPU/Memory metrics.
*   **VPA:** Scales pod resource requests/limits (requires pod restart).
*   **Cluster Autoscaler:** Adds/removes nodes based on pending pods.
*   **KEDA:** Scales based on external events (e.g., Kafka queue depth, AWS SQS).
*   **KEDA vs HPA:** Use KEDA when scaling is driven by non-standard metrics (queues, events) rather than just CPU/Memory.

CL7 — **Disaster Recovery (DR) Strategy:**
*   **RPO/RTO:** RPO (data loss tolerance) dictates replication sync (sync for low RPO). RTO (recovery time) dictates active-active vs passive.
*   **Active-Active:** Both regions serve traffic. High cost, low RTO/RPO.
*   **Active-Passive:** One region active, one standby. Lower cost, higher RTO.
*   **Database:** Async replication for cost/latency; Sync for consistency.
*   **DNS Failover:** Route53 Health Checks trigger failover.
*   **State:** Externalize state (S3, DynamoDB) to avoid local state loss.
*   **Models:** Pilot Light (DB running, app off), Warm Standby (Scaled down app), Multi-site (Active-Active).

CL8 — **GitOps:**
*   **Model:** Git repository is the single source of truth. Controllers sync cluster state to Git state.
*   **ArgoCD vs Flux:** ArgoCD has a UI, Flux is CLI-first. Both use reconciliation loops.
*   **Secrets:** Sealed Secrets (encrypted in Git), External Secrets Operator (fetches from Vault/AWS Secrets Manager).
*   **Drift:** If manual change occurs, the controller detects drift and reverts the cluster to match Git.

CL9 — **Stateful Legacy App on K8s:**
*   **StatefulSet:** Ensures stable pod identity and ordered startup/deletion.
*   **PV/PVC:** Use `local` storage class for low latency (node-local disk).
*   **Headless Service:** Provides stable DNS names for StatefulSet pods.
*   **Init Containers:** Prepare filesystem or config before main app starts.
*   **ConfigMaps:** Mount as volumes for config files at specific paths.
*   **PDB:** Ensure minimum available pods during disruptions to maintain quorum.

CL10 — **Identity & Credentials:**
*   **Service Account:** K8s identity for pods.
*   **Workload Identity:** Maps K8s SA to cloud IAM role (GCP/Azure).
*   **IRSA:** Maps K8s SA to AWS IAM role via OIDC.
*   **Mounting Credentials:** Bad practice (leaks secrets, hard to rotate, violates least privilege).
*   **Federation:** Uses OIDC tokens to exchange K8s identity for temporary cloud credentials without long-lived secrets.

CL11 — **Storage Types:**
*   **Object (S3/Blob):** Unstructured data, high durability (11 9s), low latency. *Use:* Backups, media, logs.
*   **Block (EBS/Disk):** Attached to VM, low latency, persistent. *Use:* OS, databases.
*   **File (EFS/Azure Files):** Shared file system, NFS/SMB. *Use:* Shared configs, legacy apps.
*   **Durability:** Object storage is highest (replicated across AZs/Regions). Block is AZ-specific unless snapshotted.

CL12 — **Zero Trust K8s:**
*   **Network Policies:** Deny all by default, allow specific pod-to-pod traffic.
*   **Service Mesh (mTLS):** Encrypts service-to-service traffic (Istio/Linkerd).
*   **Pod Identity:** Workload identity federation (no static keys).
*   **Admission Controllers:** OPA/Kyverno to enforce policies on deployment.
*   **Image Signing:** Cosign/Notary to verify image integrity.
*   **Runtime Security:** Falco to detect anomalous behavior.
*   **Perimeter vs Zero Trust:** Perimeter trusts inside; Zero Trust verifies every request regardless of location.

CL13 — **CDN:**
*   **Definition:** Distributed network of edge servers caching content.
*   **Mechanism:** Caches static assets at edge locations closer to users.
*   **Services:** CloudFront, Azure CDN.
*   **Use CDN:** For static content (images, JS, CSS) to reduce latency and origin load.
*   **Use Origin:** For dynamic, personalized, or real-time data not suitable for caching.

CL14 — **AZs and Regions:**
*   **Region:** Geographic area with multiple AZs.
*   **AZ:** Isolated data center with independent power/cooling.
*   **Multi-AZ:** Protects against single data center failure.
*   **Zonal:** Resources tied to one AZ (e.g., EBS).
*   **Regional:** Resources available across AZs (e.g., S3, RDS Multi-AZ).

CL15 — **Managed Database:**
*   **Definition:** DBaaS (RDS, Cloud SQL).
*   **Provider Tasks:** Patching, backups, failover, hardware provisioning, monitoring.
*   **User Tasks:** Schema, queries, data, access control.
*   **Benefit:** Reduces operational overhead and improves availability.

CL16 — **IAM & Least Privilege:**
*   **Users:** Human identities.
*   **Groups:** Collections of users for policy assignment.
*   **Roles:** Temporary credentials for services/users (no long-term keys).
*   **Policies:** JSON permissions defining actions/resources.
*   **Least Privilege:** Grant only necessary permissions. Apply via Roles and scoped policies, not admin access.

CL17 — **Serverless:**
*   **Definition:** Event-driven compute without managing servers (Lambda, Functions).
*   **Cold Starts:** Latency when a function is invoked after inactivity (initialization).
*   **Poor Fit:** Long-running processes, high stateful requirements, strict latency needs, predictable high throughput (reserved instances cheaper).

CL18 — **Ingress Options:**
*   **Ingress Resource:** Standard K8s resource for HTTP routing.
*   **Gateway API:** Next-gen standard, supports TCP/UDP, better separation of concerns.
*   **Cloud LB:** Provider-specific annotations (e.g., AWS NLB).
*   **Mesh Gateway:** Service mesh ingress (Istio).
*   **TLS Termination:** At LB (Cloud), Ingress Controller, or Mesh.
*   **Gateway API:** Use for complex routing, multi-cluster, or TCP/UDP needs beyond HTTP.

CL19 — **Cost Optimization ($80k/mo):**
*   **RI/SP:** Commit to usage for 1-3 years (20-70% savings).
*   **Spot/Preemptible:** Use for fault-tolerant workloads (up to 90% savings).
*   **Right-sizing:** Match instance size to actual CPU/Memory usage.
*   **Storage Tiering:** Move old data to Glacier/Archive.
*   **Idle Detection:** Find unattached EBS, idle LBs.
*   **FinOps:** Cross-team accountability, showback/chargeback dashboards.
*   **Tools:** AWS Cost Explorer, CloudHealth, Kubecost.

CL20 — **Multi-Cloud Strategy:**
*   **Abstraction:** Kubernetes (runtime), Terraform (IaC), Crossplane (API).
*   **Networking:** Direct Connect/ExpressRoute or VPN for hybrid/multi-cloud.
*   **Identity:** Federation (OIDC) to central IdP.
*   **Trade-offs:** Increased complexity, networking latency, data egress costs, skill fragmentation.
*   **Justified:** Regulatory requirements, avoiding single-vendor outage, leveraging specific best-of-breed services.
*   **Unnecessary:** If one cloud meets all needs, multi-cloud adds risk without value.

OT1 — **Purdue Model:**
*   **Level 0:** Physical Process (Sensors/Actuators).
*   **Level 1:** Basic Control (PLCs/RTUs).
*   **Level 2:** Area Supervisory (SCADA/HMI).
*   **Level 3:** Site Operations (MES/Production).
*   **Level 4:** Enterprise (ERP/Business).

OT2 — **IT vs OT:**
*   **IT:** Confidentiality/Integrity focus, frequent patching, high availability.
*   **OT:** Safety/Availability focus, stability over security, legacy systems.
*   **Patching:** OT patching requires validation to prevent downtime/safety hazards; IT patches are routine.

OT3 — **DMZ (IDMZ):**
*   **Role:** Buffer zone between IT (Level 4) and OT (Level 0-2).
*   **Traffic:** Unidirectional or strictly controlled bidirectional (e.g., OPC UA pull).
*   **Direct L4-L1 Risk:** Allows IT malware to reach control layer directly, risking physical safety.

OT4 — **OT Protocols:**
*   **Modbus TCP:** Level 1-2, Master/Slave, unencrypted.
*   **OPC UA:** Level 2-3, Secure, platform-independent.
*   **EtherNet/IP:** Level 1-2, CIP protocol over Ethernet.
*   **PROFINET:** Level 1-2, Real-time industrial Ethernet.
*   **MQTT:** Level 2-4, Pub/Sub for telemetry to cloud.

OT5 — **Data Diode vs Firewall:**
*   **Firewall:** Filters traffic bidirectionally.
*   **Data Diode:** Hardware-enforced unidirectional flow (OT to IT only).
*   **Use Case:** High-security telemetry export where IT cannot send commands back.
*   **Limitation:** Cannot receive commands or updates from IT side.

OT6 — **Data Flow Design:**
*   **Flow:** PLC -> Historian -> OPC UA Server -> Edge Gateway -> MQTT Broker -> Cloud.
*   **Security:** DMZ between Historian and Edge. Edge Gateway authenticates to Cloud.
*   **Boundaries:** Firewall between OT network and Edge. Encryption (TLS) for data in transit.

OT7 — **IEC 62443 Zones & Conduits:**
*   **Zone:** Logical grouping of assets with similar security requirements.
*   **Conduit:** Communication path between zones.
*   **Design:**
    *   Zone 1: Corporate (Level 4).
    *   Zone 2: Engineering (Level 3).
    *   Zone 3: SCADA/Control (Level 2).
    *   Zone 4: Field Devices (Level 1).
    *   Conduits: Firewalls between zones. SIS in isolated Zone 5.

OT8 — **Segmentation Strategy:**
*   **VLANs:** Segment by production line and function (PLC, HMI, Server).
*   **Firewall:** Place at VLAN boundaries and DMZ.
*   **Switch:** Managed Layer 3 switches for routing.
*   **Remote Access:** Jump host in DMZ with MFA and session recording.
*   **Transition:** Phase rollout during maintenance windows; parallel run if possible.

OT9 — **SIS vs BPCS:**
*   **BPCS:** Basic Process Control (PLC/DCS) for normal operation.
*   **SIS:** Safety Instrumented System (Independent) for emergency shutdown.
*   **IEC 61511:** Standard for Safety Instrumented Systems.
*   **Isolation:** SIS must be independent to ensure safety if BPCS/IT is compromised.
*   **Ransomware:** If SIS is compromised, safety shutdowns may fail, causing physical damage.

OT10 — **PLC vs DCS vs SCADA:**
*   **PLC:** Localized control, fast logic, rugged.
*   **DCS:** Distributed control, integrated process control, large scale.
*   **SCADA:** Supervisory, data acquisition, wide area.
*   **Coexist:** Yes, PLCs feed SCADA; DCS often integrates PLCs.

OT11 — **Remote Access:**
*   **Architecture:** Jump host in DMZ, MFA required, session recording.
*   **Vendor:** Time-limited credentials, specific IP whitelisting.
*   **RDP Risk:** Direct RDP exposes OT assets to IT network threats; no audit trail.

OT12 — **Incident Response (Nation-State):**
*   **Monitoring:** IDS/Anomaly detection on OT traffic (e.g., unexpected Modbus commands).
*   **Inventory:** Identify compromised assets immediately.
*   **IR Steps:** Isolate network segments, preserve logs, notify leadership.
*   **Ops Coordination:** Operators must maintain safe state (manual control if needed).
*   **Operator Role:** Prioritize physical safety over data recovery; execute emergency procedures.

OT13 — **HMI:**
*   **Definition:** Interface for operators to view/process data.
*   **Role:** Visualization and manual control.
*   **Vs Engineering Workstation:** HMI is for operations (read-only/limited write); Engineering is for configuration/programming.

OT14 — **Historian:**
*   **Definition:** Time-series database for process data.
*   **Importance:** Trend analysis, compliance, troubleshooting.
*   **Plant (L3):** Real-time process data.
*   **Enterprise (L4):** Aggregated data for business reporting.

OT15 — **Ladder Logic:**
*   **Definition:** Graphical programming language for PLCs.
*   **Device:** PLC.
*   **Dominance:** Familiar to electricians, deterministic, robust, legacy code base.

OT16 — **RTU vs PLC:**
*   **RTU:** Remote Terminal Unit, rugged, long-distance comms (cellular/RF).
*   **PLC:** Local control, high-speed I/O.
*   **RTU Use:** Pipelines, SCADA sites far from control room.

OT17 — **Analog vs Digital:**
*   **Analog:** Continuous signal (4-20mA).
*   **Digital:** Discrete signal (0/1).
*   **4-20mA:** 4mA indicates live wire (fault detection), 0mA indicates break.

OT18 — **Asset Inventory:**
*   **Foundation:** You can't secure what you don't know.
*   **Tools:** Passive monitoring (IDS), Active scanning (Nmap).
*   **Active Risk:** Scanning can crash legacy devices or disrupt control loops.

OT19 — **MITRE ATT&CK ICS:**
*   **Tactics:** Initial Access, Execution, Persistence, Privilege Escalation, Defense Evasion.
*   **Techniques:** Spearphishing, Exploit Public-Facing App, Valid Accounts.
*   **Difference:** Focuses on physical impact and industrial protocols.
*   **Usage:** Map current defenses to tactics to find gaps (e.g., no network segmentation).

OT20 — **Converged SOC:**
*   **Logs:** Collect via agents/forwarders without impacting OT latency.
*   **SIEM:** Centralize IT/OT logs.
*   **Rules:** OT-specific (e.g., PLC write attempts).
*   **Triage:** OT engineers validate physical impact; IT analysts handle malware.
*   **False Positives:** Tune rules to avoid alert fatigue during production.
*   **Firmware Alert:** Isolate PLC, verify backup, restore from known good image, investigate source.
