T1 — Easy: `tofu plan` calculates the execution plan, showing what changes will be made without applying them. `tofu apply` executes the plan, creating, updating, or destroying resources to match the configuration. `plan` is for review; `apply` is for execution.

T2 — Easy: State is the database mapping your configuration to real-world resources. Remote state is important because it enables collaboration (multiple users), state locking (prevents concurrent writes), and persistence (survives local machine failures).

T3 — Medium: `count` uses integer indices; `for_each` uses a map or set of strings. `for_each` is preferred when resources are identified by unique names (e.g., instance IDs) because removing one item doesn't shift indices and cause unintended recreation of other resources.

T4 — Medium: This is "drift". OpenTofu detects it during `plan` by comparing the state file against the live infrastructure API. If drift is detected, `apply` will attempt to reconcile it (recreate or modify the resource) unless `ignore_changes` is set.

T5 — Medium:
```hcl
resource "aws_instance" "web" {
  for_each = toset(["web1", "web2", "web3"])
  ami      = "ami-12345"
  instance_type = "t2.micro"
  tags = {
    Name = each.key
  }
}
```

T6 — Hard: `terraform_remote_state` reads a state file from a backend (decoupled). `output` values are passed directly via module arguments (coupled). The recommended alternative is passing outputs as arguments if modules share the same state, as it is explicit, type-safe, and avoids parsing state files.

T7 — Hard: The `null_resource` provisioner runs asynchronously; the output is generated before the provisioner finishes. Fix: Add `depends_on = [null_resource.kubeconfig]` to the dependent resource, or use a `local-exec` provisioner that polls the Kubernetes API until ready before generating the kubeconfig.

T8 — Medium: `import` CLI manually adds existing resources to state. Limitations: Requires manual mapping of attributes. The `import` block (1.5+) allows declarative import within the configuration, enabling version-controlled import logic and better integration with `plan`.

T9 — Easy: A `variable` is input data passed into the module (e.g., region, size). An `output` is data exposed from the module to the caller (e.g., instance ID, DNS name).

T10 — Easy: A provider is a plugin that interacts with a cloud API (e.g., AWS, Azure). Without `tofu init`, providers are not downloaded, and `plan` will fail with "Provider not installed" errors.

T11 — Easy: `.terraform.lock.hcl` records the exact provider versions and checksums used. It should be committed to ensure reproducible builds and prevent accidental provider upgrades that might break infrastructure.

T12 — Easy: `tofu destroy` removes all managed resources. To target a single resource: `tofu destroy -target=aws_instance.example`.

T13 — Easy: A data source reads existing infrastructure data without managing it. Example: `data "aws_ami" "ubuntu"` to find the latest Ubuntu AMI ID instead of hardcoding it.

T14 — Medium: `variable` blocks define inputs from outside. `locals` define computed values internal to the module. Use `locals` for intermediate calculations or formatting to keep configuration clean and avoid cluttering input variables.

T15 — Medium: `lifecycle` controls resource behavior. `create_before_destroy` creates new resource before deleting old (zero-downtime). `prevent_destroy` stops accidental deletion. `ignore_changes` ignores drift in specific attributes (e.g., tags managed externally).

T16 — Hard: `moved` block allows renaming resources or moving them between modules without destroying them. It updates the state file mapping so OpenTofu recognizes the resource still exists under the new address.

T17 — Hard: The dependency graph determines execution order. Implicit dependencies come from references (e.g., `aws_instance.id`). Explicit dependencies use `depends_on`. `depends_on` reduces parallelism because it forces serialization, potentially slowing down apply.

T18 — Hard: 1. **Split State:** Break monolithic state into smaller modules/states (Tradeoff: More management overhead). 2. **Increase Parallelism:** Set `parallelism` in config (Tradeoff: Higher API rate limits). 3. **Refresh=False:** Use `plan -refresh=false` (Tradeoff: May miss drift). 4. **Prune State:** Remove unused resources (Tradeoff: Risk of accidental data loss).

T19 — Hard: State locking prevents concurrent writes. If a lock isn't released, subsequent runs hang. Manually break with `terraform force-unlock <LOCK_ID>`. Risk: If another process is actually running, this can corrupt the state file.

T20 — Hard: 1. Backup local state. 2. Add `backend` block with S3/DynamoDB config. 3. Run `terraform init -migrate-state`. 4. Run `terraform plan` to verify state is remote. 5. Verify state file is in S3 and lock table in DynamoDB.

A1 — Easy: A **Playbook** is a YAML file defining a sequence of tasks. A **Role** is a structured directory for reusable logic (tasks, handlers, vars). A **Task** is a single action (e.g., install package).

A2 — Easy: `ansible_facts` contains gathered system info. Access OS family via `{{ ansible_facts['os_family'] }}`.

A3 — Medium: `command` runs without shell (no pipes). `shell` runs with shell (supports pipes). `raw` runs without setup (no Python module transfer), useful for initial bootstrap.

A4 — Medium: Idempotency means running a task multiple times yields the same result. `apt: name=nginx` is idempotent (installs only if missing). `shell: rm -rf /tmp/*` is not (deletes every time).

A5 — Medium:
```yaml
- name: Install htop
  package:
    name: htop
  when: ansible_os_family == "Debian"

- name: Install htop and epel
  package:
    name:
      - htop
      - epel-release
  when: ansible_os_family == "RedHat"
```

A6 — Hard: Priority (High to Low): Extra vars > Play vars > Role vars > Group vars > Host vars > Default. Extra vars override everything.

A7 — Hard: Use `serial: 3` in the play to run 3 nodes at a time. Use `wait_for` or `meta: wait_for` to ensure the first batch is ready before the next batch starts.

A8 — Medium: `ansible-vault` encrypts secrets. To encrypt a single variable, use the `!vault` block syntax inside the YAML file: `secret: !vault | $ANSIBLE_VAULT;1.1;AES256...`.

A9 — Easy: Inventory defines hosts. **Static** is a file (INI/YAML). **Dynamic** is a script/API that generates inventory dynamically (e.g., from AWS EC2).

A10 — Easy: `ansible webservers -m shell -a "df -h"`.

A11 — Easy: A **handler** is a task that runs only when notified. It differs from a regular task by running asynchronously after the play finishes. Triggered via `notify: handler_name`.

A12 — Easy: `ansible.cfg` configures Ansible behavior. Common settings: `inventory`, `remote_user`, `forks`, `host_key_checking`.

A13 — Easy: `register` saves a task's output to a variable. Use it in subsequent tasks via `when: my_var.stdout == "success"`.

A14 — Medium: `include_tasks` is evaluated at runtime (dynamic). `import_tasks` is evaluated at compile time (static). Use `include` for conditionals; `import` for performance.

A15 — Medium: Ansible Galaxy is a repository for roles/collections. Use `requirements.yml` with `roles:` and `collections:` lists, then run `ansible-galaxy install -r requirements.yml`.

A16 — Medium: Connection plugins manage communication. **ssh** (remote Linux), **local** (control node), **network_cli** (network devices). Use `ssh` for servers, `network_cli` for routers/switches.

A17 — Hard: `delegate_to: localhost` runs a task on the control node. `run_once` ensures it runs once even in a loop. Example: Gather facts from remote, then generate a config file on the control node.

A18 — Hard: 1. **Forks:** Increase `forks` (Tradeoff: Memory usage). 2. **Pipelining:** Enable `pipelining` (Tradeoff: SSH overhead). 3. **Fact Caching:** Use `fact_caching` (Tradeoff: Stale facts). 4. **Mitogen:** Use Mitogen plugin (Tradeoff: Complexity/Compatibility).

A19 — Hard: `{{ }}` outputs variables. `{% %}` controls logic (loops/if). `{# #}` comments.
Snippet:
```jinja
upstream backend {
{% for server in backend_servers %}
    server {{ server }}; # Health check: {{ server_health_check }}
{% endfor %}
}
```

A20 — Hard: 1. Set `serial: 5`. 2. Pre-task: Deregister from LB. 3. Deploy. 4. Post-task: Health check via `uri` module. 5. `block/rescue`: If health check fails, rollback (revert config, re-register old version).
