AA1 — **Monolith vs. Microservices:** A monolith is a single, unified codebase deployed as one unit. Microservices are a collection of small, independent services communicating over a network.
**3 Problems Microservices Introduce:**
1.  **Distributed Complexity:** Debugging requires tracing across multiple services (distributed tracing).
2.  **Data Consistency:** ACID transactions are hard; requires eventual consistency or sagas.
3.  **Network Latency:** Inter-service communication adds latency compared to in-process calls.

AA2 — **Sync vs. Async:** Synchronous communication blocks the sender until a response is received (request/response). Asynchronous communication allows the sender to continue without waiting (fire-and-forget).
**Examples:** Synchronous: HTTP/REST, gRPC. Asynchronous: RabbitMQ, Apache Kafka.

AA3 — **API Gateway vs. Reverse Proxy:** A reverse proxy routes traffic to backend servers (L4/L7). An API Gateway adds business logic (auth, rate limiting, transformation).
**Replacement:** An API Gateway *can* replace a reverse proxy functionally, but often a Layer 4 load balancer (e.g., HAProxy, F5) is kept in front of the Gateway for SSL termination and TCP routing before the Gateway handles HTTP logic.

AA4 — **Saga Pattern:** A sequence of local transactions where each updates data and triggers the next. If one fails, compensating transactions undo previous steps.
**Choreography vs. Orchestration:** Choreography uses events (services talk to each other); Orchestration uses a central coordinator.
**Appropriateness:** Choreography for simple, decoupled flows. Orchestration for complex flows requiring centralized control and visibility.

AA5 — **CQRS:** Command Query Responsibility Segregation separates read (Query) and write (Command) models.
**Pairing with Event Sourcing:** Event Sourcing stores state changes as events. CQRS uses these events to build optimized read models.
**Problem Solved:** Solves scaling reads vs. writes independently and provides a complete audit trail/history of state changes, which traditional CRUD does not.

AA6 — **Circuit Breaker:** Prevents repeated calls to a failing service.
**States:** Closed (normal), Open (failures detected, requests blocked), Half-Open (testing if service recovered).
**Vs. Retry:** Retry attempts to fix the failure; Circuit Breaker stops traffic to prevent cascading failures.
**Use Together:** Use retries for transient network glitches; use Circuit Breaker when the service is consistently down to save resources.

AA7 — **Tenant Isolation Models:**
1.  **Shared DB/Schema:** Lowest cost, highest operational complexity (schema changes affect all), high noisy-neighbor risk, lower security.
2.  **Shared DB/Schema per Tenant:** Medium cost, better isolation (schema per tenant), moderate noisy-neighbor risk.
3.  **Separate DB per Tenant:** Highest cost, lowest operational complexity (per-tenant DB), lowest noisy-neighbor risk, highest security.

AA8 — **Eventual vs. Strong Consistency:** Eventual consistency guarantees data will match eventually; Strong consistency guarantees immediate match.
**Inventory Architecture:** Use **Strong Consistency** (SQL/ACID).
**Data Store:** Relational DB (PostgreSQL/SQL Server).
**Patterns:** Pessimistic locking (`SELECT FOR UPDATE`) or Optimistic locking (versioning) to prevent overselling.

AA9 — **Sidecar Proxy:** A container deployed alongside the application to handle network traffic.
**Envoy/Istio:** Envoy handles data plane traffic; Control plane (Istiod) configures Envoy.
**Data vs. Control Plane:** Data plane = traffic forwarding; Control plane = configuration management.
**mTLS:** Solves identity verification and encryption at the network layer, removing the burden from the application code to manage certificates.

AA10 — **Strangler Fig Pattern:** Incrementally migrate functionality from a legacy system to a new one.
**Strategy:** Deploy an API Gateway in front of the legacy .NET monolith. Route specific API paths (e.g., `/api/new`) to new .NET 8 microservices. Gradually move more routes to the new services until the monolith is "strangled" and retired.

AA11 — **12-Factor App:** Methodology for building cloud-native apps.
**6 Factors:**
1.  **Codebase:** One codebase tracked in version control.
2.  **Dependencies:** Explicitly declared and isolated.
3.  **Config:** Stored in environment variables.
4.  **Backing Services:** Treated as attached resources.
5.  **Build, Release, Run:** Strictly separated.
6.  **Processes:** Stateless and short-lived.
**Why:** Ensures portability, scalability, and maintainability in cloud environments.

AA12 — **Event-Driven (Kafka) vs. Request-Driven (REST):**
*   **Coupling:** Event-driven is loosely coupled; Request-driven is tightly coupled.
*   **Scalability:** Event-driven scales better for high throughput; Request-driven scales linearly with load.
*   **Debugging:** Event-driven is harder (distributed state); Request-driven is easier (linear trace).
*   **Ordering:** Event-driven can guarantee ordering per partition; Request-driven is per-request.
*   **Failure:** Event-driven requires retry/dead-letter queues; Request-driven returns errors immediately.
**Choice:** Event-driven for decoupling/async processing; Request-driven for real-time user interaction.

AA13 — **Load Balancer:** Distributes traffic across multiple servers.
**L4 vs. L7:** L4 (Transport) routes by IP/Port. L7 (Application) routes by HTTP headers/URL.
**L7 Use Case:** SSL termination or routing `/api` to Service A and `/static` to Service B.

AA14 — **Message Queue:** Stores messages for processing.
**Queue (RabbitMQ) vs. Log (Kafka):** Queue consumes messages once (tasks). Log retains messages for replay (streaming).
**Pick:** RabbitMQ for task queues/workers. Kafka for event streaming/audit logs.

AA15 — **Container vs. VM:** A container virtualizes the OS (shares kernel); a VM virtualizes hardware (separate kernel).
**Popularity:** Containers are lightweight, start faster, and have lower overhead, making them ideal for microservices scaling.

AA16 — **Stateless vs. Stateful:** Stateless services store no session data (any instance can handle any request). Stateful services store session/data locally.
**Scaling:** Stateless is easier to scale horizontally because any node can handle any request without data migration.
**Examples:** Stateless: Web Server. Stateful: Database.

AA17 — **ORM:** Maps database tables to programming language objects.
**Examples:** Hibernate (Java), Entity Framework (.NET).
**Trade-offs:** ORM speeds up development and reduces boilerplate but can generate inefficient SQL and hide performance bottlenecks compared to raw SQL.

AA18 — **Bulkhead Pattern:** Isolates resources (threads, connections) for different services.
**Implementation:** Use separate thread pools for each downstream service call.
**Benefit:** If one service hangs, it doesn't exhaust the thread pool for other services, preventing cascading failures.

AA19 — **OT vs. CRDT:** OT requires a central server to merge operations sequentially. CRDTs allow local edits that merge mathematically without a server.
**Choice:** **CRDTs**.
**Why:** CRDTs support offline editing because conflict resolution happens locally on the client without needing server mediation.

AA20 — **Distributed Tracing Challenges:**
*   **Context Propagation:** Pass `TraceID`/`SpanID` via headers (W3C/B3) across services.
*   **Sampling:** Head-based (start) vs. Tail-based (end). Tail-based is better for debugging errors.
*   **Storage:** Jaeger/Tempo store traces; querying requires indexing.
*   **Kafka:** Inject trace context into message headers so consumers can continue the trace.

OP1 — **Hypervisor Type 1 vs. Type 2:** Type 1 runs directly on hardware (Bare-metal). Type 2 runs on an OS.
**Examples:** Type 1: VMware ESXi, Microsoft Hyper-V. Type 2: VirtualBox, VMware Workstation.

OP2 — **SAN vs. NAS:** SAN provides block storage (disk-like). NAS provides file storage (folder-like).
**Use:** SAN for databases (low latency). NAS for file shares.
**Protocols:** SAN: iSCSI, Fibre Channel. NAS: NFS, SMB/CIFS.

OP3 — **3-Tier Network:** Core (Backbone), Distribution (Policy), Access (Endpoints).
**Leaf-Spine:** Preferred because it offers non-blocking bandwidth and lower latency.
**Problem Solved:** 3-tier has oversubscription at the Core/Distribution layer; Leaf-Spine provides equal uplink/downlink ratios.

OP4 — **Proxmox HA Design:**
*   **Nodes:** Minimum 3 for quorum.
*   **Quorum:** Majority vote to prevent split-brain.
*   **Fencing:** STONITH (Shoot The Other Node In Head) to power off failed nodes.
*   **Storage:** Ceph (distributed) or iSCSI/NFS (shared).
*   **Failure:** VMs restart on surviving nodes automatically.

OP5 — **VLAN Trunking:** Carries multiple VLAN traffic over one link.
**802.1Q:** Adds a 4-byte tag to Ethernet frames identifying the VLAN.
**Linux Config:** Create sub-interfaces (e.g., `eth0.10`, `eth0.20`) using `ip link` or `vconfig` and assign IPs to each.

OP6 — **RAID Levels:**
*   **RAID 1:** Mirror (50% capacity).
*   **RAID 5:** Striping + Parity (N-1 capacity).
*   **RAID 6:** Double Parity (N-2 capacity).
*   **RAID 10:** Mirror + Striping (50% capacity).
*   **RAID Z2:** ZFS Double Parity.
**Choice:** **RAID 10** for 6 drives. Best balance of read/write performance and fault tolerance (can lose 2 drives if mirrored pairs).

OP7 — **DNS Architecture:**
*   **Internal:** AD-integrated DNS for clients.
*   **DMZ:** Public-facing authoritative servers.
*   **Split-Horizon:** Internal clients get internal IPs; External get public IPs.
*   **DNSSEC:** Sign external zone to prevent spoofing.
*   **Laptop:** Connects via DHCP, queries internal DNS, resolves internal names via AD, external via forwarders.

OP8 — **PKI Chain:** Root CA (Offline) -> Subordinate CA (Issuing) -> End-Entity Cert.
**Revocation:** CRL (List) vs. OCSP (Real-time check).
**Compromise:** Subordinate compromised = Revoke Sub, issue new. Root compromised = **Total Trust Loss**, must rebuild PKI and re-issue all certs.

OP9 — **WAN Consolidation:**
*   **MPLS:** High cost, high reliability.
*   **IPsec:** Low cost, variable reliability.
*   **SD-WAN:** Dynamic path selection, cost-effective.
*   **WireGuard:** Lightweight, mesh.
**Recommendation:** **SD-WAN**. Best for VoIP (QoS), cost management, and ease of management across 5 sites.

OP10 — **iDRAC/iLO/IPMI:** Out-of-band management interfaces on server hardware.
**Critical:** Allows server control (power, console) even if OS is down.
**Exposed:** If exposed to internet, attackers can gain root access to the server hardware (IPMI compromise).

OP11 — **LVM vs. ZFS:**
*   **LVM:** Volume manager on top of block devices.
*   **LVM-Thin:** Overprovisions storage (thin provisioning).
*   **ZFS:** Filesystem + Volume Manager + RAID.
*   **Snapshots:** LVM (Copy-on-Write, slower). ZFS (Instant, low overhead).
**Use:** LVM for simple VM storage. ZFS for data integrity and snapshots.

OP12 — **Stretch Cluster/DR:**
*   **Ceph CRUSH:** Rules to distribute OSDs across sites.
*   **Mon Placement:** 1 Monitor per site + Tiebreaker (VM/Disk).
*   **Latency:** <5ms RTT required for synchronous replication.
*   **Split-Brain:** If network fails, both sites think they are primary.
*   **Tiebreaker:** Required to ensure only one site remains writable.

OP13 — **DHCP DORA:** Discover, Offer, Request, Acknowledge.
**Unavailable:** Client cannot renew lease.
**Windows Expiry:** Falls back to APIPA (169.254.x.x) and cannot reach network resources.

OP14 — **Managed vs. Unmanaged Switch:**
*   **Unmanaged:** Plug-and-play.
*   **Managed:** Configurable (VLANs, QoS, ACLs, Port Mirroring).
*   **Importance:** Managed switches are critical for segmentation, security, and traffic prioritization in enterprises.

OP15 — **NTP:** Network Time Protocol.
**Critical:** Kerberos auth, log correlation, database replication, cluster consensus.
**Drift:** Causes auth failures, split-brain scenarios, and incorrect log timestamps.

OP16 — **Physical vs. VM vs. Container:**
*   **Physical:** Bare metal.
*   **VM:** Virtual OS on hypervisor.
*   **Container:** Virtualized app on host OS.
**Bare Metal:** Use for high I/O, low latency, or hardware passthrough (e.g., High-Performance DB, HPC).

OP17 — **UPS vs. Generator:**
*   **UPS:** Battery backup (minutes).
*   **Generator:** Fuel backup (hours).
*   **ATS:** Automatic Transfer Switch switches power from Grid to Generator when grid fails.
*   **Need Both:** UPS bridges the gap until generator starts.

OP18 — **Active-Passive vs. Active-Active:**
*   **Active-Passive:** One node handles load; standby takes over on failure.
*   **Active-Active:** Both nodes handle load.
*   **Pacemaker/Corosync:** Manages cluster state.
*   **STONITH:** Fencing mechanism to kill failed node.
*   **VIP:** Virtual IP moves to active node on failover.

OP19 — **Backup Strategy:**
*   **3-2-1:** 3 copies, 2 media types, 1 offsite.
*   **RPO/RTO:** Recovery Point/Time Objective.
*   **Types:** Full, Incremental, Differential.
*   **Immutable:** Prevents ransomware from encrypting backups.
*   **Backup vs. DR:** Backup is data copy; DR is the plan/process to restore operations at a secondary site.

OP20 — **802.1X:**
*   **Roles:** Supplicant (Client), Authenticator (Switch), Server (RADIUS).
*   **EAP:** EAP-TLS (Certificate), PEAP (Username/Password).
*   **Dynamic VLAN:** Assigns VLAN based on user identity.
*   **Fail:** Device placed in Quarantine VLAN.
*   **MAB:** Authenticates device by MAC address if 802.1X fails (fallback for printers/IoT).
