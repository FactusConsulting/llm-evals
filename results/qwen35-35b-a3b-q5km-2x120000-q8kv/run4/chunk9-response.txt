SC1-A
The failure stems from the interaction between the **Restricted Pod Security Standard (PSS)** and the container image configuration.
1.  **Restricted PSS Requirements:** Unlike Baseline, Restricted enforces that containers must run as non-root. Specifically, it requires `securityContext.runAsNonRoot: true` **AND** `securityContext.runAsUser` to be explicitly set to a non-zero value (UID > 0).
2.  **Image Behavior:** The new base image (`cgr.dev/chainguard/static:latest`) defaults `USER` to root (UID 0).
3.  **The Conflict:** The pod spec sets `runAsNonRoot: true` but omits `runAsUser`. Kubernetes defaults `runAsUser` to 0 if not specified. When the Restricted policy is enforced, the admission controller (or runtime) detects that the effective UID is 0, violating the "non-root" requirement.
4.  **Result:** The pod fails to start or crashes immediately upon startup because the security context is invalid under the Restricted profile. The upgrade likely triggered a rolling update or policy enforcement that restarted pods, exposing this incompatibility.

SC1-C
**Remediation Strategy:**
1.  **Immediate Fix (Image):** Rebuild the container image with a non-root user. Add `USER 1000` (or similar) to the Dockerfile.
2.  **Manifest Fix:** Update the Deployment manifests to explicitly set `securityContext.runAsUser: 1000` to satisfy Restricted PSS requirements regardless of image defaults.
3.  **CI Pipeline Gate:** Integrate a PSS validation tool (e.g., `kube-score` or `checkov`) into the CI pipeline. Add a step to lint manifests against the target cluster's PSS profile before merging.
4.  **Rollout:** Use a **Canary Deployment**. Deploy the fixed version to 5% of nodes first. Monitor CrashLoopBackOff rates and latency. If stable, gradually increase to 100%. This ensures no downtime and allows quick rollback if the new image has other issues.

SC2-A
**Stale Data & Replication:**
1.  **Cause:** Cross-region replication is asynchronous. When the EU replica is promoted, it only contains data up to the last WAL (Write-Ahead Log) shipped from US-East. Any writes committed in US-East after the last sync are lost or delayed.
2.  **WAL Shipping:** PostgreSQL streams WAL records. Across regions, network latency and bandwidth limit throughput.
3.  **Data Loss:** Asynchronous replication implies an RPO > 0. If the primary fails before WAL is shipped, that data is lost.
4.  **Quantification:** Check `pg_stat_replication` on the new primary (formerly replica) for `write_lag`, `flush_lag`, and `replay_lag`. Compare the `pg_current_wal_lsn()` on the old primary (if accessible) vs. the new primary to calculate the gap.

SC2-B
**Terraform Operations:**
1.  **Import:** Import the promoted replica into state using its new ARN.
    ```bash
    terraform import aws_db_instance.promoted_replica <new_replica_arn>
    ```
2.  **State Move:** If the old instance resource name is still in state, move the promoted replica to that resource name to avoid recreation.
    ```bash
    terraform state mv aws_db_instance.old_primary aws_db_instance.promoted_replica
    ```
3.  **Update Dependencies:** Update the Terraform code to reference the new ARN or use a variable for the primary instance ID. Run `terraform plan` to ensure no destroy/create actions are queued for dependent resources (Security Groups, Route53).
4.  **Apply:** Run `terraform apply` to update the state of dependent resources (e.g., update Route53 record to point to the new endpoint).

SC2-C
**Architecture Design:**
1.  **Connection Management:** Use **RDS Proxy** instead of PgBouncer. It handles connection pooling at the AWS layer, reducing overhead on the DB and providing better failover integration with RDS.
2.  **DNS Failover:** Use **Route53 Health Checks** with Latency-Based Routing. Point the primary endpoint to us-east-1 and the secondary to eu-west-1. If us-east-1 fails health checks, Route53 routes traffic to eu-west-1.
3.  **Retry Logic:** Implement exponential backoff with jitter in the application connection pool.
4.  **RPO < 1s:** Standard RDS async replication cannot guarantee < 1s RPO. **Recommend Aurora Global Database.** It uses a single storage volume replicated across regions with synchronous replication for the primary region and async for secondary, offering RPO < 1 second and faster failover.
5.  **Failover:** Aurora Global Database supports automated failover with DNS updates, reducing manual intervention.

SC6-C
**VPN Upgrade Comparison:**
1.  **AWS Direct Connect:**
    *   **Pros:** Dedicated 10Gbps+ bandwidth, lower latency, consistent performance.
    *   **Cons:** High cost, physical hardware setup.
    *   **Zero-Trust:** Fits well; traffic is encrypted (MACsec) and isolated.
2.  **Second VPN Tunnel (ECMP):**
    *   **Pros:** Low cost, quick deployment.
    *   **Cons:** Shared bandwidth, higher latency variability.
    *   **Zero-Trust:** IPsec encryption maintained, but IP-based trust remains.
3.  **Hybrid Mesh (Tailscale/Netbird):**
    *   **Pros:** Zero-trust by design (Identity-based), no IP management.
    *   **Cons:** Performance depends on relay servers (unless using P2P), not suitable for high-throughput DB replication.
    *   **Zero-Trust:** Best fit for identity, worst for raw bandwidth.
**Recommendation:** Use **Direct Connect** for the core data plane (DB/Storage) to meet bandwidth needs. Use **SPIRE/Tailscale** for management plane and service-to-service identity verification.

SC10-C
**Failover Orchestration:**
1.  **Detection:** Prometheus monitors site health. If Chicago nodes are unreachable for 2 mins, trigger failover.
2.  **DNS:** GSLB (Route53) updates A records to Dallas IP.
3.  **Patroni:** Dallas node detects Chicago is down, promotes itself to Master (via etcd quorum).
4.  **Redis:** Sentinel detects master down, promotes replica.
5.  **RabbitMQ:** Federation policies sync queues.
6.  **MinIO:** Site replication catches up (async).
7.  **State:** Ansible/Terraform updates state to reflect Dallas as primary.
8.  **Split-Brain:** Prevented by **etcd quorum** in Patroni. If Chicago is up but network partitioned, etcd prevents promotion until consensus.
9.  **Failback:** Once Chicago recovers, demote Dallas, re-sync data, and switch DNS back.
