## Go (20 questions)

**G1 — Easy:** An array is a fixed-size value type (`[N]T`), while a slice is a dynamic descriptor containing a pointer to an underlying array, length, and capacity (`[]T`). When appending to a slice that has reached its capacity, the Go runtime allocates a new, larger underlying array (typically doubling the capacity), copies the existing elements, and updates the slice header to point to the new array.

**G2:** `var x int` declares a variable with an explicit type and initializes it to the zero value (`0`). `x := 0` is a short variable declaration that infers the type from the value. You cannot use `:=` at the package level (outside functions) or if you are redeclaring a variable with the same name in the same scope.

**G3:** A goroutine is a lightweight thread managed by the Go runtime, not the OS. It differs from an OS thread by having a much smaller stack (starting at 2KB) and being multiplexed onto fewer OS threads. The Go scheduler (GMP model) manages Goroutines (G), OS Threads (M), and Processors (P). P represents logical execution contexts that schedule Gs onto Ms, allowing efficient concurrency without OS thread overhead.

**G4:** Channels are typed conduits for communication between goroutines. Unbuffered channels block both sender and receiver until the other side is ready. Buffered channels have a fixed capacity; sending blocks only if the buffer is full, and receiving blocks only if empty. Sending to a full buffered channel blocks until space is available.

**G5:** The `context` package manages deadlines, cancellation signals, and request-scoped values across API boundaries.
```go
ctx, cancel := context.WithTimeout(context.Background(), 5*time.Second)
defer cancel()
resp, err := http.GetContext(ctx, "http://example.com")
```

**G6:** Go interfaces are satisfied implicitly; a type satisfies an interface if it implements all its methods. Unlike Java/C#, Go interfaces are not declared by the implementing type. The empty interface `interface{}` (or `any`) accepts any value. It is powerful for generic-like flexibility but dangerous because it requires type assertions to use values, losing compile-time type safety.

**G7:** Go's memory model guarantees visibility only if there is a happens-before relationship (e.g., via synchronization). The code is unsafe because `ready` and `data` are accessed concurrently without synchronization, leading to a data race where the reader might see `ready=true` but `data=""`. Correct alternatives include using `sync/atomic` for the boolean or a channel to signal readiness and transfer data.

**G8:** `sync.Mutex` provides exclusive locking. `sync.RWMutex` allows multiple readers or one writer. `sync.Map` is optimized for cases where keys are added dynamically and reads vastly outnumber writes. Trade-offs: `sync.Map` has higher overhead per access and cannot be iterated safely, while a regular map with a mutex is better for dense keys or frequent writes.

**G9:** Go modules manage dependencies. `go.mod` declares the module path and required dependencies. `go.sum` stores cryptographic checksums of those dependencies. `go mod tidy` removes unused dependencies and adds missing ones. Minimum Version Selection (MVS) selects the lowest version of a dependency that satisfies all constraints in the dependency graph.

**G10:** `defer` schedules a function call to run immediately before the surrounding function returns. Multiple defers execute in Last-In-First-Out (LIFO) order. A common gotcha in loops is capturing the loop variable by reference in the deferred closure, causing all deferred calls to use the final loop value.

**G11:** A pointer holds the memory address of a value. The zero value of a pointer is `nil`. To create a pointer to a struct literal, use the address-of operator: `&MyStruct{Field: value}`.

**G12:** The `init()` function is used for package initialization. It runs automatically before `main()` executes. A package can have multiple `init()` functions; they execute in the order they appear in the source files, and within a file, in declaration order.

**G13:** `make` is a built-in function used to initialize slices, maps, and channels, returning the initialized type. `new` is a built-in function that allocates zeroed memory for any type and returns a pointer to it. `make` returns the type itself; `new` returns a pointer.

**G14:** Errors are values implementing the `error` interface (`Error() string`). The idiomatic pattern is `if err != nil { return err }`. `errors.Is` checks if an error matches a target error in the chain (semantic equality). `errors.As` checks if an error matches a specific type (type assertion). `==` only checks pointer equality or exact value match, not wrapped errors.

**G15:** Built-in composite types are `map`, `slice`, `channel`, and `struct`. Zero values: `map` and `channel` are `nil`; `slice` is `nil` (length 0); `struct` has all fields set to their respective zero values.

**G16:** A race condition occurs when two goroutines access the same memory concurrently, and at least one writes. The `-race` flag instruments code to detect these at runtime. Example: `counter++` without a lock. Fix: Wrap access in `sync.Mutex.Lock()/Unlock()` or use a channel to serialize access.

**G17:** `select` allows a goroutine to wait on multiple channel operations. Unlike `switch`, it chooses a case randomly if multiple are ready. To implement a timeout: `select { case <-ch: ... case <-time.After(1*time.Second): ... }`.

**G18:** Escape analysis determines if a variable lives on the stack or heap. If a variable's address escapes the function scope, it is allocated on the heap. Inspect with `go build -gcflags="-m"`. Heap allocation is slower due to GC pressure; stack allocation is faster.

**G19:** The `reflect` package allows runtime introspection. `reflect.Type` describes the type metadata; `reflect.Value` holds the actual data and allows manipulation. Reflection has high performance costs (indirection, type checks). It is justified for frameworks (serialization, dependency injection) but code generation (e.g., `go generate`) is preferred for performance-critical paths.

**G20:** `go:embed` embeds files into the binary at compile time. Syntax: `//go:embed path`. Works for single files, directories, or `//go:embed *`. Limitations: files must exist at compile time; cannot embed stdin/stdout. It interacts with `io/fs.FS` by providing an `embed.FS` type that implements the interface.

## Rust (20 questions)

**R1:** Ownership is Rust's core memory safety feature. Rules: 1. Each value has exactly one owner. 2. When the owner goes out of scope, the value is dropped. 3. Values can be moved or borrowed, but not both simultaneously.

**R2:** `String` is an owned, heap-allocated, growable string type. `&str` is a borrowed string slice (a view into a string). Use `&str` in function signatures for read-only access to avoid unnecessary allocations; use `String` when ownership or mutation is required.

**R3:** Borrowing allows temporary access to data. Rules: You can have either one mutable reference (`&mut T`) OR any number of immutable references (`&T`), but not both. References must always be valid.

**R4:** `Result<T, E>` represents success (`Ok(T)`) or failure (`Err(E)`). `Option<T>` represents a value that may be present (`Some(T)`) or absent (`None`). The `?` operator propagates errors up the call stack.
```rust
use std::fs;
fn read_file() -> Result<String, std::io::Error> {
    fs::read_to_string("file.txt")
}
```

**R5:** `Box<T>` owns data on the heap with a single owner. `Rc<T>` (Reference Counted) allows multiple owners in a single thread. `Arc<T>` (Atomic Reference Counted) allows multiple owners across threads. Use `Box` for heap allocation, `Rc` for shared ownership in single-threaded contexts, `Arc` for multi-threaded sharing.

**R6:** Traits define shared behavior. Unlike Go interfaces, traits can have default implementations. `dyn Trait` creates a trait object for dynamic dispatch (vtable). `T: Trait` uses generics for static dispatch (monomorphization), which is generally faster.

**R7:** The function fails because the compiler cannot determine which input reference (`x` or `y`) the returned reference refers to. Fix: Add a lifetime annotation `'a` to indicate both inputs and output share the same lifetime.
```rust
fn longest<'a>(x: &'a str, y: &'a str) -> &'a str {
    if x.len() > y.len() { x } else { y }
}
```

**R8:** `Send` means ownership can be transferred across threads. `Sync` means shared references (`&T`) can be shared across threads. `Rc<T>` is not `Send` because its reference count is not atomic. `Arc<T>` uses atomic reference counting, making it `Send` and `Sync`.

**R9:** `async/await` creates a state machine (`Future`) that can be polled. The executor (e.g., Tokio) drives the future to completion. Unlike Go's M:N goroutines (which are preemptive and OS-thread backed), Rust futures are cooperative and run on a smaller pool of OS threads.

**R10:** `unsafe` allows operations the compiler cannot verify for safety. Examples: dereferencing raw pointers, calling unsafe functions, accessing mutable statics. Justified for FFI, performance-critical low-level ops, or implementing safe abstractions over unsafe primitives.

**R11:** Pattern matching (`match`) exhaustively checks all possible values. `if let` matches only a specific pattern. The compiler enforces exhaustiveness in `match` to prevent logic errors from unhandled cases.

**R12:** `Vec<T>` is a growable heap-allocated array. `[T; N]` is a fixed-size stack-allocated array. When a `Vec` exceeds capacity, it allocates a new, larger buffer (typically doubling capacity) and moves elements.

**R13:** Enums (ADTs) can hold different types of data in variants. Unlike C/Java enums, Rust enums can carry payload. `Option<T>` is an enum (`Some(T)` or `None`) that replaces `null`, forcing explicit handling of absence.

**R14:** `impl` defines methods. `&self` borrows the instance; `&mut self` borrows mutably; `self` consumes the instance. Associated functions (e.g., `String::new()`) do not take `self` and act like constructors.

**R15:** `derive` automatically implements traits like `Debug`, `Clone`, `PartialEq`. You cannot derive traits if the trait requires custom logic or if fields do not support the trait (e.g., deriving `Clone` on a struct containing a `Rc` requires `Clone` on the inner type).

**R16:** `Cow<'a, T>` (Clone on Write) holds either a borrowed reference or an owned value. It avoids allocation if the data is only read. If mutation is needed, it clones the data internally. Useful for functions that accept `&str` but may need to return an owned `String`.

**R17:** Modules organize code. `mod` declares a module, `use` imports paths, `pub` makes items public. File system maps to modules (`mod.rs` or `mod/` directory). Rust 2018 improved path resolution (e.g., `crate::` instead of `extern crate`).

**R18:** `Pin<T>` prevents a value from being moved in memory. Needed for self-referential structs and async/await (futures hold references to themselves). `Unpin` is a marker trait indicating a type can be safely moved; most types are `Unpin`.

**R19:** Macros automate code generation. `macro_rules!` are declarative (pattern matching). Procedural macros (derive, attribute, function-like) are Rust code that generates code. Hygiene rules ensure macro variables don't leak into or shadow surrounding scope.

**R20:** Zero-cost abstractions mean high-level features compile to efficient machine code. Iterators are inlined loops. Generics use monomorphization (generating specific code per type). Traits use static dispatch (vtables only when necessary), avoiding runtime overhead compared to dynamic languages.
