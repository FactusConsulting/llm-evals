CL1 — **IaaS, PaaS, SaaS:**
*   **IaaS (Infrastructure as a Service):** Provides virtualized computing resources (VMs, storage, networks). You manage OS, middleware, runtime, data, and apps. *Example:* AWS EC2.
*   **PaaS (Platform as a Service):** Provides hardware and software tools (runtime, DB, OS) needed for development. You manage data and apps. *Example:* Google App Engine.
*   **SaaS (Software as a Service):** Fully managed application delivered over the internet. You manage only user access and configuration. *Example:* Microsoft 365.

CL2 — **Shared Responsibility Model:**
*   **Concept:** Provider secures the *Cloud* (physical, network, hypervisor); Customer secures *in the Cloud* (data, OS, config).
*   **IaaS VM:** Provider secures hardware/hypervisor. Customer secures OS patching, firewall config, app security, data encryption.
*   **Managed Kubernetes:** Provider secures control plane, node OS, networking. Customer secures pod security, RBAC, network policies, app code.
*   **SaaS (M365):** Provider secures app, infrastructure, data center. Customer secures identity (MFA), data classification, device compliance, user access.

CL3 — **Networking Layers:**
*   **VPC:** Isolated virtual network in the cloud.
*   **Subnet:** IP range within a VPC (public/private).
*   **Security Group:** Instance-level firewall. **Stateful:** Allows return traffic automatically.
*   **NACL:** Subnet-level firewall. **Stateless:** Requires explicit rules for both inbound and outbound traffic.
*   **Layering:** VPC contains Subnets. NACLs filter traffic entering/leaving Subnets. Security Groups filter traffic entering/leaving Instances.

CL4 — **Multi-Account Strategy:**
*   **Landing Zone:** Baseline environment with networking, identity, and security controls pre-configured.
*   **OUs:** Organizational Units structure accounts by function (e.g., Prod, Dev, Security).
*   **Guardrails:** SCPs or policies preventing risky actions (e.g., disabling encryption).
*   **Shared Networking:** Transit Gateway (AWS) or Hub-and-Spoke (Azure) connects accounts centrally.
*   **Centralized Logging:** Aggregates logs to a dedicated Security/Log account.
*   **Single-Account Problem:** High blast radius (compromise affects all), billing confusion, lack of isolation for compliance, shared resource contention.

CL5 — **Infrastructure as Code (IaC):**
*   **Terraform/OpenTofu:** Declarative, multi-cloud, state-based. Best for general purpose, multi-cloud, and complex provisioning.
*   **Pulumi:** Imperative/Declarative using general languages (TS, Python). Best for teams wanting code flexibility.
*   **CloudFormation/Bicep:** Native to AWS/Azure. Best for deep integration with specific provider features and CI/CD pipelines.
*   **Crossplane:** Kubernetes-native IaC. Best for managing cloud resources via K8s API (GitOps for infra).

CL6 — **Autoscaling Patterns:**
*   **HPA:** Scales pods based on CPU/Memory or custom metrics.
*   **VPA:** Adjusts resource requests/limits for pods (requires restart).
*   **Cluster Autoscaler:** Adds/removes nodes based on pending pods.
*   **KEDA:** Event-driven scaling (e.g., Kafka queue depth, RabbitMQ).
*   **KEDA vs HPA:** Use KEDA when scaling is driven by external events (message queues, HTTP requests) rather than just resource utilization.

CL7 — **Disaster Recovery (DR) Strategy:**
*   **RPO/RTO:** RPO (data loss tolerance), RTO (downtime tolerance).
*   **Active-Active:** Both regions serve traffic. Low RTO/RPO, high cost.
*   **Active-Passive:** One region active, one standby. Higher RTO, lower cost.
*   **Database:** Sync (RPO=0, latency sensitive) vs Async (RPO>0, latency tolerant).
*   **DNS Failover:** Route53 Health Checks trigger failover.
*   **Patterns:** Pilot Light (DB running, app off), Warm Standby (App scaled down), Multi-Site (Active-Active).
*   **Cost:** Active-Active doubles infra cost; Pilot Light is cheapest but higher RTO.

CL8 — **GitOps Model:**
*   **Concept:** Git repository is the single source of truth. Controllers sync cluster state to Git state.
*   **ArgoCD vs Flux:** ArgoCD (Push, UI focus), Flux (Pull, K8s-native).
*   **Secrets:** Sealed Secrets (encrypted in Git), External Secrets Operator (fetch from Vault/AWS Secrets Manager).
*   **Drift:** If manual change occurs, GitOps controller detects drift and reverts the cluster to match Git.

CL9 — **Stateful Legacy App to K8s:**
*   **StatefulSet:** Ensures stable network IDs and ordered deployment/termination.
*   **PV/PVC:** Use `local` storage class with `hostPath` for low latency (binds to specific node).
*   **Headless Service:** Provides stable DNS names for StatefulSet pods.
*   **Init Containers:** Handle config file generation before main app starts.
*   **ConfigMaps:** Mount as volumes for specific paths.
*   **PDB:** Ensure minimum available pods during disruptions to maintain quorum.

CL10 — **Identity (SA vs Workload vs IRSA):**
*   **Service Account:** K8s identity for pods.
*   **Workload Identity:** GCP/Azure native federation (K8s SA -> Cloud Identity).
*   **IRSA:** AWS specific (K8s SA -> IAM Role via OIDC).
*   **Mounting Creds:** Bad practice (leaks in logs, containers, or if pod is compromised).
*   **Federation:** Uses OIDC tokens to exchange K8s identity for temporary cloud credentials without long-lived secrets.

CL11 — **Storage Types:**
*   **Object (S3/Blob):** Unstructured data, HTTP API. High durability (11 9s), eventual consistency. Use for backups, media.
*   **Block (EBS/Managed Disk):** Raw storage for VMs. Low latency, persistent. Use for OS, DBs.
*   **File (EFS/Azure Files):** Shared file system (NFS/SMB). Use for shared configs, legacy apps.
*   **Guarantees:** Object storage offers highest durability; Block/File offer higher availability for active workloads.

CL12 — **Zero-Trust K8s:**
*   **Network Policies:** Default deny all traffic between pods.
*   **Service Mesh (mTLS):** Encrypts service-to-service traffic (Istio/Linkerd).
*   **Pod Identity:** Restrict pod permissions (SPIFFE/SPIRE).
*   **Admission Controllers:** OPA/Kyverno enforce policies on deploy.
*   **Image Signing:** Cosign/Notary verify images before pull.
*   **Runtime Security:** Falco detects anomalies (e.g., shell spawn).
*   **Difference:** Traditional relies on perimeter (firewall); Zero Trust assumes breach, verifies every request.

CL13 — **CDN:**
*   **Definition:** Distributed network of edge servers caching content closer to users.
*   **How:** Caches static assets (images, CSS) at edge nodes.
*   **Services:** AWS CloudFront, Azure CDN.
*   **Use Case:** Use CDN for global static content to reduce latency/origin load. Use Origin for dynamic, personalized, or real-time data.

CL14 — **AZ vs Region:**
*   **Region:** Geographic area (e.g., us-east-1).
*   **Availability Zone (AZ):** Isolated data center within a Region.
*   **Multi-AZ:** Deploy across AZs for High Availability (HA) against data center failure.
*   **Zonal:** Resources tied to one AZ (EBS). **Regional:** Resources span AZs (ELB, RDS Multi-AZ).

CL15 — **Managed Database:**
*   **Definition:** DB service where provider manages infrastructure.
*   **Provider Tasks:** Patching, backups, failover, scaling, monitoring.
*   **Customer Tasks:** Schema design, queries, access control, data modeling.
*   **Benefit:** Reduces operational overhead and improves availability.

CL16 — **IAM & Least Privilege:**
*   **IAM:** Users (People), Groups (Roles), Roles (Temporary access), Policies (Permissions).
*   **Least Privilege:** Grant only permissions necessary to perform a task.
*   **Application:** Use Roles instead of Users for services. Use scoped policies (e.g., `s3:GetObject` on specific bucket, not `*`).

CL17 — **Serverless:**
*   **Definition:** Run code without provisioning servers (AWS Lambda, Azure Functions).
*   **Cold Starts:** Delay when a function is invoked after inactivity (initialization).
*   **Impact:** Matters for latency-sensitive apps (real-time APIs).
*   **Poor Fit:** Long-running processes, stateful apps, high memory/CPU needs, strict compliance requiring dedicated hardware.

CL18 — **Ingress Options:**
*   **Ingress Resource:** K8s standard for HTTP routing. TLS terminated at Ingress Controller.
*   **Gateway API:** New standard, more flexible (TCP, gRPC), multi-cluster support.
*   **Cloud LB:** Cloud provider LB with annotations.
*   **Mesh Gateway:** Service mesh ingress (Istio).
*   **TLS:** Terminates at LB (Edge), Ingress Controller (Cluster), or Backend (mTLS).
*   **Gateway API:** Use for complex routing, multi-cluster, or when Ingress resource limitations hinder requirements.

CL19 — **Cost Optimization ($80k/mo):**
*   **Compute:** Reserved Instances (1-3yr) for steady state. Savings Plans (flexible). Spot/Preemptible for fault-tolerant workloads (up to 90% savings).
*   **Right-sizing:** Analyze utilization (CloudWatch/Azure Monitor) to downsize over-provisioned instances.
*   **Storage:** S3 Intelligent-Tiering for unknown access. Archive for cold data.
*   **Idle Detection:** Tagging, auto-shutdown scripts for dev/non-prod.
*   **FinOps:** Showback/Chargeback models to make teams accountable.
*   **Tools:** AWS Cost Explorer, Azure Cost Management, Kubecost for K8s.

CL20 — **Multi-Cloud Strategy:**
*   **Abstraction:** Kubernetes (runtime), Terraform (IaC), Crossplane (API).
*   **Networking:** Direct Connect/ExpressRoute or SD-WAN for low latency interconnect.
*   **Identity:** Federation via OIDC (e.g., Entra ID to AWS/Azure).
*   **Trade-offs:** Avoids lock-in but increases complexity, networking latency, and operational overhead.
*   **Justified:** Regulatory requirements (data sovereignty), avoiding single-vendor risk, best-of-breed services.
*   **Unnecessary:** If one provider meets all needs, multi-cloud adds cost without value.

OT1 — **Purdue Model:**
*   **Level 0:** Physical Process (Sensors/Actuators).
*   **Level 1:** Basic Control (PLCs, RTUs).
*   **Level 2:** Area Supervisory (SCADA, HMIs).
*   **Level 3:** Site Operations (MES, Historian).
*   **Level 4:** Enterprise (ERP, Business).
*   **Level 5:** Business Planning (Corporate).

OT2 — **IT vs OT Networks:**
*   **IT:** Confidentiality/Integrity focus. Standard patching acceptable.
*   **OT:** Safety/Availability focus. Patching requires validation/testing to avoid downtime.
*   **Difference:** OT protocols are often unencrypted, stateful, and real-time. IT traffic is bursty.

OT3 — **DMZ (IDMZ):**
*   **Role:** Buffer zone between IT (L4) and OT (L0-L3).
*   **Traffic:** IT pulls data from OT (via DMZ). OT pushes data to IT (via DMZ). No direct IT-to-OT control commands.
*   **Danger:** Direct L4->L1 allows attackers to send control commands to PLCs, causing physical damage.

OT4 — **OT Protocols:**
*   **Modbus TCP:** L1-L2 (Control). Simple, unencrypted.
*   **OPC UA:** L2-L3 (Control/Info). Secure, platform-independent.
*   **EtherNet/IP:** L1-L2 (Control). CIP protocol over Ethernet.
*   **PROFINET:** L1-L2 (Control). Real-time industrial Ethernet.
*   **MQTT:** L3-L4 (Info). Lightweight pub/sub for cloud connectivity.

OT5 — **Data Diode vs Firewall:**
*   **Firewall:** Filters traffic both ways. Can be bypassed or misconfigured.
*   **Data Diode:** Hardware-enforced one-way communication (optical).
*   **Use:** When data must flow OT->IT but IT must never touch OT (e.g., high-security zones).
*   **Limitation:** Cannot send commands back to OT; requires specific hardware.

OT6 — **Data Flow Design:**
*   **Flow:** PLC -> Historian (L3) -> Edge Gateway (Protocol Translation) -> MQTT Broker -> Cloud Ingestion.
*   **Security Boundaries:** DMZ between Historian and Edge. TLS encryption on MQTT. Firewall between Edge and Cloud.
*   **Placement:** Edge gateway handles protocol conversion and buffering to protect OT network from internet exposure.

OT7 — **IEC 62443 Zones & Conduits:**
*   **Zone:** Logical grouping of assets with similar security requirements.
*   **Conduit:** Controlled communication path between zones.
*   **Design:**
    *   Zone 4: Corporate Network.
    *   Zone 3: SCADA Servers (DMZ).
    *   Zone 2: Engineering Workstations.
    *   Zone 1: PLCs/Control.
    *   Zone 0: Safety Systems (SIS).
    *   Conduits: Firewalls between zones enforce strict allow-lists.

OT8 — **Segmentation Strategy:**
*   **VLANs:** Create VLANs per production line and function (PLC, HMI, Server).
*   **Firewalls:** Place firewalls between VLANs (e.g., Line VLAN to Server VLAN).
*   **Switches:** Use managed switches with ACLs.
*   **Remote Access:** Vendor access via Jump Host with MFA and session recording.
*   **Transition:** Phase migration during maintenance windows. Use temporary parallel networks if possible.

OT9 — **SIS vs BPCS:**
*   **BPCS:** Basic Process Control System (PLCs/DCS) for normal operation.
*   **SIS:** Safety Instrumented System (Independent) for emergency shutdown.
*   **IEC 61511:** Standard for Safety Instrumented Systems.
*   **Isolation:** SIS must be air-gapped/isolated to prevent cyber attacks from disabling safety functions.
*   **Ransomware on SIS:** If compromised, safety shutdowns may fail, leading to catastrophic physical damage or injury.

OT10 — **PLC vs DCS vs SCADA:**
*   **PLC:** Logic controller for specific machines.
*   **DCS:** Distributed Control System for continuous process control (refineries).
*   **SCADA:** Supervisory Control and Data Acquisition for monitoring remote sites.
*   **Coexist:** Yes. PLCs feed DCS/SCADA. DCS manages process, SCADA monitors.

OT11 — **Remote Access Architecture:**
*   **Jump Host:** Single entry point for all remote access.
*   **MFA:** Required for all remote sessions.
*   **Recording:** Session recording for audit.
*   **Time-Limited:** Access expires after task completion.
*   **No Direct RDP:** Prevents lateral movement from corporate network to OT devices.

OT12 — **Nation-State Incident Response:**
*   **Monitoring:** IDS/Anomaly detection on OT traffic (e.g., unexpected Modbus writes).
*   **Inventory:** Identify compromised assets immediately.
*   **Steps:** Isolate network segment (switch port shutdown). Do not reboot (preserve evidence).
*   **Coordination:** Control room prioritizes physical safety over IT recovery.
*   **Operator Role:** Monitor physical processes manually if automation is compromised.

OT13 — **HMI:**
*   **Definition:** Interface for operators to view process status and issue commands.
*   **Role:** Visualization and control at Level 2.
*   **Difference:** Engineering Workstation is for configuration/programming (Level 2/3), HMI is for operation (Level 2).

OT14 — **Historian:**
*   **Definition:** Time-series database for process data.
*   **Importance:** Trend analysis, compliance, troubleshooting.
*   **Plant Historian:** Local, real-time, high frequency (Level 3).
*   **Enterprise Historian:** Aggregated, lower frequency, for business analytics (Level 4).

OT15 — **Ladder Logic:**
*   **Definition:** Graphical programming language resembling relay logic.
*   **Device:** Executed by PLCs.
*   **Dominance:** Familiar to electricians, deterministic, safety-certified, legacy code base.

OT16 — **RTU vs PLC:**
*   **RTU:** Remote Terminal Unit. Designed for harsh environments, long-distance comms (cellular/RF).
*   **PLC:** Programmable Logic Controller. Designed for factory floors, local control.
*   **Deploy RTU:** Pipelines, water treatment plants, remote substations.

OT17 — **Analog vs Digital:**
*   **Analog:** Continuous signal (4-20mA, 0-10V).
*   **Digital:** Discrete signal (On/Off, 0/1).
*   **4-20mA:** Preferred over 0-20mA because 4mA is "live zero". 0mA indicates wire break/fault, not just "off".

OT18 — **OT Asset Inventory:**
*   **Foundation:** You cannot secure what you don't know exists.
*   **Tools:** Passive monitoring (packet sniffing) is safe. Active scanning (ping/scan) can crash legacy devices.
*   **Risk:** Active scanning can cause network loops or device resets on old PLCs.

OT19 — **MITRE ATT&CK for ICS:**
*   **Tactics:** Initial Access, Execution, Persistence, Privilege Escalation, Defense Evasion, Impact.
*   **Techniques:** Spearphishing (Initial Access), Exploitation of Remote Services (Execution), Manipulation of Control Logic (Impact).
*   **Difference:** Focuses on physical impact and safety, not just data theft.
*   **Usage:** Map defenses to tactics to find gaps (e.g., do we detect logic manipulation?).

OT20 — **Converged IT/OT SOC:**
*   **Log Collection:** Use passive taps or OT-friendly agents to avoid latency.
*   **SIEM:** Centralize logs but tag OT traffic distinctly.
*   **Rules:** OT-specific rules (e.g., "PLC write to memory").
*   **Staffing:** IT analysts for network, OT engineers for process context.
*   **False Positives:** Tune rules to avoid alert fatigue (OT ops are sensitive to downtime).
*   **Firmware Change:** High severity. Isolate device immediately. Verify with engineering before restoring.
