## Go (20 questions)

**G1 — Easy:** An array is a fixed-size value type (`[N]T`), while a slice is a dynamic reference type (`[]T`) containing a pointer, length, and capacity. When appending to a slice at capacity, the Go runtime allocates a new, larger underlying array (usually doubling capacity), copies elements, and updates the slice header.

**G2:** `var x int` declares a variable with the zero value (`0`) and requires explicit type specification. `x := 0` is a short variable declaration that infers the type. You cannot use `:=` at the package level (outside functions) or if any variable on the left side has already been declared in the same scope.

**G3:** A goroutine is a lightweight thread managed by the Go runtime, not the OS. The Go scheduler (GMP model) maps Goroutines (G) to OS Threads (M) via Processors (P). The scheduler multiplexes many goroutines onto fewer OS threads to maximize CPU utilization and minimize context-switch overhead.

**G4:** Channels are typed conduits for communication between goroutines. Unbuffered channels block both sender and receiver until the other side is ready. Buffered channels allow sending without blocking up to their capacity. Sending to a full buffered channel blocks the sender until a receiver reads from the channel.

**G5:** The `context` package manages deadlines, cancellation signals, and request-scoped values across API boundaries.
```go
ctx, cancel := context.WithTimeout(context.Background(), 5*time.Second)
defer cancel()
resp, err := http.GetContext(ctx, "http://example.com")
```

**G6:** Go interfaces are satisfied implicitly (duck typing) rather than declared explicitly. `interface{}` (or `any`) is the empty interface accepting any type. It is powerful for generic-like behavior but dangerous because it requires type assertions to use values, losing compile-time type safety.

**G7:** This is unsafe due to a data race: one goroutine writes `data` and `ready`, while the main goroutine reads `data` without synchronization. The memory model guarantees no visibility of writes across goroutines without synchronization. Fix by using a `sync.Mutex` or a channel to signal readiness.

**G8:** `sync.Mutex` provides exclusive access; `sync.RWMutex` allows multiple readers or one writer. `sync.Map` is optimized for concurrent read-heavy scenarios with sparse keys. Trade-off: `sync.Map` has higher overhead for writes and iteration compared to a regular `map` protected by a `Mutex`.

**G9:** Go modules manage dependencies via `go.mod` (declarations) and `go.sum` (checksums). `go mod tidy` removes unused dependencies and adds missing ones. Minimum Version Selection (MVS) ensures the lowest compatible versions of transitive dependencies are selected to avoid conflicts.

**G10:** `defer` schedules a function call to run after the surrounding function returns. Defers execute in Last-In-First-Out (LIFO) order. A common gotcha in loops is capturing the loop variable by reference in the deferred closure, causing all defers to see the final loop value.

**G11:** A pointer holds a memory address. Its zero value is `nil`. To create a pointer to a struct literal, use the address-of operator: `&MyStruct{Field: value}`.

**G12:** `init()` functions run automatically before `main()` when a package is loaded. A package can have multiple `init()` functions; they execute in the order they appear in the source files, and files are processed in alphabetical order.

**G13:** `new(T)` allocates zeroed memory for type `T` and returns a pointer `*T`. `make(T)` initializes built-in reference types (`map`, `slice`, `channel`) and returns the initialized value (not a pointer). `new` works with any type; `make` only works with built-ins.

**G14:** Errors are values implementing the `error` interface (`Error() string`). The idiomatic pattern is `if err != nil { return err }`. `errors.Is` checks equality against wrapped errors; `errors.As` retrieves the underlying error type. `==` only works for direct equality, not wrapped errors.

**G15:** Built-in composite types are `map`, `slice`, `channel`, and `struct`. Their zero values are: `nil` (map), `nil` (slice), `nil` (channel), and a struct with all fields set to their respective zero values.

**G16:** A race condition occurs when multiple goroutines access shared data concurrently without synchronization. The `-race` flag instruments code to detect these at runtime. Fix by protecting shared state with `sync.Mutex` or using channels for communication (CSP model).

**G17:** `select` waits on multiple channel operations, choosing one ready case randomly. Unlike `switch`, it evaluates channel readiness dynamically. To implement a timeout: `select { case <-ch: ... case <-time.After(1s): ... }`.

**G18:** Escape analysis determines if a variable lives on the stack or heap. If a variable's address escapes the function scope, it is allocated on the heap. Inspect with `go build -gcflags="-m"`. Heap allocation increases GC pressure, impacting performance.

**G19:** `reflect` allows runtime inspection of types and values. `reflect.Type` describes the type; `reflect.Value` holds the value. It is slow due to dynamic dispatch and type checks. Justified for frameworks (serialization, dependency injection) but avoided in performance-critical paths.

**G20:** `go:embed` embeds files/directories at compile time into a `embed.FS`. It supports single files (`//go:embed file.txt`) or directories (`//go:embed dir/`). Limitations: files must exist at compile time. It integrates with `io/fs.FS` for standard file system interfaces.

## Rust (20 questions)

**R1 — Easy:** Ownership is Rust's core memory safety system. Rules: 1) Each value has one owner. 2) When the owner goes out of scope, the value is dropped. 3) You can transfer ownership (move) but not copy (unless `Copy` trait).

**R2:** `String` is an owned, heap-allocated, growable buffer. `&str` is a borrowed, immutable slice of UTF-8 bytes. Use `String` for owned data; use `&str` for function arguments to avoid unnecessary allocation and copying.

**R3:** Borrowing allows temporary access without ownership. Rules: You can have multiple immutable references (`&T`) OR one mutable reference (`&mut T`), but not both simultaneously. The borrow checker enforces these rules at compile time.

**R4:** `Result<T, E>` represents success or error; `Option<T>` represents a value or `None`. The `?` operator propagates errors.
```rust
fn read_file() -> Result<String, std::io::Error> {
    std::fs::read_to_string("file.txt")
}
```

**R5:** `Box<T>` owns data on the heap with single ownership. `Rc<T>` (Reference Counted) allows multiple owners in a single thread. `Arc<T>` (Atomic Reference Counted) allows multiple owners across threads. Use `Arc` when sharing data between threads.

**R6:** Traits define shared behavior. Unlike Go, Rust requires explicit implementation (`impl Trait for Type`). `dyn Trait` enables dynamic dispatch (vtable); `T: Trait` enables static dispatch (monomorphization). Static dispatch is generally faster.

**R7:** The function fails because the compiler cannot determine which input's lifetime the returned reference belongs to. Fix by adding a lifetime parameter: `fn longest<'a>(x: &'a str, y: &'a str) -> &'a str { ... }`.

**R8:** `Send` means a type can be transferred to another thread; `Sync` means it can be referenced from another thread. `Rc<T>` is not `Send` because its reference count is not atomic. `Arc<T>` uses atomic counters, making it `Send` and `Sync`.

**R9:** `async/await` creates a state machine (`Future`) that yields control. The executor (e.g., Tokio) polls the future to completion. Unlike Go's stackful goroutines, Rust futures are stackless, allowing fine-grained control and zero-cost concurrency.

**R10:** `unsafe` allows operations the compiler cannot verify (raw pointers, FFI, union access). Justified when interfacing with C, implementing low-level primitives, or optimizing performance-critical paths where safety is manually guaranteed.

**R11:** `match` exhaustively checks all possible patterns. `if let` checks a single pattern. The compiler enforces exhaustiveness in `match`, ensuring no unhandled cases (e.g., missing `None` in `Option`), preventing runtime errors.

**R12:** Arrays `[T; N]` have fixed size known at compile time. `Vec<T>` is a growable heap-allocated vector. When capacity is exceeded, `Vec` allocates a new, larger buffer (usually doubling) and moves elements.

**R13:** Enums are Algebraic Data Types (ADTs) that can hold different types of data. `Option<T>` is `Some(T)` or `None`. Rust has no `null`; `Option` forces explicit handling of absence, preventing null pointer exceptions.

**R14:** `impl` defines methods. `&self` borrows the instance; `&mut self` mutably borrows; `self` takes ownership. Associated functions (e.g., `String::new()`) do not take `self` and act like static methods.

**R15:** `derive` automatically implements traits (e.g., `#[derive(Debug)]`). Limitations: Cannot derive traits requiring custom logic or if fields contain types that don't implement the trait (e.g., lifetimes in `Clone`).

**R16:** `Cow<'a, T>` (Clone on Write) holds either a borrowed reference or an owned value. It avoids allocation if the data is only read. If mutation is needed, it clones the data internally, optimizing read-heavy workloads.

**R17:** `mod` defines a module; `use` imports paths; `pub` makes items public. Modules map to files (`mod.rs` or `module.rs`). Rust 2018 changed the crate root to `lib.rs`/`main.rs` and improved path resolution (`use crate::...`).

**R18:** `Pin<T>` prevents a value from being moved in memory. Needed for self-referential structs and async futures (which may hold references to themselves). `Unpin` is a marker trait indicating a type can be safely moved; most types are `Unpin`.

**R19:** Declarative macros (`macro_rules!`) match patterns and generate code. Procedural macros (derive, attribute) operate on Rust AST. Hygiene ensures macro variables don't conflict with local variables, preventing accidental shadowing.

**R20:** Zero-cost abstractions mean high-level code compiles to efficient machine code. Generics use monomorphization (code duplication per type) instead of vtables. Iterators are inlined. This matches C performance while providing safety and ergonomics.
