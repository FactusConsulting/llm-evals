AA1 — **Monolith vs. Microservices:** A monolith is a single, unified codebase deployed as one unit. Microservices are small, independent services deployed separately.
**Problems introduced by Microservices:**
1.  **Distributed Complexity:** Managing transactions, data consistency, and communication across network boundaries.
2.  **Operational Overhead:** Requires sophisticated orchestration (Kubernetes), monitoring, and logging aggregation.
3.  **Network Latency:** Inter-service communication adds latency compared to in-process calls.

AA2 — **Sync vs. Async:** Synchronous communication blocks the caller until a response is received (request/response). Asynchronous communication allows the caller to continue without waiting (fire-and-forget or callback).
**Examples:** Synchronous: HTTP/REST, gRPC. Asynchronous: RabbitMQ, Apache Kafka.

AA3 — **API Gateway vs. Reverse Proxy:** A reverse proxy forwards traffic to backend servers (L4/L7). An API Gateway adds business logic like authentication, rate limiting, and API versioning.
**Replacement:** An API Gateway cannot entirely replace a reverse proxy in all cases. Pure L4 load balancing (TCP/UDP) or legacy protocol passthrough often requires a dedicated proxy or load balancer layer before the gateway.

AA4 — **Saga Pattern:** A sequence of local transactions where each updates data and triggers the next. If one fails, compensating transactions undo previous steps.
**Choreography vs. Orchestration:** Choreography uses events to trigger steps (loose coupling, hard to debug). Orchestration uses a central coordinator (tighter control, easier error handling).
**Appropriate Use:** Choreography for simple, low-coupling flows. Orchestration for complex workflows requiring compensation logic and visibility.

AA5 — **CQRS:** Command Query Responsibility Segregation separates read (Query) and write (Command) models.
**Pairing with Event Sourcing:** Event Sourcing stores state changes as immutable events. CQRS uses these events to build optimized read projections.
**Problem Solved:** Allows independent scaling of read/write workloads and provides a complete audit trail/history of state changes, which traditional CRUD models obscure.

AA6 — **Circuit Breaker:** Prevents repeated calls to a failing service.
**States:** Closed (normal), Open (fail fast), Half-Open (test recovery).
**Difference from Retry:** Retry attempts to recover immediately; Circuit Breaker stops attempts to prevent cascading failures.
**Use Together:** Use Retry for transient network glitches, Circuit Breaker for persistent service outages.

AA7 — **Multi-tenant Isolation Models:**
1.  **Shared DB/Schema:** Lowest cost, highest noisy-neighbor risk, complex security (row-level).
2.  **Shared DB/Schema:** Medium cost, better isolation (schema per tenant), harder migrations.
3.  **Separate DB:** Highest cost, best isolation/security, lowest noisy-neighbor risk, high operational complexity.

AA8 — **Consistency & Inventory:**
**Model:** Strong Consistency (to prevent overselling).
**Data Store:** Distributed SQL (e.g., CockroachDB, TiDB) or Sharded SQL with row-level locking.
**Patterns:** Optimistic Concurrency Control (version numbers) or Distributed Locking (Redis/Zookeeper) to serialize inventory decrements.

AA9 — **Sidecar Proxy:** A proxy deployed alongside the application container.
**Envoy/Istio:** Envoy handles traffic (Data Plane); Pilot configures Envoy (Control Plane).
**mTLS:** Application-level TLS secures host-to-host. mTLS between sidecars authenticates service identity (SPIFFE/SPIRE) and encrypts traffic without code changes, preventing man-in-the-middle attacks between services.

AA10 — **Strangler Fig Pattern:** Incrementally migrate by routing specific functionality to new services while keeping the legacy system.
**Strategy:** Deploy .NET 8 services behind an API Gateway. Route specific endpoints (e.g., `/api/orders`) to new services. Gradually move more endpoints. Decommission legacy .NET Framework code once all routes are migrated.

AA11 — **12-Factor App:**
1.  **Codebase:** One codebase tracked in version control.
2.  **Dependencies:** Explicitly declared and isolated.
3.  **Config:** Stored in environment variables.
4.  **Backing Services:** Treated as attached resources.
5.  **Build/Release/Run:** Strictly separated.
6.  **Processes:** Stateless, state stored in backing services.
**Why:** Ensures portability, scalability, and consistency across environments (Dev/Prod).

AA12 — **Event-Driven (Kafka) vs. Request-Driven (REST):**
**Coupling:** Event (Low) vs. Request (High).
**Scalability:** Event (High, independent scaling) vs. Request (Moderate).
**Debugging:** Event (Hard, distributed state) vs. Request (Easy, synchronous stack).
**Ordering:** Event (Possible, per-partition) vs. Request (Guaranteed per session).
**Failure:** Event (Dead-letter queues) vs. Request (Timeouts).
**Choice:** Event for decoupling/async processing; Request for real-time user interaction.

AA13 — **Load Balancer:** Distributes traffic across servers.
**L4 vs. L7:** L4 (Transport Layer) routes by IP/Port. L7 (Application Layer) routes by HTTP headers/URL.
**Use Case:** SSL Termination or Path-based routing (e.g., `/api` to Service A, `/static` to Service B) requires L7.

AA14 — **Message Queue vs. Log Broker:**
**MQ (RabbitMQ):** Tasks, acks, delete after read. Good for work queues.
**Log Broker (Kafka):** Streams, retention, replay. Good for event sourcing/audit.
**Pick:** MQ for task processing; Kafka for high-throughput event streaming and replayability.

AA15 — **Container vs. VM:** Containers share the host OS kernel; VMs virtualize hardware and run a full OS.
**Popularity:** Containers are lightweight, start faster, and have lower overhead, making them ideal for microservices scaling.

AA16 — **Stateless vs. Stateful:** Stateless services store no session data (e.g., Web Server). Stateful services store session data (e.g., Database).
**Scaling:** Stateless services are easier to scale horizontally because any instance can handle any request without data affinity.

AA17 — **ORM:** Maps database tables to programming language objects.
**Examples:** Hibernate (Java), Entity Framework (.NET).
**Trade-offs:** ORMs increase development speed but can generate inefficient SQL and reduce fine-grained control over performance.

AA18 — **Bulkhead Pattern:** Isolates resources (threads, connections) for specific services.
**Implementation:** Use separate thread pools for each downstream service call. If one service hangs, it only exhausts its pool, not the application's total resources.

AA19 — **OT vs. CRDT:**
**OT:** Server-authoritative, complex conflict resolution, requires online connection.
**CRDT:** Client-authoritative, mathematically guarantees convergence, supports offline editing.
**Choice:** CRDT for offline support. It allows local edits to merge automatically without server mediation.

AA20 — **Distributed Tracing:**
**Propagation:** W3C Trace Context (TraceID/SpanID) passed via HTTP headers or Kafka headers.
**Sampling:** Head-based (early decision) vs. Tail-based (late decision based on latency).
**Storage:** Jaeger/Tempo store spans for querying.
**Kafka:** Inject trace context into message headers to maintain continuity across async boundaries.

OP1 — **Hypervisor Type 1 vs. Type 2:**
**Type 1 (Bare-metal):** Runs directly on hardware (e.g., VMware ESXi, Microsoft Hyper-V).
**Type 2 (Hosted):** Runs on top of an OS (e.g., VirtualBox, VMware Workstation).
**Difference:** Type 1 offers better performance and security for production; Type 2 is for development/testing.

OP2 — **SAN vs. NAS:**
**SAN (Storage Area Network):** Block-level storage (e.g., iSCSI, Fibre Channel). Used for databases requiring low latency.
**NAS (Network Attached Storage):** File-level storage (e.g., NFS, SMB). Used for file shares and backups.

OP3 — **3-Tier vs. Leaf-Spine:**
**3-Tier:** Core, Distribution, Access. Can suffer from oversubscription and high latency.
**Leaf-Spine:** All-to-all connectivity. Solves oversubscription and provides consistent low latency, preferred for modern data centers.

OP4 — **Proxmox HA Architecture:**
**Nodes:** Minimum 3 for quorum.
**Quorum:** Majority vote required to avoid split-brain.
**Fencing:** STONITH (e.g., DRAC/iLO) to power off failed nodes.
**Storage:** Ceph or iSCSI shared storage.
**Failure:** VMs restart on surviving nodes automatically.

OP5 — **VLAN Trunking:**
**802.1Q:** Adds a tag to Ethernet frames to identify VLAN membership.
**Linux Config:** Create sub-interfaces (e.g., `eth0.10`, `eth0.20`) using `ip link add` or `vconfig`. Assign IP addresses to sub-interfaces.

OP6 — **RAID Levels:**
**RAID 1:** Mirroring (50% capacity).
**RAID 5:** Striping + Parity (1 drive loss).
**RAID 6:** Striping + Double Parity (2 drive loss).
**RAID 10:** Striping + Mirroring (50% capacity, high perf).
**RAID Z2:** ZFS equivalent to RAID 6.
**Choice:** RAID 10 for a DB server (6 drives) to maximize write performance and fault tolerance.

OP7 — **DNS Architecture:**
**Internal:** AD DNS for clients.
**External:** Public DNS for DMZ.
**Split-Horizon:** Internal clients get internal IPs; external get public IPs.
**Forwarders:** Internal DNS forwards external queries to ISP/Cloud.
**Laptop:** DHCP updates DNS records; split-horizon resolves based on network interface (LAN vs. VPN).

OP8 — **PKI Chain:**
**Root CA:** Offline, highly secure.
**Subordinate CA:** Online, issues end-entity certs.
**Revocation:** CRL (list) vs. OCSP (real-time check).
**Compromise:** Subordinate = Revoke and reissue. Root = Catastrophic (trust broken globally).

OP9 — **WAN Consolidation:**
**MPLS:** Reliable, expensive.
**IPsec:** Cheap, variable latency.
**SD-WAN:** Intelligent routing, cost-effective, reliable.
**WireGuard:** Simple, secure, mesh.
**Recommendation:** SD-WAN for 50 users/site with VoIP (prioritizes traffic, handles internet variability).

OP10 — **iDRAC/iLO/IPMI:**
**Function:** Out-of-band hardware management (power, console).
**Critical:** Allows recovery if OS is down.
**Risk:** Exposed to internet = Remote root access, ransomware, hardware hijacking.

OP11 — **LVM vs. ZFS:**
**LVM:** Block management. Snapshots are copy-on-write (COW).
**ZFS:** Filesystem + Volume. Snapshots are instant, COW, checksummed.
**Use:** ZFS for data integrity/replication. LVM for flexible block allocation.

OP12 — **Stretch Cluster:**
**Design:** 2 sites + Tiebreaker (3rd site or cloud witness).
**Ceph:** CRUSH rules distribute data across sites.
**Latency:** < 5ms between sites.
**Split-Brain:** Without tiebreaker, both sites may write, corrupting data. Tiebreaker ensures only one site is primary.

OP13 — **DHCP DORA:**
**Process:** Discover, Offer, Request, Acknowledge.
**Unavailable:** Client uses APIPA (169.254.x.x) if lease expires and server is unreachable.

OP14 — **Managed vs. Unmanaged Switch:**
**Managed:** Supports VLANs, QoS, ACLs, SNMP.
**Importance:** Essential for segmentation, traffic prioritization, and monitoring in enterprise networks.

OP15 — **NTP:**
**Importance:** Critical for logs, Kerberos auth, and clustering.
**Drift:** Causes auth failures, log correlation issues, and split-brain in clusters.

OP16 — **Physical vs. VM vs. Container:**
**Physical:** Direct hardware access.
**VM:** Virtualized OS.
**Container:** Virtualized App.
**Bare Metal:** Use for high-performance DBs, HPC, or low-latency requirements where virtualization overhead is unacceptable.

OP17 — **UPS vs. Generator:**
**UPS:** Short-term power (minutes) for graceful shutdown.
**Generator:** Long-term power (hours/days).
**ATS:** Automatically switches power source from Grid to Generator when grid fails.

OP18 — **Active-Passive vs. Active-Active:**
**Active-Passive:** One node handles traffic; failover to standby.
**Active-Active:** Both handle traffic.
**PostgreSQL HA:** Pacemaker/Corosync manages VIP. STONITH fencing ensures failed node is isolated. VIP moves to active node.

OP19 — **Backup/Recovery:**
**3-2-1:** 3 copies, 2 media types, 1 offsite.
**RPO/RTO:** Recovery Point/Time Objectives.
**Immutable:** Prevents ransomware deletion.
**Test:** Regular restore drills.
**Difference:** Backup = Data copy; DR = Site recovery strategy.

OP20 — **802.1X:**
**Roles:** Supplicant (Client), Authenticator (Switch), Server (RADIUS).
**EAP:** EAP-TLS (Cert-based, secure) vs. PEAP (User/Pass).
**Failure:** Port blocked or placed in Guest VLAN.
**MAB:** Authenticates device by MAC address if 802.1X fails (for printers/IoT).
