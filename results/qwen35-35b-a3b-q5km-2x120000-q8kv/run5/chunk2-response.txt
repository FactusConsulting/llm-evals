K1 — Easy: A **Deployment** manages stateless applications, ensuring a desired number of identical pod replicas are running. A **StatefulSet** manages stateful applications, providing stable, unique network identifiers (hostnames) and ordered, graceful deployment and scaling.

K2 — Easy: A **ClusterIP** Service assigns an internal virtual IP. **kube-proxy** (using iptables or IPVS) updates the node's networking rules to forward traffic sent to the Service IP to the IP addresses of the pods listed in the Service's **Endpoints** object.

K3 — Medium: A **PersistentVolume (PV)** is a cluster-level storage resource. A **PersistentVolumeClaim (PVC)** is a user request for storage. A **StorageClass** defines the provisioning class for dynamic PV creation. A PVC binds to an existing PV (static) or triggers the StorageClass to provision a new PV (dynamic).

K4 — Medium: 1. Check application logs (`kubectl logs`) for errors. 2. Check pod events (`kubectl describe`) for OOMKilled or probe failures. 3. Verify resource limits vs. node capacity. 4. Inspect liveness/readiness probe configurations for incorrect paths or timeouts.

K5 — Medium: CoreDNS provides DNS resolution for Kubernetes services (service discovery). If CoreDNS pods are down, internal service discovery (e.g., `my-service.default.svc`) fails, though external DNS resolution may still work depending on the cluster's `resolv.conf` configuration.

K6 — Hard: The scheduler **filters** nodes (removes those with taints, insufficient resources, or affinity mismatches), then **scores** remaining nodes (based on priority, affinity, and resource balance). Factors considered: CPU/Memory requests, Node Affinity rules, and Pod Affinity rules.

K7 — Hard: Yes, it will be scheduled (256Mi request < 0.5GB free RAM). At runtime, if it uses 600Mi, it exceeds the 512Mi limit, triggering an **OOMKilled** event, causing the pod to restart.

K8 — Hard: 1. Request hits External Load Balancer. 2. LB forwards to Ingress Controller Pod. 3. Ingress Controller parses Ingress rules and routes to Kubernetes Service. 4. Service resolves to Pod IPs via Endpoints. 5. kube-proxy (iptables/IPVS) directs traffic to the specific App Pod.

K9 — Medium: A **NetworkPolicy** restricts traffic flow between pods. Default behavior: All traffic is allowed. When a policy is applied, traffic is **default-deny** for the selected pods unless explicitly allowed by the policy rules.

K10 — Medium: **RollingUpdate** replaces pods gradually (zero downtime). **Recreate** kills all existing pods before starting new ones. Choose **Recreate** for stateful applications where simultaneous instances would cause data conflicts or licensing issues.

K11 — Easy: A **Namespace** provides logical isolation within a cluster. Default namespaces: `default`, `kube-system`, `kube-public`, `kube-node-lease`.

K12 — Easy: `kubectl describe` shows detailed status. Information not in `get`: **Events** (warnings/errors), **Conditions** (Ready status details), and **Mounts** (volume attachment details).

K13 — Easy: A **ConfigMap** stores non-confidential configuration data. Pods consume it via: 1. **Environment Variables** in the pod spec. 2. **Volume Mounts** to expose keys as files in the container filesystem.

K14 — Easy: **ClusterIP**: Internal cluster access only. **NodePort**: Exposes service on a static port on each node's IP. **LoadBalancer**: Provisions an external cloud load balancer to access the service.

K15 — Easy: A **DaemonSet** ensures a copy of a pod runs on all (or specific) nodes. Use instead of Deployment for system-level agents like logging (Fluentd), monitoring (Node Exporter), or storage daemons.

K16 — Medium: **Taints** repel pods from a node; **Tolerations** allow pods to schedule on tainted nodes. **Node Affinity** prefers nodes but doesn't repel. Use case: Taint for dedicated GPU nodes; Affinity for co-locating related services.

K17 — Medium: A **Kubernetes Operator** is a custom controller that manages the lifecycle of a complex application (CRD + Controller). It differs from standard controllers by automating operational tasks (backups, upgrades) specific to that application.

K18 — Hard: etcd uses **strong consistency** (linearizable). If quorum is lost, the cluster becomes read-only/unavailable. Recovery: Restore from a recent etcd snapshot, rejoin lost nodes to the cluster, and verify data integrity.

K19 — Hard: **Role** (namespaced permissions), **ClusterRole** (cluster-wide), **RoleBinding** (links Role to subject in namespace), **ClusterRoleBinding** (links ClusterRole to subject cluster-wide).
```yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata: namespace: default
name: pod-reader
rules:
- apiGroups: [""]
  resources: ["pods", "pods/log"]
  verbs: ["get", "list", "watch"]
---
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata: namespace: default
name: read-pods-binding
subjects:
- kind: User
  name: developer
  apiGroup: rbac.authorization.k8s.io
roleRef:
  kind: Role
  name: pod-reader
  apiGroup: rbac.authorization.k8s.io
```

K20 — Hard: `ndots: 5` causes the resolver to try 5 suffixes for every query (e.g., `google.com` becomes 5 DNS lookups). Fix: Set `dnsConfig` in pod spec: `options: [{ name: "ndots", value: "1" }]` to reduce queries.

D1 — Easy: **Rebase** rewrites history to create a linear commit chain. **Merge** preserves history with a merge commit. Avoid rebasing on public/shared branches to prevent history conflicts for others.

D2 — Easy: **TCP** is reliable and connection-oriented (HTTP, SSH). **UDP** is unreliable and connectionless (DNS, VoIP).

D3 — Medium: A **race condition** occurs when output depends on the sequence of execution. Example: Two threads incrementing a shared counter without a lock results in a lost update.

D4 — Medium: **Symmetric** uses one key; **Asymmetric** uses a public/private key pair. TLS uses Asymmetric encryption for the handshake (key exchange) and Symmetric encryption for the data transfer session.

D5 — Medium: A **layer** is a filesystem diff in a container image. Best practice: Place stable commands (install dependencies) before volatile commands (copy source code) to maximize Docker layer caching.

D6 — Medium: 1. Check application error logs. 2. Check upstream service health. 3. Review load balancer logs. 4. Monitor resource utilization (CPU/Memory) for throttling or OOM.

D7 — Medium: **CAP Theorem**: Consistency, Availability, Partition Tolerance. **CP** (Consistency + Partition Tolerance): Zookeeper. **AP** (Availability + Partition Tolerance): DynamoDB.

D8 — Medium: **Horizontal Scaling** adds more nodes; **Vertical Scaling** adds more resources to a node. Patterns enabling horizontal scaling: Stateless services, Sharding, Load Balancing.

D9 — Easy: An **environment variable** is runtime configuration. To persist across sessions on Linux, export it in `~/.bashrc` or `~/.profile`.

D10 — Easy: **JSON** is strict key-value syntax; **YAML** is human-readable indentation-based syntax. Advantage JSON: Machine parsing. Advantage YAML: Readability for config files.

D11 — Easy: **REST** is an architectural style. **GET** (retrieve), **POST** (create), **PUT** (replace), **DELETE** (remove).

D12 — Easy: **Version pinning** locks dependencies to exact versions. Important for reproducible builds to prevent breaking changes from upstream updates.

D13 — Easy: **Compiled** languages translate to binary before execution (C, Go). **Interpreted** languages execute code line-by-line (Python, JavaScript).

D14 — Medium: A **reverse proxy** receives client requests and forwards them to backend servers. A **forward proxy** acts on behalf of clients to access the internet. Servers: Nginx (Web server), HAProxy (Load balancer).

D15 — Medium: A **deadlock** is a permanent blocking state. Coffman conditions: Mutual exclusion, Hold and wait, No preemption, Circular wait.

D16 — Medium: **Eventual consistency** means updates propagate over time. Cassandra uses **Read Repair** and **Hinted Handoff**. Tunable levels: ONE, QUORUM, ALL.

D17 — Medium: **mTLS** authenticates both client and server via certificates. Handshake: Both sides exchange and verify certificates. Commonly used in **Service Meshes** (e.g., Istio) for secure service-to-service communication.

D18 — Medium: A **hash table** maps keys to indices. Collision resolution: **Chaining** (linked lists) or **Open Addressing** (probing). Complexity: Average O(1), Worst O(n).

D19 — Medium: **Circuit Breaker** prevents cascade failures. States: **Closed** (normal), **Open** (fail fast), **Half-open** (test recovery). Relates to retries: Do not retry if circuit is open.

D20 — Medium: **Optimistic** checks version on commit (CAS). **Pessimistic** locks data on read (`SELECT FOR UPDATE`). Choose Optimistic for low conflict/high concurrency; Pessimistic for high conflict/critical data.
