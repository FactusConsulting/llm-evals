CL1 — **IaaS** provides virtualized computing resources (servers, storage) over the internet (e.g., AWS EC2). **PaaS** provides a platform allowing customers to develop, run, and manage applications without dealing with underlying infrastructure (e.g., AWS Elastic Beanstalk). **SaaS** delivers software applications over the internet on a subscription basis (e.g., Microsoft 365).

CL2 — The **Shared Responsibility Model** divides security duties. **IaaS VMs:** Provider secures physical hardware/hypervisor; Customer secures OS, apps, data, and network config. **Managed Kubernetes:** Provider secures control plane/infrastructure; Customer secures nodes, pods, network policies, and secrets. **SaaS (M365):** Provider secures app/infrastructure; Customer secures identity, data classification, and access policies.

CL3 — **VPC** is an isolated virtual network. **Subnets** divide the VPC into IP ranges. **Security Groups** act as stateful firewalls at the instance level. **NACLs** act as stateless firewalls at the subnet level. They layer as: VPC contains Subnets, which contain Instances protected by SGs. Stateful (SG) remembers return traffic; Stateless (NACL) requires explicit rules for both directions.

CL4 — A **Landing Zone** is a secure, compliant baseline environment. **OUs** organize accounts logically. **Guardrails** enforce policies (e.g., no public S3 buckets). **Shared Networking** (Transit Gateway/Hub-and-Spoke) centralizes connectivity. **Centralized Logging** aggregates logs for security monitoring. Single-account approaches risk **blast radius** (one breach compromises all), billing confusion, and permission sprawl.

CL5 — **Infrastructure as Code (IaC)** manages infra via machine-readable definition files. **Terraform/OpenTofu:** Multi-cloud, declarative, state-managed. **Pulumi:** Code-based (Python/TS), imperative/declarative. **CloudFormation/Bicep:** AWS/Azure native, tightly integrated. **Crossplane:** K8s-native, manages cloud resources via K8s API. Choose Terraform for multi-cloud, Pulumi for dev-heavy teams, CFN/Bicep for cloud-native simplicity, Crossplane for K8s-centric ops.

CL6 — **HPA** scales pods based on CPU/Memory. **VPA** adjusts pod resource requests/limits. **Cluster Autoscaler** adds/removes nodes. **KEDA** scales based on external events (e.g., Kafka queue depth). Use **KEDA** when scaling is driven by event-driven workloads (message queues, timers) rather than standard resource metrics.

CL7 — **DR Strategy:** Define RPO (data loss tolerance) and RTO (downtime tolerance). **Active-Active** (low RPO/RTO, high cost) vs **Active-Passive** (higher RTO, lower cost). **DB:** Sync (strong consistency, latency) vs Async (replication lag). **DNS:** Route53 health checks for failover. **State:** Externalize state (S3/DB). **Pilot Light** (DB running, app off), **Warm Standby** (scaled app), **Multi-site** (full active).

CL8 — **GitOps** uses Git as the single source of truth for infrastructure/app state. **ArgoCD** (push/pull) vs **FluxCD** (pull). **Secrets:** Sealed Secrets, External Secrets Operator, or Vault integration. **Drift:** Controllers detect manual changes, automatically revert them to match the Git state to ensure consistency.

CL9 — Use **StatefulSet** for stable network IDs and ordered deployment. **Local PV** (hostPath) for low latency. **Headless Service** for direct pod DNS. **Init Containers** for setup. **ConfigMaps** mounted as volumes for config. **PDB** ensures minimum availability during updates. Ensure storage class supports local binding.

CL10 — **Service Account** is K8s identity. **Workload Identity** maps SA to cloud identity. **IRSA** (AWS) maps SA to IAM role via OIDC. Mounting credentials is bad practice (leakage, rotation issues). **Federation** solves this by issuing short-lived tokens via OIDC, eliminating static secrets.

CL11 — **Object Storage (S3):** Unstructured data, high durability (11 9s), eventual consistency. **Block Storage (EBS):** Attached disk, low latency, high IOPS. **File Storage (EFS):** Shared FS, POSIX compliant. Use Object for backups/media, Block for DBs/OS, File for shared app configs.

CL12 — **Zero Trust:** Verify every request. **Network Policies** deny all by default. **Service Mesh** (mTLS) encrypts east-west traffic. **Pod Identity** (SPIFFE) authenticates workloads. **Admission Controllers** (OPA) enforce policies. **Image Signing** ensures trust. **Runtime Security** (Falco) detects anomalies. Differs from perimeter model by assuming internal threats exist.

CL13 — **CDN** caches content at edge locations to reduce latency. **CloudFront** (AWS), **Cloudflare**. Use CDN for static assets (images, JS) to offload origin and improve global performance. Use origin directly for dynamic, personalized, or real-time data.

CL14 — **Availability Zone (AZ):** Isolated data center within a Region. **Region:** Geographic area with multiple AZs. Deploy across AZs for fault tolerance (hardware failure). **Zonal** (EBS) tied to one AZ; **Regional** (ELB) spans AZs.

CL15 — **Managed DB (RDS):** Provider handles provisioning, patching, backups, failover, and scaling. You handle schema, queries, and access control. Reduces operational overhead but limits low-level OS access.

CL16 — **IAM Users** (people), **Groups** (roles), **Roles** (temp access), **Policies** (permissions). **Least Privilege:** Grant only necessary permissions. Apply via policies attached to roles/groups, avoiding root/admin usage.

CL17 — **Serverless:** Event-driven, no server mgmt. **Lambda/Functions.** **Cold Starts:** Latency on first invocation due to provisioning. Poor fit for long-running processes, stateful apps, or predictable high-throughput workloads.

CL18 — **Ingress:** K8s resource for HTTP routing. **Gateway API:** Standardized, multi-cluster support. **LB:** Cloud provider load balancer. **Mesh:** Sidecar termination. TLS: Ingress terminates at LB/Ingress; Mesh terminates at sidecar. Use **Gateway API** for complex routing, multi-cluster, or vendor-agnostic needs.

CL19 — **Cost Opt:** Reserved Instances/Savings Plans (commitment), Spot (cheap, interruptible), Right-sizing, Storage Tiering. **FinOps:** Culture/process for cost visibility. **Tools:** AWS Cost Explorer, Kubecost. **Showback/Chargeback:** Allocate costs to teams via tags/labels to drive accountability.

CL20 — **Multi-cloud:** Use K8s for runtime, Terraform for IaC. **Networking:** Direct Connect/ExpressRoute. **Identity:** OIDC federation. **Trade-offs:** Complexity, latency, cost. **Justified:** Compliance, disaster recovery, negotiation leverage. **Unnecessary:** Startups, simple apps, where single-cloud benefits outweigh risks.

OT1 — **Purdue Model (ISA-95):** L0 (Physical Process), L1 (Basic Control/PLCs), L2 (Area Supervisory/SCADA), L3 (Operations/Manufacturing Execution), L4 (Enterprise/Business Planning).

OT2 — **IT:** Data confidentiality, patching, uptime. **OT:** Safety, availability, legacy hardware. Standard IT patching risks downtime or safety failures in OT due to untested compatibility and real-time constraints.

OT3 — **DMZ (IDMZ):** Buffer zone between IT and OT. **Traffic:** Historian sync (OT->IT), limited commands (IT->OT). **Direct L4->L1:** Dangerous as it bypasses security controls, allowing lateral movement from corporate network to critical control systems.

OT4 — **Modbus TCP:** L1/L2 (Control). **OPC UA:** L2/L3 (Data exchange). **EtherNet/IP:** L1/L2 (Control). **PROFINET:** L1/L2 (Control). **MQTT:** L2/L3/Cloud (Telemetry).

OT5 — **Data Diode:** Hardware enforcing unidirectional flow (OT->IT). **Firewall:** Bidirectional filtering. Use diode for high-security data export. **Limitation:** No feedback/control from IT to OT, limiting remote management capabilities.

OT6 — **Flow:** PLC -> Historian -> OPC UA -> Edge Gateway -> MQTT -> Cloud. **Security:** Place firewall between Edge and Cloud. Encrypt traffic (TLS). Isolate Edge Gateway in DMZ.

OT7 — **IEC 62443:** **Zones** (logical groups with similar security needs), **Conduits** (controlled traffic paths). **Model:** Corporate (L4) -> DMZ -> SCADA (L3) -> PLCs (L1). Conduits enforce strict access control between zones.

OT8 — **Segmentation:** VLANs per production line. Firewalls between VLANs and IT. **Switches:** Managed industrial switches. **Vendor Access:** Jump host with MFA. **Transition:** Phased rollout, parallel run, backup before changes to avoid production stoppage.

OT9 — **SIS:** Independent safety system (shutdown). **BPCS:** Basic process control. **IEC 61511:** Safety lifecycle standard. **SIS Isolation:** Prevents cyber attacks from disabling safety functions. **Ransomware on SIS:** Can disable safety shutdowns, leading to physical damage or injury.

OT10 — **PLC:** Logic controller for machines. **DCS:** Integrated control for continuous processes. **SCADA:** Supervisory monitoring. **Coexist:** Yes, PLCs/DCS control, SCADA monitors.

OT11 — **Remote Access:** Jump host, MFA, Session Recording, Time-limited. **Direct RDP:** Unacceptable as it exposes HMI to corporate threats, lacks audit trails, and allows malware propagation.

OT12 — **Detection:** NIDS, Anomaly detection on OT traffic. **Response:** Isolate compromised segment, notify Ops. **Ops Role:** Monitor physical safety, manual override if needed. **Coordination:** IT/OT joint incident response team.

OT13 — **HMI:** Operator interface for monitoring/control. **Difference:** HMI is for operations; Engineering Workstation is for configuration and programming.

OT14 — **Historian:** Time-series database for process data. **Plant (L3):** Real-time operational data. **Enterprise (L4):** Long-term analytics and reporting.

OT15 — **Ladder Logic:** Relay-based programming language. **Device:** PLC. **Dominant:** Reliability, safety certification, and deep industry familiarity.

OT16 — **RTU:** Remote Terminal Unit, rugged, comms-focused. **PLC:** Local, control-focused. **RTU Use:** Remote sites (pipelines, power grids) with limited connectivity.

OT17 — **Analog:** Continuous signal (4-20mA). **Digital:** Discrete (On/Off). **4-20mA:** Preferred because 4mA is "live zero" (detects wire break at 0mA).

OT18 — **Asset Inventory:** Foundation of security (know what you have). **Tools:** Passive (sniffing), Active (scanning). **Active Scanning:** Dangerous as it can crash legacy OT devices.

OT19 — **MITRE ATT&CK ICS:** Framework for ICS threats. **Tactics:** Initial Access, Execution, Persistence, Privilege Escalation, Impact. **Difference:** Focuses on safety/availability vs enterprise confidentiality. **Use:** Assess defensive gaps against known ICS attack patterns.

OT20 — **Converged SOC:** Passive log collection (IDS), SIEM integration. **Rules:** OT-specific (e.g., PLC write attempts). **Staffing:** OT engineers for triage. **False Positives:** High noise requires tuning. **Firmware Change:** Stop process, isolate, investigate, revert, patch.
