SC1-A
The pods are failing due to a conflict between the **Restricted Pod Security Standard (PSS)** and the container image configuration.
1.  **Restricted PSS:** Requires `runAsNonRoot: true` AND explicitly sets `runAsUser` to a non-zero UID. It forbids running as root.
2.  **Image Behavior:** The new base image (`cgr.dev/chainguard/static:latest`) defaults `USER` to root (UID 0).
3.  **Manifest Behavior:** The manifests set `runAsNonRoot: true` but omit `runAsUser`.
4.  **The Failure:** When the pod starts, the kubelet attempts to enforce the security context. Since `runAsNonRoot` is true, the container must not run as UID 0. However, the image defaults to UID 0, and no override is provided in the pod spec. The container runtime (containerd/cri-o) refuses to start the container as root, causing the CrashLoopBackOff.
5.  **Baseline vs. Restricted:** Baseline allows `runAsNonRoot` without `runAsUser` (allowing root if image defaults to it). Restricted enforces non-root strictly, requiring an explicit UID.

SC1-C
**Remediation Strategy:**
1.  **Immediate Fix (Image):** Rebuild the image with a non-root user. Add `USER 1000` to the Dockerfile.
2.  **Manifest Fix:** Update Deployment specs to explicitly set `securityContext.runAsUser: 1000` (matching the image) and ensure `runAsNonRoot: true`.
3.  **CI Gate:** Add an OPA Gatekeeper or Kyverno policy in CI that scans Helm charts/manifests. Policy: "If `runAsNonRoot` is true, `runAsUser` must be defined and non-zero."
4.  **Rollout:** Use a Canary deployment. Update 10% of pods to the new image/manifest. Monitor CrashLoopBackOff and error rates. If stable, roll out to 100%. This prevents a full outage if the new image has other issues.

SC2-A
**Stale Data & Replication:**
1.  **Cause:** Cross-region replication is asynchronous. Promoting a replica cuts it off from the primary. Any WAL (Write-Ahead Log) entries not shipped from `us-east-1` to `eu-west-1` before the cut are lost.
2.  **WAL Shipping:** PostgreSQL streams WAL records. In async mode, the primary acknowledges the write before the replica confirms receipt.
3.  **Data Loss:** The "lag window" represents potential data loss. If the primary crashed 5 minutes before promotion, those 5 minutes of writes are lost.
4.  **Quantification:** Check `pg_stat_replication` on the old primary (if accessible) or the replica's `pg_last_wal_receive_lsn` vs `pg_last_wal_replay_lsn` before promotion. The difference indicates the lag.

SC2-B
1.  **Import Replica:** `terraform import aws_db_instance.promoted_replica <replica-arn>`
2.  **Update State:** If the old instance ARN is hardcoded in state, use `terraform state mv` to update references if resources were renamed, or update the TF code to point to the new ARN.
3.  **Sync:** Run `terraform plan` to ensure state matches the promoted replica.
4.  **Cleanup:** Once verified, run `terraform destroy` on the old `aws_db_instance` resource (now orphaned in code but in state) to remove the old instance.
5.  **Dependencies:** Update Security Groups and Route53 records in TF to reference the new instance ID/ARN.

SC2-C
**Architecture:**
1.  **RDS Proxy:** Use RDS Proxy instead of PgBouncer. It handles connection pooling at the proxy layer, reducing connection churn during failover and supporting IAM auth.
2.  **Route53:** Use Route53 Failover Routing Policy with health checks on the primary endpoint. On failure, DNS TTL allows propagation to the secondary region.
3.  **App Retry:** Implement exponential backoff with jitter in the application connection logic.
4.  **RPO < 1s:** Standard RDS async replication cannot guarantee <1s RPO. **Recommend Aurora Global Database.** It uses a dedicated replication layer with synchronous commit options and lower latency replication (typically <1s) across regions, ensuring minimal data loss during failover.

SC6-C
**Connectivity Upgrade:**
1.  **AWS Direct Connect (DX):**
    *   *Pros:* Dedicated 10Gbps+ bandwidth, lower latency, consistent performance.
    *   *Cons:* High cost, physical hardware setup.
    *   *Zero Trust:* Fits well; traffic is encrypted (MACsec) and isolated.
2.  **ECMP VPN:**
    *   *Pros:* Faster setup, uses existing internet.

3.  **Mesh (Tailscale/Netbird):**
    *   *Pros:* Zero config, identity-based access (Zero Trust native).
    *   *Cons:* Not suitable for high-throughput data center interconnect (DCI).
    *   *Zero Trust:* Best fit for identity, worst for bandwidth.
**Recommendation:** Use **Direct Connect** for the core data plane (DB, Storage) to meet bandwidth/latency needs. Use **SPIFFE/SPIRE** for the control plane (Zero Trust identity).

SC10-C
**Failover Orchestration:**
1.  **Detection:** Prometheus monitors site health (ping, DB latency). If Chicago is down > 2 mins, trigger alert.
2.  **DNS:** GSLB (Route53) detects failure and switches DNS to Dallas.
3.  **Promotion:** Patroni detects primary loss, promotes Dallas node.
4.  **Redis/RabbitMQ:** Sentinel/Federation automatically promotes replicas.
5.  **MinIO:** Site replication catches up.
6.  **State:** Ansible/Tofu updates state to mark Dallas as Primary.
7.  **Split-Brain:** Use **Fencing** (STONITH). If Chicago comes back, it must be fenced (shutdown) before rejoining to prevent dual-primary.
8.  **Failback:** Once Chicago is stable, run `patroni switchover` to move primary back. Update DNS.
