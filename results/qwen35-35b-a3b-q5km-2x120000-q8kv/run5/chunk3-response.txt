T1 — `tofu plan` calculates and displays the changes that would be made to infrastructure without applying them. `tofu apply` executes those changes to create, update, or destroy resources.

T2 — State is a JSON file mapping real-world resources to configuration objects. Remote state is important for collaboration (sharing state among team members), security (storing secrets securely), and enabling state locking to prevent concurrent modifications.

T3 — `count` uses integer indices to create multiple instances of a resource. `for_each` uses a map or set of keys. `for_each` is preferred when adding or removing items because it avoids index shifting, which can cause unintended resource recreation.

T4 — OpenTofu detects this via "drift detection" during `tofu plan`. It compares the current state file against the live infrastructure. If differences exist, `plan` shows them as changes. `apply` can revert them to match the configuration or fail if the configuration doesn't match the desired state.

T5 —
```hcl
resource "aws_instance" "web" {
  for_each = toset(["web1", "web2", "web3"])
  ami      = "ami-12345678"
  instance_type = "t2.micro"
  tags = {
    Name = each.key
  }
}
```

T6 — `terraform_remote_state` imports the entire state file, while `output` values (accessed via `data` source) import specific values. The recommended alternative is to use `output` values via `data` source with specific keys to minimize state exposure and improve security by not sharing sensitive state data.

T7 — The output is computed before the `remote-exec` provisioner finishes writing the kubeconfig file. Fix: Add `depends_on = [null_resource.kubeconfig]` to the resource consuming the output, or use `local-exec` to generate the file locally first.

T8 — `tofu import` maps existing infrastructure to config manually. Limitations: No state history, manual mapping required. The `import` block (1.5+) improves this by allowing declarative imports in config, making it version-controllable and suitable for CI/CD pipelines.

T9 — A variable is an input parameter passed into a module or configuration. An output is a value exposed by a module or configuration for use by other modules. Variables are inputs; outputs are results.

T10 — A provider is a plugin that interacts with a specific API (e.g., AWS, Azure). Without `tofu init`, `tofu plan` fails because the provider plugins are not downloaded or initialized.

T11 — The `.terraform.lock.hcl` file records the exact versions and checksums of providers used. It should be committed to version control to ensure reproducible builds and prevent unexpected provider updates.

T12 — `tofu destroy` removes all resources managed by the configuration. To target a single resource: `tofu destroy -target=aws_instance.example`.

T13 — A data source reads information from an external system without creating or managing it. Example: Using `data "aws_ami"` to find the latest Ubuntu AMI ID instead of hardcoding it.

T14 — `locals` define computed values within the configuration scope. `variables` define inputs from outside. Use `locals` for intermediate calculations or formatting that shouldn't be exposed as inputs.

T15 — The `lifecycle` block controls resource behavior. `create_before_destroy` creates new resource before destroying old (zero downtime). `prevent_destroy` prevents accidental deletion. `ignore_changes` ignores specific attribute drifts (e.g., tags modified manually).

T16 — A `moved` block allows refactoring resource addresses (e.g., moving a resource to a new module) without destroying and recreating it. It updates the state mapping to reflect the new address.

T17 — The dependency graph is a DAG determining execution order. Implicit dependencies come from code references (e.g., `aws_instance.id`). Explicit dependencies use `depends_on`. `depends_on` reduces parallelism by forcing sequential execution, so it should be used sparingly to avoid slowing down plans.

T18 — 1. Increase `parallelism` (risk: API rate limits). 2. Use `refresh=false` (risk: drift detection skipped). 3. Modularize code (risk: increased complexity). 4. Optimize state backend (S3/DynamoDB) (risk: configuration changes).

T19 — State locking prevents concurrent writes using a backend (e.g., DynamoDB). If a lock persists after a crash, `tofu force-unlock` can break it. Risk: Data corruption if another process is actually holding the lock.

T20 — 1. Backup local state file. 2. Add `backend` block (S3/DynamoDB) to config. 3. Run `tofu init -migrate-state`. 4. Verify state exists in S3 and lock works. 5. Run `tofu plan` to confirm state matches.

A1 — A **task** is a single action (e.g., install package). A **role** is a reusable directory structure for organizing tasks, handlers, and vars. A **playbook** is a YAML file defining a list of plays (tasks + hosts) to execute.

A2 — `ansible_facts` contains gathered system information. Access OS family via `{{ ansible_os_family }}`.

A3 — `command` runs without shell features (secure). `shell` runs with shell features (pipes, vars). `raw` runs without setup (no Python module), useful for initial bootstrap.

A4 — Idempotency means running a task multiple times yields the same result. `apt: state=present` is idempotent. `shell: rm -rf /` is not.

A5 —
```yaml
- name: Install htop
  package:
    name: htop
    state: present
  when: ansible_os_family == "Debian"
- name: Install htop and epel
  package:
    name:
      - epel-release
      - htop
    state: present
  when: ansible_os_family == "RedHat"
```

A6 — Highest priority: Extra vars (`-e`) > Command line > Play vars > Role vars > Group vars > Host vars > Inventory vars > Default.

A7 — Use `serial: 3` to limit batch size. Group nodes in inventory. Use `meta: wait_for` or `wait_for` module to ensure first 3 nodes are ready before proceeding.

A8 — `ansible-vault encrypt_string "secret_value" --name "my_secret"`. This encrypts a specific variable string within a vars file.

A9 — Inventory lists hosts. **Static** is a file (INI/YAML). **Dynamic** is a script/API that generates inventory dynamically (e.g., from cloud provider).

A10 — `ansible webservers -m shell -a "df -h"`.

A11 — A handler is a task triggered by `notify` when a task reports a change. It runs after all tasks in the play complete, unlike regular tasks which run immediately.

A12 — `ansible.cfg` configures Ansible behavior. Common settings: `inventory`, `remote_user`, `forks`.

A13 — `register` saves a task's output to a variable. Use it in subsequent tasks via `when: my_var.stdout == "success"` or `debug: var=my_var`.

A14 — `import_tasks` is static (evaluated at parse time). `include_tasks` is dynamic (evaluated at runtime). Use `include_tasks` when conditionals depend on runtime data.

A15 — Ansible Galaxy is a repository for roles/collections. Use `requirements.yml` to list them, then run `ansible-galaxy install -r requirements.yml`.

A16 — Connection plugins define how Ansible talks to hosts. `ssh` (remote Linux), `local` (control node), `network_cli` (network devices). Use `ssh` for servers, `network_cli` for routers/switches.

A17 — `delegate_to` runs a task on a specific host. `run_once` runs it once across the batch. Example: Gather facts on one remote host, then write a config file to `localhost`.

A18 — 1. Increase `forks` (risk: resource usage). 2. Enable `pipelining` (risk: SSH config). 3. Use `fact_caching` (risk: stale facts). 4. Use `mitogen` (risk: compatibility).

A19 — `{{ }}` for expressions, `{% %}` for control flow, `{# #}` for comments.

A20 — 1. Set `serial: 1` for batches. 2. `pre_tasks`: Deregister from LB. 3. `tasks`: Deploy app. 4. `post_tasks`: Register + `uri` health check. 5. `block/rescue`: If health check fails, rollback (revert LB, redeploy previous version).
