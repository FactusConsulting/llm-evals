AA1 — A monolith is a single deployable unit with tightly coupled components, while microservices are independently deployable units communicating over a network. Microservices introduce: 1) Network latency and failure points, 2) Distributed tracing and debugging complexity, 3) Data consistency challenges (no global ACID transactions).

AA2 — Synchronous communication blocks the caller until a response is received (e.g., HTTP/gRPC). Asynchronous communication sends a message without waiting for an immediate response (e.g., RabbitMQ/Kafka).

AA3 — An API Gateway manages API logic (auth, rate limiting, routing) for external clients. A reverse proxy routes traffic based on host/path/headers. A Gateway cannot fully replace a proxy if low-level TCP routing or SSL termination is needed at the network edge, though they are often combined.

AA4 — The Saga pattern manages distributed transactions via a sequence of local transactions. Choreography uses events to trigger steps (loose coupling, hard to debug). Orchestration uses a central coordinator (tighter control, easier error handling). Use Choreography for simple flows; Orchestration for complex/error-prone workflows.

AA5 — CQRS separates read and write data models. It is paired with Event Sourcing (state derived from event log) to solve read/write scaling mismatches and provide a complete audit trail, which traditional CRUD cannot easily achieve.

AA6 — The Circuit Breaker stops calling a failing service to prevent cascading failures. States: Closed (normal), Open (failing), Half-Open (testing recovery). It differs from retry (which keeps hammering a failing service) by failing fast. Use Retry for transient errors; Circuit Breaker for persistent failures.

AA7 — 1. Shared DB/Schema: Low cost, high security complexity, high noisy-neighbor risk. 2. Shared DB/Schema per tenant: Medium cost, medium complexity, medium risk. 3. Separate DB: High cost, low ops complexity, low noisy-neighbor risk.

AA8 — Inventory requires Strong Consistency. Architecture: Relational DB (Postgres) with row-level locking or Redis for hot keys. Patterns: Optimistic locking, distributed locks (Redlock), or Saga for updates to prevent overselling.

AA9 — A sidecar proxy (Envoy) runs alongside the service to handle traffic. Data plane = Envoy (traffic handling). Control plane = Istio (config management). mTLS solves identity verification and encryption at the network layer, removing the need for application code to manage certificates.

AA10 — The Strangler Fig pattern incrementally migrates functionality. Strategy: Deploy .NET 8 services, configure the API Gateway to route specific paths to new services, keep legacy .NET Framework for remaining paths, and gradually shift traffic until the legacy code is removed.

AA11 — 12-Factor App: 1. Codebase, 2. Dependencies, 3. Config, 4. Backing Services, 5. Build/Release/Run, 6. Processes, 7. Port Binding, 8. Concurrency, 9. Disposability, 10. Dev/Prod Parity, 11. Logs, 12. Admin Processes. They ensure cloud-native scalability, portability, and maintainability.

AA12 — Event-driven (Kafka): Loose coupling, high scalability, hard debugging, ordered messages, async failure handling. Request-driven (REST): Tight coupling, limited scalability, easy debugging, ordered requests, sync failure handling. Choose Event for decoupling/scaling; Request for real-time synchronization.

AA13 — A load balancer distributes traffic across servers. Layer 4 = IP/Port routing. Layer 7 = HTTP content routing. Use Case: SSL termination or routing `/api` to specific backend pools.

AA14 — A message queue stores messages for processing. Queue (RabbitMQ) = Tasks/workers (delete after read). Log (Kafka) = Streaming/replay (retain for time). Pick Queue for tasks; Log for analytics/streaming.

AA15 — A container isolates a process; a VM isolates an OS. Containers are popular for microservices due to lightweight resource usage, fast startup times, and consistent environments.

AA16 — Stateless services store no session data (e.g., Web Server). Stateful services store data (e.g., Database). Stateless services scale horizontally easily because any instance can handle any request.

AA17 — An ORM maps objects to database tables. Examples: Hibernate (Java), Entity Framework (.NET). Trade-offs: ORMs increase dev speed but reduce SQL control and can impact performance.

AA18 — The Bulkhead pattern isolates resources to prevent one failure from taking down the whole system. Implement via separate thread pools per service or separate Kubernetes namespaces/deployment groups.

AA19 — OT requires server authority (complex, low latency online). CRDTs resolve conflicts mathematically (eventual consistency, high latency). CRDTs are better for offline editing as they don't require server coordination to merge changes.

AA20 — Tracing tracks request flow. Context propagation uses W3C Trace Context (traceparent header). Sampling: Head-based (start) vs Tail-based (end). Storage: Jaeger/Tempo. Kafka: Inject TraceID/SpanID into message headers to ensure continuity across async boundaries.

OP1 — Type 1 (Bare-metal): ESXi, Hyper-V. Type 2 (Hosted): VirtualBox, VMware Workstation. Type 1 runs directly on hardware; Type 2 runs on an OS.

OP2 — SAN = Block storage (iSCSI, Fibre Channel). NAS = File storage (NFS, SMB). Use SAN for databases; NAS for file shares.

OP3 — 3-tier = Core/Distribution/Access. Leaf-Spine = Low latency, no oversubscription. Solves bottlenecks at the distribution layer found in 3-tier architectures.

OP4 — Proxmox HA: 3+ nodes, Quorum (majority), Fencing (DRAC/iLO), Shared Storage (Ceph/iSCSI). Node failure: VMs restart on surviving nodes.

OP5 — VLAN Trunking = 802.1Q tagging. Linux config: `ip link add link eth0 name eth0.10 type vlan id 10`.

OP6 — RAID 10 = Best for DB (6 drives). Balances performance (striping) and fault tolerance (mirroring).

OP7 — DNS: Internal AD (Authoritative), External Public (Authoritative), Recursive Resolvers (Internal), Forwarders (External), Conditional Forwarding (Internal domains). Laptop: DHCP updates DNS, resolves internal via AD, external via forwarder.

OP8 — PKI: Root CA (Offline) -> Subordinate CA (Issuing) -> End Entity. CRL/OCSP for revocation. Templates/Auto-enrollment for management. Compromised Sub: Revoke. Compromised Root: Trust chain broken globally.

OP9 — MPLS (High cost, High reliability), IPsec (Low cost, Low reliability), SD-WAN (Balanced), WireGuard (Simple). Recommendation: SD-WAN for VoIP (latency/reliability balance).

OP10 — iDRAC/iLO/IPMI = Out-of-band management. Critical for power control/console access. Exposed to internet = Remote root access risk.

OP11 — LVM = Logical volumes. LVM-Thin = Overprovisioning. ZFS = Copy-on-write, Snapshots. Snapshots: LVM (point-in-time), ZFS (instant, COW).

OP12 — Stretch Cluster: 2 sites + Witness. Ceph CRUSH rules (placement), Monitors (quorum), Latency (<5ms). Split-brain: No consensus. 2-site bad: Latency causes split-brain without witness.

OP13 — DHCP = DORA (Discover, Offer, Request, Ack). Unavailable + Expire = APIPA (169.254.x.x).

OP14 — Managed = VLANs, QoS, ACLs, SNMP. Unmanaged = Plug & Play. Managed is critical for enterprise segmentation and traffic control.

OP15 — NTP = Time sync. Drift = Certs fail, logs broken, cluster split-brain.

OP16 — Physical = Bare metal. VM = Virtualized OS. Container = Isolated process. Bare metal for high performance/low latency (DB, HPC).

OP17 — UPS = Short term power. Generator = Long term power. ATS = Switches between them.

OP18 — Active-Passive = One active. Active-Active = Both active. Pacemaker/Corosync: Fencing (STONITH), VIP, Failover.

OP19 — 3-2-1 (3 copies, 2 media, 1 offsite). RPO/RTO. Types (Full/Inc/Diff). Dedup. Media (Tape/Disk/Cloud). Immutable (Ransomware). Test (Restore). Backup = Copy. DR = Recovery Plan.

OP20 — 802.1X: Supplicant (Client), Authenticator (Switch), Server (RADIUS). EAP-TLS (Cert), PEAP (User). VLAN: Dynamic. Fail: Quarantine. MAB: Switch sends MAC to RADIUS.
