T1 — Easy: `tofu plan` calculates the execution plan and displays the changes that would be made without applying them. `tofu apply` executes the plan, creating, updating, or destroying resources to match the configuration.

T2 — Easy: State is a JSON file tracking the mapping between OpenTofu resources and real-world infrastructure. Remote state is important for collaboration (sharing state among team members), enabling state locking to prevent concurrent writes, and storing state securely outside local machines.

T3 — Medium: `count` creates resources based on an integer index, while `for_each` creates resources based on a map or set of keys. `for_each` is preferred when adding or removing items, as it avoids resource recreation due to index shifting (e.g., removing index 0 shifts 1 to 0 in `count`, but `for_each` keeps keys stable).

T4 — Medium: This is called "drift." OpenTofu detects it during `tofu plan` by comparing the state file against the live infrastructure. If the config matches the state, OpenTofu will propose changes to revert the manual modification to match the code.

T5 — Medium:
```hcl
resource "aws_instance" "web" {
  for_each = toset(["web1", "web2", "web3"])
  ami      = "ami-12345"
  instance_type = "t2.micro"
  tags = {
    Name = each.key
  }
}
```

T6 — Hard: `terraform_remote_state` reads state from a backend, requiring credentials and exposing state data. The recommended alternative is **Module Composition**: pass outputs directly from one module to another via `module` blocks. This is preferred because it is type-safe, avoids credential management for state access, and keeps dependencies explicit within the configuration.

T7 — Hard: The `null_resource` provisioner runs asynchronously during apply. The `kubeconfig` output likely reads a file that isn't guaranteed to exist until the provisioner finishes. Fix: Add `depends_on = [null_resource.cluster_setup]` to the resource consuming the output, or use a `local-exec` provisioner that waits for the cluster to be ready before writing the config file.

T8 — Medium: `terraform import` is a CLI command to import existing resources into state manually. The `import` block (1.5+) allows importing resources declaratively within the configuration file. It improves on the CLI by making imports part of the version-controlled plan, allowing `tofu plan` to show the import as a change, and enabling better CI/CD integration.

T9 — Easy: A `variable` is an input parameter passed into a module or root module. An `output` is a value exposed from a module or root module to be used by other modules or the CLI. Variables are inputs; outputs are outputs.

T10 — Easy: A provider is a plugin that interacts with a specific API (e.g., AWS, Azure). If you run `tofu plan` without `tofu init`, OpenTofu will fail because it hasn't downloaded the required provider plugins and initialized the backend.

T11 — Easy: The `.terraform.lock.hcl` file records the exact versions and checksums of providers used. It should be committed to version control to ensure reproducible builds and prevent accidental upgrades that might break infrastructure.

T12 — Easy: `tofu destroy` removes all resources managed by the configuration. To target a single resource: `tofu destroy -target=aws_instance.example`.

T13 — Easy: A data source reads information from an existing external system (e.g., cloud provider, API) without creating or managing it. Use one when you need to reference existing resources (e.g., `data "aws_ami" "ubuntu"` to find the latest AMI ID) rather than creating new ones.

T14 — Medium: `variable` blocks define inputs for the module. `locals` define values computed within the module that are not exposed externally. Use `locals` for intermediate calculations or constants that don't need to be overridden by users.

T15 — Medium: The `lifecycle` block controls resource behavior. `create_before_destroy` creates the new resource before destroying the old (zero-downtime). `prevent_destroy` stops accidental deletion. `ignore_changes` prevents OpenTofu from tracking specific attribute drift (e.g., tags managed manually).

T16 — Hard: The `moved` block tracks resource identity changes during refactoring (e.g., moving a resource from one module to another). It updates the state file to map the old address to the new address without destroying and recreating the resource.

T17 — Hard: OpenTofu builds a Directed Acyclic Graph (DAG) of dependencies. Implicit dependencies are inferred from references (e.g., `aws_instance.web` in `aws_instance.db`). Explicit dependencies use `depends_on`. `depends_on` reduces parallelism because it forces serialization, which can slow down apply times if overused.

T18 — Hard:
1. **Increase Parallelism:** Set `parallelism` in config. Tradeoff: Higher API load on cloud provider.
2. **State Optimization:** Exclude resources from state or use `terraform plan -refresh=false`. Tradeoff: Risk of stale state or missing drift detection.
3. **Module Isolation:** Split large configs into smaller modules. Tradeoff: Increased management overhead.
4. **Backend Caching:** Use a backend that supports state locking and caching. Tradeoff: Cost/complexity of backend setup.

T19 — Hard: State locking prevents concurrent modifications. If a lock isn't released (e.g., crash), subsequent commands fail. Manually break it with `tofu force-unlock <LOCK_ID>`. Risk: Data corruption if another process is actually writing to the state simultaneously.

T20 — Hard:
1. Backup existing state file.
2. Add `backend` block with S3/DynamoDB config to `main.tf`.
3. Run `tofu init -migrate-state` to move state to remote backend.
4. Verify with `tofu state list` and ensure locking works via `tofu plan`.

A1 — Easy: A **Playbook** is a YAML file defining the automation steps. A **Role** is a structured directory layout for reusable logic. A **Task** is a single action (module execution) within a playbook or role.

A2 — Easy: `ansible_facts` is a dictionary of system information gathered by the `setup` module. Access OS family via `{{ ansible_facts['os_family'] }}`.

A3 — Medium: `command` runs a command without shell features (no pipes). `shell` runs a command through a shell (supports pipes, variables). `raw` runs without setup (no Python module transfer), useful for initial bootstrap. Use `raw` for bootstrapping SSH, `shell` for complex logic, `command` for simple binaries.

A4 — Medium: Idempotency means running a task multiple times produces the same result. Example (Idempotent): `apt: name=nginx` (installs only if missing). Example (Not Idempotent): `shell: echo "Hello" > /tmp/test` (overwrites file every time).

A5 — Medium:
```yaml
- name: Install htop
  package:
    name: htop
  become: yes
  when: ansible_os_family == "Debian"

- name: Install htop and epel on RedHat
  package:
    name:
      - epel-release
      - htop
  become: yes
  when: ansible_os_family == "RedHat"
```

A6 — Hard: Highest priority: Extra vars (`-e`) > Command line > Environment vars > Play vars > Host vars > Group vars > Role vars > Role defaults. Lowest priority: `set_fact` (unless `persist` is used).

A7 — Hard: Use `serial: 3` to limit concurrency. Group nodes into `agents` and `workers`. Use `meta: wait_for` or `wait_for` module to ensure the first 3 nodes are ready before the rest start.

A8 — Medium: `ansible-vault` encrypts sensitive data. To encrypt a single variable: `ansible-vault encrypt_string 'secret_value' --name 'my_secret' --vars-file vars.yml`.

A9 — Easy: An inventory file lists hosts and groups. **Static** inventory is a fixed file (INI/YAML). **Dynamic** inventory is a script or API call that generates the list dynamically (e.g., from AWS EC2).

A10 — Easy: `ansible webservers -m shell -a "df -h"`.

A11 — Easy: A handler is a task that runs only when notified by another task (e.g., `notify: restart service`). It differs from a regular task because it is deferred until the end of the play. Triggered via `notify: handler_name`.

A12 — Easy: `ansible.cfg` configures Ansible behavior. Common settings: `host_key_checking` (disable), `forks` (parallelism), `remote_user` (default SSH user).

A13 — Easy: `register` saves the result of a task into a variable. Use it in subsequent tasks via `when: my_var.stdout == 'success'` or `debug: var=my_var`.

A14 — Medium: `import_tasks` is processed at compile time (static). `include_tasks` is processed at runtime (dynamic). Use `import` for fixed logic, `include` for conditional logic. `include` allows `when` to skip the file entirely; `import` always loads the file.

A15 — Medium: Ansible Galaxy is a repository for roles and collections. Use `requirements.yml` to list dependencies. Run `ansible-galaxy install -r requirements.yml` to install them.

A16 — Medium: Connection plugins manage how Ansible connects to hosts. `ssh` (standard remote), `local` (control node), `network_cli` (network devices like Cisco/Juniper). Use `ssh` for Linux/Windows, `network_cli` for routers/switches.

A17 — Hard: `delegate_to: localhost` runs a task on the control node. `run_once: true` ensures it runs only once even in a loop. Example: Gather facts from remote hosts, then process them locally to generate a config file.

A18 — Hard:
1. **Forks:** Increase `forks` in config. Tradeoff: More system resources on control node.
2. **Pipelining:** Enable `pipelining` in `ansible.cfg`. Tradeoff: Requires SSH support, reduces latency.
3. **Fact Caching:** Cache facts (e.g., `jsonfile`). Tradeoff: Facts might be stale.
4. **Mitogen:** Use Mitogen plugin. Tradeoff: Additional dependency, compatibility checks.

A19 — Hard: `{{ }}` outputs variable values. `{% %}` executes logic (loops, if). `{# #}` comments.
Snippet:
```jinja2
upstream backend {
{% for server in backend_servers %}
    server {{ server.ip }} weight={{ server.weight }}; # Health check: {{ server.health }}
{% endfor %}
}
```

A20 — Hard:
1. Set `serial: 1` for one server at a time.
2. `pre_tasks`: Deregister from load balancer (e.g., `aws_elb_deregister`).
3. `tasks`: Deploy app.
4. `post_tasks`: Register to LB, run `uri` health check.
5. `block/rescue`: If health check fails, `rescue` triggers rollback (revert code, re-register).
