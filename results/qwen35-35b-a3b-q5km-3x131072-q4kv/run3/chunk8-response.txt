CL1 — **IaaS (Infrastructure as a Service):** Rent computing resources (VMs, storage, networks). *Example: AWS EC2.* **PaaS (Platform as a Service):** Rent hardware + OS + middleware. *Example: Google App Engine.* **SaaS (Software as a Service):** Rent fully managed software. *Example: Microsoft 365.*

CL2 — **Shared Responsibility:** Provider secures the cloud (hardware, physical); Customer secures what they put in the cloud. **IaaS VM:** Provider secures hardware/physical; Customer secures OS, network config, data, app. **Managed Kubernetes:** Provider secures control plane/infra; Customer secures node OS, pod config, network policies, data. **SaaS (M365):** Provider secures app/infra; Customer secures identity, data classification, access policies.

CL3 — **VPC:** Isolated virtual network. **Subnet:** IP range within VPC. **Security Group:** Stateful firewall at instance level (allow rules). **NACL:** Stateless firewall at subnet level (allow/deny rules). **Layering:** VPC contains Subnets; Subnets contain Instances; SG protects Instances; NACL protects Subnets. **Stateful:** Tracks connection state (return traffic allowed). **Stateless:** Treats every packet independently (must allow return traffic explicitly).

CL4 — **Landing Zone:** Secure, compliant baseline environment. **OUs:** Logical grouping for billing/policy. **Guardrails:** Preventative controls (e.g., no public S3 buckets). **Shared Networking:** Transit Gateway (AWS) or Hub-and-Spoke (Azure) for centralized connectivity. **Centralized Logging:** Aggregated logs for security monitoring. **Single-Account Problems:** High blast radius, billing complexity, lack of isolation, policy conflicts, no granular access control.

CL5 — **IaC:** Managing infrastructure via code. **Terraform/OpenTofu:** Multi-cloud, declarative, state file management. **Pulumi:** Multi-cloud, imperative (use general languages like Python/TS). **CloudFormation/Bicep:** Native to AWS/Azure, tightly integrated, less portable. **Crossplane:** IaC via Kubernetes API. **Choice:** Terraform for multi-cloud portability; CloudFormation/Bicep for deep native integration; Pulumi for dev-heavy teams; Crossplane for K8s-centric ops.

CL6 — **HPA:** Scales pods based on CPU/Memory. **VPA:** Scales pod resource requests/limits (requires restart). **Cluster Autoscaler:** Adds/removes nodes based on pod scheduling. **KEDA:** Scales based on external events (Kafka, Queue). **Use KEDA over HPA:** When scaling is driven by event queues (e.g., message backlog) rather than resource utilization.

CL7 — **DR Strategy:** Define RPO (data loss tolerance) and RTO (downtime tolerance). **Active-Active:** Both regions serve traffic (low RTO/RPO, high cost). **Active-Passive:** One region active, one standby (higher RTO, lower cost). **DB:** Sync (strong consistency, higher latency) vs Async (replication lag). **DNS:** Route53 Failover routing. **State:** Shared storage or replicated stateful sets. **Cost:** Active-Active doubles infra cost. **Pilot Light:** Core infra running, app deployed on demand. **Warm Standby:** App running scaled down. **Active-Active:** Full production in both regions.

CL8 — **GitOps:** Git as single source of truth for infra/app state. **ArgoCD:** Push-based (agent watches Git). **FluxCD:** Pull-based (agent pulls Git). **Secrets:** Sealed Secrets (encrypted in Git), External Secrets Operator (fetch from Vault/AWS Secrets Manager). **Drift:** GitOps controller detects manual changes and reverts them to match Git state.

CL9 — **Architecture:** Use **StatefulSet** for stable network IDs. **Local PV:** Use `local` storage class with `volumeBindingMode: WaitForFirstConsumer` for low latency. **Headless Service:** For stable DNS discovery. **Init Containers:** For config file generation before main app starts. **ConfigMaps:** Mount as volumes for specific paths. **PDB:** Ensure minimum available pods during updates to prevent data loss.

CL10 — **Service Account:** K8s identity for pods. **Workload Identity:** Maps K8s SA to Cloud IAM role. **IRSA:** AWS specific (OIDC federation). **Mounting Creds:** Bad practice (leaks secrets, hard to rotate, violates least privilege). **Federation:** Uses OIDC tokens to request temporary cloud credentials without long-lived secrets.

CL11 — **Object Storage (S3):** Unstructured data, high durability (11 9s), low latency. **Block Storage (EBS):** Attached to VM, low latency, persistent. **File Storage (EFS):** Shared file system, NFS/SMB, scalable. **Use Object:** Backups, media. **Use Block:** OS disk, DB. **Use File:** Shared configs, home dirs. **Durability:** Object (11 9s), Block (99.999%), File (99.999%).

CL12 — **Zero Trust:** Verify every request. **Network Policies:** Restrict pod-to-pod traffic. **Service Mesh (mTLS):** Encrypt service-to-service. **Pod Identity:** SPIFFE/SPIRE. **Admission (OPA/Kyverno):** Enforce policies on deploy. **Image Signing:** Cosign/Notary. **Runtime (Falco):** Detect anomalies. **Difference:** Traditional uses perimeter (firewall); Zero Trust assumes breach and verifies identity/context for every interaction.

CL13 — **CDN:** Distributed network of edge servers caching content. **How:** Caches static assets closer to users. **Services:** AWS CloudFront, Cloudflare. **Use CDN:** For static content, global latency reduction, DDoS protection. **Use Origin:** For dynamic content, real-time data, or when caching is not beneficial.

CL14 — **Region:** Geographic area (e.g., us-east-1). **AZ:** Isolated data centers within a region. **Multi-AZ:** Protects against data center failure. **Zonal Resources:** Tied to specific AZ (e.g., EBS). **Regional Resources:** Span multiple AZs (e.g., ELB, RDS Multi-AZ).

CL15 — **Managed DB:** RDS, Cloud SQL. **Provider Handles:** Patching, backups, failover, hardware, OS updates. **You Handle:** Schema, queries, user access, data. **Benefit:** Reduced operational overhead, higher availability SLAs.

CL16 — **IAM Users:** Human identities. **Groups:** Collections of users. **Roles:** Temporary credentials for services/users. **Policies:** JSON permissions. **Least Privilege:** Grant only necessary permissions. **Apply:** Use roles instead of users, scope policies to specific resources, use MFA.

CL17 — **Serverless:** Run code without managing servers. **Compare:** Lambda (AWS), Functions (Azure), Cloud Functions (GCP). **Cold Starts:** Latency when function is invoked after inactivity (initialization). **Poor Fit:** Long-running processes, high CPU usage, stateful apps, strict latency requirements.

CL18 — **Ingress:** K8s resource for HTTP/HTTPS routing. **Gateway API:** Standardized, multi-protocol, advanced routing. **Cloud LB:** Cloud provider specific annotations. **Mesh:** Istio/Linkerd ingress gateways. **TLS:** Terminates at LB (Cloud), Ingress Controller, or Mesh. **Use Gateway API:** For complex routing, multi-cluster, or when standard Ingress is insufficient.

CL19 — **Strategy:** **RI/SP:** Commit to usage for discount. **Spot:** Interruptible for fault-tolerant workloads. **Right-sizing:** Match instance to actual usage. **Tiering:** Move old data to Glacier/S3 IA. **Idle Detection:** Tag and remove unused resources. **FinOps:** Cross-team cost accountability. **Showback:** Report costs to teams. **Chargeback:** Bill teams directly. **Tools:** AWS Cost Explorer, Kubecost, CloudHealth.

CL20 — **Strategy:** **Abstraction:** Kubernetes (K8s) for runtime, Terraform for IaC. **Networking:** Direct Connect/ExpressRoute for low latency. **Identity:** OIDC federation across clouds. **Trade-offs:** Increased complexity, higher networking costs, harder security management. **Justified:** Regulatory compliance, vendor risk mitigation, disaster recovery. **Unnecessary:** If single cloud meets all needs, multi-cloud adds cost without value.

OT1 — **Purdue Model:** ISA-95 hierarchy. **Level 0:** Physical Process (Sensors/Valves). **Level 1:** Basic Control (PLCs/RTUs). **Level 2:** Supervisory Control (SCADA/HMI). **Level 3:** Site Operations (Manufacturing Execution). **Level 4:** Enterprise (ERP/Business).

OT2 — **IT vs OT:** IT focuses on Confidentiality/Integrity; OT focuses on Availability/Safety. **Patching:** IT patches regularly; OT systems often run legacy OS, require stability, and downtime can cause physical damage or production loss.

OT3 — **DMZ (IDMZ):** Buffer zone between IT and OT. **Traffic:** IT pulls data from OT (e.g., historian sync); OT pushes data to IT. **Direct L4->L1:** Dangerous because IT vulnerabilities can directly compromise control logic, risking physical safety.

OT4 — **Protocols:** 1. **Modbus TCP:** L1/L2, simple polling. 2. **OPC UA:** L2/L3, secure, platform-independent. 3. **EtherNet/IP:** L1/L2, Allen-Bradley dominant. 4. **PROFINET:** L1/L2, real-time industrial Ethernet. 5. **MQTT:** L3/L4, lightweight pub/sub for cloud.

OT5 — **Data Diode:** Hardware enforcing one-way data flow. **Firewall:** Two-way traffic control. **Use Diode:** When data must leave OT but nothing can enter (e.g., high security export). **Limitation:** Cannot receive commands or updates from outside.

OT6 — **Flow:** PLCs -> SCADA Server -> Historian -> Edge Gateway -> MQTT Broker -> Cloud. **Security:** IDMZ between SCADA and Edge. Edge Gateway authenticates to Cloud. Firewall rules restrict OT->IT to specific ports/protocols.

OT7 — **Zones:** Groups of assets with similar security requirements. **Conduits:** Controlled paths between zones. **Design:** Zone 1 (Corporate), Zone 2 (Engineering), Zone 3 (SCADA), Zone 4 (PLC/SIS). Conduits between zones require firewalls. Engineering workstations access SCADA via secure conduit.

OT8 — **Segmentation:** **VLANs:** Separate lines/areas. **Firewall:** Between VLANs and IDMZ. **Switch:** Managed Layer 2/3 switches. **Remote Access:** Jump host in IDMZ with MFA. **Transition:** Phase rollout during maintenance windows, test in parallel, monitor traffic closely.

OT9 — **SIS vs BPCS:** SIS (Safety Instrumented System) stops process safely; BPCS (Basic Process Control) manages normal operation. **IEC 61511:** Safety lifecycle standard. **Isolation:** SIS must be air-gapped to prevent cyber attacks from disabling safety functions. **Ransomware on SIS:** Can disable safety interlocks, leading to catastrophic physical failure.

OT10 — **PLC:** Logic controller for machines. **DCS:** Distributed control for continuous processes. **SCADA:** Supervisory monitoring over wide area. **Coexist:** Yes, PLCs often feed into SCADA/DCS systems.

OT11 — **Architecture:** Jump host in DMZ. **MFA:** Required for all access. **Recording:** Session recording for audit. **Vendor:** Time-limited accounts. **Direct RDP:** Unacceptable because it bypasses DMZ controls and exposes OT assets directly to corporate network threats.

OT12 — **Plan:** **Monitor:** IDS for anomalous traffic. **Inventory:** Identify compromised assets. **IR:** Isolate network segments, do not reboot (preserve evidence). **Ops:** Switch to manual control if needed. **Operator:** Prioritize physical safety over data recovery. **Coordination:** IT/OT joint command center.

OT13 — **HMI:** Operator interface for monitoring/control. **Role:** Visualize process, allow manual overrides. **Difference:** Engineering Workstation is for configuration/programming; HMI is for real-time operation.

OT14 — **Historian:** Time-series database for process data. **Importance:** Trend analysis, compliance, troubleshooting. **Plant Historian:** Real-time, high frequency, local. **Enterprise Historian:** Aggregated, lower frequency, for business analytics.

OT15 — **Ladder Logic:** Graphical programming language resembling relay logic. **Device:** PLC. **Dominance:** Standardized (IEC 61131-3), safety certified, widely understood by electricians/engineers.

OT16 — **RTU:** Remote Terminal Unit. **Difference:** RTUs are rugged, designed for long-distance comms (cellular/RF), often fewer I/O. **PLC:** Local, high-speed I/O. **RTU Use:** Pipelines, water distribution, remote sites.

OT17 — **Analog:** Continuous voltage/current. **Digital:** Discrete on/off. **4-20mA:** Preferred over 0-20mA because 0mA indicates a broken wire (fault), whereas 4mA indicates minimum process value (live zero).

OT18 — **Inventory:** Knowing all assets is critical for patching/vulnerability management. **Tools:** Passive (packet sniffing), Active (scanning). **Active Scanning:** Dangerous because it can crash legacy devices or trigger alarms in safety systems.

OT19 — **MITRE ATT&CK ICS:** Framework for ICS threats. **Tactics:** Reconnaissance, Initial Access, Execution, Persistence, Impact. **Example:** "Exploit Public-Facing Application" (Initial Access). **Difference:** Focuses on physical impact and safety, not just data theft. **Use:** Map current defenses to tactics to find gaps.

OT20 — **SOC:** **Logs:** Collect via passive taps or lightweight agents. **SIEM:** Correlate IT/OT logs. **Rules:** OT-specific (e.g., PLC write attempts). **Triage:** OT engineer validates before IT acts. **Staffing:** Mix of IT security and OT engineers. **False Positives:** Tune rules to avoid production stoppages. **Firmware Change:** Alert -> Isolate -> Verify integrity -> Re-image if compromised.
