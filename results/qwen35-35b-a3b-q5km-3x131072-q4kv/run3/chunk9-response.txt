SC1-A
The pods are failing because the **Restricted Pod Security Standard (PSS)** forbids containers running as root unless explicitly allowed, and it requires `runAsNonRoot: true` to be paired with an explicit `runAsUser` > 0.
1.  **Baseline vs. Restricted:** Baseline allows most privileged settings. Restricted forbids `runAsNonRoot: false`, `privileged: true`, and requires `runAsUser` to be set if `runAsNonRoot` is true.
2.  **Image Behavior:** The new base image (`cgr.dev/chainguard/static:latest`) defaults `USER` to root (UID 0).
3.  **The Conflict:** The manifest sets `securityContext.runAsNonRoot: true` but omits `runAsUser`. Kubernetes defaults `runAsUser` to 0 (root) if not specified. Restricted PSS rejects this combination (Non-root required, but running as root).
4.  **Result:** The kubelet rejects the pod creation or the container runtime kills the process immediately, causing CrashLoopBackOff.

kubectl get pods --all-namespaces -o json | jq -r '.items[] | select(.status.phase=="Running" and .status.containerStatuses[]?.state.waiting.reason=="CrashLoopBackOff") | "\(.metadata.namespace) \(.metadata.name) \(.spec.securityContext | tojson)"'

SC1-C
**Remediation Strategy:**
1.  **Image Fix:** Rebuild the base image with a non-root user (e.g., `USER 1000`) or use a pre-built non-root Chainguard image variant.
2.  **Manifest Fix:** Add `securityContext.runAsUser: 1000` to the pod spec.
3.  **CI Gate:** Integrate a PSS scanner (e.g., `kube-score` or `checkov`) into the CI pipeline. Fail the build if `runAsNonRoot` is true without `runAsUser`.
4.  **Rollout:** Use a **Canary Deployment**. Deploy the fixed version to 5% of traffic, monitor for stability, then gradually increase to 100%. This ensures immediate rollback capability if the fix introduces new issues.

SC2-A
**Stale Data Issue:**
1.  **Replication Lag:** Cross-region asynchronous replication introduces latency. Promoting a replica before WAL shipping completes causes data loss (uncommitted transactions on primary not shipped).
2.  **WAL Shipping:** PostgreSQL ships Write-Ahead Logs (WAL) asynchronously. If the primary crashes before shipping, those transactions are lost.
3.  **Quantification:** Check `pg_stat_replication` on the new primary (formerly replica) for `pg_wal_lsn_diff` (write_lsn vs replay_lsn). The difference indicates the lag in bytes.
4.  **Data Loss:** Any transactions committed on the old primary between the last successful WAL send and the failover are lost.

SC2-B
1.  **Import New Instance:** `terraform import aws_db_instance.new_primary <new_instance_arn>`
2.  **Remove Old Instance:** `terraform state rm aws_db_instance.old_primary`
3.  **Update Dependencies:** Run `terraform plan` to ensure dependent resources (SGs, Route53) reference the new ARN. If they reference the old ARN hardcoded in variables, update the variable file and run `terraform apply`.
4.  **Verify:** `terraform state list` to confirm only the new instance exists.

SC2-C
**Architecture Design:**
1.  **RDS Proxy:** Use RDS Proxy instead of PgBouncer to handle connection pooling and failover transparently at the application layer.
2.  **Route53:** Use Latency-Based Routing with Health Checks. Point the primary endpoint to the US region; if unhealthy, Route53 fails over to EU.
3.  **Retry Logic:** Implement exponential backoff with jitter in the application connection pool.
4.  **RPO < 1s:** Use **Aurora Global Database**. It uses a single storage volume replicated across regions with sub-second replication latency, unlike standard RDS async replication.
5.  **Recommendation:** Yes, Aurora Global DB is recommended over standard RDS for this requirement due to lower RPO and faster failover times.

SC3-A
**Incident Response:**
1.  **Containment:** Cordon the affected nodes (`kubectl cordon <node>`) and drain pods (`kubectl drain <node> --ignore-daemonsets --delete-emptydir-data`) to stop new pods from scheduling, but **do not delete** the compromised pods immediately.
2.  **Forensics:**
    *   **Process Tree:** `kubectl exec -it <pod> -- ps auxf`
    *   **Network:** `kubectl exec -it <pod> -- ss -tulpn` or `netstat -tulpn`
    *   **Filesystem:** `kubectl exec -it <pod> -- tar czf /tmp/forensics.tar.gz /tmp/.x /var/log`
    *   **Diff:** Mount the container filesystem to a host or use `docker diff` equivalent logic if possible, or snapshot the EBS volume.

SC3-C
**Hardened Pipeline:**
1.  **Signing:** Use **Cosign** to sign images upon build. `cosign sign --key cosign.key <image>`.
2.  **Admission Controller:** Deploy **Kyverno** or **OPA Gatekeeper** to enforce signature verification on pod admission. Reject unsigned images.
3.  **Pinning:** Pin base images by SHA256 digest (e.g., `node:20-slim@sha256:...`) rather than tags to prevent tag swapping.
4.  **SBOM:** Generate Software Bill of Materials (SBOM) using **Syft** and scan with **Grype** in CI.
5.  **Notary v2:** Notary v2 (TUF) provides a decentralized trust model. It can be integrated to verify image signatures before pulling, acting as a secondary check to Cosign.

SC4-A
**Top 5 Blockers:**
1.  **Local Sessions:** Kubernetes pods are ephemeral. **Solution:** Move sessions to Redis.
2.  **Local NFS:** Files are lost on pod restart. **Solution:** Use Persistent Volumes (EFS/NFS CSI) with `ReadWriteMany`.
3.  **Sticky Sessions:** F5 relies on IP affinity. **Solution:** Use Kubernetes Ingress with session affinity or move session state to Redis.
4.  **Oracle DB:** On-prem DB access. **Solution:** Use VPC Peering or Direct Connect to access on-prem DB, or migrate DB to cloud.
5.  **IBM MQ:** Stateful messaging. **Solution:** Use StatefulSets for MQ or migrate to cloud-native messaging (SQS/RabbitMQ).

SC5-C
**Architectural Solution:**
1.  **CQRS:** Separate read (inventory view) from write (reconciliation).
2.  **Read Replica:** Run the reconciliation job against a read replica to avoid locking production tables.
3.  **Bulkheads:** Configure separate connection pools for the reconciliation job vs. serving pods.
4.  **Scheduling:** Move CronJob to off-peak hours or use `concurrencyPolicy: Forbid`.
5.  **PDBs:** Set `minAvailable` for serving pods so the CronJob cannot evict them during resource contention.

SC6-C
**Connectivity Upgrade:**
1.  **AWS Direct Connect:** High bandwidth, low latency, dedicated. Best for performance. Costly. Fits Zero Trust (private link).
2.  **VPN ECMP:** Cheaper, easier. Limited bandwidth (1Gbps x 2). Encryption is IPsec.
3.  **Mesh (Tailscale/Netbird):** Zero-config, identity-based. Good for dev/staging. Not ideal for core production due to control plane dependency.
**Routing:** DX requires BGP peering. VPN requires static routes. Mesh requires agent routing.
**Recommendation:** Direct Connect for production core, VPN for backup.

SC7-A
**Drift Sources:**
1.  **Manual `tofu apply`:** Bypasses CI/CD. Dangerous because state locking is missing, leading to race conditions and potential resource destruction.
