K1 — Easy: A **Deployment** manages stateless applications, creating pods with random names and allowing rolling updates. A **StatefulSet** manages stateful applications, providing stable, unique network IDs (hostnames), ordered deployment/termination, and persistent storage retention.

K2 — Easy: A **ClusterIP** Service creates a virtual IP (VIP) within the cluster. **kube-proxy** (using iptables or IPVS) configures the node's networking rules to route traffic destined for the VIP to the backend Pod IPs based on the Service's selector.

K3 — Medium: A **PersistentVolume (PV)** is a cluster-level storage resource. A **PersistentVolumeClaim (PVC)** is a user request for storage. A **StorageClass** defines the provisioning policy (dynamic vs. static). A PVC binds to a PV; if no PV exists, the StorageClass triggers dynamic provisioning of a new PV.

K4 — Medium: 1. `kubectl logs <pod>` to check application errors. 2. `kubectl describe pod <pod>` to check events (e.g., OOMKilled, probe failures). 3. Check liveness/readiness probe configuration. 4. Verify resource limits (CPU/Memory) to ensure the container isn't being killed by the node.

K5 — Medium: **CoreDNS** provides DNS resolution for services within the cluster. If CoreDNS pods are down, internal service discovery (e.g., `my-service.default.svc`) fails, breaking inter-pod communication, though external DNS resolution may still work depending on the node's `/etc/resolv.conf`.

K6 — Hard: The scheduler uses **Filtering** (bin-packing, removing nodes that don't meet requirements) and **Scoring** (ranking nodes based on preferences). Factors considered: 1. CPU/Memory requests vs. available capacity. 2. Node Affinity/Taints. 3. Pod Affinity/Anti-Affinity. 4. Topology spread constraints.

K7 — Hard: Yes, it will be scheduled (3.5GB + 256Mi < 4GB). If it uses 600Mi at runtime, it exceeds the `limits.memory: 512Mi`. The OOM Killer will terminate the container, causing a CrashLoopBackOff.

K8 — Hard: 1. External LB forwards traffic to Ingress Controller IP. 2. NGINX Ingress Controller matches Host/Path rules. 3. Controller forwards traffic to the Service ClusterIP. 4. kube-proxy (iptables/IPVS) maps Service IP to Pod IPs. 5. Traffic reaches the specific Application Pod.

K9 — Medium: A **NetworkPolicy** restricts traffic flow between pods. Default behavior (no policy): All traffic is allowed. When a policy is applied: Traffic is denied by default unless explicitly allowed by Ingress/Egress rules in the policy.

K10 — Medium: **RollingUpdate** replaces pods gradually (zero downtime). **Recreate** kills all existing pods before starting new ones. Choose **Recreate** for stateful applications where downtime is acceptable or for database migrations requiring a full restart.

K11 — Easy: A **Namespace** provides logical isolation for resources. Default namespaces: `default`, `kube-system`, `kube-public`, `kube-node-lease`.

K12 — Easy: `kubectl describe` shows detailed status and events. 1. **Events** (e.g., scheduling failures, OOMKilled). 2. **Conditions** (e.g., Ready status). 3. **Mounts** (Volumes, ConfigMaps). `kubectl get` only shows basic status/age.

K13 — Easy: A **ConfigMap** stores non-confidential configuration data. Pods consume it via: 1. **Environment Variables** (in container spec). 2. **Volume Mounts** (files inside the container).

K14 — Easy: **ClusterIP** (internal cluster access), **NodePort** (exposes service on a static port on each node's IP), **LoadBalancer** (provisions an external cloud load balancer).

K15 — Easy: A **DaemonSet** ensures a copy of a pod runs on all (or specific) nodes. Use it for node-level agents like logging (Fluentd), monitoring (Prometheus Node Exporter), or storage daemons.

K16 — Medium: **Taints** repel pods (node attribute); **Tolerations** allow pods to schedule on tainted nodes. **Node Affinity** pulls pods to nodes. Use Taints for dedicated hardware (e.g., GPU nodes); use Affinity for co-location (e.g., app + cache).

K17 — Medium: An **Operator** is a custom controller that manages complex stateful applications using CRDs. It differs from a standard controller by automating lifecycle management (backup, scaling, upgrades) specific to the application, solving operational complexity.

K18 — Hard: etcd uses **Raft consensus** (strong consistency). If quorum is lost, the cluster becomes read-only/unavailable. Recovery: Restore a snapshot on a new node, then rejoin it to the cluster to restore quorum (requires at least one surviving node or a full restore from backup).

K19 — Hard: **Role** (namespace-scoped permissions), **ClusterRole** (cluster-wide), **RoleBinding** (links Role to Subject in namespace), **ClusterRoleBinding** (links ClusterRole to Subject cluster-wide).
```yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  name: pod-reader
  namespace: dev
rules:
- apiGroups: [""]
  resources: ["pods", "pods/log"]
  verbs: ["get", "list"]
---
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  name: read-pods
  namespace: dev
subjects:
- kind: User
  name: developer
roleRef:
  kind: Role
  name: pod-reader
  apiGroup: rbac.authorization.k8s.io
```

K20 — Hard: `ndots` (default 5) dictates how many dots trigger a search domain lookup. If a domain has fewer dots than `ndots`, the resolver tries search domains first, causing multiple queries. Fix: Set `dnsConfig.options.ndots: 1` in the pod spec to query FQDNs immediately.

D1 — Easy: **Rebase** rewrites history to create a linear commit path. **Merge** preserves history with a merge commit. Avoid rebasing on public/shared branches to prevent disrupting other developers.

D2 — Easy: **TCP** is connection-oriented and reliable (HTTP, SSH). **UDP** is connectionless and fast (DNS, VoIP).

D3 — Medium: A **race condition** occurs when output depends on the sequence of events. Example: Two threads incrementing a shared counter without a lock results in lost updates.

D4 — Medium: **Symmetric** uses one key for encryption/decryption. **Asymmetric** uses a public/private key pair. TLS uses Asymmetric for the handshake (key exchange) and Symmetric for data transfer (performance).

D5 — Medium: A **layer** is a diff of the filesystem in a Docker image. Layer caching speeds up builds. Best practice: Copy dependency files (e.g., `package.json`) before source code to cache dependency installation.

D6 — Medium: 1. Check application logs for errors. 2. Check upstream service health. 3. Verify resource limits (OOM). 4. Inspect load balancer health checks.

D7 — Medium: **CAP** states a distributed system can only guarantee 2 of 3: Consistency, Availability, Partition Tolerance. **CP** (Zookeeper, HBase) prioritizes consistency. **AP** (Cassandra, DynamoDB) prioritizes availability.

D8 — Medium: **Horizontal** adds nodes; **Vertical** adds resources (CPU/RAM). Patterns enabling horizontal scaling: Stateless services, sharding, load balancing, microservices.

D9 — Easy: An **environment variable** is a dynamic value passed to processes. To persist across sessions on Linux, export it in `~/.bashrc` or `~/.profile`.

D10 — Easy: **JSON** is strict key-value (machine-friendly). **YAML** is human-readable (indentation-based). Advantage JSON: Standardized parsing. Advantage YAML: Readable for config.

D11 — Easy: **REST** is an architectural style for APIs. **GET** (retrieve), **POST** (create), **PUT** (replace), **DELETE** (remove).

D12 — Easy: **Version pinning** locks dependencies to exact versions. Important for reproducible builds to prevent breakage from upstream updates.

D13 — Easy: **Compiled** (source to binary before run: C, Go). **Interpreted** (source executed line-by-line: Python, JavaScript).

D14 — Medium: A **reverse proxy** sits in front of servers (client -> proxy -> server). A **forward proxy** sits in front of clients (client -> proxy -> internet). NGINX (web serving), HAProxy (load balancing).

D15 — Medium: A **deadlock** is a circular wait where processes block each other. Coffman conditions: 1. Mutual Exclusion, 2. Hold and Wait, 3. No Preemption, 4. Circular Wait.

D16 — Medium: **Eventual consistency** ensures updates propagate over time. Cassandra uses **Read Repair** and **Hinted Handoff**. Consistency levels: ONE, QUORUM, ALL, LOCAL_QUORUM.

D17 — Medium: **mTLS** authenticates both client and server via certificates. Handshake: Client sends cert, Server validates; Server sends cert, Client validates. Used in Service Meshes (Istio) for secure service-to-service communication.

D18 — Medium: A **hash table** maps keys to indices. **Collision resolution**: Chaining (linked lists) or Open Addressing (probing). Average complexity: O(1). Worst case (all collisions): O(n).

D19 — Medium: **Circuit breaker** prevents cascade failures. States: **Closed** (normal), **Open** (fail fast), **Half-open** (test recovery). Relates to retries/backoff by stopping retries when the circuit is open.

D20 — Medium: **Optimistic** (check version on commit, no lock). **Pessimistic** (lock on read). Choose Optimistic for high concurrency/low conflict. Choose Pessimistic for critical data/high conflict (e.g., `SELECT FOR UPDATE`).
