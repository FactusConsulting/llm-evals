K1 — A **Deployment** manages stateless applications, ensuring a desired number of identical pod replicas are running. It handles rolling updates and rollbacks easily. A **StatefulSet** manages stateful applications, providing stable, unique network identifiers (hostnames) and ordered, graceful deployment/deletion of pods. StatefulSets are used for databases or services requiring persistent identity and ordered state.

K2 — A **ClusterIP** Service exposes a stable virtual IP within the cluster. Traffic routing is handled by **kube-proxy** on each node. kube-proxy maintains network rules (using iptables or IPVS) that map the Service IP:Port to the backend Pod IPs. When a pod sends traffic to the Service IP, the node's iptables rules DNAT the packet to one of the healthy backend Pod IPs.

K3 — A **PersistentVolume (PV)** is a cluster-wide storage resource (e.g., NFS, EBS). A **PersistentVolumeClaim (PVC)** is a user request for storage with specific size/access modes. A **StorageClass** defines the "class" of storage and how PVs are dynamically provisioned. A PVC binds to a PV (static) or triggers dynamic provisioning via a StorageClass.

K4 — Troubleshooting CrashLoopBackOff:
1.  **Logs:** `kubectl logs <pod> --previous` to see the last exit code/error.
2.  **Events:** `kubectl describe pod <pod>` to check for OOMKills, Liveness probe failures, or ImagePullBackOff.
3.  **Resource Limits:** Check if the container is exceeding memory/CPU limits causing termination.
4.  **Application Config:** Verify environment variables, ConfigMaps, or secrets are correctly mounted and valid.

K5 — **CoreDNS** provides DNS resolution for services within the cluster (Service discovery). If CoreDNS pods are down, internal service discovery (e.g., `my-service.default.svc.cluster.local`) fails. External DNS resolution may still work if the pod uses the host network or if the node's `/etc/resolv.conf` has fallback nameservers, but internal Kubernetes networking breaks.

K6 — The scheduler uses **Filtering** (Predicate) and **Scoring** (Priority).
1.  **Filtering:** Nodes must satisfy constraints (nodeSelector, taints/tolerations, resource requests, affinity).
2.  **Scoring:** Remaining nodes are ranked.
3.  **Factors:** CPU/Memory availability, Node Affinity/Anti-Affinity rules, Pod Priority/Preemption, and Topology Spread Constraints.

K7 — **Scheduling:** Yes. Node has 4GB total, 3.5GB requested. Free = 0.5GB. Pod requests 256Mi (~0.25GB). 0.5GB > 0.25GB, so it fits.
**Runtime:** If it uses 600Mi, it exceeds the `limits.memory: 512Mi`. The OOM Killer will terminate the container (OOMKilled), causing a CrashLoopBackOff.

K8 — 1. **External LB** forwards traffic to Ingress Controller NodePort.
2. **Ingress Controller** (Pod) receives request, parses Ingress rules (host/path).
3. **Service** (ClusterIP) is matched; kube-proxy routes traffic to **Endpoints** (Pod IPs).
4. **kube-proxy** (iptables/IPVS) performs DNAT to the specific backend Pod IP.
5. **App Pod** receives request on its container port.

K9 — A **NetworkPolicy** restricts traffic flow between pods. Default behavior (no policy): All traffic is allowed. When a policy is applied: Traffic is denied by default unless explicitly allowed by the policy's `ingress` or `egress` rules. It acts as a firewall for pods.

K10 — **RollingUpdate:** Gradually replaces old pods with new ones (zero downtime). **Recreate:** Terminates all existing pods before starting new ones. Choose **Recreate** for stateful applications that cannot handle concurrent versions or require a full shutdown before update (e.g., single-writer databases).

K11 — A **Namespace** provides logical isolation for resources within a cluster. Default namespaces: `default`, `kube-system`, `kube-public`, and `kube-node-lease`.

K12 — `kubectl describe` shows detailed status and events. `kubectl get` shows summary.
**Info in describe not in get:**
1.  **Events:** Warnings, scheduling issues, probe failures.
2.  **Conditions:** Pod status details (Ready, Initialized).
3.  **Mounts/Env:** Detailed volume mounts and environment variable values.

K13 — A **ConfigMap** stores non-sensitive configuration data as key-value pairs.
**Consumption Methods:**
1.  **Environment Variables:** Injected into the container environment.
2.  **Volume Mounts:** Mounted as files inside the container filesystem.

K14 — **ClusterIP:** Internal IP, accessible only within the cluster. **NodePort:** Exposes the service on a static port on each node's IP. **LoadBalancer:** Provisions an external cloud load balancer to expose the service externally.

K15 — A **DaemonSet** ensures a copy of a pod runs on all (or selected) nodes. Use a DaemonSet for cluster-level agents like logging (Fluentd), monitoring (Prometheus Node Exporter), or network plugins (CNI) that need to run on every node.

K16 — **Taints** are applied to nodes to repel pods (unless the pod has a **Toleration**). **Node Affinity** allows pods to be scheduled on nodes matching specific labels.
**Use Case:** Taint a node for GPU workloads (only pods with GPU tolerations run there). Use Affinity to co-locate pods on the same zone for low latency.

K17 — A **Kubernetes Operator** is a custom controller that manages the lifecycle of a complex application (e.g., database) using Custom Resource Definitions (CRDs). It differs from a standard controller by encoding operational knowledge (backup, scaling, failover) into the cluster. It solves the complexity of managing stateful, stateful applications declaratively.

K18 — **etcd** uses a strong consistency model (Raft consensus). If quorum is lost (e.g., 2 of 3 nodes lost), the cluster becomes read-only or unavailable.
**Recovery:** Restore etcd from a snapshot on a new node, rejoin it to the cluster, and ensure the remaining nodes are healthy to re-establish quorum.

K19 — **RBAC** controls access. **Role/ClusterRole** defines permissions (verbs/resources). **RoleBinding/ClusterRoleBinding** assigns roles to users/groups.
**Manifest:**
```yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  namespace: default
  name: pod-reader
rules:
- apiGroups: [""]
  resources: ["pods"]
  verbs: ["get", "list", "watch"]
- apiGroups: [""]
  resources: ["pods/log"]
  verbs: ["get", "list"]
---
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  name: read-pods-binding
  namespace: default
subjects:
- kind: User
  name: developer
  apiGroup: rbac.authorization.k8s.io
roleRef:
  kind: Role
  name: pod-reader
  apiGroup: rbac.authorization.k8s.io
```

K20 — **ndots** defines how many dots a domain name must have before it is considered absolute. Default is 5. If a domain has fewer dots, the resolver tries all search domains, causing multiple queries.
**Fix:** Set `dnsConfig` in the pod spec: `options: [{ name: "ndots", value: "1" }]`. This reduces unnecessary search domain queries for short hostnames.

D1 — **Rebase** rewrites commit history to create a linear sequence (avoid merge commits). **Merge** creates a new commit preserving history. Avoid rebasing on shared/public branches as it rewrites history and disrupts collaborators.

D2 — **TCP** is connection-oriented, reliable (guaranteed delivery). **UDP** is connectionless, faster (no guarantee).
**TCP:** HTTP, SSH. **UDP:** DNS, VoIP (SIP/RTP).

D3 — A **race condition** occurs when the output depends on the sequence/timing of uncontrollable events.
**Example:** Two threads increment a shared counter `x++`. Without a lock, both read `x`, increment, and write back, resulting in a lost update (count increases by 1 instead of 2).

D4 — **Symmetric:** Same key for encryption/decryption (fast). **Asymmetric:** Public key encrypts, private key decrypts (secure key exchange).
**TLS:** Asymmetric keys exchange a session key securely; Symmetric key encrypts the actual data transfer for performance.

D5 — A **container image layer** is a read-only filesystem snapshot added to the image.
**Caching:** Docker caches layers. If a layer hasn't changed, it's reused. Best practice: Copy dependency files (`package.json`) before source code so dependency installation layers are cached even if code changes.

D6 — Troubleshooting HTTP 503:
1.  **Health Checks:** Verify upstream service health endpoints.
2.  **Dependencies:** Check database or external API connectivity.
3.  **Rate Limiting:** Check if the client or service is rate-limited.
4.  **Logs:** Check application logs for specific error messages or timeouts.

D7 — **CAP Theorem:** A distributed system can only guarantee two of three: Consistency, Availability, Partition Tolerance.
**CP:** Zookeeper, HBase (prioritize data consistency over availability during partition).
**AP:** Cassandra, DynamoDB (prioritize availability, data eventually becomes consistent).

D8 — **Horizontal Scaling:** Adding more nodes/instances. **Vertical Scaling:** Increasing resources (CPU/RAM) of existing nodes.
**Patterns:** Stateless services, Load Balancing, Sharding, Microservices architecture enable horizontal scaling.

D9 — An **environment variable** is a dynamic value affecting process behavior.
**Persist:** Add `export VAR=value` to `~/.bashrc` or `~/.profile` to persist across sessions.

D10 — **JSON** is a strict key-value format (text-based). **YAML** is a human-readable data serialization (indentation-based).
**JSON Advantage:** Native support in most languages, strict parsing. **YAML Advantage:** Easier to read/write for humans, supports comments.

D11 — **REST API:** Architectural style using HTTP methods.
**GET:** Retrieve resource. **POST:** Create resource. **PUT:** Replace resource (idempotent). **DELETE:** Remove resource.

D12 — **Version pinning** specifies exact versions of dependencies (e.g., `package@1.2.3`).
**Importance:** Ensures reproducible builds, prevents unexpected breakage from upstream updates, and aids security auditing.

D13 — **Compiled:** Source code is translated to machine code before execution (e.g., C, Go). **Interpreted:** Source code is executed line-by-line at runtime (e.g., Python, JavaScript).

D14 — **Reverse Proxy:** Receives client requests and forwards them to backend servers (e.g., Nginx). **Forward Proxy:** Client requests go through proxy to reach internet (e.g., corporate firewall).
**Use Case:** Nginx (SSL termination, caching), HAProxy (Layer 4 load balancing).

D15 — A **deadlock** is a situation where processes wait indefinitely for each other.
**Coffman Conditions:**
1.  Mutual Exclusion.
2.  Hold and Wait.
3.  No Preemption.
4.  Circular Wait.

D16 — **Eventual Consistency:** Updates propagate asynchronously; reads may return stale data temporarily.
**Cassandra:** Uses Read Repair and Hinted Handoff.
**Levels:** ONE (fastest, least consistent), QUORUM (balanced), ALL (strongest, slowest).

D17 — **mTLS:** Mutual TLS where both client and server present certificates.
**Handshake:** Client sends cert -> Server validates -> Server sends cert -> Client validates.
**Usage:** Service Meshes (Istio), internal microservice communication security.

D18 — A **hash table** maps keys to values using a hash function.
**Collision:** **Chaining** (linked list at bucket), **Open Addressing** (probe next slot).
**Complexity:** Average O(1) for insert/lookup/delete. Worst case O(n) if all keys hash to same bucket.

D19 — **Circuit Breaker:** Prevents repeated calls to failing services.
**States:** **Closed** (normal), **Open** (fail fast, no calls), **Half-Open** (test if service recovered).
**Relation:** Prevents retries from overwhelming a failing service; exponential backoff is used when transitioning from Open to Half-Open.

D20 — **Optimistic:** Assumes no conflict; checks version on commit (e.g., `SELECT ... FOR UPDATE` is not used). **Pessimistic:** Locks data immediately upon read.
**Choice:** Optimistic for low contention (web apps). Pessimistic for high contention (banking).
**Example:** Optimistic uses `version` column; Pessimistic uses `SELECT FOR UPDATE`.
