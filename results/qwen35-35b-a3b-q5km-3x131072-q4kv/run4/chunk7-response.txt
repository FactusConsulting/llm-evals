AA1 — **Monolith vs. Microservices:** A monolith is a single deployable unit where all components are tightly coupled. Microservices are independently deployable, loosely coupled services. **3 Problems Microservices Introduce:** 1. **Network Latency:** Communication is over the network, not in-process. 2. **Distributed Consistency:** ACID transactions are hard; require sagas or eventual consistency. 3. **Operational Complexity:** Requires orchestration (Kubernetes), distributed tracing, and service discovery.

AA2 — **Sync vs. Async:** Synchronous communication waits for an immediate response (blocking). Asynchronous communication sends a message and continues processing without waiting (non-blocking). **Examples:** Synchronous: REST/HTTP or gRPC. Asynchronous: RabbitMQ (Queue) or Kafka (Log).

AA3 — **API Gateway vs. Reverse Proxy:** A reverse proxy routes traffic (L4/L7) and handles SSL termination. An API Gateway adds business logic (auth, rate limiting, transformation, routing). **Can it replace entirely?** Yes, for external entry points, as a Gateway is a specialized reverse proxy. However, internal service-to-service communication often uses simpler proxies or service mesh sidecars to avoid Gateway bottlenecks.

AA4 — **Saga Pattern:** A sequence of local transactions where each updates data and triggers the next. If one fails, compensating transactions undo previous steps. **Choreography:** Services publish events; others react (loose coupling, hard to debug). **Orchestration:** A central coordinator tells services what to do (centralized control, single point of failure risk). **Use Choreography** for simple flows; **Orchestration** for complex workflows requiring visibility and control.

AA5 — **CQRS:** Separates read (Query) and write (Command) models into different schemas. **Paired with Event Sourcing:** ES stores state changes as immutable events, which naturally feed the write model and can be projected to the read model. **Problem Solved:** Allows independent scaling of read/write workloads and provides a complete audit trail of state changes, which traditional CRUD (current state only) does not.

AA6 — **Circuit Breaker:** Prevents repeated calls to a failing service. **States:** Closed (normal), Open (failures threshold met, fail fast), Half-Open (testing if service recovered). **Differs from Retry:** Retry attempts to fix transient failures; Breaker stops hammering persistent failures. **Use Together:** Retry for transient network blips; Breaker for service outages to prevent cascading failures.

AA7 — **Tenant Isolation Models:**
1.  **Shared DB/Schema:** Lowest cost, highest noisy-neighbor risk, complex security (row-level).
2.  **Shared DB/Schema:** Medium cost, better isolation (schema per tenant), moderate ops complexity.
3.  **Separate DB:** Highest cost, best isolation/security, highest ops complexity (backup/migration per tenant).
*Recommendation:* Model 2 for mid-size SaaS; Model 3 for enterprise/high-security clients.

AA8 — **Consistency:** Strong = immediate visibility of changes. Eventual = changes propagate over time. **Inventory Architecture:** Requires Strong Consistency to prevent overselling. **Solution:** Use a relational DB (ACID) with distributed locking (e.g., Redis Redlock) or optimistic locking (versioning). At scale, partition inventory by SKU (sharding) to maintain strong consistency locally, then sync asynchronously for reporting.

AA9 — **Sidecar Proxy:** A proxy deployed alongside the application container. **Envoy/Istio:** Envoy handles traffic (Data Plane); Istio Control Plane configures Envoy. **mTLS:** Encrypts traffic between services and authenticates identity at the network layer, solving trust and encryption without modifying application code (unlike app-level TLS).

AA10 — **Strangler Fig:** Incrementally replace legacy functionality with new services. **Strategy:** Place a reverse proxy in front of the .NET Framework monolith. Route specific URL paths (e.g., `/api/new`) to new .NET 8 microservices. Gradually migrate features until the monolith is empty and can be decommissioned.

AA11 — **12-Factor App:** 1. Codebase (one repo), 2. Dependencies (explicit), 3. Config (env vars), 4. Backing Services (treat as attached resources), 5. Build/Run/Execute (separate), 6. Processes (stateless), 7. Port Binding (export via env), 8. Concurrency (scale via processes), 9. Disposability (fast start/stop), 10. Dev/Prod parity, 11. Logs (stream), 12. Admin (one-off processes). **Why:** Ensures portability, scalability, and maintainability in cloud environments.

AA12 — **Event-Driven vs. Request-Driven:**
*   **Coupling:** Event (Low) vs. Request (High).
*   **Scalability:** Event (High, independent scaling) vs. Request (Tied to load).
*   **Debugging:** Event (Hard, distributed) vs. Request (Easy, linear).
*   **Ordering:** Event (Hard to guarantee) vs. Request (Guaranteed).
*   **Failure:** Event (Compensating transactions) vs. Request (Timeouts/Retries).
*   **Choice:** Event for decoupling/async processing; Request for real-time user interactions.

AA13 — **Load Balancer:** Distributes traffic across servers. **L4:** Routes based on IP/Port (TCP/UDP). **L7:** Routes based on HTTP headers/URL. **Use Case:** SSL termination or routing `/api` to backend A and `/static` to backend B requires L7.

AA14 — **Message Queue vs. Log:** **MQ (RabbitMQ):** Messages consumed once, then deleted. Good for task queues. **Log (Kafka):** Messages retained, replayable. Good for event streaming/audit. **Pick MQ** for fire-and-forget tasks; **Pick Kafka** for data pipelines or replaying history.

AA15 — **Container vs. VM:** **Container:** Shares host OS kernel, isolates processes. **VM:** Virtualizes hardware, runs full OS. **Why Popular:** Containers are lighter (faster startup, higher density), portable, and align with microservices.

AA16 — **Stateless vs. Stateful:** **Stateless:** No session data stored on server (e.g., Web Server). **Stateful:** Stores session/data locally (e.g., Database). **Scaling:** Stateless scales horizontally easily (any node handles any request). Stateful requires data replication or sticky sessions.

AA17 — **ORM:** Maps database tables to programming objects. **Examples:** Entity Framework (.NET), Hibernate (Java). **Trade-offs:** ORMs increase dev speed but can generate inefficient SQL and hide performance issues. Raw SQL offers performance/complexity control but is verbose.

AA18 — **Bulkhead:** Isolates resources (threads, connections) for specific services. **Implementation:** Assign dedicated thread pools per downstream service. If one service hangs, it exhausts only its pool, not the whole application. Prevents cascading failures.

AA19 — **OT vs. CRDT:** **OT:** Server authoritative, calculates transformations. **CRDT:** Mathematically mergeable state, client authoritative. **Offline Support:** CRDT wins. **Choice:** CRDT for offline editing (Google Docs style) because conflicts resolve automatically without server round-trips.

AA20 — **Distributed Tracing:** **Context Propagation:** TraceID/SpanID passed via headers (W3C/B3). **Sampling:** Head (random at start) vs. Tail (based on latency/error). **Storage:** Jaeger/Tempo. **Kafka:** Inject trace context into message headers so downstream consumers can continue the trace. **Continuity:** Ensure all services (including Kafka producers/consumers) propagate headers.

OP1 — **Hypervisor Type 1:** Bare-metal (runs directly on hardware). Examples: VMware ESXi, Microsoft Hyper-V Server. **Type 2:** Hosted (runs on OS). Examples: VirtualBox, VMware Workstation. Type 1 is for production; Type 2 is for development.

OP2 — **SAN vs. NAS:** **SAN:** Block storage (LUNs). Protocols: FC, iSCSI. **NAS:** File storage (files/folders). Protocols: NFS, SMB. **Use SAN** for databases (low latency, block access). **Use NAS** for file shares, backups, home directories.

OP3 — **3-Tier:** Core (Backbone), Distribution (Policy), Access (Edge). **Leaf-Spine:** Preferred for modern DCs. **Problem Solved:** 3-Tier has oversubscription and high latency between tiers. Leaf-Spine provides non-blocking, low-latency, predictable bandwidth between any two nodes.

OP4 — **Proxmox HA:** **Nodes:** 3+ (odd for quorum). **Quorum:** Majority vote. **Fencing:** STONITH (power off failed node). **Storage:** Ceph (distributed) or iSCSI/NFS (shared). **Failure:** VMs restart on surviving nodes automatically.

OP5 — **VLAN Trunking:** Carries multiple VLANs over one link. **802.1Q:** Adds VLAN ID tag to Ethernet frame. **Linux Config:** `ip link add link eth0 name eth0.10 type vlan id 10` (repeat for each VLAN).

OP6 — **RAID:** **1:** Mirror (50% cap). **5:** Striping + Parity (1 drive loss). **6:** Double Parity (2 drive loss). **10:** Mirror + Stripe (High perf, 50% cap). **ZFS2:** RAID-Z2 (2 drive loss, checksums). **Choice:** RAID 10 for 6 drives (Best perf + fault tolerance for DB).

OP7 — **DNS Architecture:** **Internal:** AD DNS (Active Directory integrated). **External:** Public DNS (e.g., Route53). **Split-Horizon:** Internal resolves private IPs; External resolves public. **Laptop:** Connects via DHCP, queries internal DNS for AD resources, forwards external to ISP/Cloud. **DNSSEC:** Signs external zone to prevent spoofing.

OP8 — **PKI Chain:** Root CA (Offline) -> Subordinate CA (Issuing) -> End Entity. **Compromised Sub:** Revoke Sub, issue new certs. **Compromised Root:** Catastrophic; must rebuild entire PKI and trust store. **Auto-enrollment:** Clients request certs from Sub CA automatically.

OP9 — **WAN Consolidation:** **MPLS:** High cost, reliable. **IPsec:** Low cost, variable latency. **SD-WAN:** Low cost, intelligent routing, QoS. **WireGuard:** Simple, encrypted. **Recommendation:** SD-WAN. Best for cost, management, and VoIP QoS across 5 sites.

OP10 — **iDRAC/iLO/IPMI:** Out-of-band management (remote power/BIOS access). **Critical:** Allows recovery if OS is down. **Exposed to Internet:** High risk. Firmware vulnerabilities can lead to full server compromise (RCE) without OS access.

OP11 — **LVM vs. ZFS:** **LVM:** Logical volume manager. Thin provisioning possible. **ZFS:** Filesystem + Volume manager. Copy-on-write, checksums. **Snapshots:** LVM (fast, thin) vs. ZFS (instant, space-efficient, immutable). **Use ZFS** for data integrity; **LVM** for simple block management.

OP12 — **Stretch Cluster:** **CRUSH:** Data placement rules. **Monitors:** Must be in both sites. **Latency:** < 5ms required. **Split-Brain:** If network fails, both sites think they are primary. **Tiebreaker:** Use a witness node (quorum disk) to decide which site survives. **2-Site Ceph:** Bad without tiebreaker due to split-brain risk.

OP13 — **DHCP DORA:** **D**iscover, **O**ffer, **R**equest, **A**cknowledge. **Unavailable:** Client uses APIPA (169.254.x.x) or retries lease. **Expired:** Client broadcasts Discover to find new server.

OP14 — **Managed vs. Unmanaged:** **Unmanaged:** Plug-and-play. **Managed:** Configurable. **Features:** VLANs, QoS, Port Mirroring, ACLs, SNMP monitoring. **Importance:** Critical for security segmentation and traffic prioritization in enterprise.

OP15 — **NTP:** Network Time Protocol. **Critical:** Cluster consensus, log correlation, Kerberos auth (tickets expire based on time). **Drift:** Causes auth failures, log confusion, and database replication errors.

OP16 — **Physical vs. VM vs. Container:** **Physical:** Hardware only. **VM:** Virtual OS. **Container:** Virtualized process. **Bare Metal:** Use for high I/O, low latency (DB, HPC), or when licensing requires physical cores.

OP17 — **UPS vs. Generator:** **UPS:** Battery backup (seconds/minutes). **Generator:** Fuel backup (hours). **Both:** UPS bridges gap until generator starts. **ATS:** Automatic Transfer Switch switches power source from Utility to UPS/Generator.

OP18 — **Active-Passive vs. Active-Active:** **Active-Passive:** One node runs, one waits. **Active-Active:** Both run. **PostgreSQL HA:** Pacemaker/Corosync cluster. **Fencing:** STONITH to kill failed node. **VIP:** Virtual IP moves to active node. **Failover:** Client reconnects to VIP.

OP19 — **Backup Strategy:** **3-2-1:** 3 copies, 2 media, 1 offsite. **Immutable:** Prevents ransomware deletion. **Test:** Regular restore drills. **Backup vs. DR:** Backup is data copy; DR is the plan/process to recover operations after disaster.

OP20 — **802.1X:** **Supplicant:** Client. **Authenticator:** Switch. **Server:** RADIUS. **EAP-TLS:** Cert-based (secure). **PEAP:** Password-based (tunneled). **Fail:** Port blocked or placed in Guest VLAN. **MAB:** Switch sends MAC address to RADIUS as credential for non-802.1X devices (printers, IoT).
