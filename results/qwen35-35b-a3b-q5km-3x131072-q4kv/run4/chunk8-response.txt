CL1 — **IaaS, PaaS, SaaS**
*   **IaaS (Infrastructure as a Service):** Provides virtualized computing resources (servers, storage, networking). You manage OS, middleware, runtime, data, and apps. *Example:* AWS EC2.
*   **PaaS (Platform as a Service):** Provides hardware and software tools (usually for app development) hosted on the cloud. You manage data and apps. *Example:* AWS Elastic Beanstalk.
*   **SaaS (Software as a Service):** Provides fully functional software applications over the internet. You manage nothing but user access. *Example:* Microsoft 365.

CL2 — **Shared Responsibility Model**
*   **Concept:** Security is a shared duty. The provider secures the cloud (infrastructure); the customer secures what is in the cloud (data, config).
*   **IaaS VM:** Provider secures physical host, hypervisor, network. Customer secures OS, patching, firewall config, data.
*   **Managed Kubernetes:** Provider secures control plane, nodes, networking. Customer secures node OS (if self-managed), pod security, RBAC, network policies, app config.
*   **SaaS (M365):** Provider secures app, infrastructure, data center. Customer secures identity (MFA), data classification, device compliance, user access.

CL3 — **Networking Components (VPC, Subnet, SG, NACL)**
*   **VPC:** Isolated virtual network in the cloud.
*   **Subnet:** Range of IP addresses within a VPC (public/private).
*   **Security Group (SG):** Stateful firewall attached to instances. Allows all return traffic automatically.
*   **Network ACL (NACL):** Stateless firewall attached to subnets. Must explicitly allow return traffic.
*   **Layering:** VPC contains Subnets. Instances in Subnets have SGs. Subnets have NACLs.
*   **Stateful vs. Stateless:** SG is stateful (if request in, response out is allowed). NACL is stateless (inbound and outbound rules must be defined separately).

CL4 — **Multi-Account Strategy**
*   **Landing Zone:** Secure, compliant baseline environment for onboarding accounts.
*   **Organizational Units (OUs):** Logical grouping (e.g., Prod, Dev, Security) for policy application.
*   **Guardrails:** SCPs (Service Control Policies) to prevent risky actions (e.g., disabling CloudTrail).
*   **Shared Networking:** Transit Gateway (AWS) or Hub-and-Spoke (Azure) connects accounts centrally.
*   **Centralized Logging:** All logs sent to a dedicated Security/Logging account.
*   **Single-Account Problems:** High blast radius (one breach compromises all), billing confusion, compliance violations, lack of isolation between environments.

CL5 — **Infrastructure as Code (IaC)**
*   **IaC:** Managing infrastructure via machine-readable definition files rather than physical hardware configuration.
*   **Terraform/OpenTofu:** Declarative, multi-cloud, state-based. Best for general-purpose, multi-cloud, or complex dependencies.
*   **Pulumi:** Imperative/Declarative using general languages (TS, Python). Best for teams wanting code flexibility.
*   **CloudFormation/Bicep:** Native to AWS/Azure. Best for single-cloud, deep integration, and CI/CD pipelines within that ecosystem.
*   **Crossplane:** Kubernetes-native IaC. Best for managing cloud resources via K8s API (GitOps for infra).

CL6 — **Autoscaling Patterns**
*   **HPA:** Scales pods based on CPU/Memory or custom metrics.
*   **VPA:** Adjusts resource requests/limits for pods (requires restart).
*   **Cluster Autoscaler:** Adds/removes nodes based on pending pods.
*   **KEDA:** Event-driven autoscaler (e.g., queue length, Kafka).
*   **KEDA vs. HPA:** Use KEDA when scaling is driven by external events (message queues, HTTP requests) rather than internal resource usage.

CL7 — **Disaster Recovery (DR) Strategy**
*   **RPO/RTO:** RPO (Data loss tolerance), RTO (Downtime tolerance).
*   **Active-Active:** Traffic split across regions. High cost, low RTO/RPO.
*   **Active-Passive:** One region active, one standby. Lower cost, higher RTO.
*   **Database:** Sync (Synchronous replication, RPO=0, high latency) vs. Async (Replication lag, RPO>0).
*   **DNS Failover:** Route53/Traffic Manager health checks route traffic to healthy region.
*   **State:** Externalize state (S3, RDS) to avoid local state loss.
*   **Models:** Pilot Light (DB running, app off), Warm Standby (Scaled down app), Multi-site Active-Active (Full app in both).

CL8 — **GitOps Model**
*   **Model:** Git repository is the single source of truth. Controllers sync cluster state to Git state.
*   **ArgoCD vs. Flux:** ArgoCD (Push, UI-focused), Flux (Pull, CLI-focused).
*   **Secrets:** Sealed Secrets (encrypted in Git), External Secrets Operator (fetches from Vault/AWS Secrets Manager).
*   **Drift:** If manual change occurs, GitOps controller detects drift and reverts the cluster to match Git state automatically.

CL9 — **Stateful Legacy App to Kubernetes**
*   **StatefulSet:** Ensures stable pod identity and ordered startup/deletion.
*   **PV/PVC:** Use `local` storage class with `volumeBindingMode: WaitForFirstConsumer` for low latency.
*   **Headless Service:** Provides stable DNS names for StatefulSet pods (no load balancing).
*   **Init Containers:** Initialize config files or permissions before main app starts.
*   **ConfigMaps:** Mount as volumes to specific paths required by the legacy app.
*   **PDB:** Ensure minimum available pods during updates to prevent total outage.

CL10 — **Identity (SA, Workload, IRSA)**
*   **Service Account:** K8s identity for pods.
*   **Workload Identity:** GCP/Azure native mapping of K8s SA to cloud IAM roles.
*   **IRSA:** AWS IAM Roles for Service Accounts (OIDC federation).
*   **Mounting Credentials:** Bad practice (leaks keys, hard to rotate, violates least privilege).
*   **Federation:** Uses short-lived OIDC tokens instead of static keys, allowing pods to assume IAM roles securely.

CL11 — **Storage Types**
*   **Object (S3/Blob):** Unstructured data, HTTP API, high durability (11 9s). Best for backups, media, logs.
*   **Block (EBS/Managed Disk):** Raw storage, attached to VM. Low latency. Best for OS, databases.
*   **File (EFS/Azure Files):** Shared file system (NFS/SMB). Best for shared configs, legacy apps.
*   **Guarantees:** Object/Block usually 99.999999999% durability. Availability varies by tier (Standard vs. Archive).

CL12 — **Zero-Trust Kubernetes**
*   **Network Policies:** Restrict pod-to-pod traffic (default deny).
*   **Service Mesh (mTLS):** Encrypts service-to-service traffic (Istio/Linkerd).
*   **Pod Identity:** Workload Identity/IRSA for cloud access.
*   **Admission Controllers:** OPA/Kyverno to enforce policies on deployment (e.g., no root user).
*   **Image Signing:** Cosign to verify image integrity before deploy.
*   **Runtime Security:** Falco to detect anomalies (e.g., shell in container).
*   **Diff:** Traditional = Trust inside perimeter. Zero Trust = Verify every request, assume breach.

CL13 — **CDN (Content Delivery Network)**
*   **Function:** Edge cache servers deliver content closer to users.
*   **Services:** AWS CloudFront, Cloudflare.
*   **Use Case:** Use CDN for static assets (images, CSS, JS) to reduce latency and origin load. Do not use for dynamic, personalized, or real-time data.

CL14 — **Availability Zones (AZ) and Regions**
*   **AZ:** Isolated datacenter within a region (power, cooling, network).
*   **Region:** Geographic area with multiple AZs.
*   **Multi-AZ:** Protects against single datacenter failure.
*   **Zonal:** EBS volumes (tied to AZ). **Regional:** S3 buckets (replicated across AZs).

CL15 — **Managed Database Service**
*   **Definition:** RDS, Cloud SQL, Azure SQL.
*   **Provider Handles:** Hardware provisioning, patching, backups, failover, scaling, monitoring.
*   **Customer Handles:** Schema design, queries, user permissions, data encryption at rest.

CL16 — **Cloud Identity Services**
*   **Users:** Individual identities. **Groups:** Collections of users. **Roles:** Temporary credentials for access. **Policies:** JSON rules defining permissions.
*   **Least Privilege:** Grant only necessary permissions. Apply by using Groups/Roles, not individual user policies, and regularly auditing access.

CL17 — **Serverless Computing**
*   **Definition:** Event-driven code execution without managing servers.
*   **Services:** AWS Lambda, Azure Functions, GCP Cloud Functions.
*   **Cold Starts:** Latency when a function is invoked after inactivity (initialization). Matters for latency-sensitive apps.
*   **Poor Fit:** Long-running processes, stateful apps, predictable high-throughput (reserved capacity cheaper).

CL18 — **Kubernetes Ingress Options**
*   **Ingress Resource:** Standard K8s resource for HTTP routing.
*   **Gateway API:** New standard, supports multiple protocols, better separation of concerns.
*   **Cloud LB:** Provider-specific annotations (e.g., AWS NLB).
*   **Mesh Ingress:** Istio/Linkerd entry point.
*   **TLS:** Terminates at LB (Edge), Ingress Controller, or Sidecar.
*   **Gateway API:** Use for complex routing, multi-cluster, or when Ingress limitations are hit.

CL19 — **Cost Optimization Strategy**
*   **Compute:** Reserved Instances (1-3yr commitment), Savings Plans (flexible), Spot/Preemptible (interruptible, cheap).
*   **Right-sizing:** Match instance size to actual utilization.
*   **Storage:** Intelligent-Tiering, Archive (Glacier) for old data.
*   **Idle Detection:** Tagging, AWS Cost Explorer, CloudHealth to find unused resources.
*   **FinOps:** Culture of cost accountability. Showback/Chargeback models to bill teams.
*   **Tools:** AWS Cost Explorer, Kubecost, CloudHealth.

CL20 — **Multi-Cloud Strategy**
*   **Abstraction:** Kubernetes (runtime), Terraform (IaC), Crossplane (API).
*   **Networking:** Direct Connect/ExpressRoute or VPN for low latency.
*   **Identity:** Federation (OIDC) across clouds.
*   **Trade-offs:** Increased complexity, higher networking costs, skill fragmentation.
*   **Justified:** Regulatory compliance, avoiding single-vendor outage, leveraging best-of-breed services.
*   **Unnecessary:** Simple apps, high operational overhead outweighs benefits.

OT1 — **Purdue Model (ISA-95)**
*   **Level 0:** Physical Process (Sensors, Actuators).
*   **Level 1:** Basic Control (PLCs, RTUs).
*   **Level 2:** Area Supervisory Control (SCADA, HMIs).
*   **Level 3:** Operations Management (MES, Historian).
*   **Level 4:** Enterprise Business Planning (ERP).
*   *(Note: Level 5 is Business/Internet, often grouped with 4).*

OT2 — **IT vs. OT Networks**
*   **IT:** Confidentiality focus, frequent patching, standard protocols (TCP/IP).
*   **OT:** Safety/Availability focus, legacy systems, real-time requirements.
*   **Patching:** OT patching risks downtime or safety hazards. OT systems often run unsupported OS (XP/7) and cannot be rebooted easily.

OT3 — **DMZ (IDMZ)**
*   **Role:** Buffer zone between IT (L4) and OT (L0-L3).
*   **Traffic:** IT -> OT (Updates, Commands), OT -> IT (Data, Logs).
*   **Direction:** Strictly controlled.
*   **Danger:** Direct L4->L1 connectivity allows malware to jump from corporate network to control systems without inspection, risking physical safety.

OT4 — **OT Protocols**
*   **Modbus TCP:** L1/L2, Master/Slave, simple.
*   **OPC UA:** L2/L3, Secure, platform-independent, modern.
*   **EtherNet/IP (CIP):** L1/L2, Allen-Bradley dominant.
*   **PROFINET:** L1/L2, Real-time industrial Ethernet.
*   **MQTT:** L3/L4, Lightweight, publish/subscribe (Cloud/IoT).

OT5 — **Data Diode vs. Firewall**
*   **Data Diode:** Hardware enforcing one-way data flow (physically).
*   **Firewall:** Software/Hardware filtering two-way traffic.
*   **Use Diode:** When data must leave OT but nothing can enter (e.g., sending data to cloud).
*   **Limitation:** No feedback loop, cannot send commands back to OT.

OT6 — **Data Flow Design (SCADA to Cloud)**
*   **Flow:** PLC -> Historian (L3) -> OPC UA Server -> Edge Gateway -> MQTT Broker -> Cloud Ingestion.
*   **Security:** Edge Gateway acts as DMZ. TLS encryption between Edge and Cloud. Firewall rules allow only specific ports/protocols.

OT7 — **IEC 62443 Zones and Conduits**
*   **Zone:** Group of assets with same security requirements.
*   **Conduit:** Logical path for traffic between zones.
*   **Design:**
    *   Zone 1: Corporate (L4).
    *   Zone 2: Engineering (L3).
    *   Zone 3: SCADA Servers (L3).
    *   Zone 4: PLCs/SIS (L1/L2).
    *   Conduits: Firewalls between zones. Engineering Zone can access SCADA Zone. SCADA Zone accesses PLC Zone.

OT8 — **Network Segmentation Strategy**
*   **VLANs:** Separate VLANs per production line.
*   **Firewalls:** Layer 3 firewalls between VLANs and DMZ.
*   **Switches:** Managed switches with ACLs.
*   **Remote Access:** Jump host with MFA, session recording, time-limited tokens.
*   **Transition:** Parallel run (new network alongside old), phased cutover during maintenance windows, rollback plan.

OT9 — **SIS vs. BPCS**
*   **SIS (Safety Instrumented System):** Independent system to bring process to safe state (e.g., emergency shutdown).
*   **BPCS (Basic Process Control System):** Normal control (e.g., PID loops).
*   **IEC 61511:** Standard for Safety Instrumented Systems.
*   **Isolation:** SIS must be air-gapped to prevent cyber attacks from disabling safety functions.
*   **Ransomware on SIS:** Could prevent emergency shutdown, leading to physical damage or injury.

OT10 — **PLC, DCS, SCADA**
*   **PLC:** Programmable Logic Controller. Localized logic, fast response.
*   **DCS:** Distributed Control System. Integrated control for continuous processes (refineries).
*   **SCADA:** Supervisory Control and Data Acquisition. Monitoring and high-level control over wide areas.
*   **Coexistence:** Yes. PLCs/DCS handle control, SCADA aggregates data and provides HMI.

OT11 — **Remote Access Architecture**
*   **Components:** Jump Host (bastion), MFA, Session Recording, Time-limited access.
*   **Vendor Access:** Dedicated accounts, no direct network access, monitored sessions.
*   **Unacceptable:** Direct RDP to HMI from corporate network bypasses segmentation, increases lateral movement risk, lacks audit trail.

OT12 — **Incident Response (Nation-State)**
*   **Detection:** IDS/Anomaly detection on OT traffic, asset inventory verification.
*   **Response:** Isolate compromised segment (VLAN/ACL), do not pull power (safety risk).
*   **Coordination:** Notify Control Room immediately. Shift to manual operation if needed.
*   **Operator Role:** Maintain physical safety, monitor process variables, execute manual procedures.
*   **Goal:** Contain threat without compromising safety or production integrity.

OT13 — **HMI (Human-Machine Interface)**
*   **Role:** Operator interface to view process status and issue commands.
*   **Difference:** HMI is for operations (Level 2). Engineering Workstation is for configuration/programming (Level 3/4).

OT14 — **Historian**
*   **Role:** Time-series database for process data.
*   **Importance:** Trend analysis, compliance, troubleshooting.
*   **Plant Historian (L3):** Real-time ops data, high frequency.
*   **Enterprise Historian (L4):** Aggregated data for analytics, reporting, lower frequency.

OT15 — **Ladder Logic**
*   **Definition:** Graphical programming language resembling relay logic.
*   **Device:** PLC.
*   **Dominance:** Familiar to electricians, deterministic execution, proven reliability in harsh environments.

OT16 — **RTU vs. PLC**
*   **RTU:** Remote Terminal Unit. Designed for harsh environments, long-distance comms, low power.
*   **PLC:** Local control, high I/O density.
*   **RTU Use:** Pipelines, water distribution, remote sites.

OT17 — **Analog vs. Digital Signals**
*   **Analog:** Continuous voltage/current (4-20mA, 0-10V).
*   **Digital:** Discrete states (On/Off).
*   **4-20mA:** Preferred over 0-20mA because 4mA is "live zero" (indicates wire break if current drops to 0mA).

OT18 — **OT Asset Inventory**
*   **Foundation:** You cannot secure what you don't know exists.
*   **Tools:** Passive monitoring (packet sniffing), Active scanning (ping/scan).
*   **Risk:** Active scanning can crash legacy devices or disrupt control loops. Passive is safer.

OT19 — **MITRE ATT&CK for ICS**
*   **Tactics:** Reconnaissance, Initial Access, Execution, Persistence, Impact.
*   **Techniques:** E.g., "Exploit Public-Facing Application", "Data Manipulation", "Inhibit System Recovery".
*   **Difference:** Focuses on physical impact and safety, not just data theft.
*   **Usage:** Map existing controls to tactics to find gaps (e.g., no detection for "Data Manipulation").

OT20 — **Converged IT/OT SOC**
*   **Log Collection:** Agents on OT assets (non-intrusive), SIEM integration.
*   **Rules:** OT-specific signatures (e.g., Modbus command anomalies).
*   **Staff:** IT analysts (threat hunting) + OT Engineers (process knowledge).
*   **False Positives:** High tuning required to avoid alert fatigue.
*   **Firmware Change Alert:** Stop traffic to asset, verify change request, isolate if unauthorized, restore from backup, patch.
