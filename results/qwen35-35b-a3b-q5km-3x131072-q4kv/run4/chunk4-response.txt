G1 — **Slice vs Array:** An array is a fixed-size value type `[T]N`. A slice is a dynamic descriptor containing a pointer to an underlying array, length, and capacity `[]T`. When appending to a slice at capacity, Go allocates a new, larger underlying array (typically doubling capacity), copies existing elements, and updates the slice header.

G2 — **`var` vs `:=`:** `var x int` declares a variable with an explicit type and zero value. `x := 0` is a short declaration that infers the type. You cannot use `:=` to redeclare a variable in the same scope (e.g., inside a function block), only to declare new ones.

G3 — **Goroutine vs Thread:** A goroutine is a lightweight thread managed by the Go runtime (user-space), whereas an OS thread is managed by the kernel. The Go scheduler uses the GMP model: **G** (goroutine), **M** (OS thread), **P** (logical processor). The scheduler multiplexes Gs onto Ms, allowing thousands of goroutines on fewer OS threads.

G4 — **Channels:** Channels are typed conduits for communication. **Unbuffered** channels block both sender and receiver until the other side is ready (synchronous). **Buffered** channels allow sending without blocking until the buffer is full. Sending to a full buffered channel blocks until a receiver reads.

G5 — **Context:** The `context` package manages deadlines, cancellation, and request-scoped values.
```go
ctx, cancel := context.WithTimeout(context.Background(), 5*time.Second)
defer cancel()
resp, err := http.GetContext(ctx, "http://example.com")
```

G6 — **Interfaces:** Go interfaces are satisfied implicitly (duck typing). Unlike Java/C#, you do not declare implementation. `interface{}` (or `any`) is the empty interface. It is powerful for generic storage but dangerous because it requires type assertions, losing compile-time safety.

G7 — **Memory Model:** The code has a data race. `ready` is not atomic, and `data` is written without synchronization. The loop `for !ready {}` may spin forever or read stale data.
**Fix:** Use `sync.Once` or `atomic.Bool`.
```go
var ready atomic.Bool
var data string
go func() {
    data = "hello"
    ready.Store(true)
}()
for !ready.Load() {}
fmt.Println(data)
```

G8 — **Mutex vs RWMutex:** `Mutex` is for exclusive access. `RWMutex` allows multiple readers or one writer. `sync.Map` is optimized for sparse maps where keys are unique and rarely deleted. It avoids lock contention for reads but has higher overhead for writes compared to a `map` with a `Mutex`.

G9 — **Modules:** `go.mod` declares dependencies and module path. `go.sum` stores checksums for integrity. `go mod tidy` removes unused imports and adds missing ones. **MVS** (Minimum Version Selection) selects the lowest compatible version of a dependency that satisfies all requirements.

G10 — **Defer:** `defer` schedules a function call until the surrounding function returns. Multiple defers execute in **LIFO** (Last In, First Out) order. A common gotcha in loops is capturing the loop variable by reference in a closure; use a local copy variable to capture the current value.

G11 — **Pointers:** A pointer holds a memory address. The zero value is `nil`. Create a pointer to a struct literal using `&Struct{Field: value}`.

G12 — **Init:** `init()` runs automatically before `main()` when a package is loaded. A package can have multiple `init()` functions; they execute in the order they appear in the source file.

G13 — **Make vs New:** `new(T)` allocates zeroed memory of type `T` and returns a pointer `*T`. `make(T)` initializes slices, maps, and channels and returns the type itself (not a pointer).

G14 — **Errors:** Errors are values implementing the `error` interface (`Error() string`). Idiom: `if err != nil { return err }`. `errors.Is` checks if an error matches a target (wrapping). `errors.As` performs type assertion on wrapped errors. `==` only works for exact error values.

G15 — **Composite Types:**
*   **Map:** Zero value is `nil`.
*   **Slice:** Zero value is `nil`.
*   **Channel:** Zero value is `nil`.
*   **Struct:** Zero value has all fields set to their respective zero values.

G16 — **Race Condition:** Occurs when two goroutines access shared data concurrently, at least one writes, without synchronization. The `-race` flag instruments code to detect this at runtime.
**Fix:** Use `sync.Mutex` to protect the shared variable or use channels for communication.

G17 — **Select:** `select` chooses a ready channel operation. Unlike `switch`, it is non-blocking and waits for a channel to be ready. If multiple are ready, one is chosen randomly. Timeout is implemented via `time.After` case.

G18 — **Escape Analysis:** Determines if a variable lives on the stack or heap. If a variable's address escapes the function scope, it is allocated on the heap. Inspect with `go build -gcflags="-m"`. Heap allocation is slower and triggers GC; stack allocation is faster.

G19 — **Reflection:** `reflect.Type` describes type metadata; `reflect.Value` holds the actual data. Reflection is slow (dynamic dispatch) and bypasses compile-time checks. Justified for frameworks (e.g., serialization) but avoided in hot paths.

G20 — **Embed:** `//go:embed` embeds files/directories at compile time into a `fs.FS`. Works with single files, directories, or glob patterns. Limitations: No symlinks, static content only. Interacts with `io/fs` to serve embedded files as standard file systems.

R1 — **Ownership:** Rust's core system. 1. Each value has a single owner. 2. When owner goes out of scope, value is dropped. 3. Values can be moved (transferred ownership) or borrowed.

R2 — **String vs &str:** `String` is an owned, heap-allocated, growable buffer. `&str` is a borrowed slice of bytes (often a string literal). Use `&str` for function arguments (zero-copy), `String` for owned/modifiable data.

R3 — **Borrowing:** You can have multiple immutable references (`&T`) OR one mutable reference (`&mut T`), but not both simultaneously. This prevents data races at compile time.

R4 — **Result/Option:** `Result<T, E>` handles errors. `Option<T>` handles nullability (`Some`, `None`). The `?` operator propagates errors up the call stack.
```rust
fn read_file() -> Result<String, io::Error> {
    std::fs::read_to_string("file.txt")
}
```

R5 — **Box/Rc/Arc:** `Box<T>` owns data on heap (single owner). `Rc<T>` (Reference Counted) allows multiple owners in single-threaded contexts. `Arc<T>` is thread-safe `Rc` for multi-threaded contexts.

R6 — **Traits:** Define shared behavior. Like Go interfaces but can have default implementations. `dyn Trait` uses dynamic dispatch (vtable, runtime cost). `T: Trait` uses static dispatch (monomorphization, zero cost).

R7 — **Lifetimes:** Ensures references are valid. The function `longest` compiles in modern Rust due to lifetime elision rules (return lifetime tied to input). Explicitly: `fn longest<'a>(x: &'a str, y: &'a str) -> &'a str`. It fails in older versions or without elision because the compiler cannot infer which input outlives the output.

R8 — **Send/Sync:** `Send` means a type can be transferred across threads. `Sync` means `&T` is safe to share across threads. `Rc<T>` is not `Send` because ref-counting is not atomic. `Arc<T>` is `Send`/`Sync` because it uses atomic ref-counting.

R9 — **Async/Await:** `async` creates a `Future` (state machine). The executor (e.g., Tokio) polls the future until ready. Unlike Go's preemptive OS threads, Rust's async is cooperative and typically runs on a thread pool, reducing overhead but requiring explicit yielding.

R10 — **Unsafe:** Allows operations the compiler can't verify. 1. Dereferencing raw pointers. 2. Calling unsafe functions. 3. Accessing mutable statics. Justified for FFI, low-level optimizations, or implementing safe abstractions.

R11 — **Pattern Matching:** `match` is exhaustive (must handle all cases). `if let` handles a single pattern. The compiler enforces exhaustiveness in `match` to prevent logic errors.

R12 — **Vec vs Array:** `Vec<T>` is heap-allocated, dynamic size. `[T; N]` is stack-allocated, fixed size. `Vec` reallocates (grows capacity) when pushing beyond current capacity, copying elements to a new buffer.

R13 — **Enums:** Tagged unions (can hold different types). `Option<T>` replaces `null` with `Some(T)` or `None`. This forces explicit handling of missing values, preventing null pointer exceptions.

R14 — **Impl:** Defines methods. `&self` borrows, `&mut self` mutably borrows, `self` consumes the value. Associated functions (e.g., `String::new()`) do not take `self` and act like static methods.

R15 — **Derive:** Macros generating boilerplate code (e.g., `#[derive(Debug)]`). Cannot derive if fields contain types that don't support the trait (e.g., `Clone` on `&T` or functions).

R16 — **Cow:** "Clone on Write". Holds a borrowed reference until mutated, then clones. Avoids allocation if the data is only read. Example: Formatting a string that might already be owned.

R17 — **Modules:** `mod` defines a module. `use` imports paths. `pub` exports. Filesystem maps to modules (`mod.rs` or `mod/`). Rust 2018 improved path resolution (`crate::` vs `self::`) and module visibility.

R18 — **Pin:** Prevents a value from being moved in memory. Needed for self-referential structs and `Future`s (which may hold internal pointers). `Unpin` is the trait for types that can be moved safely.

R19 — **Macros:** `macro_rules!` (declarative) matches patterns at compile time. Procedural macros (derive, attribute, function-like) operate on AST. Hygiene ensures macro variables don't collide with local scope variables.

R20 — **Zero-Cost Abstractions:** High-level features compile to efficient machine code. Generics use monomorphization (copy-paste code per type). Iterators are inlined loops. No runtime overhead compared to C, while providing safety.
