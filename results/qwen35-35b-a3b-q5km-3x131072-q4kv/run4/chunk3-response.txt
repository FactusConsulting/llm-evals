T1 — `tofu plan` generates an execution plan showing what changes will be made without applying them. `tofu apply` executes the plan, creating, updating, or destroying resources to match the configuration.

T2 — State is a JSON file tracking the mapping between your configuration and real-world resources. Remote state is important for collaboration (sharing state among team members), security (storing secrets away from local machines), and enabling state locking to prevent concurrent modifications.

T3 — `count` uses integer indices to create resources; `for_each` uses map or set keys. `for_each` is preferred when the number of resources is dynamic or when you need stable identifiers (keys) that don't shift if items are removed, preventing unintended resource recreation.

T4 — This is "drift". OpenTofu detects it during `tofu plan` by comparing the state file to the actual infrastructure. If the configuration matches the desired state, `tofu apply` will revert the manual change. If the configuration differs, it will update the resource to match the config.

T5 —
```hcl
resource "aws_instance" "web" {
  for_each = toset(["web1", "web2", "web3"])
  ami      = "ami-12345678"
  instance_type = "t2.micro"
  tags = {
    Name = each.key
  }
}
```

T6 — `terraform_remote_state` reads the state file of another stack (cross-state), creating tight coupling. `output` values are an explicit interface between modules (intra-state). Recommended alternative: Use `output` values via module calls if resources are in the same state to reduce coupling and improve performance. If cross-state is required, use `terraform_remote_state` but only read specific outputs.

T7 — The `null_resource` provisioner runs after the resource is created. If the kubeconfig is generated by the provisioner, the output is empty until the provisioner finishes. Fix: Use a `local_file` resource to store the kubeconfig, have the `null_resource` provisioner write to it, and have dependent resources read the `local_file`. Alternatively, add `depends_on` to the dependent resource.

T8 — `tofu import` adds existing infrastructure to state without creating config. Limitations: No config generation, manual mapping. The `import` block (1.5+) allows declarative import within the configuration file, enabling version-controlled import logic and better integration with CI/CD pipelines.

T9 — A `variable` is an input parameter passed into the module or run. An `output` is data exposed from the module to be used by other modules or the CLI. Variables are for configuration; outputs are for data retrieval.

T10 — A provider is a plugin that interacts with a specific API (e.g., AWS, Azure). If you run `tofu plan` without `tofu init`, the command fails because the provider binaries are not downloaded and initialized.

T11 — The `.terraform.lock.hcl` file records the exact versions of providers and plugins used. It should be committed to version control to ensure reproducible builds and prevent unexpected provider upgrades that might break infrastructure.

T12 — `tofu destroy` removes all resources managed by the state. To target a single resource: `tofu destroy -target=aws_instance.web`.

T13 — A data source reads information about existing infrastructure rather than managing it. Example: Using `data "aws_ami" "ubuntu"` to find the latest Ubuntu AMI ID instead of managing an AMI resource.

T14 — `locals` define computed values used internally within a module. `variable` blocks define inputs from outside. Use `locals` for logic, transformations, or constants that shouldn't be configurable by the user.

T15 — `lifecycle` controls resource behavior. `create_before_destroy`: Creates new resource before destroying old (zero downtime). `prevent_destroy`: Prevents accidental deletion (e.g., databases). `ignore_changes`: Ignores drift on specific attributes (e.g., tags managed externally).

T16 — A `moved` block tracks resource identity changes during refactoring (e.g., renaming a module or resource type). It updates the state file to map the old address to the new address without destroying and recreating the resource.

T17 — The dependency graph determines execution order. Implicit dependencies are inferred from references (e.g., `aws_instance.web.id`). Explicit dependencies use `depends_on`. `depends_on` reduces parallelism by forcing sequential execution, which should be used sparingly to avoid slowing down plans.

T18 — 1. **State Splitting:** Break large state into modules. Tradeoff: More state files to manage. 2. **`-refresh=false`:** Skip API calls to check state. Tradeoff: Plan may be stale. 3. **Reduce `depends_on`:** Allow parallelism. Tradeoff: Risk of race conditions. 4. **Provider Caching:** Use local provider cache. Tradeoff: Requires disk space and management.

T19 — State locking prevents concurrent writes to the state file. If a lock is not released (e.g., crash), subsequent commands fail. Manually break lock with `tofu force-unlock <LOCK_ID>`. Risk: Data corruption if another process is actually running.

T20 — 1. Backup existing state file. 2. Configure the new backend in `backend` block. 3. Run `tofu init -migrate-state`. 4. Verify state is in S3 and lock works. 5. Run `tofu plan` to ensure state matches config.

A1 — A **Playbook** is a YAML file defining the execution steps. A **Role** is a structured directory for organizing tasks, handlers, and variables. A **Task** is a single action (e.g., install a package) within a playbook or role.

A2 — `ansible_facts` is a dictionary containing gathered system information. Access OS family via `{{ ansible_facts['os_family'] }}`.

A3 — `command`: Executes without shell (no pipes). `shell`: Executes with shell (supports pipes). `raw`: Executes without setup (no module setup, no fact gathering), useful for early bootstrapping.

A4 — Idempotency means running a task multiple times produces the same result. Example (Idempotent): `apt: name=nginx` (installs only if missing). Example (Not Idempotent): `shell: rm -rf /tmp/*` (deletes every time).

A5 —
```yaml
- name: Install htop
  package:
    name: htop
  become: yes
  when: ansible_os_family == "Debian"

- name: Install htop on RedHat
  package:
    name: htop
  become: yes
  when: ansible_os_family == "RedHat"
  # Add epel-release task if needed
```

A6 — Precedence (High to Low): Extra_vars > Command line > Playbook vars > Host vars > Group vars > Role defaults > Role vars > Inventory vars. `set_fact` overrides most but is runtime.

A7 — Use `serial: 3` in the play to process 3 nodes at a time. Use `meta: wait_for` or `wait_for` module to ensure the first 3 are ready before the next batch starts.

A8 — `ansible-vault` encrypts sensitive data. To encrypt a single variable: `ansible-vault encrypt_string 'secret_value' --name 'my_secret' -e`.

A9 — An inventory file defines the hosts and groups Ansible manages. **Static:** Defined in a file (INI/YAML). **Dynamic:** Generated by a script or API (e.g., AWS EC2), useful for cloud environments.

A10 — `ansible webservers -m shell -a "df -h"`.

A11 — A **handler** is a task that runs only when notified by another task. It differs from a regular task by running asynchronously after the play finishes (or at the end of the block). Triggered via `notify: handler_name`.

A12 — `ansible.cfg` configures Ansible behavior. Common settings: `inventory` (path), `remote_user` (default SSH user), `forks` (parallel connections).

A13 — `register` saves the result of a task into a variable. Use it in subsequent tasks via `{{ registered_var.stdout }}` or `{{ registered_var.failed }}`.

A14 — `import_tasks` is static (evaluated at parse time, no conditionals). `include_tasks` is dynamic (evaluated at runtime, supports conditionals). Use `import` for performance, `include` for dynamic logic.

A15 — Ansible Galaxy is a repository for roles and collections. Use `requirements.yml` to list them, then run `ansible-galaxy install -r requirements.yml` to install.

A16 — Connection plugins manage how Ansible connects to hosts. `ssh`: Remote Linux/Unix. `local`: Control node itself. `network_cli`: Network devices (Cisco, Juniper).

A17 — `delegate_to: localhost` runs a task on the control node. `run_once: true` ensures it runs once even in a loop. Example: Gather facts from remote, then process them locally to generate a config file.

A18 — 1. **Increase `forks`:** More parallel connections. Tradeoff: Control node resource usage. 2. **Enable `pipelining`:** Reduces SSH connections. Tradeoff: SSH config requirements. 3. **Fact Caching:** Store facts to avoid gathering. Tradeoff: Stale facts. 4. **Mitogen:** Optimized SSH. Tradeoff: Compatibility/Complexity.

A19 — `{{ }}` for expressions (output). `{% %}` for logic (loops/ifs). `{# #}` for comments.
```jinja
upstream backend_servers {
{% for server in servers %}
    server {{ server.ip }} weight={{ server.weight }}; # Health check: {{ server.health }}
{% endfor %}
}
```

A20 — 1. Set `serial: 5` (batches). 2. `pre_tasks`: Deregister from LB. 3. `tasks`: Deploy app. 4. `post_tasks`: Health check via `uri` module. 5. `block/rescue`: If health check fails, rollback (revert deployment, re-register LB).
