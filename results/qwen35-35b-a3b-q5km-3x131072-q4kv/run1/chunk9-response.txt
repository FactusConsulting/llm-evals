SC1-A
The pods are failing because the **Restricted Pod Security Standard** is stricter than **Baseline**.
1.  **Restricted Policy:** Requires `securityContext.runAsNonRoot: true` AND an explicit `securityContext.runAsUser` (non-zero UID). It forbids running as root.
2.  **Baseline Policy:** Allowed `runAsNonRoot: true` without an explicit UID, defaulting to the image's `USER` directive.
3.  **The Conflict:** The new base image (`cgr.dev/chainguard/static:latest`) defaults `USER` to root (UID 0).
4.  **Result:** The namespace enforces Restricted (no root allowed). The pod spec has `runAsNonRoot: true` but no `runAsUser`. Kubernetes defaults `runAsUser` to the image's USER (0). Since 0 is not allowed under Restricted, the container runtime (e.g., containerd) refuses to start the container, causing a CrashLoopBackOff (often with `Error: container failed to start` or similar security context violation).

SC1-B
```bash
kubectl get pods --all-namespaces -o json | jq -r '.items[] | select(.status.containerStatuses[]?.state.waiting.reason == "CrashLoopBackOff") | "\(.metadata.namespace) \(.metadata.name) \(.spec.securityContext)"'
```

SC1-C
**Immediate Remediation:**
1.  **Image Fix:** Rebuild the base image or override the `USER` in the Dockerfile to a non-root UID (e.g., `USER 1000`).
2.  **Manifest Fix:** Add `securityContext.runAsUser: 1000` to the Pod/Container spec to explicitly satisfy Restricted.
3.  **Rollout:** Use a rolling update (`kubectl rollout restart`) with a `maxSurge` of 0 and `maxUnavailable` of 10% to ensure zero downtime.

**Prevention:**
1.  **CI Gate:** Integrate `kube-score` or `checkov` in CI to validate manifests against the target Pod Security Standard before merge.
2.  **Admission Controller:** Deploy OPA Gatekeeper or Kyverno in the cluster to enforce `runAsUser` presence when `runAsNonRoot` is true.
3.  **Image Policy:** Enforce image signatures (Cosign) and scan for `USER` directives in CI to ensure base images are compliant.

SC2-A
**Stale Data:** Promoting a cross-region read replica creates a standalone primary. Since replication is asynchronous, the replica may not have received all WAL (Write-Ahead Log) records from the primary before the failover.
**Replication Lag:** PostgreSQL streaming replication ships WAL files. Across regions, network latency and bandwidth limit throughput. If the primary crashes before WAL is shipped to EU, that data is lost.
**Data Loss Window:** The window is the time between the last committed transaction on US and the last WAL record shipped to EU.
**Quantification:** Check `pg_stat_replication` on the primary (before outage) for `sent_lsn` vs `write_lsn`. Post-failover, check `pg_last_wal_receive_lsn` on the new primary to estimate the gap.

SC2-B
1.  **Import Replica:** `terraform import aws_db_instance.promoted_replica <arn_of_promoted_replica>`
2.  **Update State:** `terraform state mv aws_db_instance.old_primary aws_db_instance.promoted_replica` (if using the same resource name).
3.  **Detach Old:** `terraform state rm aws_db_instance.old_primary` (ensure old instance is terminated or detached).
4.  **Update References:** Update Terraform variables to point to the new ARN.
5.  **Apply:** `terraform apply` to sync dependent resources (Security Groups, Route53) to the new instance.

SC2-C
**Architecture:**
1.  **Database:** Use **Aurora Global Database**. It offers synchronous cross-region replication (RPO < 1s) and automatic failover with minimal lag compared to standard RDS.
2.  **Connection Management:** Use **RDS Proxy** instead of PgBouncer. It handles connection pooling centrally and supports failover transparently without client-side retry logic.
3.  **DNS:** Use **Route53 Health Checks** with Failover Routing Policy. Point the primary endpoint to the US endpoint and secondary to EU.
4.  **Application:** Implement exponential backoff retry logic for transient connection errors.
5.  **Recommendation:** Aurora Global DB is recommended over standard RDS because standard RDS cross-region replication is asynchronous and manual promotion risks data loss. Aurora Global DB manages the replication lag and failover logic natively.

SC3-C
**Hardened Pipeline:**
1.  **Image Signing:** Use **Cosign** to sign images upon build. `cosign sign --key cosign.key <image>`.
2.  **Admission Control:** Deploy **Kyverno** or **OPA Gatekeeper** to verify signatures on pod creation.
3.  **Pinning:** Pin base images by **digest** (e.g., `node:20-slim@sha256:...`) in Dockerfiles, not tags.

SC4-A
1.  **Local Sessions:** K8s pods are ephemeral. **Solution:** Move sessions to Redis.
2.  **Local NFS:** K8s pods don't share local filesystems. **Solution:** Use NFS CSI driver with RWX access.
3.  **Oracle DB:** Stateful, on-prem. **Solution:** Keep on-prem or migrate to RDS/Aurora (complex).
4.  **IBM MQ:** Stateful, on-prem. **Solution:** Use managed MQ or Kubernetes Operator for MQ.
5.  **Sticky Sessions:** F5 relies on IP/cookie. **Solution:** Use Kubernetes Ingress with sticky sessions or move to Redis.

SC5-C
**Solution:**
1.  **CQRS:** Separate read/write models. Reconciliation writes to a write model; serving reads from a read replica.
2.  **Read Replica:** Run reconciliation job against a replica to avoid locking production tables.
3.  **Bulkheads:** Configure separate connection pools for serving vs. background jobs.
4.  **PDBs:** Set `minAvailable` for serving pods to ensure they aren't evicted during maintenance.
5.  **Scheduling:** Move CronJob to off-peak hours or use `concurrencyPolicy: Forbid`.

SC6-C
**Comparison:**
1.  **Direct Connect:** High bandwidth, low latency, dedicated. Best for RPO/RTO. Fits Zero Trust (private link).
2.  **ECMP VPN:** Cheaper, shared bandwidth. Higher latency/jitter.
3.  **Mesh (Tailscale):** E2E encryption, easy setup. Lower throughput than DX.
**Routing:** DX requires BGP peering. VPN uses IPsec tunnels. Mesh uses overlay routing.
**Recommendation:** Upgrade to Direct Connect for core traffic to meet bandwidth needs. Use Mesh for developer access.

SC7-A
**Drift Sources:**
1.  **Manual `tofu apply`:** Bypasses CI/CD. Dangerous because state locking is missing; concurrent runs cause corruption.
2.  **Helm Values Divergence:** Untracked changes lead to config mismatch between environments.
3.  **`kubectl edit`:** Bypasses GitOps. Dangerous because changes are lost on next `apply` or cause state inconsistency.
**State Locking:** Without locking, two engineers running `apply` simultaneously can overwrite each other's changes, leading to resource destruction or configuration loss.

SC8-A
**Diagnosis:**
1.  **Prometheus OOM:** High cardinality (8.2M series). Memory usage scales with series count. Check `prometheus_tsdb_head_series`.

SC10-A
**Architecture:**
1.  **Active-Passive (Velero):** Chicago active, Dallas passive. Velero backups to S3. RPO depends on backup frequency (15m). RTO depends on restore time.
2.  **Active-Active (Istio):** Both sites serve traffic. Requires state sync.
3.  **Stateful Replication:**
    *   **PostgreSQL:** Patroni synchronous replication (Chicago -> Dallas).
    *   **Redis:** Sentinel with cross-region sync (or Redis Enterprise).
    *   **RabbitMQ:** Federation or Shovel.
    *   **MinIO:** Site Replication (S3 compatible).
**RPO/RTO:** Active-Passive with synchronous DB replication meets RPO < 15m. RTO < 1h is achievable with Velero restore. Active-Active is harder for stateful DBs with 12ms latency.

SC10-C
**Failover Sequence:**
1.  **Detection:** Prometheus monitors site health. If Chicago unreachable for 30s, trigger alert.
2.  **DNS:** GSLB updates Route53 to point to Dallas.
3.  **DB:** Patroni promotes Dallas replica (synchronous commit ensures no data loss).
4.  **Cache:** Redis Sentinel promotes Dallas master.
5.  **Queue:** RabbitMQ federation syncs pending messages.
6.  **Storage:** MinIO site replication catches up.
7.  **State:** OpenTofu state updated to reflect Dallas as primary.
**Split-Brain:** Use a quorum-based DCS (etcd) in Patroni. If Chicago is down, Dallas cannot promote unless it has quorum.
**Failback:** Once Chicago recovers, demote Dallas, sync data, and switch DNS back.
