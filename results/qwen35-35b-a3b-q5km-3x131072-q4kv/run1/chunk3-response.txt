T1 — **`tofu plan`** performs a dry run, calculating the execution plan and displaying the differences (add, change, destroy) without making changes. **`tofu apply`** executes the plan, creating, updating, or destroying resources to match the configuration.

T2 — **State** is a JSON file mapping OpenTofu resources to real-world infrastructure. **Remote state** is crucial for team collaboration (sharing state), security (storing secrets away from local machines), and state locking (preventing concurrent writes).

T3 — **`count`** uses integer indices; changing the list order can cause resource recreation. **`for_each`** uses map/set keys; it is preferred when the set of items is dynamic or keys are stable, as it avoids index shifting issues and allows for more readable resource addressing.

T4 — OpenTofu detects this via **state drift**. Running `tofu plan` compares the state file against the real infrastructure. If drift is found, OpenTofu plans to revert the resource to match the configuration (unless `ignore_changes` is set).

T5 —
```hcl
resource "aws_instance" "web" {
  for_each = toset(["web1", "web2", "web3"])
  ami      = "ami-12345678"
  instance_type = "t2.micro"
  tags = {
    Name = each.key
  }
}
```

T6 — `terraform_remote_state` reads the entire state file from a backend, creating tight coupling. **Output values** via `module` blocks are explicit interfaces. The recommended alternative is using **`module` blocks** to consume outputs from other root modules (if in the same plan) or refactoring to modules, as it enforces a contract and reduces state file coupling.

T7 — The `null_resource` provisioner runs during the apply phase, but the output might be evaluated before the provisioner finishes writing the file. **Fix:** Add `depends_on = [null_resource.kubeconfig]` to the dependent resource, or use a `local_file` resource to write the kubeconfig and read from that file path in the output.

T8 — `terraform import` is a CLI command to import existing resources into state manually. The **`import` block** (1.5+) is declarative, version-controlled, and allows OpenTofu to plan the import as part of the normal workflow, reducing manual steps and errors.

T9 — A **variable** is an input parameter for the configuration. An **output** is a value exposed by the configuration for other modules or users to consume. Variables are inputs; outputs are results.

T10 — A **provider** is a plugin that interacts with a specific cloud or service (e.g., AWS, Azure). If you run `tofu plan` without `tofu init`, OpenTofu fails because it hasn't downloaded the required provider plugins yet.

T11 — The `.terraform.lock.hcl` file contains **provider version constraints and checksums**. It should be committed to version control to ensure all team members and CI pipelines use the exact same provider versions, preventing unexpected behavior.

T12 — `tofu destroy` removes all resources managed by the configuration. To target a single resource: `tofu destroy -target=aws_instance.example`.

T13 — A **data source** reads data from an external system without creating or managing it. Use one instead of a managed resource when you need to look up existing data (e.g., `aws_ami` to find the latest Ubuntu image) rather than creating new infrastructure.

T14 — **Variables** are inputs passed from outside (CLI, files). **Locals** are computed values defined within the configuration for reuse. Use `locals` for intermediate calculations or complex expressions that don't need to be exposed as inputs.

T15 — The **lifecycle block** controls resource behavior.
*   `create_before_destroy`: Creates new resource before destroying old (zero downtime).
*   `prevent_destroy`: Prevents accidental deletion (safety).
*   `ignore_changes`: Ignores drift in specific attributes (e.g., tags managed manually).

T16 — The **`moved` block** allows you to change a resource's address (e.g., module path) in the configuration without destroying and recreating it. OpenTofu maps the old state address to the new one, preserving the resource.

T17 — The **dependency graph** determines execution order. **Implicit dependencies** are inferred from references (e.g., `aws_instance.web.id`). **Explicit dependencies** use `depends_on`. `depends_on` forces serialization, reducing parallelism and slowing down apply, so it should be used only when implicit references are insufficient.

T18 — Strategies to reduce plan time:
1.  **`-refresh=false`**: Skips state refresh (faster, but may miss drift).
2.  **Split State Files**: Modularize to reduce state file size and API calls.
3.  **Prune State**: Remove unused resources from state to reduce graph size.
4.  **`-target`**: Plan only specific resources if changes are isolated.

T19 — State locking prevents concurrent modifications. If a lock is not released (e.g., crash), subsequent commands fail. **Break lock:** `tofu force-unlock <LOCK_ID>`. **Risk:** Data corruption if two processes write simultaneously.

T20 — Process:
1.  Backup local state file.
2.  Configure backend in `backend "s3" { ... }` with DynamoDB lock.
3.  Run `tofu init -migrate-state` to move state to S3.
4.  Run `tofu plan` to verify state is accessible.
5.  Verify state in S3 console/DynamoDB.

A1 — A **Playbook** is a YAML file defining a sequence of plays. A **Role** is a reusable directory structure for organizing tasks, vars, and templates. A **Task** is a single action (e.g., install a package) within a play.

A2 — `ansible_facts` contains gathered system information. Access OS family via: `{{ ansible_facts['ansible_os_family'] }}`.

A3 — **`command`**: Executes without shell (no pipes/redirection). **`shell`**: Executes with shell (supports pipes/redirection). **`raw`**: Executes without setup (no Python module transfer), useful for early bootstrapping.

A4 — **Idempotency** means running a task multiple times produces the same result. **Idempotent:** `apt: name=nginx` (installs only if missing). **Not Idempotent:** `shell: echo "1" > /tmp/test` (overwrites file every time).

A5 —
```yaml
- name: Install htop
  package:
    name: htop
  when: ansible_os_family == "Debian"

- name: Install htop on RedHat
  package:
    name: htop
  when: ansible_os_family == "RedHat"
  # Add epel-release task separately if needed
```

A6 — **Precedence (High to Low):** Command line extra vars > Play vars > Role vars > Host vars > Group vars > Role defaults. `set_fact` overrides most at runtime.

A7 — Use **`serial: 3`** in the play to limit concurrency. Group nodes into `agents` and `workers`. Use `wait_for` in a pre-task to ensure the first 3 nodes are ready before the rest start, or rely on `serial` to batch them.

A8 — **`ansible-vault`** encrypts sensitive data. To encrypt a single variable: `ansible-vault encrypt_string 'secret_value' --name 'my_secret'`.

A9 — An **inventory** lists hosts and groups. **Static inventory** is a file (INI/YAML). **Dynamic inventory** is a script/API that generates inventory dynamically (e.g., from cloud metadata).

A10 — `ansible webservers -m shell -a "df -h"`.

A11 — A **handler** is a task that runs only when notified. It differs from a regular task by running after all tasks in a play complete. Triggered via `notify: handler_name` in a task.

A12 — **`ansible.cfg`** configures Ansible behavior. Common settings: `inventory`, `remote_user`, `forks` (parallelism), `host_key_checking`.

A13 — **`register`** saves a task's output to a variable. Use it in subsequent tasks via `when: result.stdout == "success"` or `debug: var=result`.

A14 — **`include_tasks`** is evaluated at runtime (dynamic). **`import_tasks`** is evaluated at compile time (static). Use `import` for performance and `include` for conditional loading.

A15 — **Ansible Galaxy** is a repository for roles and collections. Use `requirements.yml` to list dependencies and run `ansible-galaxy install -r requirements.yml`.

A16 — **Connection plugins** manage how Ansible connects. **`ssh`**: Remote Linux/Unix. **`local`**: Control node itself. **`network_cli`**: Network devices (Cisco, Juniper).

A17 — **`delegate_to`** runs a task on a different host. **`run_once`** runs a task once even in a loop. Example: Gather facts from remote hosts, then write a config file to `localhost` (control node) once.

A18 — Strategies to improve performance:
1.  **`forks`**: Increase parallelism (e.g., `forks: 50`).
2.  **`pipelining`**: Reduces SSH connections (faster).
3.  **`fact_caching`**: Caches gathered facts to avoid re-gathering.
4.  **`mitogen`**: Optimizes SSH transport (reduces overhead).

A19 — **`{{ }}`**: Expressions (variables). **`{% %}`**: Control structures (loops/if). **`{# #}`**: Comments.
Snippet:
```jinja2
upstream backend_servers {
{% for server in backend_servers %}
    server {{ server.ip }} weight={{ server.weight }}; # Health check: {{ server.health_check }}
{% endfor %}
}
```

A20 — **Rolling Deploy:**
1.  Set `serial: 1` (or batch size).
2.  **Pre-task:** Deregister from load balancer (e.g., AWS ELB).
3.  **Deploy:** Update app.
4.  **Health Check:** Use `uri` module to verify endpoint.
5.  **Post-task:** Register with load balancer.
6.  **Rollback:** Use `block/rescue` to revert changes if health check fails.
