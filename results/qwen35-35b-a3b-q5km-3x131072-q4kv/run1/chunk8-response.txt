CL1 — **IaaS (Infrastructure as a Service)** provides virtualized computing resources (servers, storage, networking). Example: AWS EC2. **PaaS (Platform as a Service)** provides hardware and software tools (runtime, database) for developers. Example: Google App Engine. **SaaS (Software as a Service)** delivers software over the internet. Example: Microsoft 365.

CL2 — **Shared Responsibility Model:** Provider secures the cloud (hardware, hypervisor); Customer secures what's in the cloud. **IaaS VM:** Provider secures physical host; Customer secures OS, patching, config. **Managed Kubernetes:** Provider secures control plane/infrastructure; Customer secures nodes, workloads, network policies. **SaaS (M365):** Provider secures app/infra; Customer secures identity, data, access policies.

CL3 — **VPC:** Isolated virtual network. **Subnet:** IP range within VPC. **Security Group:** Instance-level firewall (stateful). **NACL:** Subnet-level firewall (stateless). **Layering:** VPC contains Subnets; Subnets contain Instances protected by SGs; Subnets are filtered by NACLs. **Stateful:** Allows return traffic automatically (SG). **Stateless:** Requires explicit rules for return traffic (NACL).

CL4 — **Landing Zone:** Secure, compliant foundation. **OUs:** Group accounts by function/environment. **Guardrails:** Policy constraints (e.g., no public S3). **Shared Networking:** Central hub for inter-VPC traffic. **Centralized Logging:** Aggregated audit trails. **Single-Account Problems:** High blast radius, billing opacity, inability to enforce least privilege across teams, compliance audit difficulty.

CL5 — **IaC:** Managing infrastructure via code. **Terraform/OpenTofu:** Multi-cloud, declarative, state file management. **Pulumi:** Multi-cloud, imperative (general purpose languages). **CloudFormation/Bicep:** AWS/Azure native, tightly integrated. **Crossplane:** IaC via Kubernetes API. **Choice:** Terraform for multi-cloud portability; CloudFormation for AWS-native simplicity; Pulumi for dev-friendly languages; Crossplane for K8s-centric workflows.

CL6 — **HPA:** Scales pods based on CPU/Memory. **VPA:** Scales pod resources (CPU/RAM) vertically. **Cluster Autoscaler:** Adds/removes nodes. **KEDA:** Event-driven scaling (e.g., queue length). **Use KEDA over HPA:** When scaling is triggered by external events (Kafka, SQS) rather than internal resource metrics.

CL7 — **DR Strategy:** Define RPO (data loss tolerance) and RTO (downtime tolerance). **Active-Active:** Both regions serve traffic (low RTO, high cost). **Active-Passive:** One active, one standby (higher RTO, lower cost). **DB:** Sync (RPO=0, latency) vs Async (RPO>0, lower latency). **DNS:** Route53 failover routing. **State:** Externalize state (S3, RDS). **Cost:** Active-Active doubles infra cost. **Pilot Light:** Minimal infra running. **Warm Standby:** Scaled-down replica. **Multi-site:** Full active-active.

CL8 — **GitOps:** Git as single source of truth for infra/app state. **ArgoCD:** Push-based (syncs cluster to Git). **Flux:** Pull-based (cluster pulls Git). **Secrets:** Sealed Secrets (encrypted in Git), External Secrets Operator (fetch from Vault/AWS Secrets Manager). **Drift:** GitOps controller detects manual changes and reverts them to match Git state.

CL9 — **Architecture:** Use **StatefulSet** for stable network IDs. **PV/PVC:** Use `local` storage class for low latency (hostPath). **Headless Service:** For stable DNS names. **Init Containers:** For config file generation before main app starts. **ConfigMaps:** Mount as volumes for specific paths. **PDB:** Ensure minimum available pods during updates to prevent data loss.

CL10 — **SA:** K8s identity for pods. **Workload Identity:** Maps K8s SA to cloud identity (GCP/AWS). **IRSA:** AWS OIDC federation for SAs. **Mounting Creds:** Bad practice (leakage, rotation difficulty, stale keys). **Federation:** Uses short-lived OIDC tokens, no long-term secrets, automatic rotation.

CL11 — **Object (S3):** Unstructured data, high durability (11 9s), low latency. **Block (EBS):** Attached storage, OS/DB, high IOPS, lower durability than object. **File (EFS):** Shared file system, NFS/SMB, moderate IOPS. **Use Object:** Backups, media. **Use Block:** Databases, OS. **Use File:** Shared configs, home dirs.

CL12 — **Zero Trust:** Verify every request. **Network Policies:** Restrict pod-to-pod traffic. **Service Mesh:** mTLS for L7 encryption. **Pod Identity:** Least privilege access. **Admission Controllers:** Enforce policies (OPA/Kyverno). **Image Signing:** Verify integrity. **Runtime Security:** Detect anomalies (Falco). **Differs:** Traditional assumes internal trust; Zero Trust assumes breach and verifies continuously.

CL13 — **CDN:** Distributed network caching content at edge. **Services:** AWS CloudFront, Azure CDN. **Use CDN:** Static assets, global latency reduction, DDoS protection. **Use Origin:** Dynamic content, real-time data, low traffic.

CL14 — **Region:** Geographic area. **AZ:** Isolated data center within region. **Deploy across AZ:** Fault tolerance (if one AZ fails, others survive). **Zonal:** Resource tied to one AZ (e.g., EBS). **Regional:** Resource spans AZs (e.g., ELB, RDS Multi-AZ).

CL15 — **Managed DB:** Service like RDS. **Provider Handles:** Patching, backups, failover, hardware, OS updates. **You Handle:** Schema, queries, user access, data. **Benefit:** Reduced operational overhead, higher availability.

CL16 — **IAM:** **Users:** Human identities. **Groups:** Collections of users. **Roles:** Temporary access (cross-account/service). **Policies:** JSON permissions. **Least Privilege:** Grant only necessary permissions. **Apply:** Use roles for services, groups for users, deny-by-default policies.

CL17 — **Serverless:** Event-driven execution, no server mgmt. **Cold Starts:** Latency during first invocation after inactivity. **Matter:** Affects real-time performance. **Poor Fit:** Long-running processes, stateful apps, predictable high-throughput workloads.

CL18 — **Ingress:** L7 routing rules. **Gateway API:** Standardized, extensible L7/L4. **LB:** L4 load balancing. **Mesh:** L7 service-to-service. **TLS:** Ingress (terminates at LB), Mesh (terminates at sidecar). **Use Gateway API:** For complex routing, multi-cluster, or when Ingress limitations hinder requirements.

CL19 — **Strategy:** **RI/SP:** Commit to usage for discount. **Spot:** Interruptible for fault-tolerant workloads. **Right-sizing:** Match instance to actual usage. **Tiering:** Move old data to Glacier. **Idle Detection:** Turn off dev envs nights/weekends. **FinOps:** Culture of cost accountability. **Tools:** Cost Explorer, CloudHealth. **Showback:** Charge teams for usage to drive behavior.

CL20 — **Strategy:** **Abstraction:** K8s (EKS/AKS), Terraform (stateless). **Networking:** Direct Connect/ExpressRoute. **Identity:** OIDC federation. **Trade-offs:** Multi-cloud increases complexity, latency, and cost. **Justified:** Regulatory compliance, catastrophic redundancy, vendor leverage. **Unnecessary:** If single cloud meets SLAs and cost is lower.

OT1 — **Purdue Model:** ISA-95 hierarchy. **Level 0:** Physical Process (Sensors). **Level 1:** Basic Control (PLCs). **Level 2:** Supervisory Control (SCADA/HMI). **Level 3:** Operations (MES). **Level 4:** Enterprise (ERP).

OT2 — **IT:** Data confidentiality, patching, uptime. **OT:** Safety, availability, legacy hardware. **Patching:** OT patching risks downtime or safety failure; legacy OS often unsupported; vendor support required.

OT3 — **DMZ (IDMZ):** Buffer between IT and OT. **Traffic:** Historian pulls from OT, pushes to IT. **Direction:** OT to IT (push), IT to Historian (pull). **Direct L4-L1:** Dangerous because it bypasses security controls, allowing direct manipulation of safety-critical devices.

OT4 — **Modbus TCP:** L1-2 (Control). **OPC UA:** L2-3 (Data exchange). **EtherNet/IP:** L1-2 (Control). **PROFINET:** L1-2 (Control). **MQTT:** L3-4 (Cloud/Edge).

OT5 — **Data Diode:** Hardware enforcing one-way data flow. **Firewall:** Two-way filtering. **Use Diode:** When data must leave OT but nothing can enter (e.g., telemetry). **Limitation:** No feedback loop, no remote configuration from outside.

OT6 — **Flow:** PLC -> SCADA -> Historian -> Edge Gateway -> MQTT Broker -> Cloud. **Security:** Firewall between SCADA and Edge. Encryption (TLS) for MQTT. Edge Gateway authenticates to Cloud.

OT7 — **Zones:** Logical grouping of assets with similar security requirements. **Conduits:** Controlled paths between zones. **Model:** Zone 4 (Corp), Zone 3 (Eng/SCADA), Zone 2 (PLC/Control), Zone 1 (SIS). Conduits between zones require firewalls/DMZs.

OT8 — **Segmentation:** VLANs per production line. Firewalls between VLANs and DMZ. **Switch:** Managed Layer 3 switches. **Remote Access:** Jump host in DMZ, MFA, session recording. **Transition:** Phased rollout, parallel testing, backup configs, vendor coordination.

OT9 — **SIS:** Independent safety system (stops process safely). **BPCS:** Basic control (maintains process). **IEC 61511:** Safety lifecycle standard. **Isolation:** Prevents cyber attacks from disabling safety functions. **Ransomware on SIS:** Physical damage, injury, or inability to shut down safely.

OT10 — **PLC:** Logic controller for machines. **DCS:** Distributed control for continuous processes. **SCADA:** Supervisory monitoring. **Coexist:** Yes, PLCs often feed SCADA; DCS may contain PLCs.

OT11 — **Architecture:** Jump host in DMZ. MFA required. Session recording enabled. Time-limited credentials. **Direct RDP:** Unacceptable because it exposes HMI to IT network threats, allowing lateral movement to control devices.

OT12 — **Plan:** Isolate compromised segment (network quarantine). Verify physical safety (manual ops). Inventory assets. Monitor IDS for lateral movement. **Operator Role:** Maintain physical safety, manual control if automation fails, coordinate with IT/Security.

OT13 — **HMI:** Interface for operators to view/process data. **Eng Workstation:** Used by engineers to configure/program logic. **Diff:** HMI is read-only for ops; Eng is write-access for config.

OT14 — **Historian:** Time-series database for process data. **Importance:** Trend analysis, compliance, troubleshooting. **Plant (L3):** Real-time ops data. **Enterprise (L4):** Business analytics, long-term storage.

OT15 — **Ladder Logic:** Graphical programming language mimicking relay circuits. **Device:** PLC. **Dominant:** Safety certification, deterministic execution, legacy familiarity, ease of maintenance for electricians.

OT16 — **RTU:** Remote Terminal Unit. **Diff:** RTU optimized for comms/telemetry in harsh environments; PLC optimized for local logic. **RTU Use:** Pipelines, remote substations, long-distance comms.

OT17 — **Analog:** Continuous signal (voltage/current). **Digital:** Discrete (on/off). **4-20mA:** Preferred because 4mA indicates live wire (vs 0mA fault), allowing detection of wire breaks.

OT18 — **Inventory:** Foundation for security (know what to protect). **Tools:** Passive (packet sniffing), Active (scanning). **Active Scanning:** Dangerous because legacy OT devices may crash or reboot when probed.

OT19 — **MITRE ATT&CK ICS:** Framework for ICS threats. **Tactics:** Initial Access (Phishing), Execution (Command Line), Persistence (Registry), Privilege Escalation (Exploit), Defense Evasion (Obfuscation). **Diff:** Focuses on physical impact/safety. **Use:** Map controls to tactics to find gaps.

OT20 — **SOC:** Passive log collection (SPAN ports). SIEM integration. OT-specific rules (e.g., PLC write attempts). **Staffing:** OT engineers for triage. **False Positives:** Tuned to avoid production stoppages. **Firmware Alert:** Isolate device, verify safety, rollback, investigate source, patch vendor.
