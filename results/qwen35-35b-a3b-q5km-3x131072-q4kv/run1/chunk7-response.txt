AA1 — **Monolith vs. Microservices:** A monolith is a single, unified codebase deployed as one unit. Microservices decompose this into independent, loosely coupled services deployed separately.
**3 Problems Microservices Introduce:**
1.  **Distributed Complexity:** Network latency, partial failures, and eventual consistency issues replace local function calls.
2.  **Operational Overhead:** Requires orchestration (Kubernetes), service discovery, and complex CI/CD pipelines for multiple services.
3.  **Data Consistency:** Maintaining ACID transactions across services requires complex patterns (Sagas) rather than simple database transactions.

AA2 — **Sync vs. Async:** Synchronous communication blocks the sender until the receiver responds (request/response). Asynchronous communication allows the sender to continue without waiting (fire-and-forget or callback).
**Examples:** Synchronous: REST (HTTP) or gRPC. Asynchronous: RabbitMQ (Message Queue) or Apache Kafka (Event Stream).

AA3 — **API Gateway vs. Reverse Proxy:** A reverse proxy routes traffic to backend servers (L4/L7). An API Gateway adds business logic like authentication, rate limiting, and API versioning.
**Replacement:** An API Gateway *can* replace a reverse proxy functionally as it includes routing capabilities. However, in large architectures, a dedicated ingress controller (reverse proxy) often sits in front of the gateway to handle SSL termination and global routing before the gateway applies business logic.

AA4 — **Saga Pattern:** A sequence of local transactions where each updates data within a service. If a step fails, compensating transactions undo previous changes.
**Choreography:** Services emit events; other services listen and react. *Appropriate:* Simple flows, low coupling.
**Orchestration:** A central coordinator tells services what to do. *Appropriate:* Complex flows, need for centralized visibility and error handling.

AA5 — **CQRS:** Separates read (Query) and write (Command) operations into different models.
**Pairing with Event Sourcing:** Event Sourcing stores state changes as an immutable log. CQRS uses this log to project read models.
**Problem Solved:** Allows independent scaling of read/write workloads and provides a complete audit trail/history of state changes, which traditional CRUD (current state only) does not.

AA6 — **Circuit Breaker:** Prevents repeated calls to a failing service.
**States:** Closed (normal), Open (failures detected, requests blocked), Half-Open (testing if service recovered).
**Vs. Retry:** Retry attempts to fix transient failures; Circuit Breaker stops hammering a dead service.
**Use Together:** Use retry for transient network blips; use Circuit Breaker when the service is persistently down to prevent cascading failures.

AA7 — **Multi-Tenant Isolation:**
1.  **Shared DB/Schema:** Lowest cost, highest noisy-neighbor risk, complex row-level security logic.
2.  **Shared DB/Schema per Tenant:** Moderate cost, better isolation (schema level), moderate ops complexity (schema migrations).
3.  **Separate DB per Tenant:** Highest cost, best security/isolation, highest ops complexity (backup/migration per tenant).

AA8 — **Consistency:** Strong consistency ensures all nodes see the same data immediately. Eventual consistency allows temporary divergence.
**Architecture:** Use **Strong Consistency**.
**Data Store:** Relational DB (e.g., PostgreSQL) with row-level locking or a distributed lock manager (e.g., Redis Redlock).
**Pattern:** Pessimistic locking (lock row on read) or Optimistic locking (version check on commit) to prevent overselling.

AA9 — **Sidecar Proxy:** A companion container running alongside the main app to handle network traffic.
**Envoy/Istio:** Envoy handles the data plane (traffic routing/mTLS). Istio Control Plane configures Envoy.
**mTLS:** Application-level TLS secures the app-to-client channel. mTLS between sidecars secures service-to-service communication transparently, ensuring identity verification and encryption without code changes.

AA10 — **Strangler Fig:** Incrementally replace legacy functionality with new services.
**Strategy:** Place a reverse proxy in front of the legacy .NET monolith. Route specific URL paths (e.g., `/api/new`) to new .NET 8 microservices. Gradually migrate features. Once all traffic is routed to new services, decommission the monolith.

AA11 — **12-Factor App:**
1.  **Codebase:** One codebase tracked in version control.
2.  **Dependencies:** Explicitly declared and isolated.
3.  **Config:** Stored in environment variables.
4.  **Backing Services:** Treated as attached resources.
5.  **Build/Release/Run:** Strictly separated stages.
6.  **Processes:** Stateless, stateless processes.
7.  **Port Binding:** Expose services via port binding.
8.  **Concurrency:** Scale via process model.
9.  **Disposability:** Fast startup/shutdown.
10. **Dev/Prod Parity:** Keep environments similar.
11. **Logs:** Treat logs as event streams.
12. **Admin Processes:** Run as one-off commands.

AA12 — **Event-Driven (Kafka) vs. Request-Driven (REST):**
*   **Coupling:** Event-driven is loosely coupled; Request-driven is tightly coupled.
*   **Scalability:** Event-driven scales better for high throughput; Request-driven scales linearly with load.
*   **Debugging:** Request-driven is easier to trace; Event-driven requires distributed tracing.
*   **Ordering:** Kafka guarantees ordering per partition; REST does not guarantee global order.
*   **Failure:** Event-driven uses retries/compensation; Request-driven uses timeouts.
*   **Choice:** Use Event-driven for decoupled, high-volume data pipelines. Use Request-driven for real-time user interactions requiring immediate feedback.

AA13 — **Load Balancer:** Distributes traffic across servers.
**L4 vs. L7:** L4 (Transport) routes based on IP/Port. L7 (Application) routes based on HTTP headers/URL.
**Use Case:** L7 is required for SSL termination or routing `/api` to backend A and `/static` to backend B.

AA14 — **Message Queue vs. Log Broker:**
*   **MQ (RabbitMQ):** Messages are consumed once and deleted. Good for task processing.
*   **Log Broker (Kafka):** Messages are retained and replayable. Good for event streaming/audit.
*   **Pick:** Use MQ for fire-and-forget tasks. Use Kafka for event sourcing or replaying historical data.

AA15 — **Container vs. VM:** A container shares the host OS kernel and is lightweight. A VM virtualizes hardware and includes a full guest OS.
**Popularity:** Containers are faster to start, more resource-efficient, and portable, fitting microservices' need for rapid scaling and deployment.

AA16 — **Stateless vs. Stateful:** Stateless services store no session data (e.g., Web Server). Stateful services store data locally (e.g., Database).
**Scaling:** Stateless services scale horizontally easily because any instance can handle any request. Stateful services require data replication or sharding to scale.

AA17 — **ORM:** Maps database tables to programming language objects.
**Examples:** Entity Framework (.NET), Hibernate (Java).
**Trade-offs:** ORMs increase developer productivity but can generate inefficient SQL and hide database-specific optimizations.

AA18 — **Bulkhead:** Isolates resources (threads, connections) for specific services.
**Implementation:** Use separate thread pools for each downstream dependency. If one service hangs, it only exhausts its specific pool, not the application's total thread count.

AA19 — **OT vs. CRDT:**
*   **OT:** Server authoritative, complex conflict resolution, high latency for offline.
*   **CRDT:** Mathematically guaranteed convergence, client authoritative, low latency.
*   **Choice:** **CRDTs** for offline support. They allow clients to edit locally and merge automatically without server intervention, ensuring consistency upon reconnection.

AA20 — **Distributed Tracing:**
*   **Context Propagation:** Pass Trace ID/Parent ID via headers (W3C/B3).
*   **Sampling:** Head-based (random at start) reduces storage; Tail-based (based on latency/errors) captures failures.
*   **Storage:** Jaeger/Tempo store spans for querying.
*   **Kafka:** Inject trace context into message headers. Consumers extract and continue the trace chain to ensure end-to-end visibility.

OP1 — **Hypervisor Type 1 (Bare-metal):** Runs directly on hardware (e.g., VMware ESXi, Microsoft Hyper-V Server).
**Hypervisor Type 2 (Hosted):** Runs on top of an OS (e.g., VMware Workstation, VirtualBox).
**Difference:** Type 1 has better performance and security for production; Type 2 is for development/testing.

OP2 — **SAN vs. NAS:**
*   **SAN:** Block storage (LUNs). High performance. Protocol: iSCSI, Fibre Channel. Use for Databases.
*   **NAS:** File storage (Files/Folders). Easier sharing. Protocol: NFS, SMB/CIFS. Use for File Shares, Home Directories.

OP3 — **3-Tier:** Core (Backbone), Distribution (Policy), Access (Edge).
**Leaf-Spine:** All-to-all connectivity between leaf (access) and spine (core).
**Problem Solved:** Eliminates oversubscription and reduces latency compared to 3-tier, which has bottlenecks at the distribution layer.

OP4 — **Proxmox HA:**
*   **Architecture:** 3+ nodes for quorum.
*   **Quorum:** Majority vote required to form cluster.
*   **Fencing:** STONITH (e.g., DRAC/iLO) to power off failed nodes.
*   **Storage:** Ceph (distributed) or Shared iSCSI/NFS.
*   **Failure:** HA manager detects node down, fences it, and restarts VMs on surviving nodes.

OP5 — **VLAN Trunking:** Carries multiple VLAN traffic over one link using 802.1Q tags.
**Linux Config:** Create sub-interfaces. `ip link add link eth0 name eth0.10 type vlan id 10`. Assign IP to `eth0.10`.

OP6 — **RAID:**
*   **RAID 1:** Mirroring (50% capacity).
*   **RAID 5:** Striping + Parity (N-1 capacity).
*   **RAID 6:** Striping + Double Parity (N-2 capacity).
*   **RAID 10:** Mirroring + Striping (50% capacity, high perf).
*   **RAID Z2:** ZFS double parity.
*   **Choice:** **RAID 10** for 6 drives. Best balance of read/write performance and fault tolerance (can survive 2 drive failures if not in same mirror) for a DB.

OP7 — **DNS Architecture:**
*   **Internal:** AD DNS servers (Active Directory integrated).
*   **External:** Public authoritative servers (e.g., Route53).
*   **Split-Horizon:** Internal clients get internal IPs; external get public IPs.
*   **Forwarders:** Internal DNS forwards unknown queries to ISP/Cloud.
*   **Laptop:** Connects via VPN -> DNS tunnel -> Resolves internal names via internal forwarders.

OP8 — **PKI Chain:** Root CA (Offline, trust anchor) -> Subordinate CA (Issuing) -> End Entity.
**Revocation:** CRL (List) vs OCSP (Real-time check).
**Compromise:** Root CA compromised = All trust revoked (catastrophic). Subordinate CA compromised = Revoke subordinate cert, issue new subordinate, reissue end entities.

OP9 — **WAN Consolidation:**
*   **MPLS:** High cost, high reliability, low latency.
*   **IPsec:** Low cost, variable latency, lower security.
*   **SD-WAN:** Dynamic path selection, cost-effective, manageable.
*   **WireGuard:** Simple, encrypted, mesh.
*   **Recommendation:** **SD-WAN**. Handles VoIP QoS, optimizes path selection over internet, lower cost than MPLS, manageable for 50 users/site.

OP10 — **iDRAC/iLO/IPMI:** Out-of-band management interfaces.
**Critical:** Allows server control (power on/off, console) even if OS is down.
**Exposure:** If exposed to internet, attackers can gain root access to the hardware (IPMI vulnerability), bypassing OS security.

OP11 — **LVM vs. ZFS:**
*   **LVM:** Volume management only. Snapshots are copy-on-write.
*   **ZFS:** Filesystem + Volume Manager + RAID. Snapshots are instant, low overhead.
*   **Use:** LVM for simple Linux volume management. ZFS for data integrity, snapshots, and built-in RAID (RAID-Z).

OP12 — **Stretch Cluster:**
*   **Design:** 2 sites + 1 Tiebreaker (Quorum Disk/Cloud).
*   **Ceph:** CRUSH rules must distribute OSDs across sites.
*   **Latency:** <5ms recommended for monitors.
*   **Split-Brain:** If network fails, both sites think they are primary.
*   **Tiebreaker:** Needed to ensure only one site remains active to prevent data corruption.

OP13 — **DHCP DORA:** Discover, Offer, Request, Acknowledge.
**Unavailable Server:** Client retries. If lease expires and no server found, Windows assigns APIPA address (169.254.x.x) and loses network connectivity.

OP14 — **Managed vs. Unmanaged:**
*   **Unmanaged:** Plug-and-play.
*   **Managed:** Configurable.
*   **Features:** VLANs, QoS (prioritize VoIP), Port Security, SNMP monitoring, ACLs.

OP15 — **NTP:** Network Time Protocol.
**Critical:** Certificates fail if time drifts > tolerance. Cluster consensus (Pacemaker) fails if clocks differ. Logs become uncorrelatable.

OP16 — **Physical vs. VM vs. Container:**
*   **Physical:** Bare metal.
*   **VM:** Virtualized OS.
*   **Container:** Virtualized App.
*   **Bare Metal Use:** High-performance DBs, security compliance (no hypervisor layer), hardware passthrough (GPUs).

OP17 — **UPS vs. Generator:**
*   **UPS:** Short-term battery power (minutes).
*   **Generator:** Long-term power (hours/days).
*   **ATS:** Automatic Transfer Switch switches power from Grid -> UPS -> Generator seamlessly.

OP18 — **Active-Passive vs. Active-Active:**
*   **Active-Passive:** One node handles load; standby takes over on failure.
*   **Active-Active:** Both handle load.
*   **PostgreSQL HA:** Pacemaker/Corosync cluster. VIP floats between nodes. STONITH (fencing) ensures old master doesn't write during failover.

OP19 — **Backup/Recovery:**
*   **3-2-1:** 3 copies, 2 media types, 1 offsite.
*   **RPO/RTO:** Recovery Point/Time Objective.
*   **Immutable:** WORM storage prevents ransomware deletion.
*   **Test:** Regular restore drills.
*   **Backup vs. DR:** Backup is the data copy; DR is the strategy/process to restore operations.

OP20 — **802.1X:**
*   **Roles:** Supplicant (Client), Authenticator (Switch), Server (RADIUS).
*   **EAP:** EAP-TLS (Certificate auth, secure); PEAP (Username/Password, tunneled).
*   **Dynamic VLAN:** Switch assigns VLAN based on RADIUS response.
*   **Failure:** Port blocked or placed in Guest VLAN.
*   **MAB:** Switch sends MAC address to RADIUS as credentials for devices that don't support 802.1X.
