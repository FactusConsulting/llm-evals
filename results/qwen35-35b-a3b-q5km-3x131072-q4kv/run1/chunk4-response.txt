G1 — Easy: An array is a fixed-size value type `[T]N`. A slice is a dynamic view over an array, defined as a struct containing a pointer to data, length, and capacity (`[T]`). When appending to a slice beyond its capacity, Go allocates a new underlying array (usually doubling capacity), copies data, and returns a new slice header pointing to it.

G2 — Easy: `var x int` declares a variable with the zero value (0) and requires a type. `x := 0` is a short variable declaration that infers the type. You cannot use `:=` if the variable already exists in the current scope (redeclaration error) or outside a function (must use `var`).

G3 — Medium: A goroutine is a lightweight thread managed by the Go runtime, not the OS. It uses less memory and context-switching overhead than OS threads. The Go scheduler (GMP model) maps Goroutines (G) to OS Threads (M) via Processors (P). The scheduler multiplexes Gs onto Ms, handling work stealing and preemption to maximize CPU utilization.

G4 — Medium: Channels are typed conduits for communication between goroutines. Unbuffered channels block both sender and receiver until the other side is ready. Buffered channels allow sending without blocking up to their capacity. Sending to a full buffered channel blocks until a receiver reads from it.

G5 — Medium: The `context` package manages deadlines, cancellation signals, and request-scoped values across API boundaries.
```go
ctx, cancel := context.WithTimeout(context.Background(), 5*time.Second)
defer cancel()
req, _ := http.NewRequestWithContext(ctx, "GET", url, nil)
resp, err := client.Do(req) // Cancels if timeout exceeded
```

G6 — Medium: Go interfaces are satisfied implicitly (duck typing); a type implements an interface by implementing its methods. Unlike Java/C#, you don't declare implementation. The empty interface `interface{}` (or `any`) accepts any value. It is powerful for generic containers but dangerous because it requires type assertions, losing compile-time safety.

G7 — Hard: This is unsafe due to a data race: one goroutine writes `data` and `ready`, while the main goroutine reads them without synchronization. The compiler/CPU may reorder operations. Fix using `sync.Once` for initialization or `sync/atomic` for the flag:
```go
var data string
var ready atomic.Bool
go func() {
    data = "hello"
    ready.Store(true)
}()
for !ready.Load() {}
fmt.Println(data)
```

G8 — Medium: `sync.Mutex` provides exclusive access. `sync.RWMutex` allows multiple readers or one writer. `sync.Map` is optimized for sparse key sets or read-heavy workloads where keys are rarely deleted. Trade-off: `sync.Map` has higher overhead per access than a mutex-protected map but avoids lock contention for reads.

G9 — Medium: Go modules manage dependencies. `go.mod` declares the module path and direct dependencies. `go.sum` stores checksums for integrity. `go mod tidy` removes unused dependencies and adds missing ones. MVS (Minimum Version Selection) selects the lowest compatible version of a dependency that satisfies all requirements in the dependency tree.

G10 — Medium: `defer` schedules a function call to run after the surrounding function returns. Multiple defers execute in LIFO (Last In, First Out) order. A common gotcha in loops: `defer` captures the loop variable by reference. If used in a closure (e.g., `defer func() { ... i }()`), all deferred calls might see the final value of `i`.

G11 — Easy: A pointer holds the memory address of a value. The zero value of a pointer is `nil`. To create a pointer to a struct literal: `&MyStruct{Field: value}`.

G12 — Easy: `init()` runs automatically before `main()` when a package is loaded. It runs once per package. A package can have multiple `init()` functions (in different files); they execute in declaration order within a file, and files are processed in alphabetical order.

G13 — Easy: `new(T)` allocates zeroed memory for type `T` and returns a pointer `*T`. `make(T)` initializes slices, maps, and channels and returns the initialized type (not a pointer). `new` works for any type; `make` only works for slices, maps, and channels.

G14 — Easy: Errors are values implementing the `error` interface (`Error() string`). Idiomatic pattern: `if err != nil { return err }`. `errors.Is(err, target)` checks if `err` wraps `target` (logical equality). `errors.As(err, &target)` performs type assertion to extract a specific error type. `==` only checks pointer equality, not wrapped errors.

G15 — Easy:
*   `map`: Zero value is `nil`.
*   `slice`: Zero value is `nil`.
*   `channel`: Zero value is `nil`.
*   `struct`: Zero value is all fields set to their respective zero values.

G16 — Medium: A race condition occurs when multiple goroutines access shared data concurrently, with at least one write, without synchronization. The `-race` flag instruments code to detect these at runtime.
Fix with `sync.Mutex`:
```go
var mu sync.Mutex
mu.Lock()
data = "hello"
mu.Unlock()
```
Or use a channel to serialize access.

G17 — Medium: `select` waits on multiple channel operations. It differs from `switch` by evaluating channel readiness dynamically. If multiple cases are ready, one is chosen pseudo-randomly. Timeout is implemented by selecting on a channel created by `time.After(duration)`.

G18 — Medium: Escape analysis determines if a variable lives on the stack (fast) or heap (slower, GC managed). If a variable's address escapes the function scope (e.g., returned), it is allocated on the heap. Inspect with `go build -gcflags="-m"`. Heap allocation matters for performance in tight loops or high-throughput systems.

G19 — Hard: `reflect` allows inspecting and modifying values at runtime. `reflect.Type` describes the type; `reflect.Value` holds the value. Reflection is slow due to dynamic dispatch and type checks. Justified for frameworks (serialization, dependency injection) but avoided in hot paths where code generation or generics are preferred.

G20 — Hard: `//go:embed` embeds files/directories at compile time into a `embed.FS`. Usage: `var f embed.FS`. It supports single files, directories, and glob patterns. Limitations: content is static (no runtime modification), no symlinks. It implements `io/fs.FS`, allowing standard file system operations on embedded content.

R1 — Easy: Ownership ensures memory safety without garbage collection. Rules: 1. Each value has exactly one owner. 2. When the owner goes out of scope, the value is dropped. 3. Values can be moved (transferred ownership) but not copied.

R2 — Easy: `String` is an owned, mutable, heap-allocated string. `&str` is a borrowed, immutable slice of bytes (string slice). Use `String` when you need to own/modify data. Use `&str` for function arguments to avoid allocation and allow flexibility.

R3 — Medium: Borrowing allows temporary access without ownership. Rules: 1. You can have many immutable references (`&T`) OR one mutable reference (`&mut T`), but not both. 2. References must always be valid. 3. You cannot mutate through an immutable reference.

R4 — Medium: `Result<T, E>` represents success (`Ok(T)`) or failure (`Err(E)`). `Option<T>` represents `Some(T)` or `None`. The `?` operator propagates errors.
```rust
use std::fs;
fn read_file() -> Result<String, std::io::Error> {
    fs::read_to_string("file.txt")
}
```

R5 — Medium: `Box<T>` owns data on the heap with a single owner. `Rc<T>` (Reference Counted) allows multiple owners in a single thread. `Arc<T>` (Atomic Reference Counted) allows multiple owners across threads. Use `Box` for heap allocation, `Rc` for shared single-threaded ownership, `Arc` for shared multi-threaded ownership.

R6 — Medium: Traits define shared behavior. Unlike Go interfaces, traits can have default implementations. `dyn Trait` enables dynamic dispatch (vtable, runtime cost). `T: Trait` enables static dispatch (monomorphization, zero cost). Use `dyn Trait` for heterogeneous collections; use generics for performance.

R7 — Hard: The compiler cannot infer that the returned reference lives as long as the inputs. Fix by adding explicit lifetimes:
```rust
fn longest<'a>(x: &'a str, y: &'a str) -> &'a str {
    if x.len() > y.len() { x } else { y }
}
```
This ties the return lifetime `'a` to the input lifetimes.

R8 — Medium: `Send` means a type can be transferred safely across threads. `Sync` means a type can be safely referenced across threads. `Rc<T>` is not `Send` because its reference count is not atomic. `Arc<T>` uses atomic reference counting, making it `Send` and `Sync`.

R9 — Medium: `async/await` creates a `Future` (state machine). The executor (e.g., Tokio) polls the future until completion. Unlike Go's preemptive green threads, Rust async is cooperative (yields at `await` points). This avoids stack overhead but requires explicit executor management.

R10 — Medium: `unsafe` allows operations the compiler cannot verify for safety. 1. Dereferencing raw pointers. 2. Calling unsafe functions. 3. Accessing mutable statics. Justified for FFI, low-level memory manipulation, or implementing safe abstractions over unsafe primitives.

R11 — Easy: Pattern matching deconstructs values. `match` requires exhaustiveness (all cases covered). `if let` is a shorthand for matching a single pattern. The compiler enforces that `match` handles all possible enum variants, preventing logic errors.

R12 — Easy: `Vec<T>` is a growable heap-allocated array. `[T; N]` is a fixed-size stack-allocated array. When `Vec` capacity is exceeded, it allocates a new, larger buffer, copies elements, and drops the old one. Arrays have fixed size known at compile time.

R13 — Easy: Enums are tagged unions (can hold different types). Unlike C/Java, Rust enums can carry data. `Option<T>` is `Some(T)` or `None`. Rust has no `null`; `None` is a type-safe alternative, forcing explicit handling of absence.

R14 — Easy: `impl` defines methods. `&self` borrows the instance. `&mut self` mutably borrows. `self` consumes the instance (move). Associated functions (e.g., `String::new()`) do not take `self` and act like constructors.

R15 — Easy: `derive` macros generate boilerplate code. Common: `Debug`, `Clone`, `PartialEq`. Cannot derive if fields contain types that don't support the trait (e.g., `Fn` closures, raw pointers).

R16 — Medium: `Cow<'a, T>` (Clone on Write) holds either a borrowed reference or an owned value. It avoids allocation if no mutation occurs. Example: Parsing a string slice into an owned string only if modification is needed, otherwise returning the slice.

R17 — Medium: `mod` defines a module. `use` imports paths. `pub` makes items visible outside the crate. File system maps to modules: `mod.rs` or `mod_name/mod.rs`. Rust 2018 edition improved path resolution (no need for `crate::` in many cases) and consolidated module declarations.

R18 — Hard: `Pin<T>` prevents a value from being moved in memory. Needed for self-referential structs and `Future` (executors borrow the future). If a `Future` moves, internal pointers become invalid. `Unpin` is a marker trait indicating a type can be safely moved.

R19 — Hard: `macro_rules!` are declarative macros (pattern matching). Procedural macros (derive, attribute, function-like) are Rust code that generates Rust code at compile time. Hygiene ensures macro variables don't collide with local variables, preventing namespace pollution.

R20 — Hard: Zero-cost abstractions mean high-level code compiles to efficient machine code. Generics use monomorphization (copying code per type). Iterators are inlined loops. Traits use static dispatch. This avoids runtime overhead (vtables, GC) compared to C while maintaining safety and ergonomics.
