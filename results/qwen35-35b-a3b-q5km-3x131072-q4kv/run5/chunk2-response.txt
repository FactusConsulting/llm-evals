K1 — **Deployment vs. StatefulSet:** A Deployment manages stateless applications with interchangeable pods (random names, no guaranteed order). A StatefulSet manages stateful applications requiring stable network IDs, ordered deployment/termination, and persistent storage per pod.

K2 — **ClusterIP Routing:** A ClusterIP creates a virtual IP within the cluster. `kube-proxy` (using iptables or IPVS) programs the node's network rules to forward traffic sent to this virtual IP to the active pod IPs associated with the Service.

K3 — **PV, PVC, StorageClass:** A **PersistentVolume (PV)** is a cluster storage resource. A **PersistentVolumeClaim (PVC)** is a user request for storage. A **StorageClass** defines the provisioning class (dynamic provisioning). A PVC binds to a PV based on access modes and storage class requirements.

K4 — **CrashLoopBackOff Troubleshooting:** 1. Check `kubectl logs` for application errors. 2. Check `kubectl describe pod` for events (OOMKilled, probe failures). 3. Verify resource limits (CPU/Mem) are sufficient. 4. Check liveness/readiness probe configuration (is it killing the pod prematurely?).

K5 — **CoreDNS Role:** CoreDNS provides internal DNS resolution for services and pods. If CoreDNS pods are down, internal service discovery (e.g., `service-name.namespace.svc`) fails, though external DNS resolution may still work if configured.

K6 — **Scheduler Decision:** The scheduler **filters** nodes (taints, resources, affinity) then **scores** them. Factors considered: 1. Available CPU/Memory vs. Pod Requests. 2. Node Affinity rules. 3. Pod Anti-Affinity rules (spread pods across nodes).

K7 — **Memory Scheduling & Runtime:** Node has 4GB, 3.5GB requested (500MB free). Pod requests 256Mi (~262MB). **Yes**, it will be scheduled (262MB < 500MB). If it uses 600Mi at runtime, it exceeds the 512Mi limit and will be **OOMKilled**.

K8 — **Ingress Request Lifecycle:** 1. External LB forwards traffic to Ingress Controller (NGINX). 2. NGINX matches Ingress rules to a Service. 3. Service selects Pods via Endpoints. 4. `kube-proxy` (iptables) routes traffic from Service IP to Pod IP. 5. Pod receives request.

K9 — **NetworkPolicy:** A policy restricting traffic flow. Default behavior (no policy): Allow all traffic. When applied: Traffic is denied by default unless explicitly allowed by the policy's ingress/egress rules for selected pods.

K10 — **RollingUpdate vs. Recreate:** RollingUpdate updates pods gradually (zero downtime). Recreate kills all old pods before starting new ones. Choose **Recreate** for stateful apps requiring full shutdown or when zero-downtime is not required.

K11 — **Namespace:** A logical cluster partition for resource isolation. Default namespaces: `default`, `kube-system`, `kube-public`, `kube-node-lease`.

K12 — **kubectl describe pod:** Shows detailed status. Information not in `get`: 1. Events (warnings/errors). 2. Conditions (Ready, etc.). 3. Mount paths and Environment Variables.

K13 — **ConfigMap:** Stores non-sensitive configuration. Pod consumption methods: 1. As Environment Variables. 2. As Volume Mounts (files).

K14 — **Service Types:** **ClusterIP:** Internal cluster IP only. **NodePort:** Exposes service on a static port on every node's IP. **LoadBalancer:** Provisions an external cloud load balancer.

K15 — **DaemonSet:** Ensures one pod runs on every node (or subset). Use instead of Deployment for cluster-level agents (logging, monitoring, CNI).

K16 — **Taints vs. Affinity:** **Taints** repel pods (node pushes away). **Tolerations** allow pods to ignore taints. **Affinity** pulls pods to nodes. Use Case: Taint for dedicated GPU nodes; Affinity to co-locate app with cache.

K17 — **Kubernetes Operator:** A custom controller that manages complex stateful applications via Custom Resource Definitions (CRDs). Differs from standard controllers by automating lifecycle management (backup, scaling, upgrades) specific to that app.

K18 — **etcd Consistency:** Strongly consistent (linearizable). Quorum loss (e.g., 2/3 nodes lost) makes the cluster read-only/unavailable. Recovery: Restore from a snapshot on a new node and rejoin the cluster to restore quorum.

K19 — **RBAC:** **Role** (namespaced perms), **ClusterRole** (cluster perms), **RoleBinding** (links Role to Subject in namespace), **ClusterRoleBinding** (links ClusterRole to Subject).
```yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata: namespace: my-ns
rules: [{apiGroups: ["", "apps"], resources: ["pods", "pods/log"], verbs: ["get", "list"]}]
---
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata: namespace: my-ns
subjects: [{kind: User, name: dev-user}]
roleRef: {kind: Role, name: pod-reader, apiGroup: rbac.authorization.k8s.io}
```

K20 — **ndots & DNS:** `ndots` (default 5) dictates how many dots trigger FQDN lookup. High ndots causes queries for every search domain before treating input as FQDN. Fix: Set `dnsConfig: options: [{name: ndots, value: "1"}]` in pod spec.

D1 — **Rebase vs. Merge:** Rebase rewrites history to create a linear commit chain. Merge preserves history with a merge commit. Avoid rebasing on shared/public branches to prevent history conflicts for others.

D2 — **TCP vs. UDP:** TCP is connection-oriented and reliable. UDP is connectionless and fast. TCP: HTTP, SSH. UDP: DNS, VoIP.

D3 — **Race Condition:** Unpredictable behavior due to execution order. Example: Two threads increment a shared counter without locking, resulting in lost updates.

D4 — **Symmetric vs. Asymmetric:** Symmetric uses one key for encryption/decryption. Asymmetric uses a public/private key pair. TLS uses Asymmetric for handshake/key exchange, then Symmetric for data transfer.

D5 — **Image Layer:** A diff of the filesystem. Layer caching improves build speed. Best practice: Copy dependency files first, install deps, then copy source code to maximize cache hits.

D6 — **HTTP 503 Troubleshooting:** 1. Check upstream service health. 2. Check database connectivity. 3. Check resource limits (OOM). 4. Check rate limiting or circuit breaker status.

D7 — **CAP Theorem:** Consistency, Availability, Partition Tolerance. **CP:** Zookeeper (consistency over availability during partition). **AP:** Cassandra (availability over consistency during partition).

D8 — **Scaling:** Horizontal (add nodes), Vertical (add resources). Patterns enabling Horizontal: Stateless services, Sharding, Load Balancing.

D9 — **Environment Variable:** Runtime config. Persist across sessions on Linux by exporting in `~/.bashrc` or `~/.profile`.

D10 — **JSON vs. YAML:** JSON is strict key-value (machine-friendly). YAML is human-readable (indentation-based). JSON advantage: Strict parsing. YAML advantage: Concise config files.

D11 — **REST API:** Architectural style using HTTP. **GET:** Read. **POST:** Create. **PUT:** Update/Replace. **DELETE:** Remove.

D12 — **Version Pinning:** Specifying exact dependency versions (e.g., `1.2.3`). Important for reproducible builds to prevent breakage from upstream updates.

D13 — **Compiled vs. Interpreted:** Compiled translates to binary before execution (C, Go). Interpreted executes code line-by-line (Python, JavaScript).

D14 — **Reverse Proxy:** Server receives requests and forwards them to backend servers. Differs from Forward Proxy (client-side). Servers: Nginx, HAProxy. Use Case: SSL Termination, Load Balancing.

D15 — **Deadlock:** Processes waiting indefinitely. Coffman Conditions: 1. Mutual Exclusion. 2. Hold and Wait. 3. No Preemption. 4. Circular Wait.

D16 — **Eventual Consistency:** Updates propagate asynchronously. Cassandra uses Read Repair and Hinted Handoff. Tunable Levels: ONE, QUORUM, LOCAL_QUORUM, ALL.

D17 — **mTLS:** Mutual TLS where both client and server present certificates. Handshake: Client Hello, Server Hello, Server Cert, Client Cert, Key Exchange. Used in: Service Meshes (Istio), Internal API security.

D18 — **Hash Table:** Array with linked lists (chaining) or probing (open addressing). Avg Time: O(1). Worst Case (all collisions): O(n).

D19 — **Circuit Breaker:** Prevents cascade failures. States: **Closed** (normal), **Open** (fail fast), **Half-open** (test recovery). Relates to Retry/Backoff: Retry until open, backoff to reduce load during recovery.

D20 — **Optimistic vs. Pessimistic:** Optimistic checks version on write (conflict detection). Pessimistic locks data on read (prevents conflict). Choose Optimistic for high concurrency/low conflict. Choose Pessimistic for critical data/high conflict.
