AA1 — **Monolith vs. Microservices:** A monolith is a single deployable unit where all components are tightly coupled. Microservices are independently deployable, loosely coupled services.
**3 Problems Microservices Introduce:**
1.  **Network Latency & Reliability:** Communication is over the network, introducing latency and failure points not present in local calls.
2.  **Distributed Data Consistency:** ACID transactions span multiple services, requiring complex patterns (e.g., Sagas) instead of simple DB transactions.
3.  **Operational Complexity:** Requires orchestration (Kubernetes), distributed tracing, and complex CI/CD pipelines for multiple services.

AA2 — **Sync vs. Async:** Synchronous communication blocks the sender until the receiver responds (request/response). Asynchronous communication allows the sender to continue without waiting (fire-and-forget or callback).
**Examples:** Synchronous: REST (HTTP), gRPC. Asynchronous: RabbitMQ (Message Queue), Kafka (Event Log).

AA3 — **API Gateway vs. Reverse Proxy:** A reverse proxy forwards traffic to backend servers (Layer 7 routing, SSL termination). An API Gateway adds business logic capabilities like authentication, rate limiting, API versioning, and analytics.
**Replacement:** No. An API Gateway cannot fully replace a reverse proxy if the proxy is needed for low-level traffic management (e.g., TCP load balancing) or if the gateway is overkill for internal service-to-service routing. However, a Gateway often *includes* reverse proxy functionality.

AA4 — **Saga Pattern:** A sequence of local transactions where each updates the database and triggers the next. If one fails, compensating transactions undo previous steps.
**Choreography vs. Orchestration:** Choreography uses events to trigger the next step (decentralized). Orchestration uses a central coordinator to tell services what to do.
**Appropriateness:** Choreography for simple, short-lived flows. Orchestration for complex, long-running workflows requiring visibility and control.

AA5 — **CQRS:** Command Query Responsibility Segregation separates read (Query) and write (Command) data models.
**Pairing with Event Sourcing:** Event Sourcing stores state changes as a sequence of events. CQRS uses these events to build optimized read models.
**Problem Solved:** Allows independent scaling of read and write workloads and provides a complete audit trail of state changes, which traditional CRUD (single model) does not easily support.

AA6 — **Circuit Breaker:** A pattern to prevent cascading failures.
**Three States:** Closed (normal operation), Open (fail fast, no calls), Half-Open (test if service recovered).
**Difference from Retry:** Retry attempts to recover from transient failures immediately. Circuit Breaker stops traffic to a failing service to allow recovery.
**Usage:** Use Retry with Backoff for transient network blips. Use Circuit Breaker when a service is persistently down to prevent resource exhaustion.

AA7 — **Tenant Isolation Models:**
1.  **Shared DB, Shared Schema:** Lowest cost, highest operational complexity (row-level security), highest noisy-neighbor risk.
2.  **Shared DB, Separate Schema:** Medium cost, medium complexity, moderate noisy-neighbor risk.
3.  **Separate DB:** Highest cost, lowest complexity (per tenant), lowest noisy-neighbor risk, best security isolation.

AA8 — **Consistency:** Eventual consistency allows temporary divergence; Strong consistency ensures immediate agreement.
**Architecture for Inventory:** Use **Strong Consistency**.
**Data Store:** Relational DB (SQL) with ACID guarantees.
**Patterns:** Optimistic Locking (version columns) or Distributed Locking (Redis/Zookeeper) to reserve stock before committing.

AA9 — **Sidecar Proxy:** A container running alongside the application to handle network traffic.
**Envoy/Istio:** Envoy is the data plane (handles traffic). Istio is the control plane (configures Envoy).
**mTLS:** Encrypts traffic between services. Application-level TLS requires app code changes; mTLS is transparent to the app, ensuring encryption without code changes and preventing MITM attacks.

AA10 — **Strangler Fig Pattern:** Incrementally migrate by routing traffic from legacy to new services.
**Strategy:**
1.  Deploy API Gateway in front of the .NET Framework monolith.
2.  Build new .NET 8 microservices for specific features.
3.  Configure Gateway to route specific paths to new services, rest to legacy.
4.  Gradually move more routes to new services until legacy is decommissioned.

AA11 — **12-Factor App:**
1.  **Codebase:** One repo, many deploys.
2.  **Dependencies:** Explicitly declared.
3.  **Config:** Stored in environment.
4.  **Backing Services:** Treated as attached resources.
5.  **Build/Run/Exec:** Strict separation.
6.  **Ports:** Export services via port binding.
7.  **Concurrency:** Scale via process model.
8.  **Disposability:** Fast startup/shutdown.
9.  **Dev/Prod Parity:** Minimize differences.
10. **Logs:** Stream as event streams.
11. **Admin Processes:** Run as one-off commands.
12. **Backing Services:** Treat as resources.
**Why:** Ensures portability, scalability, and maintainability in cloud environments.

AA12 — **Event-Driven (Kafka) vs. Request-Driven (REST/gRPC):**
*   **Coupling:** Event-driven is loosely coupled; Request-driven is tightly coupled.
*   **Scalability:** Event-driven scales better for high throughput; Request-driven is simpler for low latency control.
*   **Debugging:** Request-driven is easier (traceable flow); Event-driven is harder (eventual visibility).
*   **Ordering:** Kafka guarantees ordering per partition; REST is request-order dependent.
*   **Failure:** Event-driven requires retry/dead-letter queues; Request-driven fails immediately.
**Choice:** Use Event-driven for high-volume data processing/decoupling. Use Request-driven for real-time user interaction.

AA13 — **Load Balancer:** Distributes traffic across servers.
**L4 vs. L7:** L4 (Transport) routes based on IP/Port. L7 (Application) routes based on URL, Headers, Cookies.
**L7 Use Case:** SSL Termination or routing `/api/v1` to specific backend versions.

AA14 — **Message Queue:** Stores messages for delivery.
**MQ (RabbitMQ) vs. Log (Kafka):** MQ is point-to-point (consumed once, deleted). Log is broadcast (retained, replayable).
**Pick:** RabbitMQ for task queues/workflows. Kafka for event streaming/logging.

AA15 — **Container vs. VM:** Container shares the host OS kernel but isolates processes. VM virtualizes hardware and runs a full OS.
**Popularity:** Containers are lightweight, start faster, and have consistent environments, ideal for microservices.

AA16 — **Stateless vs. Stateful:** Stateless services store no session data (any instance can handle any request). Stateful services store data (e.g., DB).
**Scaling:** Stateless scales horizontally easily (add nodes). Stateful requires data replication/migration.
**Examples:** Stateless: Web Server. Stateful: Database.

AA17 — **ORM:** Maps objects to database tables.
**Examples:** Hibernate (Java), Entity Framework (.NET).
**Trade-offs:** ORM increases development speed but can obscure SQL performance and limit complex query optimization.

AA18 — **Bulkhead Pattern:** Isolates resources (threads, connections) for different services.
**Implementation:** Use separate thread pools for each downstream service call. If one service hangs, it only exhausts its pool, not the whole application.

AA19 — **OT vs. CRDT:** OT requires a central server to merge changes (server authoritative). CRDTs allow clients to merge changes mathematically (client authoritative).
**Choice:** **CRDTs**.
**Why:** CRDTs support offline editing and conflict resolution without a central server, essential for real-time collaboration with latency tolerance.

AA20 — **Distributed Tracing Challenges:**
*   **Context Propagation:** Pass TraceID/SpanID via headers (W3C/B3).
*   **Sampling:** Head-based (cost control) vs. Tail-based (accuracy for errors).
*   **Storage:** Jaeger/Tempo for querying.
*   **Kafka:** Inject trace context into message headers so downstream consumers can continue the trace.

OP1 — **Hypervisor Type 1 vs. Type 2:** Type 1 runs directly on hardware (ESXi, Hyper-V). Type 2 runs on an OS (VirtualBox, VMware Workstation).
**Examples:** Type 1: VMware ESXi, Microsoft Hyper-V. Type 2: Oracle VirtualBox, VMware Workstation.

OP2 — **SAN vs. NAS:** SAN provides block storage (iSCSI, Fibre Channel). NAS provides file storage (NFS, SMB).
**Use:** SAN for databases (low latency, block access). NAS for file shares (collaboration).

OP3 — **3-Tier vs. Leaf-Spine:** 3-Tier (Core/Distribution/Access) has oversubscription and latency at the distribution layer. Leaf-Spine is flat, non-blocking, and reduces latency.
**Problem Solved:** Leaf-Spine solves oversubscription and provides predictable latency for east-west traffic in modern data centers.

OP4 — **Proxmox HA Design:**
*   **Nodes:** 3+ nodes for quorum.
*   **Quorum:** Majority vote to prevent split-brain.
*   **Fencing:** STONITH (DRAC/iLO) to power off failed nodes.
*   **Storage:** Ceph (distributed) or iSCSI/NFS (shared).
*   **Failure:** VMs restart on remaining nodes automatically.

OP5 — **VLAN Trunking:** 802.1Q tagging adds a VLAN ID to Ethernet frames.
**Linux Config:** Create sub-interfaces. `ip link add link eth0 name eth0.10 type vlan id 10`. Assign IP to `eth0.10`.

OP6 — **RAID Levels:** RAID 1 (Mirror), RAID 5 (Striped+Parity), RAID 6 (Double Parity), RAID 10 (Mirror+Stripe), RAID Z2 (ZFS Double Parity).
**Choice:** **RAID 10** (or Z2 equivalent).
**Why:** With 6 drives, RAID 10 offers best performance and fault tolerance (survives 2 drive failures if mirrored pairs are intact) for a database.

OP7 — **DNS Architecture:**
*   **Internal:** AD Integrated DNS (Active Directory).
*   **External:** Public DNS (e.g., Route53) with DNSSEC.
*   **Split-Horizon:** Internal clients resolve internal IPs; External resolve public IPs.
*   **Forwarders:** Internal DNS forwards unknown queries to external.
*   **Laptop:** DHCP assigns DNS server. Laptop queries internal DNS for internal resources, external for internet.

OP8 — **PKI Chain:** Root CA (Offline) -> Subordinate CA (Issuing) -> End Entity.
**CRL vs. OCSP:** CRL is a list of revoked certs; OCSP is real-time status check.
**Compromise:** Root CA compromise = total trust loss (must rebuild PKI). Subordinate CA compromise = limited scope (revoke sub, issue new).

OP9 — **WAN Consolidation:**
*   **MPLS:** High cost, high reliability.
*   **IPsec:** Low cost, variable reliability.
*   **SD-WAN:** Low cost, managed, QoS.
*   **WireGuard:** Simple, secure, mesh.
*   **Recommendation:** **SD-WAN**. Best for cost, management, and VoIP QoS for 50 users/site.

OP10 — **iDRAC/iLO/IPMI:** Out-of-band management interfaces.
**Critical:** Allows server control (power, console) even if OS is down.
**Exposure:** Exposed to internet = high risk of unauthorized root access to hardware.

OP11 — **LVM vs. ZFS:** LVM manages volumes on top of block devices. ZFS is a filesystem + volume manager.
**Snapshots:** LVM (Copy-on-Write, slower). ZFS (Instant, efficient).
**Use:** LVM for simple volume mgmt. ZFS for data integrity and snapshots.

OP12 — **Stretch Cluster:**
*   **Design:** 2 sites + 1 Witness node (tiebreaker).
*   **Ceph:** CRUSH rules must distribute replicas across sites.
*   **Latency:** <5ms for monitors.
*   **Split-Brain:** Without a witness, both sites might think they are primary. 2-site Ceph is bad without a tiebreaker.

OP13 — **DHCP DORA:** Discover, Offer, Request, Acknowledge.
**Unavailable Server:** Client retries. If lease expires, client releases IP and starts DORA again. Windows may use APIPA (169.254.x.x) if no server found.

OP14 — **Managed vs. Unmanaged:** Managed switches support VLANs, QoS, ACLs, SNMP. Unmanaged are plug-and-play.
**Importance:** Managed switches are critical for segmentation (VLANs) and traffic prioritization (QoS) in enterprises.

OP15 — **NTP:** Network Time Protocol.
**Critical:** Certificates, logs, and cluster consensus rely on accurate time.
**Drift:** Clocks drift = Kerberos auth fails, logs become uncorrelatable, cluster nodes diverge.

OP16 — **Physical vs. VM vs. Container:** Physical = Bare metal. VM = Virtualized OS. Container = Isolated processes.
**Bare Metal:** Use for high-performance computing, low-latency databases, or hardware passthrough needs.

OP17 — **UPS vs. Generator:** UPS provides short-term power (minutes) to shut down gracefully. Generator provides long-term power (hours).
**ATS:** Automatic Transfer Switch switches power from Utility to Generator when utility fails.

OP18 — **Active-Passive vs. Active-Active:** Active-Passive (one handles load, one waits). Active-Active (both handle load).
**PostgreSQL HA:** Pacemaker/Corosync cluster. Virtual IP floats. STONITH (fencing) ensures failed node is powered off. Failover moves VIP to new master.

OP19 — **Backup Strategy:**
*   **3-2-1:** 3 copies, 2 media types, 1 offsite.
*   **RPO/RTO:** Define acceptable data loss and downtime.
*   **Types:** Full, Incremental, Differential.
*   **Immutable:** Protects against ransomware.
*   **Test:** Regular restore drills.
*   **Backup vs. DR:** Backup is data copy; DR is site recovery plan.

OP20 — **802.1X:**
*   **Roles:** Supplicant (Client), Authenticator (Switch), Server (RADIUS).
*   **EAP:** EAP-TLS (Certificate-based, secure). PEAP (Password-based, tunneled).
*   **Dynamic VLAN:** Assign VLAN based on user identity.
*   **Failure:** Port blocked or placed in Guest VLAN.
*   **MAB:** Fallback for devices without 802.1X (e.g., printers) using MAC address as credential.
