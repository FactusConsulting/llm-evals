**DN1**
**Class vs. Struct:** `class` is a reference type (heap-allocated, mutable by default), while `struct` is a value type (stack-allocated or inline, immutable by default).
**When to use Struct:** Use `struct` for small, immutable data objects (e.g., `Point`, `Color`) where copying is cheap and value semantics are required. Avoid for large objects or those with complex state.

**DN2**
**IEnumerable<T> vs. IQueryable<T>:** `IEnumerable<T>` executes queries in-memory (client-side). `IQueryable<T>` represents a query expression tree that can be translated to a provider-specific language (e.g., SQL) and executed server-side.
**Database Distinction:** This matters because `IQueryable` allows filtering, projection, and sorting to happen on the database server, reducing network traffic and improving performance. `IEnumerable` pulls all data to memory first.

**DN3**
**Async/Await:** `async/await` enables non-blocking I/O by compiling state machines. `await` pauses execution until the `Task` completes without blocking the thread.
**Task vs. ValueTask:** `Task` is a reference type (heap allocation). `ValueTask` is a struct (stack allocation) that wraps a `Task` or a `ValueTask` result.
**Use ValueTask:** Use `ValueTask` in hot paths where the operation completes synchronously often, to avoid the allocation overhead of `Task`.

**DN4**
**Dependency Injection (DI):** DI is a pattern where dependencies are provided to a class rather than created internally.
**Lifetimes:**
1.  **Transient:** New instance every time requested.
2.  **Scoped:** One instance per request (e.g., HTTP request).
3.  **Singleton:** One instance for the application lifetime.
**Common Bug:** Injecting a `Scoped` service into a `Singleton` service. The Singleton captures the Scoped instance at creation time, leading to stale data or disposal errors when the request scope ends.

**DN5**
**String vs. ReadOnlySpan<char>:** `string` is immutable and heap-allocated. `ReadOnlySpan<char>` is a stack-only view into memory (can be on stack, heap, or stack).
**Use Span<T>:** Use `Span<T>` for high-performance string processing (parsing, slicing) to avoid allocations and copying. It enables zero-copy operations.

**DN6**
**Garbage Collector (GC):** The GC automatically reclaims memory from objects no longer in use.
**Generations:** Objects are promoted through Gen 0 (short-lived), Gen 1, and Gen 2 (long-lived) to optimize collection frequency.
**Large Object Heap (LOH):** Objects >85KB are allocated here and collected less frequently.
**IDisposable:** Needed for unmanaged resources (files, DB connections) that the GC cannot track. `Dispose` ensures timely cleanup; relying solely on GC is non-deterministic.

**DN7**
**ConfigureAwait(false):** Prevents capturing the current `SynchronizationContext` after an `await`.
**When to Use:** In library code or background services to avoid deadlocks and improve performance.
**When NOT to Use:** In UI applications (WPF, WinForms) where you need to marshal back to the UI thread to update controls.
**SynchronizationContext:** It marshals execution back to a specific thread. `ConfigureAwait(false)` bypasses this, allowing the continuation to run on any thread pool thread.

**DN8**
**Features:**
*   **Primary Constructors:** Define constructor parameters directly in the class declaration (C# 12/.NET 8).
*   **Collection Expressions:** Concise syntax for creating arrays/lists (`[1, 2, 3]`).
*   **FrozenDictionary:** Immutable dictionary with O(1) lookup and no allocation on modification.
**Example:**
```csharp
public class Service(int Id) : IService { }
// DI Registration: services.AddSingleton(new Service(1));
```

**DN9**
**Diagnosis:** The `DbContext` is likely registered as a `Singleton` instead of `Scoped`. Under high concurrency, multiple requests share the same context instance, causing race conditions and `ObjectDisposedException` when one request disposes it while another uses it.
**Fix:** Register `DbContext` as `Scoped` in DI (`AddDbContext`).
**Interaction:** EF Core `DbContext` is not thread-safe. It must align with the request scope in ASP.NET Core.

**DN10**
**Record vs. Class:** `record` is a reference type with value-based equality (by position) and is immutable by default. `class` has reference equality.
**Record Struct:** A value type with value-based equality.
**Choice:** Use `record` for immutable data models (DTOs, Entities). Use `record struct` for small, immutable value objects. Use `class` for mutable entities or when identity matters.

**DN11**
**Modifiers:**
*   `ref`: Pass by reference (read/write). Caller must initialize.
*   `out`: Pass by reference (write-only). Must be assigned inside method.
*   `in`: Pass by reference (read-only). Avoids copy of large structs.
**Usage:** `out` for multiple return values. `ref` for modifying existing values. `in` for performance with large structs.

**DN12**
**Nullable Reference Types (NRT):** Compile-time warnings for potential null dereferences on reference types. `string?` allows null.
**Nullable Value Types:** `int?` is syntactic sugar for `Nullable<int>`. It is a struct that can hold a value or null.
**Difference:** NRT tracks nullability at compile-time for references; `int?` is a runtime value type wrapper.

**DN13**
**LINQ:** Language Integrated Query.
**Syntax:** Method (`list.Where(...)`) vs Query (`from x in list`). They compile to the same IL.
**Deferred Execution:** Queries are not executed until enumerated (e.g., `ToList()`, `foreach`). This allows composition and optimization.

**DN14**
**Generic Constraints:** Restrict types passed to generics.
*   `where T : class`: Reference type.
*   `where T : struct`: Value type.
*   `where T : new()`: Parameterless constructor.
*   `where T : IComparable<T>`: Implements interface.
**Combination:** `where T : class, IComparable<T>` ensures T is a reference type implementing comparison.

**DN15**
**Abstract Class vs. Interface:** Abstract class provides partial implementation and state. Interface defines a contract (methods/properties).
**C# 8+ Default Interface Methods:** Interfaces can now have method implementations. This blurs the line, allowing interfaces to provide default behavior without forcing inheritance. Prefer interfaces for contracts, abstract classes for shared code.

**DN16**
**Middleware Pipeline:** Components that handle requests/responses in order.
*   `app.Use()`: Adds middleware to the pipeline.
*   `app.Map()`: Branches the pipeline for specific paths.
*   `app.Run()`: Terminal middleware (no next).
**Ordering:** Critical (e.g., Auth before App logic).
**Exception Handling:** Middleware can catch exceptions and return a response, preventing the pipeline from crashing.

**DN17**
**Channel<T>:** Async-safe queue for producer-consumer patterns.
**vs. BlockingCollection<T>:** `Channel` is async-native; `BlockingCollection` is sync-based.
**Bounded Channels:** Set a capacity limit. If full, producers block (backpressure), preventing memory exhaustion.

**DN18**
**Source Generators:** Compile-time code generation.
**vs. Reflection/IL Weaving:** Reflection is runtime (slow). IL Weaving is post-build (complex). Generators are compile-time (fast, type-safe).
**Incremental Generator:** Only regenerates code when source files change, improving build performance over `ISourceGenerator` which runs on every build.

**DN19**
**TPL Dataflow:** Library for building data pipelines.
*   **ActionBlock:** Processes items (void return).
*   **TransformBlock:** Maps items (input -> output).
*   **BufferBlock:** Stores items (queue).
**vs. Channels:** Dataflow offers more complex flow control (fan-out, fan-in, cancellation), while Channels are simpler for basic queues.

**DN20**
**Minimal APIs:** Lightweight API endpoints without controllers.
**Trade-offs:** Less boilerplate, but harder to scale complex logic, less built-in model binding attributes, and OpenAPI support requires extensions (Swashbuckle).
**Route Groups:** Organize endpoints by feature (`MapGroup`).
**Endpoint Filters:** Apply logic (e.g., validation, auth) to specific endpoints.

**P1**
**List vs. Tuple:** `List` is mutable, ordered collection. `Tuple` is immutable, fixed-size collection.
**Set vs. Frozenset:** `Set` is mutable, unordered unique items. `Frozenset` is immutable (can be used as a dictionary key).

**P2**
**GIL:** Global Interpreter Lock allows only one thread to execute Python bytecode at a time.
**Affected Workloads:** CPU-bound multi-threaded tasks.
**Workarounds:** Use `multiprocessing` (separate processes), C-extensions (release GIL), or async I/O.

**P3**
**Deepcopy vs. Copy:** `copy` creates a shallow copy (new container, same objects). `deepcopy` creates a new container and recursively copies all objects.
**Shallow Copy Bug:**
```python
a = [[1], [2]]
b = copy(a)
b[0].append(3)
# a is now [[1, 3], [2]] because inner lists are shared.
```

**P4**
**Decorators:** Functions that wrap other functions to extend behavior.
**Time Decorator:**
```python
import time
def timer(func):
    def wrapper(*args, **kwargs):
        start = time.time()
        result = func(*args, **kwargs)
        print(f"{func.__name__} took {time.time() - start}s")
        return result
    return wrapper
```

**P5**
**Generators/Yield:** Functions that return an iterator. `yield` pauses execution and saves state.
**vs. List:** Generators are lazy (memory efficient).
**Yield From:** Delegates to another generator (`yield from sub_gen()`).

**P6**
**Event Loop:** Single-threaded scheduler for async tasks.
**run()**: Starts the loop. **gather()**: Runs tasks concurrently and waits for all. **create_task()**: Schedules a coroutine immediately.
**time.sleep()**: Blocks the event loop. Use `await asyncio.sleep()` instead.

**P7**
**MRO:** Method Resolution Order determines which method is called in inheritance. Python 3 uses C3 Linearization.
**Diamond Inheritance:** `D(B, C)`. MRO: `D -> B -> C -> A`.
**Output:** `D().who()` prints "C". `B` inherits `who` from `A`, but `C` overrides it. MRO visits `C` before `A`.

**P8**
**Metaclass:** Class of a class. Controls class creation.
**__init_subclass__**: Simpler hook for subclass initialization.
**Use Case:** Metaclass for registry (auto-registering plugins). `__init_subclass__` for enforcing config on subclasses.

**P9**
**Multiprocessing:** Separate processes (no GIL, heavy overhead). Good for CPU-bound.
**Threading:** Shared memory (GIL). Good for I/O-bound.
**Concurrent.Futures:** High-level API for both.
**Serialization:** Multiprocessing requires arguments to be picklable (cannot pass lambdas or open file handles easily).

**P10**
**__slots__:** Declares fixed attributes.
**Use:** Save memory and prevent dynamic attribute creation.
**Impact:** Reduces `__dict__` overhead per instance. Prevents accidental typos in attribute names.

**P11**
**Virtual Environment:** Isolated Python environment with its own packages.
**Why:** Prevent dependency conflicts between projects.
**Differences:** `venv` (stdlib, Python 3.3+), `virtualenv` (3rd party, older), `conda` (binary packages, data science).

**P12**
**Comprehensions:** Concise syntax for creating lists/dicts/sets.
**vs. map/filter:** Comprehensions are more Pythonic and readable.
**Preference:** Use comprehensions for simple transformations. Use `map/filter` for functional pipelines or when reusing functions.

**P13**
**== vs. is:** `==` checks value equality. `is` checks object identity (memory address).
**Interning:** Small integers (-5 to 256) are cached. `a is b` returns True for small ints because they point to the same object.

**P14**
**\*args/\*\*kwargs:** `*args` collects positional args into a tuple. `**kwargs` collects keyword args into a dict.
**Markers:** `/` forces args before it to be positional-only. `*` forces args after it to be keyword-only.

**P15**
**Context Managers:** Manage resources (setup/teardown).
**Implementation:** `__enter__` (setup), `__exit__` (cleanup). `@contextmanager` uses a generator.
**Example:** Database connection, file locks, or temporary directory creation.

**P16**
**Type Hinting:** Static type annotations.
**Types:** `Union` (A|B), `Optional` (A|None), `Literal` (specific value), `TypeVar` (generic type), `Protocol` (structural typing).
**Usage:** `mypy`/`pyright` check types before runtime.

**P17**
**Dunder Methods:** Magic methods (`__init__`, `__str__`).
**__repr__ vs __str__:** `__repr__` (dev/debug), `__str__` (user).
**__eq__ vs __hash__:** If `__eq__` is defined, `__hash__` becomes None (unhashable) unless explicitly defined.
**Impact:** Objects with custom `__eq__` cannot be used in sets/dicts unless `__hash__` is implemented.

**P18**
**Descriptors:** Objects that define `__get__`, `__set__`, `__delete__`.
**Usage:** `property`, `classmethod`, `staticmethod` are descriptors.
**Mechanism:** Attribute access triggers descriptor protocol, allowing custom logic before/after access.

**P19**
**Import System:** Finders locate modules, loaders load them.
**Relative vs Absolute:** Relative (`from . import x`) vs Absolute (`from pkg import x`).
**sys.path:** Search path. `__init__.py` marks a directory as a package (PEP 420 allows namespace packages without it).
**Finder/Loader:** `sys.meta_path` contains finders.

**P20**
**Dataclasses:** Reduces boilerplate for `__init__`, `__repr__`, etc.
**vs. attrs/Pydantic:** `attrs` is more flexible. `Pydantic` adds validation.
**__post_init__:** Logic after `__init__`.
**Frozen:** Immutable dataclass.
**Pydantic Choice:** Use Pydantic when input validation (types, constraints) is required (e.g., API payloads).
