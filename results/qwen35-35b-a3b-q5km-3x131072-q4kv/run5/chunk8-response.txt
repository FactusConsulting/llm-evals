**CL1**
**IaaS (Infrastructure as a Service):** Provides virtualized computing resources (servers, storage, networking). You manage OS, middleware, runtime, data, and apps. *Example: AWS EC2.*
**PaaS (Platform as a Service):** Provides hardware and software tools (usually for app development) without the cost of buying/maintaining the underlying infrastructure. You manage data and apps. *Example: AWS Elastic Beanstalk.*
**SaaS (Software as a Service):** Delivers software applications over the internet. You manage only user access and data configuration. *Example: Microsoft 365.*

**CL2**
**Shared Responsibility Model:** The cloud provider secures the infrastructure (hardware, software, networking, facilities). The customer secures what they put in the cloud (OS, data, apps, config).
**IaaS VM:** Provider secures hypervisor/physical host. Customer secures OS, patching, firewall, data.
**Managed Kubernetes:** Provider secures control plane/infrastructure. Customer secures node OS, pod security, network policies, secrets.
**SaaS (M365):** Provider secures app/infrastructure. Customer secures identity, data classification, device compliance, user access.

**CL3**
**VPC:** Isolated virtual network in the cloud. **Subnet:** Range of IP addresses within a VPC (can be public/private). **Security Group:** Stateful firewall at the instance level (allow rules only). **NACL:** Stateless firewall at the subnet level (allow/deny rules).
**Layering:** VPC contains Subnets. Subnets contain Instances. SG protects Instances. NACL protects Subnets.
**Stateful vs Stateless:** SG is stateful (return traffic allowed automatically). NACL is stateless (must explicitly allow return traffic).

**CL4**
**Landing Zone:** Secure, compliant baseline environment. **OUs:** Organizational Units for grouping accounts (e.g., Prod, Dev). **Guardrails:** SCPs or policies to prevent risky actions (e.g., disable root). **Shared Networking:** Transit Gateway (AWS) or Hub-and-Spoke (Azure) for centralized connectivity. **Centralized Logging:** Aggregates logs to a central account for monitoring.
**Single-Account Problems:** High blast radius (one breach compromises all), billing complexity, lack of isolation between environments, difficulty enforcing compliance boundaries.

**CL5**
**IaC:** Managing infrastructure via code files rather than manual console clicks.
**Terraform/OpenTofu:** Multi-cloud, declarative, state-based. Best for heterogeneous environments.
**Pulumi:** Multi-cloud, imperative (uses general languages like Python/TS). Best for devs comfortable with code.
**CloudFormation/Bicep:** Cloud-native (AWS/Azure). Best for deep integration with specific provider features.
**Crossplane:** IaC using Kubernetes CRDs. Best for managing cloud resources from within a K8s cluster.

**CL6**
**HPA:** Scales pods based on CPU/Memory metrics. **VPA:** Adjusts resource requests/limits for pods (requires restart). **Cluster Autoscaler:** Adds/removes nodes based on pending pods. **KEDA:** Event-driven scaling (e.g., queue length).
**KEDA vs HPA:** Use KEDA when scaling based on external events (Kafka, SQS, HTTP) rather than just CPU/Memory.

**CL7**
**RPO/RTO:** RPO (Recovery Point Objective) = max data loss. RTO (Recovery Time Objective) = max downtime.
**Active-Active:** Both regions run traffic. Low RTO/RPO, high cost. **Active-Passive:** One region runs, other waits. Higher RTO, lower cost.
**DB Replication:** Sync (RPO=0, latency risk) vs Async (RPO>0, latency tolerant). **DNS Failover:** Route53/Manager routes traffic to healthy region.
**DR Types:** **Pilot Light:** DB running, app off. **Warm Standby:** App running, scaled down. **Multi-site Active-Active:** Full redundancy in both regions.

**CL8**
**GitOps:** Git repository is the single source of truth for infrastructure/app state. **ArgoCD:** Push-based (polls Git, applies to cluster). **Flux:** Pull-based (cluster pulls Git, applies).
**Secrets:** Sealed Secrets (encrypted in Git), External Secrets Operator (fetches from Vault/AWS Secrets Manager).
**Drift:** GitOps controllers detect manual changes and automatically revert them to match Git state (or alert).

**CL9**
**StatefulSet:** Ensures stable network IDs and ordered deployment/termination. **PV/PVC:** Use `local` storage class with `volumeBindingMode: WaitForFirstConsumer` for low latency. **Headless Service:** Provides DNS for StatefulSet pods. **Init Containers:** Prepare config files before main app starts. **ConfigMaps:** Mount as volumes for specific paths. **PDB:** Ensures minimum available pods during disruptions.

**CL10**
**Service Account:** K8s identity for pods. **Workload Identity:** GCP/Azure native identity binding. **IRSA:** AWS IAM Role bound to K8s SA via OIDC.
**Mounting Creds:** Bad practice because credentials are static, hard to rotate, and visible in pod spec/logs.
**Federation:** Uses OIDC tokens to exchange K8s identity for temporary cloud credentials, eliminating static secrets.

**CL11**
**Object Storage (S3):** Unstructured data, high durability (11 9s), low latency. Use for backups, media.
**Block Storage (EBS):** Attached to VM, low latency, persistent. Use for OS, DBs.
**File Storage (EFS):** Shared file system, NFS/SMB. Use for shared app configs, home dirs.
**Guarantees:** Object storage offers highest durability (11 9s). Block/File offer high availability but lower durability than object.

**CL12**
**Zero Trust:** Verify every request, never trust by default. **Network Policies:** Restrict pod-to-pod traffic. **Service Mesh (mTLS):** Encrypts service-to-service traffic. **Pod Identity:** Assign roles to pods. **Admission Controllers (OPA):** Enforce policies on deploy. **Image Signing:** Verify integrity. **Runtime Security (Falco):** Detect anomalies.
**Difference:** Traditional perimeter trusts internal traffic. Zero Trust assumes breach and validates every interaction.

**CL13**
**CDN:** Distributed network of edge servers caching content closer to users.
**Cloud-Native:** AWS CloudFront, Azure CDN.
**Use CDN:** For static assets, global latency reduction, DDoS protection. **Use Origin:** For dynamic, personalized, or real-time data not suitable for caching.

**CL14**
**Region:** Geographic area with multiple AZs. **AZ:** Isolated data center within a region.
**Multi-AZ:** Protects against data center failure.
**Zonal:** Resources tied to one AZ (e.g., EBS). **Regional:** Resources span AZs (e.g., ELB, Route53).

**CL15**
**Managed DB:** Service like RDS/Cloud SQL.
**Provider Handles:** Patching, backups, failover, hardware provisioning, monitoring.
**You Handle:** Schema, queries, user access, data encryption.

**CL16**
**IAM:** Users (people), Groups (collections), Roles (temporary access), Policies (permissions).
**Least Privilege:** Grant only necessary permissions.
**Application:** Use roles instead of long-term keys, group users by function, audit permissions regularly.

**CL17**
**Serverless:** Event-driven code execution without managing servers.
**Cold Starts:** Latency when a function is invoked after inactivity (initialization).
**Poor Fit:** Long-running processes, stateful apps, high-frequency invocations with strict latency needs.

**CL18**
**Ingress:** L7 routing rules (K8s native). **Gateway API:** Extensible, standardized API for L7/L4. **LB:** Cloud provider load balancer. **Mesh:** Sidecar proxy routing.
**TLS:** Terminates at LB (external) or Ingress Controller (internal).
**Gateway API:** Use for complex routing, multi-cluster, or when Ingress resource is too limited.

**CL19**
**Commit:** Reserved Instances/Savings Plans (1-3yr) for baseline. **Spot:** For fault-tolerant, interruptible workloads.
**Right-sizing:** Match instance size to actual usage. **Tiering:** Move old data to Glacier/S3 IA.
**Tools:** AWS Cost Explorer, Azure Cost Management, CloudHealth.
**Process:** FinOps team, monthly reviews, showback/chargeback to teams for accountability.

**CL20**
**Strategy:** Kubernetes (runtime), Terraform (IaC), Interconnect (Direct Connect/ExpressRoute).
**Trade-offs:** Multi-cloud increases complexity, cost, and skill requirements.
**Justified:** Regulatory compliance, avoiding single-vendor lock-in, geographic redundancy.
**Unnecessary:** If one cloud meets all needs, multi-cloud adds operational overhead without benefit.

**OT1**
**Purdue Model:** ISA-95 standard for IT/OT integration.
**Levels:**
1.  **Level 0:** Physical Process (Sensors/Actuators).
2.  **Level 1:** Basic Control (PLCs/RTUs).
3.  **Level 2:** Monitoring (SCADA/HMI).
4.  **Level 3:** Operations (MES/Plant Historian).
5.  **Level 4:** Enterprise (ERP/Business).

**OT2**
**IT vs OT:** IT focuses on Confidentiality (CIA triad). OT focuses on Safety and Availability (Safety, Reliability).
**Patching:** OT systems often run legacy OS (XP/7) and cannot tolerate downtime. Patching can break control logic or cause safety hazards.

**OT3**
**DMZ (IDMZ):** Buffer zone between IT and OT.
**Traffic:** Controlled data flow (e.g., Historian pulls from OT, pushes to IT).
**Direct L4-L1:** Dangerous because IT vulnerabilities can directly compromise control logic, risking physical safety.

**OT4**
1.  **Modbus TCP:** L1/L2, simple, unencrypted.
2.  **OPC UA:** L2/L3, secure, platform-independent.
3.  **EtherNet/IP:** L1/L2, Allen-Bradley dominant.
4.  **PROFINET:** L1/L2, real-time industrial Ethernet.
5.  **MQTT:** L3/L4, lightweight, publish/subscribe for cloud.

**OT5**
**Data Diode:** Hardware allowing data flow in one direction only.
**Firewall:** Allows bidirectional traffic with rules.
**Use Diode:** When OT data must go to IT but IT must never touch OT (e.g., high security).
**Limitation:** Cannot receive commands or updates from IT side.

**OT6**
**Flow:** PLC -> Historian -> Edge Gateway -> MQTT Broker -> Cloud.
**Security:** Edge Gateway authenticates to Cloud. DMZ separates Historian from Internet. Encryption (TLS) for data in transit.

**OT7**
**Zones:** Areas with similar security requirements. **Conduits:** Controlled paths between zones.
**Model:** Corporate (L4) -> DMZ -> SCADA (L3) -> PLCs (L1).
**SIS:** Isolated zone. Engineering workstations connect via secure jump host. Conduits allow data flow but block direct control commands from IT.

**OT8**
**VLANs:** Segment by production line or function. **Firewalls:** Between VLANs and DMZ. **Switches:** Managed industrial switches with ACLs.
**Remote Access:** Jump host with MFA and session recording.
**Transition:** Phased migration during maintenance windows, parallel testing before cutover.

**OT9**
**SIS vs BPCS:** SIS (Safety Instrumented System) trips process to safe state. BPCS (Basic Process Control) manages normal operation.
**IEC 61511:** Standard for safety instrumented systems.
**Isolation:** SIS must be air-gapped to prevent cyber attacks from disabling safety functions.
**Ransomware:** If SIS is compromised, physical safety is lost, risking injury or explosion.

**OT10**
**PLC:** Logic controller for specific machines. **DCS:** Distributed control for continuous processes. **SCADA:** Supervisory monitoring over wide area.
**Coexist:** Yes, PLCs often feed into SCADA/DCS.

**OT11**
**Architecture:** Jump host (bastion) -> MFA -> Session Recording.
**Vendor Access:** Time-limited, audited, no direct network access.
**Direct RDP:** Unacceptable because it exposes OT assets to IT threats and lacks granular control.

**OT12**
**Detection:** IDS/Anomaly detection on OT network. **Inventory:** Know all assets.
**Response:** Isolate compromised segment, verify physical safety, manual override if needed.
**Operator Role:** Maintain safe physical state, communicate with SOC, execute emergency procedures.

**OT13**
**HMI:** Operator interface for monitoring and controlling processes.
**Difference:** Engineering Workstation is for programming/configuring logic. HMI is for runtime operation.

**OT14**
**Historian:** Time-series database for process data.
**Importance:** Trend analysis, compliance, troubleshooting.
**Plant vs Enterprise:** Plant (L3) is local, real-time. Enterprise (L4) is aggregated for business analytics.

**OT15**
**Ladder Logic:** Graphical programming language for PLCs.
**Dominance:** Proven reliability, easy for electricians to read, industry standard for decades.

**OT16**
**RTU:** Remote Terminal Unit, ruggedized for harsh environments (long distance).
**PLC:** Local, factory floor.
**Deploy RTU:** Pipelines, water treatment plants, remote sites.

**OT17**
**Analog:** Continuous signal (voltage/current). **Digital:** Discrete (0/1).
**4-20mA:** Preferred because 0mA indicates a broken wire (fault), whereas 0-20mA cannot distinguish between "zero value" and "broken wire".

**OT18**
**Inventory:** Foundation for security (you can't protect what you don't know).
**Tools:** Passive monitoring (safe), Active scanning (risky).
**Active Scanning:** Dangerous because it sends traffic that can crash legacy OT devices.

**OT19**
**MITRE ICS:** Framework for ICS threats.
**Tactics:**
1.  **Initial Access:** Spearphishing (e.g., via email).
2.  **Execution:** Command and Scripting Interpreter.
3.  **Persistence:** Scheduled Task/Job.
4.  **Privilege Escalation:** Exploiting Vulnerabilities.
5.  **Impact:** Data Destruction or Process Disruption.
**Use:** Map defenses to tactics to find gaps (e.g., no email filtering = Initial Access gap).

**OT20**
**SOC:** SIEM collects logs (passive). OT-specific rules detect anomalies (e.g., PLC write).
**Staffing:** IT analysts for IT logs, OT engineers for OT alerts.
**Firmware Change:** Alert triggers immediate investigation. Stop production if unsafe. Revert firmware. Investigate root cause.
**False Positives:** Tune rules to reduce noise (e.g., allow maintenance windows).
