SC1-A
The failure stems from the interaction between the **Restricted Pod Security Standard (PSS)** and the container image configuration.
1.  **Restricted PSS:** Unlike Baseline, Restricted forbids privileged containers, forbids `runAsUser: 0` (root), and requires `runAsNonRoot: true`. It also restricts capabilities and volume mounts.
2.  **Image USER Directive:** The new base image (`cgr.dev/chainguard/static:latest`) defaults `USER` to `root` (UID 0).
3.  **The Conflict:** The pod spec sets `securityContext.runAsNonRoot: true` but omits `runAsUser`. Kubernetes attempts to run the container as the image's default user (root). The Restricted PSS admission controller rejects this because running as root violates the policy, even if `runAsNonRoot` is true (which implies "do not run as root", but without an explicit UID, the image defaults to 0).
4.  **Result:** The kubelet rejects the pod creation or the container crashes immediately upon startup due to security policy enforcement, resulting in CrashLoopBackOff.

SC1-C
**Remediation Strategy:**
1.  **Immediate Fix (Image):** Rebuild the container image with a non-root user. Add `USER 1000` (or similar) to the Dockerfile. Ensure the application can run with these permissions.
2.  **Manifest Fix:** Update the Deployment manifests to explicitly set `securityContext.runAsUser: 1000` alongside `runAsNonRoot: true`.
3.  **CI Pipeline Gate:** Add a step in the CI pipeline using `trivy` or `kube-score` to scan images for `USER root` and reject builds if found. Add a policy check in the deployment pipeline (e.g., OPA Gatekeeper) to enforce `runAsUser` is set and non-zero.
4.  **Rollout:** Use a **Canary Deployment**. Deploy the fixed version to 10% of traffic first. Monitor error rates and latency. If stable, gradually increase to 100%. This ensures the fix works without impacting the entire user base immediately.

SC2-A
**Stale Data & Replication Lag:**
1.  **Cause:** Cross-region asynchronous replication introduces latency. When the primary (us-east-1) fails, the replica (eu-west-1) is promoted. Any WAL (Write-Ahead Log) entries generated in us-east-1 *after* the last successful replication to eu-west-1 are lost.
2.  **PostgreSQL Streaming:** WAL is shipped asynchronously. Promoting a replica cuts the link to the primary. Queries issued immediately after failover may read data from the replica's state before the last committed transaction was shipped.
3.  **Data Loss:** The window of loss equals the replication lag at the moment of failover.
4.  **Quantification:** Check `pg_stat_replication` on the primary before failure (look at `sent_lsn`, `write_lsn`, `flush_lsn`, `replay_lsn`). Calculate the difference between `write_lsn` and `replay_lsn` to estimate the lag in bytes/time.

SC2-B
1.  **Import Replica:** Import the promoted EU instance into the Terraform state using the existing resource name (to avoid recreation).
    ```bash
    terraform import aws_db_instance.primary_rds <eu-instance-id>
    ```
2.  **Update State:** If the resource name in code is `primary_rds` but the ARN changed, update the state to reflect the new ID.
    ```bash
    terraform state mv aws_db_instance.primary_rds aws_db_instance.promoted_replica
    ```
3.  **Detach Old:** Remove the old US instance from state (it is now orphaned/terminated).
    ```bash
    terraform state rm aws_db_instance.primary_rds
    ```
4.  **Update References:** Update Terraform variables or outputs that reference the old ARN to point to the new EU instance ID/ARN. Run `terraform apply` to update dependent resources (Security Groups, Route53) to the new ARN.

SC2-C
**Architecture Design:**
1.  **Aurora Global Database:** Recommend Aurora Global DB over standard RDS. It offers sub-second replication lag (1 second RPO) across regions, automatic failover, and a single logical database. Standard RDS async replication is too slow for <1s RPO.
2.  **Connection Management:** Use **RDS Proxy** instead of PgBouncer. RDS Proxy handles connection pooling centrally and supports failover transparently to the application, reducing connection churn during failover.
3.  **DNS Failover:** Use **Route53 Latency-Based Routing** with Health Checks. Point the application endpoint to a Route53 Alias record. If us-east-1 fails health checks, Route53 routes traffic to eu-west-1.
4.  **Application Logic:** Implement exponential backoff with jitter in the connection pooler and application retry logic to handle transient connection errors during the failover window.

SC3-C
**Hardened Pipeline:**
1.  **Image Signing:** Use **Cosign** to sign images upon build. `cosign sign --key <key> <image>`.
2.  **Admission Control:** Deploy **Kyverno** or **OPA Gatekeeper** in the cluster to enforce `verify-image-signatures`. Pods cannot start without a valid signature.
3.  **Base Image Pinning:** Pin base images by **digest** (e.g., `node:20-slim@sha256:...`) in the Dockerfile, not tag. This prevents tag swapping attacks.
4.  **SBOM:** Generate Software Bill of Materials using **Syft** during build and store in artifact registry.
5.  **Notary v2:** Notary v2 (part of Sigstore) provides the trust infrastructure for signing and verifying. It integrates with Cosign to ensure the image hasn't been tampered with since signing.

SC4-A
**Top 5 Blockers & Solutions:**
1.  **Local Sessions (Sticky):** Kubernetes pods are ephemeral.
    *   *Solution:* Move sessions to Redis (ElastiCache) or a distributed session store.
2.  **Local NFS Mounts:** Pods cannot share local filesystems across nodes.
    *   *Solution:* Use **ReadWriteMany (RWX)** StorageClass (e.g., EFS or NFS CSI) to mount shared storage.
3.  **Stateful Oracle DB:** Oracle is stateful and licensed per core.
    *   *Solution:* Keep Oracle on-prem or use RDS/Aurora. If on-prem, use DB Proxy or JDBC connection pooling to handle network latency.
4.  **IBM MQ Integration:** External dependency with specific network requirements.
    *   *Solution:* Use a sidecar adapter or dedicated MQ queue manager pods with persistent volumes for local queues, or expose MQ via a secure API gateway.
5.  **F5 Load Balancer:** F5 handles health checks and sticky sessions.
    *   *Solution:* Replace F5 with Kubernetes Ingress Controller (e.g., NGINX or ALB Ingress) and Service Mesh for traffic management.

SC5-C
**Architectural Solution:**
1.  **CQRS:** Separate the reconciliation job (Write/Read heavy) from the checkout service (Read heavy).
2.  **Read Replica:** Run the reconciliation job against a **PostgreSQL Read Replica**. This prevents locks on the primary DB used by checkout.
3.  **Bulkhead Isolation:** Configure separate connection pools for the reconciliation job vs. serving pods. Use Kubernetes ResourceQuotas to limit DB connections per namespace.
4.  **CronJob Scheduling:** Move the CronJob to off-peak hours or use a Job with a backoff limit to prevent repeated failures.
5.  **PDBs:** Define a `PodDisruptionBudget` for the inventory-service pods to ensure at least 1 pod remains available during voluntary disruptions (like node maintenance), preventing total starvation.

**Connectivity Upgrade:**
1.  **AWS Direct Connect:**
    *   *Pros:* Dedicated 10Gbps+ bandwidth, lower latency, consistent throughput.
    *   *Cons:* High cost, physical provisioning.
    *   *Zero-Trust:* Fits well; traffic is encrypted (IPsec) and private.

    *   *Principle:* Sensitive Data Exposure.

SC10-C
**Automated Failover Orchestration:**
1.  **Detection:** Prometheus monitors site health (ping, DB latency). If Chicago is unreachable for > 30s, trigger alert.
2.  **DNS Failover:** GSLB (Route53) detects health check failure and switches DNS to Dallas.
3.  **DB Promotion:** Patroni detects primary loss and promotes Dallas node automatically (Synchronous commit ensures no data loss).
4.  **Redis/RabbitMQ:** Sentinel/Federation detects master loss and promotes replica.
5.  **MinIO:** Site replication catches up.
6.  **State Update:** Ansible/Terraform updates OpenTofu state to mark Dallas as Primary.
7.  **Split-Brain:** Use **Quorum** (Patroni etcd). If Chicago is up but partitioned, it steps down to avoid split-brain.
8.  **Failback:** Once Chicago recovers, re-establish replication. Promote Chicago back to Primary manually or via automated script after data sync verification.
