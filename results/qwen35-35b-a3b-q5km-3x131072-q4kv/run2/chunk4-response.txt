G1 — **Slice vs Array:** An array is a fixed-size value type `[T]N`. A slice is a dynamic view over an array, consisting of a pointer, length, and capacity `[T]`. **Append to full capacity:** Go allocates a new backing array (typically doubling capacity), copies existing elements, and returns a new slice header pointing to the new array.

G2 — **`var x int` vs `x := 0`:** `var x int` declares a variable with explicit type and zero value. `x := 0` is a short variable declaration (inferred type). **Cannot use `:=`:** When redeclaring a variable in the same scope, or when the type must be explicit (e.g., `var x int = 0` is valid, `x := 0` is not if `x` exists).

G3 — **Goroutine vs Thread:** A goroutine is a lightweight thread managed by the Go runtime (stack starts small, grows/shrinks). An OS thread is managed by the kernel (fixed stack size). **Scheduler (GMP):** Maps Goroutines (G) to OS Threads (M) via Processors (P). The scheduler multiplexes Gs onto Ms, handling blocking syscalls and load balancing.

G4 — **Channels:** Channels are typed pipes for communication. **Buffered vs Unbuffered:** Buffered has a queue size; unbuffered blocks sender until receiver is ready. **Send to full buffered:** The sender blocks until a receiver reads from the channel.

G5 — **Context:** Used for cancellation, deadlines, and request-scoped values. **Example:**
```go
ctx, cancel := context.WithTimeout(context.Background(), 1*time.Second)
defer cancel()
resp, err := http.GetContext(ctx, "http://example.com")
```

G6 — **Interface Satisfaction:** Go uses implicit satisfaction (duck typing); no `implements` keyword. Java/C# require explicit declaration. **Empty Interface (`interface{}`/`any`):** Holds any type. Powerful for generic containers, dangerous because type assertions can panic if types don't match, losing compile-time safety.

G7 — **Memory Model:** Go guarantees visibility only via synchronization. **Unsafe:** The loop `for !ready {}` creates a data race; the writer goroutine may cache `data` in a register, and the reader may not see the write to `data` before `ready` flips. **Fix:** Use `sync/atomic` for the flag or a `sync.Mutex` to protect `data`.

G8 — **Mutex vs RWMutex:** `Mutex` is exclusive. `RWMutex` allows multiple readers or one writer. **`sync.Map`:** Optimized for read-heavy workloads or keys that are deleted. **Trade-offs:** `sync.Map` has higher memory overhead and slower writes than a `map` + `Mutex` due to internal sharding and atomic operations.

G9 — **Modules:** `go.mod` defines module path and dependencies. `go.sum` stores checksums for verification. **`go mod tidy`:** Removes unused imports and adds missing dependencies. **MVS:** Selects the lowest version of a dependency that satisfies all constraints in the dependency tree.

G10 — **Defer:** Delays function execution until the surrounding function returns. **Order:** LIFO (Last In, First Out). **Loop Gotcha:** Deferring a closure inside a loop captures the loop variable by reference; all defers see the final value of the variable. Fix by creating a new variable inside the loop.

G11 — **Pointer:** A memory address. **Zero value:** `nil`. **Struct literal pointer:** `&MyStruct{Field: value}`.

G12 — **`init()`:** Runs before `main()` during package initialization. **Multiple `init()`:** Yes. They execute in source file order, then package dependency order.

G13 — **`make` vs `new`:** `make` allocates and initializes map, slice, channel (returns typed value). `new` allocates zeroed memory (returns pointer). `make` works only with built-in types; `new` works with any type.

G14 — **Error Handling:** Errors are values implementing `error` interface (`Error() string`). **Idiom:** `if err != nil { return err }`. **`errors.Is` vs `errors.As`:** `Is` checks logical equality (wrapping). `As` performs type assertion to extract specific error types. `==` only works for exact type matches.

G15 — **Composite Types:**
*   **Map:** Zero value is `nil`.
*   **Slice:** Zero value is `nil`.
*   **Channel:** Zero value is `nil`.
*   **Struct:** Zero value has all fields set to their zero values.

G16 — **Race Condition:** Unsynchronized concurrent access to shared data. **`-race` flag:** Instrumented build detects data races at runtime. **Fix:** Use `sync.Mutex` to lock access or channels to pass ownership.

G17 — **Select:** Multiplexes channel operations. **vs Switch:** Select waits for channel readiness; switch matches values. **Multiple ready:** Randomly chosen. **Timeout:** Add a `case <-time.After(duration)` to the select.

G18 — **Escape Analysis:** Compiler determines if a variable lives on stack or heap. **Heap:** If pointer escapes function scope. **Inspect:** `go build -gcflags="-m"`. **Performance:** Heap allocations trigger GC; stack is faster.

G19 — **Reflection:** Runtime introspection. **Type vs Value:** `Type` is metadata (kind, methods); `Value` is the actual data container. **Cost:** Slow, no compile-time checks. **Justified:** Frameworks (JSON, ORM) where code generation is too complex.

G20 — **`go:embed`:** Embeds files at compile time. **Usage:** `//go:embed dir/*`. **Limitations:** Static at compile time, no symlinks on some OSs. **`io/fs.FS`:** Returns an `embed.FS` implementing `fs.FS` interface for standard file operations.

R1 — **Ownership:** Resource management without GC. **Rules:** 1. Each value has one owner. 2. When owner goes out of scope, value is dropped. 3. Values can be moved, not copied (unless `Clone`).

R2 — **`String` vs `&str`:** `String` is owned, heap-allocated, mutable. `&str` is a borrowed slice of bytes (immutable). **Signatures:** Use `&str` for arguments (no allocation), `String` for owned data.

R3 — **Borrowing:** References to data. **Rules:** 1. Immutable refs (`&T`) can coexist. 2. Mutable ref (`&mut T`) is exclusive. 3. Cannot have mutable and immutable refs simultaneously.

R4 — **`Result`/`Option`:** `Result<T, E>` (Ok/Err), `Option<T>` (Some/None). **`?` operator:** Propagates error/None up the call stack. **Example:**
```rust
fn read_file(path: &str) -> Result<String, std::io::Error> {
    std::fs::read_to_string(path)
}
```

R5 — **Box/Rc/Arc:** `Box<T>`: Heap allocation, single owner. `Rc<T>`: Reference counted, single thread. `Arc<T>`: Atomic reference counted, thread-safe. Use `Arc` for shared data across threads.

R6 — **Traits:** Interfaces. **vs Go:** Go is implicit; Rust is explicit (`impl Trait for Type`). **`dyn Trait` vs `T: Trait`:** `dyn Trait` is dynamic dispatch (vtable, heap). `T: Trait` is static dispatch (monomorphization, inline).

R7 — **Lifetimes:** Scope of references. **Fails:** Compiler cannot guarantee which input (`x` or `y`) outlives the other. **Fix:**
```rust
fn longest<'a>(x: &'a str, y: &'a str) -> &'a str {
    if x.len() > y.len() { x } else { y }
}
```

R8 — **Send/Sync:** `Send` = transferable across threads. `Sync` = referenceable across threads. **`Rc` not `Send`:** Ref count is not atomic. **`Arc` safe:** Uses atomic ref counting for thread-safe sharing.

R9 — **Async/Await:** State machine. **Future:** Pending computation. **Executor:** Polls futures. **vs Go:** Go uses green threads (stackful). Rust uses stackless futures (state machine), requiring explicit executor (e.g., Tokio).

R10 — **Unsafe:** Bypasses safety checks. **3 things:** Raw pointer dereference, FFI, union access. **Justified:** FFI, performance-critical low-level ops, interfacing with hardware.

R11 — **Pattern Matching:** `match` is exhaustive (compiler enforces all cases). `if let` is partial. **Exhaustiveness:** `match` requires a catch-all (`_`) or all variants covered.

R12 — **Vec vs Array:** `Vec` is dynamic heap array. `[T; N]` is fixed stack array. **Vec Capacity:** When full, allocates new larger buffer, copies elements, drops old buffer.

R13 — **Enums:** Algebraic Data Types (variants with data). **vs C/Java:** Rust enums can hold data per variant. **`Option<T>`:** `Some(T)` or `None`. **No null:** Compile-time safety prevents null pointer dereferences.

R14 — **`impl` block:** Defines methods. **`&self`:** Borrow. **`&mut self`:** Mutate. **`self`:** Consume. **Associated functions:** `fn new()` (no instance needed, often `Self::new()`).

R15 — **Derive Macro:** Generates trait implementations. **Common:** `Debug`, `Clone`, `PartialEq`. **Cannot derive:** If fields contain types that don't implement the trait (e.g., `Fn` closures).

R16 — **`Cow<'a, T>`:** Clone on Write. **Use:** Avoids allocation if data is not modified. **Example:** Parsing config; if no changes, return borrowed `&str`. If changed, clone to owned `String`.

R17 — **Module System:** `mod` declares module, `use` imports, `pub` exports. **File System:** `mod.rs` or `mod.rs` in directory. **2015 vs 2018:** 2018 allows `use` paths relative to crate root (e.g., `crate::`), simplifying imports.

R18 — **`Pin<T>`:** Prevents moving memory. **Needed:** For self-referential structs (common in async await where `&mut self` is stored). **`Unpin`:** Marker trait indicating type can be moved safely.

R19 — **Macros:** Code generation. **`macro_rules!`:** Declarative (pattern matching). **Proc Macros:** Rust code (derive, attribute, function-like). **Hygiene:** Prevents variable capture; macro variables don't leak into caller scope.

R20 — **Zero-Cost Abstractions:** No runtime overhead. **Iterators:** Loop unrolled at compile time. **Generics:** Monomorphization (code copied per type). **Traits:** Static dispatch (vtable only for `dyn`). Performance matches C.
