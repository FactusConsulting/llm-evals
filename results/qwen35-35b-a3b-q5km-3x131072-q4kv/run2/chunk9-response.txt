SC1-A
The pods are failing due to a conflict between the **Restricted Pod Security Standard (PSS)** and the container image configuration.
1.  **Restricted PSS:** Requires `runAsNonRoot: true` AND explicitly sets `runAsUser` to a non-zero UID (or the image must define a non-root USER). It forbids privileged containers and requires specific capabilities.
2.  **Image Change:** The new base image (`cgr.dev/chainguard/static:latest`) defaults `USER` to `root` (UID 0).
3.  **The Conflict:** The namespace enforces `runAsNonRoot: true` but lacks an explicit `runAsUser`. Kubernetes defaults `runAsUser` to 0 if not specified. Since the image also runs as root, the admission controller rejects the pod creation (or the kubelet refuses to start it) because it violates the "non-root" requirement of the Restricted profile.
4.  **Baseline vs. Restricted:** Baseline allows `runAsNonRoot: true` without enforcing a specific UID or forbidding root if the image defines it. Restricted enforces non-root execution strictly.

SC1-C
**Remediation Strategy:**
1.  **Immediate Fix:** Update the container image to a non-root user (e.g., `cgr.dev/chainguard/static:latest@sha256:...` with a specific user) or update the Pod spec to include `securityContext.runAsUser: 1000`.
2.  **Manifest Change:** Add `securityContext.runAsUser: 1000` to all affected Deployment/StatefulSet specs to ensure compliance with Restricted PSS regardless of image defaults.
3.  **CI Gate:** Implement an admission controller (OPA Gatekeeper or Kyverno) in CI/CD that validates manifests against the Restricted PSS policy before merging. Add a Trivy/Grype scan step to verify the base image user.
4.  **Rollout:** Use a **Canary Deployment**. Update 10% of pods first. Monitor CrashLoopBackOff rates and health checks. If stable, increment to 50%, then 100%. This prevents a full outage if the fix is incomplete.

SC2-A
**Stale Data Issue:**
1.  **Replication Lag:** Cross-region asynchronous replication introduces latency. Promoting a replica before WAL shipping completes causes data loss (the "data loss window").
2.  **WAL Shipping:** PostgreSQL ships Write-Ahead Logs (WAL) asynchronously. The primary acknowledges writes before the replica receives them.
3.  **Data Loss:** Any transactions committed on the primary after the last WAL record shipped to the replica are lost upon failover.
4.  **Quantification:** Check `pg_stat_replication` on the primary before failover. Look at `write_lag`, `flush_lag`, and `replay_lag`. The difference between the primary's `pg_current_wal_lsn()` and the replica's `pg_last_wal_receive_lsn()` quantifies the potential loss.

SC2-B
1.  **Import Replica:** `terraform import aws_db_instance.promoted_replica <new_replica_arn>`
2.  **Update State:** If the old instance is still in state, move it: `terraform state mv aws_db_instance.old_primary aws_db_instance.promoted_replica` (or remove old and import new).
3.  **Update Variables:** Update the Terraform variable file (e.g., `prod.tfvars`) to point to the new ARN.
4.  **Apply:** `terraform plan` to verify no destructive changes, then `terraform apply`.
5.  **Detach Old:** If the old instance is no longer needed, `terraform destroy` the old resource (ensure it's not the active DB).
*Crucial:* Ensure `skip_destroy` is set on the old instance during the transition to prevent accidental deletion during state drift.

SC2-C
**Architecture Design:**
1.  **RDS Proxy:** Use RDS Proxy instead of PgBouncer. It handles connection pooling at the AWS layer, reducing connection churn during failover and supporting IAM auth.
2.  **Route53:** Use **Latency-Based Routing** with Health Checks. Point the application endpoint to a Route53 Alias record. If us-east-1 fails, Route53 routes to eu-west-1.
3.  **App Logic:** Implement exponential backoff with jitter for retries. Detect `DatabaseConnectionError` and switch to a secondary endpoint.
4.  **RPO < 1s:** Standard RDS async replication cannot guarantee <1s RPO. **Recommend Aurora Global Database.** It uses a dedicated replication network with sub-second replication latency and automatic failover (usually <60s).
5.  **Failover:** Aurora Global DB handles promotion automatically. RDS Proxy updates DNS endpoints automatically.

5.  **F5 Load Balancer:** F5 is hardware. **Fix:** Replace with Kubernetes Ingress Controller (NGINX/ALB) and use Service Mesh for traffic management.

SC6-C
**Comparison:**
1.  **AWS Direct Connect:**
    *   *Pros:* Dedicated bandwidth (10Gbps+), lower latency, consistent throughput.
    *   *Cons:* High cost, hardware provisioning.
    *   *Zero Trust:* Fits well (private link).
2.  **ECMP VPN:**
    *   *Pros:* Cheaper, uses existing internet.
    *   *Cons:* Shared bandwidth, variable latency, encryption overhead.
    *   *Zero Trust:* Less secure (public internet).
3.  **Mesh (Tailscale/Netbird):**
    *   *Pros:* Easy setup, identity-based access, encrypted.
    *   *Cons:* Not for high-throughput data transfer (200TB).
    *   *Zero Trust:* Excellent for management, poor for bulk data.
**Recommendation:** Use **Direct Connect** for data plane (DB/Storage sync) and **Mesh** for management plane. Update routing to use BGP over DX for dynamic failover.

SC10-A
**Architecture Comparison:**
1.  **Active-Passive (Velero):**
    *   *RPO/RTO:* RPO depends on backup frequency (15m). RTO depends on restore time (1h).
    *   *Feasibility:* Meets requirements.
    *   *State:* MinIO needs Site Replication. DB needs Async Replication.
2.  **Active-Active (Istio/Cilium):**
    *   *RPO/RTO:* Near zero RPO (sync), RTO < 1m.
    *   *Feasibility:* Harder to maintain consistency across 12ms latency.
    *   *State:* Requires distributed DB (CockroachDB) or strong sync (Patroni Sync).
**Recommendation:** **Active-Passive** for simplicity and compliance. Use **Patroni Sync** for DB (RPO=0), **Redis Sentinel** for cache, **RabbitMQ Federation** for queues, **MinIO Site Replication** for storage.

SC10-C
**Failover Orchestration:**
1.  **Detection:** Prometheus monitors site health (ping, DB latency). If Chicago > 2m down, trigger alert.
2.  **DNS:** GSLB (Route53) switches DNS to Dallas endpoint.
3.  **DB:** Patroni promotes Dallas node (Synchronous commit ensures no data loss).
4.  **Cache/Queue:** Redis Sentinel promotes new master. RabbitMQ Federation syncs.
5.  **Storage:** MinIO Site Replication catches up (eventual consistency).
6.  **State:** Ansible/Terraform updates state to mark Dallas as Primary.
**Split-Brain:** Use **etcd quorum**. If Chicago etcd is down, Dallas cannot promote unless it has quorum. Use **fencing** (power off Chicago nodes) to prevent split-brain.
**Failback:** Once Chicago is up, demote Dallas, sync data, and switch DNS back.
