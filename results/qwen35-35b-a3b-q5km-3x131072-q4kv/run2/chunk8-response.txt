CL1 — **IaaS, PaaS, SaaS**
*   **IaaS (Infrastructure as a Service):** Provides virtualized computing resources (servers, storage, networking). You manage OS, middleware, runtime, data, and apps. *Example:* AWS EC2.
*   **PaaS (Platform as a Service):** Provides hardware and software tools (usually for app development) over the internet. You manage data and apps. *Example:* AWS Elastic Beanstalk.
*   **SaaS (Software as a Service):** Delivers software applications over the internet. You manage nothing but user access. *Example:* Microsoft 365.

CL2 — **Shared Responsibility Model**
*   **Concept:** Security is a joint effort. Provider secures the cloud; you secure what you put in the cloud.
*   **IaaS VM:** Provider secures physical hardware, hypervisor, and network. You secure OS, patching, firewall config, and data.
*   **Managed Kubernetes:** Provider secures control plane and underlying nodes. You secure worker node OS, K8s config, network policies, and application code.
*   **SaaS (M365):** Provider secures everything including app, data, and infrastructure. You secure identity (MFA), data classification, and user permissions.

CL3 — **VPC, Subnet, SG, NACL**
*   **VPC:** Isolated virtual network in the cloud.
*   **Subnet:** Range of IP addresses within a VPC (can be public or private).
*   **Security Group (SG):** Stateful firewall attached to an ENI (instance). Allows whitelisting traffic.
*   **Network ACL (NACL):** Stateless firewall attached to a Subnet. Allows/denies traffic at subnet entry/exit.
*   **Layering:** VPC contains Subnets. Subnets contain NACLs. Instances within Subnets have SGs.
*   **Stateful vs Stateless:** SG is stateful (return traffic allowed automatically). NACL is stateless (must explicitly allow return traffic).

CL4 — **Multi-Account Strategy**
*   **Landing Zone:** Secure, compliant foundation for cloud accounts.
*   **Organizational Units (OUs):** Logical grouping of accounts (e.g., Prod, Dev, Security).
*   **Guardrails:** SCPs (Service Control Policies) to prevent risky actions (e.g., disabling CloudTrail).
*   **Shared Networking:** Transit Gateway (AWS) or Hub-and-Spoke (Azure) connects accounts centrally for traffic flow.
*   **Centralized Logging:** Aggregates logs to a dedicated Security/Logging account for analysis.
*   **Single-Account Problems:** High blast radius (one breach compromises all), billing complexity, lack of isolation for compliance, shared resource contention.

CL5 — **Infrastructure as Code (IaC)**
*   **Concept:** Managing infrastructure via machine-readable definition files rather than physical hardware configuration.
*   **Terraform/OpenTofu:** Declarative, multi-cloud, state-based. Best for multi-cloud or heterogeneous environments.
*   **Pulumi:** Imperative/Declarative using general languages (TS, Python). Best for devs comfortable with code.
*   **CloudFormation/Bicep:** Native to AWS/Azure. Best for deep integration with specific cloud features and CI/CD pipelines.
*   **Crossplane:** Kubernetes-native IaC. Best if you want to manage cloud resources via K8s API.

CL6 — **Autoscaling Patterns**
*   **HPA:** Scales pods based on CPU/Memory metrics.
*   **VPA:** Scales pod resource requests/limits (requires pod restart).
*   **Cluster Autoscaler:** Adds/removes nodes based on pending pod resource needs.
*   **KEDA:** Event-driven autoscaler (scales based on external sources like Kafka, SQS).
*   **KEDA vs HPA:** Use KEDA when load is driven by external events (e.g., queue depth) rather than internal resource usage (CPU).

CL7 — **Disaster Recovery (DR) Strategy**
*   **RPO/RTO:** RPO (max data loss), RTO (max downtime).
*   **Active-Active:** Both regions serve traffic. Low RTO/RPO, high cost, complex data sync.
*   **Active-Passive:** One region active, one standby. Higher RTO, lower cost.
*   **Database:** Async replication (performance, eventual consistency) vs Sync (consistency, latency).
*   **DNS Failover:** Route53 Health Checks trigger failover to secondary region.
*   **State Management:** Externalize state (S3, DynamoDB) to avoid local storage dependency.
*   **Models:** Pilot Light (min infra running), Warm Standby (scaled down prod), Multi-site Active-Active (full redundancy).

CL8 — **GitOps Model**
*   **Concept:** Git repository is the single source of truth for infrastructure and app state.
*   **ArgoCD vs Flux:** ArgoCD pushes changes (push model), Flux pulls changes (pull model). Both sync cluster state to Git.
*   **Secrets:** Use Sealed Secrets (encrypted in Git) or External Secrets Operator (fetch from Vault/AWS Secrets Manager).
*   **Drift:** If manual changes occur, GitOps controllers detect drift and revert the cluster to match the Git state automatically.

CL9 — **Stateful Legacy App to Kubernetes**
*   **StatefulSet:** Ensures stable network IDs and ordered deployment/termination.
*   **PV/PVC:** Use `local` storage class for low latency (hostPath) or network storage (NFS) if shared.
*   **Headless Service:** Provides stable DNS names for StatefulSet pods.
*   **Init Containers:** Handle config file generation or DB initialization before main app starts.
*   **ConfigMaps:** Mount config files as volumes at specific paths.
*   **PDB:** Ensure minimum available pods during updates to prevent downtime.

CL10 — **Identity (SA, Workload, IRSA)**
*   **Service Account:** K8s identity for pods (internal).
*   **Workload Identity:** GCP/Azure native federation (K8s SA -> Cloud Identity).
*   **IRSA:** AWS specific (K8s SA -> IAM Role via OIDC).
*   **Mounting Credentials:** Bad practice (leaks keys, hard to rotate, violates least privilege).
*   **Federation:** Uses OIDC tokens to assume IAM roles dynamically without long-lived static keys.

CL11 — **Storage Types**
*   **Object (S3/Blob):** Unstructured data, HTTP access. High durability (11 9s). Use for backups, media.
*   **Block (EBS/Managed Disk):** Raw storage, attached to VM. Low latency. Use for OS, DBs.
*   **File (EFS/Azure Files):** Shared file system (NFS/SMB). Use for shared configs, legacy apps.
*   **Durability:** Object storage typically offers highest durability guarantees.

CL12 — **Zero-Trust Network Architecture**
*   **Network Policies:** Restrict pod-to-pod traffic (L3/L4).
*   **Service Mesh (mTLS):** Encrypts service-to-service traffic (L7).
*   **Pod Identity:** Assign specific roles to pods (no shared node identity).
*   **Admission Controllers (OPA/Kyverno):** Enforce policies on deployment (e.g., no root user).
*   **Image Signing:** Verify container images before deployment.
*   **Runtime Security (Falco):** Detects anomalous behavior (e.g., shell in container).
*   **Diff:** Traditional = Trust inside perimeter. Zero Trust = Verify every request, assume breach.

CL13 — **CDN**
*   **Concept:** Distributed network of servers caching content closer to users.
*   **Services:** AWS CloudFront, Azure CDN.
*   **Usage:** Use CDN for static assets (images, CSS) to reduce latency and origin load. Use origin for dynamic/private data.

CL14 — **Availability Zones & Regions**
*   **Region:** Geographic area (e.g., us-east-1).
*   **AZ:** Isolated data centers within a Region (power, cooling, network).
*   **Multi-AZ:** Deploy across AZs to survive data center failure.
*   **Zonal vs Regional:** Zonal resources (e.g., EBS) tied to one AZ. Regional resources (e.g., S3, RDS Multi-AZ) span AZs.

CL15 — **Managed Database**
*   **Definition:** DB service where provider manages hardware/OS.
*   **Provider Tasks:** Patching, backups, failover, scaling, monitoring.
*   **User Tasks:** Schema design, queries, data modeling, access control.

CL16 — **Cloud Identity (IAM)**
*   **Users:** Human identities.
*   **Groups:** Collections of users for permission management.
*   **Roles:** Temporary credentials for services/users (no password).
*   **Policies:** JSON rules defining permissions.
*   **Least Privilege:** Grant only necessary permissions. Apply via roles/groups, not individual users.

CL17 — **Serverless Computing**
*   **Definition:** Event-driven code execution without managing servers.
*   **Services:** AWS Lambda, Azure Functions, GCP Cloud Functions.
*   **Cold Starts:** Latency when a function is invoked after inactivity (initialization).
*   **Poor Fit:** Long-running processes, high CPU usage, stateful apps, strict latency requirements.

CL18 — **Kubernetes Ingress Options**
*   **Ingress Resource:** Standard K8s resource for HTTP routing.
*   **Gateway API:** New standard, more flexible, supports multiple protocols.
*   **Cloud LB Annotations:** Cloud provider specific LB config.
*   **Mesh Ingress:** Service mesh entry point (e.g., Istio).
*   **TLS:** Terminates at LB (Edge), Ingress Controller, or Mesh.
*   **Gateway API:** Use for complex routing, multi-cluster, or when Ingress limitations are hit.

CL19 — **Cost Optimization Strategy**
*   **Compute:** Reserved Instances (1-3yr commitment), Savings Plans (flexible commitment), Spot (interruptible, cheap).
*   **Right-sizing:** Match instance size to actual usage metrics.
*   **Storage:** Lifecycle policies (S3 Intelligent-Tiering, Archive).
*   **Idle Detection:** Tagging and reporting unused resources (EBS, IPs).
*   **FinOps:** Culture of cost accountability, showback/chargeback to teams.
*   **Tools:** AWS Cost Explorer, CloudHealth, Kubecost.

CL20 — **Multi-Cloud Strategy**
*   **Abstraction:** Kubernetes (EKS/AKS) as common runtime.
*   **IaC:** Terraform for portability (avoid provider-specific resources).
*   **Networking:** Direct Connect/ExpressRoute or VPN for inter-cloud traffic.
*   **Identity:** Federation (OIDC) across clouds.
*   **Trade-offs:** Avoid lock-in vs increased complexity/operational overhead.
*   **Justified:** Regulatory requirements, geo-redundancy, vendor negotiation leverage.
*   **Unnecessary:** Simple apps, high cost of management outweighs benefits.

OT1 — **Purdue Model (ISA-95)**
*   **Level 0:** Physical Process (Sensors, Actuators).
*   **Level 1:** Basic Control (PLCs, RTUs).
*   **Level 2:** Supervisory Control (SCADA, HMIs).
*   **Level 3:** Operations (MES, Historians).
*   **Level 4:** Enterprise (ERP, Business Systems).

OT2 — **IT vs OT Networks**
*   **IT:** Confidentiality focus, frequent patching, TCP/IP heavy.
*   **OT:** Safety/Availability focus, stability over security, proprietary protocols.
*   **Patching:** OT patching risks downtime or safety hazards; often requires vendor approval and maintenance windows.

OT3 — **DMZ (IDMZ)**
*   **Role:** Buffer zone between IT (L4) and OT (L0-L3).
*   **Traffic:** Pull-based (OT pulls data from IT). Push from IT to OT is restricted.
*   **Danger:** Direct L4->L1 connectivity allows lateral movement from corporate network to critical control systems, risking physical safety.

OT4 — **OT Protocols**
*   **Modbus TCP:** L1/L2, Master/Slave, simple.
*   **OPC UA:** L2/L3, Secure, platform-independent.
*   **EtherNet/IP (CIP):** L1/L2, Allen-Bradley dominant.
*   **PROFINET:** L1/L2, Real-time industrial Ethernet.
*   **MQTT:** L2/L3, Lightweight, publish/subscribe for IoT/Cloud.

OT5 — **Data Diode vs Firewall**
*   **Data Diode:** Hardware-enforced one-way data flow.
*   **Firewall:** Two-way traffic filtering.
*   **Use Case:** Diode for exporting data from secure OT to IT without any return path risk.
*   **Limitation:** Cannot send commands back to OT; no feedback loop.

OT6 — **Data Flow to Cloud**
*   **Flow:** PLC -> Historian (L3) -> Edge Gateway -> Cloud Ingestion.
*   **Security:** Edge Gateway acts as IDMZ. TLS encryption for cloud transmission.
*   **Boundaries:** Firewall between Edge and Cloud. Network segmentation between PLC and Historian.

OT7 — **IEC 62443 Zones & Conduits**
*   **Zone:** Group of assets with similar security requirements.
*   **Conduit:** Logical path for traffic between zones.
*   **Design:**
    *   Zone 1: Corporate (L4).
    *   Zone 2: Engineering (L3).
    *   Zone 3: SCADA/Control (L2).
    *   Zone 4: PLC/SIS (L1).
    *   Conduits: Firewalls between zones, strictly controlled traffic flow.

OT8 — **Network Segmentation Strategy**
*   **VLANs:** Segment by production line and function (Control vs Monitoring).
*   **Firewalls:** Place between VLANs and at network edge.
*   **Switches:** Managed industrial switches with ACLs.
*   **Remote Access:** Jump hosts with MFA and session recording.
*   **Transition:** Parallel run (new network alongside old), test in staging, migrate during maintenance windows.

OT9 — **SIS vs BPCS**
*   **BPCS:** Basic Process Control System (maintains process).
*   **SIS:** Safety Instrumented System (shuts down process safely on hazard).
*   **IEC 61511:** Standard for Safety Instrumented Systems.
*   **Isolation:** SIS must be air-gapped to prevent cyber attacks from disabling safety functions.
*   **Ransomware on SIS:** Could prevent safe shutdown during emergencies, causing physical damage or injury.

OT10 — **PLC, DCS, SCADA**
*   **PLC:** Programmable Logic Controller (Logic execution, discrete control).
*   **DCS:** Distributed Control System (Process control, continuous, integrated).
*   **SCADA:** Supervisory Control and Data Acquisition (Monitoring, HMI, data collection).
*   **Coexistence:** Yes, common in large plants (PLCs for machines, DCS for process, SCADA for overview).

OT11 — **Remote Access Architecture**
*   **Components:** Jump Host, MFA, Session Recording, Time-limited access.
*   **Vendor Access:** Provisioned via Jump Host, not direct network access.
*   **Unacceptable:** Direct RDP to HMI exposes OT assets to IT threats and lacks audit trails.

OT12 — **Nation-State Incident Response**
*   **Monitoring:** IDS/Anomaly detection on OT traffic.
*   **Inventory:** Identify compromised assets immediately.
*   **Steps:** Isolate network segments, preserve evidence, assess safety impact.
*   **Coordination:** Control room prioritizes physical safety over IT containment.
*   **Operator Role:** Monitor physical processes, execute manual overrides if needed, report anomalies.

OT13 — **HMI vs Engineering Workstation**
*   **HMI:** Human-Machine Interface. Used by operators to monitor/control process in real-time.
*   **Engineering Workstation:** Used by engineers to configure logic, download code, and troubleshoot.
*   **Role:** HMI = Operations; EWS = Maintenance/Configuration.

OT14 — **Historian**
*   **Definition:** Time-series database for process data.
*   **Importance:** Trend analysis, compliance, troubleshooting.
*   **Plant Historian (L3):** Local, real-time, high frequency.
*   **Enterprise Historian (L4):** Aggregated, long-term storage, analytics.

OT15 — **Ladder Logic**
*   **Definition:** Graphical programming language mimicking relay logic.
*   **Device:** PLC.
*   **Dominance:** Visual, easy for electricians to understand, highly reliable, legacy code base.

OT16 — **RTU vs PLC**
*   **RTU:** Remote Terminal Unit. Designed for harsh environments, long-distance comms (cellular/RF).
*   **PLC:** Local control, factory floor.
*   **RTU Use:** Pipelines, water distribution, geographically dispersed assets.

OT17 — **Analog vs Digital Signals**
*   **Analog:** Continuous voltage/current (4-20mA, 0-10V).
*   **Digital:** Discrete on/off states.
*   **4-20mA:** Preferred over 0-20mA because 4mA is "live zero" (detects wire break at 0mA).

OT18 — **OT Asset Inventory**
*   **Foundation:** You cannot secure what you don't know exists.
*   **Tools:** Passive monitoring (sniffing traffic), Active scanning (pinging).
*   **Risk:** Active scanning can crash legacy devices (no TCP stack). Use passive monitoring primarily.

OT19 — **MITRE ATT&CK for ICS**
*   **Tactics:** Reconnaissance, Initial Access, Execution, Persistence, Impact.
*   **Example:** Initial Access (Supply Chain Compromise), Impact (Disrupt Process).
*   **Difference:** Focuses on physical impact and safety, not just data theft.
*   **Usage:** Map defenses to tactics to find gaps (e.g., no detection for PLC logic changes).

OT20 — **Converged IT/OT SOC**
*   **Log Collection:** Syslog/OT-specific agents (non-intrusive).
*   **SIEM:** Centralized correlation of IT and OT logs.
*   **Rules:** OT-specific (e.g., unexpected PLC write).
*   **Staffing:** OT engineers needed to validate alerts (avoid false positives).
*   **Firmware Change Alert:** Stop production, isolate device, verify change authorization, revert if unauthorized.
