K1: A **Deployment** manages stateless applications, creating pods with random names and handling rolling updates. A **StatefulSet** manages stateful applications, providing stable, unique network IDs (hostnames), ordered deployment/termination, and persistent storage guarantees.

K2: A **ClusterIP** Service creates a virtual IP within the cluster. **kube-proxy** (using iptables or IPVS) configures the node's network rules to redirect traffic sent to the Service IP to the backend Pod IPs based on the Service's selector.

K3: A **PersistentVolume (PV)** is a cluster-level storage resource. A **PersistentVolumeClaim (PVC)** is a user request for storage. A **StorageClass** defines the provisioning mechanism (dynamic). A PVC binds to a PV; if a StorageClass is specified, the system auto-provisions a PV to match the PVC.

K4: Troubleshooting CrashLoopBackOff:
1. `kubectl logs <pod>`: Check application error messages.
2. `kubectl describe pod <pod>`: Check events, liveness probe failures, or OOMKilled status.
3. Check resource limits: Ensure memory/CPU limits aren't causing kills.
4. Verify dependencies: Check if the app can connect to DBs or external services.

K5: **CoreDNS** provides DNS resolution for Kubernetes services (Service discovery). If CoreDNS pods are down, internal service discovery (e.g., `my-service.default.svc`) fails. External DNS might still work if the pod uses the host network or fallback, but cluster-internal communication breaks.

K6: The scheduler uses **Filtering** (feasibility) and **Scoring** (preference).
1. **Filtering**: Checks node resources, taints/tolerations, and node selectors.
2. **Scoring**: Ranks nodes based on resource usage (LeastAllocated), affinity rules, and topology spread constraints.
3. **Factors**: CPU/Memory availability, Pod Affinity/Anti-Affinity, Taints/Tolerations.

If it uses 600Mi at runtime: It exceeds the `limits.memory` (512Mi). The OOM Killer will terminate the pod (OOMKilled).

K8: 1. **External LB** forwards traffic to Ingress Controller IP. 2. **Ingress Controller** (NGINX) parses Ingress rules and routes to Service. 3. **Service** resolves to Pod IPs via **Endpoints**. 4. **kube-proxy** (iptables/IPVS) on the node redirects traffic to the specific **Pod IP**.

K9: A **NetworkPolicy** restricts traffic flow between pods. Default behavior (no policy): All traffic is allowed. When a policy is applied: Traffic is denied by default unless explicitly allowed by the policy's `ingress` or `egress` rules.

K10: **RollingUpdate** gradually replaces old pods with new ones (zero downtime). **Recreate** kills all old pods before starting new ones. Choose **Recreate** for stateful applications requiring exclusive access or atomic updates where downtime is acceptable.

K11: A **Namespace** provides logical isolation for resources. Default namespaces: `default`, `kube-system`, `kube-public`, `kube-node-lease`.

K12: `kubectl describe` shows detailed status and events. `kubectl get` shows summary.
Info in `describe` not in `get`:
1. **Events** (warnings/errors).
2. **Conditions** (Ready status details).
3. **Mounts** (Volumes/ConfigMaps attached).

K13: A **ConfigMap** stores non-confidential configuration data. Pods consume it via:
1. **Environment Variables**: Injected into the container.
2. **Volume Mounts**: Mounted as files in the container filesystem.

K14: **ClusterIP**: Internal virtual IP (default). **NodePort**: Exposes service on a static port on each node. **LoadBalancer**: Provisions an external cloud load balancer to access the service.

K15: A **DaemonSet** ensures a copy of a pod runs on all (or specific) nodes. Use instead of Deployment for system-level daemons like logging agents (Fluentd), monitoring (Prometheus Node Exporter), or storage daemons (Ceph).

K16: **Taints** are applied to nodes to repel pods; **Tolerations** are applied to pods to allow them to schedule on tainted nodes. **Node Affinity** attracts pods to nodes.
*Use Case:* Taint a GPU node to repel general pods; use Affinity to spread pods across availability zones.

K17: A **Kubernetes Operator** is a custom controller that manages the lifecycle of a complex application (CRD). It differs from a standard controller by encoding human operational knowledge (e.g., backups, upgrades) into the cluster. It solves the complexity of managing stateful, distributed systems.

K18: etcd uses **strong consistency** (linearizable). If quorum is lost (e.g., 2 of 3 nodes lost), the cluster becomes read-only/unavailable. Recovery: Restore from a snapshot on a new etcd cluster (3 nodes) and update the API server endpoint.

K19: **Role** (namespace-scoped permissions), **ClusterRole** (cluster-wide), **RoleBinding** (links Role to subject in namespace), **ClusterRoleBinding** (links ClusterRole to subject cluster-wide).
*Manifest:*
```yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata: namespace: my-ns
rules:
- apiGroups: [""]
  resources: ["pods", "pods/log"]
  verbs: ["get", "list", "watch"]
---
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata: namespace: my-ns
subjects:
- kind: User
  name: developer
roleRef:
  kind: Role
  name: pod-reader
```

K20: `ndots` defines how many dots must appear in a name to be treated as FQDN. `ndots: 5` means short names trigger queries for every search domain until 5 dots are reached, causing latency. Fix: Set `dnsConfig` in pod spec with `ndots: 1` or empty `search` list.

D1: **Rebase** rewrites history (linear), **Merge** preserves history (branching). Avoid rebasing on shared/public branches as it rewrites history others may rely on.

D2: **TCP** is connection-oriented/reliable (HTTP, SSH). **UDP** is connectionless/fast (DNS, VoIP).

D3: A **race condition** occurs when output depends on the sequence/timing of uncontrollable events. Example: Two threads incrementing a shared counter without a lock, resulting in lost updates.

D4: **Symmetric** uses one key for encryption/decryption. **Asymmetric** uses a public/private key pair. TLS uses Asymmetric for the handshake (key exchange/auth) and Symmetric for the data transfer (performance).

D5: A **layer** is a diff of the filesystem in a Docker image. Layer caching speeds up builds: place stable instructions (install deps) before volatile ones (copy code) so changes to code don't invalidate dependency layers.

D6: Troubleshooting HTTP 503:
1. Check upstream service health (is the backend down?).
2. Check database connectivity.
3. Review application logs for errors.
4. Check resource limits (OOM/CPU throttling).

D7: **CAP Theorem**: Consistency, Availability, Partition Tolerance.
*CP:* Zookeeper (consistency over availability during partition).
*AP:* Cassandra (availability over consistency during partition).

D8: **Horizontal Scaling**: Adding more nodes. **Vertical Scaling**: Adding more resources to a node. Patterns enabling horizontal scaling: Stateless services, sharding, load balancing.

D9: An **environment variable** is a dynamic value affecting process behavior. To persist across sessions on Linux, export it in `~/.bashrc` or `~/.profile`.

D10: **JSON** is strict key-value (machine-readable). **YAML** is human-readable (indentation-based). Advantage JSON: Standard for APIs. Advantage YAML: Concise for config files.

D11: **REST** is an architectural style using HTTP. **GET** (retrieve), **POST** (create), **PUT** (replace), **DELETE** (remove).

D12: **Version pinning** locks dependencies to exact versions (e.g., `1.0.1` vs `^1.0.0`). Important for reproducible builds to prevent unexpected breakage from upstream updates.

D13: **Compiled** (source to binary before run): C, Go. **Interpreted** (source executed line-by-line): Python, JavaScript.

D14: **Reverse Proxy** (Client -> Proxy -> Server) hides backend details. **Forward Proxy** (Client -> Proxy -> Internet) hides client identity.
*NGINX:* Web server/Reverse proxy. *HAProxy:* Load balancer/Reverse proxy.

D15: A **deadlock** is a permanent blocking of processes waiting for resources. **Coffman Conditions**: 1. Mutual Exclusion, 2. Hold and Wait, 3. No Preemption, 4. Circular Wait.

D16: **Eventual Consistency**: Updates propagate asynchronously. Cassandra uses **Tunable Consistency** (ONE, QUORUM, ALL). For read-after-write, use `LOCAL_QUORUM` to ensure the writer's node is included in the read.

D17: **mTLS** requires both client and server to present certificates. Handshake: Client Hello -> Server Hello + Cert + Request Client Cert -> Client Cert -> Key Exchange -> Finished. Used in Service Meshes (Istio) for zero-trust internal communication.

D18: A **hash table** maps keys to indices. **Collision Resolution**: Chaining (linked lists) or Open Addressing (probing). **Complexity**: Avg O(1), Worst O(n) (all keys hash to same index).

D19: **Circuit Breaker** prevents cascade failures. **States**: Closed (normal), Open (fail fast), Half-open (test recovery). Relates to retries: If circuit is Open, retries are skipped to prevent overload.

D20: **Optimistic**: Check conflicts at commit (e.g., row versioning). **Pessimistic**: Lock data at read (e.g., `SELECT FOR UPDATE`). Choose Optimistic for high concurrency/low conflict; Pessimistic for high conflict/critical data.
