**AA1** — **Monolith vs. Microservices:** A monolith is a single, unified codebase deployed as one unit. Microservices are a collection of loosely coupled, independently deployable services.
**3 Microservices Problems:**
1.  **Distributed Complexity:** Debugging requires tracing across network calls; no single stack trace.
2.  **Data Consistency:** ACID transactions across services are hard; requires eventual consistency or sagas.
3.  **Operational Overhead:** Requires orchestration (Kubernetes), service discovery, and complex CI/CD pipelines.

**AA2** — **Sync vs. Async:** Synchronous communication blocks the caller until a response is received (request-response). Asynchronous communication allows the caller to continue processing without waiting for an immediate response (fire-and-forget or callback).
**Examples:** Synchronous: REST/HTTP, gRPC. Asynchronous: RabbitMQ, Apache Kafka.

**AA3** — **API Gateway vs. Reverse Proxy:** A reverse proxy routes traffic to backend servers (Layer 7 routing, SSL termination). An API Gateway adds business logic capabilities like authentication, rate limiting, API versioning, and analytics.
**Replacement:** An API Gateway cannot fully replace a reverse proxy in all cases. While modern gateways (like Kong) act as proxies, a dedicated reverse proxy (like Nginx) is often used at the edge for high-performance SSL offloading and static content caching before traffic hits the gateway.

**AA4** — **Saga Pattern:** A sequence of local transactions where each updates data within a service. If one fails, compensating transactions undo previous changes.
**Choreography:** Services communicate via events; no central coordinator. *Appropriate:* Simple workflows, low coupling.
**Orchestration:** A central coordinator tells services what to do. *Appropriate:* Complex workflows, easier debugging, centralized control.

**AA5** — **CQRS:** Separates read (Query) and write (Command) operations into different models.
**Pairing with Event Sourcing:** Event Sourcing stores state changes as a sequence of events. CQRS uses these events to build optimized read models.
**Problem Solved:** Allows independent scaling of read and write workloads and provides a complete audit trail of state changes, which traditional CRUD (overwriting data) obscures.

**AA6** — **Circuit Breaker:** Prevents repeated calls to a failing service.
**States:** Closed (normal), Open (failures detected, requests blocked), Half-Open (testing if service recovered).
**Retry vs. Breaker:** Retry attempts to recover immediately; Breaker stops attempts to prevent cascading failure. Use together: Retry transient failures (e.g., network blip) but trigger the Breaker if failures persist to protect the system.

**AA7** — **Tenant Isolation Models:**
1.  **Shared DB/Schema:** Lowest cost, highest noisy-neighbor risk, complex security (row-level), easiest ops.
2.  **Shared DB/Schema per Tenant:** Medium cost, better security (schema isolation), moderate ops complexity.
3.  **Separate DB per Tenant:** Highest cost, best security/isolation, highest ops complexity (backup/migration per tenant).
**Trade-off:** Choose based on compliance needs and tenant count. High compliance = Separate DB. High scale/low cost = Shared.

**AA8** — **Consistency:** Strong consistency ensures all nodes see the same data immediately. Eventual consistency allows temporary divergence.
**Inventory Architecture:** Requires **Strong Consistency** to prevent overselling.
**Implementation:** Use a relational database (PostgreSQL) with **Serializable Isolation** or **Optimistic Locking** (version columns). For scale, implement a **Distributed Lock** (Redis) or a **Saga** with a reservation step before committing stock.

**AA9** — **Sidecar Proxy:** A container deployed alongside the application container to handle network traffic.
**Envoy/Istio:** Envoy handles the data plane (traffic routing, mTLS). The Control Plane (Istiod) configures Envoy.
**mTLS:** Encrypts traffic between services. Application-level TLS secures client-to-server; mTLS secures service-to-service, ensuring identity verification without app code changes.

**AA10** — **Strangler Fig:** Incrementally migrate functionality from a legacy system to new services.
**Strategy:**
1.  Place an API Gateway in front of the legacy .NET monolith.
2.  Build new .NET 8 microservices for specific features.
3.  Configure the Gateway to route traffic for those features to the new services.
4.  Gradually move more routes to the new services.
5.  Decommission the legacy monolith once all routes are migrated.

**AA11** — **12-Factor App:** Methodology for building SaaS apps.
**6 Factors:**
1.  **Codebase:** One codebase tracked in version control.
2.  **Dependencies:** Explicitly declared and isolated.
3.  **Config:** Stored in environment variables.
4.  **Backing Services:** Treated as attached resources.
5.  **Build/Release/Run:** Strictly separated stages.
6.  **Concurrency:** Scale out via process model.
**Importance:** Ensures portability, scalability, and maintainability in cloud environments.

**AA12** — **Event-Driven (Kafka) vs. Request-Driven (REST):**
*   **Coupling:** Event-driven is loosely coupled; Request-driven is tightly coupled.
*   **Scalability:** Event-driven scales better for high throughput; Request-driven is simpler for low volume.
*   **Debugging:** Request-driven is easier (traceable); Event-driven is harder (distributed state).
*   **Ordering:** Kafka guarantees ordering per partition; REST is request-order dependent.
*   **Failure:** Event-driven uses dead-letter queues; Request-driven uses timeouts/retries.
**Choice:** Use Event-driven for decoupled, high-volume data pipelines. Use Request-driven for real-time user interactions requiring immediate feedback.

**AA13** — **Load Balancer:** Distributes traffic across servers.
**L4 vs. L7:** L4 (Transport) routes based on IP/Port. L7 (Application) routes based on HTTP headers/URL.
**L7 Use Case:** SSL Termination or routing `/api` to backend A and `/static` to backend B based on URL path.

**AA14** — **Message Queue vs. Log Broker:**
*   **Queue (RabbitMQ):** Messages are consumed and deleted. Good for task distribution.
*   **Log Broker (Kafka):** Messages are retained and replayable. Good for event streaming/audit.
**Pick:** Use Queue for work items (email sending). Use Log Broker for analytics, audit logs, or event sourcing.

**AA15** — **Container vs. VM:** A container virtualizes the OS (shares kernel), while a VM virtualizes hardware (runs full OS).
**Popularity:** Containers are lightweight, start instantly, and have lower overhead, making them ideal for microservices density and rapid scaling.

**AA16** — **Stateless vs. Stateful:** Stateless services store no session data (e.g., Web Server). Stateful services store data (e.g., Database).
**Scaling:** Stateless services are easier to scale horizontally because any instance can handle any request without affinity.
**Example:** Stateless: Nginx. Stateful: PostgreSQL.

**AA17** — **ORM:** Maps database tables to programming language objects.
**Examples:** Hibernate (Java), Entity Framework (.NET).
**Trade-offs:** ORMs increase developer productivity but can generate inefficient SQL and hide database-specific optimizations. Raw SQL offers performance control but increases maintenance burden.

**AA18** — **Bulkhead Pattern:** Isolates resources (threads, connections) for specific services to prevent failures from cascading.
**Implementation:** Use separate thread pools for each downstream service call. If the "Payment Service" pool fills up, the "Inventory Service" pool remains available. In Kubernetes, use separate namespaces or resource quotas.

**AA19** — **OT vs. CRDT:**
*   **OT:** Server-authoritative, resolves conflicts centrally. High latency, complex logic.
*   **CRDT:** Client-authoritative, mathematically guarantees convergence. Low latency, supports offline.
**Choice:** **CRDTs** for offline support. OT requires a server to resolve conflicts, which breaks offline editing. CRDTs allow local edits to merge automatically upon reconnection.

**AA20** — **Distributed Tracing Challenges:**
*   **Context Propagation:** Pass `TraceID`/`SpanID` via headers (W3C/B3).
*   **Sampling:** Head-based (random at start) vs. Tail-based (sample based on latency/errors).
*   **Storage:** High-volume data requires efficient stores (Jaeger/Tempo).
*   **Kafka:** Inject trace context into message headers. Ensure consumers read headers and continue the trace chain.

**OP1** — **Hypervisor Type 1 vs. 2:**
*   **Type 1 (Bare-metal):** Runs directly on hardware. Examples: VMware ESXi, Microsoft Hyper-V Server.
*   **Type 2 (Hosted):** Runs on top of an OS. Examples: VirtualBox, VMware Workstation.
**Difference:** Type 1 has better performance and resource control; Type 2 is easier for development.

**OP2** — **SAN vs. NAS:**
*   **SAN (Storage Area Network):** Block-level storage. Protocols: iSCSI, Fibre Channel. Use for: Databases, VMs.
*   **NAS (Network Attached Storage):** File-level storage. Protocols: NFS, SMB/CIFS. Use for: File shares, home directories.

**OP3** — **3-Tier vs. Leaf-Spine:**
*   **3-Tier:** Core, Distribution, Access. Can suffer from oversubscription and high latency in large fabrics.
*   **Leaf-Spine:** Every leaf connects to every spine.
**Benefit:** Solves oversubscription and provides predictable, low-latency bisection bandwidth in modern data centers.

**OP4** — **Proxmox HA Design:**
*   **Nodes:** Minimum 3 for quorum.
*   **Quorum:** Majority vote required to avoid split-brain.
*   **Fencing:** STONITH (e.g., IPMI) to power off failed nodes.
*   **Storage:** Ceph (distributed) or Shared iSCSI/NFS.
*   **Failure:** Pacemaker detects node down, fences it, and restarts VMs on remaining nodes.

**OP5** — **VLAN Trunking:** Carries multiple VLAN traffic over one link.
**802.1Q:** Adds a 4-byte tag to Ethernet frames identifying the VLAN ID.
**Linux Config:** Create sub-interfaces (e.g., `eth0.10`, `eth0.20`) using `ip link` or `vconfig`, assign IPs, and tag traffic.

**OP6** — **RAID Levels:**
*   **RAID 1:** Mirroring (50% capacity).
*   **RAID 5:** Striping + Parity (N-1 capacity).
*   **RAID 6:** Striping + Dual Parity (N-2 capacity).
*   **RAID 10:** Mirroring + Striping (50% capacity, high perf).
*   **RAID Z2:** ZFS equivalent of RAID 6.
**Choice:** **RAID 10** (or Z2) for 6 drives. RAID 10 offers best performance for DBs with 3-drive fault tolerance (if mirrored pairs). RAID 6/Z2 offers more capacity but slower writes. For DBs, performance is key -> RAID 10.

**OP7** — **DNS Architecture:**
*   **Internal:** AD-integrated DNS (Active Directory).
*   **External:** Public authoritative servers (e.g., Route53).
*   **Split-Horizon:** Internal clients get internal IPs; external get public IPs.
*   **Forwarders:** Internal DNS forwards external queries to public resolvers.
*   **VPN:** Split tunneling sends internal DNS queries to internal servers; full tunnel sends all traffic through HQ.

**OP8** — **PKI Chain:**
*   **Root CA:** Offline, highly secure.
*   **Subordinate CA:** Issues end-entity certs.
*   **End-Entity:** Servers/Devices.
*   **Validation:** CRL (list) or OCSP (real-time).
*   **Compromise:** Root = Trust collapse (revoke all). Subordinate = Revoke subordinate, issue new chain.

**OP9** — **WAN Consolidation:**
*   **MPLS:** High cost, high reliability, low latency.
*   **IPsec:** Low cost, variable reliability.
*   **SD-WAN:** Dynamic path selection, cost-effective, manages QoS.
*   **WireGuard:** Lightweight, modern crypto.
**Recommendation:** **SD-WAN**. It optimizes VoIP traffic over the internet (lower cost than MPLS) while maintaining QoS and reliability.

**OP10** — **iDRAC/iLO/IPMI:** Out-of-band management interfaces.
**Critical:** Allows server control (power, console) even if OS is down.
**Risk:** If exposed to internet, attackers can gain root access to the hardware (e.g., DRAC vulnerabilities).

**OP11** — **LVM vs. ZFS:**
*   **LVM:** Block device manager. Snapshots are copy-on-write (Cow).
*   **ZFS:** Filesystem + Volume Manager. Snapshots are instant, cheap, and immutable.
*   **Use:** LVM for simple block storage. ZFS for data integrity, compression, and snapshots.

**OP12** — **Stretch Cluster:**
*   **Design:** 2 sites + 1 Tiebreaker (witness).
*   **Ceph:** CRUSH rules must distribute OSDs across sites.
*   **Latency:** < 5ms round-trip required for monitors.
*   **Split-Brain:** Without a tiebreaker, both sites might think they are primary. Tiebreaker breaks the tie.

**OP13** — **DHCP DORA:**
*   **Process:** Discover, Offer, Request, Acknowledge.
*   **Unavailable:** Client uses APIPA (169.254.x.x) if lease expires and server is down.

**OP14** — **Managed vs. Unmanaged Switch:**
*   **Unmanaged:** Plug-and-play.
*   **Managed:** Configurable VLANs, QoS, ACLs, SNMP monitoring.
*   **Importance:** Essential for security segmentation and traffic prioritization in enterprises.

**OP15** — **NTP:** Network Time Protocol.
**Critical:** Certificates, logs, and database transactions rely on accurate time.
**Drift:** Can cause Kerberos auth failures, log analysis errors, and cluster consensus issues.

**OP16** — **Physical vs. VM vs. Container:**
*   **Physical:** Hardware only.
*   **VM:** OS virtualization.
*   **Container:** Process virtualization.
*   **Bare Metal:** Use for high-performance IO (DB), latency-sensitive apps, or hardware passthrough needs.

**OP17** — **UPS vs. Generator:**
*   **UPS:** Short-term battery power (minutes).
*   **Generator:** Long-term power (hours/days).
*   **ATS:** Automatic Transfer Switch switches load from utility to generator when power fails.
*   **Need Both:** UPS bridges the gap until generator starts.

**OP18** — **Active-Passive vs. Active-Active:**
*   **Active-Passive:** One node handles load; standby takes over on failure.
*   **Active-Active:** Both handle load.
*   **PostgreSQL HA:** Pacemaker/Corosync cluster.
*   **STONITH:** Fence failed node to prevent data corruption.
*   **VIP:** Virtual IP moves to active node.

**OP19** — **Backup Strategy:**
*   **3-2-1:** 3 copies, 2 media types, 1 offsite.
*   **Immutable:** Prevents ransomware deletion.
*   **Test:** Regularly restore random files/volumes.
*   **Backup vs. DR:** Backup is data copy; DR is the plan/process to restore operations after a disaster.

**OP20** — **802.1X:**
*   **Roles:** Supplicant (Client), Authenticator (Switch), Server (RADIUS).
*   **EAP:** EAP-TLS (Certificate auth, secure); PEAP (User/Pass auth).
*   **Dynamic VLAN:** Switch assigns VLAN based on RADIUS response.
*   **Failure:** Port blocked or placed in Guest VLAN.
*   **MAB:** Switch sends MAC address to RADIUS as username/password for non-802.1X devices.
