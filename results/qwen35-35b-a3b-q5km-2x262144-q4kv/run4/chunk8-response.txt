CL1 — **IaaS** (Infrastructure as a Service) provides virtualized computing resources (e.g., AWS EC2). **PaaS** (Platform as a Service) provides a framework for developing/runing apps without managing infrastructure (e.g., AWS Elastic Beanstalk). **SaaS** (Software as a Service) delivers software over the internet (e.g., Microsoft 365).

CL2 — **Shared Responsibility Model:** Provider secures the cloud (hardware, hypervisor); Customer secures what is *in* the cloud.
*   **IaaS VM:** Provider secures physical host; Customer secures OS, patching, data, firewall config.
*   **Managed Kubernetes:** Provider secures control plane; Customer secures nodes, workloads, network policies, secrets.
*   **SaaS (M365):** Provider secures everything; Customer secures identity, access management, and data classification.

CL3 — **VPC:** Isolated virtual network. **Subnet:** IP range within VPC. **Security Group:** Stateful firewall at instance level (allow rules only). **NACL:** Stateless firewall at subnet level (allow/deny rules).
*   **Layering:** VPC contains Subnets; Subnets contain Instances protected by SGs; NACLs filter traffic entering/leaving Subnets.
*   **Stateful:** SG tracks connection state (return traffic allowed automatically). **Stateless:** NACL requires explicit rules for both request and response.

CL4 — **Landing Zone:** Secure, compliant baseline environment. **OUs:** Group accounts by function (Prod, Dev). **Guardrails:** SCPs/Policy to prevent risky actions (e.g., disabling encryption). **Shared Networking:** Transit Gateway/Hub-and-Spoke centralizes connectivity. **Centralized Logging:** Aggregates logs for security monitoring.
*   **Single-Account Problem:** High blast radius (one breach compromises all), billing opacity, permission sprawl, lack of isolation between environments.

CL5 — **IaC:** Managing infrastructure via code.
*   **Terraform/OpenTofu:** Multi-cloud, declarative, state-file based. Best for heterogeneous environments.
*   **Pulumi:** Imperative/Declarative using general languages (TS, Python). Best for devs comfortable with code.
*   **CloudFormation/Bicep:** Native to AWS/Azure. Best for single-cloud, deep integration.
*   **Crossplane:** K8s-native IaC. Best for managing cloud resources via K8s API.

CL6 — **HPA:** Scales pods based on CPU/Memory. **VPA:** Adjusts pod resource requests/limits (requires restart). **Cluster Autoscaler:** Adds/removes nodes based on pod scheduling. **KEDA:** Event-driven scaling (e.g., queue length).
*   **Use KEDA:** When scaling is triggered by external events (Kafka, SQS, HTTP) rather than just resource utilization.

CL7 — **Strategy:** Define RPO (data loss tolerance) and RTO (downtime tolerance).
*   **Active-Active:** Both regions serve traffic. Low RTO/RPO, high cost.
*   **Active-Passive:** One region active. Higher RTO, lower cost.
*   **DB:** Async replication (lower latency, potential data loss) vs Sync (strong consistency, latency).
*   **DNS:** Route53 Health Checks/Traffic Manager for failover.
*   **DR Types:** **Pilot Light:** Core DB running, app scaled down. **Warm Standby:** Scaled-down app running. **Multi-site Active-Active:** Full capacity in both regions.

CL8 — **GitOps:** Git repository is the single source of truth. **ArgoCD:** Push/Pull model, UI-driven. **Flux:** Pull model, K8s-native.
*   **Secrets:** Sealed Secrets (encrypted in Git), External Secrets Operator (fetch from Vault/AWS Secrets Manager).
*   **Drift:** Reconciliation loop detects manual changes and reverts them to match Git state.

CL9 — **Architecture:**
*   **StatefulSet:** Ensures stable pod identity/ordering.
*   **PV/PVC:** Use `local` storage class for low latency (hostPath).
*   **Headless Service:** Provides stable DNS for StatefulSet pods.
*   **Init Containers:** Prepare config files before main app starts.
*   **ConfigMap:** Mount as volume for specific paths.
*   **PDB:** Ensure minimum availability during disruptions.

CL10 — **SA:** K8s internal identity. **Workload Identity:** Maps K8s SA to Cloud IAM Role. **IRSA:** AWS-specific implementation of Workload Identity.
*   **Mounting Creds:** Bad practice (risk of leakage, rotation difficulty).
*   **Federation:** Uses OIDC tokens to exchange K8s identity for short-lived cloud credentials without storing keys.

CL11 — **Object (S3):** Unstructured data, high durability (11 9s), low latency. **Block (EBS):** Attached to VM, low latency, high IOPS. **File (EFS):** Shared file system, NFS/SMB protocol.
*   **Use Object:** Backups, media, logs. **Use Block:** OS disks, DBs. **Use File:** Shared configs, home directories.

CL12 — **Zero Trust:** Verify every request.
*   **Network Policies:** Restrict pod-to-pod traffic.
*   **Service Mesh (mTLS):** Encrypts service-to-service traffic.
*   **Admission (OPA/Kyverno):** Enforce policies on deployment.
*   **Image Signing:** Verify integrity before run.
*   **Runtime (Falco):** Detect anomalies.
*   **Diff:** Traditional relies on perimeter (firewall); Zero Trust assumes breach and verifies identity/context.

CL13 — **CDN:** Distributed network caching content at edge locations.
*   **Services:** AWS CloudFront, Azure CDN.
*   **Use CDN:** Static assets (images, JS), global latency reduction, DDoS mitigation.
*   **Direct Origin:** Dynamic content, real-time data, or when caching is not desired.

CL14 — **Region:** Geographic area (e.g., us-east-1). **AZ:** Isolated data centers within a region.
*   **Multi-AZ:** Protects against data center failure (high availability).
*   **Zonal:** EC2, EBS (tied to AZ). **Regional:** ELB, Route53 (span AZs).

CL15 — **Managed DB:** RDS, Cloud SQL.
*   **Provider Handles:** Patching, backups, failover, hardware, scaling storage.
*   **User Handles:** Schema, queries, user permissions, data logic.

CL16 — **IAM:** **Users** (People), **Groups** (Roles), **Roles** (Assumable identities), **Policies** (Permissions).
*   **Least Privilege:** Grant only necessary permissions.
*   **Application:** Use Groups for permissions, Roles for cross-account access, avoid long-term keys.

CL17 — **Serverless:** Event-driven, no server management (Lambda, Functions).
*   **Cold Start:** Delay when function initializes (memory/code load). Matters for latency-sensitive apps.
*   **Poor Fit:** Long-running processes, stateful apps, predictable high-throughput (costlier than provisioned).

CL18 — **Ingress:** K8s resource for HTTP/HTTPS routing. **Gateway API:** Standardized, multi-protocol, more granular control. **LB:** Cloud provider load balancer. **Mesh:** Istio ingress gateway.
*   **TLS:** Terminates at LB (Edge), Ingress Controller, or Mesh.
*   **Gateway API:** Use for complex routing, multiple protocols, or when standard Ingress is insufficient.

CL19 — **Strategy:**
*   **Commit:** Reserved Instances/Savings Plans (up to 72% savings).
*   **Spot:** Interruptible instances (up to 90% savings) for fault-tolerant workloads.
*   **Right-sizing:** Match instance to actual usage.
*   **Storage:** Intelligent-Tiering, Archive for old data.
*   **FinOps:** Tagging, Showback/Chargeback dashboards.
*   **Tools:** AWS Cost Explorer, Kubecost, CloudHealth.

CL20 — **Strategy:**
*   **Abstraction:** Kubernetes (K8s) as runtime, Terraform for IaC.
*   **Networking:** Direct Connect/ExpressRoute + VPN.
*   **Identity:** OIDC Federation.
*   **Trade-offs:** Multi-cloud increases complexity, cost, and skill requirements.
*   **Justified:** Regulatory compliance, catastrophic vendor risk, specific regional capabilities.
*   **Unnecessary:** If single-cloud provider meets all needs (complexity outweighs benefit).

OT1 — **Purdue Model:** ISA-95 standard for IT/OT integration.
*   **Level 0:** Physical Process (Sensors/Actuators).
*   **Level 1:** Basic Control (PLCs, RTUs).
*   **Level 2:** Supervisory Control (SCADA, HMI).
*   **Level 3:** Operations (MES, Historian).
*   **Level 4:** Enterprise (ERP, Business).

OT2 — **IT:** Data confidentiality, patching, frequent changes. **OT:** Safety, availability, legacy hardware.
*   **Patching:** OT systems often run on unsupported OS; patching can cause downtime or safety failures.

OT3 — **DMZ:** Buffer zone between IT and OT. **IDMZ:** Industrial DMZ.
*   **Traffic:** Historian pulls from SCADA (Push), Engineering stations access DMZ.
*   **Danger:** Direct L4 to L1 allows attackers to bypass controls and impact physical safety directly.

OT4 — **Protocols:**
1.  **Modbus TCP:** L2/L3, Master-Slave.
2.  **OPC UA:** L2-L7, Secure, Platform-independent.
3.  **EtherNet/IP:** L2/L3, CIP protocol.
4.  **PROFINET:** L2, Real-time Ethernet.
5.  **MQTT:** L4/L7, Pub/Sub, lightweight.

OT5 — **Data Diode:** Hardware enforcing one-way data flow. **Firewall:** Two-way filtering.
*   **Use Diode:** When data must leave OT but nothing can enter (prevent exfiltration/injection).
*   **Limitation:** No bidirectional control or feedback from IT to OT.

OT6 — **Flow:** PLCs -> Historian -> OPC UA Server -> Edge Gateway (Protocol conversion) -> MQTT Broker -> Cloud Ingestion.
*   **Boundaries:** Firewall between Edge and Cloud; DMZ between Edge and Historian.

OT7 — **IEC 62443:** **Zone:** Logical grouping of assets with similar security requirements. **Conduit:** Controlled path between zones.
*   **Model:** Corporate (L4) -> DMZ -> SCADA (L3) -> PLCs (L2) -> SIS (L1). Firewalls between zones. Engineering workstations in DMZ or L3.

OT8 — **Segmentation:**
*   **VLANs:** Separate lines, HMIs, Servers.
*   **Firewalls:** Between VLANs and DMZ.
*   **Switches:** Managed Layer 3 switches.
*   **Remote Access:** Jump hosts with MFA and session recording.
*   **Transition:** Parallel run (new network alongside old), test offline, switch over during maintenance window.

OT9 — **SIS:** Independent safety system (stops process safely). **BPCS:** Basic Process Control System.
*   **IEC 61511:** Safety lifecycle standard.
*   **Isolation:** Prevents cyber attacks from disabling safety systems.
*   **Ransomware on SIS:** Safety system failure, potential physical damage or injury.

OT10 — **PLC:** Logic controller for machines. **DCS:** Distributed control for continuous processes. **SCADA:** Monitoring/Supervisory.
*   **Coexist:** Yes. PLCs/DCS feed data to SCADA.

OT11 — **Architecture:** Jump host (bastion) -> MFA -> Session Recording -> Time-limited access.
*   **No Direct RDP:** Exposes OT assets to IT network threats; violates segmentation.

OT12 — **Response:**
*   **Monitor:** IDS for anomalies, asset inventory check.
*   **Steps:** Isolate compromised segment, notify plant manager, engage OT engineers.
*   **Operator:** Maintain manual control, prioritize safety over production.
*   **Role:** Operator ensures physical safety while IT/OT teams contain the threat.

OT13 — **HMI:** Operator interface for monitoring/control.
*   **Diff:** Engineering Workstation (EW) is for programming/configuring logic; HMI is for viewing/operating.

OT14 — **Historian:** Time-series database for process data.
*   **Plant (L3):** Real-time ops, alarms.
*   **Enterprise (L4):** Long-term analytics, reporting.

OT15 — **Ladder Logic:** Graphical programming language resembling relay circuits.
*   **Device:** PLC.
*   **Dominant:** Deterministic, safety-certified, widely understood by electricians.

OT16 — **RTU:** Remote Terminal Unit. **PLC:** Programmable Logic Controller.
*   **Diff:** RTU optimized for comms and harsh environments; PLC optimized for local logic.
*   **RTU:** Pipelines, remote substations.

OT17 — **Analog:** Continuous signal (4-20mA). **Digital:** Discrete (0/1).
*   **4-20mA:** 4mA = "Live Zero" (detects wire break if signal drops to 0).

OT18 — **Inventory:** Foundation for security (know what to protect).
*   **Tools:** Passive (packet sniffing), Active (scanning).
*   **Active Risk:** Can crash legacy devices or trigger alarms.

OT19 — **MITRE ATT&CK ICS:** Framework for ICS threats.
*   **Tactics:** Initial Access (Phishing), Execution (Command & Scripting), Persistence (Modify Config), Privilege Escalation (Exploit), Impact (Stop Process).
*   **Diff:** Focuses on physical impact and safety, not just data.
*   **Use:** Map defenses to tactics to find gaps.

OT20 — **Converged SOC:**
*   **Logs:** Passive collection (SPAN ports) to avoid OT impact.
*   **SIEM:** Correlate IT/OT logs.
*   **Rules:** OT-specific (e.g., PLC write attempts).
*   **Staffing:** OT engineers for triage (IT analysts lack context).
*   **False Positives:** High; tune rules to reduce noise.
*   **Firmware Change:** Alert -> Isolate -> Verify -> Revert -> Patch -> Restore.
