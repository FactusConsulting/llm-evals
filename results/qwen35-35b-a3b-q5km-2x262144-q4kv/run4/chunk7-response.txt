AA1 — **Monolith vs. Microservices:** A monolith is a single unified codebase deployed as one unit. Microservices decompose this into independent, loosely coupled services deployed separately.
**3 Microservices Problems:**
1.  **Distributed Complexity:** Network latency, partial failures, and distributed transactions are harder to manage than local calls.
2.  **Data Consistency:** Maintaining ACID transactions across multiple databases requires patterns like Sagas.
3.  **Operational Overhead:** Requires robust CI/CD, monitoring, and service discovery infrastructure (e.g., Kubernetes, Service Mesh).

AA2 — **Sync vs. Async:** Synchronous communication waits for an immediate response (request-response). Asynchronous communication sends a message and continues processing without waiting.
**Examples:** Synchronous: REST/HTTP, gRPC. Asynchronous: RabbitMQ, Apache Kafka.

AA3 — **API Gateway vs. Reverse Proxy:** A reverse proxy routes traffic to backend servers (L4/L7). An API Gateway adds business logic (auth, rate limiting, transformation) specifically for APIs.
**Replacement:** An API Gateway often includes reverse proxy functionality, but a pure reverse proxy (like Nginx) is sometimes preferred for simple L4 load balancing or SSL termination where API-specific logic isn't needed. They are not always interchangeable depending on feature requirements.

AA4 — **Saga Pattern:** A sequence of local transactions where each updates data within a service. If one fails, compensating transactions undo previous changes.
**Choreography:** Services publish events; others react. *Appropriate:* Simple flows, low coupling.
**Orchestration:** A central coordinator tells services what to do. *Appropriate:* Complex flows, need for visibility/rollback control.

AA5 — **CQRS:** Separates read (Query) and write (Command) operations into different models.
**Pairing with Event Sourcing:** Event Sourcing stores state changes as immutable events. CQRS uses these events to build optimized read models.
**Problem Solved:** Allows independent scaling of read/write workloads and provides a complete audit trail of state changes, which traditional CRUD (overwriting state) obscures.

AA6 — **Circuit Breaker:** Prevents repeated calls to a failing service.
**States:** Closed (normal), Open (failures threshold met, requests fail immediately), Half-Open (testing if service recovered).
**Difference from Retry:** Retry attempts to fix the failure; Circuit Breaker stops traffic to prevent cascading failures.
**Use Together:** Use retries for transient network glitches, but wrap them in a Circuit Breaker to stop hammering a truly down service.

AA7 — **Tenant Isolation Models:**
1.  **Shared DB/Schema:** Lowest cost, highest noisy-neighbor risk, complex security (row-level).
2.  **Shared DB/Schema per Tenant:** Balanced cost/security. Easier backup per tenant, moderate complexity.
3.  **Separate DB per Tenant:** Highest security/isolation, highest cost/operational complexity. Best for compliance-heavy tenants.

AA8 — **Consistency:** Strong consistency ensures all nodes see the same data immediately. Eventual consistency allows temporary divergence.
**Inventory Architecture:** Use **Strong Consistency**.
**Data Store:** Relational DB (e.g., PostgreSQL) with row-level locking or a distributed store with strong consistency guarantees (e.g., CockroachDB).
**Patterns:** Optimistic locking (versioning) or Distributed Locking (Redis/Zookeeper) to reserve stock before committing.

AA9 — **Sidecar Proxy:** A container deployed alongside the main application container to handle network traffic.
**Envoy/Istio:** Envoy handles the data plane (traffic routing, mTLS). Istio Control Plane configures Envoy.
**Data vs. Control Plane:** Data plane processes traffic; Control plane manages configuration and policy.
**mTLS:** Solves identity verification and encryption at the network layer, independent of application code, preventing man-in-the-middle attacks between services.

AA10 — **Strangler Fig:** Incrementally migrate functionality from a legacy system to new services.
**Strategy:** Deploy an API Gateway in front of the legacy monolith. Route specific endpoints (e.g., `/api/orders`) to new .NET 8 microservices while others go to the legacy .NET Framework app. Gradually move more routes until the monolith is "strangled" and removed.

AA11 — **12-Factor App:** Methodology for building SaaS apps.
**6 Factors:**
1.  **Codebase:** One codebase tracked in revision control.
2.  **Dependencies:** Explicitly declared and isolated.
3.  **Config:** Stored in environment variables.
4.  **Backing Services:** Treated as attached resources.
5.  **Build, Release, Run:** Strictly separated stages.
6.  **Processes:** Stateless, state stored in backing services.
**Why:** Ensures portability, scalability, and ease of deployment in cloud environments.

AA12 — **Event-Driven (Kafka) vs. Request-Driven (REST/gRPC):**
*   **Coupling:** Event-driven is loosely coupled; Request-driven is tightly coupled.
*   **Scalability:** Event-driven scales better for high throughput; Request-driven is simpler for low latency user interaction.
*   **Debugging:** Event-driven is harder (distributed state); Request-driven is easier (linear flow).
*   **Ordering:** Kafka guarantees ordering per partition; REST does not.
*   **Failure:** Event-driven requires retries/replay; Request-driven fails immediately.
**Choice:** Use Event-driven for background processing/analytics; Request-driven for user-facing APIs.

AA13 — **Load Balancer:** Distributes traffic across multiple servers.
**L4 vs. L7:** L4 (Transport) routes based on IP/Port. L7 (Application) routes based on HTTP headers/URL.
**L7 Use Case:** SSL termination or routing `/api` traffic to backend A and `/static` to backend B.

AA14 — **Message Queue:** Stores messages for processing.
**MQ (RabbitMQ) vs. Log (Kafka):** MQ focuses on task delivery (acknowledged once). Kafka focuses on event streaming (logs retained, multiple consumers).
**Pick:** RabbitMQ for task queues/workers. Kafka for event sourcing, analytics, or high-throughput logging.

AA15 — **Container vs. VM:** A container virtualizes the OS kernel (shares host OS). A VM virtualizes hardware (full OS per VM).
**Popularity:** Containers are lighter, start faster, and have higher density, making them ideal for microservices scaling.

AA16 — **Stateless vs. Stateful:** Stateless services store no session data (any instance handles any request). Stateful services store session/data locally.
**Scaling:** Stateless scales horizontally easily (add instances). Stateful requires data replication/migration.
**Examples:** Stateless: Web server. Stateful: Database, Redis.

AA17 — **ORM:** Maps database tables to objects in code.
**Examples:** Hibernate (Java), Entity Framework (.NET).
**Trade-offs:** ORM speeds development and reduces boilerplate but can generate inefficient SQL and hide performance issues compared to raw SQL.

AA18 — **Bulkhead Pattern:** Isolates resources (threads, connections) for different services to prevent one failure from exhausting all resources.
**Implementation:** Use separate thread pools for each downstream service call. Or deploy critical services on separate node groups in Kubernetes.

AA19 — **OT vs. CRDT:** OT requires a central server to resolve conflicts sequentially. CRDTs allow mathematically guaranteed convergence without coordination.
**Offline Support:** CRDTs are superior for offline editing as clients can merge changes locally without server validation.
**Choice:** CRDTs for a system requiring offline editing and eventual consistency.

AA20 — **Distributed Tracing Challenges:**
*   **Context Propagation:** TraceID/SpanID must be passed via headers (W3C/B3) across services.
*   **Sampling:** Head-based (random) for volume; Tail-based (error-based) for debugging.
*   **Storage:** Jaeger/Tempo store spans; querying requires indexing.
*   **Kafka:** Context must be injected into Kafka headers (baggage) to maintain continuity across async boundaries.

OP1 — **Hypervisor Type 1:** Bare-metal (runs directly on hardware). Examples: VMware ESXi, Microsoft Hyper-V Server.
**Hypervisor Type 2:** Hosted (runs on OS). Examples: VMware Workstation, VirtualBox.

OP2 — **SAN vs. NAS:** SAN provides block storage (appears as disk). NAS provides file storage (appears as folder).
**Use:** SAN for databases (high I/O). NAS for file shares/backups.
**Protocols:** SAN (iSCSI, Fibre Channel). NAS (NFS, SMB/CIFS).

OP3 — **3-Tier:** Core (Backbone), Distribution (Policy), Access (Edge).
**Leaf-Spine:** Preferred for modern DCs. Solves oversubscription and latency issues of 3-tier by providing non-blocking, equal-cost paths between any leaf and spine.

OP4 — **Proxmox HA:**
*   **Architecture:** 3+ nodes for quorum.
*   **Quorum:** Majority vote required to form cluster.
*   **Fencing:** STONITH (e.g., DRAC/iLO) to power off failed nodes.
*   **Storage:** Ceph (distributed) preferred over iSCSI/NFS (single point of failure).
*   **Failure:** VMs on failed node restart on surviving nodes automatically.

OP5 — **VLAN Trunking:** Carries multiple VLANs over one link using 802.1Q tags.
**Linux Config:** Create sub-interfaces (e.g., `eth0.10`, `eth0.20`) with `ip link add link eth0 name eth0.10 type vlan id 10`. Assign IPs to sub-interfaces.

OP6 — **RAID:**
*   **RAID 1:** Mirror (50% capacity).
*   **RAID 5:** Striping + Parity (N-1 capacity).
*   **RAID 6:** Double Parity (N-2 capacity).
*   **RAID 10:** Mirror + Stripe (50% capacity, high perf).
*   **RAID Z2:** ZFS double parity (similar to RAID 6).
*   **Choice:** RAID 10 for 6 drives. Best performance and fault tolerance (can lose 2 drives if mirrored pairs) for a DB server.

OP7 — **DNS Architecture:**
*   **Internal:** AD DNS servers (Active Directory integrated).
*   **External:** Public authoritative servers (e.g., Route53).
*   **Split-Horizon:** Internal clients resolve internal names; external resolve public.
*   **Forwarders:** Internal DNS forwards unknown queries to public resolvers.
*   **Laptop:** Connects via VPN -> DNS traffic routed to internal AD DNS -> Resolves internal names.

OP8 — **PKI Chain:** Root CA (Offline) -> Subordinate CA (Issuing) -> End Entity.
**CRL vs. OCSP:** CRL is a list of revoked certs; OCSP checks status in real-time.
**Compromise:** Root CA compromised = All trust broken (must rebuild PKI). Subordinate CA compromised = Revoke that CA and issue new certs (contained).

OP9 — **WAN Consolidation:**
*   **MPLS:** High cost, high reliability, low latency.
*   **IPsec:** Low cost, variable latency.
*   **SD-WAN:** Dynamic path selection, cost-effective, easy management.
*   **WireGuard:** Lightweight, secure, but lacks enterprise management features.
*   **Recommendation:** SD-WAN. Balances cost, management, and VoIP QoS requirements.

OP10 — **iDRAC/iLO/IPMI:** Out-of-band management interfaces for hardware control (power, console).
**Critical:** Allows recovery if OS is down.
**Exposed to Internet:** High risk of unauthorized hardware access, firmware tampering, or ransomware.

OP11 — **LVM vs. ZFS:**
*   **LVM:** Volume management only. Snapshots are copy-on-write, slower.
*   **ZFS:** Filesystem + Volume + RAID. Snapshots are instant, immutable, and space-efficient.
*   **Use:** LVM for simple Linux volume management. ZFS for data integrity and advanced snapshotting.

OP12 — **Stretch Cluster:**
*   **Design:** 2 sites + 1 witness node (tiebreaker).
*   **Ceph:** CRUSH rules must distribute replicas across sites.
*   **Latency:** <5ms recommended for monitors.
*   **Split-Brain:** Without a tiebreaker, both sites might think they are primary, causing data corruption. 2-site Ceph is bad without a witness.

OP13 — **DHCP DORA:** Discover, Offer, Request, Acknowledge.
**Unavailable:** Client uses APIPA (169.254.x.x) if lease expires and server is unreachable.

OP14 — **Managed vs. Unmanaged:**
*   **Unmanaged:** Plug-and-play.
*   **Managed:** Configurable VLANs, QoS, ACLs, Port Mirroring, SNMP monitoring. Critical for security and traffic prioritization.

OP15 — **NTP:** Network Time Protocol.
**Critical:** Cluster consensus, log correlation, certificate validation (Kerberos/SSL).
**Drift:** Can cause authentication failures, data corruption in distributed systems, and log analysis issues.

OP16 — **Physical vs. VM vs. Container:**
*   **Physical:** Hardware only.
*   **VM:** OS virtualization.
*   **Container:** Process virtualization.
*   **Bare Metal:** Use for high I/O, low latency, or specific hardware licensing requirements.

OP17 — **UPS vs. Generator:**
*   **UPS:** Short-term battery power (seconds/minutes).
*   **Generator:** Long-term power (hours/days).
*   **ATS:** Switches load from utility to UPS, then to generator during outage.
*   **Need Both:** UPS bridges the gap until generator starts.

OP18 — **Active-Passive vs. Active-Active:**
*   **Active-Passive:** One node handles traffic; standby takes over on failure.
*   **Active-Active:** All nodes handle traffic.
*   **PostgreSQL HA:** Pacemaker/Corosync manages VIP. STONITH ensures failed node is powered off. VIP floats to active node.

OP19 — **Backup Strategy:**
*   **3-2-1:** 3 copies, 2 media types, 1 offsite.
*   **Immutable:** Prevents ransomware deletion.
*   **Test:** Regular restore drills.
*   **Backup vs. DR:** Backup is data copy; DR is the plan/process to restore operations after a site failure.

OP20 — **802.1X:**
*   **Roles:** Supplicant (Client), Authenticator (Switch), Server (RADIUS).
*   **EAP:** EAP-TLS (Certificate-based, secure). PEAP (Username/Password).
*   **Dynamic VLAN:** Assigns VLAN based on user identity.
*   **Failure:** Port blocked or placed in Guest VLAN.
*   **MAB:** Fallback for devices without 802.1X (e.g., printers) using MAC address as credential.
