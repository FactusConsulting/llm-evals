SC1-A
The failure stems from a conflict between the **Restricted Pod Security Standard (PSS)** and the container image configuration.
1.  **PSS Restricted Requirement:** The Restricted standard mandates `securityContext.runAsNonRoot: true`. It also implicitly requires that the effective user ID (UID) of the container process is non-zero.
2.  **Image Default:** The new base image (`cgr.dev/chainguard/static:latest`) defaults `USER` to root (UID 0).
3.  **Manifest Gap:** The pod spec sets `runAsNonRoot: true` but omits `runAsUser`. Kubernetes defaults `runAsUser` to the image's `USER` directive (UID 0).
4.  **Conflict:** The effective UID is 0, violating `runAsNonRoot: true`.
5.  **Baseline vs. Restricted:** Under **Baseline**, running as root is allowed. Under **Restricted**, running as root is forbidden.
6.  **Result:** The Pod Security Admission Controller (PSA) should have blocked creation. If pods are CrashLooping, they were likely admitted (e.g., policy applied after creation or PSA misconfigured) but the container runtime or application crashes immediately because it attempts to run as root while the security context enforces non-root, or the application fails permission checks on startup due to the mismatch.

SC1-C
1.  **Image Fix:** Rebuild the image with a non-root user (e.g., `USER 1000`) or pin the image digest to a known-good version that complies with Restricted PSS.
2.  **Manifest Fix:** Explicitly set `securityContext.runAsUser: 1000` (or matching non-root UID) in the Pod/Deployment spec.
3.  **CI Gate:** Integrate a PSS compliance check (e.g., `kube-score` or `conftest`) in the CI pipeline to reject manifests that violate Restricted PSS before deployment.
4.  **Rollout:** Use a rolling update strategy (`maxSurge: 1`, `maxUnavailable: 0`) to replace pods gradually. Monitor CrashLoopBackOff counts in the new pods before scaling up. If failures persist, rollback the image immediately.

SC2-A
1.  **Stale Data:** Caused by **replication lag**. Cross-region async replication means the replica receives WAL records after the primary commits.
2.  **WAL Shipping:** PostgreSQL streams WAL records asynchronously. If the primary crashes before WAL is shipped to the replica, that data is lost.
3.  **Data Loss Window:** The time between the last committed transaction on the primary and the last shipped WAL record on the replica.
4.  **Quantification:** Check `pg_stat_replication` on the primary (specifically `write_lag`, `flush_lag`, `replay_lag`) and the replica's `pg_last_wal_receive_lsn()` vs `pg_last_wal_replay_lsn()`.

SC2-B
1.  **Import Replica:** `terraform import aws_db_instance.promoted_replica <arn_of_promoted_replica>`
2.  **Remove Old:** `terraform state rm aws_db_instance.old_primary`
3.  **Update State:** Run `terraform plan` to verify the promoted replica is now the primary in state.
4.  **Update References:** Update Terraform variables/outputs to point to the new instance ARN. Run `terraform apply` to update dependent resources (Security Groups, Route53) to reference the new ARN.

SC2-C
1.  **Architecture:** Use **Aurora Global Database**. It provides native cross-region replication with RPO < 1 second and automatic failover.
2.  **Connection Management:** Use **RDS Proxy** instead of PgBouncer. It handles connection pooling centrally and supports failover transparently without application reconnection storms.
3.  **DNS Failover:** Use **Route53 Health Checks** with Failover Routing Policy. Point the primary endpoint to the Chicago region and the secondary to EU.
4.  **Retry Logic:** Implement exponential backoff with jitter in the application connection pool.
5.  **Recommendation:** Aurora Global DB is recommended over standard RDS because it manages replication lag and failover at the storage layer, ensuring lower RPO/RTO and eliminating manual promotion steps.

SC4-A
1.  **Local Sessions:** Kubernetes pods are ephemeral. **Solution:** Move sessions to Redis.
2.  **NFS Mount:** Kubernetes requires CSI drivers for persistent storage. **Solution:** Use NFS CSI driver with ReadWriteMany.
3.  **On-Prem DB:** Kubernetes is in the cloud. **Solution:** Use AWS RDS or migrate DB to cloud (VPC Peering/Transit Gateway).
4.  **IBM MQ:** On-prem mainframe integration. **Solution:** Use MQ Bridge or migrate to RabbitMQ/Kafka in cloud.
5.  **Sticky Sessions:** F5 relies on IP affinity. **Solution:** Use Kubernetes Ingress with session affinity or move session state to Redis.

SELECT blocked_locks.pid AS blocked_pid,
       blocked_activity.query AS blocked_query,
       blocking_locks.pid AS blocking_pid,
       blocking_activity.query AS blocking_query
FROM pg_catalog.pg_locks blocked_locks
JOIN pg_catalog.pg_stat_activity blocked_activity ON blocked_locks.pid = blocked_activity.pid
JOIN pg_catalog.pg_locks blocking_locks ON blocking_locks.locktype = blocked_locks.locktype
AND blocking_locks.database IS NOT DISTINCT FROM blocked_locks.database
AND blocking_locks.relation IS NOT DISTINCT FROM blocked_locks.relation
AND blocking_locks.page IS NOT DISTINCT FROM blocked_locks.page
AND blocking_locks.tuple IS NOT DISTINCT FROM blocked_locks.tuple
AND blocking_locks.virtualxid IS NOT DISTINCT FROM blocked_locks.virtualxid
AND blocking_locks.transactionid IS NOT DISTINCT FROM blocked_locks.transactionid
AND blocking_locks.classid IS NOT DISTINCT FROM blocked_locks.classid
AND blocking_locks.objid IS NOT DISTINCT FROM blocked_locks.objid
AND blocking_locks.objsubid IS NOT DISTINCT FROM blocked_locks.objsubid
AND blocking_locks.pid != blocked_locks.pid
JOIN pg_catalog.pg_stat_activity blocking_activity ON blocking_locks.pid = blocking_activity.pid
WHERE NOT blocked_locks.granted;

SC6-C
1.  **Direct Connect:** Best for bandwidth (10Gbps+), low latency, dedicated. Costly. Fits Zero-Trust (private link).
2.  **ECMP VPN:** Cheaper, uses existing internet. Higher latency, less reliable. Fails over via BGP.
3.  **Mesh (Tailscale/Netbird):** Easiest to deploy, zero-config. Good for dev/staging. Not recommended for production due to control plane dependency.
4.  **Routing:** DX requires BGP peering. VPN uses static routes. Mesh uses overlay routing.
5.  **Recommendation:** Upgrade to **Direct Connect** for production to meet bandwidth and reliability requirements while maintaining Zero-Trust via private links.

SC10-A
1.  **Active-Passive (Velero):** RPO depends on backup frequency (15 mins). RTO depends on restore time. 200TB MinIO requires significant restore time.
2.  **Active-Active (Istio):** RPO/RTO better, but complex. Requires stateful replication.
3.  **PostgreSQL:** Patroni with synchronous replication (Chicago -> Dallas).
4.  **Redis:** Sentinel with cross-region replication.
5.  **RabbitMQ:** Federation or Quorum Queues.
6.  **MinIO:** Site Replication (S3 compatible).
7.  **Recommendation:** Active-Passive with Velero + Patroni Sync is safer for RPO < 15 mins given 12ms latency. Active-Active is risky for stateful apps with 200TB data.
