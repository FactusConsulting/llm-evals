K1 — Easy: A **Deployment** manages stateless applications, allowing pods to be replaced freely with no guaranteed identity. A **StatefulSet** manages stateful applications, providing stable, unique network identities (hostnames) and ordered, graceful deployment/termination.

K2 — Easy: A **ClusterIP** Service creates a virtual IP inside the cluster. **kube-proxy** (using iptables or IPVS) updates the node's network rules to route traffic destined for the Service IP to the IP addresses of the backing pods (Endpoints).

K3 — Medium: A **PersistentVolume (PV)** is a cluster-level storage resource. A **PersistentVolumeClaim (PVC)** is a user request for storage. A **StorageClass** defines the provisioner and parameters for dynamic provisioning. A PVC binds to a PV (static) or triggers a PV creation via StorageClass (dynamic).

K4 — Medium: 1. Check `kubectl logs` for application errors. 2. Check `kubectl describe pod` for events (OOMKilled, CrashLoopBackOff). 3. Verify resource limits (CPU/Memory) are not exceeded. 4. Check liveness/readiness probe configurations and health endpoint availability.

K5 — Medium: **CoreDNS** provides DNS resolution for services and pods within the cluster. If CoreDNS pods are down, internal service discovery (e.g., `service-name.namespace.svc`) fails, breaking inter-service communication.

K6 — Hard: The scheduler uses **Filtering** (Predicate) to eliminate nodes that don't meet requirements (e.g., taints, resource capacity) and **Scoring** (Preference) to rank remaining nodes. Factors: 1. Resource requests (CPU/Memory). 2. Node Affinity/Taints. 3. Pod Anti-Affinity.

K7 — Hard: **Yes**, it will be scheduled. 4GB - 3.5GB = 500MB free; 256Mi request fits. At runtime, if it uses 600Mi (exceeding 512Mi limit), the kernel will **OOMKill** the container, causing a CrashLoopBackOff.

K8 — Hard: 1. Request hits External LB. 2. LB forwards to Ingress Controller (NGINX). 3. NGINX matches Ingress rules to Service. 4. Service routes to Pod IPs via **Endpoints**. 5. **kube-proxy** (iptables/IPVS) on the node directs traffic to the specific Pod IP.

K9 — Medium: A **NetworkPolicy** restricts traffic flow between pods. Default behavior (no policy): All traffic is allowed. When a policy is applied: Traffic is denied by default unless explicitly allowed by the policy rules.

K10 — Medium: **RollingUpdate** replaces pods gradually (zero downtime). **Recreate** kills all existing pods before starting new ones. Choose **Recreate** for stateful applications that cannot handle concurrent versions or require full shutdown before upgrade.

K11 — Easy: A **Namespace** provides logical isolation within a cluster. Default namespaces: `default`, `kube-system`, `kube-public`, `kube-node-lease`.

K12 — Easy: `kubectl describe` shows detailed status and events. 1. **Events** (warnings/errors). 2. **Conditions** (Ready, PodScheduled). 3. **Mounts/Env vars** (detailed configuration). `kubectl get` only shows summary status.

K13 — Easy: A **ConfigMap** stores non-sensitive configuration data. Pods consume it via: 1. **Environment Variables**. 2. **Volume Mounts** (files).

K14 — Easy: **ClusterIP**: Internal cluster access only. **NodePort**: Exposes service on a static port on every node's IP. **LoadBalancer**: Provisions an external cloud load balancer with a public IP.

K15 — Easy: A **DaemonSet** ensures a copy of a pod runs on all (or specific) nodes. Use for logging agents, monitoring daemons, or storage drivers where node-level access is required.

K16 — Medium: **Taints** repel pods from nodes; **Tolerations** allow pods to schedule on tainted nodes. **Node Affinity** attracts pods to nodes. Use Case: Taint/Toleration for dedicated hardware (e.g., GPU); Affinity for grouping related services.

K17 — Medium: An **Operator** is a custom controller that manages complex stateful applications using CRDs. It differs from a standard controller by encoding operational knowledge (lifecycle, scaling, backup). It solves the complexity of managing stateful apps (e.g., databases) via automation.

K18 — Hard: etcd uses **Strong Consistency** (linearizable). If quorum is lost, the cluster becomes read-only/unavailable. Recovery: Restore from a snapshot on a new node, rejoin it to the cluster, and ensure quorum is restored (e.g., 2 out of 3 nodes must be healthy).

K19 — Hard: **Role** (namespace-scoped permissions), **ClusterRole** (cluster-wide), **RoleBinding** (links Role to users/NS), **ClusterRoleBinding** (links ClusterRole to users/cluster).
```yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata: namespace: dev
name: pod-reader
rules: - apiGroups: [""] resources: ["pods", "pods/log"] verbs: ["get", "list", "watch"]
---
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata: namespace: dev
name: read-pods
subjects: - kind: User name: developer
roleRef: kind: Role name: pod-reader apiGroup: rbac.authorization.k8s.io
```

K20 — Hard: **ndots** defines how many dots trigger an FQDN search. Default 5 causes queries for `google.com` to try `google.com`, `google.com.ns1`, etc. Fix: Set `dnsConfig.options: [{name: "ndots", value: "1"}]` to reduce queries.

D1 — Easy: **Rebase** rewrites history to create a linear commit path. **Merge** preserves history with a merge commit. Avoid rebasing on shared/public branches to prevent disrupting collaborators.

D2 — Easy: **TCP** is connection-oriented and reliable (HTTP, SSH). **UDP** is connectionless and faster (DNS, VoIP).

D3 — Medium: A **race condition** occurs when output depends on the sequence of uncontrollable events. Example: Two threads incrementing a shared counter without a lock results in lost updates.

D4 — Medium: **Symmetric** uses one key for encryption/decryption. **Asymmetric** uses a public/private key pair. TLS uses Asymmetric for the handshake (key exchange) and Symmetric for the data session (performance).

D5 — Medium: A **layer** is a read-only filesystem diff in a Docker image. **Caching**: Place frequently changing commands (e.g., `COPY .`) after stable commands (e.g., `RUN apt-get`) to maximize cache hits during builds.

D6 — Medium: 1. Check upstream service health. 2. Review application logs for errors. 3. Verify database connectivity. 4. Check resource limits (CPU/Memory throttling).

D7 — Hard: **CAP** states a distributed system can only guarantee 2 of 3: Consistency, Availability, Partition Tolerance. **CP**: ZooKeeper (chooses consistency during partition). **AP**: Cassandra (chooses availability during partition).

D8 — Medium: **Horizontal** adds more nodes; **Vertical** increases node capacity. Patterns enabling horizontal scaling: Stateless architecture, Sharding, Load Balancing.

D9 — Easy: An **environment variable** is a dynamic value affecting process behavior. Persist across sessions on Linux by adding `export VAR=value` to `~/.bashrc` or `/etc/environment`.

D10 — Easy: **JSON** is strict key-value pairs; **YAML** is human-readable with indentation. Advantage JSON: Native language support. Advantage YAML: Less verbose for config.

D11 — Easy: **REST** is an architectural style for APIs. **GET** (retrieve), **POST** (create), **PUT** (replace), **DELETE** (remove).

D12 — Easy: **Version pinning** locks dependencies to specific versions (e.g., `1.0.1` vs `^1.0.0`). Important for reproducibility to prevent unexpected breakage from upstream updates.

D13 — Easy: **Compiled** (source to binary, e.g., C, Go). **Interpreted** (source executed directly, e.g., Python, JavaScript).

D14 — Medium: **Reverse Proxy** sits in front of servers (client -> proxy -> server). **Forward Proxy** sits in front of clients (client -> proxy -> internet). Examples: Nginx (web traffic), HAProxy (L4 load balancing).

D15 — Medium: A **deadlock** is a permanent blocking of processes. **Coffman Conditions**: 1. Mutual Exclusion. 2. Hold and Wait. 3. No Preemption. 4. Circular Wait.

D16 — Medium: **Eventual Consistency** ensures updates propagate over time. Cassandra uses **Read Repair** and **Hinted Handoff**. Tunable levels: **ONE** (fastest), **QUORUM** (balanced), **ALL** (strongest).

D17 — Medium: **mTLS** authenticates both client and server via certificates. Handshake: Server sends cert, Client verifies; Client sends cert, Server verifies. Used in **Service Meshes** (e.g., Istio) for secure service-to-service communication.

D18 — Medium: A **Hash Table** maps keys to indices via a hash function. **Collision Resolution**: Chaining (linked lists) or Open Addressing (probing). Complexity: Avg O(1), Worst O(n) (all keys collide).

D19 — Medium: **Circuit Breaker** prevents cascade failures. **Closed** (normal), **Open** (fail fast), **Half-Open** (test recovery). Relates to **Retry/Backoff** by stopping retries when the circuit is open to reduce load.

D20 — Medium: **Optimistic** (check conflicts at commit, e.g., versioning). **Pessimistic** (lock resources immediately, e.g., `SELECT FOR UPDATE`). Choose Optimistic for high concurrency/low conflict; Pessimistic for high conflict/consistency critical.
