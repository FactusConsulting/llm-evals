G1 — Easy: An array is a fixed-size value type `[T; N]` where the size is part of the type. A slice is a dynamic view into an array `[]T`, containing a pointer, length, and capacity. When appending to a slice that reaches capacity, the Go runtime allocates a new, larger underlying array (usually doubling capacity), copies the existing elements, and returns a new slice header pointing to it.

G2 — Easy: `var x int` declares a variable with an explicit type and initializes it to the zero value (0). `x := 0` uses short variable declaration, inferring the type from the value. You cannot use `:=` to declare a variable that already exists in the same scope (e.g., redeclaring `x` in the same block).

G3 — Medium: A goroutine is a lightweight thread managed by the Go runtime, not the OS. It differs from an OS thread by having a smaller stack (starting at 2KB) and being multiplexed onto fewer OS threads. The Go scheduler (GMP model) maps Goroutines (G) to OS Threads (M) via Processors (P), which are logical CPUs. The scheduler handles work stealing and preemption to maximize CPU utilization.

G4 — Medium: Channels are typed conduits for communication between goroutines. Unbuffered channels block both sender and receiver until both are ready. Buffered channels allow sending without blocking up to their capacity. Sending to a full buffered channel blocks the sender until a receiver reads from the channel.

G5 — Medium: The `context` package manages deadlines, cancellation signals, and request-scoped values across API boundaries.
```go
ctx, cancel := context.WithTimeout(context.Background(), 5*time.Second)
defer cancel()
req, _ := http.NewRequestWithContext(ctx, "GET", url, nil)
resp, err := client.Do(req)
```

G6 — Medium: Go interfaces are satisfied implicitly (duck typing); a type satisfies an interface if it implements the methods, without explicit declaration. Java/C# require explicit `implements` or `: interface`. The empty interface `any` (formerly `interface{}`) accepts any type. It is powerful for generic containers but dangerous because it requires type assertions, losing compile-time type safety.

G7 — Hard: This code is unsafe due to a data race: one goroutine writes `data` and `ready`, while the main goroutine reads `data` without synchronization. The compiler may reorder instructions, and the main goroutine might see `ready=true` before `data` is updated. Fix by using `sync.Mutex` to protect `data` and `ready`, or use a channel to signal readiness and pass data.

G8 — Medium: `sync.Mutex` allows exclusive access (one writer or one reader). `sync.RWMutex` allows multiple concurrent readers but exclusive writers. `sync.Map` is optimized for concurrent reads with sparse keys or when keys are known at runtime. It trades some write performance for read performance compared to a `map` protected by a `Mutex`.

G9 — Hard: Go modules manage dependencies. `go.mod` lists direct dependencies and the Go version. `go.sum` contains cryptographic checksums for reproducible builds. `go mod tidy` removes unused imports and adds missing dependencies. Minimum Version Selection (MVS) selects the lowest version of a dependency that satisfies all requirements in the dependency graph.

G10 — Medium: `defer` schedules a function call to run after the surrounding function returns. Multiple defers execute in LIFO (Last In, First Out) order. A common gotcha in loops is capturing the loop variable by reference in a deferred closure, causing all deferred calls to use the final loop value. Capture the variable in a new scope or parameter.

G11 — Easy: A pointer holds the memory address of a value. The zero value of a pointer is `nil`. To create a pointer to a struct literal, use the address-of operator: `&MyStruct{Field: value}`.

G12 — Easy: `init()` is a function executed automatically before `main()` runs. It runs during package initialization. A package can have multiple `init()` functions; they execute in source file order, and within a file, in declaration order.

G13 — Easy: `new(T)` allocates zeroed memory for type `T` and returns a pointer `*T`. `make(T)` initializes built-in types (slice, map, channel) and returns the initialized type (not a pointer). `new` works with any type; `make` only works with built-in reference types.

G14 — Easy: Errors are values implementing the `error` interface (`Error() string`). The idiomatic pattern is `if err != nil { return err }`. `errors.Is` checks if an error matches a target (unwrapping). `errors.As` checks if an error matches a specific type and assigns it to a variable. `==` only checks exact error values, not wrapped errors.

G15 — Easy: Built-in composite types are `map`, `slice`, `channel`, and `struct`. Their zero values are: `map` (nil), `slice` (nil), `channel` (nil), `struct` (fields set to their zero values).

G16 — Medium: A race condition occurs when multiple goroutines access shared data concurrently, with at least one write, without synchronization. The `-race` flag instruments code to detect these at runtime. Fix by using `sync.Mutex` to lock access or channels to transfer ownership of data.

G17 — Medium: `select` listens on multiple channel operations simultaneously. Unlike `switch`, it evaluates channel readiness at runtime. If multiple cases are ready, one is chosen pseudo-randomly. Implement timeout by selecting on a channel and a `time.After` channel.

G18 — Hard: Escape analysis determines if a variable lives on the stack or heap. If a variable's address escapes the function scope, it is allocated on the heap. Inspect with `go build -gcflags="-m"`. Heap allocation increases GC pressure; stack allocation is faster. Understanding this helps optimize performance-critical code.

G19 — Hard: `reflect` allows runtime inspection of types and values. `reflect.Type` describes the type; `reflect.Value` holds the value. Reflection is slow due to dynamic dispatch and type checking. It is justified for frameworks (serialization, dependency injection) but avoided in hot paths. Code generation (e.g., `go generate`) is preferred for performance.

G20 — Hard: `go:embed` embeds files/directories at compile time into the binary. `//go:embed file.txt` embeds a single file. `//go:embed dir/` embeds a directory recursively. It interacts with `io/fs.FS` by returning an `embed.FS` type. Limitations: files must exist at compile time, and paths are relative to the build directory.

R1 — Easy: Ownership is Rust's core memory safety feature. Rules: 1) Each value has one owner. 2) When the owner goes out of scope, the value is dropped. 3) Values can be moved, but not copied unless `Copy` is implemented.

R2 — Easy: `String` is an owned, heap-allocated, growable string. `&str` is a borrowed string slice (reference to UTF-8 bytes). Use `String` for owned data or when mutation is needed. Use `&str` for function arguments to avoid allocation and allow borrowing from string literals.

R3 — Medium: Borrowing allows temporary access to data. Rules: 1) You can have either one mutable reference `&mut T` OR any number of immutable references `&T`. 2) References must always be valid. The borrow checker enforces these rules at compile time to prevent data races.

R4 — Medium: `Result<T, E>` represents success (`Ok(T)`) or failure (`Err(E)`). `Option<T>` represents a value that may be present (`Some(T)`) or absent (`None`). The `?` operator propagates errors.
```rust
fn read_file(path: &str) -> Result<String, std::io::Error> {
    std::fs::read_to_string(path)
}
```

R5 — Medium: `Box<T>` allocates `T` on the heap with single ownership. `Rc<T>` (Reference Counted) allows multiple owners in a single thread. `Arc<T>` (Atomic Reference Counted) allows multiple owners across threads. Use `Box` for heap allocation, `Rc` for shared single-threaded ownership, `Arc` for shared multi-threaded ownership.

R6 — Medium: Traits define shared behavior. Unlike Go interfaces, traits can have default implementations. `dyn Trait` enables dynamic dispatch (vtable) at runtime. Generic `T: Trait` enables static dispatch (monomorphization) at compile-time, which is faster but less flexible for heterogeneous collections.

R7 — Hard: The function fails because the compiler cannot determine which input's lifetime the return value refers to. Fix by adding a lifetime parameter:
```rust
fn longest<'a>(x: &'a str, y: &'a str) -> &'a str {
    if x.len() > y.len() { x } else { y }
}
```

R8 — Hard: `Send` means a type can be transferred to another thread. `Sync` means `&T` can be shared safely. `Rc<T>` is not `Send` because its reference count is not atomic, leading to races if shared across threads. `Arc<T>` uses atomic reference counting, making it `Send` and `Sync`.

R9 — Medium: `async/await` creates a state machine (Future) that can be paused and resumed. The executor (e.g., Tokio) polls the Future to completion. Unlike Go's green threads (which are preemptive), Rust futures are cooperative; the executor drives them, allowing fine-grained control over concurrency without OS thread overhead.

R10 — Medium: `unsafe` allows operations the compiler cannot verify for safety. 1) Dereferencing raw pointers. 2) Calling unsafe functions. 3) Accessing mutable statics. Justified when interfacing with C (FFI), implementing low-level primitives, or optimizing performance-critical code where safety can be manually verified.

R11 — Easy: Pattern matching deconstructs values. `match` requires exhaustive handling of all cases. `if let` is a shorthand for matching a single pattern. The compiler enforces exhaustiveness in `match` to prevent logic errors from unhandled cases.

R12 — Easy: `Vec<T>` is a growable heap-allocated array. `[T; N]` is a fixed-size stack-allocated array. When `Vec` capacity is exceeded, it allocates a new larger buffer (usually doubling), copies elements, and drops the old buffer.

R13 — Easy: Enums in Rust are Algebraic Data Types (ADTs) that can hold data. Unlike C/Java enums, they can store variants with different types. `Option<T>` is an enum (`Some(T)` or `None`) used instead of `null` to enforce handling of missing values at compile time.

R14 — Easy: `impl` blocks define methods for a type. `&self` borrows the instance, `&mut self` mutably borrows it, `self` takes ownership. Associated functions (e.g., `String::new()`) do not take `self` and act like constructors.

R15 — Easy: `derive` macros automatically implement traits. Common traits: `Debug`, `Clone`, `PartialEq`. You cannot derive traits that require custom logic or have lifetimes that cannot be inferred (e.g., `Clone` on types with lifetimes).

R16 — Medium: `Cow<'a, T>` (Clone on Write) holds either a borrowed `&'a T` or an owned `Box<T>`. It avoids allocation if the data is borrowed. If mutation is requested, it clones the data to owned memory. Useful for functions that return data that might be borrowed or owned.

R17 — Medium: `mod` declares a module. `use` imports items. `pub` makes items public. Modules map to files (`mod.rs` or `module.rs`). Rust 2018 edition improved path resolution (e.g., `crate::` vs `self::`) and allowed `use` paths to be relative to the crate root more intuitively.

R18 — Hard: `Pin<T>` prevents a value from being moved in memory. Needed for self-referential structs and async/await (where the Future might be polled while pinned). `Unpin` is a marker trait indicating a type can be moved safely. Most types are `Unpin`; `Pin` is used for types that are not.

R19 — Hard: `macro_rules!` are declarative macros (pattern matching on tokens). Procedural macros (derive, attribute, function-like) are Rust code that generates Rust code. Hygiene ensures macro variables don't leak into the caller's scope. Procedural macros are more powerful but harder to write.

R20 — Hard: Zero-cost abstractions mean high-level constructs compile to efficient machine code. Generics use monomorphization (code duplication per type) instead of virtual dispatch. Iterators are inlined. This allows Rust to match C performance while providing safety and ergonomics, as the abstraction overhead is eliminated at compile time.
