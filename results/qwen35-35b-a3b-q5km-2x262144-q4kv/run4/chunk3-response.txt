T1 — Easy: `tofu plan` performs a dry run, calculating the execution plan and displaying the changes without modifying infrastructure. `tofu apply` executes the plan, creating, updating, or destroying resources to match the configuration.

T2 — Easy: State is a JSON file tracking the current state of managed resources. Remote state is important for team collaboration, enabling state locking to prevent concurrent writes, and securing sensitive data (like credentials) outside local machines.

T3 — Medium: `count` uses integer indices, while `for_each` uses a map or set of strings. `for_each` is preferred when the number of resources changes dynamically, as it avoids index shifting and ensures stable resource IDs when items are added or removed.

T4 — Medium: This is "drift." OpenTofu detects it during `tofu plan` (via state refresh). If the resource is modified outside OpenTofu, the next plan will show a difference. You can either revert the manual change or use `tofu import` to sync the state.

T5 — Medium:
```hcl
resource "aws_instance" "web" {
  for_each = toset(["web1", "web2", "web3"])
  ami        = "ami-12345678"
  instance_type = "t2.micro"
  tags = {
    Name = each.key
  }
}
```

T6 — Hard: `terraform_remote_state` reads the entire state file of another module, coupling them tightly. Output values are preferred because they expose only specific data, decoupling modules and improving security. The recommended alternative is using `output` values in the source module and referencing them via `module` blocks in the consumer.

T7 — Hard: The `null_resource` provisioner runs asynchronously during apply. If dependent resources read the output before the provisioner finishes writing the kubeconfig, the value is empty. Fix: Add `depends_on = [null_resource.kubeconfig]` to the dependent resources to enforce execution order.

T8 — Medium: `tofu import` is a CLI command to import existing resources into state. Limitations: requires manual mapping of attributes. The `import` block (1.5+) allows importing within configuration, enabling version-controlled import logic and better CI/CD integration.

T9 — Easy: A `variable` is an input parameter passed into a module or root configuration. An `output` is a value exposed from a module or root configuration to be used by other modules or the CLI.

T10 — Easy: A provider is a plugin that interacts with a specific API (e.g., AWS, Azure). Running `tofu plan` without `tofu init` fails because the provider plugins are not downloaded or initialized, causing "Provider not installed" errors.

T11 — Easy: `.terraform.lock.hcl` records the exact provider versions and checksums used. It should be committed to ensure reproducible builds and prevent accidental provider upgrades that might break infrastructure.

T12 — Easy: `tofu destroy` removes all resources defined in the configuration. To target a single resource, use the `-target` flag (e.g., `tofu destroy -target=aws_instance.web`).

T13 — Easy: A data source reads data from an external system without managing its lifecycle. Use one when you need to reference existing resources (e.g., looking up an AMI ID by name) rather than creating new ones.

T14 — Medium: `variable` blocks define inputs from outside the module. `locals` define computed values used internally within the module. Use `locals` for intermediate calculations or formatting to keep configuration clean, avoiding the need for external inputs.

T15 — Medium: `lifecycle` manages resource behavior. `create_before_destroy` creates the new resource before destroying the old (zero-downtime). `prevent_destroy` prevents accidental deletion (e.g., production DBs). `ignore_changes` ignores drift for specific attributes (e.g., tags managed externally).

T16 — Medium: A `moved` block allows renaming or moving a resource to a new module path without changing its state ID. It helps refactor code structure without triggering resource recreation, preserving the state mapping.

T17 — Hard: The dependency graph determines execution order. Implicit dependencies are inferred from references (e.g., `aws_instance.id`). Explicit dependencies use `depends_on`. `depends_on` reduces parallelism by forcing serialization, which should be used sparingly to avoid slowing down apply times.

T18 — Hard: 1. `refresh=false`: Skips API calls to check drift (Tradeoff: Misses external changes). 2. Reduce Data Sources: Minimize API calls (Tradeoff: Less dynamic data). 3. Modularize State: Split large state into smaller modules (Tradeoff: More complex management). 4. Prune State: Remove unused resources (Tradeoff: Loss of history).

T19 — Hard: State locking prevents concurrent writes. If a lock isn't released (e.g., CI crash), subsequent plans fail. Manually break it with `tofu force-unlock <LOCK_ID>`. Risk: Concurrent writes can corrupt the state file if not handled carefully.

T20 — Hard: 1. Backup local state. 2. Add `backend` block to config with S3/DynamoDB settings. 3. Run `tofu init -migrate-state`. 4. Verify state exists in S3. 5. Run `tofu plan` to ensure consistency. Existing state file is uploaded to S3 during init.

A1 — Easy: A **Playbook** is a YAML file defining the execution steps. A **Role** is a structured directory for organizing tasks, handlers, and variables. A **Task** is a single action (e.g., install a package) within a playbook or role.

A2 — Easy: `ansible_facts` is a dictionary of system information gathered by the `setup` module. Access the OS family via `{{ ansible_os_family }}`.

A3 — Medium: `command` runs without a shell (no pipes/redirection). `shell` runs via a shell (supports pipes/redirection). `raw` runs without SSH setup (no Python module transfer), useful for initial bootstrap.

A4 — Medium: Idempotency means running a task multiple times yields the same result. Example (Idempotent): `copy` module (only writes if content differs). Example (Not Idempotent): `shell: rm -rf /tmp/data` (deletes every time).

A5 — Medium:
```yaml
- name: Install htop
  package:
    name: htop
  become: yes
  when: ansible_os_family == "Debian"

- name: Install htop on RedHat
  package:
    name:
      - epel-release
      - htop
  become: yes
  when: ansible_os_family == "RedHat"
```

A6 — Hard: Precedence (High to Low): Extra vars > Play vars > Host vars > Group vars > Role vars > Role defaults. `set_fact` overrides most at runtime.

A7 — Hard: Use `serial: 3` in the play to limit concurrency. Group nodes into `rke2_first` and `rke2_rest`. Use `wait_for` to ensure the first 3 nodes are ready (e.g., checking a port) before the rest start.

A8 — Medium: `ansible-vault` encrypts sensitive data. To encrypt a single variable: `ansible-vault edit --encrypt-string --vault-password-file <file> -c "variable_name: value"`.

A9 — Easy: An inventory file lists hosts and groups. **Static** inventory is a manual file (INI/YAML). **Dynamic** inventory is a script or API that generates host lists dynamically (e.g., from cloud providers).

A10 — Easy: `ansible webservers -m shell -a "df -h"`.

A11 — Easy: A **handler** is a task that runs only when notified by another task (e.g., `notify: restart service`). It differs from a regular task by being conditional on a `notify` trigger and running at the end of the play.

A12 — Easy: `ansible.cfg` configures Ansible behavior. Common settings: `forks` (parallelism), `host_key_checking` (security), `timeout` (connection timeout).

A13 — Easy: `register` saves a task's output to a variable. Use it in subsequent tasks via `{{ registered_variable.stdout }}` or `{{ registered_variable.failed }}`.

A14 — Medium: `import_tasks` is evaluated at parse time (static). `include_tasks` is evaluated at runtime (dynamic). Use `include_tasks` when you need conditionals to determine which tasks to run.

A15 — Medium: Ansible Galaxy is a repository for roles and collections. Use `requirements.yml` to list dependencies and run `ansible-galaxy install -r requirements.yml` to install them.

A16 — Medium: Connection plugins manage how Ansible connects to hosts. `ssh` (remote Linux), `local` (control node), `network_cli` (network devices). Use `ssh` for servers, `network_cli` for routers/switches.

A17 — Hard: `delegate_to: localhost` runs a task on the control node. `run_once: true` ensures it runs once even in a loop. Example: Gather facts on remote hosts, then process the aggregated data on localhost.

A18 — Hard: 1. Increase `forks` (Tradeoff: More memory). 2. Enable `pipelining` (Tradeoff: SSH config changes). 3. Enable `fact_caching` (Tradeoff: Cache invalidation complexity). 4. Use `Mitogen` (Tradeoff: Additional dependency).

A19 — Hard: `{{ }}` outputs variables. `{% %}` controls logic (loops/if). `{# #}` comments.
```jinja
upstream backend_servers {
{% for server in backend_servers %}
    server {{ server.ip }} weight={{ server.weight }}; # Health check: {{ server.health_check }}
{% endfor %}
}
```

A20 — Hard: 1. Set `serial: 5` for batches. 2. `pre_tasks`: Deregister from LB. 3. `tasks`: Deploy app. 4. `post_tasks`: Health check via `uri` module. 5. `block/rescue`: If health check fails, `rescue` triggers rollback (revert code, re-register LB).
