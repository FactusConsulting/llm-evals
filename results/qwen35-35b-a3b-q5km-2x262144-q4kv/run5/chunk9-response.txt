SC1-A
The failure is caused by a conflict between the **Restricted Pod Security Standard (PSS)** and the container configuration.
1.  **Restricted PSS:** Requires containers to run as non-root users (`runAsNonRoot: true`) and explicitly defines `runAsUser` (UID) to prevent privilege escalation. It forbids `privileged: true` and requires specific capabilities.
2.  **Image Behavior:** The new base image (`cgr.dev/chainguard/static:latest`) defaults `USER` to root (UID 0).
3.  **Manifest Behavior:** The pods have `securityContext.runAsNonRoot: true` but **no** `securityContext.runAsUser` set.
4.  **The Conflict:** When the Restricted PSS admission controller evaluates the pod, it sees `runAsNonRoot: true` (good) but the image defaults to root. Since `runAsUser` is not explicitly set in the pod spec to override the image default, the container attempts to run as root. The Restricted policy forbids running as root (UID 0).
5.  **Baseline vs. Restricted:** Baseline allows `runAsNonRoot: true` without enforcing a specific UID, allowing the container to run as root if the image defaults to it. Restricted enforces a non-zero UID.
6.  **Result:** The kubelet rejects the pod creation or the container crashes immediately upon startup because it cannot satisfy the security policy, leading to `CrashLoopBackOff`.

SC1-C
**Immediate Remediation:**
1.  **Rollback:** Revert the affected pods to the previous image version (before the base image change) to restore service immediately.
2.  **Manifest Fix:** Update the Deployment manifests to explicitly set `securityContext.runAsUser: 1000` (or the specific non-root UID required by the new image) and ensure `runAsNonRoot: true` remains.
3.  **Image Fix:** Rebuild the container images with a non-root user defined in the Dockerfile (`USER 1000`) to align with the Restricted policy.

**Prevention & CI Gate:**
1.  **CI Gate:** Integrate a PSS validation tool (like `kube-score` or `checkov`) into the CI pipeline to validate manifests against the target namespace's PSS level before merging.
2.  **Admission Controller:** Ensure the `PodSecurity` admission controller is enabled in the cluster with `enforce: restricted` for production namespaces.
3.  **Image Scanning:** Enforce image scanning (Trivy/Grype) to detect root users in the final image layer.

**Rollout Strategy:**
1.  **Canary:** Deploy the fixed manifests to a single namespace or a small subset of pods (5%) first.
2.  **Monitor:** Watch for CrashLoopBackOff and readiness probe failures.
3.  **Expand:** Gradually increase traffic to 100% once stability is confirmed.
4.  **Rollback Plan:** Maintain the previous image tag in the registry for immediate rollback if the new image fails validation.

SC2-A
**Stale Data & Replication Lag:**
1.  **Cause:** Cross-region replication is typically asynchronous. When the primary (us-east-1) fails, the replica (eu-west-1) is promoted. Any data written to the primary but not yet shipped to the replica is lost.
2.  **WAL Shipping:** PostgreSQL streams Write-Ahead Logs (WAL) to the replica. In async mode, the primary does not wait for the replica to acknowledge receipt before committing.
3.  **Data Loss:** The "data loss window" is the time between the last committed transaction on the primary and the last received transaction on the replica.
4.  **Quantification:** Check `pg_stat_replication` on the promoted replica (before promotion) for `replay_lag` and `write_lag`. After promotion, check the application logs for "stale data" timestamps vs. database transaction IDs.

SC2-C
**Architecture Design:**
1.  **Database:** Switch to **Amazon Aurora Global Database**. It provides native cross-region replication with sub-second RPO (<1s) and automatic failover. Standard RDS async replication has higher RPO risk.
2.  **Connection Management:** Use **RDS Proxy** instead of PgBouncer. RDS Proxy handles connection pooling and failover transparently at the AWS layer, reducing connection churn during failover.
3.  **DNS Failover:** Use **Route53 Latency-Based Routing** with Health Checks. The primary endpoint is us-east-1; the secondary is eu-west-1. If us-east-1 fails health checks, Route53 routes traffic to eu-west-1.
4.  **Application Logic:** Implement exponential backoff retry logic with jitter in the application code to handle transient connection errors during failover.
5.  **RPO < 1s:** Achieved via Aurora Global Database's synchronous replication across regions (using the Aurora Multi-Master or Global Database feature).

SC3-A

1.  **Containment:** Do not kill pods immediately. Cordon the nodes running the affected pods to prevent scheduling new workloads.

SC4-A
**Top 5 Technical Blockers:**
1.  **Local Sessions (Sticky Sessions):**
    *   *Problem:* Kubernetes pods are ephemeral; local memory sessions break on restart.
    *   *Solution:* Migrate sessions to a distributed store like **Redis** or **Memcached**.
2.  **Local NFS Mounts:**
    *   *Problem:* Kubernetes pods are scheduled on different nodes; local disk is not shared.
    *   *Solution:* Use **NFS CSI Driver** with `ReadWriteMany` access mode or migrate to object storage (S3).
3.  **Oracle 19c (On-Prem):**
    *   *Problem:* Network latency and licensing for cloud DBs.
    *   *Solution:* Keep Oracle on-prem but optimize network (Direct Connect) or migrate to **Amazon RDS for Oracle** or **Aurora PostgreSQL**.
4.  **IBM MQ Integration:**
    *   *Problem:* Legacy mainframe communication requires specific adapters.
    *   *Solution:* Use **IBM MQ Kubernetes Operator** or a sidecar adapter pattern to bridge MQ to the cloud.
5.  **F5 Load Balancer:**
    *   *Problem:* F5 handles sticky sessions and SSL termination.
    *   *Solution:* Replace with **Kubernetes Ingress Controller** (e.g., NGINX or ALB Ingress) and use **Istio** for service mesh features.

SC5-C
**Architectural Solution:**
1.  **CQRS:** Separate the read model (inventory view) from the write model. The reconciliation job updates the write model; the checkout service reads from a read replica or cache.
2.  **Read Replica:** Run the reconciliation job against a **PostgreSQL Read Replica** to avoid locking production tables.
3.  **Bulkhead Isolation:** Configure the database connection pool in `checkout-service` to have a dedicated pool size separate from the reconciliation job's pool.
4.  **CronJob Scheduling:** Move the reconciliation job to off-peak hours (e.g., 03:00 UTC).
5.  **PDB & Quotas:**
    *   **PDB:** Set `minAvailable: 1` for the reconciliation CronJob pod to ensure it doesn't kill serving pods during disruption.
    *   **Resource Quotas:** Limit the CronJob's CPU/Memory to prevent it from starving the main pods.

SC6-C
**Connectivity Comparison:**
1.  **AWS Direct Connect:**
    *   *Pros:* High bandwidth (10Gbps+), low latency, dedicated connection.
    *   *Cons:* High cost, hardware provisioning.
    *   *Zero-Trust:* Fits well; traffic is encrypted (MACsec) and isolated.
2.  **Second VPN Tunnel (ECMP):**
    *   *Pros:* Low cost, quick setup.
    *   *Cons:* Bandwidth limited by tunnel (1Gbps), encryption overhead.
    *   *Zero-Trust:* IPsec encrypts traffic, but IP-based trust remains.
3.  **Hybrid Mesh (Tailscale/Netbird):**
    *   *Pros:* Zero-trust by design (WireGuard), no IP management.
    *   *Cons:* Performance overhead, not suitable for 10Gbps backbone.
    *   *Zero-Trust:* Best fit for identity-based access, but bandwidth is the bottleneck.
**Recommendation:** Use **Direct Connect** for the backbone bandwidth and **SPIRE/WireGuard** overlay for zero-trust identity verification on top of the encrypted link.
