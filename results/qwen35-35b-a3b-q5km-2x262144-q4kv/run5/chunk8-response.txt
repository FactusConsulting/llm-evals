CL1 — **IaaS (Infrastructure as a Service):** Provides virtualized computing resources (servers, storage, networking). *Example:* AWS EC2. **PaaS (Platform as a Service):** Provides hardware and software tools (runtime, middleware) for development. *Example:* Google App Engine. **SaaS (Software as a Service):** Delivers software applications over the internet. *Example:* Microsoft Office 365.

CL2 — **Shared Responsibility Model:** Provider secures the cloud (physical, network, hypervisor); Customer secures what is in the cloud (data, OS, apps). **IaaS VM:** Provider secures hardware/hypervisor; Customer secures OS, patching, firewall, data. **Managed Kubernetes:** Provider secures control plane; Customer secures node OS, pod security, network policies, data. **SaaS (M365):** Provider secures app/infrastructure; Customer secures identity, access, data classification, device compliance.

CL3 — **VPC:** Isolated virtual network. **Subnet:** Range of IP addresses within a VPC. **Security Group:** Stateful firewall at the instance level (allow rules only). **NACL:** Stateless firewall at the subnet level (allow/deny rules). **Layering:** VPC contains Subnets; Instances sit in Subnets protected by SGs; Subnets protected by NACLs. **Stateful:** SG tracks connection state (return traffic allowed automatically). **Stateless:** NACL requires explicit rules for both inbound and outbound traffic.

CL4 — **Landing Zone:** Secure, compliant baseline environment. **OUs:** Organizational structure for policy application. **Guardrails:** Preventative controls (e.g., no public S3 buckets). **Shared Networking:** Central hub (Transit Gateway/Hub) connects spokes (workload accounts) for centralized egress/ingress. **Centralized Logging:** Aggregates logs for security monitoring. **Single-Account Problem:** High blast radius (one breach compromises all), poor cost visibility, compliance violations, lack of isolation.

CL5 — **IaC:** Managing infrastructure via machine-readable definition files. **Terraform/OpenTofu:** Declarative, multi-cloud, state-managed. **Pulumi:** Imperative/Declarative, uses general-purpose languages (TS/Python). **CloudFormation/Bicep:** Native to AWS/Azure, tightly coupled. **Crossplane:** Kubernetes-native IaC. **Choice:** Multi-cloud/General -> Terraform/Pulumi. AWS/Azure specific -> CFN/Bicep. K8s-centric -> Crossplane.

CL6 — **HPA:** Scales pods based on CPU/Memory. **VPA:** Scales pod resource requests/limits. **Cluster Autoscaler:** Adds/removes nodes based on pod scheduling. **KEDA:** Scales based on external events (e.g., queue length). **KEDA over HPA:** Use KEDA when scaling is driven by external triggers (Kafka, SQS, HTTP) rather than internal resource metrics.

CL7 — **DR Strategy:** **RPO/RTO:** Define data loss tolerance and downtime. **Active-Active:** Both regions serve traffic (low RTO, high cost). **Active-Passive:** One region active, one standby (higher RTO, lower cost). **DB:** Sync (strong consistency, higher latency) vs Async (eventual consistency, lower latency). **DNS:** Route53 failover routing policies. **State:** Externalize state (S3/RDS) to avoid region lock-in. **DR Types:** Pilot Light (min infra, DB replicated), Warm Standby (scaled down app), Multi-site (full active-active).

CL8 — **GitOps:** Git repository is the single source of truth for infrastructure/app state. **ArgoCD:** Push-based (agent in cluster watches Git). **Flux:** Pull-based (agent pulls Git). **Secrets:** Sealed Secrets (encrypted in Git), External Secrets Operator (fetches from Vault/AWS Secrets Manager). **Drift:** GitOps controller detects manual changes and automatically reverts them to match Git state.

CL9 — **Architecture:** Use **StatefulSet** for stable network IDs and storage binding. **Local PV/PVC:** Use `local` storage class for low latency (data on node). **Headless Service:** Enable DNS resolution for StatefulSet pods. **Init Containers:** Handle config file generation before main app starts. **ConfigMaps:** Mount as volumes for specific paths. **PDB:** Ensure minimum available pods during disruptions.

CL10 — **SA:** K8s internal identity for pods. **Workload Identity:** GCP/Azure native identity federation. **IRSA:** AWS IAM roles mapped to K8s Service Accounts via OIDC. **Mounting Creds:** Bad practice because credentials are static, hard to rotate, and visible in pod spec/logs. **Federation:** Uses short-lived OIDC tokens from K8s to request cloud credentials, eliminating long-term secrets.

CL11 — **Object Storage (S3):** Unstructured data, high durability (11 9s), low latency. **Block Storage (EBS):** Attached to VM, low latency, used for OS/DB. **File Storage (EFS):** Shared file system, NFS/SMB, used for shared configs/logs. **Durability:** Object (11 9s), Block (99.999%), File (99.999%). **Availability:** Object (99.99%), Block/File (99.99%+).

CL12 — **Zero Trust:** Assume breach, verify explicitly. **Network Policies:** Restrict pod-to-pod traffic. **Service Mesh (mTLS):** Encrypts service-to-service traffic. **Pod Identity:** Restricts cloud permissions per pod. **Admission Controllers:** Enforce policies (OPA/Kyverno) on deploy. **Image Signing:** Verify integrity before run. **Runtime Security:** Detect anomalies (Falco). **Difference:** Traditional relies on perimeter; Zero Trust assumes internal threats and enforces micro-segmentation.

CL13 — **CDN:** Distributed network of servers caching content at edge locations. **Mechanism:** Caches static assets closer to users to reduce latency and origin load. **Services:** AWS CloudFront, Azure CDN. **Use CDN:** For static content (images, CSS, JS) to improve performance. **Use Origin:** For dynamic, personalized, or real-time data.

CL14 — **Region:** Geographic area with multiple AZs. **AZ:** Isolated data center within a region. **Multi-AZ:** Deploy across AZs for High Availability (HA) against data center failure. **Zonal:** Resources tied to one AZ (e.g., EC2). **Regional:** Resources span AZs (e.g., ELB, RDS Multi-AZ).

CL15 — **Managed DB:** Service where provider manages hardware, OS, DB software. **Provider Tasks:** Patching, backups, failover, scaling, monitoring. **Customer Tasks:** Schema design, queries, user management, data security.

CL16 — **IAM Users:** Individual identities. **Groups:** Collections of users. **Roles:** Temporary credentials for entities (users, services). **Policies:** JSON permissions defining actions/resources. **Least Privilege:** Grant only necessary permissions. **Application:** Use roles for services, groups for users, deny-by-default policies, and audit regularly.

CL17 — **Serverless:** Event-driven code execution without managing servers. **Comparison:** Lambda (AWS), Functions (Azure), Cloud Functions (GCP). **Cold Starts:** Latency when code initializes on first/infrequent invocation. **Poor Fit:** Long-running processes, stateful apps, high CPU requirements, strict latency needs.

CL18 — **Ingress:** K8s resource for L7 routing. **Gateway API:** Standardized API for L7/L4 traffic management. **LB Annotations:** Cloud-specific LB configuration. **Mesh Ingress:** Service mesh entry point. **TLS:** Terminates at LB (cloud), Ingress Controller, or Mesh. **Gateway API:** Use for complex routing, multi-cluster, or when standard Ingress is insufficient.

CL19 — **Strategy:** **RI/SP:** Commit to usage for discount. **Spot:** Use for fault-tolerant workloads (up to 90% savings). **Right-sizing:** Match instance size to actual usage. **Tiering:** Move old data to Glacier/Archive. **Idle Detection:** Tag and terminate unused resources. **FinOps:** Cross-team cost accountability. **Tools:** AWS Cost Explorer, CloudHealth. **Showback:** Report costs to teams. **Chargeback:** Bill teams directly.

CL20 — **Strategy:** **Abstraction:** Kubernetes (K8s) for runtime, Terraform for infra. **Networking:** Direct Connect/ExpressRoute for low latency. **Identity:** OIDC federation across clouds. **Trade-offs:** Multi-cloud increases complexity, cost, and skill requirements. **Justified:** Regulatory compliance, avoiding single-vendor risk, geographic requirements. **Unnecessary:** If one cloud meets all needs, multi-cloud adds complexity without value.

OT1 — **Purdue Model:** ISA-95 standard for IT/OT integration. **Level 0:** Physical Process (sensors/actuators). **Level 1:** Basic Control (PLCs, RTUs). **Level 2:** Supervisory Control (SCADA, HMIs). **Level 3:** Operations (MES, Historian). **Level 4:** Enterprise (ERP, Business).

OT2 — **IT:** Information Technology (data, security, business logic). **OT:** Operational Technology (physical processes, control systems). **Patching:** IT patches frequently; OT patches rarely due to stability requirements, legacy hardware, and risk of downtime affecting physical safety.

OT3 — **DMZ:** Buffer zone between IT and OT. **IDMZ:** Industrial DMZ. **Traffic:** Historian pulls from SCADA (push to IT allowed). **Direction:** IT to OT should be restricted (pull-based). **Danger:** Direct L4 to L1 allows attackers to manipulate control logic directly, risking physical safety.

OT4 — **Modbus TCP:** L2/L3, Master/Slave, simple. **OPC UA:** L3, secure, platform-independent. **EtherNet/IP:** L2/L3, CIP protocol, Allen-Bradley dominant. **PROFINET:** L2, real-time, Siemens dominant. **MQTT:** App layer, lightweight, publish/subscribe. **Location:** Modbus/OPC UA/Modbus TCP (L2-L3), EtherNet/IP/PROFINET (L2-L3), MQTT (L4/L5).

OT5 — **Data Diode:** Hardware allowing data flow in one direction only. **Firewall:** Bidirectional traffic control. **Use Diode:** When data must leave OT but nothing can enter (e.g., telemetry to cloud). **Limitation:** Cannot receive commands or updates from the outside.

OT6 — **Flow:** PLC -> Historian -> Edge Gateway -> Cloud. **Security:** Edge Gateway acts as DMZ. TLS encryption for data in transit. Firewall rules restrict traffic to specific ports/protocols. Cloud ingestion uses secure APIs (not direct DB access).

OT7 — **Zones:** Logical grouping of assets with similar security requirements. **Conduits:** Controlled paths between zones. **Model:** Zone 4 (Corporate), Zone 3 (Engineering/SCADA), Zone 2 (Control), Zone 1 (Field). Conduits between zones require firewalls. Engineering workstations in Zone 3 access Zone 2 via secure conduit. SIS in Zone 1 isolated.

OT8 — **Segmentation:** VLANs per production line. **Firewalls:** Between VLANs and DMZ. **Switches:** Managed Layer 3 switches. **Remote Access:** Jump host in DMZ with MFA. **Transition:** Parallel network setup, test offline, migrate during maintenance windows, rollback plan.

OT9 — **SIS:** Safety Instrumented System (independent safety shutdown). **BPCS:** Basic Process Control System (normal operation). **IEC 61511:** Standard for safety lifecycle. **Isolation:** SIS must be independent to ensure safety function works even if BPCS/IT is compromised. **Ransomware:** If SIS is compromised, safety shutdowns may fail, leading to physical damage or injury.

OT10 — **PLC:** Programmable Logic Controller (logic execution). **DCS:** Distributed Control System (process control, integrated). **SCADA:** Supervisory Control and Data Acquisition (monitoring). **Coexist:** Yes, PLCs often feed SCADA; DCS may contain PLC-like logic.

OT11 — **Architecture:** Jump host in DMZ. **MFA:** Required for all access. **Recording:** Session recording for audit. **Vendor:** Time-limited, scoped access. **No Direct RDP:** Prevents lateral movement from corporate network to OT devices.

OT12 — **Detection:** Network IDS, anomaly detection on traffic patterns. **Inventory:** Verify compromised assets. **Response:** Isolate network segment, notify control room. **Coordination:** Operator maintains manual control if needed. **Operator Role:** Monitor physical safety, execute manual procedures, verify system status.

OT13 — **HMI:** Human-Machine Interface (operator display/control). **Role:** Visualize process, allow manual overrides. **Engineering Workstation:** Used for programming/configuring PLCs/HMIs, not for daily operation.

OT14 — **Historian:** Time-series database for process data. **Importance:** Trend analysis, compliance, troubleshooting. **Plant (L3):** Real-time ops data. **Enterprise (L4):** Aggregated data for business analytics.

OT15 — **Ladder Logic:** Graphical programming language resembling relay logic. **Device:** PLC. **Dominance:** Familiarity, deterministic execution, safety certification, ease of maintenance for electricians.

OT16 — **RTU:** Remote Terminal Unit (rugged, long-distance comms). **PLC:** Local control. **RTU Use:** Pipelines, SCADA, harsh environments, long-distance telemetry.

OT17 — **Analog:** Continuous signal (voltage/current). **Digital:** Discrete signal (on/off). **4-20mA:** Preferred over 0-20mA because 4mA represents "live zero" (detects wire break at 0mA).

OT18 — **Inventory:** Knowing all assets is critical for vulnerability management. **Tools:** Passive (packet sniffing), Active (scanning). **Active Risk:** Scanning can crash legacy devices or disrupt control loops.

OT19 — **MITRE ICS:** Framework for ICS threats. **Tactics:** Reconnaissance (Scanning), Initial Access (Phishing), Execution (Command), Persistence (Backdoor), Impact (Disruption). **Example:** SMB Exploit (Initial Access), File Deletion (Impact). **Difference:** Focuses on physical safety and availability, not just confidentiality. **Usage:** Map defenses to tactics to find gaps.

OT20 — **SOC:** SIEM collects logs (OT agents). **Rules:** OT-specific (e.g., PLC write attempts). **Triage:** OT engineer validates before IT action. **Staffing:** Mix of IT analysts and OT engineers. **False Positives:** Tuned to avoid alert fatigue. **Firmware Alert:** Isolate PLC, verify change log, rollback if unauthorized, investigate source.
