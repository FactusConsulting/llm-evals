T1 — `tofu plan` calculates the execution plan, showing what changes will be made without applying them. `tofu apply` executes the plan, creating, updating, or destroying resources to match the configuration.

T2 — State is a JSON file mapping real-world resources to your configuration. Remote state is important for team collaboration (sharing state), enabling state locking (preventing concurrent writes), and storing state securely outside the local machine.

T3 — `count` uses integer indices; `for_each` uses maps or sets. `for_each` is preferred when resources are identified by unique keys (e.g., names) because removing an item doesn't shift indices for remaining resources, preventing unintended recreation.

T4 — OpenTofu detects this via state drift during `tofu plan`. If the resource exists but config differs, it shows a change. If config matches desired state, `apply` will revert the manual change. If config expects different values, it may cause conflicts or require manual intervention.

T5 —
```hcl
resource "null_resource" "vm" {
  for_each = toset(["web1", "web2", "web3"])
  triggers = {
    name = each.key
  }
}
```

T6 — `terraform_remote_state` reads the raw state file (coupling). `output` values expose specific data via module interfaces (decoupling). Recommended alternative: Use module `output` values. It enforces an API contract and hides internal state structure.

T7 — The `null_resource` provisioner runs asynchronously; the output is read before the file is fully written. Fix: Use a `local_file` resource to store the kubeconfig. Have the `null_resource` write to that file, and have dependent resources depend on the `local_file` resource (which depends on the `null_resource`).

T8 — `terraform import` adds existing resources to state without config. `import block` (1.5+) allows importing during `apply` with config, making it version-controlled and repeatable in CI/CD pipelines.

T9 — A `variable` is an input parameter for the module. An `output` is data exposed from the module to the caller. Variables configure infrastructure; outputs share results.

T10 — A provider is a plugin that interacts with a specific API (e.g., AWS). Without `tofu init`, providers aren't downloaded, causing `tofu plan` to fail with "Provider not installed" errors.

T11 — `.terraform.lock.hcl` records exact provider versions and checksums. It should be committed to ensure reproducible builds and prevent unexpected provider upgrades.

T12 — `tofu destroy` removes all managed resources. Target a single resource with `tofu destroy -target=resource.address`.

T13 — A data source reads existing data from an external system. Use one instead of a managed resource when you need to reference existing infrastructure (e.g., looking up an AMI ID) rather than creating new infrastructure.

T14 — `variable` blocks define inputs from outside. `locals` define internal values derived from variables or other locals. Use `locals` for calculations or transformations within the module that shouldn't be exposed as inputs.

T15 — `lifecycle` manages resource behavior. `create_before_destroy` creates new resource before deleting old (zero downtime). `prevent_destroy` prevents accidental deletion. `ignore_changes` ignores drift in specific attributes.

T16 — A `moved` block allows changing a resource's address (e.g., module path) without destroying and recreating it. It updates the state mapping so OpenTofu recognizes the resource as the same object.

T17 — The dependency graph determines execution order. Implicit dependencies come from resource references. Explicit `depends_on` forces order. `depends_on` reduces parallelism because it blocks unrelated resources from running concurrently.

T18 — 1. `refresh=false` (faster, risks stale state). 2. Modularize (smaller graphs). 3. `exclude` resources (faster, ignores specific changes). 4. Remove unused state resources (cleaner graph). Tradeoff: Speed vs. accuracy of state.

T19 — State locking prevents concurrent writes. If a lock isn't released, subsequent runs fail. Break it with `tofu force-unlock <LOCK_ID>`. Risk: Data corruption if two processes are actually writing simultaneously.

T20 — 1. Add `backend` block to config. 2. Run `tofu init -backend-config="bucket=..."`. 3. Run `tofu state mv` if addresses changed. 4. Upload existing state to S3. 5. Run `tofu plan` to verify state matches remote.

A1 — A **playbook** is a YAML file defining automation steps. A **role** is a structured directory for reusable tasks/files. A **task** is a single action executed by Ansible.

A2 — `ansible_facts` is a dictionary of system information gathered by Ansible. Access OS family via `{{ ansible_facts['os_family'] }}`.

A3 — `command` runs without shell features. `shell` runs with shell features (pipes, vars). `raw` runs without setup (no fact gathering, no module setup), useful for bootstrapping SSH.

A4 — Idempotency means running a task multiple times yields the same result. `apt: name=htop` is idempotent. `shell: rm -rf /tmp/test` is not (fails if file missing, or deletes every time).

A5 —
```yaml
- name: Install htop
  package:
    name: htop
  become: true
  when: ansible_os_family == "Debian"
- name: Install htop on RedHat
  package:
    name: htop
  become: true
  when: ansible_os_family == "RedHat"
```

A6 — Priority (High to Low): Extra vars > Play vars > Host vars > Group vars > Role defaults > Role vars > Inventory vars. `set_fact` overrides most at runtime.

A7 — Use `serial: 3` in the play to limit concurrency. Group nodes into `rke2_agent` and `rke2_control`. Use `meta: flush_handlers` or `wait_for` to ensure the first 3 nodes are ready before the next batch starts.

A8 — `ansible-vault` encrypts sensitive data. Encrypt a single variable using `ansible-vault edit vars.yml` or `ansible-vault encrypt_string 'secret' --name 'my_secret'`.

A9 — An inventory file lists hosts and groups. **Static** is a file (INI/YAML). **Dynamic** is a script/API that generates inventory at runtime (e.g., AWS EC2).

A10 — `ansible webservers -m shell -a "df -h"`.

A11 — A **handler** is a task that runs only when notified by another task. Regular tasks run every time. Trigger a handler with `notify: handler_name`.

A12 — `ansible.cfg` configures Ansible behavior. Common settings: `host_key_checking`, `forks` (parallelism), `remote_user`.

A13 — `register` stores a task's output in a variable. Use it in subsequent tasks via `when: my_var.stdout == 'success'` or `debug: var=my_var`.

A14 — `include_tasks` is dynamic (evaluated at runtime, supports `when`). `import_tasks` is static (evaluated at parse time, faster, no `when`). Use `include` for conditional logic.

A15 — Ansible Galaxy is a repository for roles/collections. Use `requirements.yml` to list them, then run `ansible-galaxy install -r requirements.yml`.

A16 — Connection plugins manage how Ansible connects. `ssh` (remote Linux), `local` (control node), `network_cli` (network devices). Use `ssh` for servers, `network_cli` for routers/switches.

A17 — `delegate_to: localhost` runs a task on the control node. `run_once` ensures it runs once even in a loop. Example: Gather facts from remote, then process them on localhost to generate a config file.

A18 — 1. Increase `forks` (more parallelism, higher memory). 2. Enable `pipelining` (reduces SSH overhead). 3. Use `fact_caching` (avoids re-gathering facts). 4. Use `mitogen` (faster execution). Tradeoffs: Resource usage vs. speed.

A19 — `{{ }}` outputs variables. `{% %}` controls flow (loops/if). `{# #}` comments.
```jinja
upstream backend_servers {
{% for server in backends %}
    server {{ server.ip }} weight={{ server.weight }}; # Health check: {{ server.health }}
{% endfor %}
}
```

A20 — 1. Set `serial: 1` for one-by-one. 2. `pre_tasks`: Deregister from LB. 3. `block`: Deploy app. 4. `rescue`: Revert config, re-register. 5. Use `uri` module to check health before proceeding to next batch.
