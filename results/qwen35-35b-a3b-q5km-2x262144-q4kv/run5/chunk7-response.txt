AA1 — **Monolith vs. Microservices:** A monolith is a single deployable unit where all components are tightly coupled. Microservices are independently deployable, loosely coupled services communicating over a network.
**3 Microservices Problems:**
1.  **Distributed Complexity:** Debugging requires tracing across multiple services.
2.  **Data Consistency:** Maintaining ACID transactions across services is difficult (requires Sagas).
3.  **Operational Overhead:** Requires orchestration (Kubernetes), service discovery, and complex networking.

AA2 — **Sync vs. Async:** Synchronous communication blocks the sender until a response is received (request/response). Asynchronous communication allows the sender to continue without waiting (fire-and-forget).
**Examples:** Synchronous: HTTP/REST, gRPC. Asynchronous: RabbitMQ, Apache Kafka.

AA3 — **API Gateway vs. Reverse Proxy:** A reverse proxy routes traffic to backend servers (L4/L7). An API Gateway adds business logic (auth, rate limiting, transformation).
**Replacement:** An API Gateway *can* replace a reverse proxy for HTTP traffic, but a dedicated L4 load balancer (e.g., HAProxy, F5) is often still needed upstream for SSL termination, TCP routing, or high-throughput DDoS protection before the Gateway.

AA4 — **Saga Pattern:** A sequence of local transactions where each updates data and triggers the next. If one fails, compensating transactions undo previous steps.
**Choreography:** Services emit events; others listen. *Pros:* Loose coupling. *Cons:* Hard to debug, complex flow.
**Orchestration:** A central coordinator tells services what to do. *Pros:* Centralized control, easier debugging. *Cons:* Single point of failure, tighter coupling to orchestrator.
**Usage:** Choreography for simple flows; Orchestration for complex business logic.

AA5 — **CQRS:** Separates read (Query) and write (Command) models into different schemas.
**Pairing with Event Sourcing:** Event Sourcing stores state changes as an immutable log. CQRS reads from projections of this log.
**Problem Solved:** Allows independent scaling of read/write paths, enables temporal queries (history), and decouples complex write logic from simple read logic.

AA6 — **Circuit Breaker:** Prevents cascading failures by stopping calls to a failing service.
**States:** Closed (normal), Open (fail fast), Half-Open (test recovery).
**Vs. Retry:** Retry attempts to fix transient errors; Circuit Breaker stops traffic to prevent overload.
**Usage:** Use Retry for transient network blips; use Circuit Breaker when a service is consistently down to save resources.

AA7 — **Tenant Isolation Models:**
1.  **Shared DB/Schema:** Lowest cost, highest noisy-neighbor risk, complex security (row-level).
2.  **Shared DB/Schema per Tenant:** Moderate cost, better security, moderate ops complexity.
3.  **Separate DB per Tenant:** Highest cost, best isolation/security, highest ops complexity (backup/migration).
**Trade-off:** Choose based on compliance needs and budget. High-security tenants need separate DBs; low-risk SaaS can use shared schema.

AA8 — **Consistency:** Strong consistency ensures immediate visibility of updates. Eventual consistency allows temporary divergence.
**Architecture:** Use **Strong Consistency**.
**Data Store:** Relational DB (PostgreSQL) with ACID transactions.
**Patterns:** Row-level locking or distributed locking (Redis/Zookeeper) for inventory counts. Avoid eventual consistency for inventory to prevent overselling.

AA9 — **Sidecar Proxy:** A container running alongside the app to handle networking.
**Envoy/Istio:** Envoy is the data plane (handles traffic). Istio Control Plane configures Envoy.
**Data vs. Control Plane:** Data plane processes traffic; Control plane manages configuration.
**mTLS:** Encrypts service-to-service traffic and authenticates identity. App-level TLS requires app code changes; mTLS is transparent to the app.

AA10 — **Strangler Fig:** Incrementally migrate functionality from legacy to new system.
**Strategy:** Deploy .NET 8 services alongside the .NET Framework monolith. Configure an API Gateway (e.g., NGINX) to route specific URLs (e.g., `/api/v2`) to the new services while routing the rest to the legacy app. Gradually move endpoints until the legacy app is empty and can be decommissioned.

AA11 — **12-Factor App:**
1.  **Codebase:** One repo, many deploys.
2.  **Dependencies:** Explicitly declared.
3.  **Config:** Stored in environment.
4.  **Backing Services:** Treated as attached resources.
5.  **Build/Release/Run:** Strictly separated.
6.  **Processes:** Stateless, state stored in backing services.
7.  **Port Binding:** Expose via port.
8.  **Concurrency:** Scale via process model.
9.  **Disposability:** Fast start/shutdown.
10. **Dev/Prod Parity:** Minimize differences.
11. **Logs:** Stream to stdout.
12. **Admin Processes:** Run as one-off commands.
**Why:** Ensures portability, scalability, and maintainability in cloud environments.

AA12 — **Event-Driven (Kafka) vs. Request-Driven (REST):**
*   **Coupling:** Event = Loose; Request = Tight.
*   **Scalability:** Event = High (partitioning); Request = Moderate.
*   **Debugging:** Event = Hard (distributed state); Request = Easier (traceable).
*   **Ordering:** Event = Possible (per partition); Request = Guaranteed.
*   **Failure:** Event = Dead Letter Queue; Request = Immediate error.
**Choice:** Use Event-Driven for decoupling, async processing, and audit trails. Use Request-Driven for real-time user interactions requiring immediate feedback.

AA13 — **Load Balancer:** Distributes traffic across servers.
**L4 vs. L7:** L4 routes based on IP/Port (TCP/UDP). L7 routes based on HTTP headers/URL.
**L7 Use Case:** SSL termination or routing `/api` to backend A and `/static` to backend B.

AA14 — **Message Queue:** Stores messages for delivery.
**MQ (RabbitMQ) vs. Log (Kafka):** MQ is for task queues (consume once). Kafka is for event logs (retain, replay, multiple consumers).
**Pick:** RabbitMQ for task execution. Kafka for event streaming, analytics, or audit logs.

AA15 — **Container vs. VM:** Container shares the host OS kernel; VM virtualizes hardware.
**Popularity:** Containers are lightweight, start faster, and have consistent environments, ideal for microservices.

AA16 — **Stateless vs. Stateful:** Stateless services store no session data (e.g., Web Server). Stateful services store data locally (e.g., Database).
**Scaling:** Stateless is easier to scale horizontally because any instance can handle any request without data synchronization.

AA17 — **ORM:** Maps objects to database tables.
**Examples:** Hibernate (Java), Entity Framework (.NET).
**Trade-offs:** ORM increases productivity but can hide performance issues (N+1 queries). Raw SQL offers performance and control but is verbose and error-prone.

AA18 — **Bulkhead:** Isolates resources so one failure doesn't exhaust all capacity.
**Implementation:** Use separate thread pools for different services. In Kubernetes, use separate namespaces or resource quotas (CPU/Memory limits) per service.

AA19 — **OT vs. CRDT:** OT transforms operations sequentially (requires server). CRDTs merge states mathematically (local resolution).
**Choice:** **CRDTs**.
**Why:** CRDTs support offline editing because conflicts resolve locally without a central server, whereas OT requires a server to serialize operations.

AA20 — **Distributed Tracing Challenges:**
*   **Context Propagation:** Pass Trace ID via headers (W3C Trace Context, B3) across languages.
*   **Sampling:** Head-based (start) vs. Tail-based (end). Tail-based is better for cost but requires buffering.
*   **Storage:** Jaeger/Tempo store spans; query via Elasticsearch/ClickHouse.
*   **Kafka:** Inject trace context into message headers (producer) and extract (consumer) to maintain continuity.

OP1 — **Hypervisor Type 1:** Bare-metal (runs directly on hardware). Examples: VMware ESXi, Microsoft Hyper-V Server.
**Hypervisor Type 2:** Hosted (runs on OS). Examples: Oracle VirtualBox, VMware Workstation.

OP2 — **SAN vs. NAS:** SAN provides block storage (iSCSI, Fibre Channel). NAS provides file storage (NFS, SMB).
**Usage:** SAN for databases (low latency, block access). NAS for file shares (easy access, file locking).

OP3 — **3-Tier:** Core (backbone), Distribution (policy), Access (edge).
**Leaf-Spine:** Preferred for modern DCs. Solves oversubscription and single points of failure in the core. Provides non-blocking, low-latency connectivity between any two nodes.

OP4 — **Proxmox HA:**
*   **Nodes:** 3+ (Quorum).
*   **Quorum:** Majority vote required to make decisions.
*   **Fencing:** STONITH (Shoot The Other Node In Head) to power off failed nodes.
*   **Storage:** Ceph (distributed) preferred over iSCSI/NFS (shared storage risk).
*   **Failure:** VMs restart on surviving nodes automatically.

OP5 — **VLAN Trunking:** Carries multiple VLANs over one link.
**802.1Q:** Adds a 4-byte tag to Ethernet frames identifying the VLAN.
**Linux Config:** Create sub-interfaces (e.g., `eth0.10`, `eth0.20`) using `ip link` or `vconfig` and assign IPs to each.

OP6 — **RAID:**
*   **RAID 1:** Mirror (50% capacity).
*   **RAID 5:** Striping + 1 parity (N-1 capacity).
*   **RAID 6:** Striping + 2 parity (N-2 capacity).
*   **RAID 10:** Mirror + Stripe (50% capacity, high perf).
*   **RAID Z2:** ZFS double parity (N-2 capacity).
**Choice:** **RAID 10**.
**Why:** For a DB server, performance is critical. RAID 10 offers the best read/write performance with high fault tolerance (can lose 1 drive per mirror pair).

OP7 — **DNS Architecture:**
*   **Internal:** AD DNS servers (Forwarders to external).
*   **DMZ:** Split-horizon (Public IP for public, Private IP for internal).
*   **DNSSEC:** Sign external zone to prevent spoofing.
*   **Laptop:** Connects to internal DNS via DHCP. Internal DNS resolves internal names; forwards external to ISP/Cloud.
*   **VPN:** Tunnel traffic to internal DNS servers.

OP8 — **PKI Chain:** Root CA (Offline) -> Subordinate CA (Online) -> End Entity.
**Internal PKI:** Root offline for security. Subordinate issues certs. Use CRL/OCSP for revocation.
**Compromise:** Subordinate = Revoke sub, reissue certs. Root = Catastrophic (trust chain broken, must rebuild PKI).

OP9 — **WAN Consolidation:**
*   **MPLS:** Expensive, reliable, low latency.
*   **IPsec:** Cheap, variable latency.
*   **SD-WAN:** Smart routing, QoS, cost-effective.
*   **WireGuard:** Secure, simple, mesh.
**Recommendation:** **SD-WAN**.
**Why:** Best for VoIP (QoS prioritization), handles internet path selection dynamically, and reduces MPLS costs.

OP10 — **iDRAC/iLO/IPMI:** Out-of-band management interfaces.
**Critical:** Allows server management (power, console) even if OS is down.
**Exposed to Internet:** High risk. Attackers can reset passwords, install firmware, or access console (hardware compromise).

OP11 — **LVM vs. ZFS:**
*   **LVM:** Logical volumes on physical disks. Snapshots are copy-on-write (space intensive).
*   **LVM-Thin:** Overcommit storage.
*   **ZFS:** Filesystem + Volume manager. Snapshots are instant, space-efficient (copy-on-write).
**Usage:** LVM for simple Linux volumes. ZFS for data integrity and snapshots.

OP12 — **Stretch Cluster:**
*   **Design:** 2 sites + 1 tiebreaker (quorum disk).
*   **Ceph:** Monitors in both sites. CRUSH rules distribute data.
*   **Latency:** <5ms between sites.
*   **Split-Brain:** If network fails, both sites might think they are primary.
*   **Why 2-site Ceph is bad:** Without a tiebreaker, split-brain causes data corruption. Tiebreaker ensures only one site remains writable.

OP13 — **DHCP:** Assigns IP addresses.
**DORA:** Discover, Offer, Request, Acknowledge.
**Unavailable + Expired:** Client uses APIPA (169.254.x.x) and cannot reach network resources.

OP14 — **Managed vs. Unmanaged:**
*   **Unmanaged:** Plug-and-play.
*   **Managed:** Configurable (VLANs, QoS, ACLs, Port Mirroring).
**Importance:** Managed switches enable network segmentation, security policies, and traffic prioritization.

OP15 — **NTP:** Network Time Protocol.
**Critical:** Logs, authentication (Kerberos), clustering, and certificate validation rely on time.
**Drift:** Causes auth failures, log confusion, and cluster split-brain.

OP16 — **Physical vs. VM vs. Container:**
*   **Physical:** Hardware only.
*   **VM:** Virtualized OS.
*   **Container:** Virtualized user space.
**Bare Metal:** Use for high-performance computing, low-latency requirements, or direct hardware access (GPUs, NICs).

OP17 — **UPS vs. Generator:**
*   **UPS:** Battery backup (seconds/minutes).
*   **Generator:** Fuel backup (hours/days).
*   **ATS:** Switches power from grid to generator.
**Need Both:** UPS bridges the gap until generator starts.

OP18 — **Active-Passive vs. Active-Active:**
*   **Active-Passive:** One node handles traffic; standby waits.
*   **Active-Active:** Both handle traffic.
**PostgreSQL HA:** Pacemaker/Corosync cluster. Use STONITH for fencing. Virtual IP (VIP) floats to active node. Failover moves VIP and starts DB on new node.

OP19 — **Backup/Recovery:**
*   **3-2-1:** 3 copies, 2 media, 1 offsite.
*   **RPO/RTO:** Recovery Point/Time Objective.
*   **Types:** Full, Incremental, Differential.
*   **Protection:** Immutable backups (WORM) prevent ransomware deletion.
*   **Test:** Regular restore drills.
*   **Backup vs. DR:** Backup is data copy; DR is the capability to restore operations.

OP20 — **802.1X:** Port-based network access control.
*   **Roles:** Supplicant (Client), Authenticator (Switch), Server (RADIUS).
*   **Methods:** EAP-TLS (Certificates), PEAP (Username/Password).
*   **Dynamic VLAN:** Assign VLAN based on user identity.
*   **Fail:** Port goes to quarantine VLAN.
*   **MAB:** Authenticates device by MAC address if 802.1X fails (for printers/IoT).
