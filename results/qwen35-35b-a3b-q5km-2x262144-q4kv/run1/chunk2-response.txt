K1 — Easy: A **Deployment** manages stateless applications with interchangeable pods (random names, no guaranteed order). A **StatefulSet** manages stateful applications requiring stable network IDs, ordered deployment/termination, and persistent storage per pod.

K2 — Easy: A **ClusterIP** Service creates a virtual IP. **kube-proxy** (using iptables or IPVS) on each node sets up network rules to forward traffic destined for the Service IP:Port to the IP:Port of the healthy backend pods.

K3 — Medium: A **PersistentVolume (PV)** is a cluster storage resource. A **PersistentVolumeClaim (PVC)** is a user request for storage. A **StorageClass** defines the provisioning class (how PVs are created dynamically). A PVC binds to a PV (static) or triggers a StorageClass to provision a PV (dynamic).

K4 — Medium: 1. `kubectl logs` to check application error messages. 2. `kubectl describe pod` to check events (e.g., OOMKilled, CrashLoopBackOff). 3. Check resource limits (CPU/Memory) to see if the container is being killed. 4. Verify configuration (Env vars, ConfigMaps) and external dependencies (database connectivity).

K5 — Medium: **CoreDNS** provides internal DNS resolution for services (e.g., `service.namespace.svc`). If CoreDNS pods are down, internal service discovery fails (pods cannot resolve each other by name), though external DNS resolution may still work depending on the cluster configuration.

K6 — Hard: The scheduler uses **Filtering** (remove nodes that don't meet requirements like taints, resources, nodeSelector) and **Scoring** (rank remaining nodes). Factors considered: 1. CPU/Memory requests vs capacity. 2. Node Affinity rules. 3. Pod Affinity/Anti-Affinity rules.

K7 — Hard: **Yes**, it will be scheduled. Node has 4GB, 3.5GB requested = 0.5GB free. Pod requests 256Mi (~0.25GB). 0.5GB > 0.25GB. If it uses 600Mi at runtime, it exceeds the 512Mi limit and will be **OOMKilled** (evicted).

K8 — Hard: 1. Client request hits **External Load Balancer**. 2. LB forwards to **Ingress Controller** (NGINX). 3. NGINX matches Ingress rules to a **Service**. 4. Service forwards to **Endpoints** (Pod IPs). 5. **kube-proxy** (iptables/IPVS) on the node routes traffic to the specific **Pod**.

K9 — Medium: A **NetworkPolicy** defines how pods are allowed to communicate. Default behavior (no policy): All traffic is allowed. When a policy is applied: Traffic is **default-deny** for the selected pods (ingress/egress blocked) unless explicitly allowed by the policy rules.

K10 — Medium: **RollingUpdate** replaces pods gradually (zero downtime). **Recreate** kills all existing pods before starting new ones. Choose **Recreate** for stateful applications where simultaneous running is impossible or when configuration changes require a full restart to apply cleanly.

K11 — Easy: A **Namespace** provides logical isolation within a cluster. Default namespaces: `default`, `kube-system`, `kube-public`, `kube-node-lease`.

K12 — Easy: `kubectl describe` shows detailed status. Unlike `kubectl get`, it shows: 1. **Events** (warnings/errors). 2. **Conditions** (Ready status details). 3. **Environment Variables** and **Mounts**.

K13 — Easy: A **ConfigMap** stores non-confidential configuration data. Pods consume it via: 1. **Environment Variables**. 2. **Volume Mounts** (files inside the container).

K14 — Easy: **ClusterIP**: Internal cluster access only. **NodePort**: Exposes service on a static port on each Node's IP. **LoadBalancer**: Provisions an external cloud Load Balancer to expose the service.

K15 — Easy: A **DaemonSet** ensures a copy of a pod runs on all (or specific) nodes. Use one for system-level agents like logging (Fluentd), monitoring (Prometheus), or network plugins (Calico) rather than application workloads.

K16 — Medium: **Taints** repel pods from a node; **Tolerations** allow pods to schedule on tainted nodes. **Node Affinity** attracts pods to nodes. Use case: Taint a GPU node (repel general pods); Affinity a DB pod to co-locate with app pods.

K17 — Medium: A **Kubernetes Operator** is a custom controller that manages custom resources (CRDs) to automate complex application lifecycle management. It differs from a standard controller by encoding domain-specific operational knowledge (e.g., backup, scaling, upgrades) for stateful apps like databases.

K18 — Hard: etcd uses **strong consistency** (linearizable). If quorum is lost, the cluster becomes **read-only/unavailable**. Recovery: Restore the latest **etcd snapshot** to a new node, initialize a new cluster, and rejoin nodes to form a new quorum.

K19 — Hard: **Role** (namespace-scoped permissions), **ClusterRole** (cluster-scoped). **RoleBinding** links a Role to a Subject in a namespace. **ClusterRoleBinding** links a ClusterRole to a Subject cluster-wide.
```yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  namespace: default
  name: pod-reader
rules:
- apiGroups: [""]
  resources: ["pods", "pods/log"]
  verbs: ["get", "list", "watch"]
---
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  name: pod-reader-binding
  namespace: default
subjects:
- kind: User
  name: developer
  apiGroup: rbac.authorization.k8s.io
roleRef:
  kind: Role
  name: pod-reader
  apiGroup: rbac.authorization.k8s.io
```

K20 — Hard: **ndots** (default 5) causes the resolver to try appending search domains up to 5 levels deep. For external domains, this triggers excessive queries (e.g., `google.com`, `google.com.ns`, etc.) causing timeouts. Fix: Set `dnsConfig` in pod spec with `ndots: 1` or optimize the `search` list.

D1 — Easy: **Rebase** rewrites history to create a linear commit chain. **Merge** preserves history and creates a merge commit. Avoid rebasing on **shared/public branches** where others may have based work on the original commits.

D2 — Easy: **TCP** is connection-oriented and reliable (HTTP, SSH). **UDP** is connectionless and faster (DNS, VoIP).

D3 — Medium: A **race condition** occurs when output depends on the sequence of uncontrollable events. Example: Two threads increment a shared counter without a lock; the final value is lower than expected due to lost updates.

D4 — Medium: **Symmetric** uses one key for encryption/decryption. **Asymmetric** uses a public/private key pair. In **TLS**, Asymmetric encryption secures the handshake (key exchange), while Symmetric encryption secures the data transfer (speed).

D5 — Medium: A **layer** is an immutable filesystem snapshot in a container image. **Layer caching**: Place instructions that change frequently (e.g., `COPY app`) at the bottom of the Dockerfile to maximize cache hits for stable layers (e.g., `COPY dependencies`).

D6 — Medium: 1. Check **upstream service health** (is the backend running?). 2. Review **load balancer logs** (timeouts?). 3. Check **resource limits** (OOMKilled?). 4. Verify **dependency connectivity** (DB/Cache availability).

D7 — Hard: **CAP Theorem**: Consistency, Availability, Partition Tolerance. **CP** (Consistency/Partition): Zookeeper (blocks requests during partition). **AP** (Availability/Partition): Cassandra (serves stale data during partition).

D8 — Medium: **Horizontal Scaling** adds more nodes; **Vertical Scaling** adds resources to existing nodes. Patterns enabling horizontal scaling: **Stateless design**, **Sharding**, and **Load Balancing**.

D9 — Easy: An **environment variable** is dynamic configuration passed to processes. To persist across sessions on Linux, add `export VAR=value` to `~/.bashrc` or `~/.profile`.

D10 — Easy: **JSON** is strict key-value syntax. **YAML** is human-readable with indentation. Advantage JSON: Strict parsing. Advantage YAML: Comments and less verbose.

D11 — Easy: **REST API** is an architectural style using HTTP. **GET** (retrieve), **POST** (create), **PUT** (replace), **DELETE** (remove).

D12 — Easy: **Version pinning** locks dependency versions (e.g., `package-lock.json`). It ensures reproducible builds by preventing unexpected breaking changes from upstream updates.

D13 — Easy: **Compiled** (C, Go) translates source to binary before execution. **Interpreted** (Python, JS) translates source to machine code at runtime.

D14 — Medium: A **Reverse Proxy** sits in front of servers (protects backend). A **Forward Proxy** sits in front of clients (bypass restrictions). **Nginx** (SSL termination), **HAProxy** (High-performance load balancing).

D15 — Medium: A **deadlock** is a permanent blocking of processes. **Coffman Conditions**: 1. Mutual Exclusion. 2. Hold and Wait. 3. No Preemption. 4. Circular Wait.

D16 — Medium: **Eventual Consistency** ensures data converges over time. **Cassandra** uses **Read Repair** and **Hinted Handoff**. Tunable levels: **ONE** (fastest), **QUORUM** (balanced), **ALL** (strongest).

D17 — Medium: **mTLS** authenticates both client and server via certificates. Handshake: Client sends cert, Server verifies; Server sends cert, Client verifies. Used in **Service Meshes** (Istio) and internal API security.

D18 — Hard: A **Hash Table** maps keys to indices via a hash function. **Collision Resolution**: Chaining (linked lists) or Open Addressing (probing). **Time Complexity**: Avg O(1), Worst O(n) (all keys collide).

D19 — Hard: A **Circuit Breaker** prevents cascade failures. **Closed** (normal), **Open** (fail fast), **Half-open** (test recovery). It relates to retries by stopping retries when open; **Exponential Backoff** is used when transitioning to Half-Open to test recovery.

D20 — Hard: **Optimistic** (check on commit, e.g., version column) suits low contention. **Pessimistic** (lock on read, e.g., `SELECT FOR UPDATE`) suits high contention. Choose Optimistic for read-heavy, Pessimistic for write-heavy/conflict-prone.
