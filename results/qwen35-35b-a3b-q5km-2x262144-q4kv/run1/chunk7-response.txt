AA1 — **Monolith vs. Microservices:** A monolith is a single unified codebase deployed as one unit. Microservices decompose this into independent, loosely coupled services deployed separately.
**3 Microservices Problems:**
1.  **Distributed Complexity:** Network latency, partial failures, and eventual consistency issues replace local function calls.
2.  **Operational Overhead:** Requires orchestration (Kubernetes), service discovery, and distributed tracing.
3.  **Data Consistency:** Maintaining ACID transactions across service boundaries is difficult (requires Sagas).

AA2 — **Sync vs. Async:** Synchronous communication blocks the caller until a response is received (request-response). Asynchronous communication allows the caller to continue without waiting for an immediate response (fire-and-forget or callback).
**Examples:** Synchronous: REST/HTTP, gRPC. Asynchronous: RabbitMQ, Apache Kafka.

AA3 — **API Gateway vs. Reverse Proxy:** A reverse proxy routes traffic to backend servers (Layer 7/4). An API Gateway adds business logic (auth, rate limiting, API versioning, transformation).
**Replacement:** An API Gateway can replace a reverse proxy for external traffic because it includes proxying capabilities. However, internal service-to-service routing often still benefits from a lightweight reverse proxy or service mesh sidecar to avoid gateway bottlenecks.

AA4 — **Saga Pattern:** A sequence of local transactions where each updates the database and triggers the next. If one fails, compensating transactions undo previous changes.
**Choreography vs. Orchestration:** Choreography uses events to trigger the next step (decentralized). Orchestration uses a central coordinator service to manage the flow (centralized).
**Appropriateness:** Choreography for simple, low-coupling flows. Orchestration for complex workflows requiring visibility, error handling, and state management.

AA5 — **CQRS:** Separates read (Query) and write (Command) operations into different models.
**Pairing with Event Sourcing:** Event Sourcing stores state changes as a sequence of events. CQRS uses these events to build optimized read models.
**Problem Solved:** Allows independent scaling of read and write workloads and provides a complete audit trail of state changes, which traditional CRUD (current state only) does not.

AA6 — **Circuit Breaker:** Prevents repeated calls to a failing service.
**Three States:** Closed (normal), Open (failures detected, requests blocked), Half-Open (testing if service recovered).
**Differs from Retry:** Retry attempts to fix the failure; Circuit Breaker stops traffic to prevent cascading failures.
**Use Together:** Use retries for transient network glitches, but wrap them in a Circuit Breaker to stop retrying if the service is actually down.

AA7 — **Multi-Tenant Isolation:**
1.  **Shared DB, Shared Schema:** Lowest cost, highest noisy-neighbor risk, complex security (row-level).
2.  **Shared DB, Separate Schema:** Medium cost, moderate noisy-neighbor risk, easier security isolation.
3.  **Separate DB per Tenant:** Highest cost, lowest noisy-neighbor risk, best security, highest operational complexity (backup/migration).

AA8 — **Consistency:** Strong consistency ensures all nodes see the same data immediately. Eventual consistency allows temporary divergence.
**Inventory Architecture:** Use **Strong Consistency**.
**Data Store:** Relational DB (e.g., PostgreSQL) with row-level locking or distributed transactions (2PC).
**Patterns:** Optimistic locking (versioning) or distributed locks (Redis/Zookeeper) to reserve stock before committing.

AA9 — **Sidecar Proxy:** A container deployed alongside the application to handle network traffic.
**Envoy/Istio:** Envoy handles the data plane (traffic routing, mTLS). Istio Control Plane configures Envoy.
**Data vs. Control Plane:** Data plane processes traffic; Control plane manages configuration.
**mTLS:** Application-level TLS secures data; mTLS between sidecars authenticates service identity and encrypts traffic at the network layer, independent of app code.

AA10 — **Strangler Fig:** Incrementally migrate functionality from a legacy system to new services.
**Strategy:** Deploy an API Gateway in front of the legacy .NET monolith. Route specific endpoints (e.g., `/api/orders`) to new .NET 8 microservices. Gradually move more endpoints until the legacy monolith is "strangled" and can be decommissioned.

AA11 — **12-Factor App:**
1.  **Codebase:** One codebase tracked in revision control.
2.  **Dependencies:** Explicitly declared and isolated.
3.  **Config:** Stored in environment variables.
4.  **Backing Services:** Treated as attached resources.
5.  **Build, Release, Run:** Strictly separated.
6.  **Processes:** Stateless, state stored in backing services.
7.  **Port Binding:** Services export ports.
8.  **Concurrency:** Scale via process model.
9.  **Disposability:** Fast startup/shutdown.
10. **Dev/Prod Parity:** Keep environments similar.
11. **Logs:** Treated as event streams.
12. **Admin Processes:** Run as one-off commands.
**Why:** Ensures portability, scalability, and maintainability in cloud environments.

AA12 — **Event-Driven (Kafka) vs. Request-Driven (REST):**
*   **Coupling:** Event-driven is loosely coupled; Request-driven is tightly coupled.
*   **Scalability:** Event-driven scales better for high throughput; Request-driven is simpler for low volume.
*   **Debugging:** Request-driven is easier to trace; Event-driven requires distributed tracing.
*   **Ordering:** Kafka guarantees ordering per partition; REST does not.
*   **Failure:** Event-driven requires idempotency/retries; REST uses timeouts.
**Choice:** Use Event-driven for decoupled, high-volume data pipelines. Use Request-driven for synchronous user interactions.

AA13 — **Load Balancer:** Distributes traffic across multiple servers.
**L4 vs. L7:** L4 (Transport) routes based on IP/Port. L7 (Application) routes based on HTTP headers/URL.
**Use Case:** L7 is required for SSL termination or routing `/api` to backend A and `/static` to backend B.

AA14 — **Message Queue:** Stores messages for consumers to retrieve.
**MQ (RabbitMQ) vs. Log (Kafka):** MQ deletes messages after consumption (point-to-point). Kafka retains logs for replay (publish-subscribe).
**Pick:** Use RabbitMQ for task queues. Use Kafka for event streaming and audit logs.

AA15 — **Container vs. VM:** A container shares the host OS kernel; a VM virtualizes hardware and runs a full guest OS.
**Popularity:** Containers are lighter, start faster, and have higher density, making them ideal for microservices.

AA16 — **Stateless vs. Stateful:** Stateless services store no session data (e.g., Web Server). Stateful services store session/data locally (e.g., Database).
**Scaling:** Stateless services are easier to scale horizontally because any instance can handle any request.
**Example:** Stateless: Nginx. Stateful: Redis.

AA17 — **ORM:** Maps database tables to programming language objects.
**Examples:** Hibernate (Java), Entity Framework (.NET).
**Trade-offs:** ORMs increase developer productivity but can generate inefficient SQL and hide database complexity. Raw SQL offers performance control but is verbose.

AA18 — **Bulkhead Pattern:** Isolates resources (threads, connections) for specific services to prevent failure propagation.
**Implementation:** Use separate thread pools for each downstream service call. If one service hangs, it only exhausts its pool, not the application's total threads.

AA19 — **OT vs. CRDT:** OT requires a central server to resolve conflicts (server authoritative). CRDTs allow clients to merge changes locally without a server.
**Offline Support:** CRDTs are superior for offline editing as they guarantee convergence without server coordination.
**Choice:** Choose CRDTs for a system requiring offline editing and eventual consistency.

AA20 — **Distributed Tracing Challenges:**
*   **Context Propagation:** Pass TraceID/SpanID via headers (W3C/B3) across services.
*   **Sampling:** Head-based (random at start) vs. Tail-based (sample based on latency/errors).
*   **Storage:** High-volume data requires efficient storage (Jaeger/Tempo).
*   **Async Messaging:** Inject trace context into message headers (e.g., Kafka headers) to maintain continuity across the queue.

OP1 — **Hypervisor Type 1 vs. Type 2:** Type 1 runs directly on hardware (Bare-metal). Type 2 runs on top of an OS.
**Examples:** Type 1: VMware ESXi, Microsoft Hyper-V. Type 2: VirtualBox, VMware Workstation.

OP2 — **SAN vs. NAS:** SAN provides block-level storage (appears as a disk). NAS provides file-level storage (appears as a folder).
**Usage:** SAN for databases (high I/O). NAS for file sharing.
**Protocols:** SAN: iSCSI, Fibre Channel. NAS: NFS, SMB/CIFS.

OP3 — **3-Tier vs. Leaf-Spine:** 3-Tier (Core/Dist/Access) has bottlenecks at the core. Leaf-Spine is a Clos network with equal latency between any two nodes.
**Problem Solved:** Leaf-Spine eliminates oversubscription and reduces latency compared to 3-Tier.

OP4 — **Proxmox HA:**
*   **Architecture:** 3+ nodes for quorum.
*   **Quorum:** Majority vote required to avoid split-brain.
*   **Fencing:** STONITH (e.g., iDRAC) to power off failed nodes.
*   **Storage:** Ceph (3 replicas) or Shared iSCSI/NFS.
*   **Failure:** VMs restart on surviving nodes automatically.

OP5 — **VLAN Trunking:** Carries multiple VLANs over a single link using 802.1Q tags.
**Linux Config:** Create sub-interfaces (e.g., `eth0.10`, `eth0.20`) with `ip link add link eth0 name eth0.10 type vlan id 10`.

OP6 — **RAID Levels:** RAID 1 (Mirror), RAID 5 (Striping + Parity), RAID 6 (Double Parity), RAID 10 (Mirror + Stripe), RAID Z2 (ZFS Double Parity).
**Choice:** RAID 10 (or Z2 equivalent) for 6 drives. Best balance of read/write performance and fault tolerance (can lose 2 drives).

OP7 — **DNS Architecture:**
*   **Internal:** AD DNS for clients.
*   **External:** Public authoritative servers.
*   **Split-Horizon:** Internal clients resolve internal IPs; external resolve public IPs.
*   **Forwarders:** Internal DNS forwards unknown queries to public resolvers.
*   **Laptop:** Connects via DHCP, queries internal DNS, resolves internal resources; VPN tunnels DNS traffic securely.

OP8 — **PKI Chain:** Root CA (offline) -> Subordinate CA (issues certs) -> End-entity.
**Revocation:** CRL (list) vs. OCSP (real-time check).
**Consequences:** Compromised Root = Trust collapse (revoke all). Compromised Sub = Revoke sub, issue new certs.

OP9 — **WAN Consolidation:**
*   **MPLS:** High cost, high reliability.
*   **IPsec:** Low cost, variable reliability.
*   **SD-WAN:** Dynamic path selection, QoS for VoIP.
*   **WireGuard:** Simple, encrypted mesh.
*   **Recommendation:** SD-WAN. Best for VoIP QoS, cost efficiency, and managing 5 sites.

OP10 — **iDRAC/iLO/IPMI:** Out-of-band management interfaces.
**Critical:** Allows server control (power, console) even if OS is down.
**Exposure:** Exposed to internet = Remote root access risk (hardware compromise).

OP11 — **LVM vs. ZFS:** LVM manages block devices (thin provisioning). ZFS manages file systems (snapshots, checksums).
**Semantics:** LVM snapshots are copy-on-write. ZFS snapshots are instant, immutable, and space-efficient.
**Use:** LVM for simple block storage. ZFS for data integrity and snapshots.

OP12 — **Stretch Cluster:**
*   **Design:** Ceph monitors in both sites + tiebreaker (quorum disk).
*   **Latency:** <10ms required for Ceph consensus.
*   **Split-Brain:** If network fails, both sites think they are primary.
*   **Bad Idea:** 2-site Ceph without tiebreaker risks data corruption during network partition.

OP13 — **DHCP DORA:** Discover, Offer, Request, Acknowledge.
**Unavailable Server:** Client retries. If lease expires, client loses IP and must restart DORA.

OP14 — **Managed vs. Unmanaged:** Unmanaged is plug-and-play. Managed supports VLANs, QoS, ACLs, and monitoring.
**Importance:** VLANs and QoS are critical for security and VoIP performance in enterprises.

OP15 — **NTP:** Network Time Protocol.
**Critical:** Certificates, logs, and cluster consensus rely on accurate time.
**Drift:** Clock skew causes authentication failures (Kerberos) and log analysis issues.

OP16 — **Physical vs. VM vs. Container:** Physical (Bare metal) has no overhead. VM has OS overhead. Container has kernel overhead.
**Bare Metal:** Use for high-performance databases or latency-sensitive workloads where virtualization overhead is unacceptable.

OP17 — **UPS vs. Generator:** UPS provides immediate short-term power (minutes). Generator provides long-term power (hours).
**ATS:** Automatic Transfer Switch switches load from Utility to UPS to Generator seamlessly.

OP18 — **Active-Passive vs. Active-Active:** Active-Passive (one node handles load, one waits). Active-Active (both handle load).
**PostgreSQL HA:** Pacemaker/Corosync with STONITH fencing. Virtual IP floats to active node. Failover triggers fencing of failed node to prevent split-brain.

OP19 — **Backup Strategy:**
*   **3-2-1:** 3 copies, 2 media types, 1 offsite.
*   **Immutable:** Prevents ransomware deletion.
*   **Test:** Regular restore drills.
*   **Backup vs. DR:** Backup is data copy; DR is the plan/process to restore operations.

OP20 — **802.1X:**
*   **Roles:** Supplicant (Client), Authenticator (Switch), Server (RADIUS).
*   **Methods:** EAP-TLS (Cert-based, secure) vs. PEAP (Username/Password).
*   **Failure:** Port blocked or placed in Guest VLAN.
*   **MAB:** Fallback for devices without 802.1X (e.g., printers) using MAC address as credential.
