CL1 — **IaaS (Infrastructure as a Service)** provides virtualized computing resources (servers, storage, networking). Example: AWS EC2. **PaaS (Platform as a Service)** provides a framework for developing, running, and managing applications without dealing with underlying infrastructure. Example: Google App Engine. **SaaS (Software as a Service)** delivers software applications over the internet, managed entirely by the provider. Example: Microsoft 365.

CL2 — **Shared Responsibility Model:** The provider secures the cloud (hardware, facilities), the customer secures what is *in* the cloud. **IaaS VM:** Provider secures hypervisor/physical host; Customer secures OS, patching, apps, data. **Managed Kubernetes:** Provider secures control plane/infrastructure; Customer secures node OS, pod security, network policies, secrets. **SaaS (M365):** Provider secures everything; Customer secures identity, access management, and data classification.

CL3 — **VPC:** Isolated virtual network. **Subnet:** IP range within a VPC (public/private). **Security Group:** Stateful firewall at the instance level (allow rules only). **NACL:** Stateless firewall at the subnet level (allow/deny rules). **Layering:** VPC contains Subnets; Subnets contain Instances protected by SGs and NACLs. **Stateful:** SG remembers return traffic; **Stateless:** NACL requires explicit rules for both inbound and outbound traffic.

CL4 — **Landing Zone:** Secure, compliant baseline environment. **OUs:** Organize accounts by function/environment. **Guardrails:** Policy constraints (e.g., no public S3 buckets). **Shared Networking:** Central hub (Transit Gateway/Hub) connects spokes (VPCs) for traffic flow. **Centralized Logging:** Aggregates logs to a security account. **Single-Account Problems:** High blast radius, billing complexity, inability to enforce granular permissions, lack of isolation between dev/prod.

CL5 — **IaC:** Managing infrastructure via machine-readable definition files. **Terraform/OpenTofu:** Multi-cloud, declarative, state-managed. **Pulumi:** Multi-cloud, imperative (using general languages like Python/TS). **CloudFormation/Bicep:** Native to AWS/Azure, tightly integrated, less portable. **Crossplane:** IaC via Kubernetes API. **Choice:** Terraform for multi-cloud, Bicep for Azure-native, Pulumi for devs preferring code, Crossplane for K8s-centric orgs.

CL6 — **HPA:** Scales pods based on CPU/Memory metrics. **VPA:** Adjusts resource requests/limits per pod (requires restart). **Cluster Autoscaler:** Adds/removes nodes based on pending pods. **KEDA:** Event-driven scaling (e.g., queue length). **Use KEDA over HPA:** When scaling is triggered by external events (Kafka, SQS, RabbitMQ) rather than internal resource utilization.

CL7 — **Strategy:** Define RPO (data loss tolerance) and RTO (downtime tolerance). **Active-Active:** Both regions serve traffic (low RTO, high cost). **Active-Passive:** One region active, one standby (higher RTO, lower cost). **DB:** Async replication (performance, eventual consistency) vs Sync (consistency, latency). **DNS:** Route53 Health Checks trigger failover. **State:** Externalized (Redis/ElastiCache) to survive region loss. **DR Types:** Pilot Light (min data running), Warm Standby (scaled-down prod), Multi-site Active-Active (full redundancy).

CL8 — **GitOps:** Git repository is the single source of truth for infrastructure/app state. **ArgoCD:** Pulls from Git, syncs to cluster. **FluxCD:** Pulls from Git, syncs to cluster. **Secrets:** Sealed Secrets (encrypted in Git), External Secrets Operator (fetches from Vault/AWS Secrets Manager). **Drift:** GitOps controller detects manual changes and reverts them to match Git state during reconciliation.

CL9 — **Architecture:** Use **StatefulSet** for stable network IDs and storage. **PV/PVC:** Use `local` storage class for low latency (node-bound). **Headless Service:** For stable DNS names per pod. **Init Containers:** For config file generation before app start. **ConfigMaps:** Mount as volumes for specific paths. **PDB:** Ensure minimum available pods during updates to prevent data loss.

CL10 — **SA:** K8s identity for pods. **Workload Identity:** Maps K8s SA to Cloud IAM identity. **IRSA:** AWS specific mapping via OIDC. **Mounting Creds:** Bad practice (risk of leakage, rotation issues, violates least privilege). **Federation:** Uses OIDC tokens to request temporary IAM credentials, eliminating static secrets.

CL11 — **Object (S3):** Unstructured data, high durability (11 9s), low latency. **Block (EBS):** Attached to VM, low latency, high IOPS, limited durability (replicated). **File (EFS):** Shared file system, NFS/SMB protocol, moderate latency. **Use Object:** Backups, media. **Block:** OS disks, DBs. **File:** Shared configs, home dirs.

CL12 — **Zero Trust:** Verify every request. **Network Policies:** Restrict pod-to-pod traffic. **Service Mesh (mTLS):** Encrypts service-to-service traffic. **Pod Identity:** Workload identity federation. **Admission Controllers (OPA):** Enforce policies on deployment. **Image Signing:** Verify artifacts before deploy. **Runtime (Falco):** Detects anomalies. **Difference:** Traditional relies on perimeter (firewall); Zero Trust assumes breach and verifies identity/context continuously.

CL13 — **CDN:** Distributed network of servers caching content at edge locations. **How:** Caches static assets closer to users. **Services:** AWS CloudFront, Azure CDN. **Use CDN:** For static assets (images, CSS, JS) to reduce latency and origin load. **Direct Origin:** Only for dynamic content or when caching is not viable.

CL14 — **Region:** Geographic area with multiple AZs. **AZ:** One or more discrete data centers with independent power/cooling. **Multi-AZ:** Ensures high availability if one data center fails. **Zonal:** Resource exists in one AZ (e.g., EBS). **Regional:** Resource spans AZs (e.g., ELB, RDS Multi-AZ).

CL15 — **Managed DB:** Service where provider manages hardware, OS, DB software. **Provider Tasks:** Patching, backups, failover, scaling, monitoring. **User Tasks:** Schema design, queries, access control. **Benefit:** Reduces operational overhead and improves availability.

CL16 — **IAM:** Users (people), Groups (collections), Roles (temporary access), Policies (permissions). **Least Privilege:** Grant only necessary permissions. **Application:** Use Roles for workloads, not Users; Group users by function; Review permissions regularly; Use MFA.

CL17 — **Serverless:** Event-driven execution without managing servers. **Compare:** Lambda (AWS), Functions (Azure), Cloud Functions (GCP). **Cold Starts:** Latency when initializing a new instance. **Poor Fit:** Long-running processes, high CPU usage, stateful applications, strict latency requirements.

CL18 — **Ingress:** L7 routing rules (HTTP/HTTPS). **Gateway API:** Enhanced routing, multi-cluster support, standardized. **LB:** L4 load balancing. **Mesh Gateway:** L7 termination within service mesh. **TLS:** LB terminates (offloads app), Ingress terminates (controller handles cert), Mesh terminates (mTLS between services). **Use Gateway API:** For complex routing, multiple ingress controllers, or when standard Ingress is insufficient.

CL19 — **Strategy:** **RI/SP:** Commit to usage for discounts (predictable). **Spot:** Interruptible for fault-tolerant workloads (cheap). **Right-sizing:** Match instance size to actual usage. **Tiering:** Move old data to Glacier. **Idle Detection:** Tag and terminate unused resources. **FinOps:** Cross-team cost accountability. **Tools:** AWS Cost Explorer, Kubecost. **Process:** Monthly reviews, chargeback/showback reports to teams.

CL20 — **Strategy:** **Abstraction:** Kubernetes (K8s) for runtime, Terraform for infra. **Networking:** Direct Connect/ExpressRoute for low latency. **Identity:** OIDC federation across clouds. **Trade-offs:** Multi-cloud increases complexity, networking costs, and operational overhead. **Justified:** Regulatory compliance, avoiding single-vendor risk, best-of-breed services. **Unnecessary:** If one cloud meets all needs, multi-cloud adds cost without value.

OT1 — **Purdue Model:** ISA-95 standard for IT/OT integration. **Levels:** 0 (Physical Process), 1 (Sensors/Actuators), 2 (Control Devices/PLCs), 3 (Operations/SCADA), 4 (Enterprise/Business).

OT2 — **IT:** Focus on confidentiality, integrity, availability (CIA), frequent patching. **OT:** Focus on safety, uptime, real-time control, legacy systems. **Patching:** OT patching risks downtime or safety hazards; often requires vendor approval and maintenance windows.

OT3 — **DMZ (IDMZ):** Buffer zone between IT and OT. **Traffic:** Historian pulls data from SCADA (push allowed), Engineering stations access DMZ. **Direction:** IT to OT is restricted; OT to IT is controlled. **Direct L4 to L1:** Dangerous because it bypasses safety logic and monitoring, allowing direct manipulation of physical processes.

OT4 — **Protocols:** 1. **Modbus TCP:** L2/L3, simple, no security. 2. **OPC UA:** L2/L3, secure, platform-independent. 3. **EtherNet/IP:** L2, CIP protocol, common in automation. 4. **PROFINET:** L2, real-time Ethernet. 5. **MQTT:** L3/Cloud, lightweight pub/sub. **Location:** Modbus/PROFINET (L1/L2), OPC UA/MQTT (L3/L4).

OT5 — **Data Diode:** Hardware enforcing unidirectional data flow. **Firewall:** Bidirectional traffic control. **Use Diode:** When data must flow out (e.g., telemetry) but nothing can come in (high security). **Limitation:** Cannot send commands back to OT network.

OT6 — **Flow:** PLC -> Historian (L3) -> Edge Gateway (Protocol translation) -> Cloud (MQTT/HTTPS). **Security:** DMZ at Edge Gateway. Encrypt traffic (TLS). Isolate OT network from Internet. Use firewalls between Edge and Cloud.

OT7 — **Zones:** Logical grouping of assets with similar security requirements. **Conduits:** Controlled paths between zones. **Model:** Zone 4 (Corporate), Zone 3 (Engineering/SCADA), Zone 2 (Production Control), Zone 1 (Field Devices). **Conduits:** Firewalls between zones. SIS (Safety) in isolated Zone 0/1.

OT8 — **Segmentation:** VLANs per production line. **Firewalls:** Between VLANs and DMZ. **Switches:** Managed Layer 2/3 switches with ACLs. **Remote Access:** Jump hosts with MFA and session recording. **Transition:** Parallel run (new network alongside old), phased migration during maintenance windows, validate safety before cutting over.

OT9 — **SIS:** Independent system for safety shutdown (e.g., pressure relief). **BPCS:** Basic Process Control System (normal operation). **IEC 61511:** Standard for safety instrumented systems. **Air-gap:** Prevents cyber attacks from compromising safety logic. **Ransomware on SIS:** Can disable safety shutdowns, leading to physical damage or injury.

OT10 — **PLC:** Logic controller for specific machines. **DCS:** Distributed Control System for continuous processes (refineries). **SCADA:** Supervisory Control and Data Acquisition for geographically dispersed assets. **Coexist:** Yes, PLCs often feed SCADA, DCS manages complex process loops.

OT11 — **Architecture:** Jump host (bastion) in DMZ. **MFA:** Required for all access. **Recording:** Session recording for audit. **Vendor:** Time-limited, scoped access. **No Direct RDP:** Exposes HMI to corporate network, increasing attack surface and risk of lateral movement to control layer.

OT12 — **Plan:** **Monitor:** IDS for OT protocols (e.g., Nozomi, Claroty). **Inventory:** Verify asset status. **IR:** Isolate compromised segment (VLAN/Firewall). **Ops:** Notify control room to switch to manual/local control. **Operator:** Maintain physical safety, monitor process variables, do not reset systems without authorization.

OT13 — **HMI:** Interface for operators to view and control processes. **Role:** Visualization and manual intervention. **Difference:** HMI is for operation; Engineering Workstation is for configuration, programming, and maintenance.

OT14 — **Historian:** Time-series database for process data. **Importance:** Trend analysis, compliance, troubleshooting. **Plant (L3):** Real-time process data. **Enterprise (L4):** Aggregated data for business analytics (OEE, production totals).

OT15 — **Ladder Logic:** Graphical programming language resembling relay logic. **Device:** PLC. **Dominance:** Standardized, easy for electricians to understand, highly reliable, legacy codebases are massive.

OT16 — **RTU:** Remote Terminal Unit. **Differs:** RTU focuses on telemetry and comms over long distances; PLC focuses on local logic. **Deploy RTU:** Pipelines, wind farms, water distribution (harsh, remote environments).

OT17 — **Analog:** Continuous signal (voltage/current). **Digital:** Discrete (on/off). **4-20mA:** Preferred over 0-20mA because 0mA indicates a wire break (fail-safe), whereas 0mA could be a valid signal in 0-20mA.

OT18 — **Inventory:** Knowing all assets is critical for vulnerability management. **Tools:** Passive (packet sniffing), Active (scanning). **Active Scanning:** Dangerous because legacy OT devices may crash or malfunction under scan traffic; passive is safer.

OT19 — **MITRE ATT&CK ICS:** Framework for ICS-specific threats. **Tactics:** Reconnaissance (Scanning), Initial Access (Phishing), Execution (Malware), Persistence (Backdoor), Impact (Disruption). **Difference:** Focuses on physical impact and industrial protocols. **Usage:** Map current defenses to tactics to find gaps (e.g., no monitoring for Modbus commands).

OT20 — **SOC:** **Logs:** Passive collection (SPAN ports) to avoid load. **SIEM:** Correlate IT/OT logs. **Rules:** OT-specific (e.g., PLC write attempts). **Staff:** IT analysts for alerts, OT engineers for validation. **False Positives:** Tune rules to reduce noise. **Firmware Change:** Alert -> Isolate PLC -> Verify against baseline -> Restore from known good image -> Investigate root cause.
