SC1-A
The pods are failing because the **Restricted Pod Security Standard (PSS)** forbids running containers as the root user (UID 0).
1.  **Image Behavior**: The new base image (`cgr.dev/chainguard/static:latest`) sets `USER root` in the Dockerfile.
2.  **Pod Spec Behavior**: The pod spec sets `securityContext.runAsNonRoot: true` but omits `runAsUser`. Kubernetes defaults `runAsUser` to 0 (root) if not explicitly set.
3.  **PSS Conflict**: Restricted PSS requires `runAsUser` to be non-zero. Since the effective UID is 0, the pod violates the policy.
4.  **Enforcement**: If the namespace PSS is set to `Enforce`, the Admission Controller should block creation. If pods are in CrashLoopBackOff, it implies they started but were killed by a runtime security enforcement (e.g., container runtime enforcing non-root) or the Admission Controller is `Audit` but a runtime policy (like Falco or a CRI plugin) is terminating the process. The upgrade likely changed the namespace policy to `Enforce` retroactively or the rolling update triggered new pods that violated the new policy.

SC1-C
**Remediation Strategy:**
1.  **Immediate Fix**: Update the Deployment manifests to explicitly set `securityContext.runAsUser: 1000` (matching the image's non-root user) and ensure `runAsNonRoot: true`.
2.  **Image Fix**: Rebuild the image with `USER 1000` in the Dockerfile to ensure consistency.
3.  **CI Gate**: Add an OPA Gatekeeper or Kyverno policy in CI to validate that `runAsUser` is non-zero and `runAsNonRoot` is true for all production manifests before merge.
4.  **Rollout**: Use a **Canary Deployment**. Deploy the fixed version to 10% of traffic, monitor for stability, then gradually increase to 100%. This prevents full outage if the fix introduces new issues.

SC2-A
**Stale Data Issue**:
1.  **Replication Lag**: Cross-region asynchronous replication introduces latency. Promoting a replica cuts off the source, but any data not yet shipped from the primary (us-east-1) to the replica (eu-west-1) is lost.
2.  **WAL Shipping**: PostgreSQL ships Write-Ahead Logs (WAL) asynchronously. If the primary fails before WAL is flushed to the replica, that data is gone.
3.  **Data Loss Window**: The window equals the replication lag at the time of failure.
4.  **Quantification**: Check `pg_stat_replication` on the replica for `write_lag`, `flush_lag`, and `replay_lag` timestamps before the outage. Compare the last committed transaction ID on the primary (if accessible) vs. the replica.

SC2-B
1.  **Import Replica**: `terraform import aws_db_instance.promoted_replica <replica-arn>`
2.  **Move State**: `terraform state mv aws_db_instance.old_primary aws_db_instance.promoted_replica`
3.  **Update References**: Update Terraform code to reference the new ARN.
4.  **Apply**: `terraform apply` to sync dependent resources (Security Groups, Route53).
5.  **Destroy Old**: `terraform destroy -target=aws_db_instance.old_primary` (after confirming new state is stable).

SC2-C
**Architecture Design**:
1.  **RDS Proxy**: Use RDS Proxy instead of PgBouncer to handle connection pooling and failover transparently at the application level.
2.  **Route53**: Use **Latency-Based Routing** with Health Checks. If us-east-1 fails, Route53 routes to eu-west-1.
3.  **Application Logic**: Implement exponential backoff with jitter for retries. Detect "stale data" via versioning or timestamps.
4.  **RPO < 1s**: Standard RDS async replication cannot guarantee <1s RPO. **Recommend Aurora Global Database**. It uses a shared storage layer with synchronous replication across regions, offering RPO < 1 second and faster failover.
5.  **Failover**: Automate failover using AWS Lambda triggered by CloudWatch alarms on primary health checks.

SC3-A

SC4-A
**Top 5 Blockers**:

2.  **Local NFS Mount**: Pods cannot share local filesystems. **Solution**: Use EFS or S3 with CSI driver.

SC6-C
**Comparison**:
1.  **Direct Connect**: High bandwidth (10Gbps+), low latency, dedicated. **Pros**: Performance. **Cons**: Cost, setup time. **Zero Trust**: Requires encryption (IPsec) on top.
2.  **ECMP VPN**: Cheaper, easier. **Pros**: Redundancy. **Cons**: Bandwidth limited by tunnel, latency jitter.
3.  **Mesh (Tailscale/Netbird)**: **Pros**: Zero-trust by design (mTLS), no IP management. **Cons**: Performance overhead.
**Recommendation**: Use **Direct Connect** for core data traffic (DB, Storage) for performance. Use **Tailscale/Netbird** for management traffic and developer access to enforce Zero Trust without IP exposure.
**Routing**: DX requires BGP. Mesh uses overlay routing.

SC10-C
**Failover Orchestration**:
1.  **Detection**: Health checks (Ping, DB connectivity). If Chicago fails for 30s, trigger failover.
2.  **DNS**: GSLB updates TTL to point to Dallas.
3.  **DB**: Patroni promotes Dallas replica (Synchronous commit ensures no data loss).
4.  **App**: Update Ingress/Service endpoints to Dallas.
5.  **State**: Update OpenTofu state to reflect Dallas as primary.
**Split-Brain**: Use **Etcd Quorum**. If Chicago is down, Dallas cannot promote unless it has quorum.
**Failback**: Once Chicago is up, re-sync data (Patroni standby), then switch traffic back during maintenance window.
