T1 — `tofu plan` calculates the execution plan and displays the changes that would be made without applying them. `tofu apply` executes the plan, creating, updating, or destroying resources to match the configuration.

T2 — State is a JSON file mapping real-world resources to configuration objects. Remote state is important for collaboration (sharing state among team members), locking (preventing concurrent writes), and security (storing sensitive data off local machines).

T3 — `count` uses integer indices; `for_each` uses maps or sets. `for_each` is preferred when resources have unique identifiers (like names) because adding/removing items doesn't shift indices, preventing unintended resource recreation.

T4 — This is "state drift". `tofu plan` detects the difference between the state file and the actual infrastructure. `tofu apply` will attempt to revert the manual changes to match the configuration, potentially overwriting the manual modification.

T5 —
```hcl
resource "aws_instance" "web" {
  for_each = toset(["web1", "web2", "web3"])
  ami      = "ami-12345"
  tags     = { Name = each.key }
}
```

T6 — `terraform_remote_state` reads the entire state file, creating tight coupling. `output` values are explicit data points. Recommended alternative: Refactor to a single root module to access outputs directly, or use `terraform_remote_state` but access specific `.outputs` keys to minimize coupling and security risks.

T7 — The `null_resource` provisioner runs asynchronously during apply; the output is computed before the provisioner finishes. Fix: Use `depends_on` on the dependent resource to wait for the `null_resource`, or write the kubeconfig to a `local_file` resource and have the output read that file.

T8 — `tofu import` CLI command maps existing infra to config manually. Limitations: No state file initially, requires manual attribute mapping. `import` block (1.5+) allows importing during `apply`, integrates with state management, and supports `depends_on` for better ordering.

T9 — A `variable` is an input parameter passed into the configuration. An `output` is data exposed from the configuration to be used by other modules or the CLI.

T10 — A provider defines the API interactions (e.g., AWS, Azure). Without `tofu init`, providers aren't downloaded or initialized, causing `tofu plan` to fail with "Provider not installed" errors.

T11 — The `.terraform.lock.hcl` file records the exact versions of providers used. It should be committed to ensure reproducible builds and prevent unexpected provider updates.

T12 — `tofu destroy` removes all managed resources. Target a single resource with `tofu destroy -target=aws_instance.example`.

T13 — A `data source` reads existing infrastructure data without managing it. Example: Using `data "aws_ami"` to find the latest Ubuntu image instead of creating a new one.

T14 — `locals` are computed values within the configuration scope. `variables` are external inputs. Use `locals` for internal logic, transformations, or constants that don't need to be overridden by users.

T15 — `lifecycle` controls resource behavior. `create_before_destroy`: Zero-downtime updates (create new, destroy old). `prevent_destroy`: Protects resources from accidental deletion. `ignore_changes`: Ignores drift in specific attributes (e.g., tags managed externally).

T16 — A `moved` block tracks resource identity changes during refactoring. It tells OpenTofu to map the old resource address to the new one in the state file, preventing recreation.

T17 — The dependency graph determines execution order. Implicit dependencies come from resource references (e.g., `aws_instance.vpc.id`). Explicit dependencies use `depends_on`. `depends_on` reduces parallelism because it forces sequential execution, slowing down apply times.

T18 — Strategies: 1. Increase `parallelism` (default 10). 2. Use `plan -refresh=false` to skip state refresh. 3. Reduce `data` sources (they block parallelism). 4. Use a remote state backend (S3) for faster locking/reads. Tradeoffs: `refresh=false` risks stale state; high parallelism increases API rate limits.

T19 — State locking prevents concurrent modifications. If a lock persists (crash), `tofu apply` fails. Manually break with `tofu force-unlock <ID>`. Risk: Data corruption if another process is actually running the apply.

T20 — 1. Update backend config in `main.tf`. 2. Run `tofu init -migrate-state`. 3. OpenTofu copies local state to S3 and deletes local copy. 4. Verify state in S3 console. 5. Run `tofu plan` to ensure no drift. 6. Run `tofu apply` to confirm functionality.

A1 — A **playbook** is a YAML file defining a sequence of tasks. A **role** is a reusable directory structure for organizing tasks, vars, and templates. A **task** is a single atomic action (e.g., install a package).

A2 — `ansible_facts` is a dictionary of gathered system information. Access OS family via `{{ ansible_facts['os_family'] }}`.

A3 — `command`: Executes without shell (no pipes). `shell`: Executes with shell (supports pipes). `raw`: Executes without setup (no Python, no fact gathering), useful for bootstrapping SSH.

A4 — Idempotency means running the task multiple times yields the same result. Example (Idempotent): `apt: state=present` (installs if missing). Example (Not): `shell: rm -rf /tmp/data` (deletes every time).

A5 —
```yaml
- name: Install htop
  package:
    name: htop
    state: present
  when: ansible_os_family == "Debian"

- name: Install htop on RedHat
  package:
    name:
      - epel-release
      - htop
    state: present
  when: ansible_os_family == "RedHat"
```

A6 — Precedence (High to Low): Extra vars > Command line > Play vars > Role vars > Host vars > Group vars > Defaults. `set_fact` overrides most at runtime.

A7 — Use `serial: 3` to limit batch size. Group nodes in inventory (e.g., `rke2_agent`). Use `wait_for` or `wait_for_connection` to ensure the first 3 nodes are ready before the next batch starts.

A8 — `ansible-vault` encrypts sensitive data. Encrypt a single variable using `ansible-vault encrypt_string 'secret_value' --name 'my_secret'`.

A9 — An inventory file lists managed hosts. **Static**: Defined in a file (INI/YAML). **Dynamic**: Generated by a script or API (e.g., AWS EC2) to reflect changing infrastructure.

A10 — `ansible webservers -m shell -a "df -h"`.

A11 — A **handler** is a task triggered by `notify`. It differs from a regular task because it only runs if notified and runs once per play at the end. Triggered via `notify: handler_name`.

A12 — `ansible.cfg` configures Ansible behavior. Common settings: `inventory`, `remote_user`, `forks`, `host_key_checking`.

A13 — `register` saves a task's output to a variable. Use it in subsequent tasks via `when: my_var.stdout == "success"` or `debug: var=my_var`.

A14 — `import_tasks` is static (evaluated at parse time). `include_tasks` is dynamic (evaluated at runtime). Use `import` for performance (no runtime overhead). Use `include` for conditional execution. `import` tasks cannot be skipped based on runtime variables.

A15 — Ansible Galaxy is a repository for roles/collections. Use `requirements.yml` to list them. Install via `ansible-galaxy install -r requirements.yml`.

A16 — Connection plugins define how Ansible talks to hosts. `ssh`: Standard Linux/Unix. `local`: Runs on control node. `network_cli`: For network devices (Cisco, Juniper) using CLI.

A17 — `delegate_to: localhost` runs a task on the control node. `run_once` runs it once for the group. Example: Gather host IPs, then `delegate_to: localhost` to update a central load balancer config file.

A18 — Strategies: 1. Increase `forks` (parallel SSH connections). 2. Enable `pipelining` (reduces SSH sessions). 3. Use `fact_caching` (avoids re-gathering facts). 4. Use `Mitogen` (faster transport). Tradeoffs: Higher forks consume more resources; caching risks stale facts.

A19 — `{{ }}`: Expressions (variables). `{% %}`: Control flow (loops, if). `{# #}`: Comments.
Snippet:
```jinja
upstream backend {
{% for server in backend_servers %}
    server {{ server.name }} {{ server.ip }} check;
{% endfor %}
}
```

A20 — 1. Set `serial: 1` for one-by-one. 2. `pre_tasks`: Deregister from LB. 3. Deploy app. 4. `post_tasks`: Health check via `uri` module. 5. `block/rescue`: If health check fails, `rescue` triggers rollback (revert to previous version). 6. Register back to LB on success.
