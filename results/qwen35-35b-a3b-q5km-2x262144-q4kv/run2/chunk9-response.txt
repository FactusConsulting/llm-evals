SC1-A
The pods are failing due to a conflict between the **Restricted Pod Security Standard (PSS)** and the container image configuration.
1.  **Restricted PSS:** Requires `securityContext.runAsNonRoot: true` AND explicitly forbids running as root (UID 0). It mandates that `runAsUser` be set to a non-zero value if `runAsNonRoot` is true.
2.  **Image Behavior:** The new base image (`cgr.dev/chainguard/static:latest`) defaults `USER` to root (UID 0).
3.  **Manifest Gap:** The manifests set `runAsNonRoot: true` but omit `runAsUser`. Kubernetes defaults `runAsUser` to 0 (root) if not specified.
4.  **Enforcement Change:** In **Baseline**, running as root is allowed. In **Restricted**, the admission controller rejects pods where the effective UID is 0, even if `runAsNonRoot` is true, because the image itself runs as root. The kubelet attempts to start the container, but the security context enforcement (or the image's entrypoint failing permission checks) causes the crash loop.

SC1-C
**Remediation Strategy:**
1.  **Immediate Fix:** Update the `securityContext` in the Deployment manifests to explicitly set `runAsUser: 1000` (or a non-root UID matching the image user).
2.  **Image Fix:** Rebuild the container image with a non-root user defined in the Dockerfile (`USER 1000`) to ensure the image itself is compliant.
3.  **CI Gate:** Add an OPA Gatekeeper or Kyverno policy in CI that scans Helm charts/Deployments to ensure `runAsUser` is non-zero when `runAsNonRoot` is true.
4.  **Rollout:** Use a **Canary Deployment** strategy. Deploy the fixed version to 10% of traffic, monitor CrashLoopBackOff rates and error logs, then gradually increase to 100%. This prevents a full outage if the fix introduces new issues.

SC2-A
**Stale Data Issue:**
1.  **Replication Lag:** Cross-region asynchronous replication introduces latency. When the primary (us-east-1) fails, the replica (eu-west-1) may not have received all WAL (Write-Ahead Log) records before the failover.
2.  **Data Loss:** Any transactions committed on the primary but not shipped to the replica before the outage are lost.
3.  **Quantification:** Check `pg_stat_replication` on the primary (if accessible) or the replica's `pg_last_wal_receive_lsn` vs `pg_last_wal_replay_lsn`. The difference in LSNs indicates the data loss window.
4.  **Implication:** Applications reading immediately after failover may see data that was written to the primary but not yet replicated.

SC2-B
1.  **Import Replica:** `terraform import aws_db_instance.promoted_replica <arn_of_promoted_replica>`
2.  **Update State:** If the resource name in state is `aws_db_instance.primary`, move it: `terraform state mv aws_db_instance.primary aws_db_instance.promoted_replica`
3.  **Update Config:** Modify the Terraform code to point to the new ARN or use the promoted replica's endpoint.
4.  **Apply:** `terraform apply` to sync dependent resources (Security Groups, Route53).
5.  **Cleanup:** `terraform destroy` the old instance (if still running) after confirming stability.

SC2-C
**Architecture Design:**
1.  **Database:** Use **Aurora Global Database**. It provides synchronous cross-region replication with sub-second RPO and automatic failover, superior to standard RDS async replication.
2.  **Connection Management:** Use **RDS Proxy** instead of PgBouncer. It handles connection pooling centrally and supports failover transparently without application reconnection storms.
3.  **DNS Failover:** Use **Route53 Health Checks** with Latency-Based Routing. If us-east-1 fails, Route53 directs traffic to eu-west-1.
4.  **Application:** Implement exponential backoff retry logic with jitter to handle transient connection failures during failover.
5.  **RPO < 1s:** Aurora Global DB ensures this via synchronous replication. Standard RDS async cannot guarantee this.

SC4-A
**Top 5 Blockers:**
1.  **Local Sessions:** Kubernetes pods are ephemeral. **Solution:** Move sessions to Redis (ElastiCache).
2.  **Local NFS Mount:** Pods cannot share local filesystems across nodes. **Solution:** Use EFS or S3 for file storage.
3.  **On-Prem Oracle:** Kubernetes is in the cloud. **Solution:** Migrate to Amazon RDS for Oracle or Aurora PostgreSQL.
4.  **IBM MQ:** On-prem mainframe integration. **Solution:** Use AWS MQ (Managed IBM MQ) or bridge via API.
5.  **Sticky Sessions:** F5 relies on IP affinity. **Solution:** Use Kubernetes Ingress with sticky sessions or move session state to Redis.

SC6-C
**Comparison:**
1.  **AWS Direct Connect:** Dedicated 10Gbps+ link. High cost, low latency, high security. Fits Zero Trust (dedicated path). Requires BGP routing changes.
2.  **ECMP VPN:** Two VPN tunnels. Cheaper, but limited bandwidth (2x1Gbps). Encryption overhead. Fails over automatically.
3.  **Mesh (Tailscale/Netbird):** Overlay network. Easiest to manage, but adds latency. Good for admin access, less for core traffic.
**Recommendation:** **Direct Connect** for core traffic (bandwidth/security). **Mesh** for admin access.
**Routing:** DX requires BGP peering. VPN uses static routes. Mesh uses overlay routing.

SC10-C
**Failover Orchestration:**
1.  **Detection:** Prometheus monitors site health. If Chicago nodes unreachable for 2 mins, trigger alert.
2.  **DNS:** GSLB (Route53) detects health check failure and switches DNS to Dallas.
3.  **DB:** Patroni detects primary down, promotes Dallas node (synchronous commit ensures no data loss).
4.  **Cache:** Redis Sentinel promotes new master.
5.  **Queue:** RabbitMQ Federation syncs messages.
6.  **Storage:** MinIO Site Replication catches up.
7.  **State:** Ansible/Tofu updates state to mark Dallas as Primary.
**Split-Brain:** Use etcd quorum. If Chicago etcd is down, Dallas cannot promote unless it has quorum.
**Failback:** Once Chicago recovers, demote Dallas, re-sync MinIO, and switch DNS back.
