AA1 — **Monolith vs. Microservices:** A monolith is a single, unified codebase deployed as one unit. Microservices are a collection of small, independent services communicating over a network.
**Microservices Problems:**
1.  **Distributed Complexity:** Debugging requires tracing across multiple services.
2.  **Data Consistency:** Maintaining ACID transactions across services is difficult (requires Sagas).
3.  **Operational Overhead:** Requires orchestration (Kubernetes), service discovery, and monitoring for each service.

AA2 — **Sync vs. Async:** Synchronous communication waits for an immediate response (request-response). Asynchronous communication sends a message and continues without waiting for an immediate reply.
**Examples:** Synchronous: HTTP/REST, gRPC. Asynchronous: RabbitMQ, Apache Kafka.

AA3 — **API Gateway vs. Reverse Proxy:** A reverse proxy routes traffic to backend servers (L4/L7). An API Gateway adds business logic (auth, rate limiting, transformation) on top of routing.
**Replacement:** An API Gateway *can* replace a reverse proxy functionally, but often a lightweight reverse proxy (like Nginx) is kept in front of the Gateway for SSL termination or basic load balancing to offload the Gateway.

AA4 — **Saga Pattern:** A sequence of local transactions where each updates data within a service. If one fails, compensating transactions undo previous changes.
**Choreography:** Services react to events (loose coupling, hard to track flow). **Orchestration:** A central coordinator tells services what to do (tighter coupling, easier debugging).
**Use:** Choreography for simple flows; Orchestration for complex workflows requiring visibility and control.

AA5 — **CQRS:** Separates read (Query) and write (Command) operations into different models.
**Pairing with Event Sourcing:** Event Sourcing stores state changes as a sequence of events. CQRS uses these events to build optimized read models.
**Problem Solved:** Allows independent scaling of read/write paths and provides a complete audit trail/history of state changes, which traditional CRUD (current state only) does not.

AA6 — **Circuit Breaker:** Prevents repeated calls to a failing service.
**States:** Closed (normal), Open (failures detected, requests blocked), Half-Open (testing if service recovered).
**Retry vs. Breaker:** Retry attempts to recover immediately; Breaker stops attempts to prevent cascading failure.
**Together:** Use Retry for transient network glitches; use Circuit Breaker when the service is persistently down to save resources.

AA7 — **Tenant Isolation Models:**
1.  **Shared DB/Schema:** Lowest cost, highest noisy-neighbor risk, complex security (row-level).
2.  **Shared DB/Schema:** Medium cost, better isolation (schema level), harder migrations.
3.  **Separate DB:** Highest cost, best isolation/security, highest operational complexity.
**Trade-off:** Choose based on compliance needs and budget. High compliance = Separate DB.

AA8 — **Consistency:** Strong consistency ensures immediate visibility of updates. Eventual consistency allows temporary divergence.
**Inventory Architecture:** Use **Strong Consistency**.
**Data Store:** ACID-compliant relational DB (e.g., PostgreSQL) or distributed DB with consensus (e.g., CockroachDB).
**Patterns:** Pessimistic locking (row-level) or Optimistic locking (version numbers) to prevent overselling.

AA9 — **Sidecar Proxy:** A container deployed alongside the main application container to handle network traffic.
**Envoy/Istio:** Envoy handles the data plane (traffic routing, mTLS). Control plane (Istiod) configures Envoy.
**mTLS:** Application TLS encrypts data but doesn't verify identity between services. mTLS verifies identity and encrypts traffic between sidecars, securing service-to-service communication without code changes.

AA10 — **Strangler Fig:** Incrementally migrate functionality from a legacy system to new services.
**Strategy:** Deploy a Gateway in front of the .NET monolith. Route specific API paths (e.g., `/api/orders`) to new .NET 8 services. Gradually move more endpoints. Once all traffic is routed to new services, decommission the monolith.

AA11 — **12-Factor App:**
1.  **Codebase:** One repo, many deploys.
2.  **Dependencies:** Explicitly declared.
3.  **Config:** Stored in environment.
4.  **Backing Services:** Treated as attached resources.
5.  **Build, Release, Run:** Strictly separated.
6.  **Processes:** Stateless, state stored in backing services.
7.  **Port Binding:** Expose via port.
8.  **Concurrency:** Scale via processes.
9.  **Disposability:** Fast startup/shutdown.
10. **Dev/Prod Parity:** Minimize differences.
11. **Logs:** Stream as event streams.
12. **Admin Processes:** Run as one-off commands.
**Why:** Ensures portability, scalability, and maintainability in cloud environments.

AA12 — **Event-Driven (Kafka) vs. Request-Driven (REST):**
*   **Coupling:** Event-driven is loosely coupled; Request-driven is tightly coupled.
*   **Scalability:** Event-driven scales better for high throughput; Request-driven is simpler for low volume.
*   **Debugging:** Request-driven is easier (traceable request); Event-driven is harder (distributed state).
*   **Ordering:** Kafka guarantees ordering per partition; REST does not.
*   **Failure:** Event-driven handles failures asynchronously; Request-driven fails immediately.
**Choice:** Use Event-driven for decoupling and high volume; Request-driven for real-time user interactions.

AA13 — **Load Balancer:** Distributes traffic across multiple servers.
**L4 vs. L7:** L4 (Transport) routes based on IP/Port. L7 (Application) routes based on HTTP headers/URL.
**Use Case:** L7 is required for SSL termination or routing `/api` to one service and `/web` to another on the same IP.

AA14 — **Message Queue:** Stores messages for delivery.
**MQ (RabbitMQ):** Task-oriented, supports complex routing, acks, and dead-letter queues. Good for job queues.
**Log-based (Kafka):** Stream-oriented, high throughput, retains messages for replay. Good for event streaming.
**Pick:** RabbitMQ for task execution; Kafka for event logging/streaming.

AA15 — **Container vs. VM:** A container shares the host OS kernel but isolates processes. A VM virtualizes the hardware and runs a full OS.
**Popularity:** Containers are lighter, start faster, and have lower overhead, making them ideal for microservices scaling.

AA16 — **Stateless vs. Stateful:** Stateless services store no session data (e.g., Web Server). Stateful services store data (e.g., Database).
**Scaling:** Stateless is easier to scale horizontally because any instance can handle any request without data synchronization.

AA17 — **ORM:** Maps database tables to objects in code.
**Examples:** Entity Framework (.NET), Hibernate (Java).
**Trade-offs:** ORMs increase development speed but can generate inefficient SQL and hide performance bottlenecks compared to raw SQL.

AA18 — **Bulkhead Pattern:** Isolates resources (threads, connections) for specific services.
**Relation:** Prevents one failing service from exhausting all resources (e.g., thread pool exhaustion).
**Implementation:** Use separate thread pools for each downstream service call, or deploy services on separate node groups in Kubernetes.

AA19 — **OT vs. CRDT:** OT requires a central server to resolve conflicts sequentially. CRDTs allow clients to merge changes mathematically without a server.
**Choice:** **CRDTs**.
**Why:** CRDTs guarantee convergence even with offline editing and network partitions, whereas OT requires constant server connectivity to resolve conflicts.

AA20 — **Distributed Tracing Challenges:**
*   **Propagation:** Inject TraceID/SpanID into headers (W3C Trace Context).
*   **Sampling:** Head-based (decide early, saves cost); Tail-based (decide after, better data).
*   **Storage:** Jaeger/Tempo store spans; querying requires indexing.
*   **Kafka:** Trace context must be injected into message headers. Consumers must extract and continue the trace.
*   **Continuity:** Ensure all services (including async consumers) propagate the context ID.

OP1 — **Hypervisor Type 1:** Bare-metal (runs directly on hardware). Examples: VMware ESXi, Microsoft Hyper-V Server.
**Hypervisor Type 2:** Hosted (runs on an OS). Examples: VirtualBox, VMware Workstation.

OP2 — **SAN:** Storage Area Network (Block storage). Protocol: iSCSI, Fibre Channel. Use for databases.
**NAS:** Network Attached Storage (File storage). Protocol: NFS, SMB/CIFS. Use for file shares.

OP3 — **3-Tier:** Core (Backbone), Distribution (Policy), Access (Edge).
**Leaf-Spine:** Preferred because it provides non-blocking bandwidth and predictable latency.
**Problem Solved:** 3-tier suffers from oversubscription and high latency between tiers; Leaf-Spine provides equal hop count and full bisection bandwidth.

OP4 — **Proxmox HA:**
*   **Nodes:** Minimum 3 for quorum.
*   **Quorum:** Majority vote required to make changes.
*   **Fencing:** STONITH (e.g., DRAC/iLO) to power off failed nodes.
*   **Storage:** Ceph (distributed) or Shared iSCSI/NFS.
*   **Failure:** VMs restart on surviving nodes automatically.

OP5 — **VLAN Trunking:** Carries multiple VLANs over one link using 802.1Q tags.
**Linux Config:** Create sub-interfaces (e.g., `eth0.10`, `eth0.20`) with `ip link add link eth0 name eth0.10 type vlan id 10`. Assign IPs to sub-interfaces.

OP6 — **RAID:**
*   **RAID 1:** Mirroring (50% capacity).
*   **RAID 5:** Striping + Parity (N-1 capacity).
*   **RAID 6:** Striping + Double Parity (N-2 capacity).
*   **RAID 10:** Mirroring + Striping (50% capacity, high perf).
*   **RAID Z2:** ZFS double parity.
**Choice:** **RAID 10**.
**Why:** For a database, performance and fault tolerance are critical. RAID 10 offers the best write performance and can survive 2 drive failures (if mirrored pairs fail differently).

OP7 — **DNS Architecture:**
*   **Internal:** AD DNS servers (Active Directory integrated).
*   **External:** Public authoritative servers (e.g., Route53).
*   **Split-Horizon:** Internal clients resolve internal IPs; External resolve public IPs.
*   **Forwarders:** Internal DNS forwards external queries to public resolvers.
*   **Laptop:** Connects via DHCP, gets internal DNS. Resolves internal names locally, external via forwarder.

OP8 — **PKI Chain:** Root CA (Offline, Trust Anchor) -> Subordinate CA (Online, Issues Certs) -> End Entity.
**CRL/OCSP:** Check revocation status.
**Compromise:** Subordinate CA = Revoke certs issued by it. Root CA = Trust chain broken globally, requires rebuilding entire PKI.

OP9 — **WAN Consolidation:**
*   **MPLS:** High cost, high reliability, low latency.
*   **IPsec:** Low cost, variable reliability.
*   **SD-WAN:** Smart routing, cost-effective, manages QoS.
*   **WireGuard:** Simple, encrypted, mesh.
**Recommendation:** **SD-WAN**.
**Why:** Best balance of cost, management, and QoS for VoIP (latency-sensitive) across 5 sites.

OP10 — **iDRAC/iLO/IPMI:** Out-of-band management interfaces for hardware control (power, console).
**Critical:** Allows recovery even if OS is down.
**Exposure:** If exposed to internet, attackers can exploit firmware vulnerabilities to gain root access to the server.

OP11 — **LVM vs. ZFS:**
*   **LVM:** Volume management on top of block devices.
*   **LVM-Thin:** Overprovisions storage (thin provisioning).
*   **ZFS:** Filesystem + Volume manager with checksumming.
*   **Snapshots:** LVM (Cow, fast), ZFS (Cow, instant, immutable).
**Use:** ZFS for data integrity; LVM for simple volume management.

OP12 — **Stretch Cluster:**
*   **Design:** 2 Sites + 1 Witness Node (Tiebreaker).
*   **Ceph:** CRUSH rules must distribute OSDs across sites.
*   **Latency:** < 5ms recommended for Ceph.
*   **Split-Brain:** Without a witness, both sites might think they are primary.
*   **Bad Idea:** 2-site Ceph without witness risks data corruption during network partition.

OP13 — **DHCP:** Dynamic Host Configuration Protocol.
**DORA:** Discover, Offer, Request, Acknowledge.
**Unavailable:** Client uses APIPA (169.254.x.x) if lease expires and server is unreachable.

OP14 — **Managed vs. Unmanaged:**
*   **Unmanaged:** Plug-and-play.
*   **Managed:** Configurable.
*   **Features:** VLANs, QoS, Port Security, SNMP monitoring.
*   **Importance:** Essential for segmentation and traffic prioritization in enterprise.

OP15 — **NTP:** Network Time Protocol.
**Critical:** Certificates, logs, and database transactions rely on accurate time.
**Drift:** Cluster consensus fails, logs become uncorrelatable, Kerberos authentication fails.

OP16 — **Physical vs. VM vs. Container:**
*   **Physical:** Hardware only.
*   **VM:** Virtual OS on hardware.
*   **Container:** Isolated process on host OS.
*   **Bare Metal:** Use for high-performance computing, low-latency databases, or hardware passthrough needs.

OP17 — **UPS vs. Generator:**
*   **UPS:** Short-term power (minutes) to allow graceful shutdown or bridge to generator.
*   **Generator:** Long-term power (hours/days).
*   **ATS:** Automatically switches power source from utility to generator when utility fails.

OP18 — **Active-Passive vs. Active-Active:**
*   **Active-Passive:** One node handles traffic; standby takes over on failure.
*   **Active-Active:** Both handle traffic.
*   **PostgreSQL HA:** Pacemaker/Corosync cluster.
*   **Fencing:** STONITH ensures failed node is powered off to prevent split-brain.
*   **Failover:** VIP moves to new master.

OP19 — **Backup Strategy:**
*   **3-2-1:** 3 copies, 2 media types, 1 offsite.
*   **RPO/RTO:** Recovery Point/Time Objectives.
*   **Types:** Full, Incremental, Differential.
*   **Immutable:** Prevents ransomware deletion.
*   **Test:** Regular restore drills.
*   **Backup vs. DR:** Backup is data copy; DR is the plan/process to restore operations.

OP20 — **802.1X:**
*   **Roles:** Supplicant (Client), Authenticator (Switch), Server (RADIUS).
*   **EAP:** EAP-TLS (Certificate-based, secure); PEAP (Username/Password).
*   **Dynamic VLAN:** Switch assigns VLAN based on RADIUS response.
*   **Failure:** Port blocked or placed in Guest VLAN.
*   **MAB:** Falls back to authenticating device MAC address if 802.1X fails (less secure).
