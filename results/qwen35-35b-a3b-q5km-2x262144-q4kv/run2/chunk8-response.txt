CL1 — **IaaS (Infrastructure as a Service):** Provides virtualized computing resources (servers, storage, networking). *Example:* AWS EC2. **PaaS (Platform as a Service):** Provides hardware and software tools (runtime, middleware) for development. *Example:* Google App Engine. **SaaS (Software as a Service):** Delivers software applications over the internet. *Example:* Microsoft Office 365.

CL2 — **Shared Responsibility Model:** Provider secures the cloud (hardware, facilities); Customer secures what is in the cloud. **IaaS VM:** Provider secures hypervisor/physical host; Customer secures OS, patching, data, firewall. **Managed Kubernetes:** Provider secures control plane/etcd; Customer secures nodes, workloads, network policies. **SaaS (M365):** Provider secures app/infra; Customer secures identity, data classification, access management.

CL3 — **VPC:** Isolated virtual network. **Subnet:** Range of IP addresses within a VPC. **Security Group:** Stateful firewall at the instance level (allow rules only). **NACL:** Stateless firewall at the subnet level (allow/deny rules). **Layering:** VPC contains Subnets; Subnets contain Instances; SG protects Instance; NACL protects Subnet. **Stateful:** SG remembers return traffic. **Stateless:** NACL requires explicit rules for inbound and outbound.

CL4 — **Landing Zone:** Secure, compliant baseline environment. **OUs:** Logical grouping for policy application. **Guardrails:** Preventative controls (e.g., "no public S3 buckets"). **Shared Networking:** Transit Gateway (AWS) or Hub-and-Spoke (Azure) for centralized connectivity. **Centralized Logging:** Aggregated logs for security monitoring. **Single-Account Risks:** High blast radius, billing confusion, inability to isolate environments, policy enforcement difficulties.

CL5 — **IaC:** Managing infrastructure via code. **Terraform/OpenTofu:** Multi-cloud, declarative, state file management. **Pulumi:** Multi-cloud, imperative (general purpose languages). **CloudFormation/Bicep:** Native to AWS/Azure, tightly integrated. **Crossplane:** IaC via Kubernetes API. **Choice:** Terraform for multi-cloud; Bicep/CFN for native depth; Pulumi for complex logic; Crossplane for K8s-centric workflows.

CL6 — **HPA:** Scales pods based on CPU/Memory. **VPA:** Adjusts resource requests/limits per pod. **Cluster Autoscaler:** Adds/removes nodes based on pod scheduling. **KEDA:** Scales based on external events (e.g., queue length). **Use KEDA:** When scaling needs to be event-driven (e.g., Kafka messages, SQS queue depth) rather than resource metrics.

CL7 — **DR Strategy:** **RPO/RTO:** Define data loss tolerance and downtime. **Active-Active:** Both regions serve traffic (low RTO/RPO, high cost). **Active-Passive:** One active, one standby (higher RTO, lower cost). **DB:** Sync (Active-Active) vs Async (Active-Passive). **DNS:** Route53 Failover. **State:** Shared storage or replicated state. **Pilot Light:** Minimal infra running (warm). **Warm Standby:** Scaled-down version running. **Multi-site:** Full production in both regions.

CL8 — **GitOps:** Git as single source of truth for infra/app state. **ArgoCD vs Flux:** ArgoCD (UI, push/pull), Flux (Git-native, pull). **Secrets:** Sealed Secrets (encrypted in Git), External Secrets Operator (fetch from Vault/AWS Secrets Manager). **Drift:** Reconciliation loop detects manual changes and reverts them to match Git state.

CL9 — **Architecture:** **StatefulSet:** Stable network IDs, ordered deployment. **PV/PVC:** Local storage class (hostPath) for low latency. **Headless Service:** DNS for StatefulSet pods. **Init Containers:** Prepare filesystem/config before app starts. **ConfigMap:** Volume mounts for config files. **PDB:** Ensure minimum available pods during disruptions.

CL10 — **SA:** K8s identity for pods. **Workload Identity:** GCP/Azure native identity binding. **IRSA:** AWS IAM role linked to K8s SA via OIDC. **Mounting Creds:** Bad practice (leakage, rotation difficulty). **Federation:** Uses short-lived OIDC tokens instead of long-lived keys, reducing exposure.

CL11 — **Object (S3):** Unstructured data, high durability (11 9s), low latency. **Block (EBS):** Attached storage, OS/DB, high IOPS, lower durability than object. **File (EFS):** Shared file system, NFS/SMB, moderate IOPS. **Use Object:** Backups, media. **Use Block:** Databases, OS. **Use File:** Shared configs, legacy apps.

CL12 — **Zero Trust:** Verify every request. **Network Policies:** L3/L4 isolation. **Service Mesh:** mTLS for L7 encryption. **Pod Identity:** Least privilege access. **Admission (OPA):** Enforce policies on deploy. **Image Signing:** Verify integrity. **Runtime (Falco):** Detect anomalies. **Diff:** Perimeter assumes internal trust; Zero Trust assumes breach and verifies continuously.

CL13 — **CDN:** Distributed network caching content at edge. **Cloud Services:** AWS CloudFront, Azure CDN. **Use CDN:** Static assets, global latency reduction, DDoS protection. **Use Origin:** Dynamic content, unique user data, real-time processing.

CL14 — **Region:** Geographic area with multiple AZs. **AZ:** Isolated data center with independent power/cooling. **Multi-AZ:** Protects against single data center failure. **Zonal:** Resource lives in one AZ (e.g., EBS). **Regional:** Resource spans AZs (e.g., ELB, RDS Multi-AZ).

CL15 — **Managed DB:** RDS, Cloud SQL. **Provider Handles:** Patching, backups, failover, hardware provisioning, scaling. **User Handles:** Schema, queries, data, access control.

CL16 — **IAM:** Users (people), Groups (collections), Roles (temporary access), Policies (permissions). **Least Privilege:** Grant only necessary permissions. **Apply:** Use Roles for workloads, Groups for humans, deny-by-default policies.

CL17 — **Serverless:** Event-driven, no server management. **Functions:** Lambda, Azure Functions, GCF. **Cold Starts:** Latency during first invocation after inactivity. **Poor Fit:** Long-running processes, stateful apps, predictable high-throughput (costlier than provisioned).

CL18 — **Ingress:** L7 routing rules. **Gateway API:** Enhanced extensibility, multi-cluster support. **LB:** L4 load balancing. **Mesh:** L7 service-to-service. **TLS:** LB (Edge), Ingress Controller (App), Mesh (mTLS). **Use Gateway API:** When needing advanced routing, multiple ingress controllers, or standardization beyond Ingress CRD.

CL19 — **Strategy:** **RI/SP:** Commit for 1-3 years (30-60% savings). **Spot:** Interruptible (up to 90% savings). **Right-sizing:** Match instance to usage. **Tiering:** Move old data to Glacier. **FinOps:** Culture of cost accountability. **Tools:** Cost Explorer, CloudHealth. **Showback:** Report costs to teams; **Chargeback:** Bill teams directly.

CL20 — **Strategy:** **Abstraction:** Kubernetes, Terraform. **Networking:** Direct Connect/ExpressRoute. **Identity:** OIDC federation. **Trade-offs:** Multi-cloud increases complexity, networking latency, and operational overhead. **Justified:** Regulatory requirements, catastrophic vendor risk, specific regional capabilities. **Unnecessary:** If single cloud meets all needs, multi-cloud adds cost without value.

OT1 — **Purdue Model:** ISA-95 hierarchy. **Level 0:** Physical Process (sensors/actuators). **Level 1:** Basic Control (PLCs). **Level 2:** Supervisory Control (SCADA/HMI). **Level 3:** Operations (MES/Production). **Level 4:** Enterprise (ERP/Business).

OT2 — **IT vs OT:** IT focuses on data confidentiality/integrity; OT focuses on safety/availability/physical process. **Patching:** OT systems often run legacy OS, require vendor approval, and downtime causes physical risk/production loss.

OT3 — **DMZ:** Buffer zone between IT and OT. **Traffic:** Historian pulls from SCADA (OT to DMZ); Engineering pushes configs (DMZ to OT). **Direct L4 to L1:** Dangerous because it exposes control logic to IT threats, bypassing safety controls, allowing direct manipulation of physical processes.

OT4 — **Protocols:** 1. **Modbus TCP:** L1/2, simple polling. 2. **OPC UA:** L2/3, secure, platform-independent. 3. **EtherNet/IP:** L1/2, CIP protocol. 4. **PROFINET:** L1/2, real-time industrial Ethernet. 5. **MQTT:** L3/4, lightweight pub/sub for cloud.

OT5 — **Data Diode:** Hardware enforcing one-way data flow. **Firewall:** Two-way filtering. **Use Diode:** When exporting data from secure OT to IT is required but no feedback is needed. **Limitation:** Cannot send commands back to OT, limiting control capabilities.

OT6 — **Flow:** PLCs -> Historian -> OPC UA Server -> Edge Gateway (Protocol conversion) -> MQTT Broker -> Cloud Ingestion. **Boundaries:** Firewall between Edge and OT network; DMZ between Edge and IT/Cloud.

OT7 — **Zones:** Logical grouping of assets with similar security requirements. **Conduits:** Controlled paths between zones. **Model:** Zone 4 (Corporate), Zone 3 (Engineering/SCADA), Zone 2 (Control/PLC), Zone 1/0 (Field). **Conduits:** Firewalls between zones enforce traffic rules (e.g., Zone 3 to Zone 2 only allows specific OPC UA ports).

OT8 — **Segmentation:** VLANs per production line. **Firewalls:** Between VLANs and DMZ. **Switches:** Managed Layer 2/3. **Remote Access:** Jump host in DMZ with MFA. **Transition:** Phased approach, parallel run, maintenance windows, backup before changes.

OT9 — **SIS vs BPCS:** SIS (Safety Instrumented System) is independent safety layer; BPCS (Basic Process Control System) manages normal operations. **IEC 61511:** Safety lifecycle standard. **Air-gap:** Prevents cyber attacks from disabling safety systems. **Ransomware on SIS:** Safety systems fail to trip, leading to potential physical damage, injury, or environmental release.

OT10 — **PLC:** Logic controller for specific machines. **DCS:** Integrated control for continuous processes. **SCADA:** Supervisory monitoring over wide area. **Coexist:** Yes, PLCs often feed SCADA/DCS systems.

OT11 — **Remote Access:** Jump host in DMZ, MFA, Session Recording, Time-limited tokens. **Direct RDP:** Unacceptable because it exposes HMI directly to corporate network threats, lacks audit trails, and bypasses security controls.

OT12 — **Response:** Isolate compromised segment immediately. **Monitoring:** IDS for anomalies. **Inventory:** Identify affected assets. **Ops:** Coordinate with control room to maintain safe physical state. **Operator Role:** Monitor physical parameters, execute manual overrides if needed, ensure safety systems remain active.

OT13 — **HMI:** Operator interface for monitoring and controlling processes. **Diff:** Engineering Workstation is for programming/configuring logic; HMI is for real-time operation.

OT14 — **Historian:** Time-series database for process data. **Importance:** Trend analysis, compliance, troubleshooting. **Plant (L3):** Real-time process data. **Enterprise (L4):** Aggregated data for business analytics.

OT15 — **Ladder Logic:** Graphical programming language mimicking relay circuits. **Device:** PLC. **Dominant:** Familiarity, deterministic execution, safety certification, ease of maintenance for electricians.

OT16 — **RTU:** Remote Terminal Unit. **Diff:** RTU focuses on comms and harsh environments; PLC focuses on logic. **RTU Use:** Pipelines, power grids, remote sites with limited connectivity.

OT17 — **Analog:** Continuous signal (voltage/current). **Digital:** Discrete (on/off). **4-20mA:** Preferred because 4mA represents zero signal (detects wire break at 0mA).

OT18 — **Inventory:** Knowing all assets is critical for patching and threat detection. **Tools:** Passive (IDS/NetFlow), Active (Scanners). **Active Danger:** Scanners can crash legacy devices or trigger safety interlocks.

OT19 — **MITRE ICS:** Framework for ICS threats. **Tactics:** Initial Access (Phishing), Execution (Command Line), Persistence (Scheduled Task), Privilege Escalation (Exploit), Defense Evasion (Obfuscation). **Diff:** Focuses on physical impact and industrial protocols. **Use:** Map defenses to tactics to find gaps.

OT20 — **SOC:** SIEM integration, OT-specific rules (e.g., PLC write attempts). **Staff:** OT engineers for context. **False Positives:** Tuned to avoid production stoppages. **Firmware Alert:** Verify change ticket, isolate device, check physical safety, rollback if unauthorized, investigate source.
