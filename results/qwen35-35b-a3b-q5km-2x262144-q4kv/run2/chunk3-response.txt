T1 — Easy: `tofu plan` calculates the execution plan (what will change) without making modifications, allowing you to review the diff. `tofu apply` executes the plan, creating, updating, or destroying resources to match the configuration.

T2 — Easy: State is a JSON file tracking the mapping between your configuration and real-world resources. Remote state is important because it enables team collaboration, prevents race conditions via locking, and secures sensitive data (like credentials) out of local repositories.

T3 — Medium: `count` uses integer indices to create multiple instances of a resource. `for_each` uses a map or set of strings to create instances with unique keys. `for_each` is preferred when resource names are dynamic or non-sequential, as it avoids index shifting issues when items are added or removed.

T4 — Medium: This is called "drift." OpenTofu detects this during `tofu plan` by comparing the current state with the live infrastructure. It will show the resource as needing to be updated to match the configuration. If the change is manual and not in config, it will show as a change to be reverted.

T5 — Medium:
```hcl
resource "aws_instance" "web" {
  for_each = toset(["web1", "web2", "web3"])
  ami      = "ami-12345678"
  instance_type = "t2.micro"
  tags = {
    Name = each.key
  }
}
```

T6 — Hard: `terraform_remote_state` reads outputs from a separate state file, creating coupling between states. `output` values are exposed within the same state. The recommended alternative is to use direct module composition (calling modules within the same state) or pass outputs via CI/CD variables between runs to reduce state file coupling and improve security.

T7 — Hard: The `null_resource` provisioner runs during apply, but the output might be evaluated before the file is fully written or the resource is ready. Dependent resources fail because the output is empty. Fix: Add `depends_on = [null_resource.kubeconfig]` to the dependent resources to ensure the `null_resource` completes before they attempt to read the output.

T8 — Medium: `tofu import` attaches existing infrastructure to state manually. Limitations: No history, requires manual mapping of attributes. The `import` block (1.5+) allows declarative imports in code, making them version-controlled, testable, and easier to manage in CI/CD pipelines.

T9 — Easy: A variable is an input parameter passed into the configuration (e.g., from CLI or env). An output is a value exposed from the configuration to be used by other modules or the CLI. Variables are inputs; outputs are results.

T10 — Easy: A provider is a plugin that interacts with a specific API (e.g., AWS, Azure). Running `tofu plan` without `tofu init` fails because the provider binaries are not downloaded, and the configuration cannot be validated.

T11 — Easy: The `.terraform.lock.hcl` file records the exact versions of providers and plugins used. It should be committed to version control to ensure reproducible builds and prevent unexpected provider updates from breaking infrastructure.

T12 — Easy: `tofu destroy` removes all resources defined in the configuration. To target a single resource, use the flag `-target=resource_type.resource_name` (e.g., `tofu destroy -target=aws_instance.web`).

T13 — Easy: A data source reads information from an external system without managing its lifecycle. Use one when you need to look up existing data (e.g., `data "aws_ami" "ubuntu"`) rather than creating new resources.

T14 — Medium: `variable` blocks define inputs from outside the module. `locals` define computed values within the module configuration. Use `locals` for values derived from variables or other locals that shouldn't be exposed as inputs.

T15 — Medium: The `lifecycle` block controls resource behavior. `create_before_destroy` creates the new resource before destroying the old (zero downtime). `prevent_destroy` prevents accidental deletion. `ignore_changes` ignores specific attribute drifts (e.g., tags modified manually).

T16 — Hard: The `moved` block tracks resource identity changes during refactoring (e.g., renaming a resource or moving it to a module). It updates the state file to map the old address to the new address without destroying and recreating the resource.

T17 — Hard: The dependency graph is a DAG (Directed Acyclic Graph) determining execution order. Implicit dependencies are inferred from references (e.g., `aws_instance.web.id`). Explicit dependencies use `depends_on`. `depends_on` reduces parallelism because it forces sequential execution, slowing down apply times.

T18 — Hard: 1. `terraform plan -refresh=false` (faster, risks stale state). 2. Increase `parallelism` (faster, higher resource usage). 3. Split monolithic state into modules (faster, more complex management). 4. Use a remote state backend with caching (faster, requires setup).

T19 — Hard: State locking prevents concurrent writes. If a lock isn't released (e.g., crash), subsequent applies fail. Manually break it with `terraform force-unlock <LOCK_ID>`. Risk: Data corruption if two processes were actually writing simultaneously.

T20 — Hard: 1. Update `backend` block in config to S3/DynamoDB. 2. Run `terraform init -migrate-state -backend-config="bucket=..."`. 3. This moves the local state file to S3. 4. Verify with `terraform state list`. 5. Run `terraform plan` to ensure state matches config.

A1 — Easy: A **Playbook** is a YAML file defining a sequence of tasks. A **Role** is a structured directory for organizing tasks, handlers, and variables. A **Task** is a single action (e.g., install a package) within a playbook or role.

A2 — Easy: `ansible_facts` is a dictionary of system information gathered by the `setup` module. Access OS family via `{{ ansible_facts['os_family'] }}`.

A3 — Medium: `command` runs without a shell (no pipes/globs). `shell` runs with a shell (supports pipes/globs). `raw` runs without SSH setup (no Python interpreter), useful for initial bootstrap. Use `command` for security, `shell` for scripting, `raw` for bare metal setup.

A4 — Medium: Idempotency means running a task multiple times yields the same result. Example (Idempotent): `apt: name=nginx state=present`. Example (Not Idempotent): `shell: rm -rf /tmp/data` (deletes every time).

A5 — Medium:
```yaml
- name: Install htop
  package:
    name:
      - htop
      - "{{ 'epel-release' if ansible_os_family == 'RedHat' else [] }}"
    state: present
```

A6 — Hard: Priority (Low to High): Default vars → Host vars → Group vars → Role vars → Play vars → Extra vars (`-e`) → Command line args. `set_fact` overrides most but is runtime-specific.

A7 — Hard: Use `serial: 3` in the play to process 3 nodes at a time. Group nodes in inventory (e.g., `[rke2_nodes]`). Use `meta: wait_for` or `wait_for` module to ensure the first 3 nodes are ready before the next batch starts.

A8 — Medium: `ansible-vault` encrypts sensitive data. To encrypt a single variable: `ansible-vault encrypt_string 'secret_value' --name 'my_secret'`. This creates a vaulted string in the vars file.

A9 — Easy: An inventory file lists managed hosts. **Static** inventory is a manual file (INI/YAML). **Dynamic** inventory is a script or API call that generates the list dynamically (e.g., from cloud metadata).

A10 — Easy: `ansible webservers -m shell -a "df -h"`.

A11 — Easy: A handler is a task triggered by `notify` when another task reports a change. It runs after all tasks in the play complete. Regular tasks run immediately; handlers run only if notified and at the end.

A12 — Easy: `ansible.cfg` configures Ansible behavior. Common settings: `inventory` (path), `remote_user` (SSH user), `forks` (parallel connections), `host_key_checking` (SSH warnings).

A13 — Easy: `register` saves a task's output to a variable. Use it in subsequent tasks via `when: my_var.stdout == 'success'` or `debug: var=my_var`.

A14 — Medium: `import_tasks` is static (evaluated at parse time, no conditionals). `include_tasks` is dynamic (evaluated at runtime, supports conditionals). Use `import` for performance, `include` for dynamic logic.

A15 — Medium: Ansible Galaxy is a repository for roles and collections. Use `requirements.yml` to list dependencies. Run `ansible-galaxy install -r requirements.yml` to install them.

A16 — Medium: Connection plugins define how Ansible talks to hosts. `ssh` (standard remote), `local` (control node), `network_cli` (network devices via CLI). Use `ssh` for servers, `local` for localhost tasks, `network_cli` for routers/switches.

A17 — Hard: `delegate_to: localhost` runs the task on the control node. `run_once: true` ensures it runs once even in a loop. Example: Gather facts from remote, then write a config file to the control node for processing.

A18 — Hard: 1. Increase `forks` (more parallelism, more RAM). 2. Enable `pipelining` (reduces SSH connections, faster). 3. Use `fact_caching` (avoids re-gathering facts). 4. Use `Mitogen` (replaces SSH, faster). Tradeoffs: Resource usage vs. speed.

A19 — Hard: `{{ }}` outputs variables. `{% %}` executes logic (loops/if). `{# #}` comments.
Snippet:
```yaml
upstream backend {
{% for server in backend_servers %}
    server {{ server }}; # Health check: {{ server_health_check }}
{% endfor %}
}
```

A20 — Hard: 1. Set `serial: 5` (batches). 2. Pre-task: Deregister from LB. 3. Deploy. 4. Post-task: Health check via `uri` module. 5. `block/rescue`: If health check fails, `rescue` triggers rollback (re-register old version). 6. `max_fail_percentage: 10` to stop if too many fail.
