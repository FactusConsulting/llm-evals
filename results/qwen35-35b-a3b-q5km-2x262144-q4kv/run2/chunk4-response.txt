## Go (20 questions)

**G1 — Easy:** An array is a fixed-size value type (`[N]T`), while a slice is a dynamic view over an array (`[]T`) containing a pointer, length, and capacity. When appending to a slice at capacity, the Go runtime allocates a new, larger underlying array (usually doubling capacity), copies existing elements, and returns a new slice header pointing to it.

**G2:** `var x int` declares a variable with explicit type and zero value (`0`), usable anywhere. `x := 0` is a short variable declaration that infers type and is only valid inside functions. You cannot use `:=` at package level (outside functions) or to redeclare an existing variable in the same scope.

**G3:** A goroutine is a lightweight thread managed by the Go runtime, not the OS. It uses less memory and context-switching overhead than OS threads. The Go scheduler uses the GMP model: **G** (Goroutine), **M** (OS Thread), **P** (Processor, logical CPU context). The scheduler maps Gs to Ps, which run on Ms, enabling efficient concurrency.

**G4:** Channels are typed conduits for communication between goroutines. Unbuffered channels block both sender and receiver until the other side is ready. Buffered channels have a fixed size; sending blocks only if the buffer is full. Receiving blocks if the buffer is empty.

**G5:** The `context` package manages deadlines, cancellation signals, and request-scoped values across API boundaries.
```go
ctx, cancel := context.WithTimeout(context.Background(), 5*time.Second)
defer cancel()
req, _ := http.NewRequestWithContext(ctx, "GET", url, nil)
client.Do(req) // Cancels if timeout exceeded
```

**G6:** Go uses implicit interface satisfaction: a type implements an interface if it implements all methods. Unlike Java/C#, no `implements` keyword is needed. The empty interface `any` (formerly `interface{}`) accepts any type. It is powerful for generic-like flexibility but dangerous because it bypasses type safety, requiring type assertions.

**G7:** This is unsafe due to a data race. The main goroutine reads `data` without synchronization after `ready` becomes true. The Go memory model requires synchronization (like a mutex or channel) to guarantee visibility of writes across goroutines.
*Fix:* Use a `sync.Mutex` to protect `data` and `ready`, or use a channel to signal readiness and pass data.

**G8:** `sync.Mutex` allows exclusive access (one writer or one reader). `sync.RWMutex` allows multiple concurrent readers but exclusive writers. `sync.Map` is optimized for high-read/low-write contention or when keys are known at compile time. Regular maps with mutexes are generally faster for mixed workloads due to better cache locality and lower locking overhead.

**G9:** Go modules manage dependencies. `go.mod` lists direct dependencies and the Go version. `go.sum` contains cryptographic checksums of downloaded modules. `go mod tidy` removes unused imports and adds missing dependencies. MVS (Minimum Version Selection) ensures the lowest compatible versions of transitive dependencies are selected to avoid conflicts.

**G10:** `defer` schedules a function call to run after the surrounding function returns. Multiple defers execute in LIFO (Last-In, First-Out) order. A common gotcha in loops: the deferred function captures the loop variable by reference (in older Go versions) or value depending on scope, often leading to all deferred calls using the final loop value.

**G11:** A pointer holds a memory address. The zero value of a pointer is `nil`. To create a pointer to a struct literal: `&MyStruct{Field: value}`.

**G12:** `init()` is a function executed automatically before `main()` runs. It runs once per package. A package can have multiple `init()` functions; they execute in the order they appear in the source files, and files within a package are sorted alphabetically.

**G13:** `new(T)` allocates zeroed memory of type `T` and returns a pointer `*T`. `make(T)` initializes built-in types (slice, map, channel) and returns the initialized type (not a pointer). `new` is for any type; `make` is only for built-in reference types.

**G14:** Errors are values implementing the `error` interface (`Error() string`). Idiom: `if err != nil { return err }`. `errors.Is` checks if an error matches a target (unwrapping). `errors.As` checks if an error matches a specific type and assigns it to a variable. `==` only works for exact error values, not wrapped errors.

**G15:**
*   **Map:** Zero value is `nil` (uninitialized).
*   **Slice:** Zero value is `nil` (length 0).
*   **Channel:** Zero value is `nil` (blocks forever).
*   **Struct:** Zero value has all fields set to their respective zero values.

**G16:** A race condition occurs when two goroutines access shared data concurrently, and at least one writes. The `-race` flag instruments code to detect these at runtime.
*Fix:* Use `sync.Mutex` to lock access or channels to pass ownership of data instead of sharing it.

**G17:** `select` waits on multiple channel operations. Unlike `switch` (which evaluates one expression), `select` evaluates all cases; if multiple are ready, one is chosen pseudo-randomly. To implement a timeout: add a `case <-time.After(duration)` to the select block.

**G18:** Escape analysis determines if a variable lives on the stack or heap. If a variable's address escapes the function scope, it is allocated on the heap. Inspect with `go build -gcflags="-m"`. Heap allocation increases GC pressure; stack allocation is faster and zero-cost.

**G19:** `reflect` allows runtime inspection of types and values. `reflect.Type` describes the type; `reflect.Value` holds the value. It is slow due to dynamic dispatch and type checking. Justified for serialization frameworks or generic libraries, but avoided in performance-critical paths where code generation (e.g., `go:generate`) is preferred.

**G20:** `//go:embed` embeds files/directories at compile time into a `string` or `[]byte`. It works with single files, directories (`//go:embed dir/*`), and file systems. Limitations: files must exist at compile time, cannot embed symlinks. It integrates with `io/fs.FS` to serve embedded content as a virtual file system.

## Rust (20 questions)

**R1 — Easy:** Ownership is Rust's core memory safety system. Rules: 1) Each value has one owner. 2) When the owner goes out of scope, the value is dropped. 3) You can move ownership, but not copy (unless `Copy` trait).

**R2:** `String` is an owned, heap-allocated, growable string. `&str` is a borrowed slice of UTF-8 bytes (string slice). Use `String` for owned data; use `&str` for function arguments to avoid unnecessary allocation and copying.

**R3:** Borrowing allows temporary access without ownership. Rules: 1) You can have either one mutable reference OR any number of immutable references. 2) References must always be valid. 3) You cannot mutate through an immutable reference.

**R4:** `Result<T, E>` represents success (`Ok(T)`) or failure (`Err(E)`). `Option<T>` represents a value that may be `Some(T)` or `None`. The `?` operator propagates errors.
```rust
fn read_file() -> Result<String, std::io::Error> {
    std::fs::read_to_string("file.txt")
}
```

**R5:** `Box<T>` owns data on the heap (single owner). `Rc<T>` (Reference Counted) allows multiple owners in a single thread. `Arc<T>` (Atomic Reference Counted) allows multiple owners across threads. Use `Arc` when sharing data between threads.

**R6:** Traits define shared behavior. Unlike Go interfaces, traits can have default implementations. `dyn Trait` creates a trait object for dynamic dispatch (vtable). `T: Trait` uses static dispatch (monomorphization) for generics, which is generally faster.

**R7:** The function fails because the compiler cannot determine which input's lifetime the returned reference belongs to.
*Fix:* Add lifetime annotations: `fn longest<'a>(x: &'a str, y: &'a str) -> &'a str { ... }`.

**R8:** `Send` means a type can be transferred to another thread. `Sync` means `&T` can be shared across threads. `Rc<T>` is not `Send` because its reference count is not atomic (unsafe to share across threads). `Arc<T>` uses atomic reference counting, making it `Send` and `Sync`.

**R9:** `async/await` creates a state machine (`Future`). The executor (e.g., Tokio) polls the future until it completes. Unlike Go's stackful goroutines, Rust's async is stackless (heap-allocated state machine), offering more control but requiring manual executor management.

**R10:** `unsafe` allows operations the compiler cannot verify for safety: 1) Dereferencing raw pointers. 2) Calling unsafe functions. 3) Accessing mutable statics. Justified for FFI, low-level optimizations, or implementing safe abstractions over unsafe primitives.

**R11:** Pattern matching deconstructs values. `match` requires exhaustive handling of all variants. `if let` handles a single pattern and ignores others. The compiler enforces exhaustiveness in `match` to prevent logic errors.

**R12:** `Vec<T>` is a growable heap-allocated array. `[T; N]` is a fixed-size stack-allocated array. When `Vec` capacity is exceeded, it allocates a new larger buffer, copies elements, and drops the old one (amortized O(1) push).

**R13:** Enums can hold data variants (e.g., `enum Color { Red, Blue }`). Unlike C/Java, Rust enums are algebraic data types. `Option<T>` is `enum Option<T> { Some(T), None }`. Rust has no `null`; `None` is a type-safe alternative.

**R14:** `impl` defines methods. `&self` borrows the instance. `&mut self` mutably borrows. `self` consumes the instance (move). Associated functions (e.g., `String::new()`) do not take `self` and act like static methods.

**R15:** `derive` macros automatically implement traits. Common: `Debug`, `Clone`, `PartialEq`. Cannot derive if fields contain types that don't support the trait (e.g., `Clone` on a field without `Clone`).

**R16:** `Cow<'a, T>` (Clone on Write) holds either a borrowed reference or an owned value. It avoids allocation if the data is only read. If mutation is needed, it clones the data internally. Useful for functions returning data that might be borrowed or owned.

**R17:** `mod` defines a module. `use` imports items. `pub` makes items visible outside the crate. File system maps to modules (`mod.rs` or `mod.rs` in folders). Rust 2018 improved path resolution (e.g., `crate::` vs `self::`) and module visibility.

**R18:** `Pin<T>` prevents a value from being moved in memory. Needed for self-referential structs and async `Future`s (which may be polled while pinned). `Unpin` is a marker trait indicating a type can be safely moved; most types are `Unpin`.

**R19:** `macro_rules!` creates declarative macros (pattern matching). Procedural macros (derive, attribute, function-like) are Rust code that generates Rust code. Hygiene ensures macro variables don't leak into the caller's scope, preventing naming collisions.

**R20:** Zero-cost abstractions mean high-level constructs compile to efficient machine code. Generics use monomorphization (code duplication per type) instead of virtual dispatch. Iterators are inlined. This allows Rust to match C performance while providing safety and ergonomics.
