G1 — **Slice vs Array:** An array is a fixed-size value type `[T]N`. A slice is a dynamic view over an array with a pointer, length, and capacity `[]T`. When appending to a slice at capacity, Go allocates a new, larger underlying array, copies elements, and updates the slice header.

G2 — **`var` vs `:=`:** `var x int` declares a variable with explicit type and zero value. `x := 0` declares and initializes using type inference. You cannot use `:=` for redeclaration of an existing variable in the same scope, or outside a function (e.g., at package level).

G3 — **Goroutine vs Thread:** A goroutine is a lightweight thread managed by the Go runtime, not the OS. The Go scheduler (GMP model) maps Goroutines (G) to OS Threads (M) via Processors (P), allowing efficient multiplexing of thousands of goroutines on fewer OS threads.

G4 — **Channels:** Channels enable communication between goroutines. Unbuffered channels block both sender and receiver until the other side is ready. Buffered channels allow sending without blocking until the buffer is full. Sending to a full buffered channel blocks until space is available.

G5 — **Context:** The `context` package manages deadlines, cancellation, and request-scoped values.
```go
ctx, cancel := context.WithTimeout(context.Background(), 5*time.Second)
defer cancel()
req, _ := http.NewRequestWithContext(ctx, "GET", url, nil)
client.Do(req) // Cancels if timeout exceeded
```

G6 — **Interfaces:** Go interfaces are satisfied implicitly (duck typing), unlike Java/C# which require explicit declaration. The empty interface `interface{}` (or `any`) accepts any type. It is powerful for generic containers but dangerous because it requires type assertions, losing compile-time safety.

G7 — **Memory Model:** The code is unsafe due to potential race conditions and lack of memory visibility guarantees (happens-before). The reader might see `ready=true` before `data` is written. Fix: Use `sync/atomic` for `ready` and `data`, or a `sync.Mutex` to protect both.

G8 — **Mutex vs RWMutex:** `Mutex` allows exclusive access. `RWMutex` allows multiple readers or one writer. `sync.Map` is optimized for read-heavy workloads where keys are distinct and static. Trade-off: `sync.Map` has higher overhead for writes and complex lookups compared to a standard `map` with a mutex.

G9 — **Modules:** `go.mod` defines module path and dependencies. `go.sum` stores checksums for integrity. `go mod tidy` removes unused imports and adds missing ones. MVS (Minimum Version Selection) selects the lowest version of a dependency that satisfies all constraints in the dependency graph.

G10 — **Defer:** `defer` schedules a function call until the surrounding function returns. Multiple defers execute in LIFO (Last In, First Out) order. Gotcha: In a loop, the deferred function captures the loop variable by reference, often leading to all defers using the final loop value.

G11 — **Pointers:** A pointer holds a memory address. The zero value is `nil`. Create a pointer to a struct literal using `&Struct{Field: value}`.

G12 — **Init:** `init()` runs automatically before `main()` during package initialization. A package can have multiple `init()` functions; they execute in source file order, then within each file in declaration order.

G13 — **Make vs New:** `new(T)` allocates zeroed memory of type `T` and returns a pointer `*T`. `make(T)` initializes built-in types (slice, map, channel) and returns the initialized value (not a pointer).

G14 — **Error Handling:** Errors are values implementing the `error` interface. Idiom: `if err != nil { return err }`. `errors.Is` checks if an error matches a target in the chain. `errors.As` checks if an error can be assigned to a target variable (type assertion).

G15 — **Composite Types:**
*   `map`: Zero value is `nil`.
*   `slice`: Zero value is `nil`.
*   `channel`: Zero value is `nil`.
*   `struct`: Zero value is all fields set to their zero values.

G16 — **Race Condition:** Occurs when concurrent goroutines access shared data without synchronization. The `-race` flag adds instrumentation to detect data races at runtime. Fix: Use `sync.Mutex` to lock access or channels to pass ownership.

G17 — **Select:** `select` waits on multiple channel operations. Unlike `switch`, it chooses a ready case randomly if multiple are ready. Timeout pattern: `select { case <-time.After(d): ... case <-ch: ... }`.

G18 — **Escape Analysis:** Determines if a variable lives on the stack or heap. If a variable escapes the function scope (e.g., returned), it moves to the heap. Inspect with `go build -gcflags="-m"`. Heap allocation is slower due to GC overhead.

G19 — **Reflection:** `reflect.Type` describes type metadata; `reflect.Value` holds the value. Reflection is slow (runtime type inspection) and bypasses compile-time checks. Justified for frameworks (serialization, dependency injection) where code generation is too complex.

G20 — **Embed:** `//go:embed` includes files at compile time into a `fs.FS`. Works on files, directories, and glob patterns. Limitations: Files must exist at build time. Integrates with `io/fs` to treat embedded content as a virtual filesystem.

R1 — **Ownership:** Rust ensures memory safety without GC. Rules: 1) Each value has one owner. 2) When owner goes out of scope, value is dropped. 3) Values can be moved, but not copied unless `Copy` trait is implemented.

R2 — **String vs &str:** `String` is an owned, heap-allocated, growable buffer. `&str` is a borrowed slice of bytes (often a string literal). Use `String` for owned data, `&str` for function arguments to avoid unnecessary allocation.

R3 — **Borrowing:** You can have many immutable references (`&T`) or one mutable reference (`&mut T`), but not both simultaneously. The borrow checker enforces this at compile time to prevent data races and dangling pointers.

R4 — **Result/Option:** `Result<T, E>` represents success or error. `Option<T>` represents value or `None`. The `?` operator propagates errors.
```rust
fn read_file(path: &str) -> Result<String, std::io::Error> {
    std::fs::read_to_string(path)
}
```

R5 — **Box/Rc/Arc:** `Box<T>` owns data on heap (single owner). `Rc<T>` is reference-counted for single-threaded shared ownership. `Arc<T>` is atomic reference-counted for thread-safe shared ownership.

R6 — **Traits:** Traits define shared behavior. Unlike Go interfaces, traits can have default implementations. `dyn Trait` uses dynamic dispatch (vtable). `T: Trait` uses static dispatch (monomorphization), which is faster.

R7 — **Lifetimes:** The function fails because the compiler cannot determine which input reference the return value refers to. Fix: `fn longest<'a>(x: &'a str, y: &'a str) -> &'a str { if x.len() > y.len() { x } else { y } }`.

R8 — **Send/Sync:** `Send` means a type can be transferred across threads. `Sync` means `&T` can be shared across threads. `Rc<T>` is not `Send` because reference counting is not atomic. `Arc<T>` is `Send` and `Sync` due to atomic counters.

R9 — **Async/Await:** `async` functions return a `Future` (state machine). The executor (e.g., Tokio) polls the future until completion. Unlike Go goroutines (stack-based), Rust futures are heap-allocated state machines, allowing fine-grained control over concurrency.

R10 — **Unsafe:** Allows raw pointer dereferencing, union access, and calling C functions. Justified when interfacing with hardware, optimizing performance-critical paths, or implementing safe abstractions over unsafe primitives.

R11 — **Pattern Matching:** `match` exhaustively checks all possible values. `if let` matches a single pattern. The compiler enforces exhaustiveness in `match`, ensuring no logic is missed for enums.

R12 — **Vec vs Array:** `Vec<T>` is heap-allocated and dynamic. `[T; N]` is stack-allocated and fixed size. `Vec` reallocates (doubling capacity) when elements exceed current capacity, copying data to new memory.

R13 — **Enums:** Rust enums are Algebraic Data Types (ADTs) that can hold data. `Option<T>` is `enum { Some(T), None }`. Rust has no `null`; `None` is a type-safe alternative.

R14 — **Impl Block:** Defines methods for a type. `&self` borrows, `&mut self` mutably borrows, `self` takes ownership. Associated functions (e.g., `String::new()`) do not take `self` and act like static methods.

R15 — **Derive Macro:** Automatically implements traits like `Debug`, `Clone`. Cannot derive traits requiring custom logic or lifetimes (e.g., `PartialEq` on types with lifetimes).

R16 — **Cow:** `Cow<'a, T>` is Clone on Write. It borrows data if possible, cloning only when mutation is needed. Example: Reading a file into a `Cow` avoids allocation if the data is already owned, but clones if you modify it.

R17 — **Modules:** `mod` declares a module, `use` imports paths, `pub` makes items public. File system maps to modules (`mod.rs` or `mod/`). Rust 2018 improved path resolution (e.g., `crate::` vs `self::`).

R18 — **Pin:** `Pin<T>` prevents a value from moving in memory. Needed for self-referential structs and async state machines where internal pointers must remain valid. `Unpin` is the marker trait for types that can be moved safely.

R19 — **Macros:** `macro_rules!` are declarative (pattern matching). Procedural macros (derive, attribute) are Rust code generating Rust code. Hygiene ensures macro variables don't leak into the caller's scope.

R20 — **Zero-Cost Abstractions:** Abstractions like generics and iterators compile down to efficient machine code. Generics use monomorphization (copying code per type). Iterators are inlined. No runtime overhead compared to C.
