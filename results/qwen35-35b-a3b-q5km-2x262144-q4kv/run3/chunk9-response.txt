SC1-A
The pods are failing due to a conflict between the **Restricted Pod Security Standard (PSS)** and the container image configuration.
1.  **Restricted PSS:** Requires `runAsNonRoot: true` AND explicitly forbids running as root (UID 0). It mandates that `runAsUser` be set to a non-zero value.
2.  **Image Change:** The new base image (`cgr.dev/chainguard/static:latest`) defaults `USER` to root (UID 0).
3.  **Manifest:** The pods have `runAsNonRoot: true` but lack `runAsUser`.
4.  **Interaction:** When the pod starts, the kubelet attempts to run the container as root (per image default). The Restricted PSS admission controller rejects this because `runAsNonRoot` is true but the effective UID is 0. Kubernetes marks the pod as `CrashLoopBackOff` because the container is killed immediately upon startup attempt.
**Change:** Baseline allowed root containers if `runAsNonRoot` wasn't explicitly set to true. Restricted enforces non-root strictly.

SC1-C
**Remediation Strategy:**
1.  **Image Fix:** Rebuild the image with `USER 1000` in the Dockerfile or use a `securityContext` override in the image build pipeline.
2.  **Manifest Fix:** Add `runAsUser: 1000` to the `securityContext` of the pod spec.
3.  **CI Gate:** Add a policy check (e.g., OPA Gatekeeper or Checkov) in CI to ensure `runAsUser` is set if `runAsNonRoot` is true, and scan images for root users.
4.  **Rollout:** Use a RollingUpdate strategy with `maxSurge: 1` and `maxUnavailable: 0`. Ensure readiness probes are configured so traffic only shifts to healthy pods.
5.  **Prevention:** Enforce PSS Restricted at the namespace level via Admission Controllers and require signed images.

SC2-A
**Stale Data:** Caused by asynchronous replication lag. When the EU replica is promoted, it becomes the primary. Any WAL transactions not yet shipped from US-East to EU-West are lost or delayed.
**Replication:** PostgreSQL streaming replication ships WAL files asynchronously by default. If the network blips or lag exceeds the promotion threshold, data written in US-East after the last sync point is lost.
**Quantify:** Check `pg_stat_replication` on the old primary (if accessible) for `write_lag` and `flush_lag`. Check `pg_wal_lsn_diff` between the promoted replica and the old primary's last known checkpoint.
**Data Loss:** Any writes committed to the old primary after the last successful WAL sync to the replica are lost.

SC2-B
1.  **Import:** `terraform import aws_db_instance.promoted_replica <new_db_instance_identifier>` (The promoted replica gets a new identifier).
2.  **Update State:** `terraform state mv aws_db_instance.old_primary aws_db_instance.promoted_replica` (If reusing resource name).
3.  **Update Dependencies:** Update Terraform variables referencing the old ARN to the new ARN.
4.  **Apply:** `terraform apply` to sync dependent resources (Security Groups, Route53) to the new instance.
5.  **Destroy Old:** `terraform destroy` on the old instance resource (if it still exists in state) to clean up.

SC2-C
**Architecture:**
1.  **Database:** Use **Aurora Global Database**. It offers synchronous replication across regions with RPO < 1 second and automatic failover.
2.  **Connection:** Use **RDS Proxy** instead of PgBouncer. It handles connection pooling centrally and supports failover transparently.
3.  **DNS:** **Route53 Latency-Based Routing** with Health Checks. If US-East fails, traffic routes to EU-West automatically.
4.  **App Logic:** Implement exponential backoff retry logic with jitter for transient connection errors.
5.  **Recommendation:** Aurora Global DB is superior to standard RDS for this requirement because standard RDS cross-region replication is asynchronous and manual promotion is required, increasing RPO/RTO risk.

SC4-A
1.  **Local Sessions:** Kubernetes pods are ephemeral. **Solution:** Move sessions to Redis.
2.  **Local NFS:** Pods cannot share local filesystems. **Solution:** Use EFS or NFS CSI driver with `ReadWriteMany`.
3.  **Oracle DB:** On-prem Oracle needs network access. **Solution:** Use AWS RDS for Oracle or migrate to Aurora PostgreSQL.
4.  **IBM MQ:** On-prem mainframe integration. **Solution:** Use MQ Cluster or AWS MQ with VPC peering.
5.  **Sticky Sessions:** F5 relies on IP affinity. **Solution:** Use Kubernetes Ingress with sticky sessions or stateless design.

SC6-C
**Comparison:**
1.  **Direct Connect:** High cost, high bandwidth (10Gbps+), low latency. Fits Zero Trust (dedicated link). Requires BGP routing.
2.  **ECMP VPN:** Low cost, limited bandwidth (2x1Gbps), higher latency. Fits Zero Trust (IPsec encryption).
3.  **Mesh (Tailscale):** Low cost, easy setup, high latency. Security model relies on identity, not network.
**Recommendation:** **Direct Connect** for production core traffic to meet bandwidth/latency. Use **Tailscale** for admin access. Update routing to use BGP for DX and static routes for VPN.

SC10-C
**Failover Orchestration:**
1.  **Detection:** Health checks (heartbeat) every 5s. Site down if > 30s no heartbeat.
2.  **DNS:** GSLB (Route53) switches traffic to Dallas.
3.  **DB:** Patroni promotes Dallas node (synchronous commit ensures no data loss).
4.  **Cache/Queue:** Redis Sentinel promotes new master. RabbitMQ federation syncs.
5.  **Storage:** MinIO site-replication catches up.
6.  **State:** Ansible/Terraform updates state to reflect Dallas as primary.
**Split-Brain:** Use etcd quorum. If Chicago is down, Dallas cannot promote unless it has quorum.
**Failback:** Once Chicago recovers, re-establish replication (Chicago becomes standby). Update DNS back to Chicago.
