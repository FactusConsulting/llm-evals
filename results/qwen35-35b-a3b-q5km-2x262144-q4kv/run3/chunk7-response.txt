AA1 — **Monolith vs. Microservices:** A monolith is a single, unified codebase deployed as one unit. Microservices are independent, loosely coupled services deployed separately. **Problems introduced by microservices:** 1. **Network Latency:** Communication is over the network, not in-process calls. 2. **Distributed Tracing:** Debugging requires correlating logs across multiple services. 3. **Data Consistency:** ACID transactions across services are impossible; eventual consistency must be managed.

AA2 — **Sync vs. Async:** Synchronous communication blocks the caller until a response is received (request/response). Asynchronous communication allows the caller to continue without waiting (fire-and-forget or callback). **Examples:** Synchronous: REST/HTTP. Asynchronous: Apache Kafka or RabbitMQ.

AA3 — **API Gateway vs. Reverse Proxy:** A reverse proxy routes traffic (L4/L7) and handles SSL termination. An API Gateway adds business logic (auth, rate limiting, transformation). **Replacement:** An API Gateway cannot entirely replace a reverse proxy in all cases. A dedicated L4 load balancer (e.g., AWS NLB) is often needed before the Gateway for DDoS protection and SSL offloading to reduce Gateway load.

AA4 — **Saga Pattern:** Manages distributed transactions via a sequence of local transactions. **Choreography:** Services emit events; others react. **Orchestration:** A central coordinator tells services what to do. **Appropriateness:** Choreography for simple, decoupled flows. Orchestration for complex workflows requiring compensation logic and visibility.

AA5 — **CQRS:** Separates read (Query) and write (Command) operations into different models. **Paired with Event Sourcing:** Event Sourcing stores state changes as an immutable log. **Problem Solved:** Allows independent scaling of read/write workloads and provides a full audit trail/time-travel capability, which traditional CRUD (single table per entity) does not.

AA6 — **Circuit Breaker:** Prevents repeated calls to a failing service. **States:** Closed (normal), Open (fail fast), Half-Open (test recovery). **Difference from Retry:** Retry attempts to fix the issue; Breaker stops traffic to prevent cascading failure. **Use Together:** Use Retry for transient network glitches, but trigger the Breaker if retries fail persistently.

AA7 — **Tenant Isolation Models:**
1. **Shared DB/Schema:** Lowest cost, highest noisy-neighbor risk, complex row-level security logic.
2. **Shared DB/Schema:** Better isolation, moderate cost, harder schema migrations.
3. **Separate DB:** Highest security/isolation, highest cost/ops complexity, easiest backup/restore per tenant.

AA8 — **Consistency:** Strong consistency ensures all nodes see the same data immediately. Eventual consistency allows temporary divergence. **Inventory Architecture:** Use **Strong Consistency** to prevent overselling. **Data Store:** Distributed SQL (e.g., CockroachDB) or Relational (Postgres) with distributed locking. **Patterns:** Optimistic locking (versioning) or Redis-based distributed locks to serialize inventory decrements.

AA9 — **Sidecar Proxy:** A container running alongside the main app (e.g., Envoy) handling network traffic. **Istio:** Envoy is the data plane (traffic); Control Plane (Pilot) configures Envoy. **mTLS:** Solves service identity verification and encryption without modifying application code, unlike app-level TLS which requires code changes.

AA10 — **Strangler Fig:** Incrementally migrate by routing specific traffic to new services. **Strategy:** Deploy a reverse proxy in front of the legacy .NET monolith. Route specific API paths to new .NET 8 microservices. Gradually move functionality until the monolith is empty and can be decommissioned.

AA11 — **12-Factor App:** 1. Codebase (one repo), 2. Dependencies (explicit), 3. Config (env vars), 4. Backing Services (treat as resources), 5. Build/Run/Exec (separate), 6. Processes (stateless). **Importance:** Ensures portability, scalability, and separation of config from code in cloud environments.

AA12 — **Event-Driven (Kafka) vs. Request-Driven (REST):**
*   **Coupling:** Event = Low (decoupled). Request = High (tight).
*   **Scalability:** Event = High (horizontal). Request = Moderate.
*   **Debugging:** Event = Complex (async flow). Request = Simple (traceable).
*   **Ordering:** Event = Guaranteed (per partition). Request = Best effort.
*   **Failure:** Event = Retry/Compensate. Request = Timeout/Retry.
*   **Choice:** Event for decoupling/scaling; Request for real-time user interaction.

AA13 — **Load Balancer:** Distributes traffic across servers. **L4:** Routes based on IP/Port (TCP/UDP). **L7:** Routes based on HTTP headers/URL. **Use Case:** SSL termination or routing `/api` to backend A and `/static` to backend B requires L7.

AA14 — **Message Queue:** Stores messages for consumers. **MQ (RabbitMQ):** Messages consumed once, then deleted. **Log Broker (Kafka):** Messages retained, replayable. **Pick MQ:** For task queues. **Pick Kafka:** For event streaming/audit logs.

AA15 — **Container vs. VM:** Container shares the host OS kernel; VM virtualizes hardware. **Popularity:** Containers are lightweight, start instantly, and ensure environment consistency, ideal for microservices.

AA16 — **Stateless vs. Stateful:** Stateless services store no session data (e.g., Web Server). Stateful services store session/data (e.g., Database). **Scaling:** Stateless is easier to scale horizontally because any instance can handle any request without data synchronization.

AA17 — **ORM:** Maps database tables to objects. **Examples:** Hibernate (Java), Entity Framework (.NET). **Trade-offs:** ORMs increase development speed but can generate inefficient SQL and hide performance bottlenecks compared to raw SQL.

AA18 — **Bulkhead Pattern:** Isolates resources (threads, connections) for specific services. **Implementation:** Use separate thread pools for each downstream service call. If one service fails, it exhausts its pool but leaves others available, preventing cascade failure.

AA19 — **OT vs. CRDT:** OT requires a central server to resolve conflicts (high latency, complex). CRDTs allow mathematically guaranteed convergence without a server (low latency, complex math). **Choice:** **CRDT** for offline editing support, as clients can merge changes locally without server coordination.

AA20 — **Distributed Tracing:** **Context Propagation:** Inject trace IDs into headers (W3C/B3). **Sampling:** Head-based (start) vs. Tail-based (end, based on latency). **Storage:** Jaeger/Tempo query traces. **Kafka:** Trace context must be injected into message headers so downstream consumers can continue the trace chain.

OP1 — **Hypervisor Type 1:** Bare-metal (e.g., VMware ESXi, Microsoft Hyper-V Server). **Type 2:** Hosted on OS (e.g., VirtualBox, VMware Workstation). Type 1 is for production; Type 2 is for development.

OP2 — **SAN vs. NAS:** SAN provides block storage (e.g., iSCSI, Fibre Channel) for databases. NAS provides file storage (e.g., NFS, SMB) for shared files. **Use SAN:** For high-performance DBs. **Use NAS:** For file shares/backups.

OP3 — **3-Tier Network:** Core (backbone), Distribution (policy), Access (endpoints). **Leaf-Spine:** Preferred for modern DCs. **Problem Solved:** Eliminates oversubscription and reduces latency compared to the multi-hop 3-tier architecture.

OP4 — **Proxmox HA:** **Architecture:** 3+ nodes for quorum. **Storage:** Ceph or iSCSI. **Fencing:** DRAC/iLO to power off failed nodes. **Failure:** Quorum lost? No. Node down? VMs restart on surviving nodes via Pacemaker.

OP5 — **VLAN Trunking:** Carries multiple VLANs over one link (802.1Q tagging). **Linux Config:** Create sub-interfaces: `ip link add link eth0 name eth0.10 type vlan id 10`. Assign IP to `eth0.10`.

OP6 — **RAID:** RAID 1 (Mirror), 5 (Striped+Parity), 6 (Double Parity), 10 (Mirror+Striped), Z2 (ZFS Double Parity). **Choice:** **RAID 10** for 6 drives. Best balance of read/write performance and fault tolerance (can lose 2 drives if mirrored pairs).

OP7 — **DNS Architecture:** **Internal:** AD-integrated DNS (Forwarders to external). **External:** Public authoritative servers. **Split-Horizon:** Internal resolves private IPs, external resolves public. **Laptop:** DHCP updates DNS records; resolves internal via AD, external via ISP/Cloud.

OP8 — **PKI Chain:** Root CA (Offline) -> Subordinate CA (Online) -> End Entity. **Compromise:** Root compromised = Trust lost globally (must rebuild PKI). Subordinate compromised = Revoke subordinate, issue new certs. **Auto-enrollment:** Windows AD CS auto-enrolls certs for users/machines.

OP9 — **WAN Consolidation:** **Recommendation:** **SD-WAN.** **Reason:** MPLS is expensive. IPsec is complex to manage. SD-WAN offers cost savings, better reliability (bonding), and QoS for VoIP with lower management complexity.

OP10 — **iDRAC/iLO/IPMI:** Out-of-band management interfaces. **Critical:** Allows server management even if OS is down. **Exposed:** If exposed to internet, attackers can gain root access, reboot, or steal data (IPMI is often unencrypted).

OP11 — **LVM vs. ZFS:** LVM manages volumes; ZFS manages filesystem + volumes + snapshots. **Snapshots:** LVM (Copy-on-Write, slower). ZFS (Instant, Copy-on-Write, efficient). **Use ZFS:** For data integrity and snapshots. **Use LVM:** For simple volume management on standard filesystems.

OP12 — **Stretch Cluster:** **Design:** Ceph monitors in both sites. **Latency:** <5ms required. **Split-Brain:** 2-site Ceph is bad because network latency causes quorum issues. **Tiebreaker:** Use a disk witness or third site to break ties and prevent data corruption.

OP13 — **DHCP:** **DORA:** Discover, Offer, Request, Acknowledge. **Unavailable:** Client uses APIPA (169.254.x.x) if lease expires and server is unreachable.

OP14 — **Managed vs. Unmanaged:** Unmanaged = Plug-and-play. Managed = Configurable (VLANs, QoS, ACLs, SNMP). **Importance:** Managed switches allow traffic prioritization (VoIP) and security segmentation in enterprises.

OP15 — **NTP:** Network Time Protocol. **Critical:** Certificates, logs, and cluster consensus rely on accurate time. **Drift:** Causes authentication failures (Kerberos), log correlation issues, and data inconsistency in distributed systems.

OP16 — **Physical vs. VM vs. Container:** Physical = Hardware. VM = Virtualized Hardware. Container = Virtualized OS. **Bare Metal:** Use for high-performance workloads (DB, HPC) where virtualization overhead is unacceptable.

OP17 — **UPS vs. Generator:** UPS provides immediate short-term power (minutes). Generator provides long-term power (hours). **ATS:** Automatic Transfer Switch switches load from Utility to UPS to Generator seamlessly.

OP18 — **Active-Passive vs. Active-Active:** Active-Passive (one node works, one waits). Active-Active (both work). **PostgreSQL HA:** Pacemaker/Corosync with STONITH fencing (power off failed node). Virtual IP floats to active node.

OP19 — **Backup Strategy:** **3-2-1:** 3 copies, 2 media, 1 offsite. **Immutable:** Protects against ransomware. **Test:** Regularly restore random files. **Backup vs. DR:** Backup is the data copy; DR is the documented process to restore operations after a disaster.

OP20 — **802.1X:** **Roles:** Supplicant (Client), Authenticator (Switch), Server (RADIUS). **Methods:** EAP-TLS (Cert-based, secure) vs PEAP (Password-based). **Failure:** Port closed. **MAB:** Switch sends MAC address to RADIUS as fallback for printers/IoT devices that don't support 802.1X.
