K1 — Easy: A **Deployment** manages stateless applications, allowing pods to be replaced freely without guaranteed identity or ordering. A **StatefulSet** manages stateful applications, providing stable, unique network identities (e.g., `pod-0`, `pod-1`), ordered, graceful deployment/termination, and stable persistent storage.

K2 — Easy: A **ClusterIP** Service creates a virtual IP inside the cluster. The kube-proxy (using iptables or IPVS) updates rules on each node to route traffic destined for the Service IP to the backend Pod IPs listed in the Service's **Endpoints** object.

K3 — Medium: A **PersistentVolume (PV)** is a cluster-wide storage resource. A **PersistentVolumeClaim (PVC)** is a user request for storage. A **StorageClass** defines the provisioner and parameters for dynamic provisioning. They relate as: PVC requests storage; if a StorageClass matches, a PV is dynamically created; otherwise, a static PV must exist for the PVC to bind to.

K4 — Medium: 1. Check `kubectl logs <pod>` for application errors. 2. Check `kubectl describe pod <pod>` for Events (e.g., OOMKilled, CrashLoopBackOff reasons). 3. Verify Liveness/Readiness probes are not failing due to misconfiguration. 4. Check resource limits (CPU/Memory) to ensure the container isn't being throttled or killed.

K5 — Medium: CoreDNS provides DNS resolution for Services and pods within the cluster. If CoreDNS pods are down, internal DNS resolution fails (pods cannot resolve Service names), breaking service discovery, though external internet access might still work if node DNS is configured.

K6 — Hard: The scheduler uses **Filtering** (Predicates) to eliminate nodes that don't meet requirements (e.g., taints, resource availability) and **Scoring** (Priorities) to rank remaining nodes. Factors considered: 1. Resource requests (CPU/Memory), 2. Node Affinity/Taints, 3. Pod Anti-Affinity (spread).

K7 — Hard: **Scheduling:** Yes. Node has 4GB (4096Mi). 3.5GB (3584Mi) requested + 256Mi pod request = 3840Mi < 4096Mi. **Runtime:** The pod will be scheduled. If it uses 600Mi (exceeding 512Mi limit), the kernel will OOM Kill the container, causing a CrashLoopBackOff.

K8 — Hard: 1. External LB forwards traffic to Ingress Controller NodePort. 2. Ingress Controller (NGINX) parses Ingress resource rules. 3. NGINX routes traffic to the Kubernetes Service IP. 4. kube-proxy (iptables/IPVS) redirects Service IP to Pod IPs in the Endpoints list. 5. Traffic reaches the Application Pod.

K9 — Medium: A **NetworkPolicy** restricts traffic flow between pods. Default behavior: If no NetworkPolicy exists in a namespace, all traffic is allowed (intra-namespace and inter-namespace). When a policy is applied, it acts as a whitelist; traffic not matching the policy is denied (unless explicitly allowed by other policies).

K10 — Medium: **RollingUpdate** replaces pods incrementally, ensuring zero downtime. **Recreate** kills all existing pods before creating new ones, causing downtime. Choose **Recreate** for stateful applications that cannot handle concurrent versions or for quick, non-critical updates where downtime is acceptable.

K11 — Easy: A **Namespace** is a virtual cluster within a physical cluster for resource isolation. Default namespaces: `default`, `kube-system`, `kube-public`, and `kube-node-lease`.

K12 — Easy: `kubectl describe` shows detailed status and events. Unlike `get`, it shows: 1. **Events** (e.g., scheduling failures, OOMKilled), 2. **Conditions** (e.g., Ready status details), 3. **Mounts** (Volumes, ConfigMaps, Secrets attached).

K13 — Easy: A **ConfigMap** stores non-sensitive configuration data. Pods consume it via: 1. **Environment Variables** (injected into the container), 2. **Volume Mounts** (mounted as files in the filesystem).

K14 — Easy: **ClusterIP:** Internal cluster access only. **NodePort:** Exposes service on a static port on every node's IP. **LoadBalancer:** Provisions an external cloud load balancer to expose the service publicly.

K15 — Easy: A **DaemonSet** ensures a copy of a pod runs on all (or specific) nodes. Use it for system daemons (e.g., logging agents, monitoring, network plugins) rather than Deployments, which target a specific count of replicas.

K16 — Medium: **Taints** repel pods (node-side), **Tolerations** allow pods to schedule on tainted nodes (pod-side). **Node Affinity** pulls pods to nodes based on labels. Use Taints/Tolerations for exclusive node usage (e.g., GPU nodes); use Affinity for preference (e.g., schedule on zone A).

K17 — Medium: An **Operator** is a custom controller that manages stateful applications using CRDs. It differs from a standard controller by automating complex lifecycle tasks (backup, scaling, upgrades) specific to an app. It solves the problem of managing stateful apps (like databases) that require manual intervention.

K18 — Hard: etcd uses **Strong Consistency** (linearizable). If quorum is lost (e.g., 2/3 nodes lost in a 3-node cluster), the cluster becomes read-only/unavailable. Recovery: Restore from the last snapshot on new etcd nodes and rejoin the cluster; you cannot recover the lost state without a snapshot.

K19 — Hard: **Role/ClusterRole** define permissions (verbs/resources). **RoleBinding/ClusterRoleBinding** bind roles to subjects (users/groups).
```yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata: namespace: my-ns
rules:
- apiGroups: [""]
  resources: ["pods", "pods/log"]
  verbs: ["get", "list", "watch"]
---
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata: namespace: my-ns
subjects:
- kind: User
  name: developer
  apiGroup: rbac.authorization.k8s.io
roleRef:
  kind: Role
  name: pod-reader
  apiGroup: rbac.authorization.k8s.io
```

K20 — Hard: `ndots` defines how many dots trigger an FQDN search. Default `ndots:5` causes queries for every search domain for short names, causing latency. Fix: Set `dnsConfig: { ndots: 1 }` in the pod spec to reduce search domain queries.

D1 — Easy: **Rebase** rewrites history to create a linear commit chain. **Merge** preserves history with a merge commit. Avoid rebasing on public/shared branches to prevent disrupting collaborators' history.

D2 — Easy: **TCP** is connection-oriented, reliable (ordered delivery). **UDP** is connectionless, unreliable (fast). TCP uses HTTP/HTTPS. UDP uses DNS/Video Streaming.

D3 — Medium: A **race condition** occurs when output depends on the sequence/timing of uncontrollable events. Example: Two threads increment a shared counter `x++` without locking. Expected result 2, actual result 1 due to read-modify-write overlap.

D4 — Medium: **Symmetric** uses one key for encryption/decryption (fast). **Asymmetric** uses public/private key pairs (secure key exchange). TLS uses Asymmetric for the handshake (key exchange) and Symmetric for the data session (performance).

D5 — Medium: A **layer** is a read-only filesystem snapshot in a Docker image. Layer caching speeds up builds. Best practice: Order commands by frequency of change (e.g., copy `package.json` and install deps before copying source code) to maximize cache hits.

D6 — Medium: 1. Check application logs for errors. 2. Verify upstream dependencies (DB, APIs) are healthy. 3. Check resource limits (CPU/Memory) for throttling. 4. Review rate limiting or circuit breaker configurations.

D7 — Hard: **CAP** states a distributed system can only guarantee 2 of 3: Consistency, Availability, Partition Tolerance. **CP:** ZooKeeper (sacrifices availability during partition). **AP:** Cassandra (sacrifices consistency during partition).

D8 — Medium: **Horizontal Scaling** adds more nodes; **Vertical Scaling** adds more resources to a node. Patterns enabling horizontal scaling: Stateless architecture, Load Balancers, Distributed Caching (Redis), Sharding.

D9 — Easy: An **environment variable** is a dynamic value affecting a process. To persist across sessions on Linux, export it in `~/.bashrc` or `~/.profile` (e.g., `export VAR=value`).

D10 — Easy: **JSON** is strict, machine-readable. **YAML** is human-readable, supports comments. Advantage JSON: Native support in web APIs. Advantage YAML: Readability for config files.

D11 — Easy: **REST** is an architectural style using HTTP. **GET:** Retrieve resource. **POST:** Create resource. **PUT:** Replace resource. **DELETE:** Remove resource.

D12 — Easy: **Version pinning** specifies exact dependency versions (e.g., `package@1.0.0`). It ensures reproducible builds by preventing unexpected breaking changes from upstream updates.

D13 — Easy: **Compiled** (C, Go) translates code to binary before execution. **Interpreted** (Python, JS) translates at runtime. Compiled is faster; Interpreted is more flexible.

D14 — Medium: A **Reverse Proxy** sits in front of servers (client sees proxy). A **Forward Proxy** sits in front of clients (server sees proxy). **Nginx:** API Gateway. **Squid:** Caching/Filtering.

D15 — Medium: A **deadlock** is a permanent block where processes wait for each other. **Coffman Conditions:** 1. Mutual Exclusion, 2. Hold and Wait, 3. No Preemption, 4. Circular Wait.

D16 — Hard: **Eventual Consistency** guarantees data converges over time. Cassandra uses **Tunable Consistency Levels** (e.g., ONE, QUORUM, ALL). Read-after-write consistency is ensured by setting `CONSISTENCY=ALL` or using `SERIAL` consistency.

D17 — Hard: **mTLS** authenticates both client and server via certificates. Handshake: ClientHello, ServerHello, ServerCert, ClientCert (requested), Key Exchange, Finished. Both sides verify the other's cert chain. Used in Service Meshes (Istio) and secure DB connections.

D18 — Hard: A **hash table** maps keys to indices via a hash function. **Collision Resolution:** Chaining (linked lists) or Open Addressing (probing). **Complexity:** Insert/Lookup/Delete is O(1) average, O(n) worst case (all collisions).

D19 — Hard: A **Circuit Breaker** prevents cascade failures. **Closed:** Normal operation. **Open:** Fail fast (reject requests). **Half-Open:** Test if service recovered. Relates to **Retry/Backoff** by stopping retries when the circuit is open to avoid overloading the failing service.

D20 — Hard: **Optimistic:** Assumes no conflict, checks version on commit (e.g., `UPDATE ... WHERE version = x`). **Pessimistic:** Locks data immediately (e.g., `SELECT FOR UPDATE`). Choose Optimistic for low contention; Pessimistic for high contention or strict serializability.
