CL1 — **IaaS (Infrastructure as a Service)** provides virtualized computing resources (servers, storage, networking). Example: AWS EC2. **PaaS (Platform as a Service)** provides a framework for developing, running, and managing applications without dealing with infrastructure. Example: Google App Engine. **SaaS (Software as a Service)** delivers software applications over the internet. Example: Microsoft Office 365.

CL2 — **Shared Responsibility Model:** The provider secures the cloud (hardware, facilities), the customer secures what is *in* the cloud. **IaaS VM:** Provider secures hypervisor/physical host; you secure OS, patching, data, firewall. **Managed Kubernetes:** Provider secures control plane/infrastructure; you secure nodes, workloads, network policies, secrets. **SaaS (M365):** Provider secures app, infrastructure, data; you secure identity, access management, data classification.

CL3 — **VPC:** Isolated virtual network. **Subnet:** Range of IP addresses within a VPC. **Security Group:** Stateful firewall at the instance level (allow rules). **NACL:** Stateless firewall at the subnet level (allow/deny rules). **Layering:** VPC contains Subnets; Subnets contain Instances protected by SGs; NACLs filter traffic entering/leaving Subnets. **Stateful:** SG remembers return traffic. **Stateless:** NACL requires explicit rules for both request and response.

CL4 — **Landing Zone:** Pre-configured secure environment. **OUs:** Logical grouping for billing/policy. **Guardrails:** Preventative controls (e.g., no public S3 buckets). **Shared Networking:** Hub-and-Spoke centralizes connectivity. **Centralized Logging:** Aggregates logs for visibility. **Single-Account Problems:** High blast radius (security), billing complexity, lack of isolation for compliance, shared resource contention.

CL5 — **IaC:** Managing infrastructure via code. **Terraform/OpenTofu:** Multi-cloud, declarative, state-managed. **Pulumi:** Multi-cloud, imperative (general-purpose languages). **CloudFormation/Bicep:** Native to AWS/Azure, tightly integrated. **Crossplane:** IaC via Kubernetes API. **Choice:** Terraform for multi-cloud flexibility; CloudFormation for AWS-native simplicity; Pulumi for dev-friendly languages; Crossplane for K8s-centric teams.

CL6 — **HPA:** Scales pods based on CPU/Memory. **VPA:** Adjusts pod resource requests/limits. **Cluster Autoscaler:** Adds/removes nodes. **KEDA:** Event-driven scaling (queues, metrics). **Use KEDA over HPA:** When scaling is driven by external events (e.g., Kafka message count, SQS queue depth) rather than internal resource utilization.

CL7 — **DR Strategy:** Define RPO (data loss tolerance) and RTO (downtime tolerance). **Active-Active:** Both regions serve traffic (low RTO/RPO, high cost). **Active-Passive:** One active, one standby (higher RTO, lower cost). **DB:** Sync (synchronous replication, low RPO, latency) vs Async (higher RPO, lower latency). **DNS:** Route53 failover routing. **State:** Shared storage vs replicated state. **DR Types:** Pilot Light (min data running), Warm Standby (scaled down prod), Multi-site (full active-active).

CL8 — **GitOps:** Git repository is the single source of truth for infrastructure/app state. **ArgoCD vs Flux:** ArgoCD is UI-focused, Flux is CLI/K8s-native. **Secrets:** Sealed Secrets (encrypted in Git), External Secrets Operator (fetch from Vault/AWS Secrets Manager). **Drift:** Controllers detect manual changes and automatically revert them to match Git state during reconciliation.

CL9 — **Architecture:** Use **StatefulSet** for stable network IDs and storage. **PV/PVC:** Use `local` storage class for low latency (hostPath). **Headless Service:** For stable DNS names. **Init Containers:** For config file preparation. **ConfigMaps:** Mount as volumes for specific paths. **PDB:** Ensure minimum available pods during updates.

CL10 — **SA:** K8s identity for pods. **Workload Identity:** Maps K8s SA to cloud identity. **IRSA:** AWS specific mapping. **Mounting Creds:** Bad practice (static keys, risk of leakage, hard to rotate). **Federation:** Uses OIDC to exchange K8s SA token for short-lived cloud credentials, eliminating static secrets.

CL11 — **Object (S3):** Unstructured data, high durability (11 9s), low latency. **Block (EBS):** Attached storage for OS/DB, high IOPS, attached to single instance. **File (EFS):** Shared file system, NFS/SMB, multiple mounts. **Use Case:** Object for backups/media, Block for DBs/OS, File for shared configs/logs.

CL12 — **Zero Trust:** Verify every request. **Network Policies:** Restrict pod-to-pod traffic. **Service Mesh (mTLS):** Encrypts service-to-service traffic. **Pod Identity:** Least privilege access. **Admission Controllers:** Enforce policies (OPA/Kyverno). **Image Signing:** Verify integrity. **Runtime Security:** Detect anomalies (Falco). **Differs:** Traditional relies on perimeter; Zero Trust assumes breach and verifies identity/context continuously.

CL13 — **CDN:** Distributed network of servers caching content closer to users. **Cloud Services:** AWS CloudFront, Azure CDN. **Use CDN:** For static assets (images, JS, CSS) to reduce latency and origin load. **Use Origin:** For dynamic, personalized, or real-time data not suitable for caching.

CL14 — **Region:** Geographic area with multiple AZs. **AZ:** Isolated data center with independent power/cooling. **Multi-AZ:** Ensures high availability if one data center fails. **Zonal:** Resource tied to one AZ (e.g., EBS). **Regional:** Resource spans AZs (e.g., ELB, RDS Multi-AZ).

CL15 — **Managed DB:** Service where provider handles provisioning, patching, backups, failover, and scaling. **Tasks Handled:** OS patching, DB engine updates, storage provisioning, HA setup. **Your Tasks:** Schema design, queries, user access, data modeling.

CL16 — **IAM:** **Users** (human), **Groups** (collections), **Roles** (temporary access), **Policies** (permissions). **Least Privilege:** Grant only necessary permissions. **Application:** Use roles instead of long-term keys, group users by function, review permissions regularly, use SCPs for guardrails.

CL17 — **Serverless:** Event-driven execution without server management. **Comparison:** Lambda (AWS), Functions (Azure), Cloud Functions (GCP). **Cold Starts:** Latency when initializing a new instance. **Poor Fit:** Long-running processes, stateful apps, predictable high-throughput workloads (costlier than provisioned).

CL18 — **Ingress:** L7 routing rules (HTTP/HTTPS). **Gateway API:** Extensible, standardized routing. **LB:** L4 load balancing. **Mesh:** L7 service-to-service. **TLS:** Terminates at LB/Ingress (external) or Mesh (internal). **Use Gateway API:** For complex routing, multi-cluster, or when Ingress resource limitations hinder requirements.

CL19 — **Strategy:** **RI/SP:** Commit to usage for discounts. **Spot:** Use for fault-tolerant workloads. **Right-sizing:** Match instance to actual usage. **Tiering:** Move old data to Glacier. **FinOps:** Cross-team cost accountability. **Tools:** AWS Cost Explorer, Kubecost, CloudHealth. **Process:** Monthly reviews, tagging enforcement, showback reports to teams.

CL20 — **Strategy:** **Abstraction:** Kubernetes (K8s) as runtime. **IaC:** Terraform (portable). **Networking:** Direct Connect/ExpressRoute + VPN. **Identity:** Federation (OIDC). **Trade-offs:** Multi-cloud increases complexity, networking latency, and operational overhead. **Justified:** Regulatory requirements, avoiding single-vendor risk, best-of-breed services. **Unnecessary:** If one cloud meets all needs, multi-cloud adds cost without value.

OT1 — **Purdue Model:** ISA-95 standard for IT/OT integration. **Levels:** 0 (Physical Process), 1 (Sensors/Actuators), 2 (Control Devices/PLCs), 3 (Operations/SCADA), 4 (Enterprise/Business).

OT2 — **IT:** Focus on confidentiality, data integrity, patching cycles. **OT:** Focus on safety, availability, uptime. **Patching:** OT systems often run legacy OS/hardware; patching can cause instability or downtime, risking physical safety.

OT3 — **DMZ:** Buffer zone between IT and OT. **Traffic:** Historian to SCADA (allowed), IT to OT (restricted to specific protocols). **Direct L4 to L1:** Dangerous because it bypasses security controls, allowing malware to directly reach control devices without inspection.

OT4 — **Protocols:** 1. **Modbus TCP:** L1-2, Master/Slave. 2. **OPC UA:** L2-3, Secure, platform-independent. 3. **EtherNet/IP:** L1-2, CIP protocol. 4. **PROFINET:** L1-2, Real-time Ethernet. 5. **MQTT:** L3-4, Lightweight pub/sub for cloud.

OT5 — **Data Diode:** Hardware allowing data flow in only one direction. **Firewall:** Allows bidirectional traffic with rules. **Use Diode:** When data must leave OT but nothing can enter (e.g., telemetry to cloud). **Limitation:** Cannot send commands back to OT devices.

OT6 — **Flow:** PLCs -> Historian -> Edge Gateway (protocol translation) -> MQTT Broker -> Cloud. **Security:** Edge Gateway in DMZ. Encryption (TLS) between Edge and Cloud. Firewall rules restricting traffic to specific ports/protocols.

OT7 — **Zones:** Logical groupings of assets with similar security requirements. **Conduits:** Controlled paths between zones. **Model:** Zone 4 (Corporate), Zone 3 (SCADA/Engineering), Zone 2 (PLCs), Zone 1 (Sensors). Conduits between zones require firewalls/DMZs.

OT8 — **Segmentation:** VLANs per production line. Firewalls between VLANs and DMZ. Switches with port security. **Remote Access:** Jump host in DMZ with MFA. **Transition:** Phased rollout during maintenance windows, parallel testing, backup of configs before changes.

OT9 — **SIS:** Independent safety system (stops process safely). **BPCS:** Basic process control (maintains process). **IEC 61511:** Safety lifecycle standard. **Isolation:** Prevents cyber attacks on BPCS from disabling safety functions. **Ransomware on SIS:** Could prevent emergency shutdown, leading to physical damage or injury.

OT10 — **PLC:** Local logic controller. **DCS:** Distributed control for continuous processes. **SCADA:** Supervisory system for monitoring. **Coexist:** Yes, PLCs/DCS control devices, SCADA supervises them.

OT11 — **Architecture:** Jump host in DMZ, MFA required, session recording enabled. **Time-limited:** Access expires after task. **Direct RDP:** Unacceptable because it exposes HMI directly to corporate network, increasing attack surface and bypassing OT security controls.

OT12 — **Plan:** Isolate compromised segment immediately. Notify plant operator. Activate manual control procedures. Analyze logs for lateral movement. Restore from clean backups. **Operator Role:** Maintain physical safety, monitor process variables, execute manual overrides if needed.

OT13 — **HMI:** Interface for operators to view and control processes. **Difference:** HMI is for operation (real-time), Engineering Workstation is for configuration/programming (maintenance).

OT14 — **Historian:** Time-series database for process data. **Importance:** Trend analysis, compliance, troubleshooting. **Plant:** Real-time ops (Level 3). **Enterprise:** Long-term storage, analytics (Level 4).

OT15 — **Ladder Logic:** Graphical programming language resembling relay circuits. **Device:** PLC. **Dominance:** Familiar to electricians, deterministic execution, proven reliability over decades.

OT16 — **RTU:** Remote Terminal Unit, designed for harsh environments, long-distance comms. **PLC:** Local control. **RTU Use:** Pipelines, water distribution, remote sites.

OT17 — **Analog:** Continuous signal (voltage/current). **Digital:** Discrete (on/off). **4-20mA:** Preferred because 0mA indicates wire break (fail-safe), whereas 0-20mA cannot distinguish between zero signal and broken wire.

OT18 — **Inventory:** Knowing all assets is critical for patching and threat detection. **Tools:** Passive (packet sniffing), Active (scanning). **Active Risk:** Scanning can crash legacy devices or trigger alarms in safety systems.

OT19 — **MITRE ICS:** Framework for ICS threats. **Tactics:** 1. Initial Access (Phishing), 2. Execution (Command Line), 3. Persistence (Scheduled Task), 4. Impact (Disruption), 5. Defense Evasion (Obfuscation). **Difference:** Focuses on physical impact and safety, not just data. **Usage:** Map defenses to tactics to find gaps.

OT20 — **SOC:** SIEM collects logs (via agents/forwarders). **Rules:** OT-specific (e.g., PLC write attempts). **Triage:** OT engineer validates before IT action. **Firmware Alert:** Isolate PLC, verify change authorization, restore from known good image, investigate source of unauthorized access.
