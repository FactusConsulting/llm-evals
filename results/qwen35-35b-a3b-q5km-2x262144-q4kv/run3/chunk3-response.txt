T1 — `tofu plan` performs a dry run to calculate and display the changes required to reach the desired state without making any modifications. `tofu apply` executes those changes to update the infrastructure.

T2 — State is a JSON file mapping OpenTofu resources to real-world infrastructure. Remote state is important for team collaboration (sharing state), state locking (preventing concurrent writes), and security (storing state off local machines).

T3 — `count` uses integer indices, which can cause resource shifting if the list changes. `for_each` uses a map or set of keys, ensuring stable resource IDs even if the list order changes. `for_each` is preferred for dynamic sets to prevent unintended resource recreation.

T4 — This is called "drift". OpenTofu detects it during `tofu plan` by comparing the state file to the live infrastructure. If the configuration matches the desired state, `tofu apply` will fix the drift; if not, it may fail or require manual intervention.

T5 —
```hcl
resource "aws_instance" "web" {
  for_each = toset(["web1", "web2", "web3"])
  ami      = "ami-12345678"
  instance_type = "t2.micro"
  tags = {
    Name = each.key
  }
}
```

T6 — `terraform_remote_state` reads state from a separate backend, creating tight coupling. `output` values allow modules to expose data directly. The recommended alternative is using **Module Outputs** (passing data via `module "..." { ... }`) as it is more secure, decoupled, and supported by OpenTofu 1.5+ features.

T7 — The `remote-exec` provisioner runs asynchronously. The output block may read the kubeconfig file before the provisioner finishes writing it. Fix: Add `depends_on = [null_resource.kubeconfig]` to the resource consuming the output, or use a `local-exec` provisioner that waits for the file to exist before completing.

T8 — `terraform import` is a CLI command to add existing resources to state. `import block` (1.5+) is declarative, allowing import logic to be version-controlled and automated within the configuration. It improves maintainability and reduces manual CLI steps.

T9 — A `variable` is an input parameter passed into the configuration (e.g., from CLI or `.tfvars`). An `output` is a value exposed from the configuration to be used by other modules or the CLI.

T10 — A provider is a plugin that interacts with a specific API (e.g., AWS, Azure). Running `tofu plan` without `tofu init` fails because the provider binaries are not downloaded or initialized.

T11 — The `.terraform.lock.hcl` file records the exact versions of providers used. It should be committed to ensure reproducible builds and prevent accidental provider upgrades that might break infrastructure.

T12 — `tofu destroy` removes all resources managed by the configuration. To target a single resource: `tofu destroy -target=aws_instance.web`.

T13 — A data source reads information from an external system without creating or managing it. Example: Using `data "aws_ami" "ubuntu"` to find the latest Ubuntu AMI ID instead of managing an AMI resource.

T14 — `variable` blocks define inputs for the module (external). `locals` define internal values computed from variables or other locals. Use `locals` for intermediate calculations not meant to be exposed as inputs.

T15 — `lifecycle` controls resource behavior. `create_before_destroy` creates the new resource before destroying the old (zero downtime). `prevent_destroy` prevents accidental deletion. `ignore_changes` ignores drift in specific attributes (e.g., tags modified manually).

T16 — A `moved` block allows you to change the address of a resource (e.g., moving from one module to another) without destroying and recreating it. OpenTofu updates the state mapping automatically.

T17 — The dependency graph determines execution order. Implicit dependencies are inferred from references (e.g., `aws_instance.web.id`). Explicit dependencies use `depends_on`. `depends_on` reduces parallelism because it forces sequential execution, slowing down plans.

T18 — 1. **Remote State Backend**: Enables parallelism and faster state access. 2. **`refresh=false`**: Skips state refresh to save API calls. 3. **`-exclude`**: Exclude unused resources from the plan. 4. **Reduce Graph Complexity**: Avoid large `for_each` loops in plan-heavy modules.

T19 — State locking prevents concurrent writes. If a lock isn't released (e.g., CI crash), subsequent runs fail. Manually break it with `tofu force-unlock <LOCK_ID>`. Risk: Concurrent writes may corrupt the state file.

T20 — 1. Backup local state file. 2. Update `backend "s3"` block in config. 3. Run `tofu init -migrate-state -backend-config="bucket=..."`. 4. Verify state in S3 console. 5. Run `tofu plan` to confirm state integrity.

A1 — A **Playbook** is a YAML file defining the execution flow. A **Role** is a directory structure for reusable logic. A **Task** is a single action (e.g., install package) within a playbook or role.

A2 — `ansible_facts` contains gathered system information. Access OS family via `{{ ansible_os_family }}`.

A3 — `command` runs without a shell (no pipes). `shell` runs with a shell (supports pipes). `raw` runs without setup (no Python module transfer), useful for initial bootstrap.

A4 — Idempotency means running a task multiple times yields the same result. **Idempotent**: `apt: state=present` (installs only if missing). **Not Idempotent**: `shell: echo "Hello" > /tmp/file` (overwrites file every time).

A5 —
```yaml
- name: Install htop
  package:
    name: htop
    state: present
  when: ansible_os_family == "Debian"

- name: Install htop on RedHat
  package:
    name:
      - epel-release
      - htop
    state: present
  when: ansible_os_family == "RedHat"
```

A6 — Precedence (High to Low): Extra vars (`-e`) > Play vars > Host vars > Group vars > Role defaults. `set_fact` overrides most at runtime.

A7 — Use `serial: 3` in the play to limit concurrency. Group nodes into `rke2_first` and `rke2_rest`. Use `wait_for` or `serial` to ensure the first 3 complete before the rest start.

A8 — `ansible-vault` encrypts sensitive data. To encrypt a single variable: `ansible-vault encrypt_string 'secret_value' --name 'my_secret'`.

A9 — An inventory file lists hosts and groups. **Static**: Defined in a file (INI/YAML). **Dynamic**: Generated by a script or API (e.g., AWS EC2) to reflect live infrastructure.

A10 — `ansible webservers -m shell -a "df -h"`.

A11 — A **Handler** is a task that runs only when notified by another task (e.g., `notify: restart service`). It differs from a regular task by running asynchronously at the end of the play or immediately upon notification.

A12 — `ansible.cfg` configures Ansible behavior. Common settings: `host_key_checking`, `forks` (parallelism), `remote_user`.

A13 — `register` stores the output of a task in a variable. Use it in subsequent tasks via `when: my_var.stdout == "success"` or `debug: var=my_var`.

A14 — `import_tasks` is static (evaluated at parse time, ignores `when`). `include_tasks` is dynamic (evaluated at runtime, respects `when`). Use `import` for fixed lists, `include` for conditional logic.

A15 — Ansible Galaxy is a repository for roles and collections. Use `requirements.yml` to list dependencies. Install with `ansible-galaxy install -r requirements.yml`.

A16 — Connection plugins manage communication. **SSH**: Remote Linux/Unix hosts. **Local**: Run on control node. **Network_CLI**: Network devices (Cisco, Juniper).

A17 — `delegate_to` runs a task on a specific host (e.g., `localhost`). `run_once` ensures it runs once per play. Example: Gather facts from remote host, then write config to control node using `delegate_to: localhost`.

A18 — 1. **Increase `forks`**: More parallel connections. 2. **Enable `pipelining`**: Reduces SSH overhead. 3. **Fact Caching**: Avoids gathering facts every run. 4. **Mitogen**: Optimizes SSH transport for speed.

A19 — `{{ }}` outputs variables. `{% %}` controls logic (loops/ifs). `{# #}` comments.
```jinja2
{% for server in backends %}
# Health check for {{ server.ip }}
upstream backend {
    server {{ server.ip }};
}
{% endfor %}
```

A20 — 1. Set `serial: 5` for batches. 2. **Pre-task**: Deregister from LB. 3. **Deploy**: Update app. 4. **Post-task**: Register to LB, `uri` health check. 5. **Rollback**: Use `block/rescue` to revert if health check fails.
