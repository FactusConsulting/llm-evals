CL1 — **IaaS, PaaS, SaaS**
*   **IaaS (Infrastructure as a Service):** Provides virtualized computing resources over the internet. You manage OS, middleware, runtime, data, and apps. *Example:* AWS EC2, Azure VMs.
*   **PaaS (Platform as a Service):** Provides hardware and software tools (usually for app development) hosted on the cloud. You manage data and apps; provider manages OS/runtime. *Example:* AWS Elastic Beanstalk, Google App Engine.
*   **SaaS (Software as a Service):** Fully managed software applications delivered over the internet. You manage nothing but user access. *Example:* Salesforce, Microsoft 365.

CL2 — **Shared Responsibility Model**
*   **IaaS VM:** Provider secures physical hardware, hypervisor, and network infrastructure. You secure the OS, patching, firewall config, and data.
*   **Managed Kubernetes:** Provider secures control plane, nodes, and underlying infrastructure. You secure node OS (if self-managed), pod security, network policies, and application code.
*   **SaaS (M365):** Provider secures everything from hardware to application logic. You secure user identities, access policies, and data classification.
*   **Boundary:** Responsibility shifts from infrastructure (IaaS) to application (SaaS) as you move up the stack.

CL3 — **VPC, Subnet, SG, NACL**
*   **VPC:** Isolated virtual network in the cloud.
*   **Subnet:** Segments of the VPC (public/private) mapped to Availability Zones.
*   **Security Group (SG):** Stateful firewall attached to instances (ENI). Allows whitelisting traffic (inbound/outbound).
*   **NACL:** Stateless firewall attached to subnets. Explicitly allows/denies traffic (inbound/outbound).
*   **Layering:** Traffic hits NACL (subnet level) first, then SG (instance level).
*   **Stateful vs Stateless:** SG remembers connection state (return traffic allowed automatically). NACL requires explicit rules for return traffic.

CL4 — **Multi-Account Strategy**
*   **Landing Zone:** Pre-configured environment with networking, identity, and security baseline.
*   **Organizational Units (OUs):** Group accounts by function (e.g., Prod, Dev, Security) for policy application.
*   **Guardrails:** SCPs (Service Control Policies) to prevent risky actions (e.g., disabling CloudTrail).
*   **Shared Networking:** Transit Gateway (AWS) or Hub-and-Spoke (Azure) connects accounts centrally for traffic flow.
*   **Centralized Logging:** Aggregates logs to a dedicated Security/Log account.
*   **Single-Account Problem:** High blast radius (one breach compromises all), no isolation for compliance, difficult cost allocation, shared resource contention.

CL5 — **Infrastructure as Code (IaC)**
*   **Terraform/OpenTofu:** Declarative, multi-cloud, state-file based. Best for general-purpose, multi-cloud, and complex dependencies.
*   **Pulumi:** Imperative/Declarative using general languages (TS, Python). Best for teams wanting code flexibility and logic.
*   **CloudFormation/Bicep:** Native to AWS/Azure. Best for deep integration with specific provider features and CI/CD pipelines within that cloud.
*   **Crossplane:** Kubernetes-native IaC. Best for managing cloud resources as K8s CRDs (unified control plane).
*   **Choice:** Terraform for portability; Native for depth; Pulumi for logic; Crossplane for K8s-centric ops.

CL6 — **Cloud-Native Autoscaling**
*   **HPA:** Scales pods based on CPU/Memory or custom metrics. Good for web traffic spikes.
*   **VPA:** Scales pod resource requests/limits (CPU/RAM). Good for optimizing resource usage, but requires pod restart.
*   **Cluster Autoscaler:** Adds/removes nodes based on pending pods. Good for capacity management.
*   **KEDA:** Event-driven scaling (e.g., Kafka queue depth, RabbitMQ).
*   **KEDA vs HPA:** Use KEDA when scaling is driven by external events (queues, DB load) rather than just CPU/Memory metrics.

CL7 — **Disaster Recovery (DR) Strategy**
*   **RPO/RTO:** RPO = Max data loss (e.g., 1hr); RTO = Max downtime (e.g., 4hrs).
*   **Active-Active:** Both regions serve traffic. High cost, low RTO/RPO.
*   **Active-Passive:** One region active, one standby. Lower cost, higher RTO.
*   **Database:** Sync (synchronous replication, low RPO, high latency) vs Async (higher RPO, lower latency).
*   **DNS Failover:** Route53 Health Checks trigger failover to secondary region.
*   **Patterns:** Pilot Light (DB running, app off), Warm Standby (scaled-down app running), Multi-Site (Active-Active).
*   **Cost:** Active-Active is most expensive; Pilot Light is cheapest but highest RTO.

CL8 — **GitOps Model**
*   **Model:** Git repository is the single source of truth. Controllers sync cluster state to Git state.
*   **ArgoCD vs Flux:** ArgoCD has a UI and push/pull models; Flux is CLI-first, Git-native, and pull-based.
*   **Secrets:** Sealed Secrets (encrypted in Git), External Secrets Operator (fetches from Vault/AWS Secrets Manager), Vault Agent.
*   **Drift:** If manual changes occur, the GitOps controller detects drift and reverts changes to match Git state.

CL9 — **Stateful Legacy App to K8s**
*   **StatefulSet:** Ensures stable network IDs and ordered deployment/termination.
*   **PV/PVC:** Use `local` storage class for low latency (node-local disks). `volumeClaimTemplates` in StatefulSet automate PVC creation.
*   **Headless Service:** Provides stable DNS names for pods (e.g., `pod-0.app.svc`).
*   **Init Containers:** Handle config file generation or DB initialization before main app starts.
*   **ConfigMaps:** Mount as volumes for config files at specific paths.
*   **PDB:** Ensure minimum available pods during disruptions to maintain quorum.

CL10 — **Identity (SA vs Workload vs IRSA)**
*   **Service Account:** K8s native identity for pods.
*   **Workload Identity:** Maps K8s SA to cloud IAM identity (Azure/GCP).
*   **IRSA:** Maps K8s SA to AWS IAM Role via OIDC.
*   **Mounting Credentials:** Bad practice (static keys, rotation difficulty, risk of leakage in logs).
*   **Federation:** Uses OIDC tokens to request temporary AWS credentials dynamically, eliminating static secrets.

CL11 — **Storage Types**
*   **Object (S3/Blob):** Scalable, HTTP-based, metadata-rich. Use for backups, media, logs. Durability: 99.999999999%.
*   **Block (EBS/Disk):** Low latency, attached to single VM. Use for OS, DBs. Durability: High, but single AZ usually.
*   **File (EFS/Azure Files):** Shared file system (NFS/SMB). Use for shared configs, legacy apps.
*   **Guarantees:** Object storage offers highest durability; Block offers lowest latency; File offers shared access.

CL12 — **Zero-Trust Network Architecture**
*   **Network Policies:** Restrict pod-to-pod traffic (default deny).
*   **Service Mesh (mTLS):** Encrypts service-to-service traffic (Istio/Linkerd).
*   **Pod Identity:** Workload identity (IRSA/Workload Identity) for cloud access.
*   **Admission Controllers:** OPA/Kyverno enforce policies (e.g., no root containers) at deploy time.
*   **Image Signing:** Cosign/Notary to verify image integrity.
*   **Runtime Security:** Falco detects anomalies (e.g., shell spawn).
*   **Difference:** Traditional relies on perimeter (firewall); Zero Trust assumes breach, verifies every request.

CL13 — **CDN**
*   **Definition:** Distributed network of servers caching content closer to users.
*   **How:** Caches static assets (images, CSS) at edge locations.
*   **Services:** CloudFront (AWS), Azure CDN.
*   **Use CDN:** For high latency reduction, DDoS protection, offloading origin.
*   **Use Origin:** For dynamic content, real-time data, or when caching is not viable.

CL14 — **Availability Zones & Regions**
*   **Region:** Geographic area (e.g., us-east-1) with multiple AZs.
*   **AZ:** Isolated data centers within a region with independent power/cooling.
*   **Multi-AZ:** Protects against data center failure.
*   **Zonal:** Resources tied to one AZ (e.g., EBS).
*   **Regional:** Resources span AZs (e.g., RDS Multi-AZ, S3).

CL15 — **Managed Database Service**
*   **Definition:** DBaaS (e.g., RDS, Cloud SQL).
*   **Provider Handles:** Hardware provisioning, patching, backups, failover, monitoring, scaling.
*   **You Handle:** Schema, queries, user access, data logic.
*   **Benefit:** Reduces operational overhead and improves availability compared to self-managed VM DBs.

CL16 — **Cloud Identity & Least Privilege**
*   **IAM Users:** Humans. **Groups:** Collections of users. **Roles:** Temporary credentials for services/users. **Policies:** JSON permissions.
*   **Least Privilege:** Grant only necessary permissions.
*   **Application:** Use roles instead of long-term keys, scope policies to specific resources (ARNs), and review permissions regularly.

CL17 — **Serverless Computing**
*   **Definition:** Run code without provisioning servers (AWS Lambda, Azure Functions).
*   **Cold Starts:** Delay when a function is invoked after inactivity (loading runtime).
*   **Matter:** Critical for latency-sensitive apps (e.g., real-time APIs).
*   **Poor Fit:** Long-running processes, high memory/CPU needs, stateful apps, or strict vendor lock-in concerns.

CL18 — **Kubernetes Ingress Options**
*   **Ingress Resource:** Standard HTTP routing. TLS termination at Ingress Controller.
*   **Gateway API:** New standard, separates routing from controller logic, supports gRPC/HTTP2 better.
*   **Cloud LB Annotations:** Provider-specific LB (e.g., AWS NLB).
*   **Service Mesh:** Ingress Gateway (e.g., Istio) for mTLS and observability.
*   **Gateway API:** Use when you need multi-cluster support, better standardization, or advanced routing rules beyond standard Ingress.

CL19 — **Cost Optimization Strategy**
*   **RI/Savings Plans:** Commit to usage for 1-3 years (up to 72% savings).
*   **Spot/Preemptible:** Use unused capacity (up to 90% savings, interruptible).
*   **Right-sizing:** Match instance size to actual CPU/Mem usage.
*   **Storage Tiering:** Move old data to Glacier/Archive.
*   **Idle Detection:** Find unattached EBS, idle LBs.
*   **FinOps:** Cross-team visibility, showback/chargeback reports, tagging enforcement.
*   **Tools:** AWS Cost Explorer, CloudHealth, Kubecost.

CL20 — **Multi-Cloud Strategy**
*   **Abstraction:** Kubernetes (runtime), Terraform (IaC), Crossplane (resource abstraction).
*   **Networking:** Direct Connect/ExpressRoute or Interconnect for low latency.
*   **Identity:** Federation (OIDC) to central IdP.
*   **Trade-offs:** Avoids lock-in but increases complexity, networking latency, and operational overhead.
*   **Justified:** Regulatory requirements, disaster recovery, negotiating leverage.
*   **Unnecessary:** If one cloud meets all needs, multi-cloud adds risk without value.

OT1 — **Purdue Model (ISA-95)**
*   **Level 0:** Physical Process (Sensors, Actuators).
*   **Level 1:** Basic Control (PLCs, RTUs).
*   **Level 2:** Area Supervisory (SCADA, HMIs).
*   **Level 3:** Site Operations (MES, Historians).
*   **Level 4:** Enterprise (ERP, Business Planning).
*   **Level 5:** (Often merged with 4) Corporate Network.

OT2 — **IT vs OT Networks**
*   **IT:** Confidentiality/Integrity focus, frequent patching, high bandwidth.
*   **OT:** Availability/Safety focus, legacy systems, real-time constraints.
*   **Patching:** OT patching requires maintenance windows and validation to avoid downtime or safety hazards. IT patching is often automated and immediate.

OT3 — **DMZ (IDMZ)**
*   **Role:** Buffer zone between IT (Level 4) and OT (Level 0-2).
*   **Traffic:** Unidirectional (OT to IT) preferred for data export. Bidirectional only for specific commands (e.g., engineering updates) via jump hosts.
*   **Danger:** Direct Level 4 to Level 1 connectivity allows attackers to bypass controls and directly manipulate physical processes.

OT4 — **OT Protocols**
*   **Modbus TCP:** Level 1-2, Master/Slave, unencrypted.
*   **OPC UA:** Level 2-3, Secure, platform-independent, client/server.
*   **EtherNet/IP:** Level 1-2, CIP protocol, common in Allen-Bradley.
*   **PROFINET:** Level 1-2, Real-time Ethernet, Siemens dominant.
*   **MQTT:** Level 2-3, Lightweight publish/subscribe, common for IIoT.

OT5 — **Data Diode vs Firewall**
*   **Firewall:** Filters traffic bidirectionally. Can be bypassed or misconfigured.
*   **Data Diode:** Hardware-enforced unidirectional flow (optical/electrical).
*   **Use:** When data must flow OT -> IT but IT must never touch OT (e.g., sensitive control data).
*   **Limitation:** Cannot send commands back to OT; requires specific hardware.

OT6 — **Data Flow to Cloud**
*   **Flow:** PLC/RTU -> Historian (Level 3) -> OPC UA Server -> Edge Gateway (MQTT) -> Cloud Ingestion (IoT Core).
*   **Security:** Firewall between Level 2/3 and Edge. Edge Gateway authenticates to Cloud.
*   **Boundary:** DMZ between OT and IT/Cloud. Edge device acts as a secure proxy.

OT7 — **IEC 62443 Zones & Conduits**
*   **Zone:** Logical grouping of assets with similar security requirements (e.g., Production, Engineering).
*   **Conduit:** Communication path between zones.
*   **Design:** Corporate (Zone 4) -> DMZ (Zone 3) -> SCADA (Zone 2) -> PLCs (Zone 1).
*   **SIS:** Isolated Zone (Safety).
*   **Conduits:** Firewalls control traffic between zones (e.g., Engineering Workstation to PLC).

OT8 — **Network Segmentation Strategy**
*   **VLANs:** Segment by function (Production, Engineering, Maintenance).
*   **Firewalls:** Place between VLANs and at DMZ.
*   **Switches:** Managed Layer 2/3 switches with ACLs.
*   **Remote Access:** Jump hosts in DMZ, MFA, session recording.
*   **Transition:** Phase migration during planned shutdowns; use temporary parallel networks; validate safety systems before cutting over.

OT9 — **SIS vs BPCS**
*   **BPCS:** Basic Process Control System (PLCs/DCS) for normal operation.
*   **SIS:** Safety Instrumented System (e.g., ESD) for emergency shutdown.
*   **IEC 61511:** Standard for safety lifecycle.
*   **Isolation:** SIS must be air-gapped to prevent cyber attacks from disabling safety functions.
*   **Ransomware:** If SIS is compromised, safety shutdowns may fail, leading to physical damage or injury.

OT10 — **PLC, DCS, SCADA**
*   **PLC:** Localized control for specific machines.
*   **DCS:** Distributed control for continuous processes (refineries).
*   **SCADA:** Supervisory control over wide geographic areas (pipelines).
*   **Coexistence:** Yes. PLCs often feed into SCADA; DCS may contain PLCs.

OT11 — **Remote Access Architecture**
*   **Jump Hosts:** Isolated bastion hosts in DMZ.
*   **MFA:** Required for all access.
*   **Session Recording:** Audit trail of all actions.
*   **Vendor Access:** Time-limited, role-based, no persistent credentials.
*   **RDP:** Direct RDP to HMI is unacceptable (exposes OT directly to IT threats).

OT12 — **Nation-State Incident Response**
*   **Monitoring:** IDS/Anomaly detection on OT traffic (e.g., unexpected Modbus commands).
*   **Inventory:** Know all assets to identify compromised ones.
*   **Steps:** Isolate network segments, preserve evidence, notify regulators.
*   **Coordination:** Control room prioritizes safety over IT containment.
*   **Operator Role:** Manual override if automation is compromised; verify physical safety.

OT13 — **HMI**
*   **Definition:** Interface for operators to monitor and control processes.
*   **Role:** Visualization, alarm management, manual control.
*   **Vs Engineering Workstation:** HMI is for operations (Level 2); Engineering Workstation is for configuration/programming (Level 3).

OT14 — **Historian**
*   **Definition:** Time-series database for process data.
*   **Importance:** Trend analysis, compliance, troubleshooting.
*   **Plant Historian:** Level 3, real-time process data.
*   **Enterprise Historian:** Level 4, aggregated data for business analytics.

OT15 — **Ladder Logic**
*   **Definition:** Graphical programming language resembling electrical relay circuits.
*   **Device:** PLCs.
*   **Dominance:** Familiar to electricians, deterministic execution, robust for decades.

OT16 — **RTU vs PLC**
*   **RTU:** Remote Terminal Unit, rugged, long-distance comms, low power.
*   **PLC:** Local control, high I/O density, faster processing.
*   **RTU Use:** Pipelines, SCADA sites, harsh environments, remote locations.

OT17 — **Analog vs Digital Signals**
*   **Analog:** Continuous voltage/current (4-20mA).
*   **Digital:** Discrete on/off or binary.
*   **4-20mA:** 4mA indicates "live zero" (wire break detected at 0mA). 0-20mA cannot distinguish break from zero signal.

OT18 — **OT Asset Inventory**
*   **Foundation:** You cannot secure what you don't know exists.
*   **Tools:** Passive monitoring (packet capture), Active scanning (ping/scan).
*   **Active Scanning Risk:** Can crash legacy devices or disrupt real-time processes. Passive is safer.

OT19 — **MITRE ATT&CK for ICS**
*   **Tactics:** Initial Access, Execution, Persistence, Privilege Escalation, Defense Evasion.
*   **Techniques:** Spearphishing (Initial Access), Exploitation of Remote Services (Execution), Data Exfiltration.
*   **Difference:** Focuses on physical impact and industrial protocols, not just data theft.
*   **Usage:** Map defenses to tactics to find gaps (e.g., no monitoring for Modbus commands).

OT20 — **Converged IT/OT SOC**
*   **Log Collection:** Passive taps or agents that don't impact latency.
*   **SIEM:** Correlate IT and OT logs.
*   **Rules:** OT-specific (e.g., PLC write attempts).
*   **Staffing:** OT engineers for triage (understand safety), IT analysts for threat intel.
*   **False Positives:** Tuned to avoid nuisance alarms that disrupt production.
*   **Firmware Alert:** Isolate PLC, verify with engineering, do not reboot without safety check.
