T1 — Easy: `tofu plan` calculates the execution plan, showing what changes will be made without applying them. `tofu apply` executes the plan, creating, updating, or destroying resources to match the configuration.

T2 — Easy: State is a JSON file mapping real-world resources to your configuration. Remote state is important because it allows multiple users to work on the same project safely (via locking), enables sharing state between modules, and prevents data loss if the local machine fails.

T3 — Medium: `count` uses integer indices; `for_each` uses a map or set of strings. `for_each` is preferred when resources are identified by unique names (e.g., usernames, IDs) because adding or removing items doesn't shift indices, preventing unintended resource recreation.

T4 — Medium: This is called "drift." OpenTofu detects it during `plan` by comparing the state file to the live infrastructure. If drift is detected, `apply` will attempt to reconcile the configuration to match the state, potentially recreating the resource if the drift is significant.

T5 — Medium:
```hcl
resource "aws_instance" "web" {
  for_each      = toset(["web1", "web2", "web3"])
  ami           = "ami-12345678"
  instance_type = "t2.micro"
  tags = {
    Name = each.key
  }
}
```

T6 — Hard: `terraform_remote_state` reads the raw state file of another module (exposes all data, security risk). `output` values expose only specific data via the module interface. Recommended: Use module outputs. Why: It enforces abstraction, security (don't expose full state), and versioning compatibility.

T7 — Hard: The `null_resource` provisioner runs during the apply phase, but the output is evaluated before the file is written. Fix: Use a `local_file` resource to write the kubeconfig, and have the `null_resource` trigger the `local_file` recreation. Output the `local_file` content instead of relying on the provisioner directly.

T8 — Medium: `tofu import` is a CLI command to add existing resources to state. Limitations: Requires manual mapping of arguments, no support for complex dependencies. Import block (1.5+) improves this by allowing declarative import definitions within the configuration, enabling version control of import logic and better dependency management.

T9 — Easy: A variable is an input parameter for the configuration. `variable` blocks define inputs passed from outside (CLI, tfvars). `output` blocks define values exposed from the configuration to be used by other modules or the CLI.

T10 — Easy: A provider is a plugin that interacts with a specific API (e.g., AWS, Azure). Running `tofu plan` without `tofu init` fails because the provider binaries and plugins are not downloaded or initialized.

T11 — Easy: `.terraform.lock.hcl` records the exact provider versions and checksums used. It should be committed to ensure reproducible builds and prevent breaking changes from provider updates.

T12 — Easy: `tofu destroy` removes all resources managed by the configuration. To target a single resource: `tofu destroy -target=aws_instance.example`.

T13 — Easy: A data source reads information from an external system without creating or managing it. Use one when you need to look up existing data (e.g., `data "aws_ami" "ubuntu"` to find the latest Ubuntu AMI) rather than creating a new resource.

T14 — Medium: `locals` define computed values within the configuration scope. `variable` blocks define inputs from outside. Use `locals` for intermediate calculations or constants that don't need to be overridden by users.

T15 — Medium: `lifecycle` controls resource behavior. `create_before_destroy`: Creates new resource before destroying old (zero downtime). `prevent_destroy`: Prevents accidental deletion (e.g., production DB). `ignore_changes`: Ignores external modifications (e.g., tags managed by another tool).

T16 — Hard: A `moved` block allows renaming resources or moving them between modules without changing the state file. It helps refactor code by telling OpenTofu that the new address refers to the existing state ID, avoiding recreation.

T17 — Hard: The dependency graph determines execution order. Implicit dependencies are inferred from references (e.g., `aws_instance.id`). Explicit dependencies use `depends_on`. `depends_on` forces serialization, reducing parallelism. Use sparingly to avoid bottlenecks.

T18 — Hard: 1. **Remote State Backend**: Enables parallelism and faster state access. 2. **Split State**: Break monolithic state into smaller modules/states to reduce parsing time. 3. **Reduce Data Sources**: Cache lookups or minimize external API calls during plan. 4. **`-refresh=false`**: Skip state refresh if drift is managed elsewhere, speeding up plan.

T19 — Hard: State locking prevents concurrent writes. If a lock isn't released (crash), subsequent runs hang. Manually break lock with `tofu force-unlock <LOCK_ID>`. Risk: Data corruption if another process is actually writing during the unlock.

T20 — Hard: 1. Add `backend` block to config. 2. Run `tofu init -migrate-state`. 3. OpenTofu copies local state to S3. 4. Verify with `tofu state list`. Ensure DynamoDB table exists for locking.

A1 — Easy: A **Task** is a single action (e.g., install package). A **Role** is a reusable collection of tasks, files, and templates. A **Playbook** is a YAML file defining a list of plays (execution flow) that run roles/tasks on hosts.

A2 — Easy: `ansible_facts` contains gathered system information. Access OS family via `{{ ansible_facts['os_family'] }}` or `{{ ansible_os_family }}`.

A3 — Medium: `command` runs without shell features (no pipes). `shell` runs with shell features (pipes, variables). `raw` runs without setup (no Python module), useful for bootstrapping SSH. Use `command` for security, `shell` for scripting, `raw` for initial setup.

A4 — Medium: Idempotency means running a task multiple times yields the same result. **Idempotent**: `apt: name=nginx` (installs only if missing). **Not Idempotent**: `shell: rm -rf /tmp/*` (deletes every time).

A5 — Medium:
```yaml
- name: Install htop on Debian
  package:
    name: htop
    state: present
  when: ansible_os_family == "Debian"

- name: Install epel-release and htop on RedHat
  package:
    name:
      - epel-release
      - htop
    state: present
  when: ansible_os_family == "RedHat"
```

A6 — Hard: Highest priority: Extra vars (CLI). Then: Play vars, Host vars, Group vars, Role vars, Role defaults. Lowest: Hardcoded values.

A7 — Hard: Use `serial: 3` in the play to limit concurrency. Group nodes into `agents` and `first_batch`. Use `wait_for` to ensure the first 3 nodes are ready before the rest start, or use `serial` with `max_fail_percentage` to control rollout.

A8 — Medium: `ansible-vault` encrypts sensitive data. To encrypt a single variable: `ansible-vault edit vars.yml` (encrypts file) or use `!vault` inline in YAML for specific values.

A9 — Easy: Inventory lists managed hosts. **Static**: Defined in a file (INI/YAML). **Dynamic**: Generated by a script or API (e.g., AWS EC2) to reflect changing infrastructure.

A10 — Easy: `ansible webservers -m shell -a "df -h"`.

A11 — Easy: A **Handler** is a task that runs only when notified by another task (e.g., `notify: restart service`). It differs from a regular task by running at the end of the play, not immediately. Triggered via `notify: handler_name`.

A12 — Easy: `ansible.cfg` configures Ansible behavior. Common settings: `host_key_checking` (disable), `forks` (parallelism), `remote_user` (default SSH user).

A13 — Easy: `register` stores task output in a variable. Use it in subsequent tasks via `{{ registered_var.stdout }}` or `{{ registered_var.failed }}`.

A14 — Medium: `import_tasks` is static (evaluated at parse time). `include_tasks` is dynamic (evaluated at runtime). `include_tasks` supports `when` on the include line; `import_tasks` does not.

A15 — Medium: Ansible Galaxy is a repository for roles/collections. Use `requirements.yml` to list them, then run `ansible-galaxy install -r requirements.yml` to download.

A16 — Medium: Connection plugins manage communication. **ssh**: Standard remote Linux/Unix. **local**: Runs on control node. **network_cli**: For network devices (Cisco, Juniper) via CLI.

A17 — Hard: `delegate_to: localhost` runs task on control node. `run_once: true` runs task once regardless of host count. Example: Gather facts from remote, then write a config file to control node for distribution.

A18 — Hard: 1. **Forks**: Increase `forks` in config for parallelism. 2. **Pipelining**: Reduces SSH connections (faster). 3. **Fact Caching**: Store facts to avoid gathering on every run. 4. **Mitogen**: Optimized SSH transport for massive scale.

A19 — Hard: `{{ }}` evaluates expressions. `{% %}` executes logic (loops, ifs). `{# #}` comments.
```jinja
upstream backend_servers {
{% for server in backend_servers %}
    server {{ server.ip }} weight={{ server.weight }}; # Health: {{ server.health_check }}
{% endfor %}
}
```

A20 — Hard: 1. Set `serial: 5` for batches. 2. Pre-task: Deregister from LB. 3. Deploy app. 4. Post-task: Health check (`uri` module). 5. Register to LB. 6. `block/rescue`: If health check fails, trigger rollback (revert config, re-register LB).
