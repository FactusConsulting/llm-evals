G1 — Easy: An array is a fixed-size value type (`[N]T`), while a slice is a dynamic reference type (`[]T`) containing a pointer to an underlying array, length, and capacity. When appending to a slice that reaches its capacity, the Go runtime allocates a new, larger backing array (typically doubling capacity), copies existing elements, and updates the slice header.

G2 — Easy: `var x int` declares a variable with an explicit type and zero value. `x := 0` is a short variable declaration that infers the type. You cannot use `:=` if the variable already exists in the current scope (e.g., redeclaring `x`), or if you are declaring a variable outside a function (e.g., package level).

G3 — Medium: A goroutine is a lightweight thread managed by the Go runtime, not the OS kernel. OS threads are heavier and managed by the kernel. The Go scheduler uses the GMP model: **G** (Goroutine), **M** (OS Thread), **P** (Processor context). The scheduler multiplexes many Gs onto fewer Ms using Ps to manage work queues and prevent blocking.

G4 — Medium: Channels are typed conduits for communication between goroutines. Unbuffered channels block both sender and receiver until the other side is ready. Buffered channels have a fixed capacity; sending blocks only if the buffer is full. Receiving blocks only if the buffer is empty.

G5 — Medium: The `context` package manages deadlines, cancellations, and request-scoped values.
```go
ctx, cancel := context.WithTimeout(context.Background(), 1*time.Second)
defer cancel()
req, _ := http.NewRequestWithContext(ctx, "GET", url, nil)
client.Do(req) // Cancels if timeout exceeded
```

G6 — Medium: Go interfaces are satisfied implicitly (duck typing), unlike Java/C# which require explicit declaration. The empty interface `interface{}` (or `any`) accepts any type. It is powerful for generic containers but dangerous because it requires type assertions at runtime, which can panic if the type is wrong.

G7 — Hard: This is unsafe due to a data race. The main goroutine reads `data` without synchronization while the worker writes to it. The `ready` flag does not guarantee visibility of `data` due to memory reordering. Correct alternatives: use `sync/atomic` for flags, `sync.Mutex` to protect `data`, or a channel to pass `data` safely.

G8 — Medium: `sync.Mutex` provides exclusive write access. `sync.RWMutex` allows multiple readers or one writer. `sync.Map` is optimized for read-heavy workloads with sparse keys. Trade-off: `sync.Map` has higher overhead for writes and lacks iteration safety compared to a regular `map` protected by a `Mutex`.

G9 — Hard: Go modules manage dependencies. `go.mod` lists direct dependencies and versions. `go.sum` stores cryptographic checksums for reproducibility. `go mod tidy` removes unused dependencies and adds missing ones. MVS (Minimum Version Selection) selects the lowest version of a dependency that satisfies all requirements in the dependency tree.

G10 — Medium: `defer` postpones function execution until the surrounding function returns. Multiple defers execute in LIFO (Last In, First Out) order. A common gotcha in loops: `defer` captures the loop variable by reference, not value. Use a copy variable inside the loop to capture the current value.

G11 — Easy: A pointer holds a memory address. Its zero value is `nil`. To create a pointer to a struct literal, use the address-of operator: `&MyStruct{Field: value}`.

G12 — Easy: `init()` is a function run automatically before `main()`. It runs once per package. A package can have multiple `init()` functions; they execute in the order they appear in the source file.

G13 — Easy: `make` is for built-in types (slice, map, channel) and returns the initialized type. `new` is for all types and returns a pointer to the zero value of that type.

G14 — Easy: Errors are returned as values. The `error` interface has a single method `Error()`. Idiomatic pattern: `if err != nil { return err }`. `errors.Is` checks if an error matches a target (wrapping). `errors.As` checks if an error matches a specific type (type assertion).

G15 — Easy: Built-in composite types are `map`, `slice`, `channel`, `struct`. Zero values: `map`/`channel`/`slice` are `nil`. `struct` fields are their respective zero values (e.g., `0` for int, `""` for string).

G16 — Medium: A race condition occurs when two goroutines access shared data concurrently, with at least one write, without synchronization. The `-race` flag adds instrumentation to detect these. Fix: Use `sync.Mutex` to lock access or channels to pass ownership.

G17 — Medium: `select` waits on multiple channel operations. Unlike `switch` (which evaluates expressions), `select` evaluates channel readiness. If multiple cases are ready, one is chosen pseudo-randomly. Timeout is implemented by adding a `case <-time.After(duration)`.

G18 — Hard: Escape analysis determines if a variable lives on the stack or heap. If a variable's address escapes the function scope, it is allocated on the heap. Inspect with `go build -gcflags="-m"`. Heap allocation increases GC pressure and can impact performance.

G19 — Hard: `reflect` allows runtime introspection. `reflect.Type` describes the type; `reflect.Value` holds the value. Reflection is slow (no compile-time checks) and bypasses type safety. Justified for serialization, ORM, or framework code where static typing is insufficient.

G20 — Hard: `go:embed` embeds files/directories at compile time. Usage: `//go:embed path`. It works with single files, directories, or `//go:embed *`. It interacts with `io/fs.FS` (e.g., `embed.FS`). Limitations: files must exist at compile time; cannot embed runtime-generated content.

R1 — Easy: Ownership is Rust's core memory safety feature. Rules: 1) Each value has one owner. 2) When the owner goes out of scope, the value is dropped. 3) Values can be moved, but not copied (unless `Copy` trait).

R2 — Easy: `String` is an owned, heap-allocated, growable buffer. `&str` is a borrowed, immutable slice of UTF-8 bytes. Use `&str` for function arguments (zero cost), `String` when you need to own or mutate the data.

R3 — Medium: Borrowing allows temporary access without ownership. Rules: You can have either one mutable reference (`&mut T`) OR any number of immutable references (`&T`), but not both simultaneously. References must always be valid.

R4 — Medium: `Result<T, E>` represents success or failure. `Option<T>` represents a value or `None`. The `?` operator propagates errors up the call stack.
```rust
fn read_file() -> Result<String, std::io::Error> {
    std::fs::read_to_string("file.txt")
}
```

R5 — Medium: `Box<T>` allocates on heap with single ownership. `Rc<T>` (Reference Counted) allows multiple owners in single-threaded contexts. `Arc<T>` (Atomic Reference Counted) allows multiple owners across threads.

R6 — Medium: Traits define shared behavior. Unlike Go interfaces, traits can have default implementations. `dyn Trait` uses dynamic dispatch (vtable, runtime cost). `T: Trait` uses static dispatch (monomorphization, zero cost).

R7 — Hard: The function fails because the compiler cannot determine which input reference (`x` or `y`) the returned reference should live for. Fix: Add lifetime annotations.
```rust
fn longest<'a>(x: &'a str, y: &'a str) -> &'a str {
    if x.len() > y.len() { x } else { y }
}
```

R8 — Hard: `Send` means a type can be transferred to another thread. `Sync` means a type can be referenced from another thread. `Rc<T>` is not `Send` because its reference count is not atomic. `Arc<T>` uses atomic reference counting, making it `Send` and `Sync`.

R9 — Medium: `async/await` creates a state machine (`Future`). The executor (e.g., Tokio) polls the future until completion. Unlike Go's preemptive goroutines, Rust's async is cooperative; the task yields control explicitly.

R10 — Medium: `unsafe` allows bypassing compiler safety checks. Actions: dereference raw pointers, call unsafe functions, access mutable statics, modify unions. Justified for FFI, low-level hardware access, or performance-critical optimizations where safety is manually verified.

R11 — Easy: Pattern matching (`match`) exhaustively checks all possible values. `if let` matches a single pattern. The compiler enforces exhaustiveness in `match` to prevent logic errors from unhandled cases.

R12 — Easy: `Vec<T>` is a growable heap-allocated array. `[T; N]` is a fixed-size stack-allocated array. When `Vec` capacity is exceeded, it allocates a new larger buffer and moves elements (reallocates).

R13 — Easy: Enums (ADTs) can hold data in variants. Unlike C/Java, Rust enums are powerful types. `Option<T>` is `enum { Some(T), None }`. Rust has no `null`; `None` is a type-safe alternative, preventing null pointer exceptions.

R14 — Easy: `impl` defines methods. `&self` borrows the instance. `&mut self` mutably borrows. `self` consumes the instance. Associated functions (e.g., `String::new()`) do not take `self` and act like constructors.

R15 — Easy: `derive` macros automatically implement traits. Common: `Debug`, `Clone`, `PartialEq`. Cannot derive if fields contain types that don't support the trait (e.g., `Fn` closures cannot be cloned).

R16 — Medium: `Cow<'a, T>` (Clone on Write) holds either a borrowed reference or an owned value. It avoids allocation if the data is only read. If modification is needed, it clones the data internally. Useful for parsing or string manipulation.

R17 — Medium: `mod` declares a module. `use` imports paths. `pub` makes items public. File system maps to modules: `mod.rs` or `lib.rs`. Rust 2018 allowed `mod foo;` to refer to `foo.rs` directly, simplifying structure compared to 2015.

R18 — Hard: `Pin<T>` prevents a value from being moved in memory. Needed for self-referential structs and `Future` implementations that hold references to their own fields. `Unpin` is a marker trait for types that can be safely moved.

R19 — Hard: Declarative macros (`macro_rules!`) transform tokens. Procedural macros (derive, attribute) operate on Rust code via the compiler API. Hygiene ensures macro variables don't conflict with local variables.

R20 — Hard: Zero-cost abstractions mean high-level code compiles to efficient machine code. Generics use monomorphization (code duplication per type). Iterators are inlined. This provides ergonomics without runtime overhead compared to C.
