AA1 — **Monolith vs. Microservices:** A monolith is a single deployable unit where all components are tightly coupled. Microservices are independently deployable, loosely coupled services. **Problems introduced by microservices:** 1. **Distributed Complexity:** Network latency, partial failures, and distributed tracing are required. 2. **Data Consistency:** ACID transactions across services are difficult (requires Sagas). 3. **Operational Overhead:** Requires orchestration (Kubernetes), CI/CD pipelines, and monitoring for each service.

AA2 — **Sync vs. Async:** Synchronous communication blocks the caller until a response is received (request/response). Asynchronous communication allows the caller to continue without waiting (fire-and-forget or callback). **Examples:** HTTP/REST (Sync), RabbitMQ or Kafka (Async).

AA3 — **API Gateway vs. Reverse Proxy:** A reverse proxy forwards traffic to backend servers (L4/L7). An API Gateway adds business logic like authentication, rate limiting, and API versioning. **Replacement:** An API Gateway cannot fully replace a reverse proxy in all cases; a reverse proxy is often needed for pure L4 load balancing or SSL termination before the Gateway, though modern Gateways often combine both roles.

AA4 — **Saga Pattern:** A sequence of local transactions where each updates the database and triggers the next. **Choreography:** Services react to events (loose coupling, harder to debug). **Orchestration:** A central coordinator directs the flow (tighter control, single point of failure risk). **Use:** Choreography for simple flows; Orchestration for complex, long-running workflows requiring compensation logic.

AA5 — **CQRS:** Separates read (Query) and write (Command) models. **Paired with Event Sourcing:** Event Sourcing stores state changes as an immutable log. **Problem Solved:** Allows independent scaling of read/write workloads and provides a complete audit trail, which traditional CRUD (single model) cannot efficiently support.

AA6 — **Circuit Breaker:** Prevents cascading failures by stopping calls to a failing service. **States:** Closed (normal), Open (fail fast), Half-Open (testing recovery). **Difference:** Retry with backoff attempts to recover immediately; Circuit Breaker stops attempts after a threshold. **Use Together:** Use Retry for transient network glitches, Circuit Breaker for persistent service outages.

AA7 — **Multi-tenant Isolation:**
1. **Shared DB/Schema:** Lowest cost, highest noisy-neighbor risk, complex security (row-level).
2. **Shared DB/Schema:** Medium cost, better isolation, moderate complexity.
3. **Separate DB:** Highest cost, best security/isolation, highest operational complexity.
**Trade-off:** Choose based on compliance needs and budget.

AA8 — **Consistency:** Strong consistency ensures immediate visibility of updates; eventual consistency allows temporary divergence. **Architecture:** Use Strong Consistency. **Data Store:** Relational DB (e.g., PostgreSQL) with row-level locking or a distributed lock manager (Redis). **Pattern:** Implement a reservation system (deduct inventory on hold) to prevent overselling before payment.

AA9 — **Sidecar Proxy:** A helper container deployed alongside the main service. **Envoy/Istio:** Envoy handles traffic (data plane); Istio control plane configures Envoy. **mTLS:** Solves service-to-service identity verification and encryption without modifying application code, unlike app-level TLS which requires code changes.

AA10 — **Strangler Fig:** Incrementally migrate by routing specific traffic to new services. **Strategy:** Deploy .NET 8 services for new features. Configure API Gateway to route requests for new features to .NET 8 and legacy requests to the .NET Framework monolith. Gradually move functionality until the monolith is removed.

AA11 — **12-Factor App:** 1. **Codebase** (one repo), 2. **Dependencies** (explicitly declared), 3. **Config** (env vars), 4. **Backing Services** (treated as attached resources), 5. **Build/Run/Execute** (separate), 6. **Processes** (stateless), 7. **Port Binding** (export via env), 8. **Concurrency** (scale via processes), 9. **Disposability** (fast start/stop), 10. **Dev/Prod Parity**, 11. **Logs** (streams), 12. **Admin Processes** (run as one-off).

AA12 — **Event-Driven vs. Request-Driven:** Event-driven (Kafka) offers loose coupling, high scalability, and ordering guarantees but higher debugging complexity. Request-driven (REST) offers tight coupling, easier debugging, and synchronous flow but lower scalability. **Choice:** Use Event-driven for decoupled, high-volume data processing; Request-driven for real-time user interactions.

AA13 — **Load Balancer:** Distributes traffic across servers. **L4:** Routes based on IP/Port (TCP/UDP). **L7:** Routes based on HTTP headers/URL. **Use Case:** SSL Termination or routing `/api` to backend services and `/static` to CDN.

AA14 — **Message Queue vs. Log Broker:** MQ (RabbitMQ) deletes messages after consumption (tasks). Log Broker (Kafka) retains messages for replay (streaming). **Pick MQ:** For task processing (email, jobs). **Pick Kafka:** For event streaming, audit logs, or data pipelines.

AA15 — **Container vs. VM:** Containers share the host OS kernel; VMs virtualize hardware with a full guest OS. **Popularity:** Containers are lighter, start faster, and have higher density, ideal for microservices.

AA16 — **Stateless vs. Stateful:** Stateless services store no session data (e.g., Web Server). Stateful services store data (e.g., Database). **Scaling:** Stateless is easier to scale horizontally because any instance can handle any request without affinity.

AA17 — **ORM:** Maps objects to database tables. **Examples:** Hibernate (Java), Entity Framework (.NET). **Trade-offs:** ORMs increase development speed but can generate inefficient SQL and obscure performance tuning compared to raw SQL.

AA18 — **Bulkhead Pattern:** Isolates resources (threads, connections) per service. **Implementation:** Use separate thread pools for each downstream dependency. If one service fails, it exhausts only its pool, preventing a cascade that takes down the whole application.

AA19 — **OT vs. CRDT:** OT requires a central server to resolve conflicts (high latency, no offline). CRDTs allow conflict-free merging on clients (low latency, offline support). **Choice:** CRDTs for offline editing support, as they guarantee convergence without server intervention.

AA20 — **Distributed Tracing:** **Propagation:** Inject trace IDs into headers (W3C/B3). **Sampling:** Head-based (start) vs. Tail-based (end, based on latency). **Storage:** Jaeger/Tempo. **Kafka:** Inject trace context into message headers (e.g., `traceparent`) so consumers can continue the trace.

OP1 — **Hypervisor:** Type 1 (Bare-metal) runs directly on hardware (e.g., VMware ESXi, Microsoft Hyper-V). Type 2 (Hosted) runs on an OS (e.g., VirtualBox, VMware Workstation). Type 1 is for production; Type 2 is for development.

OP2 — **SAN vs. NAS:** SAN provides block storage (e.g., iSCSI, Fibre Channel) for databases. NAS provides file storage (e.g., NFS, SMB) for file shares. **Use SAN:** For high-performance DBs. **Use NAS:** For shared user files.

OP3 — **3-Tier vs. Leaf-Spine:** 3-Tier (Core/Dist/Access) has bottlenecks at the distribution layer. Leaf-Spine is flat, providing non-blocking bandwidth and lower latency, solving oversubscription issues in modern data centers.

OP4 — **Proxmox HA:** 3+ nodes for quorum. **Storage:** Ceph or shared iSCSI. **Fencing:** DRAC/ILO to power off failed nodes. **Failure:** Quorum detects failure, fencing isolates node, VMs restart on healthy nodes.

OP5 — **VLAN Trunking:** 802.1Q tags frames with VLAN IDs. **Linux Config:** Create sub-interfaces (e.g., `eth0.10`, `eth0.20`) with `ip link add` and assign IPs. Switch port must be configured as a trunk.

OP6 — **RAID:** RAID 1 (Mirror), 5 (Striped+Parity), 6 (Double Parity), 10 (Mirror+Striped), Z2 (ZFS Double Parity). **Choice:** RAID 10 for 6 drives. Best balance of read/write performance and fault tolerance (can lose 2 drives if mirrored correctly).

OP7 — **DNS Architecture:** Internal AD DNS for clients. External Public DNS for web. **Split-Horizon:** Internal queries resolve internal IPs. **Forwarders:** Internal DNS forwards external to public resolvers. **Laptop:** DHCP updates DNS records; VPN tunnel routes internal DNS queries to AD.

OP8 — **PKI:** Root CA (offline) signs Subordinate CA (online), which signs End-Entity certs. **Compromise:** Subordinate compromise = Revoke sub, issue new. Root compromise = Trust collapse, must rebuild entire PKI. **Auto-enrollment:** Windows AD CS auto-provisions certs.

OP9 — **WAN:** MPLS (reliable, expensive), IPsec (cheap, variable latency), SD-WAN (dynamic routing, cost-effective), WireGuard (simple, secure). **Recommendation:** SD-WAN. Best for VoIP (QoS), cost management, and ease of management across 5 sites.

OP10 — **iDRAC/iLO/IPMI:** Out-of-band management interfaces. **Critical:** Allows server control even if OS is down. **Exposed:** If exposed to internet, attackers can gain root access, power cycle, or install malware.

OP11 — **LVM vs. ZFS:** LVM manages block devices (flexible resizing). ZFS manages filesystems (integrity, snapshots). **LVM-Thin:** Overprovisioning. **ZFS:** Copy-on-write snapshots. **Use ZFS:** For data integrity and snapshots. **Use LVM:** For simple block management.

OP12 — **Stretch Cluster:** 2 sites + 1 tiebreaker (witness). **Ceph:** Monitors must be distributed. **Latency:** <10ms between sites. **Split-Brain:** Without tiebreaker, both sites think they are primary. **Bad Idea:** 2-site Ceph without witness risks data corruption.

OP13 — **DHCP:** DORA = Discover, Offer, Request, Acknowledge. **Unavailable:** Client uses APIPA (169.254.x.x) or retries lease. **Expired:** Client attempts renewal; if failed, releases IP.

OP14 — **Managed vs. Unmanaged:** Unmanaged = Plug-and-play. Managed = VLANs, QoS, ACLs, Port Mirroring. **Importance:** Managed switches allow segmentation, traffic prioritization (VoIP), and security policies.

OP15 — **NTP:** Network Time Protocol. **Critical:** Certificates, logs, and cluster consensus rely on time. **Drift:** Causes authentication failures (Kerberos), log analysis issues, and split-brain in clusters.

OP16 — **Physical vs. VM vs. Container:** Physical = Bare metal. VM = Virtualized OS. Container = Virtualized App. **Bare Metal:** Use for high-performance DBs, HPC, or latency-sensitive workloads where virtualization overhead is unacceptable.

OP17 — **UPS vs. Generator:** UPS provides short-term power (minutes) for graceful shutdown or bridge. Generator provides long-term power (hours). **ATS:** Automatically switches load from utility to generator when power fails.

OP18 — **Active-Passive vs. Active-Active:** Active-Passive (one handles load, one waits). Active-Active (both handle load). **PostgreSQL HA:** Pacemaker/Corosync manages Virtual IP. **STONITH:** Fence failed node to prevent split-brain. **Failover:** VIP moves to standby node.

OP19 — **Backup Strategy:** 3-2-1 Rule (3 copies, 2 media, 1 offsite). **RPO/RTO:** Define acceptable data loss and downtime. **Immutable:** Protects against ransomware. **Test:** Regularly restore to verify integrity. **Backup vs. DR:** Backup is data copy; DR is the plan to recover operations.

OP20 — **802.1X:** **Roles:** Supplicant (client), Authenticator (switch), Server (RADIUS). **EAP:** EAP-TLS (certs, secure), PEAP (user/pass). **Dynamic VLAN:** Assigns VLAN based on user role. **Fail:** Port blocked or placed in guest VLAN. **MAB:** Switch sends MAC address to RADIUS for authentication if 802.1X fails (for printers).
