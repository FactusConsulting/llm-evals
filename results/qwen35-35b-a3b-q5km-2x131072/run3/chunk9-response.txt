SC1-A
The pods are failing because the **Restricted Pod Security Standard (PSS)** forbids running containers as root (UID 0).
1.  **Image Behavior:** The new base image (`cgr.dev/chainguard/static:latest`) defaults `USER` to root.
2.  **Manifest Behavior:** The manifest sets `securityContext.runAsNonRoot: true` but omits `runAsUser`. Kubernetes defaults `runAsUser` to the image's `USER` directive if not explicitly set.
3.  **PSS Enforcement:**
    *   **Baseline:** Allows `runAsNonRoot: true` without `runAsUser` (defaults to image user, even if root).
    *   **Restricted:** Requires `runAsNonRoot: true` **AND** `runAsUser` to be explicitly set to a non-zero value. It also forbids `privileged: true` and requires specific capabilities.
4.  **Failure:** The admission controller rejects the pod spec because it effectively runs as root (UID 0) despite the `runAsNonRoot` flag, violating the Restricted profile's requirement for explicit non-root UID configuration.

SC1-C
**Remediation Strategy:**
1.  **Image Fix:** Rebuild the base image with a non-root user (e.g., `USER 1000`) or use a pre-built non-root variant.
2.  **Manifest Fix:** Explicitly set `securityContext.runAsUser: 1000` in the Pod/Deployment spec to satisfy Restricted PSS.
3.  **CI Gate:** Integrate `kube-score` or `checkov` into the CI pipeline to validate manifests against the Restricted PSS profile before deployment.
4.  **Rollout:** Use a **Canary Deployment**. Deploy the fixed version to 5% of traffic first. Monitor CrashLoopBackOff rates and latency. If stable, ramp to 100%. This ensures immediate rollback capability without downtime.

SC2-A
**Stale Data & Replication Lag:**
1.  **Cause:** Cross-region replication is asynchronous. When the primary (us-east-1) fails, the replica (eu-west-1) is promoted. Any WAL (Write-Ahead Log) entries generated on the primary but not shipped to the replica before the outage are lost.
2.  **WAL Shipping:** PostgreSQL streams WAL records. In async mode, the primary acknowledges the write before the replica confirms receipt.
3.  **Data Loss:** The "data loss window" is the time between the last committed transaction on the primary and the last shipped WAL record on the replica.
4.  **Quantification:** Check `pg_stat_replication` on the primary (if accessible) or the replica's `pg_wal` logs. Look for `replay_lag` and `write_lag` metrics prior to failure. Check application logs for transaction IDs to estimate the gap.

2.  **Update State:** If the old instance ARN is hardcoded in state, use `terraform state mv` to point the resource to the new ARN, or update the Terraform config to reference the new ARN and run `terraform import`.

*Command:* `terraform import aws_db_instance.new_primary arn:aws:rds:eu-west-1:123456789:db:promoted-replica` followed by `terraform apply`.

SC2-C
**Architecture Design:**
1.  **RDS Proxy:** Use RDS Proxy instead of PgBouncer to handle connection pooling and failover transparently at the application level.
2.  **Route53:** Configure **Latency-Based Routing** with Health Checks. Point the primary endpoint to us-east-1 and the secondary to eu-west-1. If us-east-1 fails health checks, Route53 routes to eu-west-1.
3.  **Retry Logic:** Implement exponential backoff with jitter in the application connection pool.
4.  **RPO < 1s:** Standard RDS async replication cannot guarantee <1s RPO. **Recommend Aurora Global Database.** It uses a dedicated replication layer with sub-second latency and automatic failover, ensuring minimal data loss.
5.  **Why Aurora:** Native cross-region replication is faster and more consistent than standard RDS streaming replication.

SC6-C
**Connectivity Comparison:**
1.  **AWS Direct Connect:**
    *   *Pros:* High bandwidth (10Gbps+), low latency, dedicated.
    *   *Cons:* High cost, physical hardware setup.
    *   *Zero Trust:* Fits well (dedicated link), but requires BGP routing.
2.  **VPN ECMP:**
    *   *Pros:* Low cost, easy setup.
    *   *Cons:* Limited bandwidth (1Gbps x 2), higher latency, encryption overhead.
    *   *Zero Trust:* IPsec tunnels are stateful; harder to enforce zero-trust at packet level.
3.  **Hybrid Mesh (Tailscale/Netbird):**
    *   *Pros:* Zero-trust by design (WireGuard), no IP management, easy failover.
    *   *Cons:* Performance depends on relay nodes (unless using P2P), cost at scale.
    *   *Zero Trust:* Best fit. Identity-based access control.
**Routing:** DX requires BGP. VPN requires IPsec routing tables. Mesh requires DNS/Control Plane updates.

SC10-C
**Failover Orchestration:**
1.  **Detection:** Prometheus monitors site health. If Chicago nodes unreachable for 2 mins, trigger alert.
2.  **GSLB:** Route53 Health Checks detect Chicago failure. DNS TTL ensures rapid failover to Dallas.
3.  **Patroni:** Patroni DCS (etcd) detects primary loss. Promotes Dallas node automatically (Synchronous commit ensures no data loss).
4.  **Redis:** Sentinel detects master down. Promotes replica.
5.  **RabbitMQ:** Federation syncs queues.
6.  **MinIO:** Site replication catches up.
7.  **Terraform:** Update state to reflect Dallas as primary.
**Split-Brain:** Use **etcd quorum**. If Chicago etcd is down, Dallas cannot promote unless it has quorum. Use fencing (STONITH) to kill Chicago nodes.
**Failback:** Once Chicago recovers, re-establish replication. Demote Dallas. Update DNS.
