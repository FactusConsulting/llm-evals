SC1-A
The failure is caused by the interaction between the **Restricted Pod Security Standard (PSS)** and the container image configuration.
1.  **Restricted PSS:** Requires `runAsNonRoot: true` AND explicitly forbids `runAsUser: 0`. It mandates that containers must not run as root.
2.  **Image Configuration:** The new base image (`cgr.dev/chainguard/static:latest`) defaults `USER` to `root` (UID 0).
3.  **Manifest Configuration:** The pods have `securityContext.runAsNonRoot: true` but **no explicit `runAsUser`**.
4.  **The Conflict:** When `runAsNonRoot: true` is set without `runAsUser`, Kubernetes attempts to run the container as the user defined in the image (`root`). Since Restricted PSS forbids UID 0, the kubelet rejects the pod or the container crashes immediately upon startup (often resulting in CrashLoopBackOff if the container exits and restarts).
5.  **Baseline vs. Restricted:** Baseline allows `runAsUser: 0` (root) if `runAsNonRoot` is not set or if the image specifies root. Restricted enforces non-root execution strictly. The upgrade changed the namespace policy from Baseline to Restricted, triggering the enforcement on existing pods that relied on the image defaulting to root.

SC1-C
**Remediation Strategy:**
1.  **Immediate Fix (Image):** Rebuild the Go/Python images with a non-root user (e.g., `USER 1000`) in the Dockerfile.
2.  **Manifest Fix:** Update Deployment manifests to explicitly set `securityContext.runAsUser: 1000` (matching the image) and ensure `runAsNonRoot: true` remains.
3.  **CI Pipeline Gate:** Integrate a PSS scanner (e.g., `kube-score` or `checkov`) into the CI pipeline. Add a gate that fails the build if the generated pod spec violates the `Restricted` profile (specifically checking for `runAsUser: 0`).
4.  **Rollout:** Use a **Canary Deployment**. Deploy the fixed version to 10% of nodes first. Monitor CrashLoopBackOff rates and error logs. If stable, gradually increase to 100%. This prevents a full outage if the fix introduces new issues.

SC2-A
**Stale Data & Replication:**
1.  **Cause:** Cross-region asynchronous replication introduces lag. Promoting the replica cuts off the primary, but any WAL (Write-Ahead Log) not shipped from us-east-1 to eu-west-1 before promotion is lost.
2.  **WAL Shipping:** PostgreSQL streams WAL records. In async mode, the primary acknowledges the write before the replica confirms receipt.
3.  **Data Loss:** Any transactions committed on the primary between the last successful WAL sync and the failover moment are lost.
4.  **Quantification:** Check `pg_stat_replication` on the old primary (if accessible) or the new primary's logs for `replay_lag`. Query `pg_wal_lsn_diff(pg_current_wal_lsn(), pg_last_wal_receive_lsn())` to see bytes of lag. Check application logs for "transaction was aborted" errors indicating lost writes.

SC2-B
1.  **Import Replica:** `terraform import aws_db_instance.promoted_replica <arn_of_promoted_replica>`
2.  **Update State:** `terraform state mv aws_db_instance.old_primary aws_db_instance.promoted_replica` (if renaming resource).
3.  **Handle Dependencies:** Use `terraform state rm` on the old instance resource if it's no longer needed, then `terraform apply` to update Route53/Security Groups to point to the new ARN.
4.  **Force Update:** If Terraform tries to recreate, use `terraform taint aws_db_instance.promoted_replica` to force a refresh of the resource attributes, or use `-replace` flag on specific resources referencing the old ARN.
5.  **Finalize:** Run `terraform plan` to verify state matches reality, then `terraform apply`.

SC2-C
**Architecture Design:**
1.  **RDS Proxy:** Replace PgBouncer with **RDS Proxy**. It handles connection pooling at the database level, supports failover transparently, and reduces connection churn during promotion.
2.  **Route53 Health Checks:** Configure **Route53 Failover Routing Policy**. Health checks monitor the primary endpoint. If unhealthy, DNS TTL propagates the failover record to the secondary region.
3.  **Retry Logic:** Implement **Exponential Backoff with Jitter** in the application. Retry failed writes on the new primary after a short delay to handle transient connection issues.
4.  **RPO < 1s:** Standard RDS async replication cannot guarantee <1s RPO. **Recommend Aurora Global Database**. It uses a dedicated replication layer with lower latency and synchronous replication options for the primary region, ensuring near-zero data loss.
5.  **Latency:** 85ms is acceptable for reads but high for writes. Use local caching (Redis) for US services to reduce write frequency to the EU primary.
