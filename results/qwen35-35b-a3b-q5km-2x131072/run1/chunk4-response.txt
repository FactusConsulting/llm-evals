G1 — **Slice vs Array:** An array is a fixed-size value type `[N]T`. A slice is a dynamic view over an array, defined as a struct containing a pointer, length, and capacity `[]T`. When appending to a slice that has reached its capacity, the runtime allocates a new, larger underlying array (typically doubling the capacity), copies the existing elements, and updates the slice header to point to the new array.

G2 — **`var` vs `:=`:** `var x int` declares a variable with an explicit type and zero value. `x := 0` is a short variable declaration that infers the type. You cannot use `:=` to redeclare a variable in the same scope (e.g., `x := 1` after `x := 0` in the same block) or at the package level (outside functions).

G3 — **Goroutine vs Thread:** A goroutine is a lightweight thread managed by the Go runtime, not the OS. It starts with minimal stack space and grows/shrinks. The Go scheduler (GMP model) maps Goroutines (G) to OS Threads (M) via logical processors (P). The scheduler multiplexes many Gs onto fewer Ms to maximize CPU utilization and handle blocking syscalls efficiently.

G4 — **Channels:** Channels are typed conduits for communication between goroutines. Unbuffered channels block both sender and receiver until the other side is ready. Buffered channels have a fixed capacity; sending to a full buffered channel blocks until space is available. Receiving from an empty buffered channel blocks until data arrives.

G5 — **Context:** The `context` package manages deadlines, cancellation signals, and request-scoped values.
```go
ctx, cancel := context.WithTimeout(context.Background(), 5*time.Second)
defer cancel()
req, _ := http.NewRequestWithContext(ctx, "GET", url, nil)
client.Do(req) // Cancels if timeout exceeded
```

G6 — **Interfaces:** Go interfaces are satisfied implicitly (duck typing). Unlike Java/C#, you do not declare implementation. The empty interface `interface{}` (or `any`) accepts any type. It is powerful for generic containers but dangerous because it requires type assertions to use values, losing compile-time type safety.

G7 — **Memory Model:** Go's memory model guarantees visibility only if synchronization is used. The code is unsafe because `ready` is not atomic and `data` write is not synchronized with the read. The reader might see `ready=true` but stale `data`. Fix using `sync/atomic` for `ready` or a `sync.Mutex` protecting both variables.

G8 — **Mutex vs RWMutex:** `sync.Mutex` allows exclusive access. `sync.RWMutex` allows multiple readers or one writer. `sync.Map` is optimized for read-heavy workloads or keys that rarely collide. Trade-off: `sync.Map` has higher overhead per access than a mutex-protected map but avoids lock contention for reads.

G9 — **Modules:** `go.mod` declares module path and dependencies. `go.sum` stores checksums for reproducible builds. `go mod tidy` removes unused dependencies and adds missing ones. MVS (Minimum Version Selection) selects the lowest version of a dependency that satisfies all requirements in the dependency tree.

G10 — **Defer:** `defer` postpones function execution until the surrounding function returns. Multiple defers execute in LIFO (Last In, First Out) order. A common gotcha in loops is capturing the loop variable by reference in a closure; defer captures the variable's address, not its value at that iteration.

G11 — **Pointers:** A pointer holds a memory address. The zero value is `nil`. To create a pointer to a struct literal: `p := &MyStruct{Field: "value"}`. Dereference with `*p`.

G12 — **Init:** `init()` runs automatically before `main()` when a package is loaded. It runs once per package. A package can have multiple `init()` functions; they execute in the order they appear in the source file.

G13 — **Make vs New:** `make(T, args)` initializes built-in types (slice, map, channel) and returns the type itself. `new(T)` allocates zeroed memory for any type and returns a pointer `*T`.

G14 — **Error Handling:** Errors are values returned by functions. The `error` interface has one method `Error() string`. Idiomatic pattern: `if err != nil { return err }`. `errors.Is` checks semantic equality against a sentinel. `errors.As` performs type assertion to extract a specific error type.

G15 — **Composite Types:**
*   `map`: Zero value is `nil`.
*   `slice`: Zero value is `nil`.
*   `channel`: Zero value is `nil`.
*   `struct`: Zero value has all fields set to their respective zero values.

G16 — **Race Condition:** Occurs when two goroutines access the same memory concurrently, and at least one writes. The `-race` flag instruments code to detect data races at runtime. Fix by using `sync.Mutex` to serialize access or channels to transfer ownership.

G17 — **Select:** `select` waits on multiple channel operations. Unlike `switch`, it chooses a ready case randomly if multiple are ready. Implement timeout by adding a case with `time.After(duration)`.

G18 — **Escape Analysis:** The compiler determines if a variable lives on the stack or heap. If a variable's address escapes the function scope, it is allocated on the heap. Inspect with `go build -gcflags="-m"`. Heap allocation incurs GC overhead; stack allocation is faster.

G19 — **Reflection:** `reflect.Type` describes the type metadata. `reflect.Value` holds the dynamic value. Reflection is slow due to runtime type inspection and indirection. Justified for serialization frameworks or generic utilities where compile-time types are unknown.

G20 — **Embed:** `//go:embed` embeds files/directories into the binary at compile time. `//go:embed *` embeds all files in a directory. It interacts with `io/fs.FS` (e.g., `embed.FS`). Limitations: content must be static at build time; cannot embed symlinks or external paths.

R1 — **Ownership:** Rust's core system ensuring memory safety without GC. Rules: 1) Each value has one owner. 2) When owner goes out of scope, value is dropped. 3) You can move ownership, but not copy (unless `Copy` trait).

R2 — **String vs &str:** `String` is an owned, heap-allocated, growable buffer. `&str` is a borrowed slice of UTF-8 bytes. Use `&str` for function arguments (zero-copy), `String` for values you own or modify.

R3 — **Borrowing:** You can have many immutable references `&T` OR one mutable reference `&mut T`, but not both simultaneously. References must be valid and not alias mutable data. The borrow checker enforces this at compile time.

R4 — **Result/Option:** `Result<T, E>` represents success or error. `Option<T>` represents value or `None`. The `?` operator propagates errors up the call stack.
```rust
fn read_file() -> Result<String, io::Error> {
    std::fs::read_to_string("file.txt")
}
```

R5 — **Box/Rc/Arc:** `Box<T>` owns data on heap (single owner). `Rc<T>` (Reference Counted) allows multiple owners in single-threaded contexts. `Arc<T>` (Atomic Reference Counted) allows multiple owners across threads.

R6 — **Traits:** Rust interfaces. `dyn Trait` uses dynamic dispatch (vtable, runtime cost). `T: Trait` uses static dispatch (monomorphization, zero cost). Go interfaces are implicit; Rust traits are explicit.

R7 — **Lifetimes:** The function fails because the compiler cannot determine which input lifetime the return value borrows. Fix:
```rust
fn longest<'a>(x: &'a str, y: &'a str) -> &'a str {
    if x.len() > y.len() { x } else { y }
}
```

R8 — **Send/Sync:** `Send` means ownership can be transferred across threads. `Sync` means `&T` can be shared across threads. `Rc<T>` is not `Send` because its ref count is not atomic. `Arc<T>` uses atomics, making it `Send` and `Sync`.

R9 — **Async/Await:** Rust futures are state machines. `async` functions return a `Future` that must be polled by an executor (e.g., Tokio). Unlike Go's preemptive goroutines, Rust futures are cooperative; they yield control explicitly.

R10 — **Unsafe:** `unsafe` blocks bypass compiler checks. Allowed actions: dereference raw pointers, call unsafe functions, access mutable statics, union access. Justified for FFI, low-level optimizations, or implementing safe abstractions over unsafe primitives.

R11 — **Pattern Matching:** `match` exhaustively checks all possible variants. `if let` checks a single pattern. The compiler enforces exhaustiveness in `match` to prevent logic errors from unhandled cases.

R12 — **Vec vs Array:** `Vec<T>` is a heap-allocated, growable array. `[T; N]` is a stack-allocated, fixed-size array. `Vec` reallocates memory (doubling capacity) and copies elements when pushed beyond capacity.

R13 — **Enums:** Rust enums are Algebraic Data Types (ADTs) that can hold data. Unlike C/Java, variants can carry payloads. `Option<T>` is `enum { Some(T), None }`. Rust has no `null`; `None` is explicit.

R14 — **Impl Block:** `impl` defines methods. `&self` borrows the instance. `&mut self` borrows mutably. `self` consumes the instance. Associated functions (e.g., `String::new()`) do not take `self` and act as constructors.

R15 — **Derive Macro:** Automatically implements traits like `Debug`, `Clone`, `PartialEq`. Cannot derive if fields contain types that don't support the trait (e.g., `Fn` trait, or fields with lifetimes that don't match).

R16 — **Cow:** `Cow<'a, T>` (Clone on Write). Holds either a borrowed `&'a T` or an owned `Box<T>`. Useful when you want to avoid allocation if the data isn't modified, but allocate only if mutation is required.

R17 — **Modules:** `mod` declares a module. `use` imports paths. `pub` makes items visible outside the crate. File system maps to modules (`mod.rs` or `mod/` folder). Rust 2018 improved path resolution (e.g., `crate::` vs `self::`).

R18 — **Pin:** `Pin<T>` prevents a value from being moved in memory. Needed for self-referential structs and `async` futures (which may hold references to themselves). `Unpin` is a marker trait indicating a type can be moved safely.

R19 — **Macros:** `macro_rules!` are declarative (pattern matching). Procedural macros (derive, attribute) are Rust code generating Rust code. Hygiene ensures macro variables don't leak into the caller's scope.

R20 — **Zero-Cost:** Abstractions that compile to efficient machine code. Generics use monomorphization (code duplication per type). Iterators use lazy evaluation without allocation. Traits use static dispatch. Performance matches C.
