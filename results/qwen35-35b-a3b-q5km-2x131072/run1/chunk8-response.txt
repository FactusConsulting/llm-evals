CL1 — **IaaS (Infrastructure as a Service):** Provides virtualized computing resources (servers, storage, networking). *Example:* AWS EC2. **PaaS (Platform as a Service):** Provides hardware and software tools (runtime, middleware) for development. *Example:* Google App Engine. **SaaS (Software as a Service):** Delivers software applications over the internet. *Example:* Microsoft Office 365.

CL2 — **Shared Responsibility Model:** Provider secures the cloud (physical, network, hypervisor); Customer secures what is in the cloud (OS, data, config). **IaaS VM:** Provider secures hardware/hypervisor; Customer secures OS, patching, app. **Managed K8s:** Provider secures control plane; Customer secures node OS, K8s config, app. **SaaS (M365):** Provider secures app/platform; Customer secures identity, data, access policies.

CL3 — **VPC:** Isolated virtual network. **Subnet:** IP range within VPC (public/private). **Security Group:** Instance-level firewall (stateful). **NACL:** Subnet-level firewall (stateless). **Layering:** VPC contains Subnets; Subnets contain Instances protected by SGs; NACLs filter traffic entering/leaving Subnets. **Stateful:** SG remembers return traffic. **Stateless:** NACL requires explicit allow rules for both request and response.

CL4 — **Landing Zone:** Secure, compliant baseline environment. **OUs:** Logical grouping for billing/policy. **Guardrails:** Preventative controls (e.g., no public S3 buckets). **Shared Networking:** Centralized hub (Transit Gateway) for inter-VPC/On-prem connectivity. **Centralized Logging:** Aggregated logs for security monitoring. **Single-Account Risk:** High blast radius, billing confusion, permission sprawl, lack of isolation between environments (prod/dev).

CL5 — **IaC:** Managing infrastructure via code files. **Terraform/OpenTofu:** Multi-cloud, declarative, state-based. **Pulumi:** Multi-cloud, imperative (using general languages like Python/TS). **CloudFormation/Bicep:** AWS/Azure native, tightly integrated. **Crossplane:** K8s-native, manages cloud resources as K8s CRDs. **Choice:** Terraform for multi-cloud flexibility; CFN/Bicep for AWS/Azure specific depth; Pulumi for dev familiarity; Crossplane for K8s-centric orgs.

CL6 — **HPA:** Scales pods based on CPU/Memory. **VPA:** Scales pod resources (CPU/RAM) based on usage. **Cluster Autoscaler:** Adds/removes nodes based on pending pods. **KEDA:** Event-driven scaling (e.g., Kafka queue depth). **Use KEDA over HPA:** When scaling is driven by external events (message queues, HTTP requests) rather than internal resource metrics.

CL7 — **Strategy:** Define RPO (data loss tolerance) and RTO (downtime tolerance). **Active-Active:** Both regions serve traffic (low RTO, high cost). **Active-Passive:** One region active, one standby (higher RTO, lower cost). **DB:** Sync (strong consistency, latency) vs Async (replication lag, lower latency). **DNS:** Route53 Failover routing. **State:** Externalized (S3/DynamoDB) vs Local. **DR Types:** Pilot Light (DB running, app off), Warm Standby (scaled-down app), Multi-site (full active).

CL8 — **GitOps:** Git repository is source of truth for desired state. **ArgoCD:** Push/Pull model, UI focused. **FluxCD:** Pull model, CLI focused. **Secrets:** Sealed Secrets (encrypted in Git), External Secrets Operator (fetch from Vault/AWS Secrets Manager). **Drift:** GitOps controller detects manual changes, automatically reverts them to match Git state.

CL9 — **Architecture:** Use **StatefulSet** for stable network IDs and ordered deployment. **PV/PVC:** Use `local` storage class for low latency (hostPath). **Headless Service:** For stable DNS names per pod. **Init Containers:** For config file generation before main app starts. **ConfigMaps:** Mount as volumes for config files. **PDB:** Ensure minimum available pods during updates to prevent data loss.

CL10 — **SA:** K8s internal identity for pods. **Workload Identity:** GCP/Azure native identity federation. **IRSA:** AWS OIDC federation for K8s SAs. **Mounting Creds:** Bad practice (static, leaked in logs/images). **Federation:** Pod requests OIDC token from K8s metadata service, exchanges it for temporary cloud credentials.

CL11 — **Object (S3):** Unstructured data, high durability (11 9s), low latency retrieval. **Block (EBS):** Attached disk, low latency, high IOPS, single AZ (usually). **File (EFS):** Shared file system, NFS protocol, multi-AZ. **Use Object:** Backups, media. **Use Block:** OS disks, DBs. **Use File:** Shared configs, home dirs.

CL12 — **Zero Trust:** Verify explicitly, least privilege. **Network Policies:** Default deny all traffic. **Service Mesh:** mTLS between services. **Pod Identity:** SPIFFE/SPIRE. **Admission:** OPA/Kyverno to enforce policies. **Image Signing:** Cosign to verify integrity. **Runtime:** Falco for anomaly detection. **Diff:** Traditional relies on perimeter trust; Zero Trust assumes breach and verifies every request.

CL13 — **CDN:** Distributed network of edge servers caching content. **How:** Caches static assets closer to users. **Services:** AWS CloudFront, Cloudflare. **Use CDN:** For static content (images, CSS) to reduce latency and origin load. **Direct:** Only for dynamic, personalized content that cannot be cached.

CL14 — **Region:** Geographic area with multiple AZs. **AZ:** Isolated data center with independent power/cooling. **Multi-AZ:** Ensures high availability if one data center fails. **Zonal:** Resource tied to one AZ (e.g., EBS). **Regional:** Resource spans AZs (e.g., S3, Route53).

CL15 — **Managed DB:** Service where provider manages hardware, OS, DB software. **Provider Tasks:** Patching, backups, failover, scaling, monitoring. **VM Tasks:** You manage OS, DB installation, patching, backups, HA configuration.

CL16 — **IAM:** Users (people), Groups (collections), Roles (temp access), Policies (permissions). **Least Privilege:** Grant only necessary permissions. **Apply:** Assign policies to Groups/Roles, not Users. Use Role assumption for cross-account access.

CL17 — **Serverless:** Event-driven execution without server management. **Compare:** Lambda (AWS), Functions (Azure), Cloud Functions (GCP). **Cold Starts:** Latency during first invocation after inactivity. **Poor Fit:** Long-running processes, stateful apps, predictable high-throughput workloads.

CL18 — **Ingress:** K8s resource for HTTP/HTTPS routing. **Gateway API:** Standardized, multi-protocol, more flexible. **LB:** Cloud provider load balancer. **Mesh:** Sidecar proxy ingress. **TLS:** Ingress (terminates at LB/Controller), Mesh (terminates at sidecar). **Use Gateway API:** For complex routing, multi-cluster, or when standard Ingress is too limited.

CL19 — **Strategy:** **RI/SP:** Commit to usage for discount. **Spot:** Use for fault-tolerant workloads. **Right-sizing:** Match instance to actual usage. **Tiering:** Move old data to Glacier. **Idle Detection:** Tag and terminate unused resources. **FinOps:** Cross-team cost accountability. **Tools:** AWS Cost Explorer, CloudHealth. **Showback:** Report costs to teams. **Chargeback:** Bill teams directly.

CL20 — **Design:** **Abstraction:** Kubernetes (runtime), Terraform (IaC). **Networking:** Direct Connect/ExpressRoute. **Identity:** OIDC federation. **Trade-offs:** Multi-cloud increases complexity/cost. **Justified:** Compliance, regulatory data residency, avoiding single-vendor outage risk. **Unnecessary:** If one cloud meets all needs, multi-cloud adds operational overhead without significant benefit.

OT1 — **Purdue Model:** ISA-95 standard for IT/OT integration. **Level 0:** Physical Process (Sensors/Actuators). **Level 1:** Basic Control (PLCs/RTUs). **Level 2:** Supervisory Control (SCADA/HMI). **Level 3:** Operations Management (MES). **Level 4:** Enterprise Business Planning (ERP).

OT2 — **IT:** Information systems, data, frequent patching, standard security. **OT:** Operational technology, physical processes, real-time, legacy hardware. **Patching:** IT patching can cause downtime or instability in OT; OT systems often run unsupported OS versions and require stability over security updates.

OT3 — **DMZ (IDMZ):** Buffer zone between IT and OT. **Traffic:** Historian sync (OT->IT), Production data (OT->IT). **Direction:** Generally OT to IT is safer; IT to OT requires strict control. **Direct L4->L1:** Dangerous because it bypasses security controls, allowing attackers to move laterally from corporate network directly to critical control systems.

OT4 — **Modbus TCP:** L1/L2, simple request/response. **OPC UA:** L2/L3, secure, platform-independent. **EtherNet/IP:** L1/L2, Allen-Bradley dominant. **PROFINET:** L1/L2, real-time industrial Ethernet. **MQTT:** L2/L3/Cloud, lightweight pub/sub for IoT.

OT5 — **Data Diode:** Hardware allowing one-way data flow. **Firewall:** Bidirectional traffic control. **Use Diode:** When data must leave OT but nothing can enter (e.g., exporting telemetry). **Limitation:** Cannot send commands or updates back to OT; no feedback loop.

OT6 — **Flow:** PLC -> Historian -> OPC UA Server -> Edge Gateway -> MQTT Broker -> Cloud. **Security Boundaries:** Firewall between Edge and Cloud (allow only specific ports), TLS encryption, DMZ at Edge Gateway to isolate OT network from public internet.

OT7 — **Zones:** Logical groupings of assets with similar security requirements. **Conduits:** Controlled paths for traffic between zones. **Design:** Zone 1 (Corporate), Zone 2 (Engineering), Zone 3 (SCADA), Zone 4 (PLC/SIS). **Conduits:** Firewall between Zone 1-2, Zone 2-3, Zone 3-4. Engineering workstations access Zone 3 via secure conduit.

OT8 — **Segmentation:** VLANs per production line. **Firewall:** Between VLANs and DMZ. **Switches:** Managed Layer 3 switches with ACLs. **Remote Access:** Jump host with MFA, session recording, time-limited tokens. **Transition:** Phased approach, parallel run, maintenance windows, backup network plan.

OT9 — **SIS:** Safety Instrumented System (independent, failsafe). **BPCS:** Basic Process Control System (normal operation). **IEC 61511:** Safety lifecycle standard for SIS. **Isolation:** SIS must be air-gapped to prevent cyber attacks from disabling safety functions. **Ransomware on SIS:** Can disable safety shutdowns, leading to physical damage, injury, or environmental disaster.

OT10 — **PLC:** Programmable Logic Controller (local logic). **DCS:** Distributed Control System (process control, integrated). **SCADA:** Supervisory Control and Data Acquisition (wide area monitoring). **Coexist:** Yes, PLCs often feed into SCADA/DCS systems.

OT11 — **Architecture:** Jump host (bastion), MFA required, session recording, vendor-specific time-limited accounts. **Direct RDP:** Unacceptable because it exposes HMI directly to IT threats, bypasses audit trails, and increases attack surface.

OT12 — **Detection:** NIDS, anomaly detection on network traffic. **Inventory:** Verify compromised assets. **Response:** Isolate network segment, notify plant operator. **Coordination:** Operator maintains physical safety (manual control) while IT isolates. **Operator Role:** Ensure process safety, manual override if needed, do not restart systems without validation.

OT13 — **HMI:** Human-Machine Interface (operator screen). **Role:** Visualize process, allow manual control. **Difference:** HMI is for operations; Engineering Workstation is for configuration, programming, and maintenance.

OT14 — **Historian:** Time-series database for process data. **Importance:** Trend analysis, compliance, troubleshooting. **Plant (L3):** Real-time process data. **Enterprise (L4):** Aggregated data for business analytics.

OT15 — **Ladder Logic:** Graphical programming language resembling relay circuits. **Device:** PLC. **Dominant:** Familiarity, deterministic execution, safety certification, ease of maintenance for electricians.

OT16 — **RTU:** Remote Terminal Unit (communication focused, harsh environments). **PLC:** Logic focused (local). **RTU Use:** Remote sites (pipelines, power grids) where comms reliability is key.

OT17 — **Analog:** Continuous signal (4-20mA). **Digital:** Discrete signal (On/Off). **4-20mA:** Preferred because 4mA indicates live wire (vs 0mA fault), allowing detection of wire breaks.

OT18 — **Inventory:** Foundation for knowing what to protect. **Tools:** Passive (IDS, NetFlow), Active (Scanners). **Active Danger:** Can crash legacy devices, disrupt real-time processes, or trigger alarms.

OT19 — **MITRE ICS:** Framework for ICS threats. **Tactics:** Reconnaissance, Initial Access, Execution, Persistence, Impact. **Technique:** E.g., "Exploit Public-Facing Application" (Initial Access). **Difference:** Focuses on physical impact/safety, not just data. **Use:** Map defenses to tactics to find gaps (e.g., no network segmentation).

OT20 — **SOC:** Passive log collection (SPAN ports), SIEM correlation, OT-specific rules (e.g., PLC write attempts). **Staffing:** IT analysts + OT engineers. **False Positives:** Tune rules to OT baselines. **Firmware Change:** Alert -> Isolate -> Verify with Engineering -> Rollback -> Investigate root cause.
