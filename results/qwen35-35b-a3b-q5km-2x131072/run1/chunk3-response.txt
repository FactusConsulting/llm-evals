T1 — **Plan** calculates the execution plan (what will change) without making changes. **Apply** executes the plan to create, update, or destroy resources. Plan is safe for review; Apply is destructive/constructive.

T2 — **State** is a JSON file mapping real-world resources to your configuration. **Remote state** is crucial for collaboration (sharing state across team members), locking (preventing concurrent writes), and security (storing secrets securely).

T3 — `count` uses integer indices; `for_each` uses maps or sets. `for_each` is preferred when resources have unique identifiers (like names) because removing one item doesn't shift indices and cause unintended resource recreation (drift).

T4 — This is **state drift**. `tofu plan` detects the difference between the state file and the real infrastructure. `tofu apply` will attempt to reconcile the state with the real world, recreating or modifying the resource to match the configuration.

T5 —
```hcl
resource "aws_instance" "web" {
  for_each = toset(["web1", "web2", "web3"])
  ami      = "ami-12345678"
  instance_type = "t2.micro"
  tags = {
    Name = each.key
  }
}
```

T6 — `terraform_remote_state` is a data source that reads another state file (read-only, no locking). `output` values are consumed directly from a module call (type-safe, respects state locking). The recommended alternative is consuming **module outputs** directly if the modules share a state, or using `terraform_remote_state` only when cross-state access is unavoidable.

T7 — `remote-exec` provisioners run asynchronously. The output might be read before the provisioner finishes writing the kubeconfig. **Fix:** Add `depends_on = [null_resource.kubeconfig]` to the dependent resource, or use a `local-exec` provisioner to write the file synchronously before the resource is considered created.

T8 — `tofu import` is a CLI command to import existing resources into state manually. The **import block** (1.5+) is declarative, version-controlled, and allows importing resources directly in the configuration, making the import process reproducible and auditable.

T9 — A **variable** is an input parameter for the module. An **output** is a value exposed by the module to be used by the caller. Variables are inputs; outputs are outputs.

T10 — A **provider** is a plugin that interacts with a specific API (e.g., AWS, Azure). Without `tofu init`, the provider plugins are not downloaded, and `tofu plan` will fail with an error indicating missing providers.

T11 — The `.terraform.lock.hcl` file records the exact provider versions and checksums used. It should be committed to ensure **reproducible builds** and prevent accidental provider upgrades that might break infrastructure.

T12 — `tofu destroy` removes all resources managed by the state. To target a single resource: `tofu destroy -target=aws_instance.example`.

T13 — A **data source** reads information from an external system without creating or managing it. Use one when you need existing data (e.g., `data "aws_ami" "ubuntu"` to find the latest Ubuntu image) rather than creating a new resource.

T14 — **Variables** are inputs from outside the module. **Locals** are internal calculations within the module. Use `locals` for intermediate values or complex expressions that shouldn't be exposed as inputs.

T15 — The `lifecycle` block controls resource behavior.
*   `create_before_destroy`: Creates new resource before destroying old (zero downtime).
*   `prevent_destroy`: Prevents accidental deletion (safety).
*   `ignore_changes`: Ignores drift on specific attributes (e.g., tags managed externally).

T16 — The `moved` block allows renaming resources or moving them between modules without changing the state file manually. It updates the state mapping internally, preventing recreation during refactoring.

T17 — The **dependency graph** determines execution order. **Implicit** dependencies are inferred from references (e.g., `aws_instance.id`). **Explicit** dependencies use `depends_on`. `depends_on` forces serialization, reducing parallelism and slowing down apply. Use only when implicit references are insufficient.

T18 — Strategies:
1.  **`refresh=false`**: Skips state refresh, significantly faster but may miss drift.
2.  **Split State**: Break monolithic state into smaller modules/states to reduce graph complexity.
3.  **Prune State**: Remove unused resources from state (`tofu state rm`) to reduce graph size.
4.  **Optimize Providers**: Use provider caching or reduce provider calls (e.g., avoid `data` sources that query APIs frequently).

T19 — State locking prevents concurrent modifications. If a lock isn't released (crash), subsequent commands hang. **Fix:** `tofu force-unlock <LOCK_ID>`. **Risk:** If another process is actively writing, this can corrupt the state file.

T20 — 1. Configure backend in `backend "s3" {...}`. 2. Run `tofu init -migrate-state -backend-config="bucket=..."`. 3. Terraform copies local state to S3 and sets up DynamoDB lock. 4. Verify with `tofu state list` and check S3 for the state file.

A1 — A **Playbook** is a YAML file defining the automation steps. A **Role** is a structured directory for reusable logic. A **Task** is a single action within a playbook or role.

A2 — `ansible_facts` is a dictionary of system information gathered by the `setup` module. Access OS family via `{{ ansible_facts['os_family'] }}`.

A3 — `command`: Executes without shell (no pipes/globs). `shell`: Executes via shell (supports pipes/globs). `raw`: Executes without setup (no Python interpreter), useful for bootstrapping SSH.

A4 — **Idempotency** means running a task multiple times yields the same result.
*   Idempotent: `apt: name=nginx` (installs only if missing).
*   Not Idempotent: `shell: echo "timestamp" > /tmp/log` (changes file every time).

A5 —
```yaml
- name: Install htop
  package:
    name: htop
  become: yes
  when: ansible_os_family == "Debian"

- name: Install htop on RedHat
  package:
    name: htop
  become: yes
  when: ansible_os_family == "RedHat"
  # Add epel-release task before if needed
```

A6 — Precedence (High to Low): Extra vars > Play vars > Host vars > Group vars > Role vars > Role defaults. `set_fact` overrides most at runtime.

A7 — Use `serial: 3` to limit concurrency. Group nodes into `agents` and `agents_batch`. Use `meta: flush_handlers` or `wait_for` to ensure the first batch is ready before the next batch starts.

A8 — `ansible-vault` encrypts sensitive data. To encrypt a single variable: `ansible-vault encrypt_string 'secret_value' --name 'my_secret'`.

A9 — An **inventory** defines hosts and groups. **Static** is a file (INI/YAML). **Dynamic** is a script/API that generates inventory dynamically (e.g., AWS EC2).

A10 — `ansible webservers -m shell -a "df -h"`.

A11 — A **handler** is a task that runs only when notified. It differs from a regular task by running asynchronously at the end of the play. Trigger via `notify: handler_name`.

A12 — `ansible.cfg` configures Ansible behavior. Common settings: `host_key_checking`, `forks` (parallelism), `remote_user`, `inventory`.

A13 — `register` stores a task's output in a variable. Use it in subsequent tasks via `when: my_var.stdout == "success"` or `debug: var=my_var`.

A14 — `include_tasks` is evaluated at runtime (dynamic). `import_tasks` is evaluated at compile time (static). `import_tasks` is faster but cannot use variables in the path; `include_tasks` allows dynamic paths.

A15 — **Ansible Galaxy** is a repository for roles/collections. Use `requirements.yml` to list them, then run `ansible-galaxy install -r requirements.yml`.

A16 — Connection plugins manage how Ansible connects. **ssh**: Remote Linux/Unix. **local**: Runs on control node. **network_cli**: Network devices (Cisco/Juniper).

A17 — `delegate_to` runs a task on a specific host. `run_once` runs a task once across the batch. Example: Gather facts from a remote host, then `delegate_to: localhost` to process them on the control node.

A18 — Strategies:
1.  **`forks`**: Increase parallel connections (e.g., `forks: 50`).
2.  **`pipelining`**: Reduces SSH connections per task (faster).
3.  **`fact_caching`**: Caches gathered facts to avoid re-running `setup`.
4.  **`mitogen`**: Optimizes SSH transport for massive scale (requires plugin).

A19 — `{{ }}` evaluates expressions. `{% %}` controls logic (loops/if). `{# #}` comments.
```jinja2
upstream backend {
{% for server in backend_servers %}
    server {{ server.ip }} weight={{ server.weight }}; # Health: {{ server.health }}
{% endfor %}
}
```

A20 — 1. Set `serial: 1`. 2. `pre_tasks`: Deregister from LB. 3. `tasks`: Deploy app. 4. `post_tasks`: Register to LB + `uri` health check. 5. `block/rescue`: If health check fails, rollback (revert to previous version).
