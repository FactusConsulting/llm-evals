AA1 — **Monolith vs. Microservices:** A monolith is a single unified codebase deployed as one unit. Microservices decompose this into independent, loosely coupled services deployed separately.
**Microservices Problems:**
1.  **Distributed Complexity:** Network latency, partial failures, and distributed tracing are required.
2.  **Data Consistency:** Maintaining ACID transactions across services requires patterns like Sagas.
3.  **Operational Overhead:** Requires robust CI/CD, container orchestration, and service discovery infrastructure.

AA2 — **Sync vs. Async:** Synchronous communication blocks the caller until a response is received (request/response). Asynchronous communication allows the caller to continue processing without waiting for an immediate response (fire-and-forget or callback).
**Examples:** Synchronous: HTTP/REST, gRPC. Asynchronous: RabbitMQ, Apache Kafka.

AA3 — **API Gateway vs. Reverse Proxy:** A reverse proxy forwards traffic to backend servers (L4/L7). An API Gateway adds business logic like authentication, rate limiting, API versioning, and analytics.
**Replacement:** An API Gateway cannot fully replace a reverse proxy in all cases. While modern gateways (like Kong) act as proxies, a pure L4 load balancer (like HAProxy in TCP mode) is sometimes preferred for raw performance or non-HTTP traffic where gateway overhead is unnecessary.

AA4 — **Saga Pattern:** A sequence of local transactions where each updates data within a service. If a step fails, compensating transactions undo previous changes.
**Choreography vs. Orchestration:** Choreography uses events to trigger the next step (decentralized, harder to debug). Orchestration uses a central coordinator to direct steps (centralized, easier to manage flow).
**Appropriateness:** Choreography for simple, loosely coupled flows. Orchestration for complex workflows requiring visibility and control.

AA5 — **CQRS:** Command Query Responsibility Segregation separates read (Query) and write (Command) operations into different models.
**Pairing with Event Sourcing:** Event Sourcing stores state changes as a sequence of events. CQRS pairs well because the event log is the source of truth for rebuilding read models.
**Problem Solved:** Allows independent scaling of read and write paths and provides a complete audit trail, which traditional CRUD (single model) struggles with efficiently.

AA6 — **Circuit Breaker:** Prevents repeated calls to a failing service.
**States:** Closed (normal), Open (failures detected, requests fail fast), Half-Open (testing if service recovered).
**Difference from Retry:** Retry attempts to fix transient failures; Circuit Breaker stops hammering a persistent failure.
**Usage:** Use Retry for transient network blips; use Circuit Breaker when a service is down to prevent cascading failures.

AA7 — **Multi-tenant Isolation:**
1.  **Shared DB/Schema:** Lowest cost, highest noisy-neighbor risk, security relies on row-level logic.
2.  **Shared DB/Schema:** Moderate cost, better security (schema isolation), higher ops complexity (schema migrations).
3.  **Separate DB:** Highest cost, highest security/isolation, highest ops complexity (backup/restore per tenant).
**Trade-off:** Model 1 is best for cost-sensitive SaaS; Model 3 is best for enterprise/regulated tenants.

AA8 — **Consistency:** Strong consistency ensures all nodes see the same data immediately. Eventual consistency allows temporary divergence.
**Inventory Architecture:** Requires **Strong Consistency**.
**Data Store:** Relational DB (e.g., PostgreSQL) with row-level locking or a distributed key-value store with atomic operations (e.g., Redis with Lua scripts).
**Patterns:** Optimistic locking (version numbers) or Distributed Locks (Redis/Zookeeper) to serialize inventory decrements.

AA9 — **Sidecar Proxy:** A container deployed alongside the main application container to handle network traffic.
**Envoy/Istio:** Envoy handles the data plane (traffic routing, mTLS). Control plane (Pilot) configures Envoy.
**Data vs. Control Plane:** Data plane processes traffic; Control plane manages configuration.
**mTLS:** Application-level TLS secures data in transit. mTLS between sidecars authenticates the *service identity* (not just the endpoint), preventing unauthorized services from communicating even if they have valid certs.

AA10 — **Strangler Fig:** Incrementally migrate functionality from a legacy system to a new one by routing specific requests to the new system.
**Strategy:** Deploy .NET 8 services alongside the .NET Framework monolith. Place an API Gateway in front. Route specific endpoints (e.g., `/api/orders`) to the new microservices while legacy routes go to the monolith. Gradually migrate endpoints until the monolith is removed.

AA11 — **12-Factor App:** Methodology for building SaaS apps.
**Factors:** 1. Codebase (one repo), 2. Dependencies (explicitly declared), 3. Config (env vars), 4. Backing Services (treated as attached resources), 5. Build/Release/Run (strict separation), 6. Processes (stateless), 7. Port Binding (export via env), 8. Concurrency (scale via processes), 9. Disposability (fast start/stop), 10. Dev/Prod Parity, 11. Logs (stream), 12. Admin Processes (run as one-off).
**Importance:** Ensures portability, scalability, and maintainability in cloud environments.

AA12 — **Event-Driven (Kafka) vs. Request-Driven (REST):**
*   **Coupling:** Event-driven is loosely coupled; Request-driven is tightly coupled.
*   **Scalability:** Event-driven scales better for high throughput; Request-driven is simpler for low volume.
*   **Debugging:** Event-driven is harder (distributed state); Request-driven is easier (traceable).
*   **Ordering:** Kafka guarantees ordering per partition; REST does not.
*   **Failure:** Event-driven uses Dead Letter Queues; Request-driven uses timeouts/retries.
**Choice:** Use Event-driven for decoupling and high volume; Request-driven for real-time control and simplicity.

AA13 — **Load Balancer:** Distributes traffic across multiple servers.
**L4 vs. L7:** L4 (Transport) balances based on IP/Port. L7 (Application) balances based on HTTP headers/URL.
**Use Case:** L7 is required for SSL termination or routing `/api` to one backend and `/static` to another.

AA14 — **Message Queue:** Stores messages for processing.
**Queue (RabbitMQ) vs. Log (Kafka):** Queues delete messages after consumption (tasks). Logs retain messages for replay (streaming).
**Pick:** Use RabbitMQ for task processing (e.g., email). Use Kafka for event streaming (e.g., analytics).

AA15 — **Container vs. VM:** Containers share the host OS kernel; VMs virtualize hardware with a full guest OS.
**Popularity:** Containers are lightweight, start faster, and have lower overhead, making them ideal for microservices scaling.

AA16 — **Stateless vs. Stateful:** Stateless services store no session data on the server (e.g., Web Server). Stateful services store data locally (e.g., Cache).
**Scaling:** Stateless is easier to scale horizontally because any instance can handle any request without data synchronization.

AA17 — **ORM:** Maps database tables to programming language objects.
**Examples:** Entity Framework (.NET), Hibernate (Java).
**Trade-offs:** ORMs increase developer productivity but can generate inefficient SQL and hide database complexity. Raw SQL offers performance control but is verbose.

AA18 — **Bulkhead Pattern:** Isolates resources (threads, connections) for different services to prevent one failure from exhausting all resources.
**Implementation:** Use separate thread pools for different downstream calls. In Kubernetes, use separate namespaces or deployment groups with resource limits (CPU/Memory) to isolate noisy neighbors.

AA19 — **OT vs. CRDT:** OT requires a central server to resolve conflicts (server authoritative). CRDTs allow clients to merge changes mathematically (client authoritative).
**Comparison:** OT has lower latency for server sync but complex conflict logic. CRDTs support offline editing and eventual convergence.
**Choice:** Choose **CRDTs** for offline support because clients can edit locally and merge automatically when online without server intervention.

AA20 — **Distributed Tracing Challenges:**
*   **Context Propagation:** Inject Trace ID into headers (W3C Trace Context) or message headers.
*   **Sampling:** Head-based (decide at start) vs. Tail-based (decide after latency known).
*   **Storage:** Jaeger/Tempo store spans; querying requires indexing.
*   **Kafka:** Trace context must be injected into Kafka message headers to maintain continuity across async boundaries.

OP1 — **Hypervisor Type 1 (Bare-metal):** Runs directly on hardware (e.g., VMware ESXi, Microsoft Hyper-V Server). **Type 2 (Hosted):** Runs on an OS (e.g., VMware Workstation, VirtualBox). Type 1 is for production; Type 2 is for development.

OP2 — **SAN vs. NAS:** SAN provides block storage (LUNs) via Fibre Channel/iSCSI. NAS provides file storage via NFS/SMB.
**Use:** SAN for databases (low latency, block access). NAS for file shares (easy sharing).

OP3 — **3-Tier Network:** Core (Backbone), Distribution (Policy), Access (Edge).
**Leaf-Spine:** Preferred for modern DCs. Solves oversubscription and latency issues of 3-tier by providing non-blocking, equal-cost paths between any two nodes.

OP4 — **Proxmox HA:** 3+ nodes for quorum. Shared storage (Ceph/iSCSI). Fencing (DRAC/iLO) to isolate failed nodes.
**Failure:** If a node fails, fencing kills its VMs, and HA restarts them on surviving nodes.

OP5 — **VLAN Trunking:** Carries multiple VLANs over one link using 802.1Q tags.
**Linux Config:** Create sub-interfaces: `ip link add link eth0 name eth0.10 type vlan id 10`. Assign IP to `eth0.10`.

OP6 — **RAID:** RAID 1 (Mirror), RAID 5 (Striped+Parity), RAID 6 (Double Parity), RAID 10 (Mirror+Striped), RAID Z2 (ZFS Double Parity).
**Choice:** **RAID 10** for a 6-drive DB server. It offers the best performance and fault tolerance (can lose 2 drives if mirrored pairs) compared to Z2 which has write penalties.

OP7 — **DNS Arch:** Internal AD DNS (Authoritative for internal). External DNS (Public). Split-horizon: Internal clients get internal IPs.
**Laptop:** Connects to VPN -> DNS tunnel -> Internal Resolver -> Internal Zone.
**Security:** DNSSEC signs external zone to prevent spoofing.

OP8 — **PKI Chain:** Root CA (Offline) -> Subordinate CA (Issuing) -> End Entity.
**Revocation:** CRL (List) vs. OCSP (Real-time check).
**Consequences:** Compromised Subordinate = Revoke certs issued by it. Compromised Root = Trust chain broken globally; must re-issue all certs and update trust stores.

OP9 — **WAN Consolidation:** MPLS (High cost, high reliability), IPsec (Low cost, variable latency), SD-WAN (Smart routing, cost-effective), WireGuard (Simple, secure).
**Recommendation:** **SD-WAN**. It optimizes VoIP traffic over the internet, reduces MPLS costs, and manages multiple links dynamically.

OP10 — **iDRAC/iLO/IPMI:** Out-of-band management interfaces. Critical for remote reboot/power cycle when OS is down.
**Risk:** Exposed to internet = Remote code execution vulnerabilities (e.g., iDRAC bugs) leading to full server compromise.

OP11 — **LVM vs. ZFS:** LVM manages volumes; ZFS manages filesystem + volume + RAID.
**Snapshots:** LVM snapshots are copy-on-write but can degrade performance. ZFS snapshots are instant, cheap, and immutable.
**Use:** LVM for simple volume management. ZFS for data integrity and snapshots.

OP12 — **Stretch Cluster:** 2 sites + Tiebreaker node. Ceph CRUSH rules must distribute replicas across sites.
**Latency:** < 5ms between sites.
**Split-Brain:** Without a tiebreaker, both sites might think they are primary. 2-site Ceph is bad without a tiebreaker because network partition causes data corruption risk.

OP13 — **DHCP DORA:** Discover, Offer, Request, Acknowledge.
**Unavailable:** Client uses APIPA (169.254.x.x) if lease expires and server is unreachable.

OP14 — **Managed vs. Unmanaged:** Managed switches support VLANs, QoS, ACLs, SNMP.
**Importance:** VLANs segment traffic; QoS prioritizes VoIP; ACLs secure access.

OP15 — **NTP:** Network Time Protocol.
**Critical:** Kerberos auth, log correlation, clustering.
**Drift:** Clock skew causes auth failures and log confusion.

OP16 — **Physical vs. VM vs. Container:** Physical = Bare metal. VM = Virtualized OS. Container = Virtualized App.
**Bare Metal:** Use for high-performance DBs, HPC, or when hardware passthrough is needed.

OP17 — **UPS vs. Generator:** UPS provides immediate battery power. Generator provides long-term power.
**ATS:** Automatic Transfer Switch switches load from Utility to UPS to Generator.
**Need:** Both to cover short outages (UPS) and long outages (Generator).

OP18 — **Active-Passive vs. Active-Active:** Active-Passive (one works, one waits). Active-Active (both work).
**PostgreSQL HA:** Pacemaker/Corosync manages VIP. STONITH (fencing) ensures failed node is powered off. VIP moves to active node on failover.

OP19 — **Backup Strategy:** 3-2-1 Rule (3 copies, 2 media, 1 offsite). Immutable backups prevent ransomware encryption.
**Test:** Regularly restore random files and full systems.
**Backup vs. DR:** Backup is data copy; DR is the plan/process to restore operations.

OP20 — **802.1X:** Supplicant (Client), Authenticator (Switch), Server (RADIUS).
**EAP:** EAP-TLS (Certificates), PEAP (Username/Password).
**Fail:** Device placed in Quarantine VLAN.
**MAB:** Switch sends MAC address to RADIUS for auth if 802.1X fails (for printers/IoT).
