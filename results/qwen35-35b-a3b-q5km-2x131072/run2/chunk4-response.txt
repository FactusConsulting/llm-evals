## Go

**G1** — An array is a fixed-size value type `[N]T`. A slice is a dynamic descriptor containing a pointer to an array, length, and capacity. When appending to a slice that reaches capacity, the Go runtime allocates a new, larger underlying array, copies the data, and updates the slice header.

**G2** — `var x int` declares a variable with an explicit type and zero value. `x := 0` is a short declaration that infers the type. You cannot use `:=` to redeclare a variable that already exists in the same scope (e.g., `x := 1` after `var x int`).

**G3** — A goroutine is a lightweight thread managed by the Go runtime, not the OS. The GMP model manages concurrency: **G** (goroutine), **M** (OS thread), and **P** (logical processor). The scheduler maps Gs to Ps, which run on Ms, allowing efficient multiplexing of goroutines on OS threads.

**G4** — Channels enable communication between goroutines. Unbuffered channels block both sender and receiver until the other is ready. Buffered channels have a queue; sending to a full buffered channel blocks until space is available. Receiving from an empty buffered channel blocks.

**G5** — The `context` package manages deadlines, cancellation, and request-scoped values.
```go
ctx, cancel := context.WithTimeout(context.Background(), 5*time.Second)
defer cancel()
resp, err := http.GetContext(ctx, "http://example.com")
```

**G6** — Go interfaces are satisfied implicitly (duck typing), unlike Java/C# which require explicit declaration. The empty interface `interface{}` (or `any`) holds any value. It is powerful for generic storage but dangerous because type assertions can panic if the type is incorrect.

**G7** — This is unsafe due to a data race: `ready` and `data` are accessed concurrently without synchronization. The memory model guarantees no visibility of writes across goroutines without synchronization. Correct alternatives include using `sync.Once`, `atomic.Bool`, or a channel to signal readiness.

**G8** — `sync.Mutex` provides exclusive access. `sync.RWMutex` allows multiple readers or one writer. `sync.Map` is optimized for sparse keys or read-heavy workloads where keys are added rarely. Trade-off: `sync.Map` has higher overhead per access than a `map` with a `Mutex` for dense keys.

**G9** — `go.mod` declares dependencies and module path. `go.sum` stores checksums for verification. `go mod tidy` removes unused dependencies and adds missing ones. MVS (Minimum Version Selection) selects the lowest version of a dependency that satisfies all constraints in the dependency graph.

**G10** — `defer` schedules a function call to run after the surrounding function returns. Multiple defers execute in LIFO (Last In, First Out) order. A common gotcha in loops is capturing the loop variable by reference; use `i := i` to shadow it.

**G11** — A pointer holds a memory address. Its zero value is `nil`. To create a pointer to a struct literal, use the address-of operator: `&MyStruct{Field: value}`.

**G12** — `init()` runs automatically before `main()` executes. A package can have multiple `init()` functions. They execute in the order they appear in the source file, and across packages, they run in dependency order.

**G13** — `new(T)` allocates zeroed memory for type `T` and returns a pointer `*T`. `make(T)` initializes slices, maps, or channels and returns the typed value (not a pointer). `new` works with any type; `make` only works with slices, maps, and channels.

**G14** — Errors are values implementing the `error` interface. The idiomatic pattern is `if err != nil { return err }`. `errors.Is` checks if an error matches a target (wrapping), while `errors.As` checks if an error can be assigned to a specific type. `==` only checks exact equality.

**G15** — Built-in composite types: `map`, `slice`, `channel`, `struct`. Zero values: `nil` for maps, slices, channels, and pointers; `0` for numeric types; `false` for bools; empty struct for structs.

**G16** — A race condition occurs when concurrent goroutines access shared data without synchronization. The `-race` flag instruments code to detect data races at runtime. Fix by using `sync.Mutex` to protect access or channels to transfer ownership.

**G17** — `select` chooses a ready channel case, similar to a switch but for channels. If multiple cases are ready, one is chosen pseudo-randomly. Implement timeout by adding a `case <-time.After(duration)`.

**G18** — Escape analysis determines if a variable lives on the stack or heap. If a variable escapes the function scope (e.g., returned), it is allocated on the heap. Inspect with `go build -gcflags="-m"`. Heap allocation incurs GC overhead; stack is faster.

**G19** — `reflect` allows runtime introspection. `reflect.Type` describes the type; `reflect.Value` holds the value. It is slow due to dynamic dispatch and type checks. Justified for serialization frameworks or generic utilities, but avoided in performance-critical paths.

**G20** — `//go:embed` embeds files/directories into the binary at compile time. It supports single files, directories, and glob patterns. Limitations: content is static at compile time. It interacts with `io/fs.FS` by returning an `embed.FS` that implements the interface.

## Rust

**R1** — Ownership is Rust's core memory safety feature. Rules: 1. Each value has one owner. 2. When the owner goes out of scope, the value is dropped. 3. Values can be moved, but not copied (unless `Copy` trait).

**R2** — `String` is an owned, heap-allocated, growable buffer. `&str` is a borrowed slice of UTF-8 bytes. Use `String` when you need to own/modify data; use `&str` for function arguments to avoid unnecessary allocation.

**R3** — Borrowing allows referencing data without ownership. Rules: You can have either one mutable reference (`&mut T`) OR any number of immutable references (`&T`), but not both simultaneously. References must always be valid.

**R4** — `Result<T, E>` represents success or error. `Option<T>` represents a value or `None`. The `?` operator propagates errors.
```rust
fn read_file(path: &str) -> Result<String, std::io::Error> {
    std::fs::read_to_string(path)
}
```

**R5** — `Box<T>` owns data on the heap with a single owner. `Rc<T>` (Reference Counted) allows multiple owners in a single thread. `Arc<T>` (Atomic Reference Counted) allows multiple owners across threads. Use `Arc` for shared state in async/multi-threaded contexts.

**R6** — Traits define shared behavior. Unlike Go, Rust traits must be implemented explicitly. `dyn Trait` enables dynamic dispatch (vtable). Generic bounds `T: Trait` enable static dispatch (monomorphization), which is generally faster.

**R7** — The function fails because the compiler cannot determine which input lifetime the return value borrows. Fix by annotating lifetimes: `fn longest<'a>(x: &'a str, y: &'a str) -> &'a str`. This tells the compiler the return lives as long as the inputs.

**R8** — `Send` means a type can be transferred between threads. `Sync` means `&T` can be shared. `Rc<T>` is not `Send` because its reference count is not atomic. `Arc<T>` uses atomic reference counting, making it `Send` and `Sync`.

**R9** — `async/await` creates a `Future` (state machine). The executor (e.g., Tokio) polls the future until completion. Unlike Go's preemptive goroutines, Rust async is cooperative; the task yields control explicitly.

**R10** — `unsafe` allows operations the compiler cannot verify. Examples: dereferencing raw pointers, calling unsafe functions, modifying `static mut`. Justified for FFI, low-level optimizations, or implementing safe abstractions over unsafe primitives.

**R11** — Pattern matching deconstructs values. `match` requires exhaustiveness (all cases covered). `if let` is for partial matching. The compiler enforces that `match` covers all possible variants of an enum.

**R12** — `Vec<T>` is a heap-allocated, dynamically sized array. `[T; N]` is a stack-allocated, fixed-size array. When `Vec` capacity is exceeded, it allocates a new, larger buffer and moves elements (reallocates).

**R13** — Enums are tagged unions (can hold different types). Unlike C/Java, they can hold data. `Option<T>` is `enum { Some(T), None }`. Rust has no `null`; `None` is the explicit representation of absence, preventing null pointer exceptions.

**R14** — `impl` defines methods. `&self` borrows the instance. `&mut self` mutably borrows. `self` consumes the instance. Associated functions (e.g., `String::new()`) do not take `self` and act as constructors.

**R15** — `derive` macros automatically implement traits. Common: `Debug`, `Clone`, `PartialEq`. Cannot derive if fields contain types that don't support the trait (e.g., `Fn` or `unsafe` types).

**R16** — `Cow<'a, T>` (Clone on Write) holds either a borrowed `&'a T` or an owned `Box<T>`. Useful when you want to avoid allocation if no modification is needed. Example: parsing a string that might be modified later.

**R17** — `mod` defines a module. `use` imports paths. `pub` makes items visible outside the crate. Modules map to files (`mod.rs` or `module.rs`). Rust 2018 improved path resolution (e.g., `crate::` vs `self::`).

**R18** — `Pin<T>` prevents a value from being moved in memory. Needed for self-referential structs and `Future` (which may hold self-references). `Unpin` is a marker trait indicating a type can be safely moved.

**R19** — Declarative macros (`macro_rules!`) match patterns. Procedural macros (derive, attribute) operate on tokens at compile time. Hygiene ensures macro variables don't leak into the caller's scope unless explicitly passed.

**R20** — Zero-cost abstractions mean high-level code compiles to efficient machine code. Generics use monomorphization (copy-paste code per type). Iterators are inlined with no allocation. Traits use static dispatch (inline) or dynamic dispatch (vtable).
