K1 — **Deployment vs StatefulSet:** A Deployment manages stateless applications with interchangeable pods (random names, no guaranteed order). A StatefulSet manages stateful applications requiring stable network IDs, ordered deployment/termination, and persistent storage per pod.

K2 — **ClusterIP Routing:** A Service assigns a virtual IP (VIP). `kube-proxy` (using `iptables` or `IPVS`) configures node-level rules to redirect traffic destined for the VIP to the IP addresses of the backing pods (Endpoints).

K3 — **PV, PVC, StorageClass:** A **PersistentVolume (PV)** is cluster storage (e.g., NFS, cloud disk). A **PersistentVolumeClaim (PVC)** is a user request for storage. A **StorageClass** defines dynamic provisioning parameters. Relation: PVC binds to a PV (static) or triggers PV creation via StorageClass (dynamic).

K4 — **CrashLoopBackOff Troubleshooting:** 1. `kubectl logs` to check application error output. 2. `kubectl describe pod` to check events (OOMKilled, probe failures). 3. Verify resource requests/limits against node capacity. 4. Check liveness/readiness probe configuration and health endpoint availability.

K5 — **CoreDNS Role:** CoreDNS provides DNS service discovery within the cluster. If CoreDNS pods are down, internal service discovery fails (pods cannot resolve other pods/services by name), though external DNS resolution may still work if configured.

K6 — **Scheduler Decision:** The scheduler filters nodes (removes ineligible ones) then scores remaining nodes. Factors: 1. **Resource Requests** (CPU/Mem fit). 2. **Node Affinity** (preferred/required labels). 3. **Taints/Tolerations** (node restrictions). 4. **Pod Affinity/Anti-Affinity** (co-location rules).

K7 — **Memory Scheduling/Usage:** Node has 4GB, 3.5GB requested. Available = 0.5GB (512Mi). Pod request = 256Mi. **Yes**, it will be scheduled (256Mi < 512Mi). If it uses 600Mi at runtime, it exceeds the 512Mi limit and will be **OOMKilled**.

K8 — **Ingress Lifecycle:** 1. External LB forwards traffic to Ingress Controller (NGINX). 2. NGINX matches Host/Path rules. 3. Traffic routed to Kubernetes Service (ClusterIP). 4. `kube-proxy` (iptables/IPVS) redirects to Pod IPs (Endpoints). 5. Request reaches Application Pod.

K9 — **NetworkPolicy:** A firewall for pods. Default behavior (no policy): All traffic allowed. When applied: Default deny for ingress/egress to selected pods unless explicitly allowed by the policy rules.

K10 — **RollingUpdate vs Recreate:** RollingUpdate replaces pods gradually (zero downtime). Recreate kills all pods before starting new ones. Choose Recreate for stateful apps requiring full shutdown or when config changes necessitate a complete restart.

K11 — **Namespace:** A logical cluster partition for resource isolation. Default namespaces: `default`, `kube-system`, `kube-public`, `kube-node-lease`.

K12 — **kubectl describe pod:** Used for debugging. Shows: 1. **Events** (scheduling, pull errors). 2. **Conditions** (Ready status details). 3. **Mounts/Env Vars** (volume and config details). `kubectl get` only shows summary status.

K13 — **ConfigMap:** A key-value store for non-sensitive config. Pod consumption methods: 1. **Environment Variables** (injected into container). 2. **Volume Mounts** (mounted as files in the filesystem).

K14 — **Service Types:** **ClusterIP** (internal cluster access only). **NodePort** (exposes service on each node's IP at a static port). **LoadBalancer** (provisions external cloud load balancer and exposes service externally).

K15 — **DaemonSet:** Ensures one pod runs on every node (or selected nodes). Use instead of Deployment for cluster-wide agents like logging (Fluentd), monitoring (Prometheus Node Exporter), or storage daemons.

K16 — **Taints/Tolerations vs Affinity:** Taints repel pods from nodes (node-side); Tolerations allow pods to ignore taints. Affinity attracts pods to nodes (pod-side). Use Case: Taints for dedicated GPU nodes; Affinity for co-locating app pods with specific storage nodes.

K17 — **Kubernetes Operator:** A custom controller that manages the lifecycle of a Custom Resource (CRD). Differs from standard controllers by automating complex operational tasks (backup, scaling, upgrades) for stateful applications.

K18 — **etcd Consistency/Recovery:** etcd uses strong consistency (linearizable). If quorum is lost, the cluster becomes read-only/unavailable. Recovery: Restore from a snapshot on a new node, rejoin it to the cluster, and ensure a majority (quorum) of nodes are online.

K19 — **RBAC:** **Role** (namespace-scoped permissions), **ClusterRole** (cluster-scoped), **RoleBinding** (links Role to Subject in namespace), **ClusterRoleBinding** (links ClusterRole to Subject cluster-wide).
*Manifest:*
```yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata: namespace: default
name: pod-reader
rules:
- apiGroups: [""]
  resources: ["pods", "pods/log"]
  verbs: ["get", "list", "watch"]
---
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata: namespace: default
name: read-pods
subjects:
- kind: User
  name: developer
  apiGroup: rbac.authorization.k8s.io
roleRef:
  kind: Role
  name: pod-reader
  apiGroup: rbac.authorization.k8s.io
```

K20 — **DNS Slow/ndots:** `ndots` defines how many dots trigger a full FQDN search. Default `ndots: 5` causes queries for every domain to try search domains first. Fix: Set `dnsConfig` in pod spec: `options: [{name: "ndots", value: "1"}]` to reduce unnecessary queries.

D1 — **Rebase vs Merge:** Rebase rewrites commit history to create a linear timeline. Merge preserves history with a merge commit. Avoid rebasing on shared branches to prevent history conflicts for collaborators.

D2 — **TCP vs UDP:** TCP is connection-oriented, reliable, ordered (HTTP, SSH). UDP is connectionless, unreliable, fast (DNS, VoIP).

D3 — **Race Condition:** Multiple threads access shared data concurrently without synchronization, leading to unpredictable results. Example: Two threads increment a shared counter without a lock, resulting in a lost update.

D4 — **Symmetric vs Asymmetric Encryption:** Symmetric uses one key (fast). Asymmetric uses public/private key pairs (secure exchange). TLS uses Asymmetric for the handshake (key exchange) and Symmetric for the actual data transfer.

D5 — **Container Image Layer:** Read-only filesystem layers. Layer caching: Place frequently changing instructions (e.g., `COPY .`) at the bottom of the Dockerfile to maximize cache hits for base layers.

D6 — **HTTP 503 Troubleshooting:** 1. Check upstream service health/status. 2. Review load balancer logs for timeouts. 3. Check application resource usage (CPU/Mem). 4. Verify dependencies (database, cache) connectivity.

D7 — **CAP Theorem:** Consistency, Availability, Partition Tolerance. **CP:** Zookeeper (prioritizes consistency during partition). **AP:** Cassandra (prioritizes availability during partition).

D8 — **Scaling:** Horizontal = add more nodes (stateless). Vertical = add resources to existing node (stateful). Patterns enabling horizontal scaling: Stateless design, sharding, message queues.

D9 — **Environment Variable:** Runtime configuration. Persist across sessions on Linux: Add `export VAR=value` to `~/.bashrc` or `~/.profile`.

D10 — **JSON vs YAML:** JSON is strict, machine-readable. YAML is human-readable, supports comments. Advantage JSON: Standard for web APIs. Advantage YAML: Readability for config files.

D11 — **REST API:** Architectural style for stateless communication. **GET** (retrieve), **POST** (create), **PUT** (replace), **DELETE** (remove).

D12 — **Version Pinning:** Locking dependency versions (e.g., `package-lock.json`). Important for reproducible builds to ensure the same code runs in all environments.

D13 — **Compiled vs Interpreted:** Compiled translates source to binary before execution (C, Go). Interpreted translates source to machine code at runtime (Python, JavaScript).

D14 — **Reverse Proxy:** Server receives requests and forwards to backend. Differs from forward proxy (client-side). Examples: NGINX (web serving), HAProxy (load balancing).

D15 — **Deadlock:** Processes waiting indefinitely for each other. Coffman Conditions: 1. Mutual Exclusion. 2. Hold and Wait. 3. No Preemption. 4. Circular Wait.

D16 — **Eventual Consistency:** Updates propagate over time. Cassandra uses tunable consistency levels (ONE, QUORUM, ALL). Read-after-write consistency is often handled via `LOCAL_QUORUM` to ensure the writer's data is visible locally.

D17 — **mTLS:** Mutual TLS where both client and server authenticate via certificates. Handshake: Client Hello -> Server Hello + Cert -> Client Cert -> Key Exchange -> Finished. Commonly used in Service Meshes (Istio) for service-to-service security.

D18 — **Hash Table:** Maps keys to indices via hash function. Collision resolution: Chaining (linked lists) or Open Addressing (probing). Time Complexity: Avg O(1), Worst O(n) (all collisions).

D19 — **Circuit Breaker:** Prevents cascade failures. States: **Closed** (normal), **Open** (fail fast), **Half-Open** (test recovery). Relates to retry: Retries only occur when closed/half-open; exponential backoff prevents overwhelming the failing service.

D20 — **Optimistic vs Pessimistic Concurrency:** Optimistic checks conflicts on commit (row versioning). Pessimistic locks data on read (`SELECT FOR UPDATE`). Choose Optimistic for low contention; Pessimistic for high contention/conflict scenarios.
