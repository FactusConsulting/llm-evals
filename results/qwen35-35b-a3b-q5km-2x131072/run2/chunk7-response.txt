AA1 — **Monolith vs. Microservices:** A monolith is a single, unified codebase deployed as one unit. Microservices are a collection of loosely coupled, independently deployable services.
**3 Problems Microservices Introduce:**
1.  **Distributed Complexity:** Network latency, partial failures, and eventual consistency issues replace local function calls.
2.  **Operational Overhead:** Requires orchestration (Kubernetes), distributed tracing, and centralized logging.
3.  **Data Consistency:** Maintaining ACID transactions across services requires complex patterns (e.g., Sagas) rather than simple DB transactions.

AA2 — **Sync vs. Async:** Synchronous communication blocks the caller until a response is received (request-response). Asynchronous communication allows the caller to continue processing without waiting for an immediate response.
**Examples:** Synchronous: HTTP/REST, gRPC. Asynchronous: RabbitMQ, Apache Kafka.

AA3 — **API Gateway vs. Reverse Proxy:** A reverse proxy routes traffic to backend servers (L4/L7). An API Gateway adds business logic on top of routing, such as authentication, rate limiting, and API versioning.
**Replacement:** An API Gateway cannot be replaced by a simple reverse proxy because the proxy lacks the necessary business logic features (auth, throttling) required for public-facing APIs. However, a Gateway *is* a type of reverse proxy.

AA4 — **Saga Pattern:** A sequence of local transactions where each updates the database and triggers the next. If one fails, compensating transactions undo previous changes.
**Choreography vs. Orchestration:** Choreography uses events to trigger services (decentralized); Orchestration uses a central coordinator to tell services what to do (centralized).
**Appropriate:** Choreography for simple, low-coupling flows. Orchestration for complex workflows requiring centralized error handling and state management.

AA5 — **CQRS:** Command Query Responsibility Segregation separates read (Query) and write (Command) operations into different models.
**Pairing with Event Sourcing:** Event Sourcing stores state changes as an immutable log of events. CQRS uses this log to project read models.
**Problem Solved:** It solves scalability (scale reads/writes independently) and auditability (full history of state changes), which traditional CRUD models obscure.

AA6 — **Circuit Breaker:** A pattern to prevent cascading failures by stopping calls to a failing service.
**3 States:** Closed (normal operation), Open (fail fast, no calls), Half-Open (test if service recovered).
**Vs. Retry:** Retry attempts to fix transient failures immediately. Circuit Breaker stops traffic to prevent overload.
**Together:** Use Retry for transient errors within the Closed state; use Circuit Breaker when errors persist to protect the system.

AA7 — **Multi-Tenant Isolation:**
1.  **Shared DB/Schema:** Lowest cost, highest noisy-neighbor risk, complex security (row-level), easiest ops.
2.  **Shared DB/Schema:** Medium cost, better isolation (schema level), harder migrations, moderate noisy-neighbor risk.
3.  **Separate DB:** Highest cost, best security/isolation, easiest backup/restore, lowest noisy-neighbor risk.

AA8 — **Consistency:** Eventual consistency allows temporary divergence; Strong consistency ensures immediate visibility of updates.
**Inventory Architecture:** Use **Strong Consistency**. Data Store: Relational (ACID) or Strongly Consistent Distributed DB (e.g., CockroachDB). Patterns: Optimistic locking (versioning) or Distributed Locks (Redis/Zookeeper) to reserve stock atomically.

AA9 — **Sidecar Proxy:** A container deployed alongside the application to handle network traffic.
**Envoy/Istio:** Envoy is the data plane proxy. Control Plane (Pilot) configures Envoy.
**mTLS:** Encrypts service-to-service traffic. Application TLS only protects client-to-service. mTLS ensures identity and encryption between internal services, preventing eavesdropping and spoofing.

AA10 — **Strangler Fig:** Incrementally migrate functionality by routing traffic to new services while keeping legacy running.
**Strategy:** Deploy an API Gateway. Route new features to .NET 8 microservices. Route legacy endpoints to the .NET Framework monolith. Gradually migrate features until the monolith is empty and can be decommissioned.

AA11 — **12-Factor App:** Methodology for building cloud-native apps.
**6 Factors:**
1.  **Codebase:** One repo, many deploys.
2.  **Dependencies:** Explicitly declared.
3.  **Config:** Stored in environment.
4.  **Backing Services:** Treated as attached resources.
5.  **Build/Run/Deploy:** Strict separation.
6.  **Processes:** Stateless, short-lived.
**Why:** Ensures portability, scalability, and maintainability in cloud environments.

AA12 — **Event-Driven (Kafka) vs. Request-Driven (REST):**
*   **Coupling:** Event (Loose), Request (Tight).
*   **Scalability:** Event (High), Request (Moderate).
*   **Debugging:** Event (Harder), Request (Easier).
*   **Ordering:** Event (Possible), Request (Guaranteed).
*   **Failure:** Event (Retry/Dead Letter), Request (Timeout/Error).
**Choice:** Event for decoupling/async processing. Request for real-time user interaction.

AA13 — **Load Balancer:** Distributes traffic across servers.
**L4 vs. L7:** L4 routes based on IP/Port (TCP/UDP). L7 routes based on HTTP headers/content.
**Use Case:** SSL Termination or routing `/api` to microservices and `/static` to CDN requires L7.

AA14 — **Message Queue vs. Log Broker:**
*   **MQ (RabbitMQ):** Tasks, fire-and-forget, deletes message after consumption.
*   **Log (Kafka):** Streams, retains messages for replay, high throughput.
**Pick:** MQ for task processing. Kafka for event streaming, audit logs, or data pipelines.

AA15 — **Container vs. VM:**
*   **Container:** OS-level virtualization, shares host kernel, lightweight.
*   **VM:** Hardware virtualization, full guest OS, heavier.
**Popularity:** Containers are faster to start, denser, and better suited for microservices orchestration.

AA16 — **Stateless vs. Stateful:**
*   **Stateless:** No session data stored on server (e.g., Web Server).
*   **Stateful:** Session data stored on server (e.g., Database).
**Scaling:** Stateless services are easier to scale horizontally because any instance can handle any request without data synchronization.

AA17 — **ORM:** Maps objects to database tables.
**Examples:** Entity Framework (.NET), Hibernate (Java).
**Trade-offs:** ORM increases dev speed and reduces boilerplate but can introduce performance overhead and "magic" SQL generation compared to raw SQL.

AA18 — **Bulkhead Pattern:** Isolates resources so a failure in one component doesn't exhaust resources for others.
**Implementation:** Use separate thread pools for different services, or deploy critical services on separate nodes/pods to isolate CPU/Memory failures.

AA19 — **OT vs. CRDT:**
*   **OT:** Server authoritative, transforms operations. High latency, complex.
*   **CRDT:** Commutative operations, merge locally. Low latency, eventual consistency.
**Choice:** **CRDT**. It supports offline editing because conflicts resolve automatically upon reconnection without server intervention.

AA20 — **Distributed Tracing:**
*   **Challenges:** Context loss across languages/services.
*   **Propagation:** W3C Trace Context (HTTP), B3 (Zipkin).
*   **Sampling:** Head (start), Tail (end).
*   **Storage:** Jaeger/Tempo.
*   **Kafka:** Inject trace ID into message headers (e.g., `X-Trace-ID`) to maintain continuity across async boundaries.

OP1 — **Hypervisor:**
*   **Type 1 (Bare-metal):** Runs directly on hardware (e.g., VMware ESXi, Microsoft Hyper-V).
*   **Type 2 (Hosted):** Runs on an OS (e.g., VirtualBox, VMware Workstation).

OP2 — **SAN vs. NAS:**
*   **SAN:** Block storage (FC, iSCSI). Used for databases requiring low latency.
*   **NAS:** File storage (NFS, SMB). Used for file sharing.

OP3 — **3-Tier vs. Leaf-Spine:**
*   **3-Tier:** Core/Distribution/Access. Hierarchical, potential bottlenecks.
*   **Leaf-Spine:** Two-tier, non-blocking, low latency. Solves oversubscription and scaling limits of 3-tier.

OP4 — **Proxmox HA:**
*   **Architecture:** 3+ nodes for quorum. Shared storage (Ceph/iSCSI).
*   **Fencing:** STONITH (Shoot The Other Node In Head) to prevent split-brain.
*   **Failure:** VMs restart on surviving nodes automatically.

OP5 — **VLAN Trunking:**
*   **802.1Q:** Tags Ethernet frames with VLAN ID.
*   **Linux Config:** Create sub-interfaces (e.g., `eth0.10`) with `ip link add link eth0 name eth0.10 type vlan id 10`.

OP6 — **RAID:**
*   **RAID 1:** Mirror. **RAID 5:** Striping + 1 parity. **RAID 6:** Striping + 2 parity. **RAID 10:** Mirror + Striping. **Z2:** ZFS RAID with parity.
*   **Choice:** **RAID 10** or **Z2**. For 6 drives, RAID 10 offers best performance/FT balance for DBs. Z2 offers better capacity utilization.

OP7 — **DNS Architecture:**
*   **Internal:** AD DNS servers. **External:** Public DNS.
*   **Split-Horizon:** Internal resolves private IPs, External resolves public.
*   **Laptop:** DHCP provides DNS IP. Conditional forwarders route internal queries to AD DNS.

OP8 — **PKI:**
*   **Chain:** Root CA -> Subordinate CA -> End-Entity Cert.
*   **Compromise:** Subordinate CA compromise requires revoking the sub and reissuing certs. Root CA compromise requires rebuilding the entire PKI.

OP9 — **WAN Consolidation:**
*   **Comparison:** MPLS (High cost/reliability), IPsec (Low cost/variable), SD-WAN (Smart routing), WireGuard (Simple).
*   **Recommendation:** **SD-WAN**. Provides QoS for VoIP, path selection, and cost efficiency.

OP10 — **iDRAC/iLO/IPMI:**
*   **Function:** Out-of-band hardware management (power, console).
*   **Exposure:** If exposed to internet, attackers can reboot, reset BIOS, or access disk images remotely.

OP11 — **LVM vs. ZFS:**
*   **LVM:** Volume management only. **ZFS:** Filesystem + Volume + RAID.
*   **Snapshots:** LVM (Copy-on-write, slower). ZFS (Instant, cheap, immutable).

OP12 — **Stretch Cluster:**
*   **Design:** 3 nodes total (2 sites + 1 tiebreaker) or 3 nodes per site with quorum witness.
*   **Ceph:** CRUSH rules must span sites. Monitors must be distributed.
*   **Split-Brain:** Without a tiebreaker, two sites may both think they are primary, corrupting data.

OP13 — **DHCP:**
*   **DORA:** Discover, Offer, Request, Acknowledge.
*   **Unavailable:** Client uses APIPA (169.254.x.x) if lease expires and server is down.

OP14 — **Managed vs. Unmanaged:**
*   **Managed:** VLANs, QoS, ACLs, Port Mirroring.
*   **Importance:** Essential for segmentation, traffic prioritization, and security in enterprises.

OP15 — **NTP:**
*   **Importance:** Certificates, logs, and clustering rely on accurate time.
*   **Drift:** Causes auth failures, log confusion, and cluster split-brain.

OP16 — **Physical vs. VM vs. Container:**
*   **Bare Metal:** Run workloads directly on hardware.
*   **When:** High-performance computing, low-latency DBs, or when virtualization overhead is unacceptable.

OP17 — **UPS vs. Generator:**
*   **UPS:** Short-term power (minutes). **Generator:** Long-term power (hours).
*   **ATS:** Automatically switches load from utility to UPS, then to generator.

OP18 — **HA Clustering:**
*   **Active-Passive:** One node active, one standby. **Active-Active:** Both handle traffic.
*   **PostgreSQL:** Pacemaker/Corosync manages VIP. STONITH ensures failed node is powered off. Failover moves VIP to new master.

OP19 — **Backup/Recovery:**
*   **3-2-1:** 3 copies, 2 media, 1 offsite.
*   **Immutable:** Prevents ransomware deletion.
*   **Backup vs. DR:** Backup is data copy; DR is the plan/process to restore operations at a new site.

OP20 — **802.1X:**
*   **Roles:** Supplicant (Client), Authenticator (Switch), Server (RADIUS).
*   **EAP:** EAP-TLS (Cert), PEAP (User).
*   **Fail:** Device placed in Quarantine VLAN.
*   **MAB:** Authenticates device by MAC address if 802.1X fails (less secure).
