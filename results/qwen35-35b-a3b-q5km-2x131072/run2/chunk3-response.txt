T1 — `tofu plan` performs a dry run to preview changes without modifying infrastructure. `tofu apply` executes the changes proposed by the plan to create, update, or destroy resources.

T2 — State is a JSON file tracking the mapping between OpenTofu resources and real-world infrastructure. Remote state is important for team collaboration, state locking (preventing concurrent writes), and security (storing secrets away from local machines).

T3 — `count` uses integer indices; `for_each` uses a map or set of strings. `for_each` is preferred when resources have unique identifiers (like names) because it avoids index shifting issues when items are added or removed, ensuring stable resource references.

T4 — OpenTofu detects this via state drift. Running `tofu plan` compares the state file against the actual infrastructure. If changes are found, OpenTofu proposes to revert them to match the state. If the state is lost, manual changes may require `tofu import`.

T5 —
```hcl
resource "aws_instance" "web" {
  for_each = toset(["web1", "web2", "web3"])
  ami      = "ami-12345"
  instance_type = "t2.micro"
  tags = {
    Name = each.key
  }
}
```

T6 — `terraform_remote_state` reads the state file of another root module. `output` values are the interface defined in a module. The recommended alternative is using `terraform_remote_state` to access specific outputs (not the whole state) to minimize coupling and security risks. Alternatively, use Terraform Cloud/Enterprise workspaces for direct output sharing.

T7 — The `null_resource` provisioner may not finish writing the kubeconfig file before the output is evaluated. Fix: Add `depends_on = [null_resource.kubeconfig_gen]` to the resource consuming the kubeconfig. Ensure the `null_resource` waits for the cluster to be ready (e.g., via a polling script).

T8 — `terraform import` is a CLI command to add existing resources to state. Limitations: Hard to script, requires matching config. The `import` block (1.5+) is declarative, version-controlled, and allows importing resources directly in the configuration file, improving CI/CD integration.

T9 — A `variable` is an input parameter passed into the configuration. An `output` is data exposed from the configuration to be used by other modules or the CLI.

T10 — A provider is a plugin that interacts with a specific cloud or service API. Without `tofu init`, providers are not downloaded, causing `tofu plan` to fail with "Provider not installed" errors.

T11 — `.terraform.lock.hcl` records the exact provider versions and checksums used. It should be committed to ensure reproducible builds and prevent unexpected provider updates.

T12 — `tofu destroy` removes all managed resources. Target a single resource with `tofu destroy -target=aws_instance.example`.

T13 — A data source reads existing data from an external system without creating or managing it. Example: `data "aws_ami" "ubuntu"` to find the latest Ubuntu AMI ID instead of hardcoding it.

T14 — `locals` define computed values within the configuration scope. `variables` are inputs from outside. Use `locals` for intermediate calculations or formatting to keep the configuration DRY and readable.

T15 — `lifecycle` manages resource behavior. `create_before_destroy` ensures new resources are created before old ones are deleted (zero downtime). `prevent_destroy` protects resources from accidental deletion. `ignore_changes` prevents OpenTofu from tracking specific attribute drift.

T16 — A `moved` block allows refactoring resource addresses (e.g., moving a resource to a new module) without destroying and recreating it. It updates the state mapping to reflect the new address.

T17 — The dependency graph determines execution order. Implicit dependencies are inferred from resource references. Explicit dependencies use `depends_on`. `depends_on` reduces parallelism because it forces sequential execution, which should be used sparingly to avoid slowing down plans.

T18 — 1. Use `-refresh=false` (Tradeoff: Misses state drift). 2. Modularize code (Tradeoff: Increased complexity). 3. Reduce `data` sources (Tradeoff: Less dynamic data). 4. Optimize backend (S3 + DynamoDB) (Tradeoff: Cost/Setup).

T19 — State locking prevents concurrent modifications. If a lock persists (crash), use `tofu force-unlock <id>` to break it. Risk: Data corruption if another apply is actually running.

T20 — 1. Configure `backend "s3"` in `main.tf`. 2. Run `tofu init -migrate-state`. 3. Verify state in S3 console. 4. Update CI/CD to use the new backend. The existing local state file is uploaded to S3 during migration.

A1 — A playbook is a YAML file defining a sequence of tasks. A role is a reusable package of tasks, handlers, and templates. A task is a single action executed by Ansible.

A2 — `ansible_facts` contains gathered system information. Access OS family via `{{ ansible_facts['os_family'] }}`.

A3 — `command` runs without a shell (no pipes). `shell` runs with a shell (supports pipes). `raw` runs without SSH setup (no Python interpreter required), useful for initial bootstrap.

A4 — Idempotency means running a task multiple times yields the same result. `apt: name=htop` is idempotent. `shell: echo "hello" > /tmp/file` is not (overwrites every time).

A5 —
```yaml
- name: Install htop
  package:
    name: htop
  when: ansible_facts['os_family'] == 'Debian'

- name: Install htop on RedHat
  package:
    name: htop
  when: ansible_facts['os_family'] == 'RedHat'
  become: yes
  notify: restart service
```
*(Note: Use `package` module with `name` list or conditional logic in a single task using `when`)*.
*Correction for single task:*
```yaml
- name: Install htop
  package:
    name: htop
  when: ansible_facts['os_family'] == 'Debian'

- name: Install epel-release + htop on RedHat
  package:
    name:
      - epel-release
      - htop
  when: ansible_facts['os_family'] == 'RedHat'
```

A6 — Priority (High to Low): Extra vars > Command line > Play vars > Host vars > Group vars > Role defaults > Inventory vars.

A7 — Use `serial: 3` in the play to process 3 nodes at a time. Group nodes in inventory (e.g., `rke2_nodes`). Use `wait_for` or `serial` to ensure the first batch completes before the next.

A8 — `ansible-vault encrypt_string`. Encrypts a specific string. To encrypt a variable in a file, use `ansible-vault edit filename.yml` and encrypt specific lines.

A9 — Inventory lists hosts and groups. Static inventory is a file (INI/YAML). Dynamic inventory is a script or API call that generates inventory dynamically.

A10 — `ansible webservers -m shell -a "df -h"`.

A11 — A handler is a task triggered by `notify` when a task reports a change. It runs after all tasks in the play complete.

A12 — `ansible.cfg` configures Ansible behavior. Common settings: `inventory`, `remote_user`, `forks`.

A13 — `register` stores task output in a variable. Use it in subsequent tasks via `{{ registered_var.stdout }}`.

A14 — `include_tasks` is evaluated at runtime (dynamic). `import_tasks` is evaluated at compile time (static). Use `include_tasks` for conditionally loading tasks.

A15 — Ansible Galaxy is a repository for roles and collections. `requirements.yml` lists them. Install via `ansible-galaxy install -r requirements.yml`.

A16 — Connection plugins manage how Ansible connects to hosts. `ssh` (remote Linux), `local` (control node), `network_cli` (network devices).

A17 — `delegate_to: localhost` runs a task on the control node. `run_once` ensures it runs once. Example: Gather facts from remote, then push config to a central server on localhost.

A18 — 1. Increase `forks` (Tradeoff: CPU/RAM usage). 2. Enable `pipelining` (Tradeoff: SSH config). 3. Fact caching (Tradeoff: Cache invalidation). 4. Use Mitogen (Tradeoff: Compatibility).

A19 — `{{ }}` evaluates expressions. `{% %}` executes statements (loops/ifs). `{# #}` is a comment.
Snippet:
```jinja2
upstream backend {
{% for server in servers %}
    server {{ server.ip }} weight={{ server.weight }}; # Health check: {{ server.health }}
{% endfor %}
}
```

A20 — Use `serial: 1` for one server at a time. `pre_tasks`: Deregister from LB. `post_tasks`: Register, run `uri` health check. `block/rescue`: If health check fails, rollback (re-register, restart).
