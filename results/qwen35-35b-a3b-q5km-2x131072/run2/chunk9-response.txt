SC1-A
**Root Cause:** The failure stems from the Kubernetes Pod Security Standards (PSS) upgrade from Baseline to Restricted.
1.  **PSS Interaction:** Baseline allows root users and privileged containers. Restricted forbids root (`runAsUser` must be non-zero) and requires `runAsNonRoot: true` to be explicitly set and enforced.
2.  **Image vs. Spec:** The new base image (`chainguard/static:latest`) defaults `USER` to root (UID 0). The pod spec sets `securityContext.runAsNonRoot: true` but lacks an explicit `runAsUser`.
3.  **The Conflict:** Under Baseline, the cluster allowed the container to run as root (UID 0) despite `runAsNonRoot: true` being present in the spec (or the admission controller didn't enforce it strictly). Under Restricted, the admission controller enforces that the effective UID must not be 0. Since `runAsUser` is not set, the container attempts to run as root (image default). This violates the Restricted policy.
4.  **CrashLoop:** The container likely starts but crashes immediately because the application cannot perform operations restricted for root in a Restricted context (e.g., binding to ports < 1024, writing to specific volumes, or capability drops enforced by the kubelet).

SC1-C
**Remediation Strategy:**
1.  **Image Fix:** Rebuild the container images with a non-root user (e.g., `USER 1000`) in the Dockerfile.
2.  **Manifest Fix:** Update all pod specs to explicitly set `securityContext.runAsUser: 1000` (matching the image) to satisfy Restricted PSS.
3.  **CI Gate:** Add an OPA Gatekeeper or Kyverno policy in CI that scans Helm charts/manifests. It must verify `securityContext.runAsUser` is set to a non-zero value if `runAsNonRoot: true` is present.
4.  **Rollout:** Use a Canary deployment. Update 10% of pods first. Monitor CrashLoopBackOff metrics. If stable, roll out to 100%. This prevents a full outage if the fix is incomplete.

SC2-A
**Stale Data & Replication:**
1.  **Replication Lag:** Cross-region async replication introduces latency. Promoting a replica cuts off the source. Any WAL (Write-Ahead Log) not shipped from us-east-1 to eu-west-1 before promotion is lost.
2.  **WAL Shipping:** PostgreSQL streams WAL records. In async mode, the primary acknowledges the write before the replica confirms receipt.
3.  **Data Loss:** If the primary fails before WAL reaches the replica, those transactions are lost.
4.  **Quantification:** Check `pg_stat_replication` on the old primary (if accessible) or the new primary's `pg_wal` logs. Compare the `replay_lag` before promotion. The data loss window is the time between the last committed transaction on the primary and the last WAL record shipped to the replica.

SC2-B
**Terraform Operations:**
1.  **Import Replica:** `terraform import aws_db_instance.promoted_replica <new_replica_arn>`
2.  **Update State:** `terraform state mv aws_db_instance.old_primary aws_db_instance.promoted_replica` (if renaming resource) or update the resource definition in code to match the new ARN.
3.  **Detach Old:** Update the Terraform config to remove the old instance resource or set `lifecycle { ignore_changes = [arn] }` if keeping it for backup, then `terraform apply` to destroy the old instance.
4.  **Dependencies:** Use `terraform state rm` to remove old security groups/Route53 records from state if they are no longer needed, then recreate them pointing to the new instance.

SC2-C
**Architecture Design:**
1.  **RDS Proxy:** Use RDS Proxy instead of PgBouncer in K8s. It handles connection pooling centrally and supports failover better for serverless or managed DBs.
2.  **Route53:** Configure Route53 Health Checks on the RDS endpoint. Use Failover Routing Policy to switch DNS to the new region endpoint automatically upon failure.
3.  **App Retry:** Implement exponential backoff with jitter in the application connection logic to handle transient failover errors.
4.  **RPO < 1s:** Standard RDS async replication cannot guarantee < 1s RPO. **Recommend Aurora Global Database.** It uses a dedicated replication layer with sub-second latency and automatic failover, meeting the RPO requirement better than standard RDS.

SC3-C
**Hardened CI/CD:**
1.  **Image Signing:** Use Cosign to sign images upon build. `cosign sign --key cosign.key <image>`.
2.  **Admission Control:** Deploy Kyverno or OPA Gatekeeper to verify signatures on pod creation. Reject unsigned images.
3.  **Pinning:** Pin base images by digest (e.g., `node:20-slim@sha256:...`) in Dockerfiles, not tags.
4.  **SBOM:** Generate Software Bill of Materials (SBOM) using Syft during build and store in artifact registry.
5.  **Notary v2:** Integrate Notary v2 for content trust. It ensures image integrity and authenticity before pulling. It works alongside Cosign but provides a standardized trust model.

SC4-A
**Top 5 Blockers:**
1.  **Local Sessions:** Sticky sessions via F5 are incompatible with stateless K8s pods. **Solution:** Move sessions to Redis (ElastiCache).
2.  **Local NFS:** `/data/uploads` on bare metal. **Solution:** Use NFS CSI Driver or S3 for object storage.
3.  **Oracle 19c:** On-prem DB. **Solution:** Migrate to AWS RDS for Oracle or Aurora PostgreSQL.
4.  **IBM MQ:** On-prem mainframe integration. **Solution:** Use AWS MQ for RabbitMQ/ActiveMQ or IBM MQ on EC2/EKS.
5.  **Sticky Sessions:** F5 relies on IP affinity. **Solution:** Use Kubernetes Ingress with `sticky` annotations or move to Redis for session state.

SC6-C
**Network Upgrade Comparison:**
1.  **AWS Direct Connect:** Best for bandwidth/cost at scale. Dedicated 10Gbps+ link. Low latency. Fits Zero Trust (private link).
2.  **VPN ECMP:** Cheaper, but limited by 1Gbps per tunnel. Failover is slower. Encryption is IPsec (less granular).
3.  **Mesh (Tailscale/Netbird):** Best for Zero Trust (identity-based). Adds latency. Not suitable for high-throughput DB replication.
**Recommendation:** Upgrade to Direct Connect for core traffic. Use Mesh for admin access.

SC10-A
**Architecture Comparison:**
1.  **Active-Passive (Velero):** Backup to S3. Restore on failover. RPO depends on backup frequency (15 mins ok). RTO depends on restore time (1 hour ok).
2.  **Active-Active (Istio):** Traffic split. RPO depends on DB replication. 12ms latency is acceptable for async DB replication.
3.  **Stateful Replication:**
    *   **Postgres:** Patroni with async replication (RPO < 15 mins).
    *   **Redis:** Sentinel with cross-region replication.
    *   **RabbitMQ:** Federation or Quorum Queues.
    *   **MinIO:** Site Replication (S3 compatible).
**Recommendation:** Active-Passive with Velero for simplicity and guaranteed RPO/RTO compliance given the 200TB data size.
