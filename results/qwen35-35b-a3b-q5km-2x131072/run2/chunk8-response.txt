CL1 — **IaaS, PaaS, SaaS**
*   **IaaS (Infrastructure as a Service):** Provides virtualized computing resources over the internet. You manage OS, middleware, runtime, data, and apps. *Example:* AWS EC2, Azure Virtual Machines.
*   **PaaS (Platform as a Service):** Provides hardware and software tools (usually for app development) over the internet. You manage data and apps; provider manages OS/runtime. *Example:* AWS Elastic Beanstalk, Azure App Service.
*   **SaaS (Software as a Service):** Provides fully functional applications over the internet. You manage only user access/data; provider manages everything else. *Example:* Microsoft 365, Salesforce.

CL2 — **Shared Responsibility Model**
*   **Concept:** Security is a shared duty. Provider secures the cloud (physical, network, hypervisor); Customer secures what is in the cloud (data, identity, OS, config).
*   **IaaS VM:** Provider secures physical host/hypervisor. Customer secures OS, patching, firewall, data, and app.
*   **Managed Kubernetes (e.g., EKS/AKS):** Provider secures control plane and underlying nodes. Customer secures worker nodes (OS), cluster config, network policies, and application code.
*   **SaaS (M365):** Provider secures infrastructure, app, and data center. Customer secures identity (MFA), data classification, and device compliance.

CL3 — **VPC, Subnet, SG, NACL**
*   **VPC:** Isolated virtual network in the cloud.
*   **Subnet:** Range of IP addresses within a VPC (can be public or private).
*   **Security Group (SG):** Virtual firewall attached to instances (ENI). **Stateful:** Allows return traffic automatically.
*   **Network ACL (NACL):** Virtual firewall attached to subnets. **Stateless:** Must explicitly allow inbound and outbound traffic.
*   **Layering:** VPC contains Subnets. Subnets contain Instances. SG filters traffic at the Instance level. NACL filters traffic at the Subnet level.

CL4 — **Multi-Account Strategy**
*   **Landing Zone:** Secure, compliant baseline environment (networking, identity, logging) for new accounts.
*   **Organizational Units (OUs):** Logical grouping of accounts (e.g., Prod, Dev, Security) for policy application.
*   **Guardrails:** SCPs (Service Control Policies) or Azure Policy to prevent risky actions (e.g., disabling logging, creating public S3 buckets).
*   **Shared Networking:** Transit Gateway (AWS) or Hub-and-Spoke (Azure) connects VPCs/VNets centrally for traffic inspection and routing.
*   **Centralized Logging:** Aggregates logs from all accounts to a central security account for SIEM.
*   **Single-Account Problem:** High blast radius (compromise affects all), billing complexity, inability to isolate environments (Prod/Dev), and compliance violations (shared credentials).

CL5 — **Infrastructure as Code (IaC)**
*   **IaC:** Managing infrastructure via machine-readable definition files rather than physical hardware configuration.
*   **Terraform/OpenTofu:** Declarative, multi-cloud, state-based. Best for complex, multi-cloud environments.
*   **Pulumi:** Imperative/Declarative, uses general-purpose languages (TS, Python). Best for devs comfortable with code.
*   **CloudFormation/Bicep:** Native to AWS/Azure. Best for single-cloud, deep integration, and CI/CD pipelines within that ecosystem.
*   **Crossplane:** K8s-native IaC. Best for managing cloud resources via K8s API (GitOps for infra).

CL6 — **Cloud-Native Autoscaling**
*   **HPA:** Scales pods based on CPU/Memory or custom metrics.
*   **VPA:** Adjusts resource requests/limits for pods (requires restart).
*   **Cluster Autoscaler:** Adds/removes nodes based on pending pods.
*   **KEDA:** Event-driven autoscaler (e.g., queue length, Kafka).
*   **KEDA vs HPA:** Use KEDA when scaling is driven by external events (e.g., SQS queue depth) rather than internal resource usage (CPU).

CL7 — **Disaster Recovery (DR) Strategy**
*   **RPO/RTO:** RPO (data loss tolerance) dictates replication sync (sync=0 RPO, async>0). RTO (downtime tolerance) dictates active-active vs passive.
*   **Active-Active:** Traffic split across regions. Lowest RTO/RPO, highest cost.
*   **Active-Passive:** One region active, one standby. Lower cost, higher RTO.
*   **Database:** Sync replication (synchronous commit) for critical data; Async for geo-redundancy.
*   **DNS Failover:** Route53 Health Checks trigger failover to secondary region.
*   **State:** Externalize state (S3, DynamoDB) to avoid local storage dependency.
*   **Models:**
    *   *Pilot Light:* DB running, app off.
    *   *Warm Standby:* App scaled down, DB running.
    *   *Multi-Site:* Full active-active.

CL8 — **GitOps Model**
*   **Concept:** Git repository is the single source of truth. Controllers sync cluster state to Git state.
*   **ArgoCD:** Push-based (or pull), UI-focused, supports multi-cluster.
*   **FluxCD:** Pull-based, K8s-native (CRDs), lightweight.
*   **Secrets:** Sealed Secrets (encrypted in Git), External Secrets Operator (fetches from Vault/AWS Secrets Manager), Vault Agent.
*   **Drift:** If manual change occurs, GitOps controller detects drift and reverts changes to match Git state (auto-remediation).

CL9 — **Stateful Legacy App to K8s**
*   **StatefulSet:** Ensures stable network IDs and ordered deployment/termination.
*   **PV/PVC:** Use `local` storage class with `hostPath` for low latency (binds to specific node).
*   **Headless Service:** Provides stable DNS names for StatefulSet pods (no load balancing).
*   **Init Containers:** Handle config file generation or permission setup before main app starts.
*   **ConfigMaps:** Mount as volumes for config files at specific paths.
*   **PDB:** Ensure minimum available pods during updates to prevent downtime.

CL10 — **Identity (SA, Workload, IRSA)**
*   **Service Account:** K8s identity for pods.
*   **Workload Identity:** Maps K8s SA to Cloud Identity (Azure/AWS/GCP).
*   **IRSA:** AWS specific implementation of Workload Identity (OIDC).
*   **Mounting Creds:** Bad practice (leakage risk, rotation difficulty, long-lived keys).
*   **Federation:** Uses short-lived tokens via OIDC. No secrets in pod; pod assumes role via metadata service.

CL11 — **Storage Types**
*   **Object (S3/Blob):** Unstructured data, HTTP API. High durability (11 9s), high availability. Use for backups, media, logs.
*   **Block (EBS/Managed Disk):** Attached to VM, low latency. High availability (99.999%), durability varies by type. Use for OS, DBs.
*   **File (EFS/Azure Files):** Shared file system (NFS/SMB). High availability. Use for shared configs, home dirs.

CL12 — **Zero-Trust Network Architecture (K8s)**
*   **Network Policies:** Restrict pod-to-pod traffic (default deny).
*   **Service Mesh (mTLS):** Encrypts service-to-service traffic (Istio/Linkerd).
*   **Pod Identity:** Workload identity federation (no static creds).
*   **Admission Controllers:** OPA/Kyverno to enforce policies on deployment (e.g., no root user).
*   **Image Signing:** Cosign/Notary to verify image integrity.
*   **Runtime Security:** Falco to detect anomalies (e.g., shell in container).
*   **Secrets:** External secrets (Vault).
*   **Diff:** Traditional = Perimeter firewall. Zero Trust = Verify every request, assume breach, least privilege.

CL13 — **CDN**
*   **Concept:** Distributed network of edge servers caching content closer to users.
*   **Services:** AWS CloudFront, Azure CDN, Cloudflare.
*   **Use Case:** Use CDN for static assets (images, JS, CSS) to reduce latency and origin load. Do not use for dynamic, personalized, or real-time data.

CL14 — **Availability Zones & Regions**
*   **Region:** Geographic area (e.g., us-east-1).
*   **Availability Zone (AZ):** Isolated data center within a Region.
*   **Multi-AZ:** Deploy across AZs to survive data center failure.
*   **Zonal Resources:** EBS volumes (tied to AZ).
*   **Regional Resources:** ELB, RDS (Multi-AZ), S3 (Region-wide).

CL15 — **Managed Database Service**
*   **Definition:** RDS, Cloud SQL, Azure SQL.
*   **Provider Handles:** Hardware provisioning, OS patching, backups, replication, failover, scaling.
*   **Customer Handles:** Schema design, queries, user permissions, data encryption.

CL16 — **Cloud Identity & Least Privilege**
*   **IAM Users:** Individual identities.
*   **Groups:** Collections of users.
*   **Roles:** Temporary credentials for entities (users, services).
*   **Policies:** JSON permissions.
*   **Least Privilege:** Grant only necessary permissions. Apply via Groups/Roles, not Users. Use Deny statements for sensitive actions.

CL17 — **Serverless Computing**
*   **Definition:** Event-driven execution without managing servers (AWS Lambda, Azure Functions).
*   **Cold Starts:** Latency when function initializes after inactivity. Matters for latency-sensitive apps.
*   **Poor Fit:** Long-running processes, stateful apps, predictable high-throughput (reserved instances cheaper), strict compliance requiring dedicated hardware.

CL18 — **Kubernetes Ingress Options**
*   **Ingress Resource:** Standard K8s resource for HTTP routing.
*   **Gateway API:** New standard, supports TCP/UDP, better traffic splitting, multi-vendor.
*   **Cloud LB Annotations:** Cloud provider specific (e.g., AWS NLB).
*   **Service Mesh Gateway:** Istio/Linkerd ingress (advanced traffic mgmt).
*   **TLS:** LB (Edge termination), Ingress (Edge termination), Mesh (mTLS internal).
*   **Gateway API:** Use for complex routing, multi-cluster, or non-HTTP protocols.

CL19 — **Cost Optimization Strategy**
*   **RI/SP:** Commit to usage (1-3 years) for discounts.
*   **Spot/Preemptible:** Interruptible instances (up to 90% off). Use for stateless/batch.
*   **Right-sizing:** Match instance size to actual usage.
*   **Storage Tiering:** Move old data to Glacier/Archive.
*   **Idle Detection:** Find unattached EBS, idle LBs.
*   **FinOps:** Culture of cost accountability.
*   **Showback/Chargeback:** Showback = Report costs to teams. Chargeback = Bill teams directly.
*   **Tools:** AWS Cost Explorer, CloudHealth, Kubecost.

CL20 — **Multi-Cloud Strategy**
*   **Abstraction:** Kubernetes (runtime), Terraform (IaC).
*   **Networking:** Direct Connect/ExpressRoute + Transit Gateway/Hub.
*   **Identity:** OIDC federation (e.g., Entra ID to AWS/Azure).
*   **Trade-offs:** Avoid lock-in vs. increased complexity, skill fragmentation, data gravity.
*   **Justified:** Regulatory requirements, catastrophic redundancy, vendor negotiation leverage.
*   **Unnecessary:** Most startups/SMES benefit from single-cloud depth and speed.

OT1 — **Purdue Model (ISA-95)**
*   **Level 0:** Physical Process (Sensors, Actuators).
*   **Level 1:** Basic Control (PLCs, RTUs).
*   **Level 2:** Area Supervisory (SCADA, HMIs).
*   **Level 3:** Operations (MES, Historian).
*   **Level 4:** Enterprise (ERP, Business).

OT2 — **IT vs OT Networks**
*   **IT:** Focus on Confidentiality/Integrity. Patching is frequent.
*   **OT:** Focus on Availability/Safety. Legacy systems, proprietary protocols.
*   **Patching:** OT patching requires downtime validation and safety testing. Standard IT patching can crash legacy controllers.

OT3 — **DMZ (IDMZ)**
*   **Role:** Buffer zone between IT (L4) and OT (L1-L3).
*   **Traffic:** One-way (OT to IT) for data collection. Two-way (IT to OT) only for specific commands (e.g., historian pull).
*   **Danger:** Direct L4 to L1 allows attackers to bypass security controls and send malicious commands directly to PLCs.

OT4 — **OT Protocols**
*   **Modbus TCP:** L1/L2 (Control). Simple, unencrypted.
*   **OPC UA:** L2/L3 (Control/Info). Secure, platform-independent.
*   **EtherNet/IP:** L1/L2 (Control). CIP protocol over Ethernet.
*   **PROFINET:** L1/L2 (Control). Real-time industrial Ethernet.
*   **MQTT:** L3/L4 (Info). Lightweight pub/sub for cloud connectivity.

OT5 — **Data Diode vs Firewall**
*   **Firewall:** Filters traffic both ways.
*   **Data Diode:** Hardware-enforced one-way communication.
*   **Use:** When data must leave OT but nothing can enter (e.g., exporting logs to cloud).
*   **Limitation:** Cannot receive commands or updates from the other side.

OT6 — **Data Flow to Cloud**
*   **Flow:** PLC -> Historian -> OPC UA Server -> Edge Gateway -> MQTT Broker -> Cloud Ingestion.
*   **Security Boundaries:** DMZ between Edge Gateway and Cloud. Firewall between Historian and Edge. Encryption (TLS) on all links.

OT7 — **IEC 62443 Zones & Conduits**
*   **Zone:** Logical grouping of assets with similar security requirements.
*   **Conduit:** Communication path between zones.
*   **Model:**
    *   *Zone 1:* Corporate Network (L4).
    *   *Zone 2:* DMZ (IDMZ).
    *   *Zone 3:* SCADA Servers (L3).
    *   *Zone 4:* PLCs/SIS (L1/L2).
    *   *Conduits:* Firewalls between zones.

OT8 — **Network Segmentation Strategy**
*   **VLANs:** Separate VLANs per production line.
*   **Firewalls:** Layer 3 firewalls between VLANs and DMZ.
*   **Switches:** Managed switches with ACLs.
*   **Remote Access:** Jump hosts with MFA and session recording.
*   **Transition:** Parallel run (new network alongside old), phased migration during maintenance windows.

OT9 — **SIS vs BPCS**
*   **BPCS:** Basic Process Control System (PLCs/DCS). Manages normal operations.
*   **SIS:** Safety Instrumented System (ESD). Independent system to shut down safely during emergencies.
*   **IEC 61511:** Standard for Safety Instrumented Systems.
*   **Isolation:** SIS must be air-gapped/isolated to prevent cyber attacks from disabling safety functions.
*   **Ransomware on SIS:** Could prevent safe shutdown, leading to physical damage or injury.

OT10 — **PLC, DCS, SCADA**
*   **PLC:** Logic controller for specific machines.
*   **DCS:** Distributed Control System for continuous process control (refineries).
*   **SCADA:** Supervisory Control and Data Acquisition (monitoring remote sites).
*   **Coexist:** Yes. PLCs feed SCADA; DCS manages process; SCADA monitors DCS/PLCs.

OT11 — **Remote Access Architecture**
*   **Jump Host:** Bastion host for all access.
*   **MFA:** Required for all remote sessions.
*   **Recording:** Session recording for audit.
*   **Vendor Access:** Time-limited, scoped accounts.
*   **No Direct RDP:** Prevents lateral movement from corporate network to OT devices.

OT12 — **Incident Response (Nation-State)**
*   **Monitoring:** IDS/Anomaly detection on OT traffic.
*   **Inventory:** Know all assets to identify compromised ones.
*   **Steps:** Contain (segment network), Investigate, Eradicate, Recover.
*   **Coordination:** Notify Control Room immediately.
*   **Safe Ops:** Switch to manual control if automation is compromised.
*   **Operator Role:** Prioritize physical safety over data recovery.

OT13 — **HMI**
*   **Definition:** Human-Machine Interface. Visual display for operators to monitor/control process.
*   **Role:** Real-time visualization and manual override.
*   **Difference:** Engineering Workstation is for programming/configuring logic; HMI is for operation.

OT14 — **Historian**
*   **Definition:** Time-series database for process data.
*   **Importance:** Trend analysis, compliance, troubleshooting.
*   **Plant Historian (L3):** Real-time ops data.
*   **Enterprise Historian (L4):** Aggregated data for business analytics.

OT15 — **Ladder Logic**
*   **Definition:** Programming language resembling relay logic diagrams.
*   **Device:** PLC.
*   **Dominance:** Deterministic execution, safety certification, legacy familiarity, ease of troubleshooting for electricians.

OT16 — **RTU vs PLC**
*   **RTU:** Remote Terminal Unit. Optimized for comms and telemetry over long distances.
*   **PLC:** Optimized for local control logic.
*   **RTU Use:** Pipelines, SCADA remote sites, harsh environments.

OT17 — **Analog vs Digital Signals**
*   **Analog:** Continuous voltage/current (4-20mA).
*   **Digital:** Discrete on/off or binary.
*   **4-20mA:** Preferred over 0-20mA because 4mA represents 0% (Live Zero). If current drops to 0mA, it indicates a wire break, not a zero reading.

OT18 — **OT Asset Inventory**
*   **Concept:** Knowing all hardware/software on the network. Foundation of security.
*   **Tools:** Passive (Network TAPs, Sniffers) vs Active (Scanners).
*   **Active Scanning:** Dangerous. Can crash legacy devices or trigger safety interlocks. Use Passive monitoring.

OT19 — **MITRE ATT&CK for ICS**
*   **Tactics:** Reconnaissance, Initial Access, Execution, Impact, Defense Evasion.
*   **Techniques:** E.g., "Exploit Public-Facing Application" (Initial Access), "Data Manipulation" (Impact).
*   **Difference:** Focuses on physical impact and safety, not just data theft.
*   **Usage:** Map existing controls to tactics to find gaps (e.g., no network segmentation for Impact).

OT20 — **Converged IT/OT SOC**
*   **Log Collection:** Passive taps (SPAN ports) to avoid impacting OT.
*   **SIEM:** Centralized correlation.
*   **Rules:** OT-specific (e.g., PLC write attempts).
*   **Staff:** IT analysts (threat hunting) + OT Engineers (process safety).
*   **False Positives:** High risk in OT (can cause unnecessary shutdowns). Tune rules carefully.
*   **Firmware Alert:** Verify integrity, isolate device, patch during maintenance, validate safety function.
