T1 — Easy: `tofu plan` calculates the execution plan and shows what changes will be made without applying them (dry run). `tofu apply` executes the plan, creating, updating, or destroying resources to match the configuration.

T2 — Easy: State is a JSON file mapping real-world resources to your configuration. Remote state is important for collaboration (sharing state among team members), locking (preventing concurrent writes), and security (storing sensitive data off local machines).

T3 — Medium: `count` uses integer indices; `for_each` uses a map or set of strings. `for_each` is preferred when resource names are dynamic or when you need to avoid index shifting issues (e.g., deleting an item in the middle of a list) because it ties resources to unique keys rather than indices.

T4 — Medium: This is called "drift". `tofu plan` detects the difference between the state and the real infrastructure. `tofu apply` will attempt to reconcile the drift by updating the resource to match the configuration, or fail if the change is destructive and not allowed.

T5 — Medium:
```hcl
resource "aws_instance" "web" {
  for_each = toset(["web1", "web2", "web3"])
  ami      = "ami-12345678"
  instance_type = "t2.micro"
  tags = {
    Name = each.key
  }
}
```

T6 — Hard: `terraform_remote_state` reads the entire state file from a backend, which can be a security risk and performance heavy. Output values expose specific data points. The recommended alternative is using **Terraform Cloud / OpenTofu Cloud** workspaces, which allow secure, RBAC-controlled access to outputs without exposing the raw state file.

T7 — Hard: `remote-exec` provisioners run asynchronously on the target host. The output is often evaluated before the provisioner finishes writing the kubeconfig file. **Fix:** Use `depends_on` on the resource consuming the kubeconfig to ensure the `null_resource` completes, or use a `local-exec` provisioner to generate the config on the control node where it is immediately available.

T8 — Medium: `tofu import` is a CLI command to import existing resources into state. Limitations: manual, error-prone, doesn't update config. The `import` block (1.5+) allows importing resources declaratively within the configuration file, making it easier to manage in CI/CD and version control.

T9 — Easy: A `variable` is an input parameter passed into the module or root configuration. An `output` is data exposed from the module to the root configuration or CLI. Variables are inputs; outputs are results.

T10 — Easy: A provider is a plugin that interacts with a specific API (e.g., AWS, Azure). If you run `tofu plan` without `tofu init`, it fails because the provider plugins are not downloaded and the `.terraform` directory is not initialized.

T11 — Easy: `.terraform.lock.hcl` records the exact versions of providers and plugins used. It should be committed to version control to ensure reproducible builds and prevent unexpected provider updates that could break infrastructure.

T12 — Easy: `tofu destroy` removes all resources managed by the configuration. To target a single resource, use the `-target` flag (e.g., `tofu destroy -target=aws_instance.web`).

T13 — Easy: A data source reads data from an external system without creating or managing it. Example: Using `data "aws_ami" "ubuntu"` to find the latest Ubuntu AMI ID instead of hardcoding it in a managed resource.

T14 — Medium: `locals` are internal variables for calculations within a module (not inputs). `variables` are inputs from outside. Use `locals` for complex logic, string manipulation, or derived values that shouldn't be exposed as inputs.

T15 — Medium: `lifecycle` controls resource behavior. `create_before_destroy` creates new resource before destroying old (zero downtime). `prevent_destroy` protects resources from accidental deletion. `ignore_changes` ignores drift in specific attributes (e.g., tags managed externally).

T16 — Hard: A `moved` block allows you to change the address of a resource (e.g., module path) in the configuration without destroying and recreating it. OpenTofu maps the old state address to the new one, preserving the resource.

T17 — Hard: The dependency graph determines execution order. Implicit dependencies are inferred from references (e.g., `aws_instance.web.id`). Explicit dependencies use `depends_on`. `depends_on` reduces parallelism because it forces sequential execution, which should be used sparingly to avoid slowing down plans.

T18 — Hard: 1. `plan -refresh=false` (skip state refresh, tradeoff: may miss drift). 2. Increase `parallelism` (tradeoff: higher API rate limits). 3. State pruning (remove unused resources, tradeoff: manual effort). 4. Modularization (split large modules, tradeoff: refactoring effort).

T19 — Hard: State locking prevents concurrent writes using a backend (e.g., DynamoDB). If a lock isn't released, subsequent operations fail. You can manually break it with `tofu force-unlock <LOCK_ID>`. Risk: Concurrent writes may corrupt state if the lock was held due to a crash.

T20 — Hard: 1. Add `backend` block (S3/DynamoDB) to config. 2. Run `tofu init -migrate-state`. 3. Verify state file exists in S3. 4. Run `tofu plan` to ensure no drift. 5. Run `tofu apply`. The existing local state file is moved to the remote backend.

A1 — Easy: A **Playbook** is a YAML file defining the execution flow. A **Role** is a directory structure for reusable logic. A **Task** is a single action (module execution) within a playbook or role.

A2 — Easy: `ansible_facts` contains gathered system information. Access OS family via `{{ ansible_facts['os_family'] }}`.

A3 — Medium: `command` runs without shell (no pipes). `shell` runs with shell (supports pipes). `raw` runs without setup (no Python module transfer), useful for initial bootstrap. Use `command` for security, `shell` for complex logic, `raw` for bootstrapping.

A4 — Medium: Idempotency means running a task multiple times produces the same result. `apt: name=htop` is idempotent (installs only if missing). `shell: rm -rf /` is not idempotent (destructive).

A5 — Medium:
```yaml
- name: Install htop
  package:
    name: htop
  become: yes
  when: ansible_os_family == "Debian"

- name: Install htop on RedHat
  package:
    name: htop
  become: yes
  when: ansible_os_family == "RedHat"
  # Note: epel-release usually handled by a separate task or role
```
*Correction:* To do it in a single task file logic:
```yaml
- name: Install htop
  package:
    name: htop
  become: yes
  when: ansible_os_family == "Debian"

- name: Install epel-release and htop on RedHat
  package:
    name:
      - epel-release
      - htop
  become: yes
  when: ansible_os_family == "RedHat"
```

A6 — Hard: Priority (High to Low): Extra vars (`-e`) > `set_fact` > Host vars > Group vars > Role vars > Role defaults. `set_fact` is runtime and overrides static vars.

A7 — Hard: Use `serial: 3` in the play to limit concurrency. Group nodes into `first_3` and `rest`. Use `wait_for` to ensure the first 3 are ready before the rest start, or use `serial` with `max_fail_percentage` to control flow.

A8 — Medium: `ansible-vault` encrypts sensitive data. To encrypt a single variable, use the `!vault` tag inline in the YAML file: `secret: !vault | $ANSIBLE_VAULT;1.1;AES256...`.

A9 — Easy: Inventory defines hosts and groups. **Static** is a file (INI/YAML). **Dynamic** is a script/API that generates inventory at runtime (e.g., AWS EC2).

A10 — Easy: `ansible webservers -m shell -a "df -h"`.

A11 — Easy: A **handler** is a task that runs only when notified by another task (e.g., `notify: restart service`). It differs from a regular task by running at the end of the play or when triggered.

A12 — Easy: `ansible.cfg` configures Ansible behavior. Common settings: `host_key_checking` (False), `forks` (number of parallel processes), `remote_user` (default SSH user).

A13 — Easy: `register` stores the output of a task in a variable. Use it in subsequent tasks via `when: my_var.stdout == "success"`.

A14 — Medium: `import_tasks` is static (evaluated at parse time, no conditionals). `include_tasks` is dynamic (evaluated at runtime, supports conditionals). Use `import` for performance, `include` for dynamic logic.

A15 — Medium: Ansible Galaxy is a repository for roles and collections. Use `ansible-galaxy install -r requirements.yml` to install dependencies defined in the file.

A16 — Medium: Connection plugins manage communication. **SSH** (remote Linux/Unix), **Local** (control node), **Network_CLI** (network devices like Cisco). Use SSH for servers, Local for localhost tasks, Network_CLI for routers/switches.

A17 — Hard: `delegate_to` runs a task on a specific host. `run_once` runs it once across the inventory. Example: Gather facts from all hosts, then update a DNS record on `localhost` once using the aggregated data.

A18 — Hard: 1. Increase `forks` (tradeoff: memory usage). 2. Enable `pipelining` (tradeoff: SSH config changes). 3. Use `fact_caching` (tradeoff: cache invalidation). 4. Use `Mitogen` (tradeoff: installation complexity).

A19 — Hard: `{{ }}` outputs variables. `{% %}` control structures (loops/if). `{# #}` comments.
Template:
```jinja2
upstream backend_servers {
{% for server in servers %}
    server {{ server.ip }} weight={{ server.weight }}; # Health check: {{ server.health_check }}
{% endfor %}
}
```

A20 — Hard: 1. Set `serial: 1` (or batch size). 2. `pre_tasks`: Deregister from LB. 3. Deploy app. 4. `post_tasks`: Health check via `uri` module. 5. `block/rescue`: If health check fails, rollback (revert deployment, re-register LB).
