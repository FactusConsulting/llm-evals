AA1 — **Monolith vs. Microservices**
A monolith is a single, unified codebase deployed as one unit. Microservices decompose this into independent, loosely coupled services deployed separately.
**3 Problems Microservices Introduce:**
1.  **Distributed Complexity:** Network latency, partial failures, and eventual consistency issues replace local memory access.
2.  **Operational Overhead:** Requires orchestration (Kubernetes), service discovery, and complex CI/CD pipelines.
3.  **Data Consistency:** Distributed transactions are hard; maintaining ACID across services requires patterns like Sagas.

AA2 — **Sync vs. Async Communication**
**Synchronous:** Caller waits for a response (blocking). Example: HTTP/REST, gRPC.
**Asynchronous:** Caller sends a message and continues; response comes later (non-blocking). Example: Message Queues (RabbitMQ), Event Streams (Kafka).
**Use Case:** Sync for immediate user feedback; Async for background processing (e.g., sending emails).

AA3 — **API Gateway vs. Reverse Proxy**
A **Reverse Proxy** routes traffic to backend servers (L4/L7) and handles SSL termination. An **API Gateway** adds business logic: authentication, rate limiting, API versioning, and analytics.
**Can it replace entirely?** Yes for external traffic, but not necessarily for internal service-to-service traffic. Internal traffic often uses a Service Mesh (sidecar proxies) for mTLS and observability without the Gateway's business logic overhead.

AA4 — **Saga Pattern**
A Saga manages distributed transactions across services by breaking them into local transactions. If one fails, compensating transactions undo previous steps.
**Choreography:** Services emit events; others react. *Pros:* Loose coupling. *Cons:* Hard to debug, no central view. *Use:* Simple flows.
**Orchestration:** A central coordinator tells services what to do. *Pros:* Centralized control, easier debugging. *Cons:* Single point of failure (mitigated by redundancy). *Use:* Complex workflows.

AA5 — **CQRS & Event Sourcing**
**CQRS:** Separates read (Query) and write (Command) models.
**Event Sourcing:** State is derived from a sequence of immutable events.
**Why paired?** CQRS allows optimized read models (denormalized) while Event Sourcing provides the audit trail and source of truth for writes.
**Problem Solved:** Traditional CRUD struggles with high write/read scaling mismatches and lacks historical state reconstruction.

AA6 — **Circuit Breaker Pattern**
Prevents cascading failures by stopping calls to a failing service.
**States:**
1.  **Closed:** Normal operation.
2.  **Open:** Calls fail immediately (service deemed down).
3.  **Half-Open:** Allow test requests; if successful, close; if failed, reopen.
**Vs. Retry:** Retry handles transient spikes; Circuit Breaker handles persistent failures to save resources. Use both: Retry for transient errors, Circuit Breaker if retries exceed threshold.

AA7 — **Multi-Tenant Isolation Models**
1.  **Shared DB, Shared Schema:** Lowest cost, highest noisy-neighbor risk. Security relies on row-level security.
2.  **Shared DB, Separate Schema:** Medium cost. Better isolation (schema-level permissions), easier backups per tenant.
3.  **Separate DB:** Highest cost, highest security/isolation. No noisy-neighbor risk. Best for regulated industries.
**Trade-off:** Cost/Complexity increases with isolation. Shared schema is easiest to manage but riskiest for security/performance.

AA8 — **Consistency & Inventory**
**Eventual:** Updates propagate over time (e.g., DNS). **Strong:** All nodes see the same data immediately (e.g., Banking).
**Architecture:** Inventory requires **Strong Consistency** to prevent overselling.
**Solution:** Use a relational database (PostgreSQL) with **Pessimistic Locking** (`SELECT ... FOR UPDATE`) or a distributed locking service (Redis Redlock).
**Pattern:** Reservation system (hold stock for 10 mins) + ACID transactions. Avoid eventual consistency for stock counts.

AA9 — **Sidecar Proxy & Service Mesh**
A **Sidecar** is a container running alongside the main app (e.g., Envoy) handling network traffic.
**Envoy/Istio:** Envoy is the data plane (handles traffic); Istio Control Plane configures Envoy.
**mTLS:** Encrypts traffic between services. Application-level TLS requires code changes; mTLS is transparent to the app, ensuring identity verification and encryption without modifying business logic.

AA10 — **Strangler Fig Pattern**
Incrementally migrate a legacy system by routing specific functionality to new services.
**Strategy:**
1.  Deploy a Gateway in front of the .NET Framework monolith.
2.  Build a new .NET 8 microservice for a specific feature (e.g., "Checkout").
3.  Configure Gateway to route `/checkout` to the new service, all else to the monolith.
4.  Repeat until the monolith is fully replaced.

AA11 — **12-Factor App**
1.  **Codebase:** One codebase, many deploys.
2.  **Dependencies:** Explicitly declare and isolate.
3.  **Config:** Store in environment, not code.
4.  **Backing Services:** Treat as attached resources.
5.  **Build, Release, Run:** Strictly separate stages.
6.  **Processes:** Stateless, short-lived.
7.  **Port Binding:** Export services via port.
8.  **Concurrency:** Scale via process model.
9.  **Disposability:** Fast startup/shutdown.
10. **Dev/Prod Parity:** Keep environments similar.
11. **Logs:** Treat logs as event streams.
12. **Admin Processes:** Run as one-off commands.
**Why:** Ensures portability, scalability, and maintainability in cloud-native environments.

AA12 — **Event-Driven vs. Request-Driven**
**Event-Driven (Kafka):** Loose coupling, high scalability, replayability. *Cons:* Debugging is hard, ordering is complex, failure handling requires retries/sagas.
**Request-Driven (REST/gRPC):** Tight coupling, synchronous, easy debugging. *Cons:* Cascading failures, harder to scale independently.
**Choice:** Use Request-driven for user-facing APIs (real-time). Use Event-driven for background processing, data pipelines, and decoupled microservices.

AA13 — **Load Balancer**
Distributes traffic across servers.
**Layer 4 (Transport):** Routes based on IP/Port. Fast, no content inspection.
**Layer 7 (Application):** Routes based on HTTP headers, URL, cookies.
**Use Case:** Layer 7 is required for SSL termination, routing `/api` to v1 and `/v2` to v2, or A/B testing based on cookies.

AA14 — **Message Queue vs. Log Broker**
**Message Queue (RabbitMQ):** Tasks/Work items. Messages are deleted after consumption. Good for task distribution.
**Log Broker (Kafka):** Event streams. Messages persist and can be replayed. Good for auditing, analytics, and event sourcing.
**Pick:** RabbitMQ for "do this task once." Kafka for "record this event for later."

AA15 — **Container vs. VM**
**Container:** Shares host OS kernel, isolates processes. **VM:** Virtualizes hardware, runs full OS.
**Why Popular:** Containers are lightweight, start in seconds, and ensure environment consistency ("works on my machine"). Ideal for microservices density.

AA16 — **Stateless vs. Stateful**
**Stateless:** No session data stored on the server (e.g., API server). Scales horizontally easily because any instance can handle any request.
**Stateful:** Stores session/data locally (e.g., Redis, DB). Harder to scale (requires sticky sessions or data replication).
**Example:** Stateless = Web Server. Stateful = Database.

AA17 — **ORM**
Maps database tables to programming language objects.
**Examples:** Hibernate (Java), Entity Framework (.NET).
**Trade-offs:** ORMs speed up development and reduce boilerplate but can generate inefficient SQL and hide performance bottlenecks. Raw SQL offers maximum control and performance.

AA18 — **Bulkhead Pattern**
Isolates resources (threads, connections) for different services to prevent one failure from exhausting shared resources.
**Implementation:** Use separate thread pools for each downstream service. If Service A fails, it only consumes its pool, leaving Service B's pool available. In Kubernetes, use separate namespaces or resource quotas.

AA19 — **OT vs. CRDT**
**OT:** Server calculates transformations to merge edits. *Cons:* Complex, requires server authority, hard offline.
**CRDT:** Data structures mathematically guarantee convergence without coordination. *Pros:* Offline support, peer-to-peer.
**Choice:** **CRDTs** for offline-first collaborative editing. They allow clients to edit locally and merge automatically without server intervention, ensuring consistency upon reconnection.

AA20 — **Distributed Tracing**
**Context Propagation:** Pass trace ID via headers (W3C Trace Context, B3) across services.
**Sampling:** Head-based (decide at start) vs. Tail-based (decide after latency). Tail-based is better for debugging slow requests.
**Storage:** Jaeger/Tempo store spans; queryable via UI.
**Kafka:** Inject trace context into message headers. Consumers must extract and continue the trace. Without this, the trace breaks at the queue boundary.

OP1 — **Hypervisor Types**
**Type 1 (Bare-metal):** Runs directly on hardware. Examples: VMware ESXi, Microsoft Hyper-V Server.
**Type 2 (Hosted):** Runs on top of an OS. Examples: Oracle VirtualBox, VMware Workstation.
**Use:** Type 1 for production data centers; Type 2 for development/testing.

OP2 — **SAN vs. NAS**
**SAN (Storage Area Network):** Block-level storage. High performance. Protocols: Fibre Channel, iSCSI. *Use:* Databases, VMs.
**NAS (Network Attached Storage):** File-level storage. Easier sharing. Protocols: NFS, SMB/CIFS. *Use:* File shares, backups.

OP3 — **Network Architecture**
**3-Tier:** Core (Backbone), Distribution (Policy), Access (Edge). Problem: Oversubscription and latency at the core.
**Leaf-Spine:** Every leaf connects to every spine. Solves oversubscription (non-blocking) and reduces latency (max 2 hops). Preferred for modern data centers.

OP4 — **Proxmox HA Design**
**Nodes:** Minimum 3 for quorum.
**Quorum:** Majority vote required to form cluster.
**Fencing:** STONITH (e.g., DRAC/iLO) to power off failed nodes to prevent split-brain.
**Storage:** Ceph (distributed) or Shared iSCSI/NFS.
**Failure:** If a node dies, HA manager restarts VMs on surviving nodes (after fencing the dead node).

OP5 — **VLAN Trunking**
**802.1Q:** Adds a tag to Ethernet frames to identify VLAN membership.
**Linux Config:** Create sub-interfaces.
`ip link add link eth0 name eth0.10 type vlan id 10`
`ip link add link eth0 name eth0.20 type vlan id 20`
Assign IPs to `eth0.10` and `eth0.20`.

OP6 — **RAID Levels**
**RAID 1:** Mirror (50% capacity). **RAID 5:** Striping + Parity (N-1 capacity). **RAID 6:** Double Parity (N-2). **RAID 10:** Mirror + Stripe (50% capacity). **ZFS Z2:** Double Parity (N-2).
**Choice:** **RAID 10** for a 6-drive DB server. It offers the best write performance and fault tolerance (can lose 3 drives if mirrored correctly) compared to RAID 5/6 rebuild times.

OP7 — **DNS Architecture**
**Internal:** AD Domain Controllers run DNS. Clients query these for internal names.
**External:** Public DNS (e.g., Route53) for public IPs.
**Split-Horizon:** Internal DNS resolves internal IPs; External resolves public IPs.
**Forwarders:** Internal DNS forwards unknown queries to external DNS.
**Laptop:** Connects via DHCP. If on LAN, queries AD DNS. If on VPN, tunnel queries to AD DNS.

OP8 — **PKI Chain of Trust**
**Root CA:** Offline, highly secure. Signs Subordinate CA.
**Subordinate CA:** Online, issues end-entity certs.
**Revocation:** CRL (list) vs OCSP (real-time check).
**Compromise:** Root compromised = Trust entire chain broken (must re-issue all). Subordinate compromised = Revoke subordinate, issue new, re-issue certs.

OP9 — **WAN Consolidation**
**MPLS:** Reliable, expensive, managed. **IPsec:** Cheap, internet-based, variable latency. **SD-WAN:** Intelligent routing over multiple links. **WireGuard:** Lightweight encryption.
**Recommendation:** **SD-WAN.** Supports VoIP (QoS), cost-effective (uses internet), and manages latency better than static IPsec.

OP10 — **Out-of-Band Management**
**iDRAC/iLO/IPMI:** Hardware-level management independent of OS.
**Critical:** Allows reboot/recovery if OS is frozen.
**Risk:** If exposed to internet, attackers can gain root access to the server hardware (power cycle, BIOS access) even if OS is secure.

OP11 — **LVM vs. ZFS**
**LVM:** Logical volume manager. Snapshots are copy-on-write but thin provisioning can be risky.
**LVM-Thin:** Overcommits storage.
**ZFS:** Filesystem + Volume Manager. Copy-on-write, checksums, native snapshots.
**Use:** LVM for simple Linux volumes. ZFS for data integrity and snapshots (e.g., Proxmox).

OP12 — **Stretch Cluster / DR**
**Design:** 2 sites + 1 Witness node (tiebreaker).
**Ceph:** CRUSH rules must distribute OSDs across sites.
**Latency:** <5ms for Ceph monitors.
**Split-Brain:** If network fails, both sites think they are primary.
**Why 2-site Ceph is bad:** Without a tiebreaker (witness), Ceph cannot decide which site to keep, risking data corruption.

OP13 — **DHCP DORA**
**Process:** Discover, Offer, Request, Acknowledge.
**Unavailable:** Client uses APIPA (169.254.x.x) if lease expires and server is down. No internet access.

OP14 — **Managed vs. Unmanaged Switch**
**Unmanaged:** Plug-and-play.
**Managed:** VLANs, QoS, ACLs, SNMP monitoring, Port Mirroring.
**Importance:** VLANs segment traffic; QoS prioritizes VoIP; ACLs secure access.

OP15 — **NTP**
**Critical:** Logs, Kerberos auth, clustering, and distributed tracing rely on synchronized time.
**Drift:** Clock skew causes auth failures (Kerberos tickets expire), split-brain in clusters, and incorrect log correlation.

OP16 — **Physical vs. VM vs. Container**
**Physical:** Bare metal. **VM:** Virtualized OS. **Container:** Virtualized App.
**Bare Metal Use:** High-performance computing, low-latency DBs, hardware passthrough needs.

OP17 — **UPS vs. Generator**
**UPS:** Battery backup for short term (minutes). **Generator:** Long term power.
**ATS:** Automatic Transfer Switch switches load from Utility to UPS to Generator.
**Why Both:** UPS bridges the gap until the generator starts.

OP18 — **Active-Passive vs. Active-Active**
**Active-Passive:** One node handles traffic; standby takes over. **Active-Active:** Both handle traffic.
**PostgreSQL HA:** Pacemaker/Corosync cluster.
**Fencing:** STONITH ensures failed node is powered off.
**VIP:** Virtual IP floats to active node. Failover = VIP moves + DB promoted.

OP19 — **Backup & Recovery**
**3-2-1:** 3 copies, 2 media types, 1 offsite.
**RPO/RTO:** Recovery Point/Time Objective.
**Types:** Full, Incremental, Differential.
**Immutable:** WORM storage to prevent ransomware deletion.
**Test:** Regular restore drills.
**Backup vs. DR:** Backup is data copy; DR is the plan/process to restore operations.

OP20 — **802.1X**
**Roles:** Supplicant (Client), Authenticator (Switch), Server (RADIUS).
**EAP:** EAP-TLS (Certificates, secure) vs PEAP (Password, tunneled).
**Dynamic VLAN:** Switch assigns VLAN based on user identity.
**Failure:** Port blocked or placed in Guest VLAN.
**MAB:** Switch sends MAC address to RADIUS as password. Used for printers/IoT that don't support 802.1X.
