CL1 — **IaaS, PaaS, SaaS**
*   **IaaS (Infrastructure as a Service):** Provides virtualized computing resources (VMs, storage, networks). You manage OS, middleware, runtime, data, and apps. *Example:* AWS EC2.
*   **PaaS (Platform as a Service):** Provides hardware and software tools (runtime, DB, OS) needed for development. You manage data and apps. *Example:* AWS Elastic Beanstalk.
*   **SaaS (Software as a Service):** Provides fully managed software applications. You manage only user access and configuration. *Example:* Microsoft 365.

CL2 — **Shared Responsibility Model**
*   **Concept:** Security is a joint effort. Provider secures the cloud (hardware, facilities); Customer secures what is in the cloud (data, config).
*   **IaaS VM:** Provider secures physical host/network. Customer secures OS, patching, firewall, data, IAM.
*   **Managed Kubernetes:** Provider secures control plane, nodes, networking. Customer secures pod config, RBAC, network policies, app security.
*   **SaaS (M365):** Provider secures app, infrastructure, data center. Customer secures identity (MFA), data classification, device compliance, user access.

CL3 — **Networking Components (VPC, Subnet, SG, NACL)**
*   **VPC:** Isolated virtual network in the cloud.
*   **Subnet:** IP range within a VPC, mapped to an Availability Zone.
*   **Security Group (SG):** Stateful firewall attached to instances (ENI). Allows allow-listing only.
*   **Network ACL (NACL):** Stateless firewall attached to subnets. Allows allow/deny lists.
*   **Layering:** VPC contains Subnets. Instances in Subnets have SGs. Subnets have NACLs.
*   **Stateful vs Stateless:** SG tracks connection state (return traffic allowed automatically). NACL evaluates each packet independently (must explicitly allow return traffic).

CL4 — **Multi-Account Strategy**
*   **Landing Zone:** Secure, compliant baseline environment for onboarding accounts.
*   **Organizational Units (OUs):** Logical grouping (e.g., Prod, Dev, Security) for policy application.
*   **Guardrails:** SCPs (Service Control Policies) to prevent risky actions (e.g., disabling CloudTrail).
*   **Shared Networking:** Transit Gateway (AWS) or Hub-and-Spoke (Azure) connects VPCs/VNets centrally for traffic inspection.
*   **Centralized Logging:** Aggregates logs to a dedicated Security/Log account for analysis.
*   **Single-Account Problem:** High blast radius (compromise affects all), billing confusion, lack of isolation for compliance, shared resource contention.

CL5 — **Infrastructure as Code (IaC)**
*   **IaC:** Managing infrastructure via machine-readable definition files rather than physical hardware configuration.
*   **Terraform/OpenTofu:** Declarative, multi-cloud, state-based. *Choose:* Multi-cloud, large teams, ecosystem maturity.
*   **Pulumi:** Imperative/Declarative using general languages (TS, Python). *Choose:* Devs prefer coding over HCL, complex logic.
*   **CloudFormation/Bicep:** Native to AWS/Azure. *Choose:* Single-cloud, tight integration, CI/CD native support.
*   **Crossplane:** Kubernetes-native IaC. *Choose:* K8s-centric teams managing cloud resources via K8s API.

CL6 — **Autoscaling Patterns**
*   **HPA:** Scales pods based on CPU/Memory or custom metrics.
*   **VPA:** Adjusts resource requests/limits for pods (requires restart).
*   **Cluster Autoscaler:** Adds/removes nodes based on pending pods.
*   **KEDA:** Event-driven autoscaler (e.g., Kafka queue depth, HTTP requests).
*   **KEDA vs HPA:** Use KEDA when scaling is driven by external events (message queues, cron) rather than standard resource utilization.

CL7 — **Disaster Recovery (DR) Strategy**
*   **RPO/RTO:** RPO (data loss tolerance), RTO (downtime tolerance).
*   **Active-Active:** Traffic split across regions. Low RTO/RPO, high cost.
*   **Active-Passive:** One region active, one standby. Higher RTO, lower cost.
*   **Database:** Sync (strong consistency, latency) vs Async (replication lag, higher availability).
*   **DNS Failover:** Route53 Health Checks trigger failover to secondary region.
*   **State:** Externalize state (S3, RDS) to avoid local storage dependency.
*   **DR Models:**
    *   *Pilot Light:* Core DB running, app servers off.
    *   *Warm Standby:* Scaled-down app running, ready to scale up.
    *   *Multi-Site Active-Active:* Full production in both regions.

CL8 — **GitOps Model**
*   **Concept:** Git repository is the single source of truth for infrastructure and app state. Controllers sync cluster state to Git state.
*   **ArgoCD vs Flux:** ArgoCD (Push/Pull, UI focused), Flux (Pull, K8s-native, CLI focused).
*   **Secrets:** Sealed Secrets (encrypted in Git), External Secrets Operator (fetches from Vault/AWS Secrets Manager), Vault Agent.
*   **Drift:** If manual change occurs, GitOps controller detects drift and reverts the cluster to match Git state automatically.

CL9 — **Stateful Legacy App to Kubernetes**
*   **StatefulSet:** Ensures stable network IDs and ordered deployment/termination.
*   **PV/PVC:** Use `local` storage class with `volumeBindingMode: WaitForFirstConsumer` for low latency.
*   **Headless Service:** Provides stable DNS names for StatefulSet pods.
*   **Init Containers:** Handle config file generation before main app starts.
*   **ConfigMaps:** Mount as volumes for specific paths.
*   **PDB:** Ensure minimum available pods during disruptions to prevent data loss.

CL10 — **Identity (SA, Workload, IRSA)**
*   **Service Account:** K8s identity for pods.
*   **Workload Identity:** Maps K8s SA to Cloud IAM Role (Azure/GCP).
*   **IRSA:** Maps K8s SA to AWS IAM Role via OIDC.
*   **Mounting Creds:** Bad practice (static keys in pod, risk of leakage, rotation difficulty).
*   **Federation:** Uses short-lived OIDC tokens. No long-term secrets stored in pod.

CL11 — **Storage Types**
*   **Object (S3/Blob):** Unstructured data, HTTP API. High durability (11 9s). *Use:* Backups, media, logs.
*   **Block (EBS/Managed Disk):** Raw storage for VMs. Low latency. *Use:* OS disks, DBs.
*   **File (EFS/Azure Files):** Shared file system (NFS/SMB). *Use:* Shared configs, legacy apps.
*   **Guarantees:** Object storage focuses on durability; Block/File focus on availability and performance.

CL12 — **Zero-Trust Network Architecture (K8s)**
*   **Network Policies:** Restrict pod-to-pod traffic (default deny).
*   **Service Mesh (mTLS):** Encrypts service-to-service traffic (Istio/Linkerd).
*   **Pod Identity:** Workload identity for cloud access.
*   **Admission Controllers (OPA/Kyverno):** Enforce policies on resource creation.
*   **Image Signing:** Verify images before deployment (Cosign).
*   **Runtime Security (Falco):** Detects anomalous behavior (e.g., shell in container).
*   **Diff:** Traditional = Perimeter trust. Zero Trust = Verify every request, assume breach.

CL13 — **CDN (Content Delivery Network)**
*   **Function:** Distributed edge servers cache content closer to users.
*   **Services:** AWS CloudFront, Azure CDN.
*   **Use CDN:** Static assets (images, JS), high latency regions, DDoS protection.
*   **Use Origin:** Dynamic content, real-time data, sensitive data not meant for caching.

CL14 — **Availability Zones (AZ) and Regions**
*   **Region:** Geographic area (e.g., us-east-1).
*   **AZ:** Isolated data centers within a Region (power, network independent).
*   **Multi-AZ:** Protects against data center failure.
*   **Zonal Resources:** EBS (tied to AZ), EC2 (tied to AZ).
*   **Regional Resources:** ELB, RDS (Multi-AZ), S3 (Region-wide).

CL15 — **Managed Database Service**
*   **Definition:** DBaaS (e.g., RDS, Cloud SQL).
*   **Provider Handles:** Hardware provisioning, OS patching, backups, replication, failover, scaling.
*   **Customer Handles:** Schema design, queries, user access, data encryption at rest.

CL16 — **Cloud Identity Services**
*   **Users:** Individual identities.
*   **Groups:** Collections of users for permission assignment.
*   **Roles:** Temporary credentials assumed by users/services.
*   **Policies:** JSON rules defining permissions.
*   **Least Privilege:** Grant only necessary permissions. Apply via Groups/Roles, not individual users.

CL17 — **Serverless Computing**
*   **Definition:** Event-driven execution without managing servers.
*   **Comparison:** Lambda (AWS), Functions (Azure), Cloud Functions (GCP).
*   **Cold Starts:** Latency when initializing a new instance. Matters for latency-sensitive apps.
*   **Poor Fit:** Long-running processes, stateful apps, predictable high-throughput (reserved capacity cheaper).

CL18 — **Kubernetes Ingress Options**
*   **Ingress Resource:** Standard K8s resource for HTTP routing.
*   **Gateway API:** Extensible, standardized API for L7/L4 traffic.
*   **Cloud LB Annotations:** Cloud provider specific LB creation.
*   **Mesh Ingress:** Sidecar proxy handles traffic (e.g., Istio).
*   **TLS:** LB (Edge termination), Ingress (K8s termination), Mesh (mTLS between services).
*   **Gateway API:** Use for multi-cluster, complex routing, or when Ingress is too limited.

CL19 — **Cost Optimization Strategy**
*   **RI/SP:** Commit to usage for discount (1-3 years).
*   **Spot/Preemptible:** Unused capacity, 90% off, interruptible.
*   **Right-sizing:** Match instance size to actual utilization.
*   **Storage Tiering:** Move old data to Glacier/Archive.
*   **Idle Detection:** Find unattached EBS, idle LBs.
*   **FinOps:** Culture of cost accountability.
*   **Showback/Chargeback:** Allocate costs to teams.
*   **Tools:** AWS Cost Explorer, Kubecost, CloudHealth.

CL20 — **Multi-Cloud Strategy**
*   **Abstraction:** Kubernetes (runtime), Terraform (IaC).
*   **Networking:** Direct Connect/ExpressRoute or Interconnect.
*   **Identity:** Federation (OIDC) across clouds.
*   **Trade-offs:** Complexity, latency, operational overhead vs. Resilience, negotiation leverage.
*   **Justified:** Regulatory requirements, vendor lock-in risk, disaster recovery diversity.
*   **Unnecessary:** If single cloud meets SLAs, multi-cloud adds cost without value.

OT1 — **Purdue Model (ISA-95)**
*   **Level 0:** Physical Process (Sensors, Actuators).
*   **Level 1:** Basic Control (PLCs, RTUs).
*   **Level 2:** Zone Control (SCADA, HMIs).
*   **Level 3:** Operations (MES, Historian).
*   **Level 4:** Enterprise (ERP, Business).
*   **Level 5:** Business Planning (Corporate).

OT2 — **IT vs OT Networks**
*   **IT:** Data confidentiality/integrity, frequent patching, high availability.
*   **OT:** Safety, availability, real-time control, legacy systems.
*   **Patching:** OT patching requires downtime testing; IT patching is routine. OT systems often run unsupported OS.

OT3 — **DMZ (IDMZ)**
*   **Role:** Buffer zone between IT (L4) and OT (L0-L2).
*   **Traffic:** Historian sync (OT->IT), Production data (OT->IT). Commands (IT->OT) restricted/filtered.
*   **Danger:** Direct L4->L1 allows IT malware to reach control logic, risking physical safety.

OT4 — **OT Protocols**
1.  **Modbus TCP:** L1/L2 (Control), Master/Slave.
2.  **OPC UA:** L3/L4 (Data exchange), Secure, Platform independent.
3.  **EtherNet/IP (CIP):** L1/L2 (Control), Allen-Bradley dominant.
4.  **PROFINET:** L1/L2 (Control), Real-time industrial Ethernet.
5.  **MQTT:** L3/L4 (IoT/Cloud), Lightweight pub/sub.

OT5 — **Data Diode vs Firewall**
*   **Data Diode:** Hardware enforcing one-way data flow.
*   **Firewall:** Two-way traffic filtering.
*   **Use Diode:** When data must leave OT but nothing can enter (e.g., high security).
*   **Limitation:** No feedback loop, no remote configuration from IT side.

OT6 — **Data Flow to Cloud**
*   **Flow:** PLC -> Historian -> Edge Gateway -> MQTT Broker -> Cloud Ingestion.
*   **Security Boundaries:** DMZ at Edge Gateway. Encrypt traffic (TLS). Isolate OT network from Internet.

OT7 — **IEC 62443 Zones and Conduits**
*   **Zone:** Logical grouping of assets with similar security requirements.
*   **Conduit:** Controlled communication path between zones.
*   **Model:**
    *   Zone 4: Corporate Network.
    *   Zone 3: Engineering/SCADA Servers.
    *   Zone 2: Production Network (PLCs).
    *   Zone 1: Safety (SIS).
    *   Conduits: Firewalls between zones, strictly filtered.

OT8 — **Network Segmentation Strategy**
*   **VLANs:** Separate by production line and function (Control, HMI, Engineering).
*   **Firewalls:** Between VLANs and DMZ.
*   **Switches:** Managed Layer 3 switches for routing.
*   **Vendor Access:** Jump host with MFA and session recording.
*   **Transition:** Phased rollout, parallel testing, maintenance windows.

OT9 — **SIS vs BPCS**
*   **SIS (Safety Instrumented System):** Independent system to bring process to safe state.
*   **BPCS (Basic Process Control System):** Normal operation control.
*   **IEC 61511:** Standard for Safety Instrumented Systems.
*   **Isolation:** SIS must be air-gapped to prevent cyber attacks disabling safety functions.
*   **Ransomware on SIS:** Safety systems fail, potential for catastrophic physical damage or injury.

OT10 — **PLC, DCS, SCADA**
*   **PLC:** Logic controller for specific machines.
*   **DCS:** Distributed control for continuous processes (refineries).
*   **SCADA:** Supervisory monitoring and control over wide area.
*   **Coexist:** Yes, PLCs feed SCADA, DCS manages process.

OT11 — **Remote Access Architecture**
*   **Jump Host:** Single entry point.
*   **MFA:** Required for all access.
*   **Recording:** Session logging for audit.
*   **Time-Limited:** Access expires after task.
*   **No Direct RDP:** Prevents lateral movement from corporate network to OT devices.

OT12 — **Nation-State Incident Response**
*   **Monitoring:** IDS/Anomaly detection on OT traffic.
*   **Inventory:** Identify compromised assets.
*   **Steps:** Isolate segment, notify ops, preserve evidence.
*   **Coordination:** Ops prioritize safety over connectivity.
*   **Operator Role:** Manual override if automation is compromised.

OT13 — **HMI (Human-Machine Interface)**
*   **Role:** Operator interface for monitoring and controlling process.
*   **Difference:** Engineering Workstation is for programming/configuring logic; HMI is for runtime operation.

OT14 — **Historian**
*   **Role:** Time-series database for process data.
*   **Plant Historian (L3):** Real-time process data, short-term retention.
*   **Enterprise Historian (L4):** Long-term analytics, business reporting.

OT15 — **Ladder Logic**
*   **Definition:** Graphical programming language resembling relay logic.
*   **Device:** PLC.
*   **Dominance:** Deterministic execution, safety certification, legacy familiarity.

OT16 — **RTU (Remote Terminal Unit)**
*   **Definition:** Remote controller with comms focus.
*   **Difference:** PLC focuses on local control; RTU focuses on telemetry over long distances.
*   **Environment:** Pipelines, SCADA, remote sites.

OT17 — **Analog vs Digital Signals**
*   **Analog:** Continuous voltage/current (4-20mA).
*   **Digital:** Discrete states (On/Off).
*   **4-20mA:** "Live zero" (4mA) indicates wire break (0mA) vs valid low signal.

OT18 — **OT Asset Inventory**
*   **Foundation:** Cannot secure what you don't know.
*   **Tools:** Passive (Network TAPs), Active (Scanners).
*   **Active Risk:** Scanning can crash legacy devices or disrupt control loops.

OT19 — **MITRE ATT&CK for ICS**
*   **Tactics:** Reconnaissance, Initial Access, Execution, Persistence, Impact.
*   **Technique Examples:** Phishing (Initial Access), Modbus Write (Execution), Data Destruction (Impact).
*   **Difference:** Focuses on physical impact and safety, not just data theft.
*   **Usage:** Map defenses to tactics to find gaps in OT monitoring.

OT20 — **Converged IT/OT SOC**
*   **Log Collection:** Syslog/OT-specific agents (non-intrusive).
*   **SIEM:** Centralized correlation.
*   **Rules:** OT-specific (e.g., PLC write attempts).
*   **Staff:** IT analysts + OT engineers.
*   **False Positives:** Tuned to avoid production disruption.
*   **Firmware Alert:** Verify intent, isolate PLC, validate with engineering, patch during maintenance.
