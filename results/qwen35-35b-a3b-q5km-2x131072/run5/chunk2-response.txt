K1 — Easy: A **Deployment** manages stateless applications, allowing for rolling updates and rollbacks without stable pod identities. A **StatefulSet** manages stateful applications, providing stable network identities (e.g., pod-0, pod-1), ordered deployment/termination, and stable persistent storage.

K2 — Easy: A **ClusterIP** Service creates a virtual IP within the cluster. **kube-proxy** (using iptables or IPVS) updates the node's networking rules to route traffic sent to the Service IP to the backend Pod IPs based on the Service's selector.

K3 — Medium: A **PersistentVolume (PV)** is a cluster-wide storage resource. A **PersistentVolumeClaim (PVC)** is a user request for storage. A **StorageClass** defines the provisioning mechanism (dynamic). A PVC binds to a PV; if no PV exists, the StorageClass triggers dynamic provisioning.

K4 — Medium: 1. Check container logs (`kubectl logs`) for application errors. 2. Check pod events (`kubectl describe`) for OOMKilled or probe failures. 3. Verify resource limits vs. actual usage. 4. Check liveness/readiness probe configurations for false negatives.

K5 — Medium: **CoreDNS** provides DNS resolution for services and pods within the cluster. If CoreDNS pods are down, internal service discovery (e.g., `service-name.namespace.svc`) fails, though external DNS resolution may still work if the cluster uses external nameservers.

K6 — Hard: The scheduler uses **Filtering** (remove nodes that don't meet requirements like taints/resources) and **Scoring** (rank remaining nodes). Factors: 1. Resource requests (CPU/Mem), 2. Node Affinity/Pod Affinity, 3. Topology Spread Constraints.

K7 — Hard: Yes, it will be scheduled (256Mi request < 500Mi available). If it uses 600Mi at runtime, it exceeds the 512Mi limit and will be **OOMKilled** by the kernel, triggering a restart.

K8 — Hard: 1. Client hits External LB. 2. LB forwards to Ingress Controller (NGINX). 3. NGINX matches Ingress rules to Service. 4. Service routes to Endpoints (Pod IPs). 5. **kube-proxy** (iptables/IPVS) on the node directs traffic to the specific Pod.

K9 — Medium: A **NetworkPolicy** restricts traffic flow between pods. Default behavior (no policy): All traffic is allowed. When a policy is applied: Traffic is denied by default unless explicitly allowed by the policy's `ingress` or `egress` rules.

K10 — Medium: **RollingUpdate** replaces pods gradually (zero downtime). **Recreate** kills all existing pods before starting new ones. Choose **Recreate** for stateful applications that cannot handle partial downtime or require full configuration reloads.

K11 — Easy: A **Namespace** is a virtual cluster partition for resource isolation. Default namespaces: `default`, `kube-system`, `kube-public`, `kube-node-lease`.

K12 — Easy: `kubectl describe` shows detailed status and events. Information not in `get`: 1. **Events** (warnings/errors), 2. **Conditions** (Ready status details), 3. **Environment Variables** and **Mounts**.

K13 — Easy: A **ConfigMap** stores non-confidential configuration data. Pods consume it via: 1. **Environment Variables**, 2. **Volume Mounts** (files).

K14 — Easy: **ClusterIP** (internal only), **NodePort** (exposes on node IP:port), **LoadBalancer** (provisions external cloud load balancer).

K15 — Easy: A **DaemonSet** ensures a copy of a pod runs on all (or specific) nodes. Use for logging agents, monitoring, or storage daemons that need to run on every node.

K16 — Medium: **Taints** repel pods; **Tolerations** allow pods to schedule on tainted nodes. **Node Affinity** attracts pods to nodes. Use case: Taint for dedicated GPU nodes; Affinity for co-locating related services.

K17 — Medium: An **Operator** is a custom controller that manages complex stateful applications using CRDs. It differs from standard controllers by automating operational tasks (backup, scaling, upgrades). It solves the complexity of managing stateful apps at scale.

K18 — Hard: etcd uses **linearizability** (strong consistency). If quorum is lost, the cluster becomes read-only/unavailable. Recovery: Restore from backup on a new etcd cluster or replace lost nodes and restore data from snapshot.

K19 — Hard: **Role/ClusterRole** define permissions; **RoleBinding/ClusterRoleBinding** assign them to users/groups.
```yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata: namespace: my-ns
rules:
- apiGroups: [""]
  resources: ["pods", "pods/log"]
  verbs: ["get", "list", "watch"]
---
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata: namespace: my-ns
subjects:
- kind: User
  name: developer
roleRef:
  kind: Role
  name: pod-reader
```

K20 — Hard: `ndots` defines how many dots trigger an FQDN search. High `ndots` causes queries for every suffix (e.g., `google.com` -> `google.com.svc.cluster.local` -> ...). Fix: Set `dnsConfig: { options: [{ name: "ndots", value: "1" }] }` to reduce queries.

D1 — Easy: **Rebase** rewrites history to create a linear commit chain. **Merge** preserves history with a merge commit. Avoid rebase on shared/public branches to prevent history conflicts.

D2 — Easy: **TCP** is connection-oriented and reliable (HTTP, SSH). **UDP** is connectionless and faster (DNS, VoIP).

D3 — Medium: A **race condition** occurs when output depends on the sequence of execution. Example: Two threads incrementing a shared counter without a lock, resulting in lost updates.

D4 — Medium: **Symmetric** uses one key; **Asymmetric** uses a key pair. TLS uses Asymmetric for the handshake (key exchange/auth) and Symmetric for bulk data encryption (performance).

D5 — Medium: A **layer** is a read-only filesystem diff in a Docker image. Caching: Place frequently changing instructions (e.g., `COPY .`) after stable ones (e.g., `COPY package.json`) to maximize cache hits.

D6 — Medium: 1. Check upstream service health. 2. Review application logs for exceptions. 3. Check resource limits (CPU/Mem throttling). 4. Verify load balancer health checks.

D7 — Medium: **CAP** states a distributed system can only guarantee 2 of 3: Consistency, Availability, Partition Tolerance. **CP**: Zookeeper (consistency over availability during partition). **AP**: DynamoDB (availability over consistency during partition).

D8 — Medium: **Horizontal** adds nodes; **Vertical** adds resources. Patterns enabling horizontal scaling: Stateless services, sharding, load balancing, microservices.

D9 — Easy: An **environment variable** is a dynamic value. Persist across sessions on Linux by exporting in `~/.bashrc` or `~/.profile`.

D10 — Easy: **JSON** is strict key-value (machine-friendly). **YAML** is human-readable (indentation-based). Advantage JSON: Standardized parsing. Advantage YAML: Readable config files.

D11 — Easy: **REST** is an architectural style. **GET** (retrieve), **POST** (create), **PUT** (replace), **DELETE** (remove).

D12 — Easy: **Version pinning** locks dependencies to specific versions. Important for **reproducible builds** to prevent breaking changes from upstream updates.

D13 — Easy: **Compiled** (source to binary, e.g., C, Go). **Interpreted** (source to execution, e.g., Python, JavaScript).

D14 — Medium: **Reverse proxy** (client -> proxy -> server). **Forward proxy** (client -> proxy -> internet). Servers: **Nginx** (web serving), **HAProxy** (load balancing).

D15 — Medium: A **deadlock** is a permanent block. **Coffman Conditions**: 1. Mutual Exclusion, 2. Hold and Wait, 3. No Preemption, 4. Circular Wait.

D16 — Medium: **Eventual consistency** ensures updates propagate over time. **Cassandra** uses **Tunable Consistency** (e.g., ONE, QUORUM, ALL) to balance latency and consistency.

D17 — Medium: **mTLS** authenticates both client and server via certificates. Handshake: Client Hello, Server Hello, Cert exchange, Verify. Used in **Service Meshes** (e.g., Istio) for secure service-to-service communication.

D18 — Medium: **Hash Table** maps keys to indices. **Collision**: Chaining (linked lists) or Open Addressing (probing). Complexity: Avg O(1), Worst O(n) (all collisions).

D19 — Medium: **Circuit Breaker** prevents cascade failures. **Closed** (normal), **Open** (fail fast), **Half-open** (test). Relates to **Retry**: Do not retry if circuit is open; use **Exponential Backoff** on retries.

D20 — Medium: **Optimistic** (check version on commit, e.g., `SELECT ... FOR UPDATE` vs `version` column). **Pessimistic** (lock on read). Choose Optimistic for low conflict/high concurrency; Pessimistic for high conflict/critical data.
