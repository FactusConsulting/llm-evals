SC1-A
The failure stems from the Kubernetes Pod Security Standards (PSS) upgrade from **Baseline** to **Restricted**.
1.  **Baseline:** Allows `runAsNonRoot: true` without `runAsUser`. It only checks that the container *can* run as non-root, but doesn't enforce it if the image defaults to root.
2.  **Restricted:** Enforces `runAsNonRoot: true` implies the container *must* run as a non-root user. If the image defaults to `USER root` (as `cgr.dev/chainguard/static:latest` does) and no `runAsUser` is explicitly set in the `securityContext`, the pod violates the Restricted policy.
3.  **Interaction:** Kubernetes validates the Pod spec against the namespace's PSS level at admission. The Restricted policy forbids running as root. Since the image defaults to root and the manifest lacks `runAsUser`, the kubelet rejects the pod or the pod starts and immediately crashes if the security context is enforced at runtime (e.g., via seccomp or specific node policies), but primarily this is an admission failure or a runtime crash if the policy is enforced via a mutating webhook that failed to inject the UID. In this specific case, the pods are likely stuck in CrashLoopBackOff because the container process starts as root, violates the security policy enforced by the node's runtime (e.g., gVisor or specific CRI settings), or the admission controller is mutating the pod but the image itself is rejecting root execution.
4.  **Root Cause:** The image defaults to root, and the manifest relies on the old Baseline allowance. Restricted requires explicit `runAsUser` (non-zero).

SC1-C
**Remediation Strategy:**
1.  **Immediate Fix (Image):** Rebuild the base image `cgr.dev/chainguard/static:latest` to include a non-root user (e.g., `USER 1000`) or use a pre-built non-root variant.
2.  **Manifest Fix:** Update all affected Deployment/StatefulSet manifests to explicitly set `securityContext.runAsUser: 1000` (matching the image) and `securityContext.runAsNonRoot: true`.
3.  **CI Gate:** Add a policy check in CI (using `checkov`, `kube-score`, or `conftest`) that validates the Pod Security Standard compliance against the target namespace level before merging. Specifically, enforce that `runAsUser` is set if `runAsNonRoot` is true.
4.  **Rollout:** Use a **Canary Deployment**.
    *   Deploy the fixed version to 5% of pods.
    *   Monitor CrashLoopBackOff metrics and error rates.
    *   Gradually increase traffic to 100% if stable.
    *   This ensures no downtime and limits blast radius if the fix is incomplete.

SC2-A
**Stale Data & Replication:**
1.  **Cause:** Cross-region asynchronous replication introduces latency. When the replica is promoted, it only contains data up to the last committed WAL record shipped from the primary. Any writes committed to the primary but not yet shipped are lost.
2.  **WAL Shipping:** PostgreSQL streams WAL records. In async mode, the primary does not wait for the replica to acknowledge receipt.
3.  **Data Loss:** The "lag window" represents potential data loss. If the primary fails immediately after a write, that write is lost.
4.  **Quantification:** Check `pg_stat_replication` on the primary (before failure) for `write_lag`, `flush_lag`, and `replay_lag`. Post-failover, check the replica's `pg_last_wal_receive_lsn()` vs `pg_last_wal_replay_lsn()`. The difference in LSNs indicates the data loss window.

SC2-B
**Terraform Operations:**
1.  **Import Replica:** Import the promoted replica into state using its new ARN.

2.  **Update State:** Move the old instance resource to a "tainted" or separate state file to prevent recreation, or update the resource name in state.

SC2-C
**Architecture Design:**
1.  **RDS Proxy:** Replace PgBouncer with **RDS Proxy**. It handles connection pooling at the AWS layer, supports failover transparently, and reduces connection churn during failover.
2.  **Route53:** Use **Route53 Latency-Based Routing** with Health Checks. Point the application endpoint to a CNAME that resolves to the primary region. If the primary health check fails, Route53 routes to the secondary region.
3.  **Retry Logic:** Implement exponential backoff with jitter in the application connection logic to handle transient failover errors.
4.  **RPO < 1s:** Standard async RDS cannot guarantee <1s RPO. **Recommend Aurora Global Database**. It uses a single storage volume replicated across regions with sub-second replication latency (typically <1s) and synchronous commit options.
5.  **Failover:** Aurora Global Database supports automatic failover with RPO < 1s and RTO < 1 minute.

SC4-A
**Top 5 Blockers:**
1.  **Local Sessions (Sticky):** Kubernetes pods are ephemeral.
    *   *Solution:* Move sessions to Redis (ElastiCache) or a distributed session store.
2.  **Local NFS Mounts:** Pods cannot share local filesystems across nodes.
    *   *Solution:* Use **ReadWriteMany (RWX)** Persistent Volumes (EFS or NFS CSI) to share `/data/uploads`.
3.  **Oracle 19c (On-Prem):** Stateful DB outside K8s.
    *   *Solution:* Keep Oracle on-prem but use a **StatefulSet** for the app to maintain affinity, or migrate to RDS/Aurora if feasible.
4.  **IBM MQ:** Stateful messaging.
    *   *Solution:* Use **Kubernetes Operator** for MQ or keep MQ on-prem with a bridge (e.g., MQ Bridge for Kubernetes).
5.  **F5 Load Balancer:** External dependency.
    *   *Solution:* Replace F5 with **Kubernetes Ingress Controller** (NGINX or ALB Ingress) and use **Service Mesh** for internal traffic management.

SC6-C
**Network Upgrade Comparison:**
1.  **AWS Direct Connect:**
    *   *Pros:* Dedicated bandwidth, lower latency, consistent performance.
    *   *Cons:* High cost, physical provisioning.
    *   *Zero Trust:* Fits well (dedicated link), but requires BGP routing.
2.  **Second VPN Tunnel (ECMP):**
    *   *Pros:* Low cost, quick deployment.
    *   *Cons:* Shared bandwidth, higher latency, less reliable.
    *   *Zero Trust:* Encryption is IPsec, but routing is static.
3.  **Hybrid Mesh (Tailscale/Netbird):**
    *   *Pros:* Zero-trust native (mTLS), easy setup, no public IPs.
    *   *Cons:* Performance depends on relay servers (unless using DERP/Local).
    *   *Zero Trust:* Best fit for identity-based access.
**Recommendation:** Use **Direct Connect** for core data traffic (DB, Storage) to meet bandwidth needs. Use **Tailscale/Netbird** for management and developer access to enforce zero-trust identity. Update routing to use BGP for DX and Overlay routing for Mesh.

**Drift Sources & Risks:**

2.  **Diverged Helm Values:** Untracked changes.

SC10-C
**Automated Failover Orchestration:**
1.  **Detection:** **Prometheus** monitors site health. If Chicago nodes are unreachable for > 30s, trigger alert.
2.  **GSLB:** **Route53** or **Global DNS** switches traffic to Dallas.
3.  **Patroni:** Patroni DCS detects leader loss. Promotes Dallas node to Primary.
4.  **Redis/RabbitMQ:** Sentinel/MQ Federation automatically promotes standby.
5.  **MinIO:** Site Replication catches up.
6.  **OpenTofu:** **Terraform Cloud** or **Spacelift** updates state to reflect Dallas as primary.
7.  **Split-Brain:** Patroni uses **etcd** quorum. If Chicago is down, it cannot vote, preventing split-brain.
8.  **Failback:**
    *   Rebuild Chicago cluster.
    *   Re-establish replication (Dallas -> Chicago).
    *   Switch traffic back via DNS.
    *   Verify data consistency.
