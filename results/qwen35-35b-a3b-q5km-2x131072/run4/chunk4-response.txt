## Go (20 questions)

**G1 — Easy:** An array is a fixed-size value type `[N]T`. A slice is a dynamic view over an array, defined as a struct containing a pointer to the array, length, and capacity. When appending to a slice that has reached capacity, the runtime allocates a new, larger underlying array (typically double the size), copies the existing elements, and returns a new slice header pointing to it.

**G2:** `var x int` declares a variable with the zero value (`0`) and requires an explicit type. `x := 0` is a short variable declaration that infers the type. You cannot use `:=` to redeclare a variable in the same scope (e.g., inside a function), nor can you use it outside a function (e.g., at package level).

**G3:** A goroutine is a lightweight thread managed by the Go runtime, not the OS. Unlike OS threads (1:1 mapping), goroutines are multiplexed onto OS threads (M:N). The Go scheduler (GMP model) manages Goroutines (G), Machines (M, OS threads), and Processors (P, logical CPUs). It handles work stealing and preemption to maximize CPU utilization.

**G4:** Channels are typed conduits for communication between goroutines. Unbuffered channels block both sender and receiver until the other side is ready. Buffered channels allow sending without blocking up to their capacity. Sending to a full buffered channel blocks until a receiver reads from it.

**G5:** The `context` package manages deadlines, cancellation signals, and request-scoped values across API boundaries.
```go
ctx, cancel := context.WithTimeout(context.Background(), 5*time.Second)
defer cancel()
req, _ := http.NewRequestWithContext(ctx, "GET", url, nil)
client.Do(req) // Cancels if timeout exceeded
```

**G6:** Go interfaces are satisfied implicitly; a type implements an interface if it implements all its methods. Unlike Java/C#, Go does not require explicit `implements` declarations. The empty interface `any` (formerly `interface{}`) accepts any type. It is powerful for generic-like flexibility but dangerous because it requires type assertions to use values, losing compile-time safety.

**G7:** Go's memory model guarantees visibility only if operations are synchronized. The code is unsafe because `ready` and `data` are accessed concurrently without synchronization, causing a data race. The compiler may optimize `ready` to a constant. Fix by using `sync/atomic` for the boolean or a `sync.Mutex` to protect both variables.

**G8:** `sync.Mutex` provides exclusive access. `sync.RWMutex` allows multiple readers or one writer. `sync.Map` is optimized for cases where keys are added dynamically and read-heavy, avoiding lock contention on the map structure. However, `sync.Map` is slower for writes and iteration compared to a regular `map` with a mutex due to internal locking granularity.

**G9:** Go modules manage dependencies. `go.mod` declares the module path and required dependencies. `go.sum` stores cryptographic checksums to verify integrity. `go mod tidy` removes unused dependencies and adds missing ones. Minimum Version Selection (MVS) selects the lowest version of a dependency that satisfies all requirements in the dependency graph.

**G10:** `defer` schedules a function call to run immediately before the surrounding function returns. Multiple defers execute in Last-In-First-Out (LIFO) order. A common gotcha in loops is capturing the loop variable in a closure; the deferred function captures the variable by reference, so all defers might see the final loop value.

**G11:** A pointer holds the memory address of a value. The zero value of a pointer is `nil`. To create a pointer to a struct literal, use the address-of operator: `&MyStruct{Field: "value"}`.

**G12:** The `init()` function is executed automatically before `main()` runs. It runs once per package. A package can have multiple `init()` functions; they execute in the order they appear in the source files, and across files, in the order the compiler processes them (typically alphabetical by filename).

**G13:** `new(T)` allocates zeroed memory for type `T` and returns a pointer `*T`. `make(T, args)` initializes built-in reference types (slice, map, channel) and returns the initialized value (not a pointer). `new` works with any type; `make` only works with slices, maps, and channels.

**G14:** Errors are values implementing the `error` interface (`Error() string`). The idiomatic pattern is `if err != nil { return err }`. `errors.Is` checks if an error matches a target (using `Unwrap`). `errors.As` checks if an error matches a specific type (using `As`). `==` only works for exact error values, not wrapped errors.

**G15:** Built-in composite types are `map`, `slice`, `channel`, and `struct`. Their zero values are: `nil` (for map, slice, channel), and zero values for all struct fields (e.g., `0`, `""`, `false`).

**G16:** A race condition occurs when two goroutines access shared data concurrently, and at least one writes. The `-race` flag instruments code to detect these at runtime.
*Fix:* Wrap shared access with `sync.Mutex`.
```go
var mu sync.Mutex
mu.Lock()
data = "hello"
mu.Unlock()
```

**G17:** `select` waits on multiple channel operations. Unlike `switch`, it chooses a case non-deterministically if multiple are ready. To implement a timeout:
```go
select {
case res := <-ch:
    return res
case <-time.After(1 * time.Second):
    return errors.New("timeout")
}
```

**G18:** Escape analysis determines if a variable lives on the stack or heap. If a variable's address escapes the function scope, it is allocated on the heap. Inspect with `go build -gcflags="-m"`. It matters because heap allocations trigger garbage collection, impacting latency and throughput.

**G19:** `reflect` allows inspecting and modifying types/values at runtime. `reflect.Type` describes the type; `reflect.Value` holds the value. Reflection is slow due to dynamic dispatch and type checking. It is justified for frameworks (e.g., serialization, ORM) but avoided in performance-critical paths in favor of code generation (e.g., `go:generate`).

**G20:** `//go:embed` embeds files/directories at compile time into a `fs.FS`.
```go
//go:embed images/*.png
var images embed.FS
```
Limitations: Files must exist at compile time; cannot embed symlinks or non-regular files. It interacts with `io/fs` by implementing the `FS` interface, allowing standard file system operations on embedded content.

## Rust (20 questions)

**R1:** Ownership is Rust's core memory safety mechanism. Rules: 1. Each value has exactly one owner. 2. When the owner goes out of scope, the value is dropped. 3. You can have either one mutable reference or any number of immutable references, but not both simultaneously.

**R2:** `String` is an owned, growable heap-allocated string. `&str` is a borrowed, immutable slice of bytes (often a string literal). Use `String` when you need to own or modify data. Use `&str` for function arguments to avoid unnecessary cloning and for read-only access.

**R3:** Borrowing allows temporary access without ownership transfer. Rules: 1. At any time, you can have either one mutable reference (`&mut T`) OR any number of immutable references (`&T`). 2. References must always be valid. 3. You cannot mutate through an immutable reference.

**R4:** `Result<T, E>` represents success (`Ok(T)`) or failure (`Err(E)`). `Option<T>` represents a value that may be `Some(T)` or `None`. The `?` operator propagates errors.
```rust
fn read_file() -> Result<String, std::io::Error> {
    std::fs::read_to_string("file.txt")
}
```

**R5:** `Box<T>` owns data on the heap with a single owner. `Rc<T>` (Reference Counted) allows multiple owners on the same thread. `Arc<T>` (Atomic Reference Counted) is thread-safe `Rc`. Use `Box` for heap allocation, `Rc` for shared ownership within a thread, `Arc` for shared ownership across threads.

**R6:** Traits define shared behavior. Unlike Go interfaces, traits can have default implementations. `dyn Trait` creates a trait object for dynamic dispatch (vtable). `T: Trait` uses generics for static dispatch (monomorphization), which is generally faster.

**R7:** The function fails because the compiler cannot determine which input's lifetime the returned reference belongs to. Fix by adding a lifetime parameter:
```rust
fn longest<'a>(x: &'a str, y: &'a str) -> &'a str {
    if x.len() > y.len() { x } else { y }
}
```

**R8:** `Send` means a type can be transferred across thread boundaries. `Sync` means a type can be safely referenced from multiple threads. `Rc<T>` is not `Send` because its reference count is not atomic. `Arc<T>` uses atomic reference counting, making it `Send` and `Sync`.

**R9:** `async/await` creates a `Future` (a state machine). The executor (e.g., Tokio) polls the future until it completes. Unlike Go's preemptive goroutines, Rust's async is cooperative; the future must yield control explicitly. This avoids stack overflow but requires manual scheduling.

**R10:** `unsafe` allows bypassing compiler checks. Actions: 1. Dereference raw pointers. 2. Call unsafe functions. 3. Access mutable statics. Justified when interfacing with C (FFI), implementing low-level primitives, or optimizing performance where safety checks are redundant.

**R11:** Pattern matching (`match`) exhaustively checks all possible values of an enum. `if let` matches a single pattern and ignores others. The compiler enforces exhaustiveness in `match`, requiring a `_` catch-all case if not all variants are covered.

**R12:** `[T; N]` is a fixed-size array on the stack. `Vec<T>` is a growable heap-allocated vector. When `Vec` capacity is exceeded, it allocates a new, larger buffer (usually double), copies elements, and drops the old buffer.

**R13:** Enums in Rust are algebraic data types (ADTs) that can hold data. Unlike C/Java enums, they can store variants with different types. `Option<T>` is `enum Option<T> { Some(T), None }`. Rust has no `null`; `None` replaces it, forcing explicit handling of missing values.

**R14:** `impl` defines methods for a type. `&self` borrows the instance. `&mut self` mutably borrows. `self` takes ownership. Associated functions (e.g., `String::new()`) do not take `self` and act like static constructors.

**R15:** `derive` macros automatically implement traits. Common: `Debug`, `Clone`, `PartialEq`. You cannot derive traits that require custom logic (e.g., `Drop`) or if the type contains fields that don't support the trait (e.g., `Clone` on a type with `!Clone` fields).

**R16:** `Cow<'a, T>` (Clone on Write) holds either a borrowed `&'a T` or an owned `Box<T>`. It avoids allocation if no mutation occurs. If mutation is requested, it clones the data to owned memory. Useful for functions that accept `&str` but might need to return an owned `String`.

**R17:** `mod` declares a module. `use` imports items. `pub` makes items visible outside the crate. File system maps to modules: `mod.rs` or `mod name.rs`. Rust 2018 improved module paths (`crate::` vs `super::`) and macro hygiene compared to 2015.

**R18:** `Pin<T>` prevents a value from being moved in memory. Needed for self-referential structs (e.g., async futures holding references to their own stack frames). `Unpin` is a marker trait indicating a type can be safely moved. Most types are `Unpin`, but `Future` implementations often are not.

**R19:** `macro_rules!` are declarative macros (pattern matching on tokens). Procedural macros (derive, attribute, function-like) operate on the AST. Hygiene ensures macro variables don't conflict with local variables unless explicitly passed.

**R20:** Zero-cost abstractions mean high-level features compile to efficient machine code. Iterators use inlining and loop unrolling. Generics use monomorphization (code duplication per type) instead of virtual dispatch. Traits use static dispatch by default, avoiding vtable overhead compared to C++ virtual functions.
