SC1-A
The pods are failing because the **Restricted** Pod Security Standard (PSS) is stricter than **Baseline**.
1.  **Restricted Policy:** Requires `securityContext.runAsNonRoot: true` AND explicitly sets `runAsUser` to a non-zero value (usually > 1000). It also forbids `privileged: true`, `hostNetwork`, and `hostPID`.
2.  **Baseline Policy:** Only forbids privileged containers and host namespaces. It allows running as root if `runAsNonRoot` is not explicitly set to true, or if `runAsUser` is omitted (defaulting to root).
3.  **The Failure:** The new base image (`cgr.dev/chainguard/static:latest`) defaults `USER` to root (UID 0). The pod spec sets `runAsNonRoot: true` but omits `runAsUser`. Kubernetes attempts to run the container as root (UID 0) because no specific user is defined. The Restricted PSS admission controller rejects this because running as root violates the "non-root" requirement enforced by Restricted.
4.  **Interaction:** The admission controller validates the Pod spec against the namespace's PSS level. Since the namespace is Restricted, the combination of `runAsNonRoot: true` + `runAsUser: 0` (implicit) is a violation.

SC1-C
**Remediation Strategy:**
1.  **Immediate Fix (Image):** Rebuild the container image with a non-root user. Add `USER 1000` to the Dockerfile.
2.  **Manifest Fix:** Update the Deployment manifests to explicitly set `securityContext.runAsUser: 1000` (matching the image) and ensure `runAsNonRoot: true` remains.
3.  **CI Gate:** Add a policy check in CI (using `kube-score` or `conftest`) to validate that all images used have a non-root user defined or that the Pod spec enforces `runAsUser` > 0. Integrate `kubectl apply --dry-run=client` with PSS validation.
4.  **Rollout:** Use a **Canary Deployment**. Update 10% of pods first. Monitor CrashLoopBackOff metrics. If stable, roll out to 50%, then 100%. This prevents a 100% outage if the fix is incomplete.

SC2-A
**Stale Data & Replication Lag:**
1.  **Cause:** Cross-region asynchronous replication introduces latency. When the replica is promoted, it only contains data up to the last committed WAL record shipped from the primary. Any transactions committed on the primary but not yet shipped are lost.
2.  **WAL Shipping:** PostgreSQL streams WAL records. In async mode, the primary does not wait for the replica to acknowledge receipt.
3.  **Data Loss:** The "data loss window" is the time between the last successful WAL send on the primary and the promotion.
4.  **Quantification:** Check `pg_stat_replication` on the primary (before failure) for `sent_lsn` vs `write_lsn`. After promotion, check `pg_last_wal_receive_lsn()` on the new primary to see the gap.

SC2-B

2.  **Update State:** `terraform state mv aws_db_instance.old_primary aws_db_instance.promoted_replica` (if renaming resource).
3.  **Detach Old:** `terraform state rm aws_db_instance.old_primary` (to remove the old instance from state without destroying it immediately, allowing manual deletion later).
4.  **Update References:** Update Terraform variables to point to the new ARN. Run `terraform plan` to ensure dependent resources (SGs, Route53) align.
5.  **Apply:** `terraform apply` to update Route53 and SGs to point to the new instance.

SC2-C
**Architecture Design:**
1.  **RDS Proxy:** Use RDS Proxy instead of PgBouncer. It handles connection pooling at the AWS layer, reducing connection churn during failover and supporting IAM auth.
2.  **Route53:** Use **Latency-Based Routing** with Health Checks. Point the primary endpoint to us-east-1. If health check fails, Route53 routes to eu-west-1.
3.  **Retry Logic:** Implement exponential backoff with jitter in the application connection pool.
4.  **RPO < 1s:** Standard RDS async replication cannot guarantee <1s RPO. **Recommend Aurora Global Database.** It uses a dedicated replication network with lower latency and synchronous replication options for the primary region, ensuring near-zero data loss.
5.  **Failover:** Aurora Global Database supports automatic failover with RPO ~0 and RTO < 1 minute.

SC4-A
**Top 5 Blockers:**
1.  **Local Sessions (Sticky):** K8s is stateless. **Solution:** Move sessions to Redis or use K8s Session Affinity (less ideal for scaling).
2.  **Local NFS Mount:** `/data/uploads` is local. **Solution:** Use a PersistentVolume (PV) with ReadWriteMany access (NFS CSI or EFS).
3.  **On-Prem Oracle:** K8s is likely cloud-native. **Solution:** Migrate to RDS/Aurora or use a managed Oracle service.
4.  **IBM MQ:** On-prem mainframe integration. **Solution:** Use Cloud MQ or establish a secure tunnel (Direct Connect) to the mainframe.
5.  **Java Monolith Scaling:** Large heap + sticky sessions. **Solution:** Tune JVM for containers (CGroup awareness), increase memory limits, and ensure horizontal pod autoscaling (HPA) is configured.

SC6-C
**Connectivity Comparison:**
1.  **AWS Direct Connect:**
    *   **Pros:** Dedicated bandwidth, lower latency, consistent performance.
    *   **Cons:** High cost, hardware provisioning.
    *   **Zero Trust:** Fits well (private link).
2.  **VPN ECMP:**
    *   **Pros:** Lower cost, faster setup.
    *   **Cons:** Shared bandwidth, higher latency variance.
    *   **Zero Trust:** Fits (IPsec encryption).
3.  **Mesh (Tailscale/Netbird):**
    *   **Pros:** Zero config, identity-based access.
    *   **Cons:** Not suitable for high-throughput data center traffic.
    *   **Zero Trust:** Excellent for management, poor for bulk data.
**Recommendation:** Upgrade to **Direct Connect** for data plane (bandwidth). Use **Tailscale** for management plane (zero trust identity).
**Routing:** DX requires BGP. VPN uses static routes. Mesh uses overlay routing.

SC10-A
**Multi-Cluster Architecture:**

3.  **Patroni:** Dallas node promotes to Primary (Synchronous commit ensures no data loss).
