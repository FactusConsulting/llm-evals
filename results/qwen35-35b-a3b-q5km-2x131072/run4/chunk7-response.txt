AA1 — **Monolith vs. Microservices:** A monolith is a single, unified codebase deployed as one unit. Microservices decompose an application into small, independent services communicating over a network.
**Microservices Problems:**
1.  **Distributed Complexity:** Debugging requires tracing across multiple services.
2.  **Data Consistency:** Maintaining ACID transactions across services is difficult (requires Sagas).
3.  **Network Latency & Reliability:** Communication is slower and prone to failure compared to in-process calls.

AA2 — **Sync vs. Async:** Synchronous communication blocks the sender until a response is received (request-response). Asynchronous communication allows the sender to continue processing without waiting for an immediate response (fire-and-forget).
**Examples:** Synchronous: HTTP/REST or gRPC. Asynchronous: RabbitMQ (Message Queue) or Kafka (Event Stream).

AA3 — **API Gateway vs. Reverse Proxy:** A reverse proxy routes traffic to backend servers (L4/L7). An API Gateway adds business logic like authentication, rate limiting, and API versioning.
**Replacement:** An API Gateway *can* replace a reverse proxy functionally, but a dedicated reverse proxy (like Nginx) is often kept in front of the Gateway for SSL termination and DDoS protection to offload heavy traffic handling from the Gateway.

AA4 — **Saga Pattern:** A sequence of local transactions where each updates data within a service. If one fails, compensating transactions undo previous changes.
**Choreography:** Services emit events; others react. (Decentralized, harder to debug). **Orchestration:** A central coordinator tells services what to do. (Centralized, easier to manage complex flows).
**Appropriate:** Choreography for simple, loosely coupled flows. Orchestration for complex workflows requiring compensation logic visibility.

AA5 — **CQRS:** Separates read (Query) and write (Command) operations into different models.
**Pairing with Event Sourcing:** Event Sourcing stores state changes as a sequence of events. CQRS uses these events to build optimized read models.
**Problem Solved:** Allows independent scaling of read/write paths and provides a complete audit trail of state changes, which traditional CRUD (overwriting state) does not.

AA6 — **Circuit Breaker:** Prevents repeated calls to a failing service.
**States:** Closed (normal), Open (failures detected, requests blocked), Half-Open (testing if service recovered).
**Difference from Retry:** Retry attempts to fix transient failures; Circuit Breaker stops hammering a persistent failure.
**Use Together:** Use Retry for transient network glitches, but wrap the retry logic inside a Circuit Breaker to prevent cascading failures if the service is truly down.

AA7 — **Tenant Isolation Models:**
1.  **Shared DB/Schema:** Lowest cost, highest operational complexity (schema changes affect all), high noisy-neighbor risk.
2.  **Shared DB/Schema:** Moderate cost, better security (schema isolation), moderate complexity (schema management).
3.  **Separate DB:** Highest cost, lowest complexity (no cross-tenant queries), lowest noisy-neighbor risk, best security.
**Trade-off:** Choose based on compliance needs and budget. High security/compliance favors Model 3.

AA8 — **Eventual vs. Strong:** Eventual consistency guarantees data will match eventually; Strong consistency guarantees immediate visibility of updates.
**Inventory Architecture:** Use **Strong Consistency**.
**Data Store:** Relational (PostgreSQL) with row-level locking or a distributed transaction manager.
**Patterns:** Optimistic locking (versioning) or Distributed Locking (Redis/Zookeeper) to reserve stock before committing.

AA9 — **Sidecar Proxy:** A container deployed alongside the main application container to handle network traffic.
**Envoy/Istio:** Envoy handles the data plane (traffic routing, mTLS). The Control Plane (Istiod) configures Envoy.
**mTLS:** Application-level TLS secures data in transit. mTLS between sidecars authenticates the *service identity* (not just the host) and encrypts traffic without code changes, enabling zero-trust security.

AA10 — **Strangler Fig:** Incrementally migrate functionality from a legacy system to new services by routing specific requests to the new system.
**Strategy:**
1.  Deploy API Gateway in front of the .NET Framework monolith.
2.  Build new .NET 8 microservices for specific features (e.g., "Checkout").
3.  Configure Gateway to route `/checkout` to the new service and the rest to the monolith.
4.  Gradually migrate more endpoints until the monolith is "strangled" and removed.

AA11 — **12-Factor App:** Methodology for building SaaS apps.
**6 Factors:**
1.  **Codebase:** One codebase tracked in version control.
2.  **Dependencies:** Explicitly declare and isolate dependencies.
3.  **Config:** Store config in environment variables.
4.  **Backing Services:** Treat backing services (DBs) as attached resources.
5.  **Build/Run/Deploy:** Strict separation of build and run stages.
6.  **Processes:** Execute the app as one or more stateless processes.
**Why:** Ensures portability, scalability, and ease of deployment in cloud environments.

AA12 — **Event-Driven (Kafka) vs. Request-Driven (REST):**
*   **Coupling:** Event-driven is loosely coupled; Request-driven is tightly coupled.
*   **Scalability:** Event-driven scales better for high throughput; Request-driven is simpler for low latency.
*   **Debugging:** Request-driven is easier to trace; Event-driven requires distributed tracing.
*   **Ordering:** Kafka guarantees ordering per partition; REST does not.
*   **Failure:** Event-driven handles failures via retries/replay; Request-driven fails immediately.
**Choice:** Use Event-driven for decoupled, high-volume data pipelines. Use Request-driven for real-time user interactions requiring immediate feedback.

AA13 — **Load Balancer:** Distributes incoming network traffic across multiple servers.
**L4 vs. L7:** L4 (Transport) routes based on IP/Port. L7 (Application) routes based on HTTP headers/URL.
**Use Case:** L7 is required for SSL termination or routing `/api` to backend A and `/static` to backend B.

AA14 — **Message Queue:** Stores messages until consumed.
**Queue (RabbitMQ) vs. Log (Kafka):** Queues delete messages after consumption (tasks). Logs retain messages for replay (streaming).
**Pick:** RabbitMQ for task distribution (email sending). Kafka for event streaming (analytics, audit logs).

AA15 — **Container vs. VM:** Containers share the host OS kernel; VMs virtualize hardware with a full guest OS.
**Popularity:** Containers are lightweight, start instantly, and have consistent environments, making them ideal for microservices scaling.

AA16 — **Stateless vs. Stateful:** Stateless services store no session data (any instance handles any request). Stateful services store session/data locally.
**Scaling:** Stateless is easier to scale horizontally because any instance can handle any request without data synchronization.
**Examples:** Stateless: Web Server. Stateful: Database.

AA17 — **ORM:** Maps database tables to programming language objects.
**Examples:** Hibernate (Java), Entity Framework (.NET).
**Trade-offs:** ORMs increase developer productivity but can generate inefficient SQL and hide database complexity. Raw SQL offers performance control but is verbose and error-prone.

AA18 — **Bulkhead Pattern:** Isolates resources (threads, connections) for specific services to prevent a failure in one from consuming all resources.
**Implementation:** Use separate thread pools for different service calls in code, or deploy services on separate Kubernetes nodes/pods to isolate CPU/Memory.

AA19 — **OT vs. CRDT:** OT requires a central server to resolve conflicts sequentially. CRDTs allow local edits that mathematically converge without coordination.
**Choice:** **CRDTs**.
**Why:** CRDTs support true offline editing (conflicts resolve locally) and reduce server load, whereas OT requires constant server connectivity for conflict resolution.

AA20 — **Distributed Tracing Challenges:**
*   **Context Propagation:** Inject `TraceID`/`SpanID` into headers (W3C/B3).
*   **Sampling:** Head-based (sample at start) reduces storage; Tail-based (sample errors) improves debugging.
*   **Storage:** Jaeger/Tempo store spans; querying requires indexing.
*   **Kafka:** Trace context must be injected into message headers. Consumers must extract and continue the trace.
*   **Continuity:** Ensure all services (including async consumers) propagate the TraceID in headers.

OP1 — **Hypervisor Type 1 vs. Type 2:** Type 1 runs directly on hardware (Bare-metal). Type 2 runs on top of an OS (Hosted).
**Examples:** Type 1: VMware ESXi, Microsoft Hyper-V. Type 2: VirtualBox, VMware Workstation.

OP2 — **SAN vs. NAS:** SAN provides block-level storage (appears as a disk). NAS provides file-level storage (appears as a folder).
**Usage:** SAN for databases (high I/O). NAS for file shares.
**Protocols:** SAN: iSCSI, Fibre Channel. NAS: NFS, SMB/CIFS.

OP3 — **3-Tier Network:** Core (Backbone), Distribution (Policy), Access (Endpoints).
**Leaf-Spine:** Preferred in modern DCs. Solves oversubscription and latency issues of 3-tier by providing non-blocking, equal-cost paths between any two nodes.

OP4 — **Proxmox HA:**
*   **Nodes:** Minimum 3 for quorum.
*   **Quorum:** Majority vote required to make decisions.
*   **Fencing:** STONITH (e.g., iDRAC) to power off failed nodes.
*   **Storage:** Ceph (distributed) or Shared iSCSI/NFS.
*   **Failure:** VMs restart on surviving nodes automatically.

OP5 — **VLAN Trunking:** Carries multiple VLAN traffic over one link using 802.1Q tags.
**Linux Config:** Create sub-interfaces (e.g., `eth0.10`, `eth0.20`) with `ip link add link eth0 name eth0.10 type vlan id 10` and assign IPs.

OP6 — **RAID Levels:**
*   **RAID 1:** Mirroring (50% capacity).
*   **RAID 5:** Striping + Parity (N-1 capacity, 1 drive fault).
*   **RAID 6:** Striping + Double Parity (N-2 capacity, 2 drive fault).
*   **RAID 10:** Mirroring + Striping (50% capacity, fast).
*   **RAID Z2:** ZFS double parity (similar to RAID 6).
**Choice:** **RAID 10** (or Z2). Best balance of performance and fault tolerance for a DB server with 6 drives.

OP7 — **DNS Architecture:**
*   **Internal:** AD Integrated DNS (Active Directory).
*   **External:** Public DNS (e.g., Route53) with DNSSEC.
*   **Split-Horizon:** Internal clients resolve internal IPs; external resolve public IPs.
*   **Forwarders:** Internal DNS forwards external queries to public resolvers.
*   **Laptop:** Connects via DHCP. If on LAN, resolves internal names. If on VPN, DNS tunneling or split-horizon ensures correct resolution.

OP8 — **PKI Chain:** Root CA (Offline) -> Subordinate CA (Issuing) -> End Entity.
**Internal PKI:** Root kept offline. Subordinate issues certs. Use CRL/OCSP for revocation.
**Compromise:** Root compromised = Trust collapse (revoke all). Subordinate compromised = Revoke subordinate, issue new certs, trust remains.

OP9 — **WAN Consolidation:**
*   **MPLS:** High cost, high reliability.
*   **IPsec:** Low cost, variable reliability.
*   **SD-WAN:** Low cost, intelligent routing, QoS.
*   **WireGuard:** Simple, secure, mesh.
**Recommendation:** **SD-WAN**. Best for cost, management, and VoIP QoS across 5 sites.

OP10 — **iDRAC/iLO/IPMI:** Out-of-band management interfaces for hardware control (power, console).
**Critical:** Allows recovery if OS is down.
**Exposed to Internet:** High risk of remote code execution (RCE) if credentials are weak or unpatched.

OP11 — **LVM vs. ZFS:** LVM manages block devices. ZFS manages filesystem and storage pool.
**Snapshots:** LVM (Copy-on-Write, slower). ZFS (Instant, cheap, immutable).
**Usage:** LVM for simple volume management. ZFS for data integrity and snapshots.

OP12 — **Stretch Cluster:**
*   **Nodes:** 3 nodes (2 per site) + 1 Witness (tiebreaker).
*   **Ceph:** CRUSH rules must distribute replicas across sites.
*   **Latency:** <5ms recommended for monitors.
*   **Split-Brain:** Without a witness, both sites might think they are primary.
*   **Why Bad:** 2-site Ceph without tiebreaker risks data corruption during network partition.

OP13 — **DHCP:** Dynamic Host Configuration Protocol.
**DORA:** Discover, Offer, Request, Acknowledge.
**Unavailable:** Client uses APIPA (169.254.x.x) or retries lease. If lease expires, client broadcasts Discover again.

OP14 — **Managed vs. Unmanaged:** Unmanaged is plug-and-play. Managed allows configuration.
**Features:** VLANs, QoS, Port Mirroring, ACLs.
**Importance:** Critical for segmentation, traffic prioritization, and security in enterprise.

OP15 — **NTP:** Network Time Protocol.
**Critical:** Clusters (Kubernetes, DBs) rely on time for consensus and logging.
**Drift:** Clock skew causes replication failures, certificate validation errors, and log analysis issues.

OP16 — **Physical vs. VM vs. Container:** Physical = Bare metal. VM = Virtualized OS. Container = Virtualized App.
**Bare Metal:** Use for high I/O, low latency, or hardware-specific workloads (e.g., GPU compute, high-performance DB).

OP17 — **UPS vs. Generator:** UPS provides short-term power (minutes) for graceful shutdown or bridge. Generator provides long-term power (hours/days).
**ATS:** Automatic Transfer Switch switches power source from Utility to Generator when utility fails.

OP18 — **Active-Passive vs. Active-Active:** Active-Passive (one handles load, one waits). Active-Active (both handle load).
**PostgreSQL HA:** Pacemaker/Corosync cluster.
**Fencing:** STONITH ensures failed node is powered off.
**Failover:** Virtual IP moves to survivor node.

OP19 — **Backup Strategy:**
*   **3-2-1:** 3 copies, 2 media types, 1 offsite.
*   **RPO/RTO:** Recovery Point/Time Objective.
*   **Types:** Full, Incremental, Differential.
*   **Immutable:** Prevents ransomware deletion.
*   **Test:** Regular restore drills.
*   **Backup vs. DR:** Backup is data copy; DR is the plan/process to restore operations.

OP20 — **802.1X:** Port-based Network Access Control.
**Roles:** Supplicant (Client), Authenticator (Switch), Server (RADIUS).
**EAP:** EAP-TLS (Certificate) vs PEAP (Password).
**Dynamic VLAN:** Assigns VLAN based on user identity.
**Failure:** Device placed in Quarantine VLAN.
**MAB:** Uses MAC address as credential for non-802.1X devices (less secure).
