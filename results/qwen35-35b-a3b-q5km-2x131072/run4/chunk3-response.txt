T1 — **Plan** performs a dry run, calculating the execution plan and showing what changes will be made without modifying infrastructure. **Apply** executes the plan, creating, updating, or destroying resources to match the configuration.

T2 — **State** is a JSON file mapping real-world resources to your configuration. **Remote state** is crucial for collaboration (sharing state among team members), security (storing secrets away from local machines), and enabling state locking to prevent concurrent modifications.

T3 — **count** uses integer indices to create multiple instances of a resource. **for_each** uses a map or set of keys. **for_each** is preferred when resource names are dynamic or non-sequential, as it avoids index shifting issues when items are added or removed from the collection.

T4 — This is called **state drift**. OpenTofu detects it during `tofu plan` by comparing the state file against the live infrastructure. It will show the resource as "drifted" (changes detected but not in config). `tofu apply` can reconcile this if the drift is acceptable or if the config is updated to match.

T5 —
```hcl
resource "aws_instance" "web" {
  for_each = toset(["web1", "web2", "web3"])
  ami      = "ami-12345678"
  instance_type = "t2.micro"
  tags = {
    Name = each.key
  }
}
```

T6 — `terraform_remote_state` reads the entire state file of another configuration, which is a security risk and inefficient. **Output values** (passed via module outputs or Terraform Cloud workspaces) are safer and more efficient. The recommended alternative is using **Module Outputs** passed directly between modules in the same run, or `data "terraform_remote_state"` with the `outputs` argument (1.5+) to fetch only specific values if cross-project.

T7 — The `null_resource` provisioner runs during the apply phase, but the `output` is evaluated before the provisioner finishes writing the file. **Fix:** Add `depends_on = [null_resource.kubeconfig]` to the resource consuming the kubeconfig, or ensure the output reads the file content (`file()`) and the dependent resource depends on the `null_resource`.

T8 — `tofu import` manually adds existing resources to state. Limitations: No history, manual mapping required. **Import block** (1.5+) allows importing resources declaratively in the config, making imports version-controlled, repeatable, and easier to manage within CI/CD pipelines.

T9 — A **variable** is an input parameter passed into the configuration (e.g., from CLI or `.tfvars`). An **output** is a value exposed by the configuration to be used by other modules or the CLI.

T10 — A **provider** is a plugin that interacts with a specific API (e.g., AWS, Azure). Without `tofu init`, the provider plugins are not downloaded, causing `tofu plan` to fail with "Provider not installed" errors.

T11 — The `.terraform.lock.hcl` file records the exact versions of providers and plugins used. It should be committed to ensure **reproducible builds** and prevent unexpected breaking changes from provider updates.

T12 — `tofu destroy` removes all managed resources. To target a single resource, use the `-target` flag: `tofu destroy -target=aws_instance.example`.

T13 — A **data source** reads information from an external system without creating or managing it. Use one when you need existing data (e.g., `data "aws_ami" "ubuntu"` to find the latest Ubuntu AMI) rather than creating a new resource.

T14 — **Variables** are inputs from outside the module. **Locals** are computed values within the module. Use **locals** for intermediate calculations, formatting, or grouping values that don't need to be exposed as inputs.

T15 — The **lifecycle** block controls resource behavior.
*   `create_before_destroy`: Creates new resource before destroying old one (zero downtime).
*   `prevent_destroy`: Prevents accidental deletion (useful for critical data).
*   `ignore_changes`: Ignores drift on specific attributes (e.g., tags managed externally).

T16 — The **moved** block allows renaming or moving resources between modules without destroying and recreating them. It updates the state file mapping so OpenTofu recognizes the resource as the same entity despite the address change.

T17 — The **dependency graph** is a DAG determining execution order. **Implicit** dependencies are inferred from resource references. **Explicit** dependencies use `depends_on`. `depends_on` forces serialization, reducing parallelism and slowing down execution, so it should be used only when implicit detection fails.

T18 — Strategies:
1.  **Increase `parallelism`**: Run more API calls concurrently (Tradeoff: API rate limits).
2.  **`refresh=false`**: Skip state refresh (Tradeoff: May miss external drift).
3.  **Modularize State**: Split large state into smaller files (Tradeoff: More complex state management).
4.  **Reduce Data Sources**: Minimize external lookups (Tradeoff: May require manual AMI/ID management).

T19 — State locking prevents concurrent writes using a backend (e.g., DynamoDB). If a lock isn't released, subsequent commands fail. **Break lock** with `tofu force-unlock <ID>`. **Risk**: Data corruption if another process is actually running the apply.

T20 — Steps:
1.  Backup local state (`cp .terraform/terraform.tfstate backup.tfstate`).
2.  Configure `backend "s3"` in `main.tf`.
3.  Run `tofu init -migrate-state`.
4.  Verify state in S3 console and run `tofu state list` to confirm resources exist.

A1 — A **Playbook** is a YAML file defining a sequence of tasks. A **Role** is a reusable package of tasks, handlers, and templates. A **Task** is a single action (e.g., install a package).

A2 — `ansible_facts` is a dictionary of gathered system information. Access OS family via `{{ ansible_os_family }}`.

A3 — **command**: Executes without shell (no pipes/globs). **shell**: Executes with shell (supports pipes/globs). **raw**: Executes without setup (no Python interpreter), useful for initial bootstrap.

A4 — **Idempotency** means running a task multiple times yields the same result. **Idempotent**: `apt: name=nginx state=present`. **Not Idempotent**: `shell: echo "hello" > /tmp/test` (overwrites file every time).

A5 —
```yaml
- name: Install htop on Debian
  package:
    name: htop
    state: present
  when: ansible_os_family == "Debian"

- name: Install epel-release and htop on RedHat
  package:
    name:
      - epel-release
      - htop
    state: present
  when: ansible_os_family == "RedHat"
```

A6 — Precedence (High to Low): Extra vars > Command line > Play vars > Role vars > Group vars > Host vars > Default.

A7 — Use `serial: 3` in the play to process 3 nodes at a time. Group nodes in inventory. Use `wait_for` (e.g., `uri` module) to ensure the first 3 nodes are healthy before the next batch starts.

A8 — **ansible-vault** encrypts sensitive data. To encrypt a single variable, use `ansible-vault edit vars.yml` and encrypt specific lines, or use `set_fact` with a vaulted variable in a playbook.

A9 — An **inventory** lists managed hosts. **Static** is a file (INI/YAML). **Dynamic** is a script/API that generates inventory dynamically (e.g., AWS EC2).

A10 — `ansible webservers -m shell -a "df -h"`

A11 — A **handler** is a task that runs only when notified by another task (e.g., restart service). Triggered via `notify: handler_name`.

A12 — **ansible.cfg** configures Ansible behavior. Common settings: `host_key_checking`, `forks` (concurrent hosts), `remote_user`.

A13 — **register** captures task output into a variable. Use it in subsequent tasks via `{{ registered_var.stdout }}` or in `when` clauses.

A14 — **import_tasks** is static (evaluated at parse time). **include_tasks** is dynamic (evaluated at runtime). `include_tasks` supports `when` clauses on the import statement itself; `import_tasks` does not.

A15 — **Ansible Galaxy** is a repository for roles/collections. Use `requirements.yml` to list them, then run `ansible-galaxy install -r requirements.yml`.

A16 — **Connection plugins** define how Ansible connects. **ssh**: Remote Linux/Unix. **local**: Control node itself. **network_cli**: Network devices (Cisco/Juniper).

A17 — **delegate_to** runs a task on a specific host. **run_once** runs a task once across all hosts. Example: Gather facts on a remote host, then write a config file to `localhost` using `delegate_to: localhost`.

A18 — Strategies:
1.  **Increase `forks`**: More concurrent hosts (Tradeoff: Memory/CPU on control node).
2.  **Enable `pipelining`**: Reduces SSH connections (Tradeoff: SSH config requirements).
3.  **Fact Caching**: Avoids gathering facts every run (Tradeoff: Stale facts).
4.  **Mitogen**: Optimized execution engine (Tradeoff: Installation complexity).

A19 — Jinja2 templates use `{{ }}` for expressions, `{% %}` for control flow (loops/if), `{# #}` for comments.
Snippet:
```jinja2
upstream backend {
{% for server in backend_servers %}
    server {{ server }}; # Health check: {{ server_health_check }}
{% endfor %}
}
```

A20 —
1.  **Serial**: `serial: 1` (or batch size).
2.  **Pre-task**: Deregister from LB (`aws_elb_deregister`).
3.  **Task**: Deploy app.
4.  **Post-task**: Register to LB + `uri` health check.
5.  **Block/Rescue**: If health check fails, trigger `rescue` to rollback (re-register old version).
