DN1 — **Class vs Struct**: `class` is a reference type (heap, nullable, inheritance), `struct` is a value type (stack/heap, non-nullable, no inheritance). Use `struct` for small, immutable data representing a single value (e.g., `Point`, `Color`) to avoid allocation and ensure value semantics.

DN2 — **IEnumerable<T> vs IQueryable<T>**: `IEnumerable<T>` executes queries in-memory (client-side). `IQueryable<T>` builds expression trees translated to server-side queries (e.g., SQL). This matters for DB queries because `IQueryable` pushes filtering/sorting to the database, reducing data transfer and improving performance.

DN3 — **Async/Await & Task vs ValueTask**: `async/await` creates a state machine to resume execution without blocking threads. `Task` is a reference type (heap allocation). `ValueTask` is a struct (stack allocation) that can wrap a `Task` or a synchronous result. Use `ValueTask` in hot paths where the result is often available synchronously to avoid allocation overhead.

DN4 — **Dependency Injection (DI)**: DI decouples object creation from usage. Lifetimes: **Transient** (new instance every time), **Scoped** (new per request), **Singleton** (single instance per app). **Bug**: Injecting a Scoped service into a Singleton creates a "Captive Dependency," causing the Singleton to hold a stale Scoped instance, leading to memory leaks or stale data.

DN5 — **string vs ReadOnlySpan<char>**: `string` is immutable and heap-allocated. `ReadOnlySpan<char>` is a stack-only view into memory (no allocation). Use `Span<T>` for high-performance string manipulation (parsing, slicing) to avoid creating intermediate string objects and reduce GC pressure.

DN6 — **Garbage Collector (GC)**: GC manages memory by reclaiming unreachable objects. **Generations**: Gen 0 (short-lived), Gen 1 (transient), Gen 2 (long-lived). **LOH**: Large Object Heap (objects > 85KB) not compacted frequently. **IDisposable**: Needed for unmanaged resources (files, DB connections) because GC only manages managed memory; `Dispose` ensures timely cleanup via `using` blocks.

DN7 — **ConfigureAwait(false)**: Default behavior captures the `SynchronizationContext` (e.g., UI thread) to resume on the same context. `ConfigureAwait(false)` skips this. Use `false` in libraries to avoid deadlocks and improve performance. Do NOT use `false` in UI/ASP.NET Core if you need to update UI controls or rely on `HttpContext` after the await.

DN8 — **.NET 8/9 Features**: **Primary Constructors** (C# 12): `public class Service(int id) : IService { }`. **Collection Expressions**: `[1, 2, 3]` creates arrays/lists. **FrozenDictionary**: Immutable, high-performance dictionary. Example: `public class User(int Id, string Name) { }` injected via DI.

DN9 — **DbContext Concurrency Issue**: Likely cause: `DbContext` is registered as Singleton but is not thread-safe. Under high concurrency, multiple requests share the same context, causing race conditions and `ObjectDisposedException`. Fix: Register `DbContext` as **Scoped** (per HTTP request) so each request gets a fresh, thread-safe context.

DN10 — **Record vs Class**: `class` is reference-equality. `record` is value-equality (compares properties). `record struct` is value-type + value-equality. Choose `record` for immutable DTOs/Models where equality matters. Choose `class` for mutable entities or when reference identity is required.

DN11 — **ref, out, in**: `ref` passes by reference (must be initialized before call, can modify). `out` passes by reference (must be assigned inside method). `in` passes by reference (read-only, no modification). Use `ref` for swapping, `out` for multiple return values, `in` for large structs to avoid copy.

DN12 — **Nullable Reference Types (NRT)**: `?` annotation warns about potential null dereferences at compile time (e.g., `string?`). Differs from `int?` (nullable value type) which is a struct wrapper (`Nullable<T>`) allowing `null` at runtime. NRTs are for reference types only and rely on compiler analysis.

DN13 — **LINQ**: Language Integrated Query. **Method Syntax**: `list.Where(...)`. **Query Syntax**: `from x in list where ...`. Both compile to same IL. **Deferred Execution**: Query is not executed until enumerated (e.g., `ToList()`). `IQueryable` executes on DB; `IEnumerable` executes in memory.

DN14 — **Generic Constraints**: `where T : class` (reference type), `where T : struct` (value type), `where T : new()` (paramless constructor), `where T : IComparable<T>` (interface). Combine with `&` (e.g., `where T : class, IComparable<T>`). Use to enforce capabilities on generic types.

DN15 — **Abstract Class vs Interface**: Abstract class provides partial implementation + contract. Interface provides contract only. **C# 8+ Default Interface Methods**: Interfaces can now have method implementations, blurring the line. Use abstract for shared code + contract; interface for capability contracts.

DN16 — **Middleware Pipeline**: Ordered components processing requests. `app.Use()` adds middleware (calls `next`). `app.Map()` branches pipeline based on path. `app.Run()` is terminal (no `next`). Ordering matters (Auth before Routing). Exception handling middleware catches exceptions and returns error responses.

DN17 — **Channel<T>**: Async-first producer-consumer queue. `BlockingCollection<T>` is blocking-first. **Bounded Channels**: Set `maxItems` to create backpressure (producer waits if full). Use `Channel<T>` for high-throughput async pipelines; `BlockingCollection` for legacy blocking scenarios.

DN18 — **Source Generators**: Compile-time code generation. Differs from Reflection (runtime) and IL Weaving (post-build). **Incremental Generator**: Uses `IncrementalGeneratorInitializationContext` to cache results and only regenerate when source changes. Preferred over `ISourceGenerator` for performance and incremental compilation support.

DN19 — **TPL Dataflow**: High-level pipeline library. `ActionBlock` (consume), `TransformBlock` (transform), `BufferBlock` (store). Compared to `Channels`, Dataflow offers built-in backpressure, completion signaling, and complex graph orchestration. Use Dataflow for complex workflows; Channels for simple streams.

DN20 — **Minimal APIs**: API endpoints defined via delegates (`app.MapGet(...)`). No Controllers. **Trade-offs**: Less boilerplate, faster startup, but harder to scale complex auth/validation. **Route Groups**: `MapGroup` organizes endpoints. **Filters**: `EndpointFilter` injects dependencies into endpoints. OpenAPI via Swashbuckle.

P1 — **List vs Tuple**: `list` is mutable, ordered collection. `tuple` is immutable, ordered collection. `set` is mutable, unordered, unique elements. `frozenset` is immutable set (hashable, can be dict key). Use tuple for fixed data, set for uniqueness.

P2 — **GIL**: Global Interpreter Lock allows only one thread to execute Python bytecode at a time. Affects **CPU-bound** workloads (threads contend). Workarounds: **Multiprocessing** (separate processes), **C-extensions** (release GIL), or **Asyncio** (I/O bound).

P3 — **Deepcopy vs Copy**: `copy` (shallow) copies the object but references nested objects. `deepcopy` recursively copies nested objects. **Shallow bug**: Modifying a nested mutable object in the copy affects the original. Use `deepcopy` for complex nested structures.

P4 — **Decorators**: Functions wrapping other functions. Example:
```python
import time
def log_time(func):
    def wrapper(*args, **kwargs):
        start = time.perf_counter()
        result = func(*args, **kwargs)
        print(f"{func.__name__} took {time.perf_counter() - start}s")
        return result
    return wrapper
```

P5 — **Generators & Yield**: Generators produce values lazily. `yield` pauses execution and returns a value. Differs from `return` (list) which loads all data into memory. `yield from` delegates to another generator. Use generators for large datasets or infinite streams.

P6 — **Asyncio Event Loop**: Manages async tasks. `asyncio.run()` creates loop and runs entry point. `asyncio.gather()` waits for multiple tasks. `asyncio.create_task()` schedules a task. `time.sleep()` blocks the loop; use `await asyncio.sleep()` to yield control.

P7 — **MRO (Method Resolution Order)**: Python 3 uses **C3 Linearization**. For `D(B, C)`, MRO is `D -> B -> C -> A`. `D().who()` calls `C.who()` because `C` overrides `A` and appears after `B` in the MRO. Output: `"C"`.

P8 — **Metaclass vs __init_subclass__**: Metaclass controls class creation (e.g., enforcing structure). `__init_subclass__` is a simpler hook called when a class is subclassed. Use Metaclass for framework-level enforcement (e.g., Django models). Use `__init_subclass__` for simpler subclass customization.

P9 — **Multiprocessing vs Threading**: **Multiprocessing** (processes, bypasses GIL, CPU-bound, requires pickling). **Threading** (threads, GIL, I/O-bound, shared memory). **concurrent.futures** provides high-level API for both. Pickling requires objects to be importable at top level.

P10 — **__slots__**: Prevents creation of `__dict__` for instances. Saves memory (no dict overhead) and prevents adding dynamic attributes. Use when creating many instances of a class with fixed attributes.

P11 — **Virtual Environment**: Isolates project dependencies. `venv` (stdlib, Python 3.3+), `virtualenv` (older, 3rd party), `conda` (package manager + env, binary packages). Use one to avoid dependency conflicts between projects.

P12 — **Comprehensions**: Concise syntax for creating lists/dicts/sets. `map()`/`filter()` are functional. Prefer comprehensions for readability. Use `map()`/`filter()` when functions are already defined or for lazy evaluation in Python 3.

P13 — **== vs is**: `==` checks value equality. `is` checks object identity. **Object Interning**: Small integers (-5 to 256) are cached. `a is b` returns True for small ints because they reference the same object.

P14 — **args/kwargs**: `*args` (positional tuple), `**kwargs` (keyword dict). `/` (positional-only), `*` (keyword-only). Example: `def func(a, /, b, *, c):`. Use to create flexible APIs or forward arguments.

P15 — **Context Managers**: Manage resources (setup/teardown). `with` statement. Implement via `__enter__`/`__exit__` or `@contextmanager`. Example: Acquiring a lock or creating a temp directory. Ensures cleanup even on exceptions.

P16 — **Type Hinting**: Static type annotations. `Union`, `Optional`, `Literal`, `TypeVar` (generics), `Protocol` (structural typing). `mypy`/`pyright` use these to catch type errors before runtime. Improves IDE support and documentation.

P17 — **Data Model (Dunders)**: `__repr__` (dev/debug), `__str__` (user). `__eq__` defines equality. Defining `__eq__` without `__hash__` makes object unhashable (cannot use in sets/dicts). `__getattr__` (fallback), `__getattribute__` (intercept all access).

P18 — **Descriptors**: Objects implementing `__get__`, `__set__`, `__delete__`. Used by `property`, `classmethod`, `staticmethod`. Under the hood, accessing `obj.prop` calls `type(obj).prop.__get__(obj, type(obj))`. Allows custom attribute logic.

P19 — **Import System**: `sys.path` lists directories. **Finders** (search paths), **Loaders** (load modules). Relative imports (`from . import x`) vs Absolute (`from pkg import x`). `__init__.py` marks packages (implicit in 3.3+). Namespace packages allow splitting a package across directories.

P20 — **Dataclasses vs Pydantic**: `dataclasses` auto-generates `__init__`, `__repr__`. `attrs` is more feature-rich. `Pydantic` adds validation and serialization. Use `__post_init__` for post-init logic. Choose Pydantic for API models requiring validation (e.g., FastAPI).
