CL1 — **IaaS (Infrastructure as a Service)** provides virtualized computing resources (e.g., AWS EC2). **PaaS (Platform as a Service)** provides a platform for developing/runing apps without managing underlying infra (e.g., AWS Elastic Beanstalk). **SaaS (Software as a Service)** delivers software over the internet (e.g., Microsoft 365).

CL2 — **Shared Responsibility Model:** Provider secures the cloud (hardware, physical, hypervisor); Customer secures what is in the cloud (OS, data, config). **IaaS VM:** Provider secures host; Customer secures OS, patching, app. **Managed K8s:** Provider secures control plane; Customer secures nodes, pods, network policies. **SaaS (M365):** Provider secures app/infra; Customer secures identity, data classification, access policies.

CL3 — **VPC:** Isolated virtual network. **Subnet:** IP range within VPC. **Security Group:** Instance-level firewall (stateful). **NACL:** Subnet-level firewall (stateless). **Layering:** VPC contains Subnets; Subnets contain Instances protected by SGs and NACLs. **Stateful:** SG allows return traffic automatically. **Stateless:** NACL requires explicit allow rules for both request and response.

CL4 — **Landing Zone:** Secure baseline environment. **OUs:** Group accounts by function/environment. **Guardrails:** Preventive controls (e.g., deny public S3). **Shared Networking:** Transit Gateway/Hub-Spoke connects VPCs centrally. **Centralized Logging:** Aggregates logs for visibility. **Single-Account Problems:** High blast radius, billing opacity, permission sprawl, compliance risk.

CL5 — **IaC:** Managing infra via code. **Terraform/OpenTofu:** Multi-cloud, declarative, state file. **Pulumi:** Code-based (TS/Python). **CloudFormation/Bicep:** AWS/Azure native, tight integration. **Crossplane:** K8s API for infra. **Choice:** Terraform for multi-cloud, CFN/Bicep for single-cloud native, Pulumi for devs, Crossplane for K8s-centric orgs.

CL6 — **HPA:** Scales pods based on CPU/Mem. **VPA:** Scales pod resources (CPU/RAM). **Cluster Autoscaler:** Adds/removes nodes. **KEDA:** Event-driven scaling (queues, metrics). **KEDA over HPA:** Use KEDA when scaling based on external events (e.g., Kafka message count) rather than just resource utilization.

CL7 — **DR Strategy:** Define RPO (data loss) and RTO (downtime). **Active-Active:** Low RTO/RPO, high cost. **Active-Passive:** Higher RTO, lower cost. **DB:** Async (lag risk) vs Sync (latency risk). **DNS:** Route53 failover routing. **State:** Externalized (S3/DB) vs Local. **Pilot Light:** DB running, app minimal. **Warm Standby:** App running scaled down. **Active-Active:** Full redundancy.

CL8 — **GitOps:** Git is source of truth. **ArgoCD:** Push/Pull, UI focus. **FluxCD:** Pull-based, K8s native. **Secrets:** Sealed Secrets, External Secrets Operator, Vault. **Drift:** Controller detects manual changes and reconciles back to Git state automatically.

CL9 — **Architecture:** Use **StatefulSet** for stable IDs. **Local PV** for low latency (no network). **Headless Service** for DNS discovery. **Init Containers** for setup. **ConfigMaps** mounted as volumes. **PDB** ensures availability during updates. Ensure `volumeClaimTemplates` match storage class.

CL10 — **SA:** K8s identity for pods. **Workload Identity:** Maps SA to Cloud Identity. **IRSA:** AWS specific mapping. **Mounting Creds:** Bad practice (risk of leakage, rotation difficulty). **Federation:** Uses OIDC to exchange SA token for short-lived cloud credentials, eliminating static secrets.

CL11 — **Object (S3):** Unstructured data, high durability (11 9s), low latency. **Block (EBS):** Attached to VM, low latency, high IOPS. **File (EFS):** Shared file system, network latency. **Use:** Object for backups/media, Block for OS/DB, File for shared app config.

CL12 — **Zero Trust:** Verify every request. **Network Policies:** Restrict pod traffic. **Service Mesh:** mTLS for L7 encryption. **Pod Identity:** Least privilege access. **Admission Controllers:** Enforce policies (OPA). **Image Signing:** Verify integrity. **Runtime:** Falco detects anomalies. **Diff:** Traditional relies on perimeter; Zero Trust assumes breach and verifies identity/context.

CL13 — **CDN:** Distributed edge cache. **CloudFront, Cloudflare.** **Use CDN:** For static assets, global latency reduction, DDoS mitigation. **Origin:** For dynamic, unique, or real-time data.

CL14 — **Region:** Geographic area. **AZ:** Isolated data center within Region. **Multi-AZ:** Protects against data center failure. **Zonal:** EBS (tied to AZ). **Regional:** ELB (spans AZs).

CL15 — **Managed DB:** RDS/Cloud SQL. **Provider Handles:** Patching, backups, HA, failover, scaling. **Customer Handles:** Schema, queries, access control. Reduces operational overhead significantly.

CL16 — **IAM:** Users (People), Groups (Roles), Roles (Assume), Policies (Permissions). **Least Privilege:** Grant only necessary access. **Apply:** Use Groups/Roles, not Users. Avoid wildcard permissions. Regularly audit.

CL17 — **Serverless:** Event-driven, no server mgmt. **Lambda/Functions.** **Cold Start:** Initialization latency on first invocation. **Poor Fit:** Long-running processes, stateful apps, strict latency requirements.

CL18 — **Ingress:** K8s resource for HTTP routing. **Gateway API:** Extensible, multi-cluster. **LB:** Cloud provider load balancer. **Mesh:** Istio ingress. **TLS:** Ingress (Edge), LB (Edge), Mesh (mTLS). **Gateway API:** Use for complex routing, multi-cluster, or when Ingress is too limited.

CL19 — **Strategy:** **RI/SP:** Commit for baseline (30-40% savings). **Spot:** For fault-tolerant workloads (60-90% savings). **Right-sizing:** Match instance to usage. **Tiering:** Move old data to Glacier. **Idle Detection:** Stop unused resources. **FinOps:** Cross-team cost accountability. **Tools:** Cost Explorer, Kubecost. **Showback:** Charge teams for usage to drive behavior.

CL20 — **Strategy:** **Abstraction:** K8s, Terraform. **Networking:** Direct Connect/ExpressRoute. **Identity:** OIDC federation. **Trade-offs:** Complexity vs. Risk. **Justified:** Compliance, Vendor Lock-in avoidance, Geo redundancy. **Unnecessary:** If single cloud meets all needs (cost/complexity outweighs benefits).

OT1 — **Purdue Model:** ISA-95 standard. **L0:** Physical Process (Sensors/Actuators). **L1:** Basic Control (PLCs). **L2:** Area Supervisory (SCADA/HMI). **L3:** Site Operations (MES/Historian). **L4:** Enterprise (ERP).

OT2 — **IT:** Data confidentiality, patching, availability. **OT:** Safety, availability, legacy hardware. **Patching:** OT systems often run on unsupported OS; patching risks downtime or safety failure. Requires validation.

OT3 — **DMZ:** Buffer zone. **Traffic:** Historian pulls from SCADA (DMZ to OT), Engineering pushes configs (IT to DMZ). **Direct L4 to L1:** Dangerous because IT threats can directly reach control logic without inspection, risking physical safety.

OT4 — **Modbus TCP:** L1/2, simple, unencrypted. **OPC UA:** L2/3, secure, semantic. **EtherNet/IP:** L1/2, CIP protocol. **PROFINET:** L1/2, real-time industrial Ethernet. **MQTT:** L2/3/Cloud, lightweight pub/sub.

OT5 — **Data Diode:** Hardware allowing one-way data flow. **Firewall:** Two-way filtering. **Use Diode:** When data must leave OT but nothing can enter (e.g., export to cloud). **Limitation:** No feedback loop, cannot send commands back.

OT6 — **Flow:** PLC -> Historian -> OPC UA Server -> Edge Gateway -> MQTT Broker -> Cloud. **Security:** Firewall between Edge and Cloud. DMZ for OPC UA. Encrypt MQTT. Isolate Historian from direct internet.

OT7 — **IEC 62443:** **Zone:** Logical grouping of assets with similar security requirements. **Conduit:** Traffic path between zones. **Model:** Corporate (Zone 1), Engineering (Zone 2), SCADA (Zone 3), PLCs (Zone 4). Conduits enforce traffic rules between zones.

OT8 — **Segmentation:** VLANs per production line. **Firewalls:** Between VLANs and DMZ. **Switches:** L3 managed switches. **Vendor:** Jump host with MFA and session recording. **Transition:** Phased rollout, parallel run, maintenance windows.

OT9 — **SIS:** Independent safety system (stops process safely). **BPCS:** Basic control (maintains process). **IEC 61511:** Safety lifecycle standard. **Air-gap:** Prevents cyber attacks from disabling safety. **Ransomware on SIS:** Safety failure, potential physical harm, explosion, injury.

OT10 — **PLC:** Logic controller (L1). **DCS:** Integrated process control (L1/L2). **SCADA:** Supervisory monitoring (L2). **Coexist:** Yes, PLCs feed SCADA, DCS manages complex processes.

OT11 — **Architecture:** Jump host (bastion), MFA, Session Recording, Time-limited access. **Direct RDP:** Unacceptable because it bypasses audit, allows lateral movement, and exposes HMI to IT threats.

OT12 — **Plan:** Isolate compromised segment immediately. Notify ops. Preserve logs. **Ops Role:** Maintain safe state, manual override if needed. **Detection:** IDS for OT protocols. **Response:** Coordinate with vendor, patch, restore from clean image.

OT13 — **HMI:** Operator interface for monitoring/control. **Eng Workstation:** Used for programming/configuring PLCs. **Diff:** HMI is for runtime ops; Workstation is for engineering changes.

OT14 — **Historian:** Time-series database for process data. **Plant:** Local, real-time, short-term. **Enterprise:** Long-term, analytics, compliance.

OT15 — **Ladder Logic:** Visual programming resembling relay circuits. **Device:** PLC. **Dominant:** Reliable, visual, easy for electricians, legacy code base.

OT16 — **RTU:** Remote Terminal Unit. **Diff:** RTU optimized for comms/harsh env; PLC optimized for logic. **Deploy:** RTUs in pipelines, remote sites; PLCs in factories.

OT17 — **Analog:** Continuous signal (4-20mA). **Digital:** Discrete (On/Off). **4-20mA:** 4mA = zero, 0mA = wire break (live zero). 0-20mA cannot distinguish break from zero.

OT18 — **Inventory:** Foundation of security (know assets). **Tools:** Passive (Sniffing), Active (Scanning). **Active Dangerous:** Can crash legacy devices or trigger safety interlocks.

OT19 — **MITRE ICS:** Framework for ICS threats. **Tactics:** Initial Access, Execution, Persistence, Privilege Escalation, Impact. **Diff:** Focuses on physical safety/process. **Use:** Map controls to tactics to find gaps (e.g., no network segmentation).

OT20 — **SOC:** **Logs:** Syslog/NetFlow (non-intrusive). **SIEM:** Splunk/Elastic. **Rules:** OT-specific (e.g., PLC write). **Staff:** OT engineers for triage. **Alert:** Verify, isolate PLC, rollback firmware, investigate source. **False Positives:** Tune rules to avoid alert fatigue.
